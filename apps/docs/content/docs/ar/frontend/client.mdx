---

`@solana/kit` يُوفّر `createClient`، وهو مُضيف صغير تُوسّعه بالإضافات. تُضيف
كلُّ إضافة قدرةً جديدة — اتصال RPC، أو موقِّع، أو تخطيط المعاملات — والعميل
المُركَّب يدعم السكريبتات ومسارات الخادم وواجهات المستخدم على حدٍّ سواء.

<Callout type="info">
  هل تبني تطبيق React؟ يُغلِّف [دليل React](/docs/frontend/react-hooks) هذا
  العميل نفسه باستخدام `@solana/react` حتى تتمكّن المكوّنات من قراءة حالة
  المحفظة وبيانات RPC عبر الـ hooks. ابدأ من هناك وعُد إلى هنا عندما تحتاج إلى
  تحكّم أدق.
</Callout>

## التثبيت

```terminal
$ npm install @solana/kit @solana/kit-plugin-rpc @solana/kit-plugin-signer
```

استخدم أيَّ مدير حزم تشاء. Kit بإصدار 7 أو أعلى وحزم `@solana/kit-plugin-*`
بإصدار 0.13 أو أعلى.

## إنشاء عميل

اجمع الإضافات باستخدام `.use(...)`. اضبط الموقِّع أولاً، ثم اتصال RPC. تُضيف
`solanaDevnetRpc()` الـ `client.rpc` والاشتراكات ومُخطِّط/منفِّذ المعاملات
ومساعد `client.sendTransaction` — لذا نادراً ما يحتاج التطبيق إلى توصيل هذه
الأجزاء بنفسه.

```ts
import { createClient, generateKeyPairSigner } from "@solana/kit";
import { solanaDevnetRpc } from "@solana/kit-plugin-rpc";
import { signer } from "@solana/kit-plugin-signer";

const payer = await generateKeyPairSigner();

const client = createClient().use(signer(payer)).use(solanaDevnetRpc());

const slot = await client.rpc.getSlot().send();
console.log("current slot", slot);
```

تُعيِّن إضافة `signer` كلاً من دافع الرسوم وهوية التطبيق. تستلزم حزمة RPC وجود
دافع مسبقاً، وهذا هو سبب أهمية الترتيب.

### متغيّرات الموقِّع

تُتيح `@solana/kit-plugin-signer` ثلاثة أدوار حتى لا تمنح إلا ما يحتاجه العميل:

- `signer(x)` — يُعيِّن كلاً من `client.payer` و `client.identity`.
- `payer(x)` — دافع الرسوم فقط.
- `identity(x)` — سلطة التطبيق فقط.

لكلٍّ منها متغيّرات `*FromFile(path)` (Node) و `generated*()` و
`generated*WithSol(amount)` و `airdrop*()` للاختبارات والسكريبتات.

## إرسال معاملة

أنشئ التعليمات باستخدام عميل برنامج مُولَّد، ثم مرِّرها إلى
`client.sendTransaction`. يحلّ المُخطِّط blockhash ويُقدِّر الحوسبة ويوقِّع
باستخدام دافع العميل ثم يُرسل المعاملة.

```ts
import { getTransferSolInstruction } from "@solana-program/system";
import { address, lamports } from "@solana/kit";

const transfer = getTransferSolInstruction({
  source: client.payer,
  destination: address("Fke...address"),
  amount: lamports(10_000_000n) // 0.01 SOL
});

const { context } = await client.sendTransaction([transfer]);
console.log("submitted", context.signature);
```

يتحلَّل `sendTransaction` إلى كائن نتيجة؛ وتوقيع المعاملة موجود في
`result.context.signature`. يتحقّق من أن الخطة تتناسب مع معاملة واحدة ويُطلق
استثناءً في حال عدم ذلك — استخدم `sendTransactions` عندما قد تمتدّ الدفعة على
معاملات متعددة.

## اختر حزمة RPC

يوفر `@solana/kit-plugin-rpc` إعدادات مسبقة للكلاستر:

- `solanaDevnetRpc()` — إعدادات devnet الافتراضية، مع تمكين الإسقاط الجوي
  (`client.airdrop`).
- `solanaMainnetRpc()` — mainnet؛ الإسقاط الجوي يُعدّ خطأً في وقت الترجمة.
- `solanaLocalRpc()` — `http://127.0.0.1:8899` لـ validator محلي.
- `solanaRpc({ rpcUrl })` — أي نقطة وصول.

```ts
import { solanaRpc } from "@solana/kit-plugin-rpc";

const client = createClient()
  .use(signer(payer))
  .use(solanaRpc({ rpcUrl: "https://your-rpc-provider.example" }));
```

### التركيب التفصيلي

تجمع الإعدادات المسبقة بين اتصال RPC والمخطط والمنفذ ومساعدات الإرسال، لذا لا
تحتاج معظم التطبيقات إلى هذه المكونات بشكل منفرد. انتقل إلى الإضافات التفصيلية
فقط لتخصيص إحداها — على سبيل المثال مخطط مخصص أو قناة اشتراك مختلفة. يتطلب هذا
المسار أيضًا `@solana/kit-plugin-instruction-plan`، الذي تتضمنه الإعدادات
المسبقة تلقائيًا:

```ts
import {
  solanaRpcConnection,
  rpcAirdrop,
  rpcTransactionPlanner,
  rpcTransactionPlanExecutor
} from "@solana/kit-plugin-rpc";
import { planAndSendTransactions } from "@solana/kit-plugin-instruction-plan";

const client = createClient()
  .use(signer(payer))
  .use(solanaRpcConnection({ rpcUrl: "https://api.devnet.solana.com" }))
  .use(rpcAirdrop())
  .use(rpcTransactionPlanner())
  .use(rpcTransactionPlanExecutor())
  .use(planAndSendTransactions());
```

## إضافة عملاء البرامج

تُوفّر حزم `@solana-program/*` المولّدة إضافات للعميل تتيح إجراء استدعاءات
البرنامج منه مباشرةً. ثبّت البرنامج الذي تحتاجه واستخدم `.use()` لإضافة إضافته:

```ts
import { tokenProgram } from "@solana-program/token";

const client = createClient()
  .use(signer(payer))
  .use(solanaDevnetRpc())
  .use(tokenProgram()); // adds client.token
```

<Callout type="warn" title="نطاقات التبعية المتناظرة">
  لا تزال بعض إصدارات `@solana-program/*` تُعلن تبعية متناظرة لـ `@solana/kit
  ^6.x` بينما Kit في الإصدار 7.x. وهي تعمل مع Kit 7 — تجاوز نطاق التبعية
  المتناظرة في مدير الحزم الخاص بك إذا أبدى اعتراضًا.
</Callout>

## التوقيع بدون محفظة متصفح

هل تعمل بدون واجهة مستخدم في مسار API أو عامل أو سكريبت؟ وقّع باستخدام خدمة
إدارة مفاتيح خلفية بدلاً من keypair خام. راجع [Keychain](/docs/tools/keychain) و
[التوقيع في بيئة الإنتاج](/docs/core/transactions/signing-in-production).

## أنماط شائعة لمطوري سولانا

- **إعداد مشترك**: أنشئ العميل مرة واحدة في وحدة أساسية واستورده إلى السكريبتات
  ومسارات الخادم وواجهة المستخدم.
- **القراءات من جانب الخادم**: عميل يحتوي فقط على `signer` + `solanaRpc` (بدون
  إضافة محفظة) يقرأ الحسابات ويحاكي المعاملات من الخادم.
- **الاختبار المحلي**: استبدل حزمة RPC بـ `.use(litesvm())` من
  `@solana/kit-plugin-litesvm` للتشغيل مقابل validator في الذاكرة (Node فقط).
- **قابلية الاختبار**: العميل المُركَّب سهل المحاكاة — استبدل استجابات RPC أو
  الموقّعين بدون الحاجة إلى محفظة متصفح.
