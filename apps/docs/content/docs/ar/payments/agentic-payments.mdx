أصبح وكلاء الذكاء الاصطناعي أكثر استقلالية في سير عملهم: كتابة الأكواد، وجلب
البيانات، وتوفير الموارد، والمزيد. للعمل بشكل مستقل تماماً، يحتاجون إلى طريقة
لإجراء "مدفوعات صغيرة" مقابل الخدمات برمجياً، في الوقت الفعلي، بأقل تكلفة. قنوات
الدفع التقليدية لا تعمل مع المدفوعات الصغيرة. رسوم سولانا التي تقل عن السنت
والنهائية التي تقل عن الثانية تجعل هذا ممكناً تقنياً واقتصادياً.

## لماذا هذا مهم؟

على سبيل المثال، تخيل مطوراً يستخدم نموذج لغة كبير لبناء تطبيق يحتاج إلى بيانات
من واجهة برمجة تطبيقات مدفوعة. لجلب بعض البيانات المهمة، سيحتاج المطور إلى إيقاف
سير عمله، والبحث عن واجهات برمجة التطبيقات، واختيار مزود، وإنشاء حساب، وإضافة
طريقة دفع، وإنشاء مفتاح واجهة برمجة التطبيقات، ونسخ المفتاح إلى بيئته، ثم
استئناف سير عمله. هذا الاحتكاك يقطع حالة التدفق. بالنسبة لسير العمل المدعوم
بالذكاء الاصطناعي، إنه عائق.

بدلاً من ذلك، مع المدفوعات الوكيلة، يمكن للوكيل تقديم الطلب، والدفع مقابله،
ومتابعة سير عمله. لا حسابات. لا مفاتيح واجهة برمجة تطبيقات. لا تدخل يدوي.

### الدفع مقابل واجهات برمجة التطبيقات باستخدام pay.sh

[pay.sh](https://pay.sh) هي طبقة دفع لوكلاء HTTP وأدوات سطر الأوامر. راجع
[وثائق pay.sh](https://pay.sh/docs) لتثبيت `pay`. تغلف واجهة سطر أوامر `pay`
أدوات مثل `curl` و `codex` و `claude` حتى تتمكن من استدعاء واجهات برمجة
التطبيقات المدفوعة دون حسابات موفري الخدمة أو مفاتيح API. عندما ترجع واجهة برمجة
التطبيقات تحدي `402 Payment Required` من نوع MPP أو x402، يطلب `pay` من محفظتك
المحلية الموافقة على التوقيع.

## بروتوكول x402

تحتاج المدفوعات الوكيلية إلى طريقة للعملاء والخوادم للتفاوض على شروط الدفع عبر
البنية التحتية القياسية للويب دون تعطيل واجهات الويب التقليدية. لا تزال مساحة
الدفعات الصغيرة ناشئة ومن المحتمل أن تظهر أدوات ومعايير جديدة مع تطور المجال.
ومع ذلك، برز [بروتوكول x402](https://www.x402.org/) كمعيار مبكر مع دعم قوي من
النظام البيئي. يستخدم x402
[رمز الحالة 402 "الدفع مطلوب" في HTTP](https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Status/402) -
وهو معيار منذ HTTP/1.1 أصبح عمليًا فقط مع التسوية عبر البلوكشين.

### كيف يعمل

يقوم العميل بإجراء طلب، ويتلقى رمز 402 مع شروط الدفع، ثم يعيد المحاولة بدفعة
موقعة. يفوض الخادم التحقق والتسوية إلى ميسّر، وهو وسيط اختياري يتعامل مع إرسال
المعاملة على السلسلة. بمجرد أن يؤكد الميسّر التسوية، يعيد الخادم المحتوى
المطلوب. يتيح هذا الفصل لمزودي واجهات برمجة التطبيقات قبول المدفوعات دون إدارة
أي بنية تحتية على السلسلة بشكل مباشر.

![مخطط تدفق x402](/assets/guides/x402/x402-flow-solana.png)

البروتوكول عديم الحالة (لا جلسات أو مفاتيح API)، ويعمل مع البنية التحتية
القياسية لـ HTTP، ويدعم أي رمز SPL يقبله الخادم. إليك مقتطف حول كيفية إعداد خادم
Express باستخدام وسيط x402:

```ts
app.use(
  paymentMiddleware(
    {
      "GET /costly-data": {
        accepts: [
          {
            scheme: "exact",
            price: "$0.001",
            network: "solana:EtWTRABZaYq6iMfeYKouRu166VU2xqa1",
            payTo: svmAddress
          }
        ],
        description: "Costly data",
        mimeType: "application/json"
      }
    },
    new x402ResourceServer(facilitatorClient).register(
      "solana:EtWTRABZaYq6iMfeYKouRu166VU2xqa1",
      new ExactSvmScheme()
    )
  )
);

app.get("/costly-data", (req, res) => {
  res.send({
    report: {
      data: "costly data"
    }
  });
});
```
