`@solana/kit` dostarcza `createClient`, mały host, który rozszerzasz za pomocą
wtyczek. Każda wtyczka dodaje nową możliwość — połączenie RPC, podpisującego,
planowanie transakcji — a złożony klient obsługuje zarówno skrypty, trasy
serwerowe, jak i interfejsy użytkownika.

<Callout type="info">
  Tworzysz aplikację React? [Przewodnik React](/docs/frontend/react-hooks)
  opakowuje tego samego klienta w `@solana/react`, dzięki czemu komponenty
  odczytują stan portfela i dane RPC przez hooki. Zacznij tam i wróć tutaj, gdy
  potrzebujesz większej kontroli.
</Callout>

## Instalacja

```terminal
$ npm install @solana/kit @solana/kit-plugin-rpc @solana/kit-plugin-signer
```

Użyj dowolnego menedżera pakietów. Kit jest w wersji 7+ a pakiety
`@solana/kit-plugin-*` są w wersji 0.13+.

## Utwórz klienta

Łącz wtyczki za pomocą `.use(...)`. Najpierw ustaw podpisującego, następnie
połączenie RPC. `solanaDevnetRpc()` dodaje `client.rpc`, subskrypcje,
planistę/executor transakcji oraz pomocnika `client.sendTransaction` — dzięki
czemu aplikacja rzadko musi samodzielnie łączyć te elementy.

```ts
import { createClient, generateKeyPairSigner } from "@solana/kit";
import { solanaDevnetRpc } from "@solana/kit-plugin-rpc";
import { signer } from "@solana/kit-plugin-signer";

const payer = await generateKeyPairSigner();

const client = createClient().use(signer(payer)).use(solanaDevnetRpc());

const slot = await client.rpc.getSlot().send();
console.log("current slot", slot);
```

Wtyczka `signer` ustawia zarówno płatnika opłat, jak i tożsamość aplikacji.
Pakiet RPC wymaga, aby płatnik był skonfigurowany jako pierwszy — dlatego
kolejność ma znaczenie.

### Warianty podpisującego

`@solana/kit-plugin-signer` udostępnia trzy role, dzięki czemu przyznajesz tylko
te uprawnienia, których klient potrzebuje:

- `signer(x)` — ustawia zarówno `client.payer`, jak i `client.identity`.
- `payer(x)` — tylko płatnik opłat.
- `identity(x)` — tylko uprawnienia aplikacji.

Każdy ma warianty `*FromFile(path)` (Node), `generated*()`,
`generated*WithSol(amount)` i `airdrop*()` dla testów i skryptów.

## Wyślij transakcję

Zbuduj instrukcje za pomocą wygenerowanego klienta programu, a następnie przekaż
je do `client.sendTransaction`. Planista rozwiązuje blockhash, szacuje
obliczenia, podpisuje przy użyciu płatnika klienta i przesyła.

```ts
import { getTransferSolInstruction } from "@solana-program/system";
import { address, lamports } from "@solana/kit";

const transfer = getTransferSolInstruction({
  source: client.payer,
  destination: address("Fke...address"),
  amount: lamports(10_000_000n) // 0.01 SOL
});

const { context } = await client.sendTransaction([transfer]);
console.log("submitted", context.signature);
```

`sendTransaction` zwraca obiekt wynikowy; sygnatura transakcji znajduje się w
`result.context.signature`. Weryfikuje, czy plan mieści się w jednej transakcji
i w przeciwnym razie zgłasza wyjątek — użyj `sendTransactions`, gdy partia może
obejmować wiele transakcji.

## Wybierz pakiet RPC

`@solana/kit-plugin-rpc` dostarcza presety klastrów:

- `solanaDevnetRpc()` — domyślne ustawienia devnet, włączony airdrop
  (`client.airdrop`).
- `solanaMainnetRpc()` — mainnet; airdrop jest błędem kompilacji.
- `solanaLocalRpc()` — `http://127.0.0.1:8899` dla lokalnego validator.
- `solanaRpc({ rpcUrl })` — dowolny endpoint.

```ts
import { solanaRpc } from "@solana/kit-plugin-rpc";

const client = createClient()
  .use(signer(payer))
  .use(solanaRpc({ rpcUrl: "https://your-rpc-provider.example" }));
```

### Konfiguracja szczegółowa

Presety łączą połączenie RPC, planer, executor oraz pomocniki wysyłania, więc
większość aplikacji nigdy nie potrzebuje tych elementów osobno. Sięgaj po
granularne wtyczki tylko wtedy, gdy chcesz dostosować jeden z nich — na przykład
własny planer lub inny kanał subskrypcji. Ta ścieżka wymaga również
`@solana/kit-plugin-instruction-plan`, który presety w przeciwnym razie
dołączają automatycznie:

```ts
import {
  solanaRpcConnection,
  rpcAirdrop,
  rpcTransactionPlanner,
  rpcTransactionPlanExecutor
} from "@solana/kit-plugin-rpc";
import { planAndSendTransactions } from "@solana/kit-plugin-instruction-plan";

const client = createClient()
  .use(signer(payer))
  .use(solanaRpcConnection({ rpcUrl: "https://api.devnet.solana.com" }))
  .use(rpcAirdrop())
  .use(rpcTransactionPlanner())
  .use(rpcTransactionPlanExecutor())
  .use(planAndSendTransactions());
```

## Dodaj klientów programu

Wygenerowane pakiety `@solana-program/*` udostępniają wtyczki klienckie, dzięki
czemu wywołania programu wiszą na kliencie. Zainstaluj potrzebny program i
dołącz jego wtyczkę za pomocą `.use()`:

```ts
import { tokenProgram } from "@solana-program/token";

const client = createClient()
  .use(signer(payer))
  .use(solanaDevnetRpc())
  .use(tokenProgram()); // adds client.token
```

<Callout type="warn" title="Zakresy zależności peer">
  Niektóre wersje `@solana-program/*` nadal deklarują peer `@solana/kit ^6.x`,
  podczas gdy Kit jest w wersji 7.x. Działają z Kit 7 — zastąp zakres peer w
  swoim menedżerze pakietów, jeśli zgłosi zastrzeżenia.
</Callout>

## Podpisywanie bez portfela przeglądarkowego

Działasz w trybie headless w trasie API, workerze lub skrypcie? Podpisuj przy
użyciu backendowej usługi zarządzania kluczami zamiast surowego keypair. Zobacz
[Keychain](/docs/tools/keychain) oraz
[Podpisywanie na produkcji](/docs/core/transactions/signing-in-production).

## Typowe wzorce dla deweloperów Solana

- **Współdzielona konfiguracja**: Zbuduj klienta raz w module podstawowym i
  importuj go do skryptów, tras serwerowych oraz interfejsu użytkownika.
- **Odczyty po stronie serwera**: Klient z samym `signer` + `solanaRpc` (bez
  wtyczki portfela) odczytuje konta i symuluje transakcje po stronie serwera.
- **Testowanie lokalne**: Zamień pakiet RPC na `.use(litesvm())` z
  `@solana/kit-plugin-litesvm`, aby uruchamiać testy na validator działającym w
  pamięci (tylko Node).
- **Testowalność**: Złożony klient jest łatwy do mockowania — zaślepiaj
  odpowiedzi RPC lub signerów bez konieczności posiadania portfela
  przeglądarkowego.
