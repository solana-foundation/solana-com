h1: Zbuduj Swój Pierwszy Program Solana

Ten szybki start wykorzystuje projekt startowy
[frameworka Anchor](https://www.anchor-lang.com/docs) wygenerowany przez
`anchor init`. Utworzysz projekt lokalnie, uruchomisz jego testy, zbudujesz
program i przejrzysz kod programu, który generuje Anchor.

## Wymagania wstępne

Zanim zaczniesz,
[zainstaluj narzędzia deweloperskie Solana](/docs/intro/installation).
Instalacja obejmuje Rust, Solana CLI oraz Anchor CLI.

Użyj Anchor CLI w wersji 1.1.2 lub wyższej dla tego szablonu. Sprawdź
zainstalowaną wersję:

```terminal
$ anchor --version
```

<Steps>

<Step>

### Utwórz projekt

Uruchom następujące polecenia w swoim terminalu:

```terminal
$ anchor init my-program
$ cd my-program
```

Projekt startowy zawiera jeden program Solana w katalogu `programs/my-program`.
Program zawiera dwie instrukcje: jedną do inicjalizacji konta licznika i jedną
do inkrementacji licznika.

Niektóre części szablonu demonstrują typowe wzorce programów Solana:
wyprowadzanie adresów kont [PDA](/docs/core/pda), wykonywanie Cross Program
Invocation ([CPI](/docs/core/cpi)) w celu transferu SOL oraz używanie
niestandardowych sprawdzeń błędów do zatrzymania instrukcji, gdy warunek nie
jest spełniony.

<Step>

### Zbuduj Program

Uruchom `anchor build`, aby skompilować program startowy:

```terminal
$ anchor build
```

Skompilowany program jest zapisywany do `target/deploy/my_program.so`. Gdy
program zostanie wdrożony, zawartość tego pliku `.so` jest przechowywana na
koncie w łańcuchu.

</Step>

<Step>

### Uruchom test

Uruchom domyślny test:

```terminal
$ anchor test
```

`Anchor.toml` tego szablonu używa polecenia testowego Rust:

```toml title="Anchor.toml"
skip_local_validator = true

[scripts]
test = "cargo test"
```

Test ładuje skompilowany program do [LiteSVM](https://www.litesvm.com/), tworzy
płatnika, wysyła instrukcje _rs`initialize`_ i _rs`increment`_, a następnie
sprawdza stan konta licznika.

Uruchomienie `anchor test` kompiluje również program, więc nie musisz uruchamiać
`anchor build` przed testowaniem lokalnym.

</Step>

<Step>

### Wdróż program

Testy lokalne zapewniają najszybszą pętlę informacji zwrotnej. Gdy będziesz
gotowy do wdrożenia w sieci, na przykład devnet, najpierw zbuduj projekt, a
następnie wdróż go do klastra.

Wdrożenie programu Solana wymaga SOL, ponieważ program jest przechowywany na
koncie, a konto musi opłacić zajmowaną przestrzeń. W sieci devnet możesz
bezpłatnie pozyskać devnet SOL z [Solana Faucet](https://faucet.solana.com/) lub
za pomocą Solana CLI:

```terminal
$ solana airdrop 2 --url devnet
```

```terminal
$ anchor build
$ anchor deploy --provider.cluster devnet
```

</Step>

</Steps>

## Pliki źródłowe

Katalog `src` zawiera program Solana. Dokumentacja Anchor dotycząca
[struktury programu](https://www.anchor-lang.com/docs/basics/program-structure)
omawiа podstawowe makra używane tutaj, w tym _rs`declare_id!`_,
_rs`#[program]`_, _rs`#[derive(Accounts)]`_ i _rs`#[account]`_. Ta sekcja
przechodzi przez pliki szablonu.

### lib.rs

`lib.rs` jest punktem wejścia programu. Łączy pliki źródłowe, definiuje adres
programu oraz instrukcje programu, które użytkownicy mogą wywoływać.

<WithNotes>

```rust title="programs/my-program/src/lib.rs" -w
// !tooltip[/pub mod constants/] constantsMod
pub mod constants;
// !tooltip[/pub mod error/] errorMod
pub mod error;
// !tooltip[/pub mod instructions/] instructionsMod
pub mod instructions;
// !tooltip[/pub mod state/] stateMod
pub mod state;

// !tooltip[/anchor_lang::prelude/] prelude
use anchor_lang::prelude::*;

pub use constants::*;
pub use instructions::*;
pub use state::*;

// !tooltip[/declare_id!/] declareId
declare_id!("82sFkffP9wxwpyfZyeaKHH2chQoJPUGsJZSPi9mrUuXd");

// !tooltip[/#\[program\]/] programMacro
#[program]
pub mod my_program {
    use super::*;

    // !tooltip[/pub fn initialize/] publicInstructions
    pub fn initialize(ctx: Context<Initialize>) -> Result<()> {
        // !tooltip[/handle_initialize/] handlers
        crate::instructions::initialize::handle_initialize(ctx)
    }

    // !tooltip[/pub fn increment/] publicInstructions
    pub fn increment(ctx: Context<Increment>) -> Result<()> {
        // !tooltip[/handle_increment/] handlers
        crate::instructions::increment::handle_increment(ctx)
    }
}
```

### !constantsMod

_rs`pub mod constants`_ włącza `constants.rs` do skrzynki programu. Tutaj
szablon przechowuje współdzielone wartości programu, takie jak seed PDA, kwota
przelewu i maksymalna wartość licznika.

### !errorMod

_rs`pub mod error`_ włącza `error.rs` do skrzynki programu. Ten plik definiuje
niestandardowe błędy, które program może zwrócić, gdy instrukcja jest
nieprawidłowa.

### !instructionsMod

_rs`pub mod instructions`_ włącza `instructions.rs` do skrzynki programu. Ten
plik działa jak indeks dla plików specyficznych dla instrukcji w
`src/instructions/`.

### !stateMod

_rs`pub mod state`_ włącza `state.rs` do skrzynki programu. Ten plik definiuje
układ danych dla kont należących do tego programu, w tym konto _rs`Counter`_.

### !prelude

_rs`anchor_lang::prelude::*`_ importuje typy i makra Anchor używane do
definiowania programu Solana, w tym _rs`Context`_, _rs`Result`_, _rs`Pubkey`_,
_rs`Account`_, _rs`Signer`_ oraz atrybuty kont Anchor.

### !reexports

Linie _rs`pub use`_ udostępniają publiczne elementy z modułów podrzędnych z
poziomu korzenia skrzynki. Dzięki temu punkty wejścia instrukcji są krótkie, np.
_rs`Context<Initialize>`_ zamiast dłuższej ścieżki modułu.

### !declareId

_rs`declare_id!`_ definiuje adres programu. Powinien być zgodny z adresem w
`Anchor.toml`, wdrożonym program account oraz wszystkimi użytkownikami lub
testami wywołującymi program.

### !programMacro

_rs`#[program]`_ nakazuje Anchor wygenerowanie punktu wejścia programu Solana
oraz kodu wysyłania instrukcji. Publiczne funkcje w tym module stają się
instrukcjami, które użytkownicy mogą wywoływać.

### !publicInstructions

_rs`initialize`_ i _rs`increment`_ to dwie nazwy instrukcji, które użytkownicy
mogą wywoływać. Każda otrzymuje _rs`Context<T>`_. Typ wewnątrz _rs`Context`_
definiuje konta wymagane przez daną instrukcję, a Anchor weryfikuje te konta
przed uruchomieniem instrukcji.

### !handlers

Funkcja obsługi zawiera logikę instrukcji.

</WithNotes>

### constants.rs

`constants.rs` przechowuje wspólne wartości w jednym miejscu. W tym szablonie
_rs`COUNTER_SEED`_ wyprowadza PDA licznika, _rs`HELLO_WORLD_LAMPORTS`_ jest
przekazywany podczas inicjalizacji, a _rs`MAX_COUNT`_ jest sprawdzany przed
inkrementacją.

<WithNotes>

```rust title="programs/my-program/src/constants.rs" -w
use anchor_lang::prelude::*;

// !tooltip[/#\[constant\]/] anchorConstant
// !tooltip[/COUNTER_SEED/] counterSeed
#[constant]
pub const COUNTER_SEED: &[u8] = b"counter";

// !tooltip[/HELLO_WORLD_LAMPORTS/] helloLamports
#[constant]
pub const HELLO_WORLD_LAMPORTS: u64 = 1;

// !tooltip[/MAX_COUNT/] maxCount
#[constant]
pub const MAX_COUNT: u64 = 10;
```

### !anchorConstant

_rs`#[constant]`_ nakazuje Anchor uwzględnienie wartości w IDL programu. Dzięki
temu ważne stałe programu są widoczne dla generowanych klientów i narzędzi.

### !counterSeed

_rs`COUNTER_SEED`_ to ciąg bajtów używany do wyprowadzania PDA licznika. Zarówno
program, jak i testy korzystają z tego seed, dzięki czemu są zgodne co do adresu
konta bez konieczności przechowywania osobnego keypair. Ten szablon używa
jednego stałego seed, więc program ma jeden adres licznika. Prawdziwe programy
często zawierają adres użytkownika lub inny identyfikator w seeds, aby tworzyć
wiele powiązanych kont.

### !helloLamports

_rs`HELLO_WORLD_LAMPORTS`_ to ilość lamport przekazywanych od płatnika do konta
licznika wewnątrz _rs`initialize`_. Jest celowo mała, ponieważ szablon
demonstruje CPI do System Program.

### !maxCount

_rs`MAX_COUNT`_ to górna granica sprawdzana przez _rs`increment`_.
Przechowywanie jej tutaj zapewnia programowi, testom i wygenerowanemu IDL jedną
wspólną nazwę dla tej reguły.

</WithNotes>

### !right

`initialize.rs` używa dwóch stałych:

- [_rs`COUNTER_SEED`_](mention:counter-seed) wyznacza adres PDA licznika.
- [_rs`HELLO_WORLD_LAMPORTS`_](mention:hello-world-lamports) określa kwotę
  przenoszoną od płatnika na konto licznika.

</SideBySide>

<SideBySide>

### !left

```rust title="increment.rs" -w
pub fn handle_increment(ctx: Context<Increment>) -> Result<()> {
    // ...

    require!(
        // !mark
        // !mention max-count
        ctx.accounts.counter.count < MAX_COUNT,
        ErrorCode::CounterOverflow,
    );

    ctx.accounts.counter.count += 1;
    Ok(())
```

### !right

`increment.rs` używa [_rs`MAX_COUNT`_](mention:max-count) jako górnej granicznej
wartości licznika. Jeśli bieżąca wartość osiągnęła już maksimum, _rs`require!`_
zwraca _rs`CounterOverflow`_ i dane konta nie są zmieniane.

</SideBySide>

### state.rs

`state.rs` definiuje niestandardowe typy danych dla kont tworzonych i
posiadanych przez program. Program definiuje instrukcje tworzenia, inicjowania i
aktualizowania tych danych, jednak dane licznika nie są przechowywane w samym
programie. Są przechowywane w oddzielnym koncie z własnym adresem.

<WithNotes>

```rust title="programs/my-program/src/state.rs" -w
use anchor_lang::prelude::*;

// !tooltip[/#\[account\]/] accountMacro
#[account]
// !tooltip[/InitSpace/] initSpace
#[derive(InitSpace)]
pub struct Counter {
    // !tooltip[/count/] count
    pub count: u64,
    // !tooltip[/authority/] authority
    pub authority: Pubkey,
}
```

### !accountMacro

_rs`#[account]`_ informuje Anchor, że ta struktura to dane, które mogą być
przechowywane w koncie onchain należącym do tego programu. Anchor używa jej do
serializacji, deserializacji i walidacji kont _rs`Counter`_.

### !initSpace

_rs`InitSpace`_ pozwala Anchor obliczyć _rs`Counter::INIT_SPACE`_, czyli liczbę
bajtów potrzebnych dla pól tej struktury. 8-bajtowy dyskryminator konta Anchor
jest dodawany osobno podczas tworzenia konta.

### !count

_rs`count`_ to wartość licznika przechowywana w danych konta.

### !authority

_rs`authority`_ przechowuje adres uprawniony do inkrementacji licznika.
_rs`initialize`_ zapisuje ten adres, a _rs`increment`_ wymaga podpisania tym
samym adresem.

</WithNotes>

<SideBySide>

### !left

```rust title="initialize.rs" -w

    init,
    payer = payer,
    // !mark
    // !mention counter-space

    seeds = [COUNTER_SEED],
    bump

pub counter: Account<'info, Counter>,
// ...

pub fn handle_initialize(ctx: Context<Initialize>) -> Result<()> {
    // !mark(1:2)
    // !mention(1:2) initialize-fields
    ctx.accounts.counter.count = 0;
    ctx.accounts.counter.authority = ctx.accounts.payer.key();

```

```rust title="increment.rs" -w

    // ...

    // !mark
    // !mention increment-count
    ctx.accounts.counter.count += 1;
    Ok(())
}
```

### !right

`state.rs` definiuje dane konta _rs`Counter`_. Pliki instrukcji używają tego
typu podczas tworzenia i aktualizowania konta:

- [_rs`Counter::INIT_SPACE`_](mention:counter-space) określa rozmiar konta dla
  pól zdefiniowanych w `state.rs`.
- [_rs`count`_ i _rs`authority`_](mention:initialize-fields) to wartości pól
  zapisywane podczas inicjalizacji konta.
- [_rs`count += 1`_](mention:increment-count) aktualizuje przechowywaną wartość
  licznika po pomyślnej weryfikacji.

</SideBySide>

### error.rs

`error.rs` definiuje niestandardowe błędy programu. W tym szablonie błędy
przedstawiają, jak handlery instrukcji zatrzymują się, gdy wywołujący nie ma
uprawnień do aktualizacji licznika lub licznik osiągnął już _rs`MAX_COUNT`_.

<WithNotes>

```rust title="programs/my-program/src/error.rs" -w
use anchor_lang::prelude::*;

// !tooltip[/#\[error_code\]/] errorCode
#[error_code]
pub enum ErrorCode {
    // !tooltip[/Unauthorized/] unauthorized
    #[msg("Only the counter authority can update this counter")]
    Unauthorized,
    // !tooltip[/CounterOverflow/] counterOverflow
    #[msg("Counter has reached the maximum value")]
    CounterOverflow,
}
```

### !errorCode

_rs`#[error_code]`_ informuje Anchor, aby zamienił te warianty w niestandardowe
błędy programu. Klienty i testy mogą używać tych błędów zamiast zgadywać na
podstawie surowych kodów błędów.

### !unauthorized

_rs`Unauthorized`_ jest zwracany, gdy podpisujący transakcję nie jest authority
przechowywanym na koncie licznika.

### !counterOverflow

_rs`CounterOverflow`_ jest zwracany, gdy inkrementacja naruszyłaby regułę
_rs`MAX_COUNT`_ programu.

</WithNotes>

<SideBySide>

### !left

```rust title="increment.rs" -w
pub fn handle_increment(ctx: Context<Increment>) -> Result<()> {
    require_keys_eq!(
        ctx.accounts.counter.authority,
        ctx.accounts.authority.key(),
        // !mark
        // !mention unauthorized-error
        ErrorCode::Unauthorized,

    require!(

        // !mark
        // !mention overflow-error
        ErrorCode::CounterOverflow,
    );

    ctx.accounts.counter.count += 1;
    msg!("Hello, world! Counter is now {}", ctx.accounts.counter.count);
    Ok(())
}
```

### !right

`error.rs` określa błędy, które instrukcja może zwrócić:

- [_rs`ErrorCode::Unauthorized`_](mention:unauthorized-error) jest zwracany, gdy
  podpisujący nie jest authority przechowywanym na koncie licznika.
- [_rs`ErrorCode::CounterOverflow`_](mention:overflow-error) jest zwracany, gdy
  licznik osiągnął już _rs`MAX_COUNT`_.

</SideBySide>

### instructions.rs

`instructions.rs` łączy pliki instrukcji z kratem programu, dzięki czemu
`lib.rs` może uzyskać dostęp do kodu instrukcji _rs`initialize`_ i
_rs`increment`_. Każdy plik instrukcji definiuje konta wymagane przez daną
instrukcję oraz logikę obsługi wykonywaną po tym, jak Anchor zweryfikuje te
konta.

<WithNotes>

```rust title="programs/my-program/src/instructions.rs" -w

pub mod increment;

// !tooltip[/pub use initialize/] reexports
```

### !moduleDeclarations

Linie _rs`pub mod`_ sprawiają, że `src/instructions/initialize.rs` i
`src/instructions/increment.rs` stają się częścią krata programu.

### !reexports

Linie _rs`pub use`_ ponownie eksportują publiczne elementy z tych plików. Dzięki
temu struktury kont instrukcji i funkcje obsługi są dostępne poprzez moduł
_rs`instructions`_.

</WithNotes>

### initialize.rs

`initialize.rs` definiuje konta wymagane do utworzenia konta licznika, a
następnie zapisuje jego pierwsze wartości. Struktura _rs`#[derive(Accounts)]`_
korzysta z
[ograniczeń kont](https://www.anchor-lang.com/docs/references/account-constraints)
Anchor, aby określić, które konta są wymagane i w jaki sposób tworzone jest nowe
konto licznika.

<WithNotes>

```rust title="programs/my-program/src/instructions/initialize.rs" -w
use anchor_lang::prelude::*;

use crate::{constants::*, state::Counter};

// !tooltip[/derive\(Accounts\)/] accountsStruct
#[derive(Accounts)]
pub struct Initialize<'info> {
    // !tooltip[/#\[account\(mut\)\]/] payerMutable
    #[account(mut)]
    // !tooltip[/Signer<'info>/] payerSigner
    pub payer: Signer<'info>,
    // !tooltip[/#\[account\(/] counterConstraints
    #[account(
        // !tooltip[/init/] initConstraint
        init,
        // !tooltip[/payer = payer/] payerConstraint
        payer = payer,
        // !tooltip[/space = 8 \+ Counter::INIT_SPACE/] space
        space = 8 + Counter::INIT_SPACE,
        // !tooltip[/seeds = \[COUNTER_SEED\]/] seeds
        seeds = [COUNTER_SEED],
        // !tooltip[/bump/] bump
        bump
    )]
    // !tooltip[/Account<'info, Counter>/] counterAccountType
    pub counter: Account<'info, Counter>,
    // !tooltip[/Program<'info, System>/] systemProgram
    pub system_program: Program<'info, System>,
}

// !tooltip[/handle_initialize/] handler
pub fn handle_initialize(ctx: Context<Initialize>) -> Result<()> {
    // !tooltip[/counter.count = 0/] initialCount
    ctx.accounts.counter.count = 0;
    // !tooltip[/counter.authority = ctx.accounts.payer.key\(\)/] initialAuthority
    ctx.accounts.counter.authority = ctx.accounts.payer.key();

    // !tooltip[/system_program::Transfer/] cpiAccounts
    let cpi_accounts = anchor_lang::system_program::Transfer {
        from: ctx.accounts.payer.to_account_info(),
        to: ctx.accounts.counter.to_account_info(),
    };
    // !tooltip[/CpiContext::new/] cpiContext
    let cpi_ctx = CpiContext::new(anchor_lang::system_program::ID, cpi_accounts);
    // !tooltip[/system_program::transfer/] transfer
    anchor_lang::system_program::transfer(cpi_ctx, HELLO_WORLD_LAMPORTS)?;

    // !tooltip[/msg!/] programLog
    msg!("Hello, world! Counter initialized");
    Ok(())
}
```

## !!steps Kontekst konta

Struktura _rs`Initialize`_ definiuje konta, które muszą być uwzględnione, gdy
użytkownik wywołuje instrukcję _rs`initialize`_. Anchor weryfikuje te konta
przed uruchomieniem obsługi.

<CodePlaceholder title="initialize.rs" />

```rust !! title="initialize.rs"
// !focus(1:2)
#[derive(Accounts)]
pub struct Initialize<'info> {
    #[account(mut)]
    pub payer: Signer<'info>,
    #[account(
        init,
        payer = payer,
        space = 8 + Counter::INIT_SPACE,
        seeds = [COUNTER_SEED],
        bump
    )]
    pub counter: Account<'info, Counter>,
    pub system_program: Program<'info, System>,
}
```

## !!steps Konto płatnika

Konto _rs`payer`_ pokrywa koszty utworzenia konta licznika. Typ
_rs`Signer<'info>`_ oznacza, że płatnik musi podpisać transakcję, a
_rs`#[account(mut)]`_ oznacza, że konto płatnika może być modyfikowane, ponieważ
lamport zostaną pobrane.

<CodePlaceholder title="initialize.rs" />

```rust !! title="initialize.rs"
#[derive(Accounts)]
pub struct Initialize<'info> {
    // !focus(1:2)
    #[account(mut)]
    pub payer: Signer<'info>,
    #[account(
```

### !imports

Ten plik importuje współdzielone stałe oraz typ konta _rs`Counter`_. Instrukcja

### !accountsStruct

_rs`Initialize`_ jest kontekstem kont dla instrukcji _rs`initialize`_. Każde

_rs`#[account(mut)]`_ informuje Anchor, że konto płatnika musi być zapisywalne.
Jest to wymagane, ponieważ zarówno tworzenie konta, jak i transfer CPI odejmują

### !payerSigner

_rs`Signer<'info>`_ informuje Anchor, że płatnik musi podpisać transakcję.

### !counterConstraints

Blok _rs`#[account(...)]`_ informuje Anchor, jak utworzyć i zwalidować konto
licznika przed uruchomieniem handlera.

### !initConstraint

_rs`init`_ mówi Anchor, aby utworzył konto _rs`counter`_ przed uruchomieniem
handlera. Nowe konto należy do tego programu, więc program może przechowywać w

### !payerConstraint

_rs`payer = payer`_ wskazuje konto _rs`payer`_ jako płatnika za utworzenie
nowego konta licznika.

### !space

_rs`space = 8 + Counter::INIT_SPACE`_ przydziela bajty na dyskryminator konta
Anchor oraz pola _rs`Counter`_ z `state.rs`. Dyskryminator pomaga Anchor
potwierdzić, że konto przechowuje oczekiwany typ przed jego odczytaniem.

### !seeds

_rs`seeds = [COUNTER_SEED]`_ oznacza, że adres konta licznika musi być PDA
wyprowadzonym z tego seed i identyfikatora programu. Przy jednym stałym seed
istnieje jeden licznik PDA dla tego programu. Aby tworzyć osobne liczniki w
prawdziwym programie, zazwyczaj należałoby dodać kolejny seed, na przykład adres
użytkownika.

### !bump

_rs`bump`_ nakazuje Anchor znalezienie i zweryfikowanie bumpa PDA, czyli
dodatkowego bajtu, który sprawia, że wyprowadzony adres jest prawidłowy jako
PDA.

### !counterAccountType

_rs`Account<'info, Counter>`_ nakazuje Anchor traktowanie danych konta jako typu

### !handler

licznika, dzięki czemu może bezpiecznie zapisać początkowe dane konta.

### !initialAuthority

```rust title="programs/my-program/src/instructions/increment.rs" -w
        ctx.accounts.counter.count < MAX_COUNT,
        ErrorCode::CounterOverflow,
    );
```

```rust !! title="increment.rs"
    // !focus
```
```rust !! title="increment.rs"
    ctx.accounts.counter.count += 1;
    msg!("Hello, world! Counter is now {}", ctx.accounts.counter.count);
    Ok(())
}
```

_rs`Ok(())`_ oznacza, że instrukcja zakończyła się pomyślnie.

### !incrementAccounts

_rs`Increment`_ to kontekst konta dla instrukcji _rs`increment`_. Anchor używa
go do weryfikacji mutowalnego konta licznika i wymaganego sygnatariusza.

### !counterMutable

_rs`mut`_ informuje Anchor, że konto licznika musi być zapisywalne, ponieważ
handler aktualizuje przechowywaną wartość _rs`count`_.

### !counterSeeds

_rs`seeds = [COUNTER_SEED]`_ weryfikuje, że podane konto licznika jest PDA
wyprowadzonym z tego seed i identyfikatora programu.

### !counterBump

_rs`bump`_ informuje Anchor, aby znalazł i zweryfikował bump PDA dla konta
licznika.

### !incrementCounterAccountType

_rs`Account<'info, Counter>`_ informuje Anchor, aby zdeserializował dane konta
jako typ _rs`Counter`_ z `state.rs`. Anchor sprawdza również, czy konto jest
własnością tego programu.

### !authoritySigner

_rs`Signer`_ wymaga podpisu authority. Handler następnie porównuje adres tego
podpisującego z authority przechowywanym na koncie licznika.

### !incrementHandler

Handler uruchamia się po tym, jak Anchor zwaliduje listę kont, sprawdzi własność
i zdeserializuje konto licznika.

### !authorityCheck

_rs`require_keys_eq!`_ porównuje dwa klucze publiczne i zatrzymuje instrukcję,
jeśli adres podpisującego nie pasuje do authority przechowywanego na koncie
licznika. Jest to reguła autoryzacji na poziomie aplikacji tego szablonu.

### !maxCheck

_rs`require!`_ sprawdza regułę logiczną i zatrzymuje instrukcję z
_rs`CounterOverflow`_, jeśli licznik osiągnął już _rs`MAX_COUNT`_. Ta reguła
szablonu demonstruje niestandardowe błędy i przerywanie przed mutacją.

### !mutateCount

Dopiero po pomyślnym przejściu obu sprawdzeń handler aktualizuje dane konta
licznika. Jeśli którekolwiek ze sprawdzeń nie powiedzie się, transakcja kończy
się niepowodzeniem i konto nie zostaje zmienione.

### !programLog

_rs`msg!`_ zapisuje wiadomość w logach programu, gdy program jest wywoływany.

### !litesvm

_rs`LiteSVM`_ to wbudowana maszyna wirtualna Solana przeznaczona do testów.
Umożliwia testowi rejestrację skompilowanego programu, tworzenie kont, wysyłanie
transakcji oraz inspekcję stanu kont bez uruchamiania procesu validator.

### !setup

Test pobiera adres programu z wartości _rs`declare_id!`_ za pomocą
_rs`my_program::id()`_ i tworzy nowy keypair płatnika dla transakcji testowej.

### !counterPda

Adres licznika jest wyprowadzany z tego samego _rs`COUNTER_SEED`_, którego używa
program. Dzięki temu test celuje w PDA, którego oczekuje `initialize.rs`.

### !programBytes

_rs`include_bytes!`_ odczytuje skompilowany plik `my_program.so` wygenerowany
przez build. Te bajty stanowią wykonywalny kod programu, który LiteSVM uruchomi
w trakcie testu.

### !addProgram

_rs`svm.add_program`_ rejestruje skompilowane bajty programu pod adresem
programu w LiteSVM. Airdrop zapewnia płatnikowi wystarczającą ilość lokalnych
SOL, aby opłacić transakcję i utworzenie konta.

### !testInstruction

_rs`Instruction::new_with_bytes`_ tworzy instrukcję wywołania programu. Zawiera
ona identyfikator programu, instruction data oraz konta instrukcji.

### !blockhash

Każda transakcja Solana wymaga aktualnego blockhasha. W tym teście LiteSVM
dostarcza blockhash do użycia w transakcji.

### !message

Wiadomość zawiera instrukcję, płatnika opłat oraz blockhash. Są to dane
transakcji, które zostaną podpisane.

### !transaction

_rs`VersionedTransaction::try_new`_ podpisuje wiadomość za pomocą keypair
płatnika. Płatnik podpisuje, ponieważ uiszcza opłatę transakcyjną i finansuje
nowe konto licznika.

### !sendTx

_rs`svm.send_transaction`_ wykonuje transakcję w LiteSVM.

### !getCounter

Po wykonaniu transakcji test odczytuje konto licznika bezpośrednio z LiteSVM,
aby móc sprawdzić dane konta.

### !deserialize

Konto licznika przechowuje zserializowane dane. _rs`Counter::try_deserialize`_
sprawdza dyskryminator konta Anchor i przekształca dane konta z powrotem w typ
_rs`Counter`_ z `state.rs`.

### !initAssert

Po pomyślnym wykonaniu _rs`initialize`_ test deserializuje konto licznika i
sprawdza, czy stan początkowy odpowiada temu, co zapisał
_rs`handle_initialize`_.

### !incrementAssert

Po pomyślnym wykonaniu _rs`increment`_ to samo konto licznika powinno mieć
_rs`count`_ ustawione na _rs`1`_, zachowując przy tym tego samego właściciela.
