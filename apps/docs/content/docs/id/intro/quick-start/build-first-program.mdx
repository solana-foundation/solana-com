Quickstart ini menggunakan proyek awal
[framework Anchor](https://www.anchor-lang.com/docs) yang dihasilkan oleh
`anchor init`. Anda akan membuat proyek secara lokal, menjalankan pengujiannya,
mem-build program, dan menelusuri kode program yang dihasilkan oleh Anchor.

## Prasyarat

Sebelum memulai,
[instal perangkat pengembangan Solana](/docs/intro/installation). Instalasi
mencakup Rust, Solana CLI, dan Anchor CLI.

Gunakan Anchor CLI versi 1.1.2 atau lebih tinggi untuk template ini. Periksa
versi yang terinstal:

```terminal
$ anchor --version
```

<Steps>

<Step>

### Buat Proyek

Jalankan perintah berikut di terminal Anda:

```terminal
$ anchor init my-program
$ cd my-program
```

Proyek awal mencakup satu program Solana di bawah `programs/my-program`. Program
ini mencakup dua instruksi: satu untuk menginisialisasi akun counter dan satu
lagi untuk menambah nilai counter.

Beberapa bagian dari template ini mendemonstrasikan pola umum program Solana:
menurunkan alamat akun [PDA](/docs/core/pda), melakukan Cross Program Invocation
([CPI](/docs/core/cpi)) untuk mentransfer SOL, dan menggunakan pemeriksaan error
kustom untukmenghentikan instruksi ketika suatu kondisi gagal.

</Step>

<Step>

### Build Program

Jalankan `anchor build` untuk mengompilasi program awal:

```terminal
$ anchor build
```

Program yang telah dikompilasi ditulis ke `target/deploy/my_program.so`. Ketika
program di-deploy, isi dari file `.so` ini disimpan dalam sebuah akun di
onchain.

</Step>

<Step>

### Jalankan Pengujian

Jalankan pengujian default:

```terminal
$ anchor test
```

Template ini `Anchor.toml` menggunakan perintah pengujian Rust:

```toml title="Anchor.toml"
skip_local_validator = true

[scripts]
test = "cargo test"
```

Pengujian memuat program yang telah dikompilasi ke dalam
[LiteSVM](https://www.litesvm.com/), membuat payer, mengirim instruksi
_rs`initialize`_ dan _rs`increment`_, lalu memeriksa status akun counter.

Menjalankan `anchor test` juga mengompilasi program, sehingga Anda tidak perlu
menjalankan `anchor build` terlebih dahulu saat pengujian secara lokal.

</Step>

<Step>

### Deploy Program

Pengujian lokal adalah siklus umpan balik tercepat. Ketika Anda siap untuk
melakukan deploy ke suatu jaringan, misalnya devnet, build terlebih dahulu, lalu
deploy ke cluster.

Men-deploy program Solana membutuhkan SOL karena program disimpan dalam sebuah
account, dan account tersebut harus membayar untuk ruang yang digunakannya. Di
devnet, minta SOL devnet gratis dari [Solana Faucet](https://faucet.solana.com/)
atau dengan Solana CLI:

```terminal
$ solana airdrop 2 --url devnet
```

```terminal
$ anchor build
$ anchor deploy --provider.cluster devnet
```

</Step>

</Steps>

## File Sumber

Direktori `src` berisi program Solana. Dokumentasi
[Struktur Program](https://www.anchor-lang.com/docs/basics/program-structure)
dari Anchor menjelaskan makro inti yang digunakan di sini, termasuk
_rs`declare_id!`_, _rs`#[program]`_, _rs`#[derive(Accounts)]`_, dan
_rs`#[account]`_. Bagian ini memandu Anda melalui file-file template.

### lib.rs

`lib.rs` adalah titik masuk program. Ini menghubungkan file-file sumber,
mendefinisikan alamat program, dan mendefinisikan instruksi program yang dapat
dipanggil oleh pengguna.

<WithNotes>

```rust title="programs/my-program/src/lib.rs" -w
// !tooltip[/pub mod constants/] constantsMod
pub mod constants;
// !tooltip[/pub mod error/] errorMod
pub mod error;
// !tooltip[/pub mod instructions/] instructionsMod
pub mod instructions;
// !tooltip[/pub mod state/] stateMod
pub mod state;

// !tooltip[/anchor_lang::prelude/] prelude
use anchor_lang::prelude::*;

pub use constants::*;
pub use instructions::*;
pub use state::*;

// !tooltip[/declare_id!/] declareId
declare_id!("82sFkffP9wxwpyfZyeaKHH2chQoJPUGsJZSPi9mrUuXd");

// !tooltip[/#\[program\]/] programMacro
#[program]
pub mod my_program {
    use super::*;

    // !tooltip[/pub fn initialize/] publicInstructions
    pub fn initialize(ctx: Context<Initialize>) -> Result<()> {
        // !tooltip[/handle_initialize/] handlers
        crate::instructions::initialize::handle_initialize(ctx)
    }

    // !tooltip[/pub fn increment/] publicInstructions
    pub fn increment(ctx: Context<Increment>) -> Result<()> {
        // !tooltip[/handle_increment/] handlers
        crate::instructions::increment::handle_increment(ctx)
    }
}
```

### !constantsMod

_rs`pub mod constants`_ menjadikan `constants.rs` bagian dari crate program. Di
sinilah template menyimpan nilai-nilai program bersama seperti seed PDA, jumlah
transfer, dan nilai penghitung maksimum.

### !errorMod

_rs`pub mod error`_ menjadikan `error.rs` bagian dari crate program. File ini
mendefinisikan error kustom yang dapat dikembalikan program ketika suatu
instruksi tidak valid.

### !instructionsMod

_rs`pub mod instructions`_ menjadikan `instructions.rs` bagian dari crate
program. File tersebut berfungsi sebagai indeks untuk file-file instruksi
spesifik di dalam `src/instructions/`.

### !stateMod

_rs`pub mod state`_ menjadikan `state.rs` bagian dari crate program. File ini
mendefinisikan tata letak data untuk akun-akun yang dimiliki oleh program ini,
termasuk akun _rs`Counter`_.

### !prelude

_rs`anchor_lang::prelude::*`_ mengimpor tipe dan makro Anchor yang digunakan
untuk mendefinisikan program Solana, termasuk _rs`Context`_, _rs`Result`_,
_rs`Pubkey`_, _rs`Account`_, _rs`Signer`_, dan atribut akun Anchor.

### !reexports

Baris _rs`pub use`_ membuat item-item publik dari modul turunan tersedia dari
akar crate. Hal ini membuat titik masuk instruksi menjadi lebih singkat, seperti
_rs`Context<Initialize>`_ alih-alih jalur modul yang lebih panjang.

### !declareId

_rs`declare_id!`_ mendefinisikan alamat program. Nilai ini harus sesuai dengan
alamat di `Anchor.toml`, program account yang di-deploy, dan semua pengguna atau
pengujian yang memanggil program tersebut.

### !programMacro

_rs`#[program]`_ memberi tahu Anchor untuk menghasilkan titik masuk program
Solana dan kode dispatch instruksi. Fungsi-fungsi publik dalam modul ini menjadi
instruksi yang dapat dipanggil oleh pengguna.

### !publicInstructions

_rs`initialize`_ dan _rs`increment`_ adalah dua nama instruksi yang dapat
dipanggil pengguna. Masing-masing menerima sebuah _rs`Context<T>`_. Tipe di
dalam _rs`Context`_ mendefinisikan akun-akun yang diperlukan oleh instruksi
tersebut, dan Anchor memvalidasi akun-akun tersebut sebelum instruksi
dijalankan.

### !handlers

Fungsi handler berisi logika untuk instruksi tersebut.

</WithNotes>

keamanan.

### constants.rs

`constants.rs` menyimpan nilai-nilai bersama di satu tempat. Dalam template ini,
_rs`COUNTER_SEED`_ menurunkan PDA counter, _rs`HELLO_WORLD_LAMPORTS`_ ditransfer
saat inisialisasi, dan _rs`MAX_COUNT`_ diperiksa sebelum menambah nilai.

<WithNotes>

```rust title="programs/my-program/src/constants.rs" -w
use anchor_lang::prelude::*;

// !tooltip[/#\[constant\]/] anchorConstant
// !tooltip[/COUNTER_SEED/] counterSeed
#[constant]
pub const COUNTER_SEED: &[u8] = b"counter";

// !tooltip[/HELLO_WORLD_LAMPORTS/] helloLamports
#[constant]
pub const HELLO_WORLD_LAMPORTS: u64 = 1;

// !tooltip[/MAX_COUNT/] maxCount
#[constant]
pub const MAX_COUNT: u64 = 10;
```

### !anchorConstant

_rs`#[constant]`_ memberi tahu Anchor untuk menyertakan nilai tersebut dalam IDL
program. Hal ini membuat konstanta program yang penting terlihat oleh klien yang
dihasilkan dan tooling.

### !counterSeed

_rs`COUNTER_SEED`_ adalah string byte yang digunakan untuk menurunkan PDA
counter. Program dan pengujian keduanya menggunakan seed ini sehingga keduanya
sepakat mengenai alamat akun tanpa menyimpan keypair terpisah. Template ini
menggunakan satu seed tetap, sehingga program memiliki satu alamat counter.
Program nyata sering menyertakan alamat pengguna atau pengenal lain dalam seeds
untuk membuat banyak akun yang saling terkait.

### !helloLamports

_rs`HELLO_WORLD_LAMPORTS`_ adalah jumlah lamport yang ditransfer dari payer ke
akun counter di dalam _rs`initialize`_. Jumlahnya sengaja dibuat sangat kecil
karena template ini mendemonstrasikan CPI ke System Program.

### !maxCount

_rs`MAX_COUNT`_ adalah batas atas yang diperiksa oleh _rs`increment`_.
Menyimpannya di sini memberikan satu nama bersama untuk aturan tersebut kepada
program, pengujian, dan IDL yang dihasilkan.

</WithNotes>

<SideBySide>

### !left

```rust title="initialize.rs" -w
#[derive(Accounts)]
pub struct Initialize<'info> {
    // ...
    #[account(
        init,
        payer = payer,
        space = 8 + Counter::INIT_SPACE,
        // !mark
        // !mention counter-seed
        seeds = [COUNTER_SEED],
        bump
    )]
    pub counter: Account<'info, Counter>,
    // ...
}

pub fn handle_initialize(ctx: Context<Initialize>) -> Result<()> {
    // ...

    // !mark
    // !mention hello-world-lamports
    anchor_lang::system_program::transfer(cpi_ctx, HELLO_WORLD_LAMPORTS)?;

    Ok(())
}
```

### !right

`initialize.rs` menggunakan dua konstanta:

- [_rs`COUNTER_SEED`_](mention:counter-seed) mengambil alamat PDA counter.
- [_rs`HELLO_WORLD_LAMPORTS`_](mention:hello-world-lamports) menetapkan jumlah
  yang ditransfer dari payer ke akun counter.

</SideBySide>

<SideBySide>

### !left

```rust title="increment.rs" -w
pub fn handle_increment(ctx: Context<Increment>) -> Result<()> {
    // ...

    require!(
        // !mark
        // !mention max-count
        ctx.accounts.counter.count < MAX_COUNT,
        ErrorCode::CounterOverflow,
    );

    ctx.accounts.counter.count += 1;
    Ok(())
}
```

### !right

`increment.rs` menggunakan [_rs`MAX_COUNT`_](mention:max-count) sebagai batas
atas counter. Jika nilai count saat ini sudah mencapai maksimum, _rs`require!`_
mengembalikan _rs`CounterOverflow`_ dan data akun tidak berubah.

</SideBySide>

### state.rs

`state.rs` mendefinisikan tipe data kustom untuk akun yang dibuat dan dimiliki
oleh program. Program mendefinisikan instruksi untuk membuat, menginisialisasi,
dan memperbarui data tersebut, namun data counter tidak disimpan di dalam
program itu sendiri. Data tersebut disimpan dalam akun terpisah dengan alamatnya
sendiri.

<WithNotes>

```rust title="programs/my-program/src/state.rs" -w
use anchor_lang::prelude::*;

// !tooltip[/#\[account\]/] accountMacro
#[account]
// !tooltip[/InitSpace/] initSpace
#[derive(InitSpace)]
pub struct Counter {
    // !tooltip[/count/] count
    pub count: u64,
    // !tooltip[/authority/] authority
    pub authority: Pubkey,
}
```

### !accountMacro

_rs`#[account]`_ memberi tahu Anchor bahwa struct ini adalah data yang dapat
disimpan dalam akun onchain yang dimiliki oleh program ini. Anchor
menggunakannya untuk melakukan serialisasi, deserialisasi, dan validasi akun
_rs`Counter`_.

### !initSpace

_rs`InitSpace`_ memungkinkan Anchor menghitung _rs`Counter::INIT_SPACE`_, yaitu
jumlah byte yang dibutuhkan untuk field-field struct ini. Discriminator akun
Anchor 8-byte ditambahkan secara terpisah saat akun dibuat.

### !count

_rs`count`_ adalah nilai counter yang disimpan di dalam data akun.

### !authority

_rs`authority`_ menyimpan alamat yang diizinkan untuk menaikkan nilai counter.
_rs`initialize`_ menulis alamat ini, dan _rs`increment`_ mengharuskan alamat
yang sama untuk menandatangani.

</WithNotes>

<SideBySide>

### !left

```rust title="initialize.rs" -w
#[account(
    init,
    payer = payer,
    // !mark
    // !mention counter-space
    space = 8 + Counter::INIT_SPACE,
    seeds = [COUNTER_SEED],
    bump
)]
pub counter: Account<'info, Counter>,
// ...

pub fn handle_initialize(ctx: Context<Initialize>) -> Result<()> {
    // !mark(1:2)
    // !mention(1:2) initialize-fields
    ctx.accounts.counter.count = 0;
    ctx.accounts.counter.authority = ctx.accounts.payer.key();

    Ok(())
}
```

```rust title="increment.rs" -w
pub fn handle_increment(ctx: Context<Increment>) -> Result<()> {
    // ...

    // !mark
    // !mention increment-count
    ctx.accounts.counter.count += 1;
    Ok(())
}
```

### !right

`state.rs` mendefinisikan data akun _rs`Counter`_. File instruksi menggunakan
tipe tersebut saat membuat dan memperbarui akun:

- [_rs`Counter::INIT_SPACE`_](mention:counter-space) menentukan ukuran akun
  untuk field yang didefinisikan dalam `state.rs`.
- [_rs`count`_ dan _rs`authority`_](mention:initialize-fields) adalah nilai
  field yang ditulis saat akun diinisialisasi.
- [_rs`count += 1`_](mention:increment-count) memperbarui nilai counter yang
  tersimpan setelah validasi berhasil.

</SideBySide>

### error.rs

`error.rs` mendefinisikan error kustom program. Dalam template ini, error-error
tersebut mendemonstrasikan bagaimana handler instruksi berhenti ketika pemanggil
tidak diizinkan untuk memperbarui counter atau counter sudah mencapai
_rs`MAX_COUNT`_.

<WithNotes>

```rust title="programs/my-program/src/error.rs" -w
use anchor_lang::prelude::*;

// !tooltip[/#\[error_code\]/] errorCode
#[error_code]
pub enum ErrorCode {
    // !tooltip[/Unauthorized/] unauthorized
    #[msg("Only the counter authority can update this counter")]
    Unauthorized,
    // !tooltip[/CounterOverflow/] counterOverflow
    #[msg("Counter has reached the maximum value")]
    CounterOverflow,
}
```

### !errorCode

_rs`#[error_code]`_ memberi tahu Anchor untuk mengubah varian-varian ini menjadi
error program kustom. Klien dan pengujian dapat menggunakan error tersebut
daripada menebak dari kode error mentah.

### !unauthorized

_rs`Unauthorized`_ dikembalikan ketika penandatangan transaksi bukan authority
yang tersimpan di akun counter.

### !counterOverflow

_rs`CounterOverflow`_ dikembalikan ketika penambahan nilai akan melanggar aturan
_rs`MAX_COUNT`_ pada program.

</WithNotes>

<SideBySide>

### !left

```rust title="increment.rs" -w
pub fn handle_increment(ctx: Context<Increment>) -> Result<()> {
    require_keys_eq!(
        ctx.accounts.counter.authority,
        ctx.accounts.authority.key(),
        // !mark
        // !mention unauthorized-error
        ErrorCode::Unauthorized,
    );
    require!(
        ctx.accounts.counter.count < MAX_COUNT,
        // !mark
        // !mention overflow-error
        ErrorCode::CounterOverflow,
    );

    ctx.accounts.counter.count += 1;
    msg!("Hello, world! Counter is now {}", ctx.accounts.counter.count);
    Ok(())
}
```

### !right

`error.rs` menamai error yang dapat dikembalikan oleh instruksi:

- [_rs`ErrorCode::Unauthorized`_](mention:unauthorized-error) dikembalikan
  ketika penandatangan bukan authority yang tersimpan di akun counter.
- [_rs`ErrorCode::CounterOverflow`_](mention:overflow-error) dikembalikan ketika
  counter sudah mencapai _rs`MAX_COUNT`_.

</SideBySide>

### instructions.rs

`instructions.rs` menghubungkan file-file instruksi ke crate program sehingga
`lib.rs` dapat mengakses kode instruksi _rs`initialize`_ dan _rs`increment`_.
Setiap file instruksi mendefinisikan akun-akun yang diperlukan oleh instruksi
tersebut dan logika handler yang berjalan setelah Anchor memvalidasi akun-akun
tersebut.

<WithNotes>

```rust title="programs/my-program/src/instructions.rs" -w
// !tooltip[/pub mod initialize/] moduleDeclarations
pub mod initialize;
pub mod increment;

// !tooltip[/pub use initialize/] reexports
pub use initialize::*;
pub use increment::*;
```

### !moduleDeclarations

Baris _rs`pub mod`_ menjadikan `src/instructions/initialize.rs` dan
`src/instructions/increment.rs` sebagai bagian dari crate program.

### !reexports

Baris _rs`pub use`_ mengekspor ulang item publik dari file-file tersebut. Hal
ini membuat struct akun instruksi dan fungsi handler tersedia melalui modul
_rs`instructions`_.

</WithNotes>

### initialize.rs

`initialize.rs` mendefinisikan akun-akun yang diperlukan untuk membuat akun
counter, lalu menulis nilai awal akun tersebut. Struct _rs`#[derive(Accounts)]`_
menggunakan Anchor
[account constraints](https://www.anchor-lang.com/docs/references/account-constraints)
untuk menentukan akun mana yang diperlukan dan bagaimana akun counter baru
dibuat.

<WithNotes>

```rust title="programs/my-program/src/instructions/initialize.rs" -w
use anchor_lang::prelude::*;

use crate::{constants::*, state::Counter};

// !tooltip[/derive\(Accounts\)/] accountsStruct
#[derive(Accounts)]
pub struct Initialize<'info> {
    // !tooltip[/#\[account\(mut\)\]/] payerMutable
    #[account(mut)]
    // !tooltip[/Signer<'info>/] payerSigner
    pub payer: Signer<'info>,
    // !tooltip[/#\[account\(/] counterConstraints
    #[account(
        // !tooltip[/init/] initConstraint
        init,
        // !tooltip[/payer = payer/] payerConstraint
        payer = payer,
        // !tooltip[/space = 8 \+ Counter::INIT_SPACE/] space
        space = 8 + Counter::INIT_SPACE,
        // !tooltip[/seeds = \[COUNTER_SEED\]/] seeds
        seeds = [COUNTER_SEED],
        // !tooltip[/bump/] bump
        bump
    )]
    // !tooltip[/Account<'info, Counter>/] counterAccountType
    pub counter: Account<'info, Counter>,
    // !tooltip[/Program<'info, System>/] systemProgram
    pub system_program: Program<'info, System>,
}

// !tooltip[/handle_initialize/] handler
pub fn handle_initialize(ctx: Context<Initialize>) -> Result<()> {
    // !tooltip[/counter.count = 0/] initialCount
    ctx.accounts.counter.count = 0;
    // !tooltip[/counter.authority = ctx.accounts.payer.key\(\)/] initialAuthority
    ctx.accounts.counter.authority = ctx.accounts.payer.key();

    // !tooltip[/system_program::Transfer/] cpiAccounts
    let cpi_accounts = anchor_lang::system_program::Transfer {
        from: ctx.accounts.payer.to_account_info(),
        to: ctx.accounts.counter.to_account_info(),
    };
    // !tooltip[/CpiContext::new/] cpiContext
    let cpi_ctx = CpiContext::new(anchor_lang::system_program::ID, cpi_accounts);
    // !tooltip[/system_program::transfer/] transfer
    anchor_lang::system_program::transfer(cpi_ctx, HELLO_WORLD_LAMPORTS)?;

    // !tooltip[/msg!/] programLog
    msg!("Hello, world! Counter initialized");
    Ok(())
}
```

berarti akun pembayar dapat diubah karena lamport akan dipotong.

Akun _rs`counter`_ menyimpan data _rs`Counter`_ dari `state.rs`. _rs`init`_

## !!steps Ukuran Akun

Batasan _rs`space`_ memberi tahu Anchor berapa banyak data akun yang perlu
dialokasikan. Anchor menyimpan diskriminator 8-byte terlebih dahulu, kemudian
byte yang dibutuhkan untuk bidang _rs`Counter`_. Diskriminator memungkinkan

### !payerSigner

_rs`Signer<'info>`_ memberi tahu Anchor bahwa payer harus menandatangani

### !counterConstraints

Blok _rs`#[account(...)]`_ memberi tahu Anchor cara membuat dan memvalidasi akun

### !initConstraint

_rs`init`_ memberi tahu Anchor untuk membuat akun _rs`counter`_ sebelum handler

_rs`payer = payer`_ menentukan akun _rs`payer`_ sebagai pembayar untuk membuat
akun counter baru.

_rs`space = 8 + Counter::INIT_SPACE`_ mengalokasikan byte untuk discriminator
akun Anchor ditambah field _rs`Counter`_ dari `state.rs`. Discriminator membantu

### !seeds

_rs`seeds = [COUNTER_SEED]`_ menyatakan bahwa alamat akun counter harus berupa
PDA yang diturunkan dari seed ini dan ID program. Dengan satu seed tetap, hanya

program nyata, biasanya Anda perlu menambahkan seed lain seperti alamat
pengguna.

### !bump

_rs`bump`_ memberi tahu Anchor untuk menemukan dan memvalidasi bump PDA, yaitu
byte tambahan yang membuat alamat turunan valid sebagai PDA.

### !counterAccountType

_rs`Account<'info, Counter>`_ memberi tahu Anchor untuk memperlakukan data akun
sebagai tipe _rs`Counter`_ dari `state.rs`. Anchor juga memeriksa bahwa akun
dimiliki oleh program ini.

### !systemProgram

_rs`Program<'info, System>`_ mengharuskan pemanggil menyediakan System Program.
Anchor menggunakannya untuk constraint pembuatan akun _rs`init`_. Transfer CPI
pada handler juga memanggil System Program.

### !handler

Handler berjalan setelah Anchor memvalidasi akun dan membuat akun counter,

### !initialCount

Menetapkan field _rs`count`_ dalam akun counter ke _rs`0`_.

### !initialAuthority

Menetapkan field _rs`authority`_ dalam akun counter ke alamat pembayar.

_rs`Signer`_ mengharuskan tanda tangan dari authority. Handler kemudian
membandingkan alamat penanda tangan ini dengan authority yang tersimpan di akun
counter.

### !incrementHandler

Handler berjalan setelah Anchor memvalidasi daftar akun, memeriksa kepemilikan,
dan mendeserialisasi akun counter.

### !authorityCheck

_rs`require_keys_eq!`_ membandingkan dua public key dan menghentikan instruksi
kecuali alamat penanda tangan cocok dengan authority yang tersimpan di akun
counter. Ini adalah aturan otorisasi tingkat aplikasi pada template.

### !maxCheck

_rs`require!`_ memeriksa aturan boolean dan menghentikan instruksi dengan
_rs`CounterOverflow`_ jika counter telah mencapai _rs`MAX_COUNT`_. Aturan
template ini mendemonstrasikan error kustom dan penghentian sebelum mutasi.

### !mutateCount

Hanya setelah kedua pemeriksaan berhasil, handler memperbarui data akun counter.
Jika salah satu pemeriksaan gagal, transaksi gagal dan akun tidak diubah.

### !programLog

_rs`msg!`_ menulis pesan ke log program ketika program dipanggil.

</WithNotes>

## File pengujian

`programs/my-program/tests/test_initialize.rs` adalah pengujian integrasi Rust.
Ia tidak menjalankan validator lokal. Sebaliknya, ia memuat file `.so` yang
telah dikompilasi ke dalam LiteSVM, membangun transaksi yang memanggil program,
dan membaca akun counter setelah setiap transaksi. Pengujian ini membangun
instruksi untuk transaksi Solana dengan menentukan ID program yang akan
dipanggil, menyediakan instruction data, dan meneruskan akun yang diperlukan.

use {
    anchor_lang::{
        prelude::Pubkey,
        solana_program::{instruction::Instruction, system_program},
        AccountDeserialize, InstructionData, ToAccountMetas,
    },
    // !tooltip[/litesvm::LiteSVM/] litesvm
    litesvm::LiteSVM,
    solana_keypair::Keypair,
    solana_message::{Message, VersionedMessage},
    solana_signer::Signer,
    solana_transaction::versioned::VersionedTransaction,

#[test]
fn test_initialize() {
    // !tooltip[/let program_id/] setup
    let program_id = my_program::id();
    let payer = Keypair::new();
    // !tooltip[/Pubkey::find_program_address/] counterPda
    let counter = Pubkey::find_program_address(
        &[my_program::constants::COUNTER_SEED],
        &program_id,
    )
    .0;
    let mut svm = LiteSVM::new();
    // !tooltip[/include_bytes!/] programBytes
    let bytes = include_bytes!(concat!(
        env!("CARGO_TARGET_TMPDIR"),
        "/../deploy/my_program.so"
    ));
    // !tooltip[/svm.add_program/] addProgram
    svm.add_program(program_id, bytes).unwrap();
    svm.airdrop(&payer.pubkey(), 1_000_000_000).unwrap();

    let instruction = Instruction::new_with_bytes(
        program_id,
        &my_program::instruction::Initialize {}.data(),
        my_program::accounts::Initialize {
            payer: payer.pubkey(),
            counter,
            system_program: system_program::ID,
        }
        .to_account_metas(None),
    );

    let blockhash = svm.latest_blockhash();
    // !tooltip[/Message::new_with_blockhash/] message
    let msg = Message::new_with_blockhash(&[instruction], Some(&payer.pubkey()), &blockhash);
    // !tooltip[/VersionedTransaction::try_new/] transaction
    let tx = VersionedTransaction::try_new(VersionedMessage::Legacy(msg), &[&payer]).unwrap();

    // !tooltip[/svm.send_transaction/] sendTx
    let res = svm.send_transaction(tx);
    assert!(res.is_ok());

    // !tooltip[/svm.get_account/] getCounter
    let counter_account = svm.get_account(&counter).unwrap();
    let mut data: &[u8] = &counter_account.data;
    // !tooltip[/Counter::try_deserialize/] deserialize
    let counter_state = my_program::state::Counter::try_deserialize(&mut data).unwrap();
    // !tooltip[/assert_eq!\(counter_state.count, 0\)/] initAssert
    assert_eq!(counter_state.count, 0);
    assert_eq!(counter_state.authority, payer.pubkey());

    let instruction = Instruction::new_with_bytes(

### !litesvm

_rs`LiteSVM`_ adalah Solana VM dalam proses untuk pengujian. Ini memungkinkan
pengujian mendaftarkan program yang telah dikompilasi, membuat akun, mengirim
transaksi, dan memeriksa status akun tanpa memulai proses validator.

### !setup

Pengujian mendapatkan alamat program dari nilai _rs`declare_id!`_ melalui
_rs`my_program::id()`_ dan membuat keypair pembayar baru untuk transaksi
pengujian.

### !counterPda

Alamat counter diturunkan dari _rs`COUNTER_SEED`_ yang sama yang digunakan oleh
program. Ini membuat pengujian menargetkan PDA yang diharapkan oleh
`initialize.rs`.

### !programBytes

_rs`include_bytes!`_ membaca file `my_program.so` yang dikompilasi yang
dihasilkan oleh build. Byte-byte tersebut adalah kode program yang dapat
dieksekusi yang akan dijalankan LiteSVM dalam pengujian.

### !addProgram

_rs`svm.add_program`_ mendaftarkan byte program yang dikompilasi pada alamat
program di LiteSVM. Airdrop memberikan payer cukup SOL lokal untuk membayar
transaksi dan pembuatan akun.

### !testInstruction

_rs`Instruction::new_with_bytes`_ membuat instruksi untuk memanggil program.
Instruksi ini mencakup ID program, instruction data, dan akun instruksi.

### !blockhash

Setiap transaksi Solana membutuhkan blockhash terkini. Dalam pengujian ini,
LiteSVM menyediakan blockhash yang digunakan untuk transaksi.

### !message

Pesan berisi instruksi, fee payer, dan blockhash. Ini adalah data transaksi yang
akan ditandatangani.

### !transaction

_rs`VersionedTransaction::try_new`_ menandatangani pesan dengan keypair payer.
Payer menandatangani karena membayar biaya transaksi dan mendanai akun counter
baru.

### !sendTx

_rs`svm.send_transaction`_ mengeksekusi transaksi di LiteSVM.

### !getCounter

Setelah transaksi berjalan, pengujian membaca akun counter langsung dari LiteSVM
sehingga dapat memeriksa data akun.

### !deserialize

Akun counter menyimpan data yang telah diserialisasi.
_rs`Counter::try_deserialize`_ memeriksa discriminator akun Anchor dan mengubah
data akun kembali menjadi tipe _rs`Counter`_ dari `state.rs`.

### !initAssert

Setelah _rs`initialize`_ berhasil, pengujian melakukan deserialisasi akun
counter dan memeriksa bahwa kondisi awal sesuai dengan yang ditulis oleh
_rs`handle_initialize`_.

### !incrementAssert

Setelah _rs`increment`_ berhasil, akun counter yang sama seharusnya memiliki
_rs`count`_ yang diset ke _rs`1`_ sambil tetap mempertahankan authority yang
sama.

</WithNotes>

## Konfigurasi proyek

File proyek root memberi tahu Anchor dan Cargo cara membangun, menguji, dan
men-deploy program. Untuk referensi lengkap, lihat dokumentasi Anchor untuk
[konfigurasi Anchor.toml](https://www.anchor-lang.com/docs/references/anchor-toml)
dan [Anchor CLI](https://www.anchor-lang.com/docs/references/cli).

<WithNotes>

```toml title="Anchor.toml"
# !tooltip[/skip_local_validator/] skipValidator
skip_local_validator = true

[toolchain]

[features]
resolution = true
skip-lint = false

# !tooltip[/programs.localnet/] programs
[programs.localnet]
my_program = "82sFkffP9wxwpyfZyeaKHH2chQoJPUGsJZSPi9mrUuXd"

# !tooltip[/provider/] provider
[provider]
cluster = "localnet"
wallet = "~/.config/solana/id.json"

# !tooltip[/scripts/] scripts
[scripts]
test = "cargo test"

[hooks]
```

### !skipValidator

`skip_local_validator = true` memberi tahu Anchor untuk tidak memulai validator
untuk `anchor test`. Pengujian Rust menggunakan LiteSVM sebagai gantinya.

### !programs

`[programs.localnet]` memetakan nama crate Rust ke alamat program yang digunakan
untuk build dan pengujian lokal.

### !provider

`[provider]` menetapkan cluster dan wallet default untuk perintah seperti
`anchor deploy`.

### !scripts

`[scripts]` mengubah apa yang dijalankan oleh `anchor test`. Dalam template ini,
`anchor test` menjalankan `cargo test`.

</WithNotes>
