Keychainは署名バックエンドを抽象化しますが、本番環境での署名者の運用は引き続きお客様の責任です。これらのプラクティスはすべてのバックエンドに適用されます。バックエンドの選択については、
[バックエンドの選択](/docs/tools/keychain/choosing-a-backend)を参照してください。

## ホット署名者とコールド署名者を分離する

残高の少ない運用用（「ホット」）署名者を頻繁な自動トランザクションに使用し、別の高価値（「コールド」）署名者を資金移動に使用します。コールド署名者はMPCカストディまたはクラウドHSMでバックアップし、署名前に承認ポリシーを必須とします。インターフェースは同一であるため、各ロールは呼び出し箇所を変更することなく異なるバックエンドを使用できます。

## 計画的に鍵をローテーションする

Solanaでは署名鍵がアカウントアドレスそのものであるため、鍵をローテーションすると新しいアドレスが生成されます。ローテーションはマイグレーションとして計画してください。新しい鍵をプロビジョニングし、関連する権限（または資金）を新しいアドレスに再割り当てしてから、古い鍵を廃止します。環境ごとに異なる鍵を使用し、devnetの鍵がmainnetで再利用されないようにしてください。

## 最小権限アクセスを付与する

各バックエンドのクレデンシャルを署名に必要な最小限の範囲に制限します。AWS
KMSの場合、特定の鍵に対する`kms:Sign`と`kms:DescribeKey`が該当します。静的クレデンシャルよりもIAMロールを優先してください。同じ原則をVaultポリシーおよびマネージドウォレットAPIキーにも適用します。

## 可用性を計画する

リモートバックエンドはネットワークサービスに依存します。署名者に依存する前に`isAvailable()`を呼び出し、一時的な障害にはバックオフ付きのリトライを適用し、停止時のフォールバックパスを定義してください。KeychainはリモートバックエンドにHTTPSを強制し、デフォルトのリクエストタイムアウトを適用します。

## シークレットとトランスポートの保護

- すべてのリモートバックエンドはHTTPS経由で通信し、HTTP（非HTTPS）エンドポイントは拒否されます。
- APIキー、トークン、認証情報はシークレットマネージャーまたは環境変数から供給してください — ハードコーディングやソース管理へのコミットは絶対に行わないでください。
- Keychainはシークレットがログに漏洩しないようエラーレスポンスをサニタイズします。独自のログにも生の認証情報やキーマテリアルが含まれないようにしてください。

## 署名の監視と監査

署名の成功率、レイテンシ、失敗回数を追跡し、急増時にアラートを設定してください —詳細は[本番環境への対応](/docs/payments/production-readiness#monitoring)のメトリクスをご参照ください。プロバイダー側の監査ログ（KMS、Vault、またはカストディアン）を有効にし、アプリケーションが送信したトランザクションと照合してください。
