トランザクションは、すべての必須署名者から有効な
[署名](/docs/core/transactions/transaction-structure#signatures)が付与されている場合にのみ実行されます。ローカル開発時は通常、[`Keypair`](/developers/cookbook/wallets/create-keypair)として鍵を保持し、直接署名します。本番環境では、問いかけが変わります：**秘密鍵はどこに保管され、誰がその鍵を使って署名を承認できるのか？**

<Callout type="warn">
  秘密鍵をクライアント側のコードに埋め込んだり、フロントエンドにバンドルしたり、ソース管理にコミットしたりしないでください。鍵を入手した者は、そのアカウントの資金を完全に制御できます。
</Callout>

## 署名方式

| 方式                           | 鍵の保管場所                                                      | 署名者                                     | 最適なユースケース                                           |
| ------------------------------ | ----------------------------------------------------------------- | ------------------------------------------ | ------------------------------------------------------------ |
| **ローカルkeypair**            | マシン上のファイルまたは環境変数                                  | コードが直接署名                           | ローカル開発、テスト、CI                                     |
| **ブラウザウォレット**         | エンドユーザーのウォレット（Phantom、Solflareなど）               | エンドユーザーがトランザクションごとに署名 | ユーザーが自分のトランザクションに署名するフロントエンドdApp |
| **クラウドKMS / HSM**          | ハードウェアベースの鍵管理サービス（AWS KMS、GCP KMS、Vaultなど） | バックエンドがサービスに署名をリクエスト   | バックエンドサービス、トレジャリー運用、規制対応環境         |
| **マネージド / MPCウォレット** | プロバイダーのインフラ全体で分散または管理                        | プロバイダーがポリシーに従って共同署名     | 組み込みウォレット、承認ワークフロー、機関向けカストディ     |

ローカルkeypairは、生のキーマテリアルをアプリケーション内部に保持する唯一のアプローチです。本番環境向けのアプローチでは、キーを専用インフラに保管し、そのインフラに代わって署名を依頼します。

## バックエンド署名

サーバーが署名を行う必要がある場合（手数料の支払い、プログラムトランザクションの送信、またはトレジャリーの運用など）は、生のkeypairではなく専用のキー管理バックエンドを使用してください。Solana
Foundationは[Keychain](/docs/tools/keychain)を提供しています。これは統合署名ライブラリ（RustおよびTypeScript）であり、上記の表に掲載されているすべてのバックエンド（Memory、HashiCorp
Vault、AWS KMS、GCP
KMS、Fireblocks、Privy、Turnkey、CDP、Crossmint、Dfns、Para、Openfort、Utila）にわたって単一の`SolanaSigner`インターフェースを公開しています。

インターフェースはすべてのバックエンドで同一であるため、ローカル開発時はインメモリキーを使用し、アプリケーションコードを書き直すことなく設定変更だけで本番バックエンドに切り替えることができます。Keychainは[`@solana/kit`](/docs/clients/official/javascript)および[Rust SDK](/docs/clients/official/rust)と互換性があります。
