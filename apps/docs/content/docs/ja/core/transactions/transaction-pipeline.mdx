<Callout type="info" title="概要">
  トランザクションは8つの段階を通過します：受信、署名検証、サニタイズ、予算/有効期限チェック、手数料支払者検証、アカウント読み込み、instruction実行、コミット。
</Callout>

## トランザクション処理パイプライン

トランザクションがvalidatorに到着すると、一連の検証と実行段階を通過します。以下では、受信からコミットまでの完全なパイプラインを、[agave](https://github.com/anza-xyz/agave/tree/v3.1.8)
validatorクライアントへのソースファイル参照とともに説明します。

### 1. 受信と逆シリアル化

# validatorはUDP/QUIC経由でトランザクションバイト列を受信します。生のバイト列は単一のパケット内に収まる必要があります（_rs`PACKET_DATA_SIZE`_

1,232バイト）。バイト列は*rs`VersionedTransaction`*に逆シリアル化され、これには署名配列と*rs`VersionedMessage`*（レガシーまたはv0）が含まれます。

### 2. 署名検証（sigverify）

署名は、トランザクションがバンキング段階に入る前に[sigverify段階](https://github.com/anza-xyz/agave/blob/v3.1.8/perf/src/sigverify.rs)で検証されます。インデックス`i`の各署名について、検証者はEd25519(`signatures[i]`,
`account_keys[i]`,
`message_bytes`)をチェックします。署名が無効な場合、パケットは破棄されます。

検証は並列化されています：validatorはパケットバッチを[`VERIFY_PACKET_CHUNK_SIZE`](https://github.com/anza-xyz/agave/blob/v3.1.8/perf/src/sigverify.rs#L27)（128）のチャンクに分割し、並列処理します。

### 3. サニタイズ

逆シリアル化されたトランザクションはサニタイズされ、_rs`SanitizedTransaction`_（または[`RuntimeTransaction`](https://github.com/anza-xyz/agave/blob/v3.1.8/runtime-transaction/src/runtime_transaction.rs#L31)）が生成されます。サニタイズは構造的な不変条件を検証します：

- 署名の数がヘッダー内の`num_required_signatures`と一致すること
- すべてのinstructionのINLINE_CODE_PLACEHOLDER_1578cc67900f668bとINLINE_CODE_PLACEHOLDER_8c4bda500419320aが範囲内にあること
- 手数料支払者（アカウントインデックス0）が書き込み可能な署名者であること

*rs`RuntimeTransaction`*ラッパーは、[`TransactionMeta`](https://github.com/anza-xyz/agave/blob/v3.1.8/runtime-transaction/src/transaction_meta.rs#L38-L44)から事前計算されたメタデータをキャッシュします：メッセージハッシュ、投票トランザクションフラグ、プリコンパイル署名数（Ed25519/secp256k1/secp256r1）、計算予算instruction詳細、および合計instruction データ長。

### 4. 計算予算、経過期間、ステータスキャッシュの確認

[`check_transactions`](https://github.com/anza-xyz/agave/blob/v3.1.8/runtime/src/bank/check_transactions.rs#L55-L69)
メソッドは、トランザクションごとに複数のチェックを実行します:

**計算予算**: トランザクションの計算予算instructions が最初に解析および検証されます。予算の上限と優先度手数料から手数料の詳細が計算されます。計算予算が無効または競合している場合、トランザクションは*rs`DuplicateInstruction`*、`InstructionError(..., InvalidInstructionData)`、または*rs`InvalidLoadedAccountsDataSizeLimit`*などの計算予算解析エラーで失敗します。

**ブロックハッシュの経過期間**: トランザクションの`recent_blockhash`が[`BlockhashQueue`](https://github.com/anza-xyz/agave/blob/v3.1.8/accounts-db/src/blockhash_queue.rs#L33)で検索されます。ハッシュが見つかり、その経過期間が*rs`MAX_PROCESSING_AGE`*(150
slot)以内の場合、トランザクションは続行されます。見つからない場合、validatorは有効な[永続ナンス](/docs/core/transactions/durable-nonces)を確認します。

**ステータスキャッシュ**: トランザクションのメッセージハッシュがステータスキャッシュと照合されます。見つかった場合、トランザクションは*rs`AlreadyProcessed`*で拒否されます。

### 5. ナンスと手数料支払者の検証

SVMの[`validate_transaction_nonce_and_fee_payer`](https://github.com/anza-xyz/agave/blob/v3.1.8/svm/src/transaction_processor.rs#L580-L620)メソッドは、2つの検証を処理します:

**ナンス検証**(該当する場合): ナンストランザクションの場合、validatorはナンスアカウントを読み込み、以下を検証します:

- アカウントがSystem Programによって所有されていること
- *rs`State::Initialized`*として解析されること
- 保存されている永続ナンスがトランザクションの`recent_blockhash`と一致すること
- ナンスを進めることができること(現在の永続ナンスが次の永続ナンスと異なる、つまりナンスが現在のブロックで既に使用されていないこと)
- ナンス権限がトランザクションに署名していること

有効な場合、ナンスは次の永続ナンス値に進められます。[`validate_transaction_nonce`](https://github.com/anza-xyz/agave/blob/v3.1.8/svm/src/transaction_processor.rs#L676-L729)を参照してください。

**手数料支払者の検証**: 手数料支払者アカウント(常にインデックス0)は、[`validate_fee_payer`](https://github.com/anza-xyz/agave/blob/v3.1.8/svm/src/account_loader.rs#L369-L417)によって読み込まれ、確認されます:

- アカウントが存在すること(lamport > 0)、そうでない場合は*rs`AccountNotFound`*
- アカウントがシステムアカウントまたはナンスアカウントであること、そうでない場合は*rs`InvalidAccountForFee`*
- lamportが`min_balance + total_fee`をカバーしていること。`min_balance`はシステムアカウントの場合は0、ナンスアカウントの場合は`rent.minimum_balance(NonceState::size())`。そうでない場合は*rs`InsufficientFundsForFee`*
- 手数料控除後もアカウントがレント免除を維持していること(レント免除からレント支払いへの移行は不可)

この段階で、手数料は手数料支払者から差し引かれます。手数料が差し引かれた手数料支払者（該当する場合は進行したnonce）のスナップショットは
_rs`RollbackAccounts`_
として保存され、これらは実行が失敗した場合でもコミットされるアカウントです。

### 6. アカウントの読み込み

[`load_transaction`](https://github.com/anza-xyz/agave/blob/v3.1.8/svm/src/account_loader.rs#L419-L455)
は、トランザクションによって参照されるすべてのアカウントを読み込みます。[`AccountLoader`](https://github.com/anza-xyz/agave/blob/v3.1.8/svm/src/account_loader.rs#L161-L165)
は外部アカウントストアをラップし、バッチローカルキャッシュを維持することで、同じバッチ内の以前のトランザクションによって変更されたアカウントが後続のトランザクションから見えるようにします。

手数料支払者以外の各アカウントに対して、ローダーは以下を実行します：

1. キャッシュまたはaccounts-dbからアカウントを取得します
2. 必要に応じてレント免除ステータスを更新します
3. アカウントのデータサイズを
   [`loaded_accounts_data_size_limit`](https://github.com/anza-xyz/agave/blob/v3.1.8/program-runtime/src/execution_budget.rs#L53)（デフォルト64
   MiB）に累積します。各アカウントは
   [`TRANSACTION_ACCOUNT_BASE_SIZE`](https://github.com/anza-xyz/agave/blob/v3.1.8/svm/src/account_loader.rs#L45)（64バイト）の基本オーバーヘッドとそのデータ長を発生させます

トランザクションのinstructionsによって呼び出される各programに対して、ローダーはprogram
accountが存在し、有効なローダー（`NativeLoader` または _rs`PROGRAM_OWNERS`_
のいずれか）によって所有されていることを検証します。無効なprogramは
_rs`ProgramAccountNotFound`_ または _rs`InvalidProgramForExecution`_
で失敗します。

LoaderV3（アップグレード可能）programは、関連するprogramdata
accountを暗黙的に読み込み、これも読み込まれたデータサイズ制限にカウントされます。

アカウントの読み込みに失敗したが手数料支払者の検証が成功した場合、トランザクションは
[`FeesOnly`](https://github.com/anza-xyz/agave/blob/v3.1.8/svm/src/account_loader.rs#L147-L152)
の結果になります：手数料は引き続き徴収されますが、instructionsは実行されません。

### 7. instructionsの実行

[`execute_loaded_transaction`](https://github.com/anza-xyz/agave/blob/v3.1.8/svm/src/transaction_processor.rs#L847)
は、読み込まれたすべてのアカウントを含む _rs`TransactionContext`_
を作成し、[`process_message`](https://github.com/anza-xyz/agave/blob/v3.1.8/svm/src/message_processor.rs)
を呼び出します。instructionsは、メッセージ内に表示される順序で順次実行されます。各instructionの呼び出しは
_rs`InvokeContext`_ を作成し、ターゲットprogramを呼び出します。

#### instructionの処理詳細

ランタイムの
[`process_message`](https://github.com/anza-xyz/agave/blob/v3.1.8/svm/src/message_processor.rs#L15)
関数は、各instructionを反復処理し、ターゲットprogramを呼び出します：

1. 各instructionに対して、ランタイムは
   [`prepare_next_top_level_instruction`](https://github.com/anza-xyz/agave/blob/v3.1.8/program-runtime/src/invoke_context.rs#L436)を呼び出し、
   [`InstructionContext`](https://github.com/anza-xyz/agave/blob/v3.1.8/transaction-context/src/lib.rs#L519)を構築します。このコンテキストには、instructionのアカウント（コンパイルされたインデックスから解決）、instruction
   data、およびprogram accountインデックスへの参照が含まれます。
2. ランタイムは、プログラムが
   [プリコンパイル](/docs/core/programs/precompiles)（Ed25519、Secp256k1、Secp256r1）であるかどうかを確認します。プリコンパイルはBPF
   VMを呼び出さずに直接検証されます。
3. その他のすべてのプログラムについては、ランタイムが
   [`process_instruction`](https://github.com/anza-xyz/agave/blob/v3.1.8/program-runtime/src/invoke_context.rs#L477)を呼び出し、キャッシュからプログラムをロードし、BPF仮想マシンで実行します。
4. instructionが完了すると、ランタイムは、すべてのinstructionアカウント全体のlamport残高の合計が変更されていないことを
   [検証](https://github.com/anza-xyz/agave/blob/v3.1.8/transaction-context/src/lib.rs#L366-L398)します（*rs`UnbalancedInstruction`*チェック）。
5. いずれかのinstructionが失敗すると、トランザクション全体がロールバックされます。中間状態の変更はコミットされません。

各instructionはinstructionトレースを増分します。トレースには、トップレベルのinstructionsと、それらが呼び出すすべての[CPI](/docs/core/cpi)の両方が含まれます。トレースの合計長（トップレベルのinstructionsとすべてのネストされたCPIを含む）は64を超えることはできません（_rs`MAX_INSTRUCTION_TRACE_LENGTH`_）。この制限を超えると、
*rs`InstructionError::MaxInstructionTraceLengthExceeded`*が返されます。

実行後、ランタイムは次のことを検証します：

- すべてのアカウントのlamportの合計が変更されていないこと
- 賃料免除から賃料支払いにアカウントが移行していないこと

### 8. コミットまたはロールバック

実行が成功すると、`TransactionContext`から変更されたアカウント状態が
*rs`AccountLoader`*のバッチローカルキャッシュに書き戻されます。実行が失敗すると、_rs`RollbackAccounts`_（手数料が差し引かれた手数料支払者と進められたnonce）のみが書き戻されます。手数料は依然として徴収されますが、他のすべてのアカウント変更は破棄されます。

### パイプラインの概要

```
Receive packet (UDP/QUIC)
  --> Deserialize into VersionedTransaction
  --> Sigverify (parallel Ed25519 verification)
  --> Sanitize (structural validation, metadata extraction)
  --> Parse compute budget, calculate fees
  --> Check blockhash age (or verify nonce account)
  --> Check status cache (dedup)
  --> Validate nonce authority and advanceability (if nonce transaction)
  --> Validate fee payer (load, check balance, deduct fee)
  --> Load all accounts (with data size limits)
  --> Load programs (verify loaders)
  --> Execute instructions sequentially
  --> Verify post-conditions (lamport balance, rent state)
  --> Commit account changes (or rollback on failure)
```
