<Callout type="info" title="概要">
  トランザクションは8つの段階を経ます：受信、署名検証、サニタイズ、予算/経過時間チェック、手数料支払者の検証、アカウント読み込み、instructions実行、およびコミットです。
</Callout>

## トランザクション処理パイプライン

トランザクションがvalidatorに到着すると、一連の検証および実行段階を経ます。以下では、受信からコミットまでの完全なパイプラインを、[agave](https://github.com/anza-xyz/agave/tree/v3.1.8)
validatorクライアントへのソースファイル参照とともに説明します。

### 1. 受信とデシリアライズ

# validatorはUDP/QUIC経由でトランザクションバイトを受信します。生バイトは単一パケット内に収まる必要があります（_rs`PACKET_DATA_SIZE`_

1,232バイト）。バイトは*rs`VersionedTransaction`*にデシリアライズされ、署名配列と*rs`VersionedMessage`*（レガシーまたはv0）が含まれます。

### 2. 署名検証（sigverify）

署名は、トランザクションがバンキング段階に入る前に[sigverify段階](https://github.com/anza-xyz/agave/blob/v3.1.8/perf/src/sigverify.rs)で検証されます。インデックス`i`の各署名について、検証プログラムはEd25519(`signatures[i]`,
`account_keys[i]`,
`message_bytes`)をチェックします。署名が無効な場合、パケットは破棄されます。

検証は並列化されています：validatorはパケットバッチを[`VERIFY_PACKET_CHUNK_SIZE`](https://github.com/anza-xyz/agave/blob/v3.1.8/perf/src/sigverify.rs#L27)（128）のチャンクに分割し、並列処理します。

### 3. サニタイズ

デシリアライズされたトランザクションはサニタイズされ、_rs`SanitizedTransaction`_（または[`RuntimeTransaction`](https://github.com/anza-xyz/agave/blob/v3.1.8/runtime-transaction/src/runtime_transaction.rs#L31)）が生成されます。サニタイズは構造的な不変条件を検証します：

- 署名数がヘッダー内の`num_required_signatures`と一致すること
- すべてのinstructionのINLINE_CODE_PLACEHOLDER_1578cc67900f668bおよびINLINE_CODE_PLACEHOLDER_8c4bda500419320aが範囲内にあること
- 手数料支払者（アカウントインデックス0）が書き込み可能な署名者であること

*rs`RuntimeTransaction`*ラッパーは、[`TransactionMeta`](https://github.com/anza-xyz/agave/blob/v3.1.8/runtime-transaction/src/transaction_meta.rs#L38-L44)から事前計算されたメタデータをキャッシュします：メッセージハッシュ、投票トランザクションフラグ、プリコンパイル署名数（Ed25519/secp256k1/secp256r1）、計算予算instruction詳細、および合計instruction
dataの長さです。

### 4. 計算予算、経過時間、ステータスキャッシュの確認

[`check_transactions`](https://github.com/anza-xyz/agave/blob/v3.1.8/runtime/src/bank/check_transactions.rs#L55-L69)
メソッドは、トランザクションごとに複数のチェックを実行します:

**計算予算**: トランザクションの計算予算instructionsが最初に解析され、検証されます。手数料の詳細は予算制限と優先順位手数料から計算されます。計算予算が無効または競合している場合、トランザクションは*rs`DuplicateInstruction`*、`InstructionError(..., InvalidInstructionData)`、または*rs`InvalidLoadedAccountsDataSizeLimit`*などの計算予算解析エラーで失敗します。

**ブロックハッシュの経過時間**: トランザクションの`recent_blockhash`が
[`BlockhashQueue`](https://github.com/anza-xyz/agave/blob/v3.1.8/accounts-db/src/blockhash_queue.rs#L33)で検索されます。ハッシュが見つかり、その経過時間が*rs`MAX_PROCESSING_AGE`*(150スロット)以内であれば、トランザクションは続行されます。見つからない場合、validatorは有効な
[永続的ノンス](/docs/core/transactions/durable-nonces)をチェックします。

**ステータスキャッシュ**: トランザクションのメッセージハッシュがステータスキャッシュに対してチェックされます。見つかった場合、トランザクションは*rs`AlreadyProcessed`*で拒否されます。

### 5. ノンスと手数料支払者の検証

SVMの
[`validate_transaction_nonce_and_fee_payer`](https://github.com/anza-xyz/agave/blob/v3.1.8/svm/src/transaction_processor.rs#L580-L620)
メソッドは2つの検証を処理します:

**ノンス検証**(該当する場合): ノンストランザクションの場合、validatorはノンスアカウントを読み込み、以下を確認します:

- アカウントがSystem Programによって所有されている
- *rs`State::Initialized`*として解析できる
- 保存された永続的ノンスがトランザクションの`recent_blockhash`と一致する
- ノンスを進めることができる(現在の永続的ノンスが次の永続的ノンスと異なる、つまりノンスが現在のブロックでまだ使用されていない)
- ノンス権限がトランザクションに署名している

有効な場合、ノンスは次の永続的ノンス値に進められます。
[`validate_transaction_nonce`](https://github.com/anza-xyz/agave/blob/v3.1.8/svm/src/transaction_processor.rs#L676-L729)を参照してください。

**手数料支払者検証**: 手数料支払者アカウント(常にインデックス0)が読み込まれ、
[`validate_fee_payer`](https://github.com/anza-xyz/agave/blob/v3.1.8/svm/src/account_loader.rs#L369-L417)によってチェックされます:

- アカウントが存在する必要がある(lamports > 0)、そうでなければ*rs`AccountNotFound`*
- アカウントはシステムアカウントまたはノンスアカウントである必要がある、そうでなければ
  _rs`InvalidAccountForFee`_
- lamportsは`min_balance + total_fee`をカバーする必要がある。ここでINLINE*CODE_PLACEHOLDER_f8938e0775e6829bENDはシステムアカウントの場合は0、ノンスアカウントの場合は`rent.minimum_balance(NonceState::size())`。そうでなければ\_rs`InsufficientFundsForFee`*
- 手数料控除後、アカウントはレント免除状態を維持する必要がある(レント免除状態からレント支払い状態への移行は不可)

この段階で、手数料は手数料支払者から差し引かれます。手数料が差し引かれた手数料支払者（および該当する場合は進行したnonce）のスナップショットは*rs`RollbackAccounts`*として保存され、これらは実行が失敗した場合でもコミットされるアカウントです。

### 6. アカウントの読み込み

[`load_transaction`](https://github.com/anza-xyz/agave/blob/v3.1.8/svm/src/account_loader.rs#L419-L455)は、トランザクションによって参照されるすべてのアカウントを読み込みます。[`AccountLoader`](https://github.com/anza-xyz/agave/blob/v3.1.8/svm/src/account_loader.rs#L161-L165)は、外部アカウントストアをラップし、バッチローカルキャッシュを維持することで、同じバッチ内の前のトランザクションによって変更されたアカウントが後続のトランザクションから見えるようにします。

各非手数料支払者アカウントについて、ローダーは以下を実行します：

1. キャッシュまたはaccounts-dbからアカウントを取得
2. 必要に応じてレント免除ステータスを更新
3. アカウントのデータサイズを[`loaded_accounts_data_size_limit`](https://github.com/anza-xyz/agave/blob/v3.1.8/program-runtime/src/execution_budget.rs#L53)（デフォルト64
   MiB）に累積。各アカウントは、[`TRANSACTION_ACCOUNT_BASE_SIZE`](https://github.com/anza-xyz/agave/blob/v3.1.8/svm/src/account_loader.rs#L45)（64バイト）の基本オーバーヘッドとそのデータ長を負担します

トランザクションのinstructionsによって呼び出される各プログラムについて、ローダーはプログラムアカウントが存在し、有効なローダー（`NativeLoader`または*rs`PROGRAM_OWNERS`*のいずれか）によって所有されていることを検証します。無効なプログラムは*rs`ProgramAccountNotFound`*または*rs`InvalidProgramForExecution`*で失敗します。

LoaderV3（アップグレード可能）プログラムは、関連するprogramdataアカウントを暗黙的に読み込み、これも読み込まれたデータサイズの制限にカウントされます。

アカウントの読み込みが失敗したものの手数料支払者の検証が成功した場合、トランザクションは[`FeesOnly`](https://github.com/anza-xyz/agave/blob/v3.1.8/svm/src/account_loader.rs#L147-L152)結果となります：手数料は依然として徴収されますが、instructionsは実行されません。

### 7. instructionsの実行

[`execute_loaded_transaction`](https://github.com/anza-xyz/agave/blob/v3.1.8/svm/src/transaction_processor.rs#L847)は、すべての読み込まれたアカウントを含む*rs`TransactionContext`*を作成し、[`process_message`](https://github.com/anza-xyz/agave/blob/v3.1.8/svm/src/message_processor.rs)を呼び出します。instructionsは、メッセージ内に現れる順序で順次実行されます。各instruction呼び出しは*rs`InvokeContext`*を作成し、ターゲットプログラムを呼び出します。

#### instructionの処理詳細

ランタイムの[`process_message`](https://github.com/anza-xyz/agave/blob/v3.1.8/svm/src/message_processor.rs#L15)関数は、各instructionを反復処理し、ターゲットプログラムを呼び出します：

1. 各instructionに対して、ランタイムは
   [`prepare_next_top_level_instruction`](https://github.com/anza-xyz/agave/blob/v3.1.8/program-runtime/src/invoke_context.rs#L436)を呼び出し、
   [`InstructionContext`](https://github.com/anza-xyz/agave/blob/v3.1.8/transaction-context/src/lib.rs#L519)を構築します。このコンテキストには、instructionのアカウント（コンパイルされたインデックスから解決）、instruction
   data、およびprogram accountインデックスへの参照が含まれます。
2. ランタイムは、プログラムが
   [プリコンパイル](/docs/core/programs/precompiles)（Ed25519、Secp256k1、Secp256r1）であるかどうかを確認します。プリコンパイルはBPF
   VMを呼び出すことなく直接検証されます。
3. その他のすべてのプログラムについて、ランタイムは
   [`process_instruction`](https://github.com/anza-xyz/agave/blob/v3.1.8/program-runtime/src/invoke_context.rs#L477)を呼び出し、キャッシュからプログラムをロードしてBPF仮想マシン内で実行します。
4. instructionが完了すると、ランタイムは
   [検証](https://github.com/anza-xyz/agave/blob/v3.1.8/transaction-context/src/lib.rs#L366-L398)を行い、すべてのinstructionアカウント全体のlamport残高の合計が変更されていないことを確認します（*rs`UnbalancedInstruction`*チェック）。
5. いずれかのinstructionが失敗した場合、トランザクション全体がロールバックされます。中間状態の変更はコミットされません。

各instructionはinstructionトレースをインクリメントします。トレースには、トップレベルのinstructionsと、それらが呼び出すすべての[CPI](/docs/core/cpi)の両方が含まれます。トレースの合計長（トップレベルのinstructionsとすべてのネストされたCPI）は64を超えることはできません（_rs`MAX_INSTRUCTION_TRACE_LENGTH`_）。この制限を超えると
*rs`InstructionError::MaxInstructionTraceLengthExceeded`*が返されます。

実行後、ランタイムは以下を検証します：

- すべてのアカウントにわたるlamportの合計が変更されていないこと
- レント免除からレント支払いに移行したアカウントがないこと

### 8. コミットまたはロールバック

実行が成功した場合、`TransactionContext`から変更されたアカウントの状態は
*rs`AccountLoader`*のバッチローカルキャッシュに書き戻されます。実行が失敗した場合、_rs`RollbackAccounts`_（手数料が差し引かれ、nonceが進められた手数料支払者）のみが書き戻されます。手数料は引き続き徴収されますが、その他のすべてのアカウントの変更は破棄されます。

### パイプラインの概要

```
Receive packet (UDP/QUIC)
  --> Deserialize into VersionedTransaction
  --> Sigverify (parallel Ed25519 verification)
  --> Sanitize (structural validation, metadata extraction)
  --> Parse compute budget, calculate fees
  --> Check blockhash age (or verify nonce account)
  --> Check status cache (dedup)
  --> Validate nonce authority and advanceability (if nonce transaction)
  --> Validate fee payer (load, check balance, deduct fee)
  --> Load all accounts (with data size limits)
  --> Load programs (verify loaders)
  --> Execute instructions sequentially
  --> Verify post-conditions (lamport balance, rent state)
  --> Commit account changes (or rollback on failure)
```
