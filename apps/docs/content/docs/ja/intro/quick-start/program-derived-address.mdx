このセクションでは、基本的なCreate, Read, Update,
Delete（CRUD）プログラムの構築方法を学びます。

このガイドでは、ユーザーがメッセージを作成、更新、削除できるシンプルなプログラムを紹介します。各メッセージは、プログラム自体から派生した決定論的なアドレス（Program
Derived AddressまたはPDA）を持つアカウントに存在します。

このガイドでは、Anchorフレームワークを使用してSolanaプログラムを構築しテストする方法を説明しながら、Program
Derived
Addresses（PDAs）を実演します。詳細については、[Program Derived Addresses](/docs/core/pda)ページを参照してください。

参考として、PDAとCross-Program
Invocation（CPI）の両方のセクションを完了した後の[最終コード](https://beta.solpg.io/668304cfcffcf4b13384d20a)を確認できます。

### メッセージアカウントタイプの定義

まず、プログラムが作成するメッセージアカウントの構造を定義します。この構造は、プログラムによって作成されるアカウントに保存するデータを定義します。

<WithNotes>

`lib.rs` で、_rs`MessageAccount`_ 構造体を以下のように更新します:

```rs title="lib.rs"
// !tooltip[/account/] account
#[account]
pub struct MessageAccount {
    // !tooltip[/user/] user
    pub user: Pubkey,
    // !tooltip[/message/] message
    pub message: String,
    // !tooltip[/bump/] bump

}
```

### !account

Anchorプログラムにおける _rs`#[account]`_
属性は、アカウントデータ(アカウントのデータフィールドに保存するデータ型)を表す構造体に注釈を付けます。

### !user

_rs`user`_ フィールドには、メッセージアカウントを作成したユーザーを識別する
_rs`Pubkey`_ が含まれます。

### !message

_rs`message`_ フィールドには、ユーザーのメッセージを含む _rs`String`_
が保持されます。

### !bump

_rs`bump`_ フィールドには、Program Derived
Address(PDA)を導出するために使用される _rs`u8`_
["bump" seed](/docs/core/pda#canonical-bump)
が保存されます。この値を保存することで、後続のinstructionsで再計算する必要がなくなり、計算量を節約できます。

</WithNotes>

データ型にデシリアライズします。テストセクションでは、アカウントデータの作成と読み取りのプロセスを実演します。

### Create Instructionを追加する

次に、_rs`MessageAccount`_ を作成して初期化する _rs`create`_
instructionを追加します。

まず、_rs`Create`_
構造体を以下のように更新して、instructionに必要なアカウントを定義します:

<WithNotes>

```rs title="lib.rs"
#[derive(Accounts)]
// !tooltip[/instruction/] instruction
#[instruction(message: String)]
pub struct Create<'info> {
    // !tooltip[/mut/] mut
    #[account(mut)]
    // !tooltip[/Signer<'info>/] signer
    pub user: Signer<'info>,

    #[account(
        // !tooltip[/init/] init
        init,
        // !tooltip[/seeds/] seeds
        seeds = [b"message", user.key().as_ref()],
        // !tooltip[/bump/] bump
        bump,
        // !tooltip[/payer/] payer
        payer = user,
        // !tooltip[/space/] space
        space = 8 + 32 + 4 + message.len() + 1
    )]
    // !tooltip[/Account<'info, MessageAccount>/] account
    pub message_account: Account<'info, MessageAccount>,
    // !tooltip[/Program<'info, System>/] program
    pub system_program: Program<'info, System>,
}
```

### !mut

_rs`mut`_ 制約は、アカウントを可変として宣言します。

### !init

_rs`init`_ 制約は、新しいアカウントを作成します。

### !seeds

_rs`seeds`_ 制約は、PDAを導出するために使用されるオプションの入力を定義します。

### !bump

_rs`bump`_ 制約は、PDAのbump seedを宣言します。

値を指定しない場合、Anchorは自動的に計算します。

### !payer

_rs`payer`_ 制約は、新しいアカウント作成の費用を支払うアカウントを指定します。

### !space

_rs`space`_
制約は、新しいアカウントのデータフィールドに割り当てるバイト数を指定します。

### !signer

_rs`Signer<'info>`_
型は、アカウントがトランザクションに署名することを要求します。

### !account

_rs`Account<'info, T>`_
型は、アカウントが指定された型と一致することを要求します。

この場合、アカウントはカスタムの _rs`MessageAccount`_
型と一致する必要があります。

### !program

_rs`Program<'info, T>`_ 型は、アカウントがプログラムと一致することを要求します。

この場合、アカウントは _rs`System`_ 型と一致する必要があり、これはSystem
Programを指します。

### !instruction

_rs`#[instruction(message: String)]`_ アノテーションにより、_rs`Create`_
構造体が `create` instructionから _rs`message`_
パラメータにアクセスできるようになります。

</WithNotes>

<Accordion title="Explanation">

Anchorプログラムにおける _rs`#[derive(Accounts)]`_
属性は、instructionに必要なアカウントを定義する構造体に注釈を付けます。

構造体の各フィールドは、次の2つの方法で検証されるアカウントを表します：

1. プログラムが期待するアカウントの種類を指定するアカウントタイプ（_rs`Signer<'info>`_
   や _rs`Account<'info, T>`_ など）
2. 追加の要件を定義するオプションの制約（_rs`#[account(mut)]`_ や
   _rs`#[account(init)]`_ など）

これらを組み合わせることで、Anchorはinstructionに渡されたアカウントを自動的に検証し、プログラムを安全に保つことができます。

構造体のフィールド名はプログラムコード内でアカウントへのアクセスを提供しますが、検証には影響しません。明確さのために説明的な名前を使用すべきです。

この例では、_rs`Create`_ 構造体が _rs`create`_
instructionに必要なアカウントを定義しています。

1. _rs`user: Signer<'info>`_
   - メッセージアカウントを作成するユーザーを表します
   - 新しいアカウントの費用を支払うため、可変ステータス（_rs`#[account(mut)]`_）が必要です
   - このアカウントからのlamport控除を承認するため、トランザクションに署名する必要があります

2. _rs`message_account: Account<'info, MessageAccount>`_
   - ユーザーのメッセージを保存する新しいアカウント
   - `init` 制約はinstruction実行中にアカウントを作成します
   - `seeds` と `bump` 制約はアカウントアドレスをProgram Derived
     Address（PDA）として導出します
   - `payer = user` は新しいアカウントの作成費用を誰が支払うかを識別します
   - `space` はアカウントのデータフィールドに必要なバイト数を割り当てます

3. _rs`system_program: Program<'info, System>`_
   - アカウント作成に必要です
   - 内部的には、`init` 制約がSystem Programを呼び出して、指定された `space`
     で新しいアカウントを作成し、所有者を現在のプログラムに変更します。

---

_rs`#[instruction(message: String)]`_ アノテーションにより、_rs`Create`_
構造体は `create` instructionから _rs`message`_ パラメータにアクセスできます。

---

`seeds` と `bump` 制約を組み合わせることで、アカウントのアドレスをProgram
Derived Address（PDA）として定義します。

```rs title="lib.rs"
seeds = [b"message", user.key().as_ref()],
bump,
```

`seeds` 制約は、PDAを導出するために使用されるオプションの入力を定義します。

- _rs`b"message"`_ - 最初のseedとしての固定文字列。
- _rs`user.key().as_ref()`_ - 2番目のseedとしての _rs`user`_
  アカウントの公開鍵。

`bump` 制約は、Anchorに正しいbump
seedを自動的に見つけて使用するように指示します。Anchorは `seeds` と `bump`
を使用してPDAを導出します。

---

`space` の計算 _rs`(8 + 32 + 4 + message.len() + 1)`_ は、`MessageAccount`
データ型のためのスペースを割り当てます:

- Anchorアカウント識別子(discriminator): 8バイト
- ユーザーアドレス(_rs`Pubkey`_): 32バイト
- ユーザーメッセージ(_rs`String`_): 長さ用の4バイト + 可変メッセージ長
- PDA bump seed(_rs`u8`_): 1バイト

```rs title="lib.rs"
#[account]
pub struct MessageAccount {
    pub user: Pubkey,
    pub message: String,
    pub bump: u8,
}
```

Anchorプログラムを通じて作成されるすべてのアカウントには、アカウント識別子用に8バイトが必要です。これはアカウント作成時にAnchorが自動的に生成するアカウントタイプの識別子として機能します。

_rs`String`_
型は、文字列の長さを格納するために4バイトが必要で、残りの長さには実際のデータが含まれます。

</Accordion>
</Accordions>

次に、以下を使用して `create` 関数を更新し、_rs`create`_
instructionのビジネスロジックを追加します:

```rs title="lib.rs"
pub fn create(ctx: Context<Create>, message: String) -> Result<()> {
    msg!("Create Message: {}", message);
    let account_data = &mut ctx.accounts.message_account;
    account_data.user = ctx.accounts.user.key();
    account_data.message = message;
    account_data.bump = ctx.bumps.message_account;
    Ok(())
}
```

`create`
関数は、新しいメッセージアカウントのデータを初期化するためのロジックを実装します。この関数は2つのパラメータを取ります:

1. _rs`ctx: Context<Create>`_ - _rs`Create`_

関数の本体は以下のロジックを実行します：

1. _rs`msg!()`_ マクロを使用してプログラムログにメッセージを出力します。

   ```rs
   msg!("Create Message: {}", message);
   ```

2. アカウントデータの初期化:
   - コンテキストから `message_account` にアクセスします。

   ```rs
   let account_data = &mut ctx.accounts.message_account;
   ```

   - `user` フィールドを `user` アカウントの公開鍵に設定します。

   ```rs
   account_data.user = ctx.accounts.user.key();
   ```

   - `message` フィールドを関数引数の `message` に設定します。

   ```rs
   account_data.message = message;
   ```

   - PDAの導出に使用される `bump` 値を設定します。これは
     `ctx.bumps.message_account` から取得されます。

   ```rs

   ```

### 更新instructionの追加

次に、`update` instructionを追加して、`MessageAccount`
を新しいメッセージで変更します。

前のステップと同様に、まず `update` instructionに必要なアカウントを指定します。

`Update` 構造体を以下のように更新します:

<WithNotes>

```rs title="lib.rs"
#[derive(Accounts)]
#[instruction(message: String)]
pub struct Update<'info> {
    #[account(mut)]
    pub user: Signer<'info>,

    #[account(
        mut,
        seeds = [b"message", user.key().as_ref()],
        bump = message_account.bump,
        // !tooltip[/realloc/] realloc
        realloc = 8 + 32 + 4 + message.len() + 1,
        // !tooltip[/realloc::payer/] realloc::payer
        realloc::payer = user,
        // !tooltip[/realloc::zero/] realloc::zero
        realloc::zero = true,
    )]
    pub message_account: Account<'info, MessageAccount>,
    pub system_program: Program<'info, System>,
}
```

### !realloc

_rs`realloc`_ 制約はアカウントのデータを再割り当てします。

### !realloc::payer

_rs`realloc::payer`_ 制約は再割り当ての支払いを行うアカウントを指定します。

### !realloc::zero

_rs`realloc::zero`_ 制約はアカウントのデータをゼロで埋めます。

</WithNotes>

_rs`Update`_ 構造体は _rs`update`_ instructionに必要なアカウントを定義します。

1. _rs`user: Signer<'info>`_
   - メッセージアカウントを更新するユーザーを表します
   - 必要に応じて `message_account`
     のためにより多くのスペースの支払いを行う可能性があるため、可変（_rs`#[account(mut)]`_）としてマークされています

2. _rs`message_account: Account<'info, MessageAccount>`_

   - `realloc` 制約はアカウントのデータのサイズ変更を可能にします
   - `seeds` と `bump` 制約はアカウントが正しいPDAであることを検証します

   - アカウントスペースの潜在的な再割り当てに必要です

_rs`update`_ instructionから _rs`message`_

### 削除instructionを追加する

次に、_rs`MessageAccount`_ を閉じるための _rs`delete`_ instructionを追加します。

_rs`Delete`_ 構造体を以下のように更新します:

<WithNotes>

```rs title="lib.rs"
#[derive(Accounts)]
pub struct Delete<'info> {
    #[account(mut)]
    pub user: Signer<'info>,

    #[account(
        mut,
        // !tooltip[/seeds/] seeds
        seeds = [b"message", user.key().as_ref()],
        // !tooltip[/bump/] bump
        bump = message_account.bump,
        // !tooltip[/close/] close
        close = user,
    )]
    pub message_account: Account<'info, MessageAccount>,
}
```

### !seeds

_rs`seeds`_ 制約は、PDAを導出するために使用されるseedsを指定します。

### !bump

_rs`bump`_ 制約は、PDAのbump seedを指定します。

この場合、プログラムは _rs`message_account`_ に保存されている既存のbump
seedを使用します。

### !close

_rs`close`_ 制約は、アカウントを閉じます。

この場合、_rs`user`_ アカウントは、閉じられた _rs`message_account`_
からlamportsを受け取ります。

</WithNotes>

<Accordions>
<Accordion title="Diff">

```diff

- pub struct Delete {}

+ #[derive(Accounts)]
+ pub struct Delete<'info> {
+     #[account(mut)]
+     pub user: Signer<'info>,
+
+     #[account(
+         mut,
+         seeds = [b"message", user.key().as_ref()],
+         bump = message_account.bump,
+         close = user,
+     )]
+     pub message_account: Account<'info, MessageAccount>,
+ }
```

</Accordion>
<Accordion title="Explanation">

_rs`Delete`_ 構造体は _rs`delete`_ instructionに必要なアカウントを定義します:

1. _rs`user: Signer<'info>`_
   - メッセージアカウントを閉じるユーザーを表します
   - 閉じられたアカウントからlamportsを受け取るため、可変(_rs`#[account(mut)]`_)としてマークされています
   - 正しいユーザーのみが自分のメッセージアカウントを閉じられるようにするため、署名が必要です

2. _rs`message_account: Account<'info, MessageAccount>`_
   - 閉じるためのアカウント
   - `mut` 制約はデータ変更を示します
   - `seeds` と `bump` 制約はアカウントが正しいPDAであることを確認します
   - `close = user` 制約はこのアカウントを閉じるようマークし、そのlamportsを
     `user` アカウントに転送します

</Accordion>
</Accordions>

    msg!("Delete Message");
    Ok(())
}
