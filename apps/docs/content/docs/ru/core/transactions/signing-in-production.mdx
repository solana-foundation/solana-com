Транзакция выполняется только при наличии действительной
[подписи](/docs/core/transactions/transaction-structure#signatures) от каждого
обязательного подписанта. В процессе локальной разработки вы обычно храните ключ
в виде [`Keypair`](/developers/cookbook/wallets/create-keypair) и подписываете
напрямую. В продакшене ключевой вопрос звучит иначе: **где хранится приватный
ключ и кому разрешено авторизовать подпись с его помощью?**

<Callout type="warn">
  Никогда не встраивайте приватные ключи в клиентский код, не включайте их во
  фронтенд-сборку и не добавляйте в систему контроля версий. Любой, кто получит
  ключ, получит полный контроль над средствами этого аккаунта.
</Callout>

## Подходы к подписанию

| Подход                         | Где хранится ключ                                              | Кто подписывает                                         | Лучше всего подходит для                                                       |
| ------------------------------ | -------------------------------------------------------------- | ------------------------------------------------------- | ------------------------------------------------------------------------------ |
| **Локальный keypair**          | Файл или переменная окружения на машине                        | Ваш код, напрямую                                       | Локальная разработка, тесты, CI                                                |
| **Браузерный кошелёк**         | Кошелёк конечного пользователя (например, Phantom, Solflare)   | Конечный пользователь, для каждой транзакции            | Фронтенд-dApp, где пользователи сами подписывают транзакции                    |
| **Облачный KMS / HSM**         | Аппаратный ключевой сервис (AWS KMS, GCP KMS, Vault)           | Ваш бэкенд запрашивает подпись у сервиса                | Бэкенд-сервисы, операции с казначейством, регулируемые среды                   |
| **Управляемые / MPC-кошельки** | Разделены или переданы на хранение в инфраструктуру провайдера | Провайдер совместно подписывает согласно вашей политике | Встроенные кошельки, рабочие процессы согласования, институциональное хранение |

Локальный keypair — единственный подход, при котором необработанный ключевой
материал хранится непосредственно в вашем приложении. Все производственные
подходы хранят ключ в специализированной инфраструктуре и делегируют подписание
этой инфраструктуре.

## Подписание на стороне сервера

Когда серверу необходимо выполнять подписание — оплачивать комиссии, отправлять
транзакции программ или управлять казначейством — используйте специализированный
бэкенд для управления ключами, а не обычный keypair. Solana Foundation
поддерживает [Keychain](/docs/tools/keychain) — унифицированную библиотеку
подписания (Rust и TypeScript), которая предоставляет единый интерфейс
`SolanaSigner` для всех бэкендов из таблицы выше: Memory, HashiCorp Vault, AWS
KMS, GCP KMS, Fireblocks, Privy, Turnkey, CDP, Crossmint, Dfns, Para, Openfort и
Utila.

Поскольку интерфейс идентичен для всех бэкендов, вы можете вести локальную
разработку с ключом в памяти, а затем переключиться на производственный бэкенд
через конфигурацию — без переписывания кода приложения. Keychain совместим с
[`@solana/kit`](/docs/clients/official/javascript) и
[Rust SDK](/docs/clients/official/rust).
