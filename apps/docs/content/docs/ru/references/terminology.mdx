Следующие термины используются в документации и экосистеме разработки Solana.

## аккаунт

Запись в реестре Solana, которая либо содержит данные, либо является исполняемой
программой.

Как и аккаунт в традиционном банке, аккаунт Solana может хранить средства,
называемые [lamport](#lamport). Каждый аккаунт в [кластере](#cluster) имеет
уникальный [адрес](#address).

## приложение

Фронтенд-приложение, которое взаимодействует с кластером Solana.

## адрес

Местоположение в Solana, которое можно использовать для поиска аккаунта. Каждый
адрес уникален в [кластере](#cluster) Solana.

Адреса в Solana могут быть либо [публичным ключом](#public-key-pubkey) Ed25519,
либо [адресом, производным от программы](#program-derived-address-pda).

## таблица поиска адресов (ALT)

Структура данных, которая хранит список адресов аккаунтов, позволяя
[версионным транзакциям](#versioned-transaction) ссылаться на аккаунты с
использованием компактных 1-байтовых индексов вместо полных 32-байтовых
публичных ключей. Таблицы поиска адресов позволяют одной транзакции ссылаться на
большее количество аккаунтов, чем допускает исходный формат транзакции.

## Anchor

Фреймворк для создания [ончейн-программ](#onchain-program) Solana на Rust.
Anchor предоставляет макросы, которые сокращают шаблонный код,
стандартизированную систему валидации аккаунтов и генерирует [IDL](#idl),
описывающий интерфейс программы. Это наиболее широко используемый фреймворк для
разработки программ на Solana.

## associated token account (ATA)

[Token account](#token-account), адрес которого детерминированно выводится из
адреса кошелька и [минта токена](#token-mint). Программа associated token
account гарантирует, что у каждого кошелька есть один канонический token account
для каждого типа токена.

## полномочие

Адрес пользователя, имеющего определенные разрешения на аккаунт.

Например:

- Возможность создания новых токенов предоставляется аккаунту, который является
  «полномочием на минтинг» для минта токена.
- Возможность обновления программы предоставляется аккаунту, который является
  «полномочием на обновление» программы.

## состояние банка

Результат интерпретации всех программ в реестре на заданной
[высоте тика](#tick-height). Включает как минимум набор всех
[аккаунтов](#account), содержащих ненулевое количество
[нативных токенов](#native-token).

## блок

Непрерывный набор [записей](#entry) в реестре, охватываемый
[голосованием](#ledger-vote). [Лидер](#leader) производит не более одного блока
за [слот](#slot).

## blockhash

Уникальное значение ([хеш](#hash)), которое идентифицирует запись (блок). Solana
вычисляет blockhash из последнего [entry id](#entry-id) блока.

## высота блока

Количество [блоков](#block) под текущим блоком. Первый блок после
[genesis block](#genesis-block) имеет высоту один.

## bootstrap validator

[Validator](#validator), который создаёт genesis (первый) [блок](#block)
блокчейна.

## BPF loader

Программа Solana, которая владеет и загружает [BPF](/docs/core/programs)
[onchain программы](#onchain-program), позволяя программе взаимодействовать с
runtime.

## bump

Один байт (0–255), добавляемый к [seeds](#seed) при деривации
[Program Derived Address](#program-derived-address-pda), чтобы гарантировать,
что полученный адрес не находится на кривой ed25519 и, следовательно, является
валидным PDA. Канонический bump — это наибольшее значение, которое создаёт
валидный PDA, и обычно сохраняется в данных аккаунта, чтобы избежать повторного
вычисления.

## клиент

Компьютерная программа, которая обращается к сетевому [кластеру](#cluster)
серверов Solana.

## commitment

Мера сетевого подтверждения для [блока](#block).

## формат компактного массива

Компактный массив — это массив, сериализованный в следующем формате:

- Длина массива (закодированная как
  [compact-u16](https://github.com/anza-xyz/agave/blob/v2.1.13/short-vec/src/lib.rs))
- Элементы массива, перечисленные один за другим

![Формат компактного массива](/assets/docs/core/transactions/compact_array_format.png)

Формат компактного массива используется для кодирования как массива
[адресов аккаунтов](/docs/core/transactions/transaction-structure#account-addresses),
так и массива
[инструкций](/docs/core/transactions/transaction-structure#instructions) в
сообщениях транзакций.

## кластер

Набор [validators](#validator), поддерживающих единый [ledger](#ledger).

## compute budget

Максимальное количество [compute units](#compute-units), потребляемых на одну
транзакцию.

## compute units

Наименьшая единица измерения потребления вычислительных ресурсов блокчейна.
Compute units используются для измерения вычислительных ресурсов, используемых
для обработки транзакции.

## время подтверждения

Реальная продолжительность между созданием [лидером](#leader)
[записи тика](#tick) и созданием [подтверждённого блока](#confirmed-block).

## подтверждённый блок

[Блок](#block), получивший [квалифицированное большинство](#supermajority)
[голосов реестра](#ledger-vote).

## плоскость управления

Сеть gossip, соединяющая все [узлы](#node) [кластера](#cluster).

## период охлаждения

Определённое количество [эпох](#epoch) после деактивации [стейка](#stake), в
течение которых он постепенно становится доступным для вывода. В этот период
стейк считается «деактивирующимся». Подробнее:
[разогрев и охлаждение](https://docs.anza.xyz/consensus/stake-delegation-and-rewards#stake-warmup-cooldown-withdrawal)

## единицы стоимости

Также называемые `transaction cost`, это комплексная оценка всех ресурсов,
необходимых для обработки транзакции, измеряемая в
[вычислительных единицах](#compute-units). Подробнее
[здесь](https://www.anza.xyz/blog/why-solana-transaction-costs-and-compute-units-matter-for-developers).

## кредит

См. [кредит голоса](#vote-credit).

## Cross Program Invocation (CPI)

Вызов одной [ончейн-программы](#onchain-program) из другой. Подробнее см. в
разделе [вызовы между программами](/docs/core/cpi).

## плоскость данных

Многоадресная сеть, используемая для эффективной проверки [записей](#entry) и
достижения консенсуса.

## дрон

Оффчейн-сервис, выступающий в роли хранителя закрытого ключа пользователя.
Обычно используется для проверки и подписи транзакций.

## запись

Запись в [реестре](#ledger), представляющая собой либо [тик](#tick), либо
[запись транзакции](#transactions-entry).

## идентификатор записи

Устойчивый к предобразу [хеш](#hash) финального содержимого записи, выступающий
в качестве глобально уникального идентификатора [записи](#entry). Хеш служит
доказательством:

- Создания записи по истечении определённого времени
- Включения в запись указанных [транзакций](#transaction)
- Позиции записи относительно других записей в [реестре](#ledger)

См. [доказательство истории](#proof-of-history-poh).

## epoch

Время, т.е. количество [слотов](#slot), в течение которого действует
[расписание лидера](#leader-schedule).

## аккаунт комиссии

Аккаунт комиссии в транзакции — это аккаунт, который оплачивает стоимость
включения транзакции в леджер. Это первый аккаунт в транзакции. Данный аккаунт
должен быть объявлен как Read-Write (доступный для записи) в транзакции,
поскольку оплата транзакции уменьшает баланс аккаунта.

## финальность

Когда узлы, представляющие 2/3 [стейка](#stake), имеют общий [корень](#root).

## форк

[Леджер](#ledger), происходящий от общих записей, но затем разошедшийся.

## генезис-блок

Первый [блок](#block) в цепи.

## конфигурация генезиса

Конфигурационный файл, который подготавливает [леджер](#ledger) для
[генезис-блока](#genesis-block).

## хеш

Цифровой отпечаток последовательности байтов.

## инфляция

Увеличение предложения токенов с течением времени, используемое для
финансирования вознаграждений за валидацию и для финансирования дальнейшей
разработки Solana.

## внутренняя инструкция

См. [Cross Program Invocation](#cross-program-invocation-cpi).

## инструкция

Вызов для активации определённого [обработчика инструкций](#instruction-handler)
в [программе](#program). Инструкция также указывает, какие аккаунты она хочет
читать или изменять, а также дополнительные данные, которые служат
вспомогательным входом для [обработчика инструкций](#instruction-handler).
[Клиент](#client) должен включить хотя бы одну инструкцию в
[транзакцию](#transaction), и все инструкции должны быть успешно выполнены,
чтобы транзакция считалась успешной.

## обработчик инструкций

Обработчики инструкций — это функции [программы](#program), которые обрабатывают
[инструкции](#instruction) из [транзакций](#transaction). Обработчик инструкций
может содержать один или несколько
[межпрограммных вызовов](#cross-program-invocation-cpi).

## IDL

Interface Definition Language. JSON-файл, создаваемый [Anchor](#anchor), который
описывает [обработчики инструкций](#instruction-handler), аккаунты и типы
программы. IDL позволяет клиентским библиотекам автоматически генерировать
типизированные интерфейсы для взаимодействия с
[ончейн-программой](#onchain-program).

## keypair

Пара из [публичного ключа](#public-key-pubkey) и соответствующего
[приватного ключа](#private-key) для доступа к аккаунту.

## lamport

Дробная единица [нативного токена](#native-token) со значением 0,000000001
[sol](#sol).

> В рамках вычислительного бюджета величина
> _[микро-lamport](https://github.com/solana-labs/solana/blob/ced8f6a512c61e0dd5308095ae8457add4a39e94/program-runtime/src/prioritization_fee.rs#L1-L2)_
> используется при расчёте [комиссии за приоритизацию](#prioritization-fee).

## leader

Роль [validator](#validator), когда он добавляет [записи](#entry) в
[леджер](#ledger).

## leader schedule

Последовательность [публичных ключей](#public-key-pubkey)
[validator](#validator), сопоставленных со [слотами](#slot). Кластер использует
расписание лидеров для определения того, какой validator является
[лидером](#leader) в любой момент времени.

## ledger

Список [записей](#entry), содержащих [транзакции](#transaction), подписанные
[клиентами](#client). Концептуально это можно проследить до
[блока генезиса](#genesis-block), но фактический леджер [validator](#validator)
может содержать только более новые [блоки](#block) для сокращения хранилища,
поскольку более старые не необходимы для валидации будущих блоков по
архитектуре.

## ledger vote

[Хеш](#hash) [состояния validator](#bank-state) на заданной
[высоте тика](#tick-height). Он представляет собой подтверждение
[validator](#validator) того, что полученный им [блок](#block) был
верифицирован, а также обязательство не голосовать за конфликтующий
[блок](#block) \(т.е. [форк](#fork)\) в течение определённого времени — периода
[блокировки](#lockout).

## light client

Тип [клиента](#client), который может верифицировать, что он подключён к
валидному [кластеру](#cluster). Он выполняет больше проверок леджера, чем
[тонкий клиент](#thin-client), и меньше, чем [validator](#validator).

## loader

[Программа](#program), способная интерпретировать бинарное кодирование других
[ончейн-программ](#onchain-program).

## lockout

Длительность времени, в течение которого [validator](#validator) не может
[голосовать](#ledger-vote) за другой [форк](#fork).

## message

Структурированное содержимое [транзакции](#transaction). Обычно содержит
заголовок, массив адресов аккаунтов, последний [blockhash](#blockhash) и массив
[инструкций](#instruction).

Узнайте больше о
[форматировании сообщений внутри транзакций](/docs/core/transactions/transaction-structure#header)
здесь.

## микро-лампорт

1 000 000 микро-лампортов = 1 [lamport](#lamport)

## Коэффициент Накамото

Мера децентрализации, коэффициент Накамото — это наименьшее количество
независимых субъектов, которые могут коллективно остановить работу блокчейна.
Термин был введён Баладжи С. Шринивасаном и Лиландом Ли в статье
[Quantifying Decentralization](https://news.earn.com/quantifying-decentralization-e39db233c28e).

## нативный токен

[Токен](#token), используемый для отслеживания работы, выполненной
[нодами](#node) в кластере.

## нода

Компьютер, участвующий в [кластере](#cluster).

## количество нод

Количество [валидаторов](#validator), участвующих в [кластере](#cluster).

## ончейн

Описывает данные или программы, которые существуют или выполняются в блокчейне.
Предпочтительное написание — «ончейн» (слитно).

## ончейн-программа

Исполняемый код в блокчейне Solana, который интерпретирует
[инструкции](#instruction), отправленные внутри каждой
[транзакции](#transaction), для чтения и изменения аккаунтов, которыми он
управляет. Эти программы часто называют
«[_смарт-контрактами_](/docs/core/programs)» в других блокчейнах.

## владелец

Может относиться либо к [программе-владельцу](#owning-program), либо к
[кошельку-владельцу](#owning-wallet). Например,
[getTokenAccountsByOwner](https://solana.com/docs/rpc/http/gettokenaccountsbyowner)
возвращает токен-аккаунты, принадлежащие определённому кошельку, однако
программой-владельцем этих токен-аккаунтов является
[Token Program](#token-program).

Чтобы избежать путаницы, предпочтительнее использовать термины
[программа-владелец](#owning-program) или [кошелёк-владелец](#owning-wallet), а
не общий термин «владелец».

## программа-владелец

[Адрес](#address) [программы](#onchain-program), которая владеет аккаунтом.
Только программа-владелец способна изменять аккаунт.

См. также [полномочие](#authority).

## владеющий кошелёк

Кошелёк, который связан с аккаунтом. Например, Token Program используется для
хранения token account, связанных с определённым кошельком и минтом, используя
адрес, созданный из этого [кошелька](#wallet) и [минта](#token-mint).

## PoH

См. [Proof of History](#proof-of-history-poh).

## очко

Взвешенный [кредит](#credit) в системе вознаграждений. В
[системе вознаграждений](https://docs.anza.xyz/consensus/stake-delegation-and-rewards)
[validator](#validator) количество очков, причитающихся [стейку](#stake) при
погашении, представляет собой произведение заработанных
[кредитов голосования](#vote-credit) и количества застейканных
[lamport](#lamport).

## приватный ключ

Приватный ключ [keypair](#keypair).

## программа

См. [программа в сети](#onchain-program).

## Program Derived Address (PDA)

Аккаунт, адрес которого создаётся на основе [программы в сети](#onchain-program)
и одного или нескольких [сидов](#seed), созданных автором программы. Адреса PDA
не являются публичными ключами, и PDA не имеют приватных ключей.

Право подписи для PDA принадлежит [программе в сети](#onchain-program), которая
является их [владеющей программой](#owning-program).

## идентификатор программы

Публичный ключ [аккаунта](#account), содержащего [программу](#onchain-program).

## Proof of History (PoH)

Стек доказательств, каждое из которых подтверждает, что определённые данные
существовали до создания доказательства и что прошёл точный промежуток времени с
момента предыдущего доказательства. Как и [VDF](#verifiable-delay-function-vdf),
Proof of History может быть проверено за меньшее время, чем потребовалось для
его создания.

## комиссия за приоритизацию

Дополнительная комиссия, которую пользователь может указать в
[инструкции](#instruction) вычислительного бюджета для приоритизации своих
[транзакций](#transaction).

Комиссия за приоритизацию рассчитывается путём умножения запрошенных
максимальных вычислительных единиц на цену вычислительной единицы (указанную с
шагом 0,000001 [lamport](#lamport) за вычислительную единицу) с округлением до
ближайшего [lamport](#lamport).

Транзакции должны запрашивать минимальное количество вычислительных единиц,
необходимых для выполнения, чтобы минимизировать комиссии.

## публичный ключ (pubkey)

Публичный ключ [keypair](#keypair). Публичный ключ — это один из различных типов
[адресов](#address), которые аккаунт может использовать в Solana.

## rent

Комиссия, уплачиваемая [аккаунтами](#account) и [программами](#program) за
хранение данных в блокчейне. Когда аккаунты не имеют достаточного баланса для
оплаты rent, они могут быть удалены сборщиком мусора.

Смотрите также [rent exempt](#rent-exempt) ниже. Подробнее о rent читайте здесь:
[Что такое rent?](/docs/core/accounts/account-structure).

## rent exempt

Аккаунты, которые поддерживают минимальный баланс в lamport, пропорциональный
объему данных, хранящихся в аккаунте. Все вновь созданные аккаунты хранятся в
блокчейне постоянно до тех пор, пока аккаунт не будет закрыт. Невозможно создать
аккаунт, который опустится ниже порога освобождения от rent.

## корень

[Блок](#block) или [slot](#slot), который достиг максимальной
[блокировки](#lockout) на [validator](#validator). Корень — это самый высокий
блок, являющийся предком всех активных форков на validator. Все блоки-предки
корня также транзитивно являются корнем. Блоки, которые не являются предком и не
являются потомком корня, исключаются из рассмотрения для консенсуса и могут быть
отброшены.

## среда выполнения

Компонент [validator](#validator), отвечающий за выполнение
[программы](#program).

## Sealevel

Параллельная среда выполнения Solana для
[программ в блокчейне](#onchain-program).

## seed

Байтовое значение или массив байтовых значений, используемых в качестве входных
данных при формировании [Program Derived Address](#program-derived-address-pda).
Программа может использовать любую комбинацию статических строк, публичных
ключей или других данных в качестве seed. Seed комбинируются с идентификатором
программы и [bump](#bump) для создания уникального адреса PDA.

## shred

Часть [блока](#block); наименьшая единица данных, передаваемая между
[валидаторами](#validator).

## подпись

64-байтовая ed25519 подпись из R (32 байта) и S (32 байта). С требованием, чтобы
R была упакованной точкой Эдвардса не малого порядка, а S — скаляром в диапазоне
`0 <= S < L`. Это требование исключает возможность подделки подписи. Каждая
транзакция должна иметь как минимум одну подпись для
[аккаунта комиссии](#fee-account). Таким образом, первая подпись в транзакции
может рассматриваться как [идентификатор транзакции](#transaction-id)

## процент пропусков

Процент [пропущенных слотов](#skipped-slot) от общего количества лидерских
слотов в текущей epoch. Эта метрика может вводить в заблуждение, поскольку имеет
высокую дисперсию после границы epoch, когда размер выборки мал, а также для
валидаторов с небольшим количеством лидерских слотов, однако может быть полезна
для выявления неправильных конфигураций узла.

## пропущенный slot

Прошедший [slot](#slot), который не произвел [блок](#block), потому что лидер
был офлайн или [форк](#fork), содержащий этот slot, был отброшен в пользу лучшей
альтернативы по консенсусу кластера. Пропущенный slot не будет отображаться в
качестве предка для блоков в последующих слотах, не увеличит
[высоту блока](#block-height) и не истечет самый старый `recent_blockhash`.

Определить, был ли slot пропущен, можно только когда он становится старше
последнего [корневого](#root) (то есть не пропущенного) слота.

## slot

Период времени, в течение которого каждый [лидер](#leader) принимает транзакции
и создает [блок](#block).

Вместе слоты создают логические часы. Слоты упорядочены последовательно и не
перекрываются, составляя примерно равные промежутки реального времени согласно
[PoH](#proof-of-history-poh).

## System Program

Нативная программа Solana (ID программы `11111111111111111111111111111111`),
ответственная за создание новых [аккаунтов](#account), выделение данных
аккаунта, назначение аккаунтов программам и перевод [SOL](#sol) между
аккаунтами. Все новые аккаунты в Solana принадлежат System Program до
переназначения.

## sysvar

Системный [аккаунт](#account). [Sysvars](https://docs.anza.xyz/runtime/sysvars)
предоставляют информацию о состоянии кластера, такую как текущая высота тика,
значения [поинтов](#point) вознаграждений и т.д. Программы могут получать доступ
к Sysvars через аккаунт Sysvar (pubkey) или запрашивая через syscall.

## тонкий клиент

Тип [клиента](#client), который доверяет, что взаимодействует с действительным
[кластером](#cluster).

## тик

[Запись](#entry) в леджере, оценивающая продолжительность реального времени.

## высота тика

N-ный [тик](#tick) в [леджере](#ledger).

## токен

Актив, который можно передавать в цифровом виде.

## token account

Аккаунт, который хранит баланс определённого [токена](#token). Токен-аккаунты
принадлежат [Token Program](#token-program) или
[Token Extensions Program](#token-extensions-program) и хранят баланс токенов,
связанный [токен минт](#token-mint) и адрес кошелька владельца.

## Token Extensions Program

[Token Extensions Program](https://www.solana-program.com/docs/token-2022) имеет
ID программы `TokenzQdBNbLqP5VEhdkAS6EPFLC1PHnBqCXEpPxuEb` и включает все те же
функции, что и [Token Program](#token-program), но также предоставляет
расширения, такие как конфиденциальные переводы, пользовательская логика
переводов, расширенные метаданные и многое другое.

## токен минт

Аккаунт, который может создавать (или «минтить») токены. Различные токены
отличаются своими уникальными адресами токен минта.

## Token Program

[Token Program](https://www.solana-program.com/docs/token) имеет ID программы
`TokenkegQfeZyiNwAJbNbGKPFXCWuBvf9Ss623VQ5DA` и предоставляет базовые
возможности для переводов, заморозки и создания токенов.

## транзакция

Одна или несколько [инструкций](#instruction), подписанных [клиентом](#client) с

атомарно с только двумя возможными результатами: успех или неудача.

## ID транзакции

Первая [подпись](#signature) в [транзакции](#transaction), которая может

[реестра](#ledger).

## подтверждения транзакции

Количество [подтверждённых блоков](#confirmed-block) с момента принятия
транзакции в [реестр](#ledger). Транзакция считается завершённой, когда её блок
становится [корневым](#root).

## validator

Полноценный участник [кластера](#cluster) сети Solana, который производит новые

## VDF

См. [проверяемая функция задержки](#verifiable-delay-function-vdf).

## функция верифицируемой задержки (VDF)

Функция, которая выполняется за фиксированное время и создаёт доказательство
своего выполнения, которое затем может быть проверено быстрее, чем потребовалось
на его создание.

## версионная транзакция

Формат транзакции, который поддерживает дополнительные функции помимо исходного
формата [транзакции](#transaction). Версия 0 (`v0`) — это первый версионный
формат, который добавляет поддержку
[таблиц поиска адресов](#address-lookup-table-alt), позволяя транзакциям
ссылаться на больше аккаунтов, чем допускает исходный формат.

Счётчик вознаграждений для [validator](#validator). Кредит за голосование
присваивается validator в его голосовом аккаунте, когда validator достигает

## период разогрева

"активирующимся". Подробнее:
[разогрев и охлаждение](https://docs.anza.xyz/consensus/stake-delegation-and-rewards#stake-warmup-cooldown-withdrawal)
