Следующие термины используются в документации и экосистеме разработки Solana.

## аккаунт

Запись в реестре Solana, которая либо содержит данные, либо является исполняемой
программой.

Как и аккаунт в традиционном банке, аккаунт Solana может хранить средства,
называемые [lamport](#lamport). Каждый аккаунт в [кластере](#cluster) имеет
уникальный [адрес](#address).

## приложение

Фронтенд-приложение, которое взаимодействует с кластером Solana.

## адрес

Местоположение в Solana, которое можно использовать для поиска аккаунта. Каждый
адрес уникален в [кластере](#cluster) Solana.

Адреса в Solana могут быть либо [публичным ключом](#public-key-pubkey) Ed25519,
либо [адресом, производным от программы](#program-derived-address-pda).

## таблица поиска адресов (ALT)

Структура данных, которая хранит список адресов аккаунтов, позволяя
[версионным транзакциям](#versioned-transaction) ссылаться на аккаунты с
использованием компактных 1-байтовых индексов вместо полных 32-байтовых
публичных ключей. Таблицы поиска адресов позволяют одной транзакции ссылаться на
большее количество аккаунтов, чем допускает исходный формат транзакции.

## Anchor

Фреймворк для создания [ончейн-программ](#onchain-program) Solana на Rust.
Anchor предоставляет макросы, которые сокращают шаблонный код,
стандартизированную систему валидации аккаунтов и генерирует [IDL](#idl),
описывающий интерфейс программы. Это наиболее широко используемый фреймворк для
разработки программ на Solana.

## associated token account (ATA)

[Token account](#token-account), адрес которого детерминированно выводится из
адреса кошелька и [минта токена](#token-mint). Программа associated token
account гарантирует, что у каждого кошелька есть один канонический token account
для каждого типа токена.

## полномочие

Адрес пользователя, имеющего определенные разрешения на аккаунт.

Например:

- Возможность создания новых токенов предоставляется аккаунту, который является
  «полномочием на минтинг» для минта токена.
- Возможность обновления программы предоставляется аккаунту, который является
  «полномочием на обновление» программы.

## автоматический маркет-мейкер (AMM)

Модель ончейн-обмена, при которой пользователи торгуют против пула резервов
токенов, управляемого формулой ценообразования, а не против размещенного ордера
контрагента. Цены изменяются в зависимости от резервов, поэтому сделки
сопровождаются [проскальзыванием](#slippage), пропорциональным их размеру.

## состояние банка

Результат интерпретации всех программ в леджере на заданной
[высоте тика](#tick-height). Включает как минимум набор всех
[аккаунтов](#account), содержащих ненулевое количество
[нативных токенов](#native-token).

## блок

Непрерывный набор [записей](#entry) в леджере, покрытых
[голосованием](#ledger-vote). [Лидер](#leader) создает максимум один блок за
[слот](#slot).

## хеш блока

Уникальное значение ([хеш](#hash)), идентифицирующее запись (блок). Solana
вычисляет хеш блока из последнего [идентификатора записи](#entry-id) блока.

## высота блока

Количество [блоков](#block) под текущим блоком. Первый блок после
[генезис-блока](#genesis-block) имеет высоту один.

## начальный валидатор

[Валидатор](#validator), который создает генезис (первый) [блок](#block)
блокчейна.

## загрузчик BPF

Программа Solana, которая владеет и загружает [BPF](/docs/core/programs)
[ончейн-программы](#onchain-program), позволяя программе взаимодействовать с
runtime.

## бамп

Один байт (0–255), добавляемый к [сидам](#seed) при выведении
[адреса, производного от программы](#program-derived-address-pda), чтобы
гарантировать, что результирующий адрес не лежит на кривой ed25519 и,
следовательно, является валидным PDA. Канонический бамп — это наибольшее
значение, которое создает валидный PDA, и обычно сохраняется в данных аккаунта,
чтобы избежать его повторного вычисления.

## центральная книга лимитных ордеров (CLOB)

Модель обмена, которая поддерживает отсортированный список размещенных заявок на
покупку и продажу; сделка происходит, когда входящий ордер пересекает ордер на
противоположной стороне. Противопоставляется
[автоматическому маркет-мейкеру](#automated-market-maker-amm).

## клиент

Компьютерная программа, которая обращается к сети серверов Solana
[кластер](#cluster).

## подтверждение

Мера сетевого подтверждения для [блока](#block).

## формат компактного массива

Компактный массив — это массив, сериализованный в следующем формате:

- Длина массива (закодированная как
  [compact-u16](https://github.com/anza-xyz/agave/blob/v2.1.13/short-vec/src/lib.rs))
- Элементы массива, перечисленные один за другим

![Формат компактного массива](/assets/docs/core/transactions/compact_array_format.png)

Формат компактного массива используется для кодирования как массива
[адресов аккаунтов](/docs/core/transactions/transaction-structure#account-addresses),
так и массива
[инструкций](/docs/core/transactions/transaction-structure#instructions) в
сообщениях транзакций.

## кластер

Набор [валидаторов](#validator), поддерживающих единый [реестр](#ledger).

## вычислительный бюджет

Максимальное количество [вычислительных единиц](#compute-units), потребляемых на
одну транзакцию.

## вычислительные единицы

Наименьшая единица измерения потребления вычислительных ресурсов блокчейна.
Вычислительные единицы используются для измерения вычислительных ресурсов,
используемых для обработки транзакции.

## время подтверждения

Время по настенным часам между созданием [лидером](#leader)
[тиковой записи](#tick) и созданием [подтверждённого блока](#confirmed-block).

## подтверждённый блок

[Блок](#block), получивший [суперболь­шинство](#supermajority)
[голосов в реестре](#ledger-vote).

## управляющая плоскость

Сеть сплетен (gossip), соединяющая все [узлы](#node) [кластера](#cluster).

## период охлаждения

Определённое количество [эпох](#epoch) после деактивации [стейка](#stake), в
течение которых он постепенно становится доступным для вывода. В течение этого
периода стейк считается "деактивирующимся". Больше информации о:
[разогреве и охлаждении](https://docs.anza.xyz/consensus/stake-delegation-and-rewards#stake-warmup-cooldown-withdrawal)

## единицы стоимости

Также называемые `transaction cost`, представляют собой всеобъемлющую оценку
всех ресурсов, необходимых для обработки транзакции, измеряемую в
[вычислительных единицах](#compute-units). Подробнее
[здесь](https://www.anza.xyz/blog/why-solana-transaction-costs-and-compute-units-matter-for-developers).

## кредит

См. [кредит за голосование](#vote-credit).

## Cross Program Invocation (CPI)

Вызов одной [программы в сети](#onchain-program) из другой. Для получения
дополнительной информации см. [вызовы между программами](/docs/core/cpi).

## плоскость данных

Multicast-сеть, используемая для эффективной проверки [записей](#entry) и
достижения консенсуса.

## дрон

Офчейн-сервис, который выступает в качестве хранителя приватного ключа
пользователя. Обычно используется для проверки и подписания транзакций.

## запись

Запись в [реестре](#ledger) — либо [тик](#tick), либо
[запись транзакции](#transactions-entry).

## идентификатор записи

Устойчивый к прообразам [хеш](#hash) финального содержимого записи, который
служит глобально уникальным идентификатором [записи](#entry). Хеш служит
доказательством того, что:

- Запись была создана по истечении определённого времени
- Указанные [транзакции](#transaction) включены в запись
- Позиция записи относительно других записей в [реестре](#ledger)

См. [доказательство истории](#proof-of-history-poh).

## epoch

Время, т.е. количество [слотов](#slot), в течение которого действует
[расписание лидеров](#leader-schedule).

## эскроу

Паттерн, при котором [программа в сети](#onchain-program) хранит токены в
[хранилище](#vault) от имени двух или более сторон до тех пор, пока не будут
выполнены условия, определённые программой, после чего освобождает их.
Используется для устранения необходимости прямого доверия между сторонами.

## аккаунт комиссии

Аккаунт комиссии в транзакции — это аккаунт, который оплачивает стоимость
включения транзакции в реестр. Это первый аккаунт в транзакции. Данный аккаунт
должен быть объявлен как Read-Write (доступный для записи) в транзакции,
поскольку оплата транзакции уменьшает баланс аккаунта.

## финальность

Когда узлы, представляющие 2/3 [стейка](#stake), имеют общий [корень](#root).

## форк

[Леджер](#ledger), произошедший от общих записей, но затем разошедшийся.

## genesis-блок

Первый [блок](#block) в цепи.

## конфигурация genesis

Конфигурационный файл, который подготавливает [леджер](#ledger) для
[genesis-блока](#genesis-block).

## хеш

Цифровой отпечаток последовательности байтов.

## инфляция

Увеличение предложения токенов с течением времени, используемое для
финансирования вознаграждений за валидацию и для финансирования дальнейшей
разработки Solana.

## внутренняя инструкция

См. [Cross Program Invocation](#cross-program-invocation-cpi).

## инструкция

Вызов для активации определенного [обработчика инструкций](#instruction-handler)
в [программе](#program). Инструкция также указывает, какие аккаунты она хочет
читать или изменять, и дополнительные данные, которые служат вспомогательным
входом для [обработчика инструкций](#instruction-handler). [Клиент](#client)
должен включить хотя бы одну инструкцию в [транзакцию](#transaction), и все
инструкции должны успешно завершиться, чтобы транзакция считалась успешной.

## обработчик инструкций

Обработчики инструкций — это функции [программы](#program), которые обрабатывают
[инструкции](#instruction) из [транзакций](#transaction). Обработчик инструкций
может содержать один или несколько
[Cross Program Invocation](#cross-program-invocation-cpi).

## IDL

Interface Definition Language. JSON-файл, генерируемый [Anchor](#anchor),
который описывает [обработчики инструкций](#instruction-handler), аккаунты и
типы программы. IDL позволяет клиентским библиотекам автоматически генерировать
типизированные интерфейсы для взаимодействия с
[ончейн-программой](#onchain-program).

## keypair

[Публичный ключ](#public-key-pubkey) и соответствующий
[приватный ключ](#private-key) для доступа к аккаунту.

## lamport

Дробный [нативный токен](#native-token) со значением 0.000000001 [SOL](#sol).

> В рамках вычислительного бюджета количество
> _[микро-lamport](https://github.com/solana-labs/solana/blob/ced8f6a512c61e0dd5308095ae8457add4a39e94/program-runtime/src/prioritization_fee.rs#L1-L2)_
> используется при расчете [комиссии за приоритизацию](#prioritization-fee).

## лидер

Роль [validator](#validator) при добавлении [записей](#entry) в
[реестр](#ledger).

## расписание лидеров

Последовательность [открытых ключей](#public-key-pubkey)
[validator](#validator), сопоставленных со [слотами](#slot). Кластер использует
расписание лидеров для определения, какой validator является [лидером](#leader)
в любой момент времени.

## реестр

Список [записей](#entry), содержащих [транзакции](#transaction), подписанные
[клиентами](#client). Концептуально это можно проследить до
[блока генезиса](#genesis-block), но фактический реестр [validator](#validator)
может содержать только более новые [блоки](#block) для сокращения хранилища,
поскольку более старые не требуются для валидации будущих блоков по дизайну.

## голосование в реестре

[Хэш](#hash) [состояния validator](#bank-state) на заданной
[высоте тика](#tick-height). Он представляет собой подтверждение
[validator](#validator) того, что полученный им [блок](#block) был проверен, а
также обещание не голосовать за конфликтующий [блок](#block) \(т.е.
[форк](#fork)\) в течение определенного периода времени — периода
[блокировки](#lockout).

## легкий клиент

Тип [клиента](#client), который может проверить, что он указывает на
действительный [кластер](#cluster). Он выполняет больше проверок реестра, чем
[тонкий клиент](#thin-client), и меньше, чем [validator](#validator).

## ликвидность

Глубина ордеров или объединенных резервов, доступных для торговли по цене,
равной или близкой к текущей. Более глубокая ликвидность снижает
[проскальзывание](#slippage). В [CLOB](#central-limit-order-book-clob)
ликвидность обеспечивается [ордерами мейкеров](#maker); в
[AMM](#automated-market-maker-amm) она обеспечивается поставщиками ликвидности,
которые вносят пары токенов в пул.

## загрузчик

[Программа](#program), способная интерпретировать бинарное кодирование других
[ончейн-программ](#onchain-program).

## блокировка

Период времени, в течение которого [validator](#validator) не может
[голосовать](#ledger-vote) за другой [форк](#fork).

## мейкер

Трейдер, чей ордер добавляет [ликвидность](#liquidity) в
[CLOB](#central-limit-order-book-clob), оставаясь в книге ордеров, а не
немедленно исполняясь на противоположной стороне. Такой ордер называется
мейкер-ордером. Мейкеры обычно не платят комиссию или получают возврат средств
от площадки.

## маркет-мейкер

Участник, который обеспечивает [ликвидность](#liquidity) путем постоянного
размещения как заявок на покупку, так и на продажу. На Solana маркет-мейкерами
являются либо программы (такие как [AMM](#automated-market-maker-amm)), либо
внебиржевые компании, использующие ботов для размещения [мейкер-ордеров](#maker)
на [CLOB](#central-limit-order-book-clob).

## сообщение

Структурированное содержимое [транзакции](#transaction). Обычно содержит
заголовок, массив адресов аккаунтов, последний [хеш блока](#blockhash) и массив
[инструкций](#instruction).

Узнайте больше о
[формате сообщений внутри транзакций](/docs/core/transactions/transaction-structure#header)
здесь.

## микро-lamport

1 000 000 микро-lamport = 1 [lamport](#lamport)

## коэффициент Накамото

Мера децентрализации, коэффициент Накамото — это наименьшее количество
независимых участников, которые могут совместно остановить блокчейн. Термин был
введен Баладжи С. Шринивасаном и Лиландом Ли в статье
[Quantifying Decentralization](https://news.earn.com/quantifying-decentralization-e39db233c28e).

## нативный токен

[Токен](#token), используемый для отслеживания работы, выполненной
[узлами](#node) в [кластере](#cluster).

## узел

Компьютер, участвующий в [кластере](#cluster).

## количество узлов

Количество [validator](#validator), участвующих в [кластере](#cluster).

## офчейн

Описывает данные, процессы или сервисы, которые существуют или работают вне
блокчейна Solana, такие как RPC-клиенты, индексаторы, фронтенды и офчейн-мейкеры
книги ордеров. Противоположность [ончейн](#onchain). Предпочтительное написание
— "офчейн" (слитно).

## ончейн

Описывает данные или программы, которые существуют или выполняются на блокчейне.
Предпочтительное написание — "ончейн" (слитно).

## ончейн-программа

Исполняемый код в блокчейне Solana, который интерпретирует
[инструкции](#instruction), отправленные внутри каждой
[транзакции](#transaction), для чтения и изменения аккаунтов, над которыми он
имеет контроль. Эти программы часто называют
"[_смарт-контрактами_](/docs/core/programs)" в других блокчейнах.

## оракул

[Ончейн-программа](#onchain-program), которая публикует данные извне сети — чаще
всего цены реальных активов — для чтения другими программами.

## владелец

Может относиться либо к [владеющей программе](#owning-program), либо к
[владеющему кошельку](#owning-wallet). Например,
[getTokenAccountsByOwner](https://solana.com/docs/rpc/http/gettokenaccountsbyowner)
получает token account, принадлежащие конкретному кошельку, однако владеющей
программой для этих token account является [Token Program](#token-program).

Во избежание путаницы предпочтительнее использовать либо
[владеющую программу](#owning-program), либо
[владеющий кошелёк](#owning-wallet), а не общий термин «владелец».

## владеющая программа

[Адрес](#address) [программы](#onchain-program), которая владеет аккаунтом.
Только владеющая программа способна изменять аккаунт.

См. также [полномочия](#authority).

## владеющий кошелёк

Кошелёк, связанный с аккаунтом. Например, Token Program используется для
хранения token account, связанных с конкретным кошельком и минтом, используя
адрес, созданный из этого [кошелька](#wallet) и [минта](#token-mint).

## PoH

См. [Proof of History](#proof-of-history-poh).

## пойнт

Взвешенный [кредит](#credit) в системе вознаграждений. В системе вознаграждений
[validator](https://docs.anza.xyz/consensus/stake-delegation-and-rewards)
количество пойнтов, причитающихся [стейку](#stake) при погашении, является
произведением заработанных [кредитов за голосование](#vote-credit) и количества
[lamport](#lamport) в стейке.

## приватный ключ

Приватный ключ [keypair](#keypair).

## программа

См. [ончейн-программа](#onchain-program).

## Program Derived Address (PDA)

Аккаунт, адрес которого создаётся на основе [ончейн-программы](#onchain-program)
и одного или нескольких [сидов](#seed), созданных автором программы. Адреса PDA
не являются публичными ключами, и PDA не имеют приватных ключей.

Полномочия на подпись для PDA принадлежат [ончейн-программе](#onchain-program),
которая является их [программой-владельцем](#owning-program).

## program id

Публичный ключ [аккаунта](#account), содержащего [программу](#onchain-program).

## proof of history (PoH)

Стек доказательств, каждое из которых подтверждает, что некоторые данные
существовали до создания доказательства и что прошел точный промежуток времени с
момента предыдущего доказательства. Как и [VDF](#verifiable-delay-function-vdf),
Proof of History может быть проверено быстрее, чем было создано.

## prioritization fee

Дополнительная комиссия, которую пользователь может указать в
[инструкции](#instruction) вычислительного бюджета для приоритизации своих
[транзакций](#transaction).

Комиссия за приоритизацию рассчитывается путем умножения запрошенного
максимального количества вычислительных единиц на цену вычислительной единицы
(указанную с шагом 0,000001 [lamport](#lamport) за вычислительную единицу) с
округлением до ближайшего [lamport](#lamport).

Транзакции должны запрашивать минимальное количество вычислительных единиц,
необходимое для выполнения, чтобы минимизировать комиссии.

## публичный ключ (pubkey)

Публичный ключ [keypair](#keypair). Публичный ключ является одним из различных
типов [адресов](#address), которые аккаунт может использовать в Solana.

## rent

Комиссия, уплачиваемая [аккаунтами](#account) и [программами](#program) за
хранение данных в блокчейне. Когда у аккаунтов недостаточно баланса для оплаты
rent, они могут быть удалены сборщиком мусора.

См. также [rent exempt](#rent-exempt) ниже. Узнайте больше о rent здесь:
[Что такое rent?](/docs/core/accounts/account-structure).

## rent exempt

Аккаунты, которые поддерживают минимальный баланс в lamport, пропорциональный
объему данных, хранящихся в аккаунте. Все вновь созданные аккаунты хранятся
ончейн постоянно до закрытия аккаунта. Невозможно создать аккаунт с балансом
ниже порога освобождения от rent.

## root

[Блок](#block) или [slot](#slot), который достиг максимального
[lockout](#lockout) на [validator](#validator). Root — это самый высокий блок,
который является предком всех активных форков на валидаторе. Все блоки-предки
root также транзитивно являются root. Блоки, которые не являются предком и не
являются потомком root, исключаются из рассмотрения для консенсуса и могут быть
отброшены.

## runtime

Компонент [validator](#validator), отвечающий за выполнение
[программ](#program).

## Sealevel

Параллельная среда выполнения Solana для [onchain-программ](#onchain-program).

## seed

Байтовое значение или массив байтовых значений, используемые в качестве входных
данных при выведении [program derived address](#program-derived-address-pda).
Программа может использовать любую комбинацию статических строк, публичных
ключей или других данных в качестве seed. Seed объединяются с идентификатором
программы и [bump](#bump) для создания уникального PDA-адреса.

## shred

Фрагмент [блока](#block); наименьшая единица, передаваемая между
[валидаторами](#validator).

## signature

64-байтовая ed25519-подпись из R (32 байта) и S (32 байта). С требованием, что R
является упакованной точкой Эдвардса не малого порядка, а S является скаляром в
диапазоне `0 <= S < L`. Это требование обеспечивает отсутствие податливости
подписи. Каждая транзакция должна иметь как минимум одну подпись для
[fee account](#fee-account). Таким образом, первую подпись в транзакции можно
рассматривать как [transaction id](#transaction-id).

## signer

[Аккаунт](#account), который авторизовал [инструкцию](#instruction) или
[транзакцию](#transaction). Для обычной [keypair](#keypair) это означает
криптографическую [подпись](#signature) над транзакцией. Для
[program derived address](#program-derived-address-pda) авторизация утверждается
[программой-владельцем](#owning-program) при выполнении
[Cross Program Invocation](#cross-program-invocation-cpi), поскольку PDA не
имеет приватного ключа.

## процент пропусков

Процент [пропущенных слотов](#skipped-slot) от общего числа слотов лидера в
текущей эпохе. Эта метрика может вводить в заблуждение, так как имеет высокую
дисперсию после границы эпохи, когда размер выборки небольшой, а также для
валидаторов с малым количеством слотов лидера. Тем не менее, она может быть
полезна для выявления неправильных конфигураций узла.

## пропущенный slot

Прошедший [slot](#slot), который не произвёл [блок](#block), потому что лидер
был офлайн или [форк](#fork), содержащий slot, был отброшен в пользу лучшей
альтернативы консенсусом кластера. Пропущенный slot не будет отображаться как
предок для блоков в последующих слотах, не увеличит
[высоту блока](#block-height) и не завершит срок действия старейшего
`recent_blockhash`.

Определить, был ли slot пропущен, можно только тогда, когда он становится старше
последнего [корневого](#root) (то есть не пропущенного) слота.

## проскальзывание

Разница между ценой, которую ожидал трейдер, и ценой, по которой фактически была
исполнена его сделка. В [AMM](#automated-market-maker-amm) оно возникает из-за
изменения ценовой кривой при изменении резервов; в
[CLOB](#central-limit-order-book-clob) оно возникает, когда один ордер
исполняется за счёт нескольких размещённых ордеров по последовательно
ухудшающимся ценам.

## slot

Период времени, в течение которого каждый [лидер](#leader) обрабатывает
транзакции и создаёт [блок](#block).

Совокупность слотов создаёт логические часы. Слоты упорядочены последовательно и
не перекрываются, составляя примерно равные промежутки реального времени в
соответствии с [PoH](#proof-of-history-poh).

[Библиотека программ](https://www.solana-program.com/docs/) на Solana, такая как

## качество обслуживания, взвешенное по стейку (SWQoS)

SWQoS позволяет

## суперквалифицированное большинство

2/3 [кластера](#cluster).

## System Program

Нативная программа Solana (ID программы `11111111111111111111111111111111`),
ответственная за создание новых [аккаунтов](#account), выделение данных
аккаунтов, назначение аккаунтов программам и перевод [SOL](#sol) между
аккаунтами. Все новые аккаунты в Solana принадлежат System Program до момента
переназначения.

## sysvar

Системный [аккаунт](#account). [Sysvars](https://docs.anza.xyz/runtime/sysvars)
предоставляют информацию о состоянии кластера, такую как текущая высота тика,
значения [поинтов](#point) вознаграждений и т.д. Программы могут получить доступ
к Sysvars через аккаунт Sysvar (pubkey) или путём запроса через системный вызов.

## тейкер

Трейдер, чья заявка удаляет [ликвидность](#liquidity) из
[CLOB](#central-limit-order-book-clob) путём пересечения противоположной стороны
книги ордеров и сопоставления с одной или несколькими размещёнными заявками.
Такая заявка называется заявкой тейкера, и тейкер обычно платит торговую
комиссию площадки.

## тонкий клиент

Тип [клиента](#client), который доверяет тому, что он взаимодействует с
действительным [кластером](#cluster).

## тик

Запись в [реестре](#entry), которая оценивает реальную длительность времени.

## высота тика

N-ный [тик](#tick) в [реестре](#ledger).

## токен

Цифровой передаваемый актив.

## token account

[Аккаунт](#account), который хранит баланс определённого [токена](#token). Token
account принадлежат [Token Program](#token-program) или
[Token Extensions Program](#token-extensions-program) и хранят баланс токена,
связанный [token mint](#token-mint) и адрес владеющего кошелька.

## Token Extensions Program

[Token Extensions Program](https://www.solana-program.com/docs/token-2022) имеет
ID программы `TokenzQdBNbLqP5VEhdkAS6EPFLC1PHnBqCXEpPxuEb` и включает все те же
функции, что и [Token Program](#token-program), но поставляется с расширениями,
такими как конфиденциальные переводы, пользовательская логика переводов,
расширенные метаданные и многое другое.

## токен минт

Аккаунт, который может производить (или «чеканить») токены. Различные токены
отличаются своими уникальными адресами токен минтов.

## Token Program

[Token Program](https://www.solana-program.com/docs/token) имеет идентификатор
программы `TokenkegQfeZyiNwAJbNbGKPFXCWuBvf9Ss623VQ5DA` и предоставляет базовые
возможности для переводов, заморозки и чеканки токенов.

## tps

[Транзакций](#transaction) в секунду.

## tpu

[Блок обработки транзакций](https://docs.anza.xyz/validator/tpu).

## транзакция

Одна или несколько [инструкций](#instruction), подписанных [клиентом](#client) с
использованием одной или нескольких [keypair](#keypair) и выполняемых атомарно
только с двумя возможными исходами: успех или неудача.

## идентификатор транзакции

Первая [подпись](#signature) в [транзакции](#transaction), которую можно
использовать для уникальной идентификации транзакции во всём [реестре](#ledger).

## подтверждения транзакции

Количество [подтверждённых блоков](#confirmed-block) с момента принятия
транзакции в [реестр](#ledger). Транзакция считается окончательной, когда её
блок становится [корневым](#root).

## запись транзакций

Набор [транзакций](#transaction), которые могут быть выполнены параллельно.

## tvu

[Блок валидации транзакций](https://docs.anza.xyz/validator/tvu).

## validator

Полноценный участник [кластера](#cluster) сети Solana, который производит новые
[блоки](#block). Validator проверяет транзакции, добавляемые в [реестр](#ledger)

## хранилище

[Token account](#token-account), принадлежащий
[ончейн-программе](#onchain-program), который хранит объединённые активы от
имени пользователей — например, базовые и котировочные резервы
[AMM](#automated-market-maker-amm), заблокированные токены в [эскроу](#escrow)
или остаточные балансы и накопленные комиссии
[CLOB](#central-limit-order-book-clob). Только владеющая программа может
перемещать токены из хранилища, обычно в собственный
[token account](#token-account) пользователя после выполнения условия сделки или
разблокировки.

## VDF

См. [функция проверяемой задержки](#verifiable-delay-function-vdf).

## функция проверяемой задержки (VDF)

Функция, выполнение которой занимает фиксированное количество времени и создаёт
доказательство своего выполнения, которое затем может быть проверено быстрее,
чем было создано.

## версионированная транзакция

Формат транзакции, поддерживающий дополнительные возможности по сравнению с
исходным форматом [транзакции](#transaction). Версия 0 (`v0`) — это первый
версионированный формат, добавляющий поддержку
[таблиц поиска адресов](#address-lookup-table-alt), что позволяет транзакциям
ссылаться на большее количество аккаунтов, чем допускает исходный формат.

## голос

См. [голос в реестре](#ledger-vote).

## голосовой кредит

Учёт вознаграждений для [validator](#validator). Голосовой кредит начисляется
validator в его голосовом аккаунте, когда validator достигает [корня](#root).

## кошелёк

Коллекция [keypair](#keypair), позволяющая пользователям управлять своими
средствами.

## период прогрева

Определённое количество [epoch](#epoch) после делегирования [stake](#stake), в
течение которых стейк постепенно становится активным. В этот период стейк
считается «активирующимся». Подробнее:
[прогрев и остывание](https://docs.anza.xyz/consensus/stake-delegation-and-rewards#stake-warmup-cooldown-withdrawal)
