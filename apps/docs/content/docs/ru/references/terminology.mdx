Следующие термины используются в документации и экосистеме разработки Solana.

## аккаунт

Запись в реестре Solana, которая либо содержит данные, либо является исполняемой
программой.

Как и аккаунт в традиционном банке, аккаунт Solana может хранить средства,
называемые [lamport](#lamport). Каждый аккаунт в [кластере](#cluster) имеет
уникальный [адрес](#address).

## приложение

Фронтенд-приложение, которое взаимодействует с кластером Solana.

## адрес

Местоположение в Solana, которое можно использовать для поиска аккаунта. Каждый
адрес уникален в [кластере](#cluster) Solana.

Адреса в Solana могут быть либо Ed25519 [публичным ключом](#public-key-pubkey),
либо [адресом, производным от программы](#program-derived-address).

## полномочие

Адрес пользователя, который имеет определённые права доступа к аккаунту.

Например:

- Возможность создавать новые токены предоставляется аккаунту, который является
  «полномочием эмиссии» для mint токена.
- Возможность обновлять программу предоставляется аккаунту, который является
  «полномочием обновления» программы.

## состояние банка

Результат интерпретации всех программ в леджере на заданной
[высоте тика](#tick-height). Включает как минимум набор всех
[аккаунтов](#account), содержащих ненулевое количество
[нативных токенов](#native-token).

## блок

Непрерывный набор [записей](#entry) в леджере, охватываемых
[голосованием](#ledger-vote). [Лидер](#leader) производит не более одного блока
за [slot](#slot).

## хеш блока

Уникальное значение ([хеш](#hash)), которое идентифицирует запись (блок). Solana
вычисляет хеш блока из последнего [идентификатора записи](#entry-id) блока.

## высота блока

Количество [блоков](#block) под текущим блоком. Первый блок после
[блока генезиса](#genesis-block) имеет высоту один.

## начальный валидатор

[Валидатор](#validator), который производит блок генезиса (первый)
[блок](#block) блокчейна.

## загрузчик BPF

Программа Solana, которая владеет и загружает
[BPF](/docs/core/programs#berkeley-packet-filter-bpf)
[ончейн-программы](#onchain-program), позволяя программе взаимодействовать со
средой выполнения.

## client

Компьютерная программа, которая обращается к сетевому [кластеру](#cluster)
сервера Solana.

## commitment

Мера подтверждения сети для [блока](#block).

## compact array format

Компактный массив — это массив, сериализованный в следующем формате:

- Длина массива (закодированная как
  [compact-u16](https://github.com/anza-xyz/agave/blob/v2.1.13/short-vec/src/lib.rs))
- Элементы массива, перечисленные один за другим

![Формат компактного массива](/assets/docs/core/transactions/compact_array_format.png)

Формат компактного массива используется для кодирования как массива
[адресов аккаунтов](/docs/core/transactions#account-addresses), так и массива
[инструкций](/docs/core/transactions#instructions) в сообщениях транзакций.

## cluster

Набор [валидаторов](#validator), поддерживающих единый [реестр](#ledger).

## compute budget

Максимальное количество [вычислительных единиц](#compute-units), потребляемых на
одну транзакцию.

## compute units

Наименьшая единица измерения потребления вычислительных ресурсов блокчейна.
Вычислительные единицы используются для измерения вычислительных ресурсов,
используемых для обработки транзакции.

## confirmation time

Реальное время между созданием [лидером](#leader) [записи тика](#tick) и
созданием [подтвержденного блока](#confirmed-block).

## confirmed block

[Блок](#block), получивший [суперквалифицированное большинство](#supermajority)
[голосов реестра](#ledger-vote).

## control plane

Gossip-сеть, соединяющая все [узлы](#node) [кластера](#cluster).

## cooldown period

Определенное количество [эпох](#epoch) после деактивации [стейка](#stake), в
течение которых он постепенно становится доступным для вывода. В этот период
стейк считается "деактивируемым". Подробнее:
[разогрев и охлаждение](https://docs.anza.xyz/implemented-proposals/staking-rewards#stake-warmup-cooldown-withdrawal)

## cost units

Также известные как `transaction cost`, представляют собой комплексную оценку
всех ресурсов, необходимых для обработки транзакции, измеряемых в
[вычислительных единицах](#compute-units). Подробнее
[здесь](https://www.anza.xyz/blog/why-solana-transaction-costs-and-compute-units-matter-for-developers).

## кредит

См. [кредит голосования](#vote-credit).

## cross-program invocation (CPI)

Вызов одной [ончейн-программы](#onchain-program) из другой. Для получения
дополнительной информации см. [вызовы между программами](/docs/core/cpi).

## плоскость данных

Многоадресная сеть, используемая для эффективной валидации [записей](#entry) и
достижения консенсуса.

## дрон

Офчейн-сервис, выступающий в роли хранителя приватного ключа пользователя.
Обычно используется для проверки и подписания транзакций.

## запись

Запись в [леджере](#ledger) — либо [тик](#tick), либо
[запись транзакции](#transactions-entry).

## идентификатор записи

Устойчивый к атакам поиска прообраза [хеш](#hash) финального содержимого записи,
выступающий глобальным уникальным идентификатором [записи](#entry). Хеш служит
доказательством того, что:

- Запись была сгенерирована после определённого промежутка времени
- Указанные [транзакции](#transaction) включены в запись
- Позиция записи относительно других записей в [леджере](#ledger)

См. [доказательство истории](#proof-of-history-poh).

## epoch

Время, то есть количество [слотов](#slot), в течение которого действует
[расписание лидеров](#leader-schedule).

## комиссионный аккаунт

Комиссионный аккаунт в транзакции — это аккаунт, который оплачивает стоимость
включения транзакции в леджер. Это первый аккаунт в транзакции. Данный аккаунт
должен быть объявлен как Read-Write (доступный для записи), поскольку оплата
транзакции уменьшает баланс аккаунта.

## финальность

Состояние, когда узлы, представляющие 2/3 [стейка](#stake), имеют общий
[корень](#root).

## форк

[Леджер](#ledger), произошедший от общих записей, но затем разошедшийся.

## блок генезиса

Первый [блок](#block) в цепи.

## конфигурация генезиса

Конфигурационный файл, который подготавливает [леджер](#ledger) для
[genesis-блока](#genesis-block).

## хеш

Цифровой отпечаток последовательности байтов.

## инфляция

Увеличение предложения токенов с течением времени, используемое для
финансирования вознаграждений за валидацию и для финансирования дальнейшей
разработки Solana.

## внутренняя инструкция

См. [Cross Program Invocation](#cross-program-invocation-cpi).

## инструкция

Вызов для запуска определенного [обработчика инструкций](#instruction-handler) в
[программе](#program). Инструкция также указывает, какие аккаунты она хочет
читать или изменять, а также дополнительные данные, которые служат
вспомогательным входом для [обработчика инструкций](#instruction-handler).
[Клиент](#client) должен включить хотя бы одну инструкцию в
[транзакцию](#transaction), и все инструкции должны быть выполнены, чтобы
транзакция считалась успешной.

## обработчик инструкций

Обработчики инструкций — это функции [программы](#program), которые обрабатывают
[инструкции](#instruction) из [транзакций](#transaction). Обработчик инструкций
может содержать один или несколько
[межпрограммных вызовов](#cross-program-invocation-cpi).

## keypair

[Публичный ключ](#public-key-pubkey) и соответствующий
[приватный ключ](#private-key) для доступа к аккаунту.

## lamport

Дробная часть [нативного токена](#native-token) со значением 0,000000001
[SOL](#sol).

> В рамках вычислительного бюджета величина
> _[микро-lamport](https://github.com/solana-labs/solana/blob/ced8f6a512c61e0dd5308095ae8457add4a39e94/program-runtime/src/prioritization_fee.rs#L1-L2)_
> используется при расчете [комиссии за приоритизацию](#prioritization-fee).

## лидер

Роль [validator](#validator), когда он добавляет [записи](#entry) в
[леджер](#ledger).

## расписание лидеров

Последовательность [публичных ключей](#public-key-pubkey)
[validator](#validator), сопоставленных со [slot](#slot). Кластер использует
расписание лидеров для определения, какой validator является [лидером](#leader)
в любой момент времени.

## леджер

Список [записей](#entry), содержащих [транзакции](#transaction), подписанные
[клиентами](#client). Концептуально это можно проследить до
[genesis-блока](#genesis-block), но фактический леджер [validator](#validator)
может содержать только более новые [блоки](#block) для сокращения хранилища, так
как более старые не нужны для валидации будущих блоков по своей конструкции.

## голосование в реестре

[Хеш](#hash) [состояния валидатора](#bank-state) на заданной
[высоте тика](#tick-height). Он представляет собой подтверждение
[валидатором](#validator) того, что полученный [блок](#block) был проверен, а
также обещание не голосовать за конфликтующий [блок](#block) \(т.е.
[форк](#fork)\) в течение определенного периода времени — периода
[блокировки](#lockout).

## легкий клиент

Тип [клиента](#client), который может проверить, что он указывает на
действительный [кластер](#cluster). Он выполняет больше проверок реестра, чем
[тонкий клиент](#thin-client), и меньше, чем [валидатор](#validator).

## загрузчик

[Программа](#program), способная интерпретировать бинарную кодировку других
ончейн-программ.

## блокировка

Период времени, в течение которого [валидатор](#validator) не может
[голосовать](#ledger-vote) за другой [форк](#fork).

## сообщение

Структурированное содержимое [транзакции](#transaction). Обычно содержит
заголовок, массив адресов аккаунтов, недавний [хеш блока](#blockhash) и массив
[инструкций](#instruction).

Узнайте больше о
[форматировании сообщений внутри транзакций](/docs/core/transactions#message-header)
здесь.

## микро-lamport

1 000 000 микро-lamport = 1 lamport

## коэффициент Накамото

Мера децентрализации, коэффициент Накамото — это наименьшее количество
независимых субъектов, которые могут коллективно остановить блокчейн. Термин был
предложен Баладжи С. Шринивасаном и Лиландом Ли в
[Quantifying Decentralization](https://news.earn.com/quantifying-decentralization-e39db233c28e).

## нативный токен

[Токен](#token), используемый для отслеживания работы, выполненной
[узлами](#node) в кластере.

## узел

Компьютер, участвующий в [кластере](#cluster).

## количество узлов

Количество [валидаторов](#validator), участвующих в [кластере](#cluster).

## ончейн-программа

Исполняемый код на блокчейне Solana, который интерпретирует
[инструкции](#instruction), отправленные внутри каждой
[транзакции](#transaction), чтобы читать и изменять аккаунты, которыми он
управляет. Эти программы часто называются
«[_смарт-контрактами_](/docs/core/programs)» на других блокчейнах.

## owner

Может относиться либо к [программе-владельцу](#owning-program), либо к
[кошельку-владельцу](#owning-wallet). Например,
[getTokenAccountsByOwner](https://solana.com/docs/rpc/http/gettokenaccountsbyowner)
получает аккаунты токенов, принадлежащие определенному кошельку, однако
программой-владельцем для этих аккаунтов токенов является
[Token Program](#token-program).

Во избежание путаницы предпочтительнее использовать либо
[программа-владелец](#owning-program), либо [кошелек-владелец](#owning-wallet),
а не общий термин «owner».

## owning program

[Адрес](#address) [программы](#onchain-program), которая владеет аккаунтом.
Только программа-владелец способна изменять аккаунт.

См. также [authority](#authority).

## owning wallet

Кошелек, связанный с аккаунтом. Например, Token Program используется для
хранения аккаунтов токенов, связанных с конкретным кошельком и минтом, используя
адрес, созданный из этого [кошелька](#wallet) и [минта](#token-mint).

## PoH

См. [Proof of History](#proof-of-history-poh).

## point

Взвешенный [кредит](#credit) в системе вознаграждений. В
[системе вознаграждений](https://docs.anza.xyz/consensus/stake-delegation-and-rewards)
[валидаторов](#validator) количество поинтов, причитающихся [стейку](#stake) при
выкупе, является произведением заработанных
[кредитов за голосование](#vote-credit) и количества застейканных lamport.

## private key

Приватный ключ [keypair](#keypair).

## program

См. [onchain program](#onchain-program).

## program derived account (PDA)

Аккаунт, адрес которого создается на основе
[onchain-программы](#onchain-program) и одного или нескольких [сидов](#seed),
созданных автором программы. Адреса PDA не являются публичными ключами, и PDA не
имеют приватных ключей.

Полномочия на подпись для PDA принадлежат [onchain-программе](#onchain-program),
которая является их [программой-владельцем](#owning-program).

## program id

Публичный ключ [аккаунта](#account), содержащего [программу](#onchain-program).

## proof of history (PoH)

Стек доказательств, каждое из которых подтверждает, что некоторые данные
существовали до создания доказательства и что прошёл точный промежуток времени
до предыдущего доказательства. Как и [VDF](#verifiable-delay-function-vdf),
Proof of History можно проверить за меньшее время, чем потребовалось для его
создания.

## комиссия приоритизации

Дополнительная комиссия, которую пользователь может указать в
[инструкции](#instruction) вычислительного бюджета для приоритизации своих
[транзакций](#transaction).

Комиссия приоритизации рассчитывается путём умножения запрошенного максимального
количества вычислительных единиц на цену вычислительной единицы (указанную в
приращениях 0,000001 lamport на вычислительную единицу) с округлением до
ближайшего lamport.

Транзакции должны запрашивать минимальное количество вычислительных единиц,
необходимых для выполнения, чтобы минимизировать комиссии.

## публичный ключ (pubkey)

Публичный ключ [keypair](#keypair). Публичный ключ — это один из различных типов
[адресов](#address), которые может использовать аккаунт в Solana.

## rent

Комиссия, уплачиваемая [аккаунтами](#account) и [программами](#program) за
хранение данных в блокчейне. Когда у аккаунтов недостаточно баланса для оплаты
rent, они могут быть удалены сборщиком мусора.

См. также [освобождённый от rent](#rent-exempt) ниже. Узнайте больше о rent
здесь: [Что такое rent?](/docs/intro/rent).

## освобождённый от rent

Аккаунты, которые поддерживают минимальный баланс в lamport, пропорциональный
объёму данных, хранящихся на аккаунте. Все вновь созданные аккаунты хранятся в
блокчейне постоянно до тех пор, пока аккаунт не будет закрыт. Невозможно создать
аккаунт, который окажется ниже порога освобождения от rent.

## корень

[Блок](#block) или [slot](#slot), который достиг максимальной
[блокировки](#lockout) на [validator](#validator). Корень — это самый высокий
блок, который является предком всех активных форков на validator. Все
блоки-предки корня также транзитивно являются корнем. Блоки, которые не являются
предками и не являются потомками корня, исключаются из рассмотрения для
консенсуса и могут быть отброшены.

## среда выполнения

Компонент [валидатора](#validator), отвечающий за выполнение
[программ](#program).

## Sealevel

Параллельная среда выполнения Solana для [ончейн-программ](#onchain-program).

## шред

Часть [блока](#block); наименьшая единица, передаваемая между
[валидаторами](#validator).

## подпись

64-байтовая подпись ed25519, состоящая из R (32 байта) и S (32 байта). При этом
R должна быть упакованной точкой Эдвардса, не имеющей малого порядка, а S —
скаляром в диапазоне `0 <= S < L`. Это требование исключает возможность
изменяемости подписи. Каждая транзакция должна иметь как минимум одну подпись
для [аккаунта комиссий](#fee-account). Таким образом, первая подпись в
транзакции может рассматриваться как [идентификатор транзакции](#transaction-id)

## доля пропущенных слотов

Процент [пропущенных слотов](#skipped-slot) от общего числа лидерских слотов в
текущем epoch. Эта метрика может вводить в заблуждение, поскольку имеет высокую
дисперсию сразу после границы epoch, когда размер выборки мал, а также для
валидаторов с небольшим количеством лидерских слотов. Однако она также может
быть полезна для выявления ошибок в конфигурации узлов.

## пропущенный slot

Прошедший [slot](#slot), который не создал [блок](#block) из-за того, что лидер
был офлайн или [форк](#fork), содержащий этот slot, был отвергнут в пользу
лучшей альтернативы на основе консенсуса кластера. Пропущенный slot не будет
отображаться в качестве предка для блоков в последующих слотах, не увеличит
[высоту блока](#block-height) и не истечёт старейший `recent_blockhash`.

Определить, был ли slot пропущен, можно только после того, как он станет старше
последнего [рутированного](#root) (то есть не пропущенного) слота.

## slot

Период времени, в течение которого каждый [лидер](#leader) получает транзакции и
создаёт [блок](#block).

В совокупности слоты создают логические часы. Слоты упорядочены последовательно
и не перекрываются, охватывая примерно равные промежутки реального времени в
соответствии с [PoH](#proof-of-history-poh).

## SOL

[Нативный токен](#native-token) [кластера](#cluster) Solana.

## Solana Program Library (SPL)

[Библиотека программ](https://spl.solana.com/) на Solana, такая как spl-token,
которая упрощает задачи, например создание и использование токенов.

## стейк

Токены, конфискованные у [кластера](#cluster), если может быть доказано
вредоносное поведение [валидатора](#validator).

## стейк-взвешенное качество обслуживания (SWQoS)

SWQoS позволяет
[предоставить приоритетную обработку для транзакций, исходящих от застейкавших валидаторов](/developers/guides/advanced/stake-weighted-qos).

## суперболь­шинство

2/3 [кластера](#cluster).

## sysvar

Системный [аккаунт](#account). [Sysvars](https://docs.anza.xyz/runtime/sysvars)
предоставляют информацию о состоянии кластера, такую как текущая высота тика,
значения [очков](#point) наград и т. д. Программы могут получить доступ к
Sysvars через аккаунт Sysvar (pubkey) или путём запроса через syscall.

## тонкий клиент

Тип [клиента](#client), который доверяет, что взаимодействует с действительным
[кластером](#cluster).

## тик

[Запись](#entry) в леджере, которая оценивает продолжительность реального
времени.

## высота тика

N-ный [тик](#tick) в [леджере](#ledger).

## токен

Цифровой актив, который можно передавать.

## Token Extensions Program

[Token Extensions Program](https://spl.solana.com/token-2022) имеет
идентификатор программы `TokenzQdBNbLqP5VEhdkAS6EPFLC1PHnBqCXEpPxuEb` и включает
все те же функции, что и [Token Program](#token-program), но поставляется с
расширениями, такими как конфиденциальные переводы, пользовательская логика
переводов, расширенные метаданные и многое другое.

## token mint

Аккаунт, который может производить (или 'чеканить') токены. Различные токены
отличаются уникальными адресами token mint.

## Token Program

[Token Program](https://spl.solana.com/token) имеет идентификатор программы
`TokenkegQfeZyiNwAJbNbGKPFXCWuBvf9Ss623VQ5DA` и предоставляет базовые
возможности для перевода, заморозки и чеканки токенов.

## tps

[Транзакций](#transaction) в секунду.

## tpu

[Модуль обработки транзакций](https://docs.anza.xyz/validator/tpu).

## транзакция

Одна или несколько [инструкций](#instruction), подписанных [клиентом](#client) с
использованием одной или нескольких [пар ключей](#keypair) и выполняемых
атомарно с двумя возможными исходами: успех или неудача.

## идентификатор транзакции

Первая [подпись](#signature) в [транзакции](#transaction), которая может
использоваться для уникальной идентификации транзакции во всём
[реестре](#ledger).

## подтверждения транзакции

Количество [подтверждённых блоков](#confirmed-block) с момента принятия
транзакции в [реестр](#ledger). Транзакция считается финализированной, когда её
блок становится [корнем](#root).

## запись транзакций

Набор [транзакций](#transaction), которые могут выполняться параллельно.

## tvu

[Модуль валидации транзакций](https://docs.anza.xyz/validator/tvu).

## validator

Полноправный участник [кластера](#cluster) сети Solana, который производит новые
[блоки](#block). Validator валидирует транзакции, добавляемые в
[реестр](#ledger)

## VDF

См. [функция с проверяемой задержкой](#verifiable-delay-function-vdf).

## функция с проверяемой задержкой (VDF)

Функция, выполнение которой занимает фиксированное время и которая генерирует
доказательство своего выполнения, которое затем может быть верифицировано за
меньшее время, чем потребовалось на его создание.

## голосование

См. [голосование в леджере](#ledger-vote).

## vote credit

Учёт вознаграждений для [validator](#validator). Vote credit начисляется
validator в его голосующий аккаунт, когда validator достигает [корня](#root).

## кошелёк

Коллекция [keypair](#keypair), которая позволяет пользователям управлять своими
средствами.

## период разогрева

Определённое количество [epoch](#epoch) после делегирования [стейка](#stake), в
течение которых он постепенно вступает в силу. В этот период стейк считается
«активирующимся». Подробнее о:
[разогреве и охлаждении](https://docs.anza.xyz/consensus/stake-delegation-and-rewards#stake-warmup-cooldown-withdrawal)
