AI-агенты становятся всё более автономными в своих рабочих процессах: пишут код,
получают данные, выделяют ресурсы и многое другое. Чтобы работать полностью
независимо, им необходим способ совершать «микроплатежи» за сервисы программно,
в реальном времени и с минимальными затратами. Традиционные платёжные системы не
подходят для микроплатежей. Субцентовые комиссии и финализация транзакций за
доли секунды в Solana делают это технически и экономически возможным.

## Почему это важно?

Например, представьте, что разработчику, использующему LLM для создания
приложения, нужны данные из платного API. Чтобы получить важные данные,
разработчику пришлось бы прервать рабочий процесс, изучить доступные API,
выбрать провайдера, создать аккаунт, добавить способ оплаты, сгенерировать
API-ключ, скопировать его в своё окружение и только после этого продолжить
работу. Такое трение нарушает поток работы. Для AI-ассистированных процессов это
становится препятствием.

С агентскими платежами агент может сам отправить запрос, оплатить его и
продолжить работу. Без аккаунтов. Без API-ключей. Без ручного вмешательства.

### Оплата API с помощью pay.sh

[pay.sh](https://pay.sh) — это платёжный уровень для HTTP-агентов и инструментов
командной строки. Обратитесь к [документации pay.sh](https://pay.sh/docs) для
установки `pay`. CLI `pay` оборачивает такие инструменты, как `curl`, `codex` и
`claude`, чтобы они могли вызывать платные API без аккаунтов провайдеров или
API-ключей. Когда API возвращает MPP или x402 `402 Payment Required`-запрос,
`pay` запрашивает у вашего локального кошелька подтверждение подписи.

Вы можете установить Pay с помощью:

```bash
brew install pay
```

или

```bash
npm install -g @solana/pay
```

Проверьте установку с помощью:

```bash
pay --version
```

Инструментарий автоматически настроит ваш адрес. Выполните следующие команды для
настройки и пополнения кошелька:

```bash
pay setup
```

## Протокол x402

Агентным платежам необходим механизм согласования условий оплаты между клиентами
и серверами через стандартную веб-инфраструктуру без нарушения работы
традиционных веб-интерфейсов. Рынок микроплатежей ещё находится на начальном
этапе развития, и по мере его эволюции будут появляться новые инструменты и
стандарты. Тем не менее [протокол x402](https://www.x402.org/) зарекомендовал
себя как ранний стандарт с широкой поддержкой экосистемы. x402 использует
[HTTP-статус 402 «Payment Required»](https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Status/402)
— стандарт, существующий с HTTP/1.1, который стал практически применимым
благодаря блокчейн-расчётам.

### Принцип работы

Клиент отправляет запрос, получает ответ 402 с условиями оплаты, а затем
повторяет запрос с подписанным платежом. Сервер делегирует проверку и расчёт
фасилитатору — необязательному посреднику, который обрабатывает отправку
транзакций в блокчейн. После того как фасилитатор подтверждает расчёт, сервер
возвращает запрошенный контент. Такое разделение позволяет провайдерам API
принимать платежи, не управляя блокчейн-инфраструктурой напрямую.

![Диаграмма потока x402](/assets/guides/x402/x402-flow-solana.png)

Протокол не имеет состояния (без сессий и API-ключей), работает со стандартной
HTTP-инфраструктурой и поддерживает любые SPL-токены, принимаемые сервером. Вот
пример настройки Express-сервера с использованием middleware x402:

```ts
app.use(
  paymentMiddleware(
    {
      "GET /costly-data": {
        accepts: [
          {
            scheme: "exact",
            price: "$0.001",
            network: "solana:EtWTRABZaYq6iMfeYKouRu166VU2xqa1",
            payTo: svmAddress
          }
        ],
        description: "Costly data",
        mimeType: "application/json"
      }
    },
    new x402ResourceServer(facilitatorClient).register(
      "solana:EtWTRABZaYq6iMfeYKouRu166VU2xqa1",
      new ExactSvmScheme()
    )
  )
);

app.get("/costly-data", (req, res) => {
  res.send({
    report: {
      data: "costly data"
    }
  });
});
```
