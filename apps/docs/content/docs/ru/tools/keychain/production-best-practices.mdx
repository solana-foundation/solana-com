Keychain абстрагирует бэкенд подписи, однако эксплуатация подписанта в
продакшене по-прежнему остаётся вашей ответственностью. Эти практики применимы
ко всем бэкендам; см. [Выбор бэкенда](/docs/tools/keychain/choosing-a-backend)
для подбора подходящего.

## Разделяйте горячие и холодные подписанты

Используйте операционный подписант с низким балансом («горячий») для частых,
автоматизированных транзакций и отдельный подписант с высокой ценностью
(«холодный») для движения средств казначейства. Обеспечьте холодный подписант
хранением с использованием MPC или облачного HSM и требуйте политик
подтверждения перед подписанием. Поскольку интерфейс идентичен, каждая роль
может использовать разный бэкенд без изменения точек вызова.

## Выполняйте ротацию ключей осознанно

В Solana ключ подписи _является_ адресом аккаунта, поэтому ротация ключа
порождает новый адрес. Планируйте ротацию как миграцию: подготовьте новый ключ,
переназначьте соответствующие права (или переведите средства) на новый адрес,
затем выведите из эксплуатации старый. Используйте отдельные ключи для каждой
среды, чтобы ключи devnet никогда не применялись в mainnet.

## Предоставляйте доступ с минимальными привилегиями

Ограничьте учётные данные каждого бэкенда минимумом, необходимым для подписания.
Для AWS KMS это `kms:Sign` и `kms:DescribeKey` на конкретном ключе;
предпочитайте роли IAM статическим учётным данным. Применяйте тот же принцип к
политикам Vault и ключам API управляемых кошельков.

## Планируйте обеспечение доступности

Удалённые бэкенды зависят от сетевого сервиса. Вызывайте `isAvailable()` перед
использованием подписанта, применяйте повторные попытки с отступом при временных
сбоях и определите резервный путь на случай отказов. Keychain принудительно
использует HTTPS для удалённых бэкендов и применяет стандартное время ожидания
запроса.

## Защита секретов и транспорта

- Все удалённые бэкенды взаимодействуют через HTTPS; эндпоинты без HTTPS
  отклоняются.
- Передавайте API-ключи, токены и учётные данные через менеджер секретов или
  переменные окружения — никогда не храните их в коде и не вносите в систему
  контроля версий.
- Keychain очищает сообщения об ошибках, чтобы исключить утечку секретов в логи;
  следите за тем, чтобы ваши собственные логи не содержали учётных данных и
  ключевого материала в открытом виде.

## Мониторинг и аудит подписания

Отслеживайте долю успешных подписаний, задержку и количество ошибок, настройте
оповещения при аномальных всплесках — см. метрики в разделе
[Готовность к продакшену](/docs/payments/production-readiness#monitoring).
Включите аудит-логи на стороне провайдера (KMS, Vault или кастодиан) и сверяйте
их с транзакциями, отправляемыми вашим приложением.
