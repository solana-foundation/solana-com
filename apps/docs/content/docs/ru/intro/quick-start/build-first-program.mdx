h1: Создайте свою первую программу на Solana

В этом кратком руководстве используется стартовый проект
[фреймворка Anchor](https://www.anchor-lang.com/docs), сгенерированный командой
`anchor init`. Вы создадите проект локально, запустите его тесты, соберёте
программу и изучите код программы, который генерирует Anchor.

## Предварительные требования

Прежде чем начать,
[установите инструменты разработки Solana](/docs/intro/installation). В
установку входят Rust, Solana CLI и Anchor CLI.

Используйте Anchor CLI версии 1.1.2 или выше для этого шаблона. Проверьте
установленную версию:

```terminal
$ anchor --version
```

<Steps>

<Step>

### Создание проекта

Выполните следующие команды в терминале:

```terminal
$ anchor init my-program
$ cd my-program
```

Стартовый проект включает одну программу Solana в `programs/my-program`.
Программа содержит две инструкции: одна инициализирует аккаунт счётчика, другая
увеличивает значение счётчика.

Некоторые части шаблона демонстрируют распространённые паттерны программ Solana:
получение адресов аккаунтов [PDA](/docs/core/pda), выполнение Cross Program
Invocation ([CPI](/docs/core/cpi)) для перевода SOL, а также использование
пользовательских проверок ошибок для остановки инструкции при невыполнении
условия.

</Step>

<Step>

### Сборка программы

Выполните `anchor build` для компиляции стартовой программы:

```terminal
$ anchor build
```

Скомпилированная программа записывается в `target/deploy/my_program.so`. После
развёртывания программы содержимое этого файла `.so` сохраняется в аккаунте в
блокчейне.

</Step>

<Step>

### Запустить тест

Запустите тест по умолчанию:

```terminal
$ anchor test
```

`Anchor.toml` этого шаблона использует команду тестирования Rust:

```toml title="Anchor.toml"
skip_local_validator = true

[scripts]
test = "cargo test"
```

Тест загружает скомпилированную программу в [LiteSVM](https://www.litesvm.com/),
создаёт плательщика, отправляет инструкции _rs`initialize`_ и _rs`increment`_,
затем проверяет состояние аккаунта счётчика.

Запуск `anchor test` также компилирует программу, поэтому при локальном
тестировании не нужно предварительно запускать `anchor build`.

</Step>

<Step>

### Развернуть программу

Локальные тесты обеспечивают наиболее быстрый цикл обратной связи. Когда вы
готовы развернуть программу в сети, например в devnet, сначала выполните сборку,
а затем разверните её в кластере.

Для развёртывания программы Solana требуется SOL, поскольку программа хранится в
program account, и этот аккаунт должен оплачивать используемое пространство. В
devnet можно получить бесплатный devnet SOL через
[Solana Faucet](https://faucet.solana.com/) или с помощью Solana CLI:

```terminal
$ solana airdrop 2 --url devnet
```

```terminal
$ anchor build
$ anchor deploy --provider.cluster devnet
```

</Step>

</Steps>

## Исходные файлы

Директория `src` содержит программу Solana. Документация Anchor по
[структуре программы](https://www.anchor-lang.com/docs/basics/program-structure)
объясняет основные макросы, используемые здесь, включая _rs`declare_id!`_,
_rs`#[program]`_, _rs`#[derive(Accounts)]`_ и _rs`#[account]`_. В этом разделе
рассматриваются файлы шаблона.

### lib.rs

`lib.rs` — точка входа программы. Он связывает исходные файлы, определяет адрес
программы и инструкции программы, которые пользователи могут вызывать.

<WithNotes>

```rust title="programs/my-program/src/lib.rs" -w
// !tooltip[/pub mod constants/] constantsMod
pub mod constants;
// !tooltip[/pub mod error/] errorMod
pub mod error;
// !tooltip[/pub mod instructions/] instructionsMod
pub mod instructions;
// !tooltip[/pub mod state/] stateMod
pub mod state;

// !tooltip[/anchor_lang::prelude/] prelude
use anchor_lang::prelude::*;

pub use constants::*;
pub use instructions::*;
pub use state::*;

// !tooltip[/declare_id!/] declareId
declare_id!("82sFkffP9wxwpyfZyeaKHH2chQoJPUGsJZSPi9mrUuXd");

// !tooltip[/#\[program\]/] programMacro
#[program]
pub mod my_program {
    use super::*;

    // !tooltip[/pub fn initialize/] publicInstructions
    pub fn initialize(ctx: Context<Initialize>) -> Result<()> {
        // !tooltip[/handle_initialize/] handlers
        crate::instructions::initialize::handle_initialize(ctx)
    }

    // !tooltip[/pub fn increment/] publicInstructions
    pub fn increment(ctx: Context<Increment>) -> Result<()> {
        // !tooltip[/handle_increment/] handlers
        crate::instructions::increment::handle_increment(ctx)
    }
}
```

### !constantsMod

_rs`pub mod constants`_ включает `constants.rs` в программный крейт. Здесь
шаблон хранит общие значения программы, такие как seed PDA, сумма перевода и
максимальное значение счётчика.

### !errorMod

_rs`pub mod error`_ включает `error.rs` в программный крейт. Этот файл
определяет пользовательские ошибки, которые программа может возвращать, если
инструкция недействительна.

### !instructionsMod

_rs`pub mod instructions`_ включает `instructions.rs` в программный крейт. Этот
файл работает как индекс для файлов, специфичных для инструкций, в
`src/instructions/`.

### !stateMod

_rs`pub mod state`_ включает `state.rs` в программный крейт. Этот файл
определяет структуру данных для аккаунтов, принадлежащих этой программе, включая
аккаунт _rs`Counter`_.

### !prelude

_rs`anchor_lang::prelude::*`_ импортирует типы и макросы Anchor, используемые
для определения программы Solana, включая _rs`Context`_, _rs`Result`_,
_rs`Pubkey`_, _rs`Account`_, _rs`Signer`_, и атрибуты аккаунтов Anchor.

### !reexports

Строки _rs`pub use`_ делают публичные элементы дочерних модулей доступными из
корня крейта. Это позволяет сократить точки входа инструкций, например
_rs`Context<Initialize>`_ вместо более длинного пути к модулю.

### !declareId

_rs`declare_id!`_ определяет адрес программы. Он должен совпадать с адресом в
`Anchor.toml`, задеплоенном program account, и у любых пользователей или тестов,
вызывающих программу.

### !programMacro

_rs`#[program]`_ указывает Anchor сгенерировать точку входа программы Solana и
код диспетчеризации инструкций. Публичные функции в этом модуле становятся
инструкциями, которые пользователи могут вызывать.

### !publicInstructions

_rs`initialize`_ и _rs`increment`_ — два названия инструкций, которые
пользователи могут вызывать. Каждая принимает _rs`Context<T>`_. Тип внутри
_rs`Context`_ определяет аккаунты, необходимые для этой инструкции, и Anchor
проверяет эти аккаунты перед выполнением инструкции.

### !handlers

Функция-обработчик содержит логику для инструкции.

</WithNotes>

### !left

```toml title="Anchor.toml" -w
[programs.localnet]
# !mark
my_program = "82sFkffP9wxwpyfZyeaKHH2chQoJPUGsJZSPi9mrUuXd"
```

### !right

```rust title="lib.rs" -w

```

</SideBySide>

Один и тот же адрес программы указан в конфигурации и в коде. `Anchor.toml`
сообщает Anchor, на какой адрес развёртывать программу или обращаться к ней в
кластере. _rs`declare_id!`_ задаёт адрес программы внутри самой программы для
проверок безопасности.

### constants.rs

`constants.rs` хранит общие значения в одном месте. В этом шаблоне
_rs`COUNTER_SEED`_ используется для получения PDA счётчика,
_rs`HELLO_WORLD_LAMPORTS`_ переводится при инициализации, а _rs`MAX_COUNT`_
проверяется перед инкрементом.

<WithNotes>

```rust title="programs/my-program/src/constants.rs" -w
use anchor_lang::prelude::*;

// !tooltip[/#\[constant\]/] anchorConstant
// !tooltip[/COUNTER_SEED/] counterSeed
#[constant]
pub const COUNTER_SEED: &[u8] = b"counter";

// !tooltip[/HELLO_WORLD_LAMPORTS/] helloLamports
#[constant]
pub const HELLO_WORLD_LAMPORTS: u64 = 1;

// !tooltip[/MAX_COUNT/] maxCount
#[constant]
pub const MAX_COUNT: u64 = 10;
```

### !anchorConstant

_rs`#[constant]`_ указывает Anchor включить значение в IDL программы. Это делает
важные константы программы видимыми для сгенерированных клиентов и инструментов.

### !counterSeed

_rs`COUNTER_SEED`_ — это байтовая строка, используемая для получения PDA
счётчика. Программа и тесты используют этот seed, чтобы согласованно определять
адрес аккаунта без хранения отдельного keypair. В этом шаблоне используется один
фиксированный seed, поэтому программа имеет один адрес счётчика. В реальных
программах в seeds часто включают адрес пользователя или другой идентификатор
для создания множества связанных аккаунтов.

### !helloLamports

_rs`HELLO_WORLD_LAMPORTS`_ — это количество lamport, переводимых от плательщика
на аккаунт счётчика внутри _rs`initialize`_. Значение намеренно небольшое, так
как шаблон демонстрирует CPI к System Program.

### !maxCount

_rs`MAX_COUNT`_ — это верхняя граница, проверяемая в _rs`increment`_. Хранение
её здесь даёт программе, тестам и сгенерированному IDL одно общее имя для
данного правила.

</WithNotes>

<SideBySide>

### !left

```rust title="initialize.rs" -w
#[derive(Accounts)]
pub struct Initialize<'info> {
    // ...
    #[account(
        init,
        payer = payer,
        space = 8 + Counter::INIT_SPACE,
        // !mark
        // !mention counter-seed
        seeds = [COUNTER_SEED],
        bump
    )]
    pub counter: Account<'info, Counter>,
    // ...
}

pub fn handle_initialize(ctx: Context<Initialize>) -> Result<()> {
    // ...

    // !mark
    // !mention hello-world-lamports
    anchor_lang::system_program::transfer(cpi_ctx, HELLO_WORLD_LAMPORTS)?;

    Ok(())
}
```

### !right

`initialize.rs` использует две константы:

- [_rs`COUNTER_SEED`_](mention:counter-seed) получает адрес PDA счётчика.
- [_rs`HELLO_WORLD_LAMPORTS`_](mention:hello-world-lamports) задаёт количество
  лампортов, переводимых от плательщика на аккаунт счётчика.

</SideBySide>

<SideBySide>

### !left

```rust title="increment.rs" -w
pub fn handle_increment(ctx: Context<Increment>) -> Result<()> {
    // ...

    require!(
        // !mark
        // !mention max-count
        ctx.accounts.counter.count < MAX_COUNT,
        ErrorCode::CounterOverflow,
    );

    ctx.accounts.counter.count += 1;
    Ok(())
}
```

### !right

`increment.rs` использует [_rs`MAX_COUNT`_](mention:max-count) как верхнюю
границу счётчика. Если текущее значение уже достигло максимума, _rs`require!`_
возвращает _rs`CounterOverflow`_ и данные аккаунта не изменяются.

</SideBySide>

### state.rs

`state.rs` определяет пользовательские типы данных для аккаунтов, которые
программа создаёт и которыми владеет. Программа определяет инструкции для
создания, инициализации и обновления этих данных, однако данные счётчика
хранятся не внутри самой программы, а в отдельном аккаунте с собственным
адресом.

<WithNotes>

```rust title="programs/my-program/src/state.rs" -w
use anchor_lang::prelude::*;

// !tooltip[/#\[account\]/] accountMacro
#[account]
// !tooltip[/InitSpace/] initSpace
#[derive(InitSpace)]
pub struct Counter {
    // !tooltip[/count/] count
    pub count: u64,
    // !tooltip[/authority/] authority
    pub authority: Pubkey,
}
```

### !accountMacro

_rs`#[account]`_ сообщает Anchor, что эта структура является данными, которые
могут храниться в онchain-аккаунте, принадлежащем этой программе. Anchor
использует её для сериализации, десериализации и валидации аккаунтов
_rs`Counter`_.

### !initSpace

_rs`InitSpace`_ позволяет Anchor вычислить _rs`Counter::INIT_SPACE`_ —
количество байт, необходимых для полей этой структуры. 8-байтовый дискриминатор
аккаунта Anchor добавляется отдельно при создании аккаунта.

### !count

_rs`count`_ — это значение счётчика, хранящееся внутри данных аккаунта.

### !authority

_rs`authority`_ хранит адрес, которому разрешено увеличивать счётчик.
_rs`initialize`_ записывает этот адрес, а _rs`increment`_ требует подписи от
того же адреса.

</WithNotes>

<SideBySide>

### !left

```rust title="initialize.rs" -w
#[account(
    init,
    payer = payer,
    // !mark
    // !mention counter-space
    space = 8 + Counter::INIT_SPACE,
    seeds = [COUNTER_SEED],
    bump
)]
pub counter: Account<'info, Counter>,
// ...

pub fn handle_initialize(ctx: Context<Initialize>) -> Result<()> {
    // !mark(1:2)
    // !mention(1:2) initialize-fields
    ctx.accounts.counter.count = 0;
    ctx.accounts.counter.authority = ctx.accounts.payer.key();

    Ok(())
}
```

```rust title="increment.rs" -w
pub fn handle_increment(ctx: Context<Increment>) -> Result<()> {
    // ...

    // !mark
    // !mention increment-count
    ctx.accounts.counter.count += 1;
    Ok(())
}
```

### !right

`state.rs` определяет данные аккаунта _rs`Counter`_. Файлы инструкций используют
этот тип при создании и обновлении аккаунта:

- [_rs`Counter::INIT_SPACE`_](mention:counter-space) задаёт размер аккаунта для
  полей, определённых в `state.rs`.
- [_rs`count`_ и _rs`authority`_](mention:initialize-fields) — значения полей,
  записываемые при инициализации аккаунта.
- [_rs`count += 1`_](mention:increment-count) обновляет сохранённое значение
  счётчика после успешной валидации.

</SideBySide>

### error.rs

`error.rs` определяет пользовательские ошибки программы. В этом шаблоне ошибки
демонстрируют, как обработчики инструкций останавливаются, если вызывающая
сторона не имеет права обновлять счётчик или счётчик уже достиг _rs`MAX_COUNT`_.

<WithNotes>

```rust title="programs/my-program/src/error.rs" -w
use anchor_lang::prelude::*;

// !tooltip[/#\[error_code\]/] errorCode
#[error_code]
pub enum ErrorCode {
    // !tooltip[/Unauthorized/] unauthorized
    #[msg("Only the counter authority can update this counter")]
    Unauthorized,
    // !tooltip[/CounterOverflow/] counterOverflow
    #[msg("Counter has reached the maximum value")]
    CounterOverflow,
}
```

### !errorCode

_rs`#[error_code]`_ указывает Anchor преобразовать эти варианты в
пользовательские ошибки программы. Клиенты и тесты могут использовать эти ошибки
вместо того, чтобы угадывать по необработанным кодам ошибок.

### !unauthorized

_rs`Unauthorized`_ возвращается, когда подписант транзакции не является
авторизованной стороной, сохранённой в аккаунте счётчика.

### !counterOverflow

_rs`CounterOverflow`_ возвращается, когда инкремент нарушил бы правило
_rs`MAX_COUNT`_ программы.

</WithNotes>

<SideBySide>

### !left

```rust title="increment.rs" -w
pub fn handle_increment(ctx: Context<Increment>) -> Result<()> {
    require_keys_eq!(
        ctx.accounts.counter.authority,
        ctx.accounts.authority.key(),
        // !mark
        // !mention unauthorized-error
        ErrorCode::Unauthorized,
    );
    require!(
        ctx.accounts.counter.count < MAX_COUNT,
        // !mark
        // !mention overflow-error
        ErrorCode::CounterOverflow,
    );

    ctx.accounts.counter.count += 1;
    msg!("Hello, world! Counter is now {}", ctx.accounts.counter.count);
    Ok(())
}
```

### !right

`error.rs` перечисляет ошибки, которые может вернуть инструкция:

- [_rs`ErrorCode::Unauthorized`_](mention:unauthorized-error) возвращается,
  когда подписант не является авторизованной стороной, сохранённой в аккаунте
  счётчика.
- [_rs`ErrorCode::CounterOverflow`_](mention:overflow-error) возвращается, когда
  счётчик уже достиг _rs`MAX_COUNT`_.

</SideBySide>

### instructions.rs

`instructions.rs` связывает файлы инструкций с крейтом программы, чтобы `lib.rs`
мог получить доступ к коду инструкций _rs`initialize`_ и _rs`increment`_. Каждый
файл инструкции определяет аккаунты, необходимые для этой инструкции, и логику
обработчика, которая выполняется после того, как Anchor проверит эти аккаунты.

<WithNotes>

```rust title="programs/my-program/src/instructions.rs" -w
// !tooltip[/pub mod initialize/] moduleDeclarations
pub mod initialize;
pub mod increment;

// !tooltip[/pub use initialize/] reexports
pub use initialize::*;
pub use increment::*;
```

### !moduleDeclarations

Строки _rs`pub mod`_ делают `src/instructions/initialize.rs` и
`src/instructions/increment.rs` частью крейта программы.

### !reexports

Строки _rs`pub use`_ реэкспортируют публичные элементы из этих файлов. Это
делает структуры аккаунтов инструкций и функции обработчиков доступными через
модуль _rs`instructions`_.

</WithNotes>

### initialize.rs

`initialize.rs` определяет аккаунты, необходимые для создания аккаунта счётчика,
и затем записывает первоначальные значения аккаунта. Структура
_rs`#[derive(Accounts)]`_ использует
[ограничения аккаунтов](https://www.anchor-lang.com/docs/references/account-constraints)
Anchor, чтобы указать, какие аккаунты требуются и как создаётся новый аккаунт
счётчика.

<WithNotes>

```rust title="programs/my-program/src/instructions/initialize.rs" -w
use anchor_lang::prelude::*;

use crate::{constants::*, state::Counter};

// !tooltip[/derive\(Accounts\)/] accountsStruct
#[derive(Accounts)]
pub struct Initialize<'info> {
    // !tooltip[/#\[account\(mut\)\]/] payerMutable
    #[account(mut)]
    // !tooltip[/Signer<'info>/] payerSigner
    pub payer: Signer<'info>,
    // !tooltip[/#\[account\(/] counterConstraints
    #[account(
        // !tooltip[/init/] initConstraint
        init,
        // !tooltip[/payer = payer/] payerConstraint
        payer = payer,
        // !tooltip[/space = 8 \+ Counter::INIT_SPACE/] space
        space = 8 + Counter::INIT_SPACE,
        // !tooltip[/seeds = \[COUNTER_SEED\]/] seeds
        seeds = [COUNTER_SEED],

        bump

    // !tooltip[/Account<'info, Counter>/] counterAccountType
    pub counter: Account<'info, Counter>,
    // !tooltip[/Program<'info, System>/] systemProgram
    pub system_program: Program<'info, System>,

// !tooltip[/handle_initialize/] handler
pub fn handle_initialize(ctx: Context<Initialize>) -> Result<()> {
    // !tooltip[/counter.count = 0/] initialCount
    ctx.accounts.counter.count = 0;
    // !tooltip[/counter.authority = ctx.accounts.payer.key\(\)/] initialAuthority
    ctx.accounts.counter.authority = ctx.accounts.payer.key();

        from: ctx.accounts.payer.to_account_info(),
        to: ctx.accounts.counter.to_account_info(),

    // !tooltip[/system_program::transfer/] transfer
```

### !imports

Этот файл импортирует общие константы и тип аккаунта _rs`Counter`_. Инструкция
использует эти определения для получения PDA, выбора суммы передачи и выделения
аккаунта с нужным размером данных.

### !accountsStruct

_rs`Initialize`_ — это контекст аккаунтов для инструкции _rs`initialize`_.
Каждое поле в структуре представляет отдельный аккаунт, который должен быть
включён при вызове инструкции. Anchor использует структуру для валидации
входящих аккаунтов перед выполнением обработчика.

### !payerMutable

_rs`#[account(mut)]`_ указывает Anchor, что аккаунт плательщика должен быть
доступен для записи. Это необходимо, поскольку создание аккаунта и CPI передачи
оба списывают lamport с плательщика.

### !payerSigner

_rs`Signer<'info>`_ указывает Anchor, что плательщик должен подписать
транзакцию.

### !counterConstraints

Блок _rs`#[account(...)]`_ указывает Anchor, как создать и проверить аккаунт
счётчика перед выполнением обработчика.

### !initConstraint

_rs`init`_ указывает Anchor создать аккаунт _rs`counter`_ до выполнения
обработчика. Новый аккаунт принадлежит этой программе, поэтому она может хранить
в нём данные _rs`Counter`_.

### !payerConstraint

_rs`payer = payer`_ указывает аккаунт _rs`payer`_ в качестве плательщика за
создание нового аккаунта счётчика.

### !space

_rs`space = 8 + Counter::INIT_SPACE`_ выделяет байты для дискриминатора аккаунта
Anchor, а также для полей _rs`Counter`_ из `state.rs`. Дискриминатор помогает
Anchor убедиться, что аккаунт хранит ожидаемый тип данных, прежде чем читать их.

### !seeds

_rs`seeds = [COUNTER_SEED]`_ указывает, что адрес аккаунта счётчика должен быть
PDA, производным от этого seed и идентификатора программы. При использовании
одного фиксированного seed для этой программы существует один PDA счётчика.
Чтобы создать отдельные счётчики в реальной программе, обычно добавляют ещё один
seed, например адрес пользователя.

### !bump

_rs`bump`_ указывает Anchor найти и проверить bump PDA — дополнительный байт,
который делает производный адрес допустимым как PDA.

### !counterAccountType

_rs`Account<'info, Counter>`_ указывает Anchor интерпретировать данные аккаунта
как тип _rs`Counter`_ из `state.rs`. Anchor также проверяет, что аккаунт
принадлежит данной программе.

### !systemProgram

_rs`Program<'info, System>`_ требует, чтобы вызывающая сторона предоставила
System Program. Anchor использует её для ограничения создания аккаунта
_rs`init`_. CPI-перевод в обработчике также вызывает System Program.

### !handler

Обработчик запускается после того, как Anchor проверит аккаунты и создаст
аккаунт счётчика, поэтому он может безопасно записать начальные данные аккаунта.

### !initialCount

Устанавливает поле _rs`count`_ в аккаунте счётчика в значение _rs`0`_.

### !initialAuthority

Устанавливает поле _rs`authority`_ в аккаунте счётчика в адрес плательщика.

### !cpiAccounts

Это аккаунты, передаваемые в инструкцию перевода System Program. Данный
CPI-перевод включён исключительно для демонстрации того, как CPI передаёт
аккаунты другой программе.

### !cpiContext

_rs`CpiContext`_ формирует целевую программу и необходимые аккаунты для Cross

## !!steps Функция обработчика

Функция _rs`handle_increment`_ запускается после того, как Anchor проверяет
аккаунты в _rs`Increment`_. Значение _rs`ctx`_ предоставляет обработчику доступ
к этим проверенным аккаунтам.

<CodePlaceholder title="increment.rs" />

```rust !! title="increment.rs"

        bump
    )]

    pub authority: Signer<'info>,
}

// !focus
pub fn handle_increment(ctx: Context<Increment>) -> Result<()> {
    require_keys_eq!(
        ctx.accounts.counter.authority,
        ctx.accounts.authority.key(),
        ErrorCode::Unauthorized,
    );
    require!(
        ctx.accounts.counter.count < MAX_COUNT,
        ErrorCode::CounterOverflow,

    msg!("Hello, world! Counter is now {}", ctx.accounts.counter.count);
    Ok(())
}
```

## !!steps Проверка прав доступа

Первая проверка убеждается, что подписант имеет право обновлять этот счётчик.
Если адрес подписанта не совпадает с _rs`counter.authority`_, выполнение
инструкции прерывается с ошибкой _rs`ErrorCode::Unauthorized`_. Это
демонстрирует авторизацию на уровне приложения: программа владеет данными
счётчика, но реализует правило, определяющее, какой подписант может изменять эти
данные.

### !incrementAccounts

_rs`Increment`_ — это контекст аккаунтов для инструкции _rs`increment`_. Anchor
использует его для проверки изменяемого аккаунта счётчика и необходимого
подписанта.

### !counterMutable

_rs`mut`_ сообщает Anchor, что аккаунт счётчика должен быть доступен для записи,
поскольку обработчик обновляет сохранённое значение _rs`count`_.

### !counterSeeds

_rs`seeds = [COUNTER_SEED]`_ проверяет, что предоставленный аккаунт счётчика
является PDA, полученным из этого seed и идентификатора программы.

### !counterBump

_rs`bump`_ указывает Anchor найти и проверить bump PDA для аккаунта счётчика.

### !incrementCounterAccountType

_rs`Account<'info, Counter>`_ указывает Anchor десериализовать данные аккаунта
как тип _rs`Counter`_ из `state.rs`. Anchor также проверяет, что аккаунт
принадлежит данной программе.

### !authoritySigner

_rs`Signer`_ требует подписи authority. Затем обработчик сравнивает адрес этого
подписанта с authority, сохранённым в аккаунте счётчика.

### !incrementHandler

Обработчик запускается после того, как Anchor проверяет список аккаунтов, права
владения и десериализует аккаунт счётчика.

### !authorityCheck

_rs`require_keys_eq!`_ сравнивает два публичных ключа и останавливает выполнение
инструкции, если адрес подписанта не совпадает с authority, сохранённым в
аккаунте счётчика. Это правило авторизации на уровне приложения в данном
шаблоне.

### !maxCheck

_rs`require!`_ проверяет булево условие и останавливает выполнение инструкции с
ошибкой _rs`CounterOverflow`_, если счётчик уже достиг значения _rs`MAX_COUNT`_.
Это правило шаблона демонстрирует пользовательские ошибки и прерывание
выполнения до изменения данных.

### !mutateCount

Только после прохождения обеих проверок обработчик обновляет данные аккаунта
счётчика. Если хотя бы одна проверка не пройдена, транзакция завершается с
ошибкой и аккаунт не изменяется.

### !programLog

_rs`msg!`_ записывает сообщение в журналы программы при её вызове.

### !litesvm

_rs`LiteSVM`_ — это внутрипроцессная Solana VM для тестирования. Она позволяет
тесту зарегистрировать скомпилированную программу, создавать аккаунты,
отправлять транзакции и проверять состояние аккаунтов без запуска процесса
validator.

### !setup

Тест получает адрес программы из значения _rs`declare_id!`_ через
_rs`my_program::id()`_ и создаёт новый keypair плательщика для тестовой
транзакции.

### !counterPda

Адрес счётчика выводится из того же _rs`COUNTER_SEED`_, который используется
программой. Это позволяет тесту обращаться к тому PDA, который ожидает
`initialize.rs`.

### !programBytes

_rs`include_bytes!`_ считывает скомпилированный файл `my_program.so`, созданный
в процессе сборки. Эти байты представляют собой исполняемый код программы,
который LiteSVM запустит в тесте.

### !addProgram

_rs`svm.add_program`_ регистрирует скомпилированные байты программы по адресу
программы в LiteSVM. Airdrop даёт плательщику достаточно локальных SOL для
оплаты транзакции и создания аккаунта.

### !testInstruction

_rs`Instruction::new_with_bytes`_ создаёт инструкцию для вызова программы. Она
включает идентификатор программы, instruction data и аккаунты инструкции.

### !blockhash

Каждая транзакция Solana требует актуального blockhash. В данном тесте LiteSVM
предоставляет blockhash для использования в транзакции.

### !message

Сообщение содержит инструкцию, плательщика комиссии и blockhash. Это данные
транзакции, которые будут подписаны.

### !transaction

_rs`VersionedTransaction::try_new`_ подписывает сообщение с помощью keypair
плательщика. Плательщик подписывает транзакцию, поскольку оплачивает комиссию и
финансирует новый аккаунт счётчика.

### !sendTx

_rs`svm.send_transaction`_ выполняет транзакцию в LiteSVM.

### !getCounter

После выполнения транзакции тест считывает аккаунт счётчика непосредственно из
LiteSVM, чтобы проверить данные аккаунта.

### !deserialize

Аккаунт счётчика хранит сериализованные данные. _rs`Counter::try_deserialize`_
проверяет дискриминатор аккаунта Anchor и преобразует данные аккаунта обратно в
тип _rs`Counter`_ из `state.rs`.

### !initAssert

После успешного выполнения _rs`initialize`_ тест десериализует аккаунт счётчика
и проверяет, что начальное состояние соответствует тому, что записала
_rs`handle_initialize`_.

### !incrementAssert

После успешного выполнения _rs`increment`_ тот же аккаунт счётчика должен иметь
_rs`count`_, установленный в _rs`1`_, при этом сохраняя прежний authority.
