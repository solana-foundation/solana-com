`@solana/kit` toimittaa `createClient`, pienen isäntäobjektin, jota laajennat
liitännäisillä. Jokainen liitännäinen lisää ominaisuuden — RPC-yhteyden,
allekirjoittajan, transaktion suunnittelun — ja koottu asiakasohjelma palvelee
skriptejä, palvelinreittejä ja käyttöliittymiä yhtä lailla.

<Callout type="info">
  Rakennatko React-sovellusta? [React-opas](/docs/frontend/react-hooks) käärii
  tämän saman asiakkaanohjelman `@solana/react`:n avulla, jotta komponentit
  lukevat lompakkotilan ja RPC-datan hookien kautta. Aloita sieltä ja palaa
  tänne, kun tarvitset tarkempaa hallintaa.
</Callout>

## Asennus

```terminal
$ npm install @solana/kit @solana/kit-plugin-rpc @solana/kit-plugin-signer
```

Käytä mitä tahansa paketinhallintaohjelmaa. Kit on v7+ ja
`@solana/kit-plugin-*`-paketit ovat 0.13+.

## Luo asiakasohjelma

Yhdistä liitännäiset `.use(...)`:llä. Aseta allekirjoittaja ensin, sitten
RPC-yhteys. `solanaDevnetRpc()` lisää `client.rpc`:n, tilaukset, transaktion
suunnittelijan/suorittajan ja `client.sendTransaction`-apufunktion — joten
sovelluksen harvoin tarvitsee koota näitä osia itse.

```ts
import { createClient, generateKeyPairSigner } from "@solana/kit";
import { solanaDevnetRpc } from "@solana/kit-plugin-rpc";
import { signer } from "@solana/kit-plugin-signer";

const payer = await generateKeyPairSigner();

const client = createClient().use(signer(payer)).use(solanaDevnetRpc());

const slot = await client.rpc.getSlot().send();
console.log("current slot", slot);
```

`signer`-liitännäinen asettaa sekä maksun maksajan että sovelluksen
identiteetin. RPC-paketti vaatii, että maksaja on määritetty ensin, minkä vuoksi
järjestyksellä on merkitystä.

### Allekirjoittajavariantit

`@solana/kit-plugin-signer` tarjoaa kolme roolia, jotta voit myöntää vain sen,
mitä asiakasohjelma tarvitsee:

- `signer(x)` — asettaa sekä `client.payer`:n että `client.identity`:n.
- `payer(x)` — ainoastaan maksun maksaja.
- `identity(x)` — ainoastaan sovelluksen auktoriteetti.

Jokaisella on `*FromFile(path)` (Node), `generated*()`,
`generated*WithSol(amount)` ja `airdrop*()` -variantit testejä ja skriptejä
varten.

## Lähetä transaktio

Rakenna käskyt luodulla ohjelmaklientillä ja anna ne sitten
`client.sendTransaction`:lle. Suunnittelija ratkaisee lohkohashin, arvioi
laskennan, allekirjoittaa asiakkaan maksajalla ja lähettää transaktion.

```ts
import { getTransferSolInstruction } from "@solana-program/system";
import { address, lamports } from "@solana/kit";

const transfer = getTransferSolInstruction({
  source: client.payer,
  destination: address("Fke...address"),
  amount: lamports(10_000_000n) // 0.01 SOL
});

const { context } = await client.sendTransaction([transfer]);
console.log("submitted", context.signature);
```

`sendTransaction` palautuu tulososioksi; transaktion allekirjoitus on
`result.context.signature`:ssa. Se varmistaa, että suunnitelma mahtuu yhteen
transaktioon, ja heittää poikkeuksen muutoin — käytä `sendTransactions`:tä, kun
erä voi kattaa useita transaktioita.

## Valitse RPC-paketti

`@solana/kit-plugin-rpc` toimittaa klusterin esiasetukset:

- `solanaDevnetRpc()` — devnet-oletukset, airdrop käytössä (`client.airdrop`).
- `solanaMainnetRpc()` — mainnet; airdrop on käännösaikainen virhe.
- `solanaLocalRpc()` — `http://127.0.0.1:8899` paikalliselle validatorille.
- `solanaRpc({ rpcUrl })` — mikä tahansa päätepiste.

```ts
import { solanaRpc } from "@solana/kit-plugin-rpc";

const client = createClient()
  .use(signer(payer))
  .use(solanaRpc({ rpcUrl: "https://your-rpc-provider.example" }));
```

### Kokoa yksityiskohtaisesti

Esiasetukset sisältävät RPC-yhteyden, suunnittelijan, suorittajan ja
lähetysapuohjelmat, joten useimmat sovellukset eivät koskaan tarvitse osia
erikseen. Siirry yksityiskohtaisiin liitännäisiin vain silloin, kun haluat
mukauttaa yhtä niistä — esimerkiksi mukautettua suunnittelijaa tai eri
tilauskanavaa. Tämä polku vaatii myös `@solana/kit-plugin-instruction-plan`,
jonka esiasetukset muutoin sisällyttävät puolestasi:

```ts
import {
  solanaRpcConnection,
  rpcAirdrop,
  rpcTransactionPlanner,
  rpcTransactionPlanExecutor
} from "@solana/kit-plugin-rpc";
import { planAndSendTransactions } from "@solana/kit-plugin-instruction-plan";

const client = createClient()
  .use(signer(payer))
  .use(solanaRpcConnection({ rpcUrl: "https://api.devnet.solana.com" }))
  .use(rpcAirdrop())
  .use(rpcTransactionPlanner())
  .use(rpcTransactionPlanExecutor())
  .use(planAndSendTransactions());
```

## Lisää ohjelma-asiakkaat

Generoidut `@solana-program/*` -paketit tarjoavat asiakasliitännäisiä, jolloin
ohjelma-kutsut kiinnittyvät asiakkaaseen. Asenna tarvitsemasi ohjelma ja
`.use()` sen liitännäinen:

```ts
import { tokenProgram } from "@solana-program/token";

const client = createClient()
  .use(signer(payer))
  .use(solanaDevnetRpc())
  .use(tokenProgram()); // adds client.token
```

<Callout type="warn" title="Peer dependency ranges">
  Jotkin `@solana-program/*` -julkaisut ilmoittavat edelleen `@solana/kit ^6.x`
  -vertaisriippuvuuden, kun Kit on versiossa 7.x. Ne toimivat Kit 7:n kanssa —
  ohita vertaisriippuvuuden versioalue paketinhallinnassasi, jos se valittaa.
</Callout>

## Allekirjoittaminen ilman selainlompakkoa

Suoritatko headless-tilassa API-reitissä, workerissa tai skriptissä?
Allekirjoita taustajärjestelmän avainhallintopalvelulla raa'an keypair-parin
sijaan. Katso [Keychain](/docs/tools/keychain) ja
[Allekirjoittaminen tuotannossa](/docs/core/transactions/signing-in-production).

## Yleisiä malleja Solana-kehittäjille

- **Jaettu asennus**: Rakenna asiakas kerran ydinmoduulissa ja tuo se
  skripteihin, palvelureitteihin ja käyttöliittymääsi.
- **Palvelinpuolen luku**: Asiakas, jossa on vain `signer` + `solanaRpc` (ei
  lompakkoliitännäistä), lukee tilejä ja simuloi transaktioita palvelimelta.
- **Paikallinen testaus**: Vaihda RPC-paketti `.use(litesvm())` -paketiksi
  `@solana/kit-plugin-litesvm`:stä ajaaksesi muistissa olevan validatorin kanssa
  (vain Node).
- **Testattavuus**: Koostettu asiakas on helppo simuloida — stubaa
  RPC-vastaukset tai allekirjoittajat ilman selaimen lompakkoa.
