本快速入门使用由 `anchor init` 生成的
[Anchor 框架](https://www.anchor-lang.com/docs)入门项目。您将在本地创建该项目，运行其测试，构建程序，并了解 Anchor 生成的程序代码。

## 前提条件

在开始之前，请[安装 Solana 开发工具](/docs/intro/installation)。安装内容包括 Rust、Solana
CLI 和 Anchor CLI。

本模板需要使用 Anchor CLI 1.1.2 或更高版本。检查已安装的版本：

```terminal
$ anchor --version
```

<Steps>

<Step>

### 创建项目

在终端中运行以下命令：

```terminal
$ anchor init my-program
$ cd my-program
```

入门项目在 `programs/my-program`
下包含一个 Solana 程序。该程序包含两条指令：一条用于初始化计数器账户，另一条用于递增计数器。

模板的某些部分展示了常见的 Solana 程序模式：派生 [PDA](/docs/core/pda)
账户地址、通过 Cross Program
Invocation（[CPI](/docs/core/cpi)）转移 SOL，以及使用自定义错误检查在条件不满足时终止指令。

### 构建程序

运行 `anchor build` 以编译入门程序：

```terminal
$ anchor build
```

编译后的程序将写入 `target/deploy/my_program.so`。程序部署后，该 `.so`
文件的内容将存储在链上账户中。

</Step>

<Step>

### 运行测试

运行默认测试：

```terminal
$ anchor test
```

此模板的 `Anchor.toml` 使用 Rust 测试命令：

```toml title="Anchor.toml"
skip_local_validator = true

[scripts]
test = "cargo test"
```

该测试将已编译的程序加载到 [LiteSVM](https://www.litesvm.com/)
中，创建一个付款方，发送 _rs`initialize`_ 和 _rs`increment`_
指令，然后检查计数器账户状态。

运行 `anchor test` 也会编译程序，因此在本地测试时无需先运行 `anchor build`。

</Step>

<Step>

### 部署程序

本地测试是最快速的反馈方式。当您准备好部署到网络（例如 devnet）时，请先构建，再部署到集群。

部署 Solana 程序需要 SOL，因为程序存储在账户中，而账户必须为其占用的空间支付费用。在 devnet 上，可通过
[Solana Faucet](https://faucet.solana.com/) 或 Solana CLI 免费获取 devnet SOL：

```terminal
$ solana airdrop 2 --url devnet
```

```terminal
$ anchor build
$ anchor deploy --provider.cluster devnet
```

</Step>

## 源文件

`src` 目录包含 Solana 程序。Anchor 的
[程序结构](https://www.anchor-lang.com/docs/basics/program-structure)
文档说明了此处使用的核心宏，包括 _rs`declare_id!`_、
_rs`#[program]`_、_rs`#[derive(Accounts)]`_ 和
_rs`#[account]`_。本节将逐步介绍模板文件。

### lib.rs

`lib.rs`
是程序入口点。它连接各源文件，定义程序地址，并定义用户可调用的程序指令。

<WithNotes>

```rust title="programs/my-program/src/lib.rs" -w
// !tooltip[/pub mod constants/] constantsMod
pub mod constants;
// !tooltip[/pub mod error/] errorMod
pub mod error;
// !tooltip[/pub mod instructions/] instructionsMod
pub mod instructions;
// !tooltip[/pub mod state/] stateMod
pub mod state;

// !tooltip[/anchor_lang::prelude/] prelude
use anchor_lang::prelude::*;

pub use constants::*;
pub use instructions::*;
pub use state::*;

// !tooltip[/declare_id!/] declareId
declare_id!("82sFkffP9wxwpyfZyeaKHH2chQoJPUGsJZSPi9mrUuXd");

// !tooltip[/#\[program\]/] programMacro
#[program]
pub mod my_program {
    use super::*;

    // !tooltip[/pub fn initialize/] publicInstructions
    pub fn initialize(ctx: Context<Initialize>) -> Result<()> {
        // !tooltip[/handle_initialize/] handlers
        crate::instructions::initialize::handle_initialize(ctx)
    }

    // !tooltip[/pub fn increment/] publicInstructions
    pub fn increment(ctx: Context<Increment>) -> Result<()> {
        // !tooltip[/handle_increment/] handlers
        crate::instructions::increment::handle_increment(ctx)
    }
}
```

### !constantsMod

_rs`pub mod constants`_ 将 `constants.rs`
纳入程序 crate。这里存放模板中共享的程序值，例如 PDA
seed、转账金额和计数器最大值。

### !errorMod

_rs`pub mod error`_ 将 `error.rs`
纳入程序 crate。该文件定义了程序在指令无效时可返回的自定义错误。

### !instructionsMod

_rs`pub mod instructions`_ 将 `instructions.rs` 纳入程序 crate。该文件充当
`src/instructions/` 中各指令专属文件的索引。

### !stateMod

_rs`pub mod state`_ 将 `state.rs`
纳入程序 crate。该文件定义了此程序所拥有账户的数据布局，包括 _rs`Counter`_
账户。

### !prelude

_rs`anchor_lang::prelude::*`_ 导入用于定义 Solana 程序的 Anchor 类型和宏，包括
_rs`Context`_、_rs`Result`_、_rs`Pubkey`_、_rs`Account`_、_rs`Signer`_
以及 Anchor 账户属性。

### !reexports

_rs`pub use`_
这几行将子模块中的公开项暴露到 crate 根目录，从而使指令入口点更简洁，例如使用
_rs`Context<Initialize>`_ 而非更长的模块路径。

### !declareId

_rs`declare_id!`_ 定义程序地址。它应与 `Anchor.toml`、已部署的 program
account 以及所有调用该程序的用户或测试中的地址保持一致。

### !programMacro

_rs`#[program]`_
指示 Anchor 生成 Solana 程序入口点和指令分发代码。该模块中的公开函数将成为用户可调用的指令。

### !publicInstructions

_rs`initialize`_ 和 _rs`increment`_ 是用户可调用的两个指令名称。每个指令接收一个
_rs`Context<T>`_。_rs`Context`_
内的类型定义了该指令所需的账户，Anchor 会在指令执行前对这些账户进行验证。

### !handlers

处理函数包含指令的逻辑。

</WithNotes>

### constants.rs

`constants.rs` 将共享值集中在一处。在此模板中， _rs`COUNTER_SEED`_
用于派生计数器 PDA，_rs`HELLO_WORLD_LAMPORTS`_ 在初始化时转入，_rs`MAX_COUNT`_
在递增前进行检查。

<WithNotes>

```rust title="programs/my-program/src/constants.rs" -w
use anchor_lang::prelude::*;

// !tooltip[/#\[constant\]/] anchorConstant
// !tooltip[/COUNTER_SEED/] counterSeed
#[constant]
pub const COUNTER_SEED: &[u8] = b"counter";

// !tooltip[/HELLO_WORLD_LAMPORTS/] helloLamports
#[constant]
pub const HELLO_WORLD_LAMPORTS: u64 = 1;

// !tooltip[/MAX_COUNT/] maxCount
#[constant]
pub const MAX_COUNT: u64 = 10;
```

### !anchorConstant

_rs`#[constant]`_
告诉 Anchor 将该值包含在程序 IDL 中。这使重要的程序常量对生成的客户端和工具可见。

### !counterSeed

_rs`COUNTER_SEED`_
是用于派生计数器 PDA 的字节字符串。程序和测试都使用此 seed，因此无需存储单独的 keypair 即可确定账户地址。此模板使用一个固定的 seed，因此程序只有一个计数器地址。实际程序通常会在 seeds 中包含用户地址或其他标识符，以创建多个关联账户。

### !helloLamports

_rs`HELLO_WORLD_LAMPORTS`_ 是在 _rs`initialize`_
中从付款方转入计数器账户的 lamport 数量。该值有意设置得很小，因为此模板旨在演示对 System
Program 的 CPI 调用。

### !maxCount

_rs`MAX_COUNT`_ 是由 _rs`increment`_
检查的上限值。将其定义在此处，可让程序、测试和生成的 IDL 共享同一个规则名称。

</WithNotes>

<SideBySide>

### !left

### !right

`initialize.rs` 使用两个常量：

- [_rs`COUNTER_SEED`_](mention:counter-seed) 用于推导计数器 PDA 地址。
- [_rs`HELLO_WORLD_LAMPORTS`_](mention:hello-world-lamports)
  设置从付款方转入计数器账户的金额。

</SideBySide>

```rust title="increment.rs" -w
pub fn handle_increment(ctx: Context<Increment>) -> Result<()> {
    // ...

        // !mention max-count

        ErrorCode::CounterOverflow,
```

### !right

`increment.rs` 使用 [_rs`MAX_COUNT`_](mention:max-count)
作为计数器的上限。如果当前计数已达到最大值，_rs`require!`_ 将返回
_rs`CounterOverflow`_，账户数据不会被更改。

### state.rs

`state.rs`
为程序创建并拥有的账户定义自定义数据类型。程序定义了创建、初始化和更新该数据的指令，但计数器数据并不存储在程序本身内，而是存储在一个拥有独立地址的单独账户中。

<WithNotes>

```rust title="programs/my-program/src/state.rs" -w
use anchor_lang::prelude::*;

// !tooltip[/#\[account\]/] accountMacro
#[account]
// !tooltip[/InitSpace/] initSpace
#[derive(InitSpace)]
pub struct Counter {
    // !tooltip[/count/] count
    pub count: u64,
    // !tooltip[/authority/] authority
    pub authority: Pubkey,
}
```

### !accountMacro

_rs`#[account]`_
告知 Anchor 此结构体是可存储在该程序所拥有的链上账户中的数据。Anchor 使用它来序列化、反序列化和验证
_rs`Counter`_ 账户。

### !initSpace

_rs`InitSpace`_ 让 Anchor 计算
_rs`Counter::INIT_SPACE`_，即此结构体字段所需的字节数。创建账户时，8 字节的 Anchor 账户鉴别器将单独添加。

### !count

_rs`count`_ 是存储在账户数据中的计数器值。

### !authority

_rs`authority`_ 存储允许递增计数器的地址。_rs`initialize`_
写入该地址，_rs`increment`_ 则要求该地址进行签名。

</WithNotes>

<SideBySide>

### !left

```rust title="initialize.rs" -w
#[account(
    init,
    payer = payer,
    // !mark
    // !mention counter-space
    space = 8 + Counter::INIT_SPACE,
    seeds = [COUNTER_SEED],
    bump
)]
pub counter: Account<'info, Counter>,
// ...

pub fn handle_initialize(ctx: Context<Initialize>) -> Result<()> {
    // !mark(1:2)
    // !mention(1:2) initialize-fields

    ctx.accounts.counter.authority = ctx.accounts.payer.key();

```

```rust title="increment.rs" -w
pub fn handle_increment(ctx: Context<Increment>) -> Result<()> {
    // ...

    // !mark
    // !mention increment-count
    ctx.accounts.counter.count += 1;
    Ok(())
}
```

### !right

`state.rs` 定义了 _rs`Counter`_ 账户数据。指令文件在创建和更新账户时使用该类型：

- [_rs`Counter::INIT_SPACE`_](mention:counter-space) 根据 `state.rs`
  中定义的字段为账户分配空间。
- [_rs`count`_ 和 _rs`authority`_](mention:initialize-fields)
  是账户初始化时写入的字段值。
- [_rs`count += 1`_](mention:increment-count) 在验证通过后更新存储的计数器值。

</SideBySide>

### error.rs

`error.rs`
定义了程序的自定义错误。在此模板中，这些错误展示了当调用方无权更新计数器，或计数器已达到
_rs`MAX_COUNT`_ 时，指令处理程序如何停止执行。

<WithNotes>

```rust title="programs/my-program/src/error.rs" -w
use anchor_lang::prelude::*;

// !tooltip[/#\[error_code\]/] errorCode
#[error_code]
pub enum ErrorCode {
    // !tooltip[/Unauthorized/] unauthorized
    #[msg("Only the counter authority can update this counter")]
    Unauthorized,
    // !tooltip[/CounterOverflow/] counterOverflow
    #[msg("Counter has reached the maximum value")]
    CounterOverflow,
}
```

### !errorCode

_rs`#[error_code]`_
告知 Anchor 将这些变体转换为自定义程序错误。客户端和测试可以使用这些错误，而无需从原始错误码中猜测。

### !unauthorized

当交易签名者不是计数器账户中存储的授权方时，将返回 _rs`Unauthorized`_。

### !counterOverflow

当递增操作将违反程序的 _rs`MAX_COUNT`_ 规则时，将返回 _rs`CounterOverflow`_。

</WithNotes>

<SideBySide>

### !left

```rust title="increment.rs" -w
pub fn handle_increment(ctx: Context<Increment>) -> Result<()> {
    require_keys_eq!(
        ctx.accounts.counter.authority,
        ctx.accounts.authority.key(),
        // !mark
        // !mention unauthorized-error
        ErrorCode::Unauthorized,
    );
    require!(
        ctx.accounts.counter.count < MAX_COUNT,
        // !mark
        // !mention overflow-error
        ErrorCode::CounterOverflow,
    );

    ctx.accounts.counter.count += 1;
    msg!("Hello, world! Counter is now {}", ctx.accounts.counter.count);

}
```

### !right

`error.rs` 列出了指令可能返回的错误：

- 当签名者不是计数器账户中存储的授权方时，将返回
  [_rs`ErrorCode::Unauthorized`_](mention:unauthorized-error)。
- 当计数器已达到 _rs`MAX_COUNT`_ 时，将返回
  [_rs`ErrorCode::CounterOverflow`_](mention:overflow-error)。

</SideBySide>

### initialize.rs

`initialize.rs`
定义创建计数器账户所需的账户，并写入账户的初始值。_rs`#[derive(Accounts)]`_
结构体使用 Anchor
[账户约束](https://www.anchor-lang.com/docs/references/account-constraints)
来指定所需账户以及新计数器账户的创建方式。

<WithNotes>

```rust title="programs/my-program/src/instructions/initialize.rs" -w
use anchor_lang::prelude::*;

use crate::{constants::*, state::Counter};

// !tooltip[/derive\(Accounts\)/] accountsStruct
#[derive(Accounts)]
pub struct Initialize<'info> {
    // !tooltip[/#\[account\(mut\)\]/] payerMutable
    #[account(mut)]
    // !tooltip[/Signer<'info>/] payerSigner
    pub payer: Signer<'info>,
    // !tooltip[/#\[account\(/] counterConstraints
    #[account(
        // !tooltip[/init/] initConstraint
        init,
        // !tooltip[/payer = payer/] payerConstraint
        payer = payer,
        // !tooltip[/space = 8 \+ Counter::INIT_SPACE/] space
        space = 8 + Counter::INIT_SPACE,
        // !tooltip[/seeds = \[COUNTER_SEED\]/] seeds
        seeds = [COUNTER_SEED],
        // !tooltip[/bump/] bump
        bump
    )]
    // !tooltip[/Account<'info, Counter>/] counterAccountType
    pub counter: Account<'info, Counter>,
    // !tooltip[/Program<'info, System>/] systemProgram
    pub system_program: Program<'info, System>,
}

// !tooltip[/handle_initialize/] handler
pub fn handle_initialize(ctx: Context<Initialize>) -> Result<()> {
    // !tooltip[/counter.count = 0/] initialCount
    ctx.accounts.counter.count = 0;
    // !tooltip[/counter.authority = ctx.accounts.payer.key\(\)/] initialAuthority
    ctx.accounts.counter.authority = ctx.accounts.payer.key();

    // !tooltip[/system_program::Transfer/] cpiAccounts
    let cpi_accounts = anchor_lang::system_program::Transfer {
        from: ctx.accounts.payer.to_account_info(),
        to: ctx.accounts.counter.to_account_info(),
    };
    // !tooltip[/CpiContext::new/] cpiContext
    let cpi_ctx = CpiContext::new(anchor_lang::system_program::ID, cpi_accounts);
    // !tooltip[/system_program::transfer/] transfer
    anchor_lang::system_program::transfer(cpi_ctx, HELLO_WORLD_LAMPORTS)?;

    // !tooltip[/msg!/] programLog
    msg!("Hello, world! Counter initialized");
    Ok(())
}
```

## !!steps 计数器账户

_rs`counter`_ 账户存储来自 `state.rs` 的 _rs`Counter`_ 数据。 _rs`init`_
告知 Anchor 在处理函数运行前创建该账户，_rs`payer = payer`_
告知 Anchor 由哪个账户支付创建费用。

```rust !! title="initialize.rs"

        payer = payer,
        space = 8 + Counter::INIT_SPACE,
        seeds = [COUNTER_SEED],

    pub counter: Account<'info, Counter>,
    pub system_program: Program<'info, System>,
```

## !!steps 账户大小

_rs`space`_
约束告诉 Anchor 需要分配多少账户数据。Anchor 首先存储一个 8 字节的鉴别器，然后存储
_rs`Counter`_
字段所需的字节数。鉴别器让 Anchor 在反序列化账户数据之前，将此账户识别为

### !imports

此文件导入共享常量和 _rs`Counter`_
账户类型。该指令使用这些定义来推导 PDA、选择转账金额，并以正确的数据大小分配账户。

### !accountsStruct

_rs`Initialize`_ 是 _rs`initialize`_
指令的账户上下文。结构体中的每个字段代表调用该指令时必须包含的一个独立账户。Anchor 在处理程序运行之前使用该结构体对传入的账户进行验证。

### !payerMutable

_rs`#[account(mut)]`_
告知 Anchor 付款账户必须是可写的。这是必须的，因为账户创建和转账 CPI 都会从付款账户中扣除 lamport。

### !payerSigner

_rs`Signer<'info>`_ 告知 Anchor 付款方必须对交易进行签名。

### !counterConstraints

_rs`#[account(...)]`_ 块告知 Anchor 如何在处理程序运行之前创建并验证计数器账户。

### !initConstraint

_rs`init`_ 指示 Anchor 在处理程序运行之前创建 _rs`counter`_
账户。新账户由本程序拥有，因此本程序可以在其中存储 _rs`Counter`_ 数据。

### !payerConstraint

_rs`payer = payer`_ 指定 _rs`payer`_ 账户作为创建新计数器账户的付款方。

### !space

_rs`space = 8 + Counter::INIT_SPACE`_ 为 Anchor 的账户判别器以及来自 `state.rs`
的 _rs`Counter`_
字段分配字节空间。判别器帮助 Anchor 在读取账户之前确认其存储的是预期类型。

### !seeds

_rs`seeds = [COUNTER_SEED]`_
表示计数器账户地址必须是由此 seed 和程序 ID 派生出的 PDA。使用单一固定 seed 时，该程序只有一个计数器 PDA。若要在实际程序中创建独立的计数器，通常需要添加另一个 seed，例如用户地址。

### !bump

_rs`bump`_
指示 Anchor 查找并验证 PDA 的 bump 值，即使派生地址作为有效 PDA 所需的额外字节。

### !counterAccountType

_rs`Account<'info, Counter>`_ 指示 Anchor 将账户数据视为来自 `state.rs` 的
_rs`Counter`_ 类型。Anchor 同时还会验证该账户归本程序所有。

### !systemProgram

_rs`Program<'info, System>`_ 要求调用方提供 System Program。Anchor 将其用于
_rs`init`_ 账户创建约束。处理程序中的转账 CPI 也会调用 System Program。

### !handler

处理程序在 Anchor 验证账户并创建计数器账户之后运行，因此可以安全地写入账户的初始数据。

### !initialCount

将计数器账户中的 _rs`count`_ 字段设置为 _rs`0`_。

### !initialAuthority

将计数器账户中的 _rs`authority`_ 字段设置为付款方的地址。

### !cpiAccounts

这些是传递给 System Program 转账指令的账户。此转账 Cross Program
Invocation 仅用于演示 CPI 如何将账户传递给另一个程序。

### !cpiContext

_rs`CpiContext`_ 将目标程序和所需账户打包，用于 Cross Program

### !incrementCounterAccountType

_rs`Account<'info, Counter>`_ 告诉 Anchor 将账户数据反序列化为 `state.rs` 中的
_rs`Counter`_ 类型。Anchor 还会检查该账户是否归此程序所有。

### !authoritySigner

_rs`Signer`_
要求权限方提供签名。处理程序随后将此签名者的地址与计数器账户中存储的权限地址进行比对。

### !incrementHandler

处理程序在 Anchor 验证账户列表、检查所有权并反序列化计数器账户后运行。

### !authorityCheck

_rs`require_keys_eq!`_
比较两个公钥，若签名者地址与计数器账户中存储的权限地址不匹配，则终止指令。这是该模板的应用层授权规则。

### !maxCheck

_rs`require!`_ 检查一个布尔规则，若计数器已达到 _rs`MAX_COUNT`_，则以
_rs`CounterOverflow`_
终止指令。该模板规则演示了自定义错误及在修改数据前提前失败的机制。

### !mutateCount

只有在两项检查均通过后，处理程序才会更新计数器账户数据。若任意一项检查失败，则交易失败，账户不会被更改。

### !programLog

_rs`msg!`_ 在程序被调用时向程序日志写入一条消息。

</WithNotes>

```rust title="programs/my-program/tests/test_initialize.rs" -w
use {
    anchor_lang::{
        prelude::Pubkey,
        solana_program::{instruction::Instruction, system_program},
        AccountDeserialize, InstructionData, ToAccountMetas,
    },
    // !tooltip[/litesvm::LiteSVM/] litesvm
    litesvm::LiteSVM,
    solana_keypair::Keypair,
    solana_message::{Message, VersionedMessage},
    solana_signer::Signer,
    solana_transaction::versioned::VersionedTransaction,
};

#[test]
fn test_initialize() {
    // !tooltip[/let program_id/] setup
    let program_id = my_program::id();
    let payer = Keypair::new();
    // !tooltip[/Pubkey::find_program_address/] counterPda
    let counter = Pubkey::find_program_address(
        &[my_program::constants::COUNTER_SEED],
        &program_id,
    )
    .0;
    let mut svm = LiteSVM::new();
    // !tooltip[/include_bytes!/] programBytes
    let bytes = include_bytes!(concat!(
        env!("CARGO_TARGET_TMPDIR"),
        "/../deploy/my_program.so"
    ));
    // !tooltip[/svm.add_program/] addProgram
    svm.add_program(program_id, bytes).unwrap();
    svm.airdrop(&payer.pubkey(), 1_000_000_000).unwrap();

    let instruction = Instruction::new_with_bytes(
        program_id,
        &my_program::instruction::Initialize {}.data(),
        my_program::accounts::Initialize {
            payer: payer.pubkey(),
            counter,
            system_program: system_program::ID,
        }
        .to_account_metas(None),
    );

    let blockhash = svm.latest_blockhash();
    // !tooltip[/Message::new_with_blockhash/] message
    let msg = Message::new_with_blockhash(&[instruction], Some(&payer.pubkey()), &blockhash);
    // !tooltip[/VersionedTransaction::try_new/] transaction
    let tx = VersionedTransaction::try_new(VersionedMessage::Legacy(msg), &[&payer]).unwrap();

    // !tooltip[/svm.send_transaction/] sendTx
    let res = svm.send_transaction(tx);
    assert!(res.is_ok());

    // !tooltip[/svm.get_account/] getCounter
    let counter_account = svm.get_account(&counter).unwrap();
    let mut data: &[u8] = &counter_account.data;
    // !tooltip[/Counter::try_deserialize/] deserialize
    let counter_state = my_program::state::Counter::try_deserialize(&mut data).unwrap();
    // !tooltip[/assert_eq!\(counter_state.count, 0\)/] initAssert
    assert_eq!(counter_state.count, 0);
    assert_eq!(counter_state.authority, payer.pubkey());

    let instruction = Instruction::new_with_bytes(
```

### !litesvm

_rs`LiteSVM`_
是一个用于测试的进程内 Solana 虚拟机。它允许测试注册已编译的程序、创建账户、发送交易并检查账户状态，而无需启动 validator 进程。

### !setup

测试通过 _rs`my_program::id()`_ 从 _rs`declare_id!`_
值中获取程序地址，并为测试交易创建一个新的 keypair 付款方。

### !counterPda

计数器地址由程序使用的同一 _rs`COUNTER_SEED`_ 派生而来。这使测试能够针对
`initialize.rs` 所期望的 PDA。

### !programBytes

_rs`include_bytes!`_ 读取构建生成的已编译 `my_program.so`
文件。这些字节是 LiteSVM 将在测试中运行的可执行程序代码。

### !addProgram

_rs`svm.add_program`_
将已编译的程序字节注册到 LiteSVM 中的程序地址。空投为付款方提供足够的本地 SOL 以支付交易和账户创建费用。

### !testInstruction

_rs`Instruction::new_with_bytes`_
创建一条调用程序的指令。它包含程序 ID、instruction data 和指令账户。

### !blockhash

每笔 Solana 交易都需要一个最新的区块哈希。在此测试中，LiteSVM 提供一个用于交易的区块哈希。

### !message

消息包含指令、手续费付款方和区块哈希。这是将被签名的交易数据。

### !transaction

_rs`VersionedTransaction::try_new`_
使用付款方 keypair 对消息进行签名。付款方签名是因为它支付交易手续费并为新计数器账户提供资金。

### !sendTx

_rs`svm.send_transaction`_ 在 LiteSVM 中执行交易。

### !getCounter

交易执行后，测试直接从 LiteSVM 读取计数器账户，以便检查账户数据。

### !deserialize

计数器账户存储序列化数据。_rs`Counter::try_deserialize`_
验证 Anchor 的账户鉴别器，并将账户数据还原为来自 `state.rs` 的 _rs`Counter`_
类型。

### !initAssert

_rs`initialize`_ 成功后，测试对计数器账户进行反序列化，并验证初始状态与
_rs`handle_initialize`_ 写入的内容一致。

### !incrementAssert

_rs`increment`_ 成功后，同一计数器账户的 _rs`count`_ 应设置为
_rs`1`_，同时保持相同的权限。

</WithNotes>
