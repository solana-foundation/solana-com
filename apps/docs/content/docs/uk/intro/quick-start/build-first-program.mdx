h1: Створіть свою першу програму Solana

Цей швидкий старт використовує стартовий проєкт
[фреймворку Anchor](https://www.anchor-lang.com/docs), згенерований командою
`anchor init`. Ви створите проєкт локально, запустите його тести, зберете
програму та ознайомитеся з кодом програми, який генерує Anchor.

## Передумови

Перш ніж почати,
[встановіть інструменти розробки Solana](/docs/intro/installation). До
встановлення входять Rust, Solana CLI та Anchor CLI.

Для цього шаблону використовуйте Anchor CLI версії 1.1.2 або вище. Перевірте
встановлену версію:

```terminal
$ anchor --version
```

<Steps>

<Step>

### Створення проєкту

Виконайте наступні команди у вашому терміналі:

```terminal
$ anchor init my-program
$ cd my-program
```

Стартовий проєкт містить одну програму Solana у `programs/my-program`. Програма
включає дві інструкції: одну для ініціалізації облікового запису лічильника та
одну для збільшення лічильника.

Деякі частини шаблону демонструють типові патерни програм Solana: виведення
адрес облікових записів [PDA](/docs/core/pda), виконання Cross Program
Invocation ([CPI](/docs/core/cpi)) для переказу SOL, а також використання
власних перевірок помилок для зупинки інструкції у разі невиконання умови.

<Files>
  <Folder name="my-program" defaultOpen>
    <File name="Anchor.toml" />
    <File name="Cargo.toml" />
    <Folder name="programs" defaultOpen>
      <Folder name="my-program" defaultOpen>

        </Folder>
        <Folder name="tests" defaultOpen>
          <File name="test_initialize.rs" />
        </Folder>
      </Folder>
    </Folder>
  </Folder>
</Files>

</Step>

<Step>

### Збірка програми

Запустіть `anchor build` для компіляції стартової програми:

```terminal
$ anchor build
```

Скомпільовану програму записано до `target/deploy/my_program.so`. Під час
розгортання програми вміст цього файлу `.so` зберігається в обліковому записі в
мережі.

</Step>

<Step>

### Запустити тест

Запустіть стандартний тест:

```terminal
$ anchor test
```

`Anchor.toml` цього шаблону використовує команду тестування Rust:

```toml title="Anchor.toml"
skip_local_validator = true

[scripts]
test = "cargo test"
```

Тест завантажує скомпільовану програму в [LiteSVM](https://www.litesvm.com/),
створює платника, надсилає інструкції _rs`initialize`_ та _rs`increment`_, а
потім перевіряє стан рахунку лічильника.

Запуск `anchor test` також компілює програму, тому вам не потрібно запускати
`anchor build` спочатку під час локального тестування.

</Step>

<Step>

### Розгортання програми

Локальні тести — це найшвидший цикл зворотного зв'язку. Коли ви готові до
розгортання в мережі, наприклад devnet, спочатку виконайте збірку, а потім
розгорніть у кластері.

Розгортання програми Solana вимагає SOL, оскільки програма зберігається в
акаунті, і акаунт повинен оплачувати використовуваний простір. У devnet
запросіть безкоштовний devnet SOL із [Solana Faucet](https://faucet.solana.com/)
або за допомогою Solana CLI:

```terminal
$ solana airdrop 2 --url devnet
```

```terminal
$ anchor build
$ anchor deploy --provider.cluster devnet
```

</Step>

</Steps>

## Вихідні файли

Директорія `src` містить програму Solana. Документація Anchor щодо
[структури програми](https://www.anchor-lang.com/docs/basics/program-structure)
пояснює основні макроси, що використовуються тут, зокрема _rs`declare_id!`_,
_rs`#[program]`_, _rs`#[derive(Accounts)]`_ та _rs`#[account]`_. У цьому розділі
розглядаються файли шаблону.

### lib.rs

`lib.rs` — це точка входу програми. Вона з'єднує вихідні файли, визначає адресу
програми та визначає інструкції програми, які можуть викликати користувачі.

<WithNotes>

```rust title="programs/my-program/src/lib.rs" -w
// !tooltip[/pub mod constants/] constantsMod
pub mod constants;
// !tooltip[/pub mod error/] errorMod
pub mod error;
// !tooltip[/pub mod instructions/] instructionsMod
pub mod instructions;
// !tooltip[/pub mod state/] stateMod
pub mod state;

// !tooltip[/anchor_lang::prelude/] prelude
use anchor_lang::prelude::*;

pub use constants::*;
pub use instructions::*;
pub use state::*;

// !tooltip[/declare_id!/] declareId
declare_id!("82sFkffP9wxwpyfZyeaKHH2chQoJPUGsJZSPi9mrUuXd");

// !tooltip[/#\[program\]/] programMacro
#[program]
pub mod my_program {
    use super::*;

    // !tooltip[/pub fn initialize/] publicInstructions
    pub fn initialize(ctx: Context<Initialize>) -> Result<()> {
        // !tooltip[/handle_initialize/] handlers
        crate::instructions::initialize::handle_initialize(ctx)
    }

    // !tooltip[/pub fn increment/] publicInstructions
    pub fn increment(ctx: Context<Increment>) -> Result<()> {
        // !tooltip[/handle_increment/] handlers
        crate::instructions::increment::handle_increment(ctx)
    }
}
```

### !constantsMod

_rs`pub mod constants`_ додає `constants.rs` до крейта програми. Тут шаблон
зберігає спільні значення програми, такі як seed PDA, сума переказу та
максимальне значення лічильника.

### !errorMod

_rs`pub mod error`_ додає `error.rs` до крейта програми. Цей файл визначає
користувацькі помилки, які програма може повертати, коли інструкція є недійсною.

### !instructionsMod

_rs`pub mod instructions`_ додає `instructions.rs` до крейта програми. Цей файл
працює як індекс для файлів, що стосуються конкретних інструкцій, у
`src/instructions/`.

### !stateMod

_rs`pub mod state`_ додає `state.rs` до крейта програми. Цей файл визначає
структуру даних для акаунтів, якими володіє ця програма, включаючи акаунт
_rs`Counter`_.

### !prelude

_rs`anchor_lang::prelude::*`_ імпортує типи та макроси Anchor, що
використовуються для визначення програми Solana, включаючи _rs`Context`_,
_rs`Result`_, _rs`Pubkey`_, _rs`Account`_, _rs`Signer`_ та атрибути акаунтів
Anchor.

### !reexports

Рядки _rs`pub use`_ роблять публічні елементи дочірніх модулів доступними з
кореня крейта. Це дозволяє скорочувати точки входу інструкцій, наприклад
_rs`Context<Initialize>`_ замість довшого шляху до модуля.

### !declareId

_rs`declare_id!`_ визначає адресу програми. Вона має збігатися з адресою у
`Anchor.toml`, розгорнутому program account та у будь-яких користувачів або
тестів, що викликають програму.

### !programMacro

_rs`#[program]`_ вказує Anchor згенерувати точку входу програми Solana та код
диспетчеризації інструкцій. Публічні функції у цьому модулі стають інструкціями,
які можуть викликати користувачі.

### !publicInstructions

_rs`initialize`_ та _rs`increment`_ — це дві назви інструкцій, які можуть
викликати користувачі. Кожна отримує _rs`Context<T>`_. Тип у _rs`Context`_
визначає акаунти, необхідні для цієї інструкції, і Anchor перевіряє ці акаунти
перед виконанням інструкції.

### !handlers

Функція-обробник містить логіку для інструкції.

</WithNotes>

<SideBySide>

### !left

```toml title="Anchor.toml" -w
[programs.localnet]
# !mark
my_program = "82sFkffP9wxwpyfZyeaKHH2chQoJPUGsJZSPi9mrUuXd"
```

### !right

```rust title="lib.rs" -w
// !mark
declare_id!("82sFkffP9wxwpyfZyeaKHH2chQoJPUGsJZSPi9mrUuXd");
```

</SideBySide>

Та сама адреса програми з'являється в конфігурації та коді. `Anchor.toml`
повідомляє Anchor, яку адресу використовувати для розгортання або виклику в
кластері. _rs`declare_id!`_ визначає адресу програми в самій програмі для
перевірок безпеки.

### constants.rs

`constants.rs` зберігає спільні значення в одному місці. У цьому шаблоні
_rs`COUNTER_SEED`_ використовується для виведення PDA лічильника,
_rs`HELLO_WORLD_LAMPORTS`_ передається під час ініціалізації, а _rs`MAX_COUNT`_
перевіряється перед інкрементуванням.

<WithNotes>

```rust title="programs/my-program/src/constants.rs" -w
use anchor_lang::prelude::*;

// !tooltip[/#\[constant\]/] anchorConstant
// !tooltip[/COUNTER_SEED/] counterSeed
#[constant]
pub const COUNTER_SEED: &[u8] = b"counter";

// !tooltip[/HELLO_WORLD_LAMPORTS/] helloLamports
#[constant]
pub const HELLO_WORLD_LAMPORTS: u64 = 1;

// !tooltip[/MAX_COUNT/] maxCount
#[constant]
pub const MAX_COUNT: u64 = 10;
```

### !anchorConstant

_rs`#[constant]`_ вказує Anchor включити значення до IDL програми. Це робить
важливі константи програми видимими для згенерованих клієнтів та інструментів.

### !counterSeed

_rs`COUNTER_SEED`_ — це рядок байтів, який використовується для виведення PDA
лічильника. Програма та тест обидва використовують цей seed, тому вони
погоджуються щодо адреси акаунта без збереження окремого keypair. Цей шаблон
використовує один фіксований seed, тому програма має одну адресу лічильника.
Реальні програми часто включають адресу користувача або інший ідентифікатор у
seeds для створення багатьох пов'язаних акаунтів.

### !helloLamports

_rs`HELLO_WORLD_LAMPORTS`_ — це кількість lamport, що передається від платника
до акаунта лічильника всередині _rs`initialize`_. Вона навмисно мала, оскільки
шаблон демонструє CPI до System Program.

### !maxCount

_rs`MAX_COUNT`_ — це верхня межа, яку перевіряє _rs`increment`_. Зберігання її
тут надає програмі, тестам та згенерованому IDL одну спільну назву для цього
правила.

</WithNotes>

<SideBySide>

### !left

```rust title="initialize.rs" -w
#[derive(Accounts)]
pub struct Initialize<'info> {
    // ...
    #[account(
        init,
        payer = payer,
        space = 8 + Counter::INIT_SPACE,
        // !mark
        // !mention counter-seed
        seeds = [COUNTER_SEED],
        bump
    )]
    pub counter: Account<'info, Counter>,
    // ...
}

pub fn handle_initialize(ctx: Context<Initialize>) -> Result<()> {
    // ...

    // !mark
    // !mention hello-world-lamports
    anchor_lang::system_program::transfer(cpi_ctx, HELLO_WORLD_LAMPORTS)?;

    Ok(())
}
```

### !right

`initialize.rs` використовує дві константи:

- [_rs`COUNTER_SEED`_](mention:counter-seed) отримує адресу PDA лічильника.
- [_rs`HELLO_WORLD_LAMPORTS`_](mention:hello-world-lamports) встановлює суму,
  яка переводиться від платника на рахунок лічильника.

</SideBySide>

<SideBySide>

### !left

```rust title="increment.rs" -w
pub fn handle_increment(ctx: Context<Increment>) -> Result<()> {
    // ...

    require!(
        // !mark
        // !mention max-count
        ctx.accounts.counter.count < MAX_COUNT,
        ErrorCode::CounterOverflow,
    );

    ctx.accounts.counter.count += 1;
    Ok(())
}
```

### !right

`increment.rs` використовує [_rs`MAX_COUNT`_](mention:max-count) як верхню межу
лічильника. Якщо поточне значення вже досягло максимуму, _rs`require!`_ повертає
_rs`CounterOverflow`_ і дані акаунту не змінюються.

</SideBySide>

### state.rs

`state.rs` визначає власні типи даних для акаунтів, які програма створює та
якими керує. Програма визначає інструкції для створення, ініціалізації та
оновлення цих даних, але дані лічильника зберігаються не всередині самої
програми. Вони зберігаються в окремому акаунті з власною адресою.

<WithNotes>

```rust title="programs/my-program/src/state.rs" -w
use anchor_lang::prelude::*;

// !tooltip[/#\[account\]/] accountMacro
#[account]
// !tooltip[/InitSpace/] initSpace
#[derive(InitSpace)]
pub struct Counter {
    // !tooltip[/count/] count
    pub count: u64,
    // !tooltip[/authority/] authority
    pub authority: Pubkey,
}
```

### !accountMacro

_rs`#[account]`_ повідомляє Anchor, що ця структура є даними, які можуть
зберігатися в onchain-акаунті, яким керує ця програма. Anchor використовує її
для серіалізації, десеріалізації та перевірки акаунтів _rs`Counter`_.

### !initSpace

_rs`InitSpace`_ дозволяє Anchor обчислити _rs`Counter::INIT_SPACE`_ — кількість
байтів, необхідних для полів цієї структури. 8-байтовий дискримінатор акаунту
Anchor додається окремо під час створення акаунту.

### !count

_rs`count`_ — це значення лічильника, що зберігається всередині даних акаунту.

### !authority

_rs`authority`_ зберігає адресу, якій дозволено збільшувати лічильник.
_rs`initialize`_ записує цю адресу, а _rs`increment`_ вимагає підпису з тієї
самої адреси.

</WithNotes>

<SideBySide>

### !left

```rust title="initialize.rs" -w
#[account(
    init,
    payer = payer,
    // !mark
    // !mention counter-space
    space = 8 + Counter::INIT_SPACE,
    seeds = [COUNTER_SEED],
    bump
)]
pub counter: Account<'info, Counter>,
// ...

pub fn handle_initialize(ctx: Context<Initialize>) -> Result<()> {
    // !mark(1:2)
    // !mention(1:2) initialize-fields
    ctx.accounts.counter.count = 0;
    ctx.accounts.counter.authority = ctx.accounts.payer.key();

    Ok(())
}
```

```rust title="increment.rs" -w
pub fn handle_increment(ctx: Context<Increment>) -> Result<()> {
    // ...

    // !mark
    // !mention increment-count
    ctx.accounts.counter.count += 1;
    Ok(())
}
```

### !right

`state.rs` визначає дані акаунта _rs`Counter`_. Файли інструкцій використовують
цей тип під час створення та оновлення акаунта:

- [_rs`Counter::INIT_SPACE`_](mention:counter-space) визначає розмір акаунта для
  полів, описаних у `state.rs`.
- [_rs`count`_ та _rs`authority`_](mention:initialize-fields) — це значення
  полів, що записуються під час ініціалізації акаунта.
- [_rs`count += 1`_](mention:increment-count) оновлює збережене значення
  лічильника після успішної перевірки.

</SideBySide>

### error.rs

`error.rs` визначає власні помилки програми. У цьому шаблоні помилки
демонструють, як обробники інструкцій зупиняються, якщо виклик не має права
оновлювати лічильник або лічильник вже досяг _rs`MAX_COUNT`_.

<WithNotes>

```rust title="programs/my-program/src/error.rs" -w
use anchor_lang::prelude::*;

// !tooltip[/#\[error_code\]/] errorCode
#[error_code]
pub enum ErrorCode {
    // !tooltip[/Unauthorized/] unauthorized
    #[msg("Only the counter authority can update this counter")]
    Unauthorized,
    // !tooltip[/CounterOverflow/] counterOverflow
    #[msg("Counter has reached the maximum value")]
    CounterOverflow,
}
```

### !errorCode

_rs`#[error_code]`_ вказує Anchor перетворити ці варіанти на власні помилки
програми. Клієнти та тести можуть використовувати ці помилки замість того, щоб
здогадуватися за необробленими кодами помилок.

### !unauthorized

_rs`Unauthorized`_ повертається, коли підписант транзакції не є authority,
збереженим в акаунті лічильника.

### !counterOverflow

_rs`CounterOverflow`_ повертається, коли інкремент порушить правило
_rs`MAX_COUNT`_ програми.

</WithNotes>

<SideBySide>

### !left

```rust title="increment.rs" -w
pub fn handle_increment(ctx: Context<Increment>) -> Result<()> {
    require_keys_eq!(
        ctx.accounts.counter.authority,
        ctx.accounts.authority.key(),
        // !mark
        // !mention unauthorized-error
        ErrorCode::Unauthorized,
    );
    require!(
        ctx.accounts.counter.count < MAX_COUNT,
        // !mark
        // !mention overflow-error
        ErrorCode::CounterOverflow,
    );

    ctx.accounts.counter.count += 1;
    msg!("Hello, world! Counter is now {}", ctx.accounts.counter.count);
    Ok(())
}
```

### !right

`error.rs` називає помилки, які може повернути інструкція:

- [_rs`ErrorCode::Unauthorized`_](mention:unauthorized-error) повертається, коли
  підписант не є authority, збереженим в акаунті лічильника.
- [_rs`ErrorCode::CounterOverflow`_](mention:overflow-error) повертається, коли
  лічильник вже досяг _rs`MAX_COUNT`_.

</SideBySide>

### instructions.rs

`instructions.rs` підключає файли інструкцій до крейту програми, щоб `lib.rs`
міг отримати доступ до коду інструкцій _rs`initialize`_ та _rs`increment`_.
Кожен файл інструкції визначає облікові записи, необхідні для цієї інструкції,
та логіку обробника, яка виконується після того, як Anchor перевірить ці
облікові записи.

<WithNotes>

```rust title="programs/my-program/src/instructions.rs" -w
// !tooltip[/pub mod initialize/] moduleDeclarations
pub mod initialize;
pub mod increment;

// !tooltip[/pub use initialize/] reexports
pub use initialize::*;
pub use increment::*;
```

### !moduleDeclarations

Рядки _rs`pub mod`_ роблять `src/instructions/initialize.rs` та
`src/instructions/increment.rs` частиною крейту програми.

### !reexports

Рядки _rs`pub use`_ повторно експортують публічні елементи з цих файлів. Це
робить структури облікових записів інструкцій та функції обробників доступними
через модуль _rs`instructions`_.

</WithNotes>

### initialize.rs

`initialize.rs` визначає облікові записи, необхідні для створення облікового
запису лічильника, а потім записує його початкові значення. Структура
_rs`#[derive(Accounts)]`_ використовує
[обмеження облікових записів](https://www.anchor-lang.com/docs/references/account-constraints)
Anchor, щоб вказати, які облікові записи є обов'язковими та як створюється новий
обліковий запис лічильника.

<WithNotes>

```rust title="programs/my-program/src/instructions/initialize.rs" -w
use anchor_lang::prelude::*;

use crate::{constants::*, state::Counter};

// !tooltip[/derive\(Accounts\)/] accountsStruct
#[derive(Accounts)]
pub struct Initialize<'info> {
    // !tooltip[/#\[account\(mut\)\]/] payerMutable
    #[account(mut)]
    // !tooltip[/Signer<'info>/] payerSigner
    pub payer: Signer<'info>,
    // !tooltip[/#\[account\(/] counterConstraints
    #[account(
        // !tooltip[/init/] initConstraint
        init,
        // !tooltip[/payer = payer/] payerConstraint
        payer = payer,
        // !tooltip[/space = 8 \+ Counter::INIT_SPACE/] space
        space = 8 + Counter::INIT_SPACE,
        // !tooltip[/seeds = \[COUNTER_SEED\]/] seeds
        seeds = [COUNTER_SEED],
        // !tooltip[/bump/] bump
        bump
    )]
    // !tooltip[/Account<'info, Counter>/] counterAccountType
    pub counter: Account<'info, Counter>,
    // !tooltip[/Program<'info, System>/] systemProgram
    pub system_program: Program<'info, System>,
}

// !tooltip[/handle_initialize/] handler
pub fn handle_initialize(ctx: Context<Initialize>) -> Result<()> {
    // !tooltip[/counter.count = 0/] initialCount
    ctx.accounts.counter.count = 0;
    // !tooltip[/counter.authority = ctx.accounts.payer.key\(\)/] initialAuthority
    ctx.accounts.counter.authority = ctx.accounts.payer.key();

    // !tooltip[/system_program::Transfer/] cpiAccounts
    let cpi_accounts = anchor_lang::system_program::Transfer {
        from: ctx.accounts.payer.to_account_info(),
        to: ctx.accounts.counter.to_account_info(),
    };
    // !tooltip[/CpiContext::new/] cpiContext
    let cpi_ctx = CpiContext::new(anchor_lang::system_program::ID, cpi_accounts);
    // !tooltip[/system_program::transfer/] transfer
    anchor_lang::system_program::transfer(cpi_ctx, HELLO_WORLD_LAMPORTS)?;

    // !tooltip[/msg!/] programLog
    msg!("Hello, world! Counter initialized");
    Ok(())
}
```

### !imports

Цей файл імпортує спільні константи та тип акаунту _rs`Counter`_. Інструкція
використовує ці визначення для виведення PDA, вибору суми переказу та виділення
акаунту з правильним розміром даних.

### !accountsStruct

_rs`Initialize`_ — це контекст акаунтів для інструкції _rs`initialize`_. Кожне
поле у структурі представляє окремий акаунт, який має бути включений під час
виклику інструкції. Anchor використовує структуру для перевірки вхідних акаунтів
перед запуском обробника.

### !payerMutable

_rs`#[account(mut)]`_ повідомляє Anchor, що акаунт платника повинен бути
доступний для запису. Це необхідно, оскільки створення акаунту та CPI переказу
обидва знімають lamport з платника.

### !payerSigner

_rs`Signer<'info>`_ повідомляє Anchor, що платник повинен підписати транзакцію.

### !counterConstraints

Блок _rs`#[account(...)]`_ повідомляє Anchor, як створити та перевірити акаунт
лічильника перед запуском обробника.

### !initConstraint

_rs`init`_ вказує Anchor створити акаунт _rs`counter`_ до виконання обробника.
Новий акаунт належить цій програмі, тому вона може зберігати в ньому дані
_rs`Counter`_.

### !payerConstraint

_rs`payer = payer`_ вказує акаунт _rs`payer`_ як платника за створення нового
акаунта лічильника.

### !space

_rs`space = 8 + Counter::INIT_SPACE`_ виділяє байти для дискримінатора акаунта
Anchor та полів _rs`Counter`_ з `state.rs`. Дискримінатор допомагає Anchor
підтвердити, що акаунт зберігає очікуваний тип перед його зчитуванням.

### !seeds

_rs`seeds = [COUNTER_SEED]`_ вказує, що адреса акаунта лічильника має бути PDA,
похідним від цього seed та ідентифікатора програми. З одним фіксованим seed
існує один PDA лічильника для цієї програми. Щоб створити окремі лічильники в
реальній програмі, зазвичай додають ще один seed, наприклад адресу користувача.

### !bump

_rs`bump`_ вказує Anchor знайти та перевірити bump PDA — додатковий байт, який
робить похідну адресу дійсною як PDA.

### !counterAccountType

_rs`Account<'info, Counter>`_ вказує Anchor інтерпретувати дані акаунта як тип
_rs`Counter`_ з `state.rs`. Anchor також перевіряє, що акаунт належить цій
програмі.

### !systemProgram

_rs`Program<'info, System>`_ вимагає від викликача надати System Program. Anchor
використовує її для обмеження створення акаунта _rs`init`_. CPI переказу в
обробнику також викликає System Program.

### !handler

Обробник виконується після того, як Anchor перевірить акаунти та створить акаунт
лічильника, тому він може безпечно записати початкові дані акаунта.

### !initialCount

Встановлює поле _rs`count`_ в акаунті лічильника на _rs`0`_.

### !initialAuthority

Встановлює поле _rs`authority`_ в акаунті лічильника на адресу платника.

### !cpiAccounts

Це акаунти, передані до інструкції переказу System Program. Цей CPI-переказ
включено лише для демонстрації того, як CPI передає акаунти іншій програмі.

### !cpiContext

_rs`CpiContext`_ пакує цільову програму та необхідні акаунти для Cross Program
Invocation. Створення контексту підготовлює CPI, пов'язуючи програму, яку ви
хочете викликати, з акаунтами, які ця програма очікує.

### !transfer

Цей CPI викликає інструкцію переказу System Program і переміщує
_rs`HELLO_WORLD_LAMPORTS`_ від платника до акаунта лічильника.

### !programLog

_rs`msg!`_ записує повідомлення до журналів програми під час її виклику.

### increment.rs

`increment.rs` визначає акаунти, необхідні для оновлення існуючого акаунта
лічильника. Обробник перевіряє, що підписант є збереженим власником, перевіряє,
що лічильник не досяг зазначеного _rs`MAX_COUNT`_, а потім збільшує лічильник.

<WithNotes>

```rust title="programs/my-program/src/instructions/increment.rs" -w
use anchor_lang::prelude::*;

use crate::{constants::*, error::ErrorCode, state::Counter};

// !tooltip[/derive\(Accounts\)/] incrementAccounts
#[derive(Accounts)]
pub struct Increment<'info> {
    #[account(
        // !tooltip[/mut/] counterMutable
        mut,
        // !tooltip[/seeds = \[COUNTER_SEED\]/] counterSeeds
        seeds = [COUNTER_SEED],
        // !tooltip[/bump/] counterBump
        bump
    )]
    // !tooltip[/Account<'info, Counter>/] incrementCounterAccountType
    pub counter: Account<'info, Counter>,
    // !tooltip[/Signer/] authoritySigner
    pub authority: Signer<'info>,
}

// !tooltip[/handle_increment/] incrementHandler
pub fn handle_increment(ctx: Context<Increment>) -> Result<()> {
    // !tooltip[/require_keys_eq!/] authorityCheck
    require_keys_eq!(
        ctx.accounts.counter.authority,
        ctx.accounts.authority.key(),
        ErrorCode::Unauthorized,
    );
    // !tooltip[/require!/] maxCheck
    require!(
        ctx.accounts.counter.count < MAX_COUNT,
        ErrorCode::CounterOverflow,
    );

    // !tooltip[/count \+= 1/] mutateCount
    ctx.accounts.counter.count += 1;
    // !tooltip[/msg!/] programLog
    msg!("Hello, world! Counter is now {}", ctx.accounts.counter.count);
    Ok(())
```

_rs`mut`_ повідомляє Anchor, що обліковий запис лічильника має бути доступним

_rs`Account<'info, Counter>`_ повідомляє Anchor десеріалізувати дані облікового

_rs`Signer`_ вимагає підпису authority. Потім обробник порівнює адресу цього

### !incrementHandler

Обробник запускається після того, як Anchor перевіряє список облікових записів,

Кореневі файли проєкту вказують Anchor та Cargo, як збирати, тестувати та
розгортати програму. Для повного довідника дивіться документацію Anchor для
[конфігурації Anchor.toml](https://www.anchor-lang.com/docs/references/anchor-toml)
та [Anchor CLI](https://www.anchor-lang.com/docs/references/cli).

<WithNotes>

```toml title="Anchor.toml"
# !tooltip[/skip_local_validator/] skipValidator
skip_local_validator = true

[toolchain]

[features]
resolution = true
skip-lint = false

# !tooltip[/programs.localnet/] programs
[programs.localnet]
my_program = "82sFkffP9wxwpyfZyeaKHH2chQoJPUGsJZSPi9mrUuXd"

# !tooltip[/provider/] provider
[provider]
cluster = "localnet"
wallet = "~/.config/solana/id.json"

# !tooltip[/scripts/] scripts
[scripts]
test = "cargo test"

[hooks]
```

### !skipValidator

`skip_local_validator = true` вказує Anchor не запускати validator для
`anchor test`. Rust-тест натомість використовує LiteSVM.

### !programs

`[programs.localnet]` зіставляє назву крейту Rust із адресою програми, що
використовується для локальних збірок і тестів.

### !provider

`[provider]` встановлює кластер і гаманець за замовчуванням для таких команд, як
`anchor deploy`.

### !scripts

`[scripts]` змінює те, що виконує `anchor test`. У цьому шаблоні `anchor test`
запускає `cargo test`.

</WithNotes>
