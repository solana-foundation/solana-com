AI-агенти стають дедалі автономнішими у своїх робочих процесах: пишуть код,
отримують дані, виділяють ресурси тощо. Щоб працювати повністю незалежно, їм
потрібен спосіб здійснювати «мікроплатежі» за сервіси програмно, у реальному
часі, з мінімальними витратами. Традиційні платіжні системи не підходять для
мікроплатежів. Комісії Solana менше цента та фінальність менше секунди роблять
це технічно та економічно вигідним.

## Чому це важливо?

Наприклад, уявіть розробника, який використовує LLM для створення застосунку та
потребує дані з платного API. Щоб отримати важливі дані, розробнику потрібно
зупинити свій робочий процес, дослідити API, обрати провайдера, створити
обліковий запис, додати спосіб оплати, згенерувати API-ключ, скопіювати ключ у
своє середовище, а потім продовжити робочий процес. Це тертя порушує стан
потоку. Для робочих процесів з підтримкою AI це перешкода.

Натомість, з агентськими платежами, агент може зробити запит, оплатити його та
продовжити свій робочий процес. Без облікових записів. Без API-ключів. Без
ручного втручання.

### Оплата API за допомогою pay.sh

[pay.sh](https://pay.sh) — це платіжний рівень для HTTP-агентів і інструментів
командного рядка. Зверніться до [документації pay.sh](https://pay.sh/docs) для
встановлення `pay`. CLI `pay` обгортає інструменти на кшталт `curl`, `codex` та
`claude`, щоб вони могли викликати платні API без облікових записів провайдерів
або ключів API. Коли API повертає MPP або x402 `402 Payment Required` виклик,
`pay` запитує ваш локальний гаманець підтвердити підписання.

## Протокол x402

Агентним платежам потрібен спосіб для клієнтів і серверів узгоджувати умови
оплати через стандартну веб-інфраструктуру без порушення традиційних
веб-інтерфейсів. Сфера мікроплатежів все ще перебуває на початковому етапі, і з
розвитком галузі, ймовірно, з'являться нові інструменти та стандарти. Однак
[протокол x402](https://www.x402.org/) виник як ранній стандарт із потужною
підтримкою екосистеми. x402 використовує
[HTTP статус 402 "Payment Required"](https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Status/402)
— стандарт з часів HTTP/1.1, який став практичним лише з появою
блокчейн-розрахунків.

### Як це працює

Клієнт надсилає запит, отримує 402 з умовами оплати, а потім повторює запит із
підписаним платежем. Сервер делегує перевірку та розрахунок фасилітатору —
опціональному посереднику, який обробляє відправку транзакції ончейн. Після
того, як фасилітатор підтверджує розрахунок, сервер повертає запитуваний
контент. Такий розподіл дозволяє постачальникам API приймати платежі без
безпосереднього управління ончейн-інфраструктурою.

![Діаграма потоку x402](/assets/guides/x402/x402-flow-solana.png)

Протокол не має стану (без сесій або ключів API), працює зі стандартною
HTTP-інфраструктурою та підтримує будь-який SPL-токен, який приймає сервер. Ось
фрагмент того, як налаштувати сервер Express за допомогою проміжного програмного
забезпечення x402:

```ts
app.use(
  paymentMiddleware(
    {
      "GET /costly-data": {
        accepts: [
          {
            scheme: "exact",
            price: "$0.001",
            network: "solana:EtWTRABZaYq6iMfeYKouRu166VU2xqa1",
            payTo: svmAddress
          }
        ],
        description: "Costly data",
        mimeType: "application/json"
      }
    },
    new x402ResourceServer(facilitatorClient).register(
      "solana:EtWTRABZaYq6iMfeYKouRu166VU2xqa1",
      new ExactSvmScheme()
    )
  )
);

app.get("/costly-data", (req, res) => {
  res.send({
    report: {
      data: "costly data"
    }
  });
});
```
