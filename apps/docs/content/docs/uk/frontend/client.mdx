`@solana/kit` постачає `createClient` — невеликий хост, який ви розширюєте за
допомогою плагінів. Кожен плагін додає певну можливість: RPC-з'єднання,
підписувач, планування транзакцій — і складений клієнт однаково підходить для
скриптів, серверних маршрутів і UI.

<Callout type="info">
  Розробляєте React-застосунок? [Посібник з React](/docs/frontend/react-hooks)
  обгортає цей самий клієнт за допомогою `@solana/react`, щоб компоненти
  зчитували стан гаманця та дані RPC через хуки. Починайте звідти та
  повертайтеся сюди, коли знадобиться більш тонке управління.
</Callout>

## Встановлення

```terminal
$ npm install @solana/kit @solana/kit-plugin-rpc @solana/kit-plugin-signer
```

Використовуйте будь-який менеджер пакетів. Kit — версія 7+, а пакети
`@solana/kit-plugin-*` — 0.13+.

## Створення клієнта

Компонуйте плагіни за допомогою `.use(...)`. Спочатку задайте підписувач, потім
RPC-з'єднання. `solanaDevnetRpc()` додає `client.rpc`, підписки,
планувальник/виконавець транзакцій та допоміжний метод `client.sendTransaction`
— тому застосунку рідко доводиться з'єднувати ці частини вручну.

```ts
import { createClient, generateKeyPairSigner } from "@solana/kit";
import { solanaDevnetRpc } from "@solana/kit-plugin-rpc";
import { signer } from "@solana/kit-plugin-signer";

const payer = await generateKeyPairSigner();

const client = createClient().use(signer(payer)).use(solanaDevnetRpc());

const slot = await client.rpc.getSlot().send();
console.log("current slot", slot);
```

Плагін `signer` встановлює як платника комісій, так і ідентифікатор застосунку.
RPC-пакет вимагає, щоб платник був визначений заздалегідь, тому порядок має
значення.

### Варіанти підписувача

`@solana/kit-plugin-signer` надає три ролі, щоб ви надавали лише ті права, які
потрібні клієнту:

- `signer(x)` — встановлює одночасно `client.payer` і `client.identity`.
- `payer(x)` — лише платник комісій.
- `identity(x)` — лише авторитет застосунку.

Кожен має варіанти `*FromFile(path)` (Node), `generated*()`,
`generated*WithSol(amount)` та `airdrop*()` для тестів і скриптів.

## Надсилання транзакції

Побудуйте інструкції за допомогою згенерованого клієнта програми, а потім
передайте їх до `client.sendTransaction`. Планувальник отримує blockhash, оцінює
обчислення, підписує ключем платника клієнта та надсилає транзакцію.

```ts
import { getTransferSolInstruction } from "@solana-program/system";
import { address, lamports } from "@solana/kit";

const transfer = getTransferSolInstruction({
  source: client.payer,
  destination: address("Fke...address"),
  amount: lamports(10_000_000n) // 0.01 SOL
});

const { context } = await client.sendTransaction([transfer]);
console.log("submitted", context.signature);
```

`sendTransaction` повертає об'єкт результату; підпис транзакції знаходиться у
`result.context.signature`. Метод перевіряє, що план вміщується в одну
транзакцію, і кидає виняток інакше — використовуйте `sendTransactions`, коли
пакет може охоплювати кілька транзакцій.

## Оберіть RPC-пакет

`@solana/kit-plugin-rpc` постачається з пресетами кластерів:

- `solanaDevnetRpc()` — налаштування devnet, аірдроп увімкнено
  (`client.airdrop`).
- `solanaMainnetRpc()` — mainnet; аірдроп спричиняє помилку під час компіляції.
- `solanaLocalRpc()` — `http://127.0.0.1:8899` для локального validator.
- `solanaRpc({ rpcUrl })` — будь-який endpoint.

```ts
import { solanaRpc } from "@solana/kit-plugin-rpc";

const client = createClient()
  .use(signer(payer))
  .use(solanaRpc({ rpcUrl: "https://your-rpc-provider.example" }));
```

### Гнучке складання

Пресети об'єднують RPC-з'єднання, планувальник, виконавець і допоміжні функції
надсилання, тому більшості застосунків ніколи не потрібні ці компоненти окремо.
Використовуйте гранулярні плагіни лише тоді, коли потрібно налаштувати один із
них — наприклад, власний планувальник або інший канал підписки. Цей підхід також
вимагає `@solana/kit-plugin-instruction-plan`, який пресети зазвичай підключають
автоматично:

```ts
import {
  solanaRpcConnection,
  rpcAirdrop,
  rpcTransactionPlanner,
  rpcTransactionPlanExecutor
} from "@solana/kit-plugin-rpc";
import { planAndSendTransactions } from "@solana/kit-plugin-instruction-plan";

const client = createClient()
  .use(signer(payer))
  .use(solanaRpcConnection({ rpcUrl: "https://api.devnet.solana.com" }))
  .use(rpcAirdrop())
  .use(rpcTransactionPlanner())
  .use(rpcTransactionPlanExecutor())
  .use(planAndSendTransactions());
```

## Додайте клієнти програм

Згенеровані пакети `@solana-program/*` надають клієнтські плагіни, завдяки яким
виклики програм доступні через клієнт. Встановіть потрібну програму та
підключіть її плагін за допомогою `.use()`:

```ts
import { tokenProgram } from "@solana-program/token";

const client = createClient()
  .use(signer(payer))
  .use(solanaDevnetRpc())
  .use(tokenProgram()); // adds client.token
```

<Callout type="warn" title="Діапазони peer-залежностей">
  Деякі релізи `@solana-program/*` досі оголошують peer `@solana/kit ^6.x`, тоді
  як Kit має версію 7.x. Вони сумісні з Kit 7 — перевизначте діапазон peer у
  вашому пакетному менеджері, якщо він скаржиться.
</Callout>

## Підписання без браузерного гаманця

Запускаєте headless-режим у маршруті API, воркері або скрипті? Підписуйте за
допомогою бекенд-сервісу управління ключами замість сирого keypair. Дивіться
[Keychain](/docs/tools/keychain) та
[Підписання у продакшені](/docs/core/transactions/signing-in-production).

## Типові патерни для Solana-розробників

- **Спільне налаштування**: Створіть клієнт один раз у базовому модулі та
  імпортуйте його до скриптів, серверних маршрутів і UI.
- **Читання на стороні сервера**: Клієнт лише з `signer` + `solanaRpc` (без
  wallet-плагіна) читає акаунти та симулює транзакції на сервері.
- **Локальне тестування**: Замініть RPC-пакет на `.use(litesvm())` з
  `@solana/kit-plugin-litesvm` для роботи з in-memory validator (лише Node).
- **Тестованість**: Складений клієнт легко мокати — замінюйте RPC-відповіді або
  підписувачів без браузерного гаманця.
