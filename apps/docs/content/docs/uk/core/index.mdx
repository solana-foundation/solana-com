Сім концепцій формують фундамент розробки на Solana. Читайте Акаунти, Програми,
Інструкції, Транзакції та Комісії саме в такому порядку, оскільки кожна наступна
базується на попередній. Прочитайте Акаунти та Програми перед PDA. Прочитайте
Програми, Інструкції та PDA перед CPI.

<Cards>
  <Card title="Акаунти" href="/docs/core/accounts">
    Solana зберігає весь стан в акаунтах — сховищі ключ-значення з 32-байтними
    адресами як ключами. Кожен акаунт має п'ять полів: lamports, data, owner,
    executable та rent_epoch.
  </Card>
  <Card title="Програми" href="/docs/core/programs">
    Смарт-контракти Solana, скомпільовані в sBPF. Програми є stateless, а весь
    змінюваний стан зберігається в окремих акаунтах даних, що передаються через
    інструкції.
  </Card>
  <Card title="Інструкції" href="/docs/core/instructions">
    Запит на виконання однієї програми, що містить ID програми, список акаунтів
    та непрозорі байти даних. Кілька інструкцій об'єднуються в одну транзакцію.
  </Card>
  <Card title="Транзакції" href="/docs/core/transactions">
    Атомарна одиниця виконання на Solana. Містить підписи та повідомлення
    (адреси акаунтів, останній blockhash та скомпільовані інструкції). Максимум
    1232 байти.
  </Card>
  <Card title="Комісії" href="/docs/core/fees">
    Базова комісія 5000 lamports за підпис, плюс опціональна пріоритетна комісія
    на основі запитаних обчислювальних одиниць. Максимум 1,4 млн обчислювальних
    одиниць на транзакцію.
  </Card>
  <Card title="Адреси, похідні від програм (PDA)" href="/docs/core/pda">
    Детерміністичні адреси, похідні від ID програми та seeds. Тільки
    програма-власник може підписувати від імені PDA через invoke_signed.
  </Card>
  <Card title="Міжпрограмні виклики (CPI)" href="/docs/core/cpi">
    Виклик однією програмою іншої через invoke або invoke_signed. Привілеї
    підписанта та запису поширюються на викликану програму. Максимальна глибина
    стеку інструкцій — 5 (верхній рівень плюс CPI).
  </Card>
</Cards>
