Транзакція виконується лише тоді, коли вона містить дійсний
[підпис](/docs/core/transactions/transaction-structure#signatures) від кожного
обов'язкового підписанта. Під час локальної розробки ви зазвичай зберігаєте цей
ключ як [`Keypair`](/developers/cookbook/wallets/create-keypair) і підписуєте
безпосередньо. У продакшні важливим є інше питання: **де зберігається приватний
ключ і хто має право авторизувати підпис за його допомогою?**

<Callout type="warn">
  Ніколи не вбудовуйте приватні ключі в клієнтський код, не включайте їх у
  фронтенд та не зберігайте в системі контролю версій. Будь-хто, хто отримає
  ключ, отримає повний контроль над коштами цього облікового запису.
</Callout>

## Підходи до підписання

| Підхід                     | Де зберігається ключ                                          | Хто підписує                                            | Найкраще підходить для                                                   |
| -------------------------- | ------------------------------------------------------------- | ------------------------------------------------------- | ------------------------------------------------------------------------ |
| **Локальний keypair**      | Файл або змінна середовища на машині                          | Ваш код, безпосередньо                                  | Локальна розробка, тести, CI                                             |
| **Браузерний гаманець**    | Гаманець кінцевого користувача (наприклад, Phantom, Solflare) | Кінцевий користувач, для кожної транзакції              | Фронтенд-dApps, де користувачі підписують власні транзакції              |
| **Хмарний KMS / HSM**      | Апаратно захищений сервіс ключів (AWS KMS, GCP KMS, Vault)    | Ваш бекенд запитує підпис у сервісу                     | Бекенд-сервіси, операції з казначейством, регульовані середовища         |
| **Керовані / MPC-гаманці** | Розподілені або зберігаються у провайдера                     | Провайдер підписує спільно відповідно до вашої політики | Вбудовані гаманці, процеси затвердження, інституційне зберігання активів |

Локальний keypair — єдиний підхід, який розміщує необроблений ключовий матеріал
безпосередньо у вашому додатку. Кожен виробничий підхід зберігає ключ у
спеціалізованій інфраструктурі та доручає їй підписувати від вашого імені.

## Підписання на бекенді

Коли серверу потрібно підписати транзакцію — сплачуючи комісії, надсилаючи
програмні транзакції або керуючи скарбницею — використовуйте спеціалізований
бекенд для управління ключами замість звичайного keypair. Solana Foundation
підтримує [Keychain](/docs/tools/keychain) — уніфіковану бібліотеку підписання
(Rust та TypeScript), яка надає єдиний інтерфейс `SolanaSigner` для всіх
бекендів із таблиці вище: Memory, HashiCorp Vault, AWS KMS, GCP KMS, Fireblocks,
Privy, Turnkey, CDP, Crossmint, Dfns, Para, Openfort та Utila.

Оскільки інтерфейс однаковий для всіх бекендів, ви можете розробляти локально з
ключем у пам'яті та перейти на виробничий бекенд через конфігурацію, не
переписуючи код додатку. Keychain сумісний з
[`@solana/kit`](/docs/clients/official/javascript) та
[Rust SDK](/docs/clients/official/rust).
