Keychain абстрагує підписувальний бекенд, однак експлуатація підписувача у
production залишається вашою відповідальністю. Ці практики застосовуються до
всіх бекендів; дивіться [Вибір бекенду](/docs/tools/keychain/choosing-a-backend)
для їх вибору.

## Розділяйте гарячі та холодні підписувачі

Використовуйте операційний («гарячий») підписувач з невеликим балансом для
частих автоматизованих транзакцій і окремий («холодний») підписувач для роботи з
великими сумами та переміщення коштів казначейства. Підкріпіть холодний
підписувач MPC-кастодією або хмарним HSM і вимагайте політик погодження перед
підписанням. Оскільки інтерфейс ідентичний, кожна роль може використовувати
різні бекенди без зміни точок виклику.

## Виконуйте ротацію ключів свідомо

У Solana підписувальний ключ _є_ адресою акаунта, тому ротація ключа призводить
до появи нової адреси. Плануйте ротацію як міграцію: надайте новий ключ,
перепризначте відповідні права (або перемістіть кошти) на нову адресу, після
чого виведіть старий ключ з обігу. Використовуйте окремі ключі для кожного
середовища, щоб ключі devnet ніколи не використовувалися повторно у mainnet.

## Надавайте доступ з мінімальними привілеями

Обмежуйте облікові дані кожного бекенда до мінімуму, необхідного для підписання.
Для AWS KMS це `kms:Sign` та `kms:DescribeKey` для конкретного ключа; надавайте
перевагу IAM-ролям над статичними обліковими даними. Застосовуйте той самий
принцип до політик Vault і ключів API керованих гаманців.

## Плануйте забезпечення доступності

Віддалені бекенди залежать від мережевого сервісу. Викликайте `isAvailable()`
перед тим, як покладатися на підписувач, застосовуйте повторні спроби з
відкладенням при тимчасових збоях і визначте резервний шлях на випадок відмов.
Keychain примусово застосовує HTTPS для віддалених бекендів і встановлює
стандартний тайм-аут запитів.

## Захищайте секрети та транспортування

- Усі віддалені бекенди спілкуються через HTTPS; кінцеві точки без HTTPS
  відхиляються.
- Надавайте API-ключі, токени та облікові дані з менеджера секретів або змінних
  середовища — ніколи не вбудовуйте їх у код і не фіксуйте в системі контролю
  версій.
- Keychain очищає відповіді з помилками, щоб уникнути витоку секретів у журнали;
  стежте, щоб ваші власні журнали не містили необроблених облікових даних та
  ключового матеріалу.

## Моніторинг та аудит підписання

Відстежуйте показники успішності підписання, затримку та кількість помилок, і
налаштовуйте сповіщення про стрибки — дивіться метрики в
[Готовність до продакшену](/docs/payments/production-readiness#monitoring).
Увімкніть журнали аудиту на стороні провайдера (KMS, Vault або кастодіана) і
звіряйте їх із транзакціями, які надсилає ваш застосунок.
