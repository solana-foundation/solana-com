`@solana/kit` incluye `createClient`, un pequeño host que se amplía con plugins.
Cada plugin añade una capacidad — una conexión RPC, un firmante, planificación
de transacciones — y el cliente compuesto da soporte a scripts, rutas de
servidor e interfaces de usuario por igual.

<Callout type="info">
  ¿Estás creando una aplicación con React? La [guía de
  React](/docs/frontend/react-hooks) envuelve este mismo cliente con
  `@solana/react` para que los componentes lean el estado de la billetera y los
  datos RPC a través de hooks. Empieza por ahí y vuelve aquí cuando necesites un
  control más preciso.
</Callout>

## Instalar

```terminal
$ npm install @solana/kit @solana/kit-plugin-rpc @solana/kit-plugin-signer
```

Usa cualquier gestor de paquetes. Kit es v7+ y los paquetes
`@solana/kit-plugin-*` son 0.13+.

## Crear un cliente

Combina plugins con `.use(...)`. Establece primero un firmante y luego la
conexión RPC. `solanaDevnetRpc()` añade `client.rpc`, suscripciones, el
planificador/ejecutor de transacciones y el helper `client.sendTransaction` — de
modo que una aplicación raramente necesita conectar esas piezas por sí misma.

```ts
import { createClient, generateKeyPairSigner } from "@solana/kit";
import { solanaDevnetRpc } from "@solana/kit-plugin-rpc";
import { signer } from "@solana/kit-plugin-signer";

const payer = await generateKeyPairSigner();

const client = createClient().use(signer(payer)).use(solanaDevnetRpc());

const slot = await client.rpc.getSlot().send();
console.log("current slot", slot);
```

El plugin `signer` establece tanto el pagador de comisiones como la identidad de
la aplicación. El paquete RPC requiere que haya un pagador configurado
previamente, por eso el orden importa.

### Variantes de firmante

`@solana/kit-plugin-signer` expone tres roles para que solo otorgues lo que un
cliente necesita:

- `signer(x)` — establece tanto `client.payer` como `client.identity`.
- `payer(x)` — solo pagador de comisiones.
- `identity(x)` — solo autoridad de la aplicación.

Cada uno tiene variantes `*FromFile(path)` (Node), `generated*()`,
`generated*WithSol(amount)` y `airdrop*()` para pruebas y scripts.

## Enviar una transacción

Construye instrucciones con un cliente de programa generado y luego pásalas a
`client.sendTransaction`. El planificador resuelve el blockhash, estima el
cómputo, firma con el pagador del cliente y lo envía.

```ts
import { getTransferSolInstruction } from "@solana-program/system";
import { address, lamports } from "@solana/kit";

const transfer = getTransferSolInstruction({
  source: client.payer,
  destination: address("Fke...address"),
  amount: lamports(10_000_000n) // 0.01 SOL
});

const { context } = await client.sendTransaction([transfer]);
console.log("submitted", context.signature);
```

`sendTransaction` se resuelve en un objeto de resultado; la firma de la
transacción se encuentra en `result.context.signature`. Comprueba que el plan
cabe en una sola transacción y lanza un error en caso contrario — usa
`sendTransactions` cuando un lote pueda abarcar múltiples transacciones.

## Elige un paquete RPC

`@solana/kit-plugin-rpc` incluye presets de clúster:

- `solanaDevnetRpc()` — configuración predeterminada de devnet, airdrop
  habilitado (`client.airdrop`).
- `solanaMainnetRpc()` — mainnet; el airdrop es un error en tiempo de
  compilación.
- `solanaLocalRpc()` — `http://127.0.0.1:8899` para un validator local.
- `solanaRpc({ rpcUrl })` — cualquier endpoint.

```ts
import { solanaRpc } from "@solana/kit-plugin-rpc";

const client = createClient()
  .use(signer(payer))
  .use(solanaRpc({ rpcUrl: "https://your-rpc-provider.example" }));
```

### Composición granular

Los presets agrupan la conexión RPC, el planificador, el ejecutor y los helpers
de envío, por lo que la mayoría de las aplicaciones nunca necesitan los
componentes de forma individual. Recurre a los plugins granulares solo para
personalizar alguno de ellos — por ejemplo, un planificador personalizado o un
canal de suscripción diferente. Esta opción también requiere
`@solana/kit-plugin-instruction-plan`, que los presets incluyen automáticamente:

```ts
import {
  solanaRpcConnection,
  rpcAirdrop,
  rpcTransactionPlanner,
  rpcTransactionPlanExecutor
} from "@solana/kit-plugin-rpc";
import { planAndSendTransactions } from "@solana/kit-plugin-instruction-plan";

const client = createClient()
  .use(signer(payer))
  .use(solanaRpcConnection({ rpcUrl: "https://api.devnet.solana.com" }))
  .use(rpcAirdrop())
  .use(rpcTransactionPlanner())
  .use(rpcTransactionPlanExecutor())
  .use(planAndSendTransactions());
```

## Añadir clientes de programa

Los paquetes `@solana-program/*` generados exponen plugins de cliente para que
las llamadas al programa se asocien al cliente. Instala el programa que
necesites y usa `.use()` para registrar su plugin:

```ts
import { tokenProgram } from "@solana-program/token";

const client = createClient()
  .use(signer(payer))
  .use(solanaDevnetRpc())
  .use(tokenProgram()); // adds client.token
```

<Callout type="warn" title="Rangos de dependencias de pares">
  Algunas versiones de `@solana-program/*` aún declaran una dependencia de par
  `@solana/kit ^6.x` mientras que Kit está en la versión 7.x. Funcionan con Kit
  7 — anula el rango de la dependencia de par en tu gestor de paquetes si
  aparece algún error.
</Callout>

## Firma sin una wallet de navegador

¿Ejecutando de forma headless en una ruta de API, un worker o un script? Firma
con un servicio de gestión de claves de backend en lugar de un keypair sin
procesar. Consulta [Keychain](/docs/tools/keychain) y
[Firma en Producción](/docs/core/transactions/signing-in-production).

## Patrones comunes para desarrolladores de Solana

- **Configuración compartida**: Crea el cliente una vez en un módulo central e
  impórtalo en scripts, rutas del servidor y tu interfaz de usuario.
- **Lecturas del lado del servidor**: Un cliente con solo `signer` + `solanaRpc`
  (sin plugin de wallet) lee cuentas y simula transacciones desde el servidor.
- **Pruebas locales**: Reemplaza el paquete RPC por `.use(litesvm())` de
  `@solana/kit-plugin-litesvm` para ejecutar contra un validator en memoria
  (solo Node).
- **Facilidad de prueba**: El cliente compuesto es fácil de simular — stub de
  respuestas RPC o firmantes sin necesidad de una wallet de navegador.
