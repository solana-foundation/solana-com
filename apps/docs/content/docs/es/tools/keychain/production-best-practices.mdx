Keychain abstrae el backend de firma, pero operar un firmante en producción
sigue siendo tu responsabilidad. Estas prácticas aplican a todos los backends;
consulta [Elegir un backend](/docs/tools/keychain/choosing-a-backend) para
seleccionar uno.

## Separa los firmantes activos y en frío

Usa un firmante operativo de saldo bajo ("activo") para transacciones frecuentes
y automatizadas, y un firmante separado de alto valor ("en frío") para
movimientos de tesorería. Respalda el firmante en frío con custodia MPC o un HSM
en la nube, y exige políticas de aprobación antes de que firme. Como la interfaz
es idéntica, cada rol puede usar un backend diferente sin modificar los puntos
de llamada.

## Rota las claves de forma deliberada

En Solana, la clave de firma _es_ la dirección de la cuenta, por lo que rotar
una clave genera una nueva dirección. Planifica la rotación como una migración:
aprovisiona una nueva clave, reasigna las autoridades relevantes (o transfiere
fondos) a la nueva dirección y luego retira la anterior. Usa claves distintas
por entorno para que las claves de devnet nunca se reutilicen en mainnet.

## Otorga acceso de mínimo privilegio

Limita las credenciales de cada backend al mínimo necesario para firmar. Para
AWS KMS, eso es `kms:Sign` e `kms:DescribeKey` sobre la clave específica;
prefiere roles de IAM en lugar de credenciales estáticas. Aplica el mismo
principio a las políticas de Vault y a las claves de API de billeteras
administradas.

## Planifica la disponibilidad

Los backends remotos dependen de un servicio de red. Llama a `isAvailable()`
antes de confiar en un firmante, aplica reintentos con retroceso exponencial
ante fallos transitorios y define una ruta de contingencia para las
interrupciones. Keychain impone HTTPS para los backends remotos y aplica un
tiempo de espera predeterminado para las solicitudes.

## Proteger secretos y el transporte

- Todos los backends remotos se comunican a través de HTTPS; los endpoints que
  no usen HTTPS son rechazados.
- Suministra claves de API, tokens y credenciales desde un gestor de secretos o
  variables de entorno — nunca los escribas directamente en el código ni los
  confirmes en el control de versiones.
- Keychain sanitiza las respuestas de error para evitar que los secretos se
  filtren en los registros; mantén tus propios registros libres de credenciales
  sin procesar y material de claves.

## Monitorear y auditar la firma

Realiza un seguimiento de la tasa de éxito de firma, la latencia y el recuento
de fallos, y configura alertas ante picos — consulta las métricas en
[Preparación para producción](/docs/payments/production-readiness#monitoring).
Habilita los registros de auditoría del proveedor (KMS, Vault o custodia) y
reconcílialos con las transacciones que tu aplicación envía.
