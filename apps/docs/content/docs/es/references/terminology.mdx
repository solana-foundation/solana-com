Los siguientes términos se utilizan en toda la documentación y el ecosistema de
desarrollo de Solana.

## cuenta

Un registro en el libro de contabilidad de Solana que contiene datos o es un
programa ejecutable.

Al igual que una cuenta en un banco tradicional, una cuenta de Solana puede
contener fondos llamados [lamports](#lamport). Cada cuenta en un
[clúster](#cluster) tiene una [dirección](#address) única.

## app

Una aplicación de interfaz de usuario que interactúa con un clúster de Solana.

## address

La ubicación en Solana que se puede utilizar para encontrar una cuenta. Cada
dirección es única en el [clúster](#cluster) de Solana.

Las direcciones en Solana pueden ser una [clave pública](#public-key-pubkey)
Ed25519 o una [dirección derivada de programa](#program-derived-address-pda).

## tabla de búsqueda de direcciones (ALT)

Una estructura de datos que almacena una lista de direcciones de cuentas,
permitiendo que las [transacciones versionadas](#versioned-transaction) hagan
referencia a cuentas utilizando índices compactos de 1 byte en lugar de claves
públicas completas de 32 bytes. Las tablas de búsqueda de direcciones permiten
que una sola transacción haga referencia a más cuentas de las que permite el
formato de transacción original.

## Anchor

Un framework para construir [programas onchain](#onchain-program) de Solana en
Rust. Anchor proporciona macros que reducen el código repetitivo, un sistema
estandarizado de validación de cuentas y genera un [IDL](#idl) que describe la
interfaz del programa. Es el framework más utilizado para el desarrollo de
programas en Solana.

## cuenta de token asociada (ATA)

Una [cuenta de token](#token-account) cuya dirección se deriva
determinísticamente de una dirección de billetera y una
[acuñación de token](#token-mint). El programa de cuenta de token asociada
garantiza que cada billetera tenga una cuenta de token canónica por tipo de
token.

## autoridad

La dirección de un usuario que tiene algún tipo de permiso sobre una cuenta.

Por ejemplo:

- La capacidad de acuñar nuevos tokens se otorga a la cuenta que es la
  'autoridad de acuñación' para la acuñación de token.
- La capacidad de actualizar un programa se otorga a la cuenta que es la
  'autoridad de actualización' de un programa.

## estado del banco

El resultado de interpretar todos los programas en el libro mayor a una
[altura de tick](#tick-height) determinada. Incluye al menos el conjunto de
todas las [cuentas](#account) que poseen [tokens nativos](#native-token)
distintos de cero.

## bloque

Un conjunto contiguo de [entradas](#entry) en el libro mayor cubierto por un
[voto](#ledger-vote). Un [líder](#leader) produce como máximo un bloque por
[slot](#slot).

## blockhash

Un valor único ([hash](#hash)) que identifica un registro (bloque). Solana
calcula un blockhash a partir del último [ID de entrada](#entry-id) del bloque.

## altura de bloque

El número de [bloques](#block) por debajo del bloque actual. El primer bloque
después del [bloque génesis](#genesis-block) tiene altura uno.

## validador bootstrap

El [validador](#validator) que produce el [bloque](#block) génesis (primero) de
una cadena de bloques.

## cargador BPF

El programa de Solana que posee y carga [programas onchain](/docs/core/programs)
[BPF](#onchain-program), permitiendo que el programa interactúe con el runtime.

## bump

Un solo byte (0–255) añadido a las [semillas](#seed) al derivar una
[dirección derivada de programa](#program-derived-address-pda) para asegurar que
la dirección resultante caiga fuera de la curva ed25519 y sea por lo tanto una
PDA válida. El bump canónico es el valor más alto que produce una PDA válida y
típicamente se guarda en los datos de la cuenta para evitar recalcularlo.

## cliente

Un programa informático que accede al [clúster](#cluster) de red del servidor de
Solana.

## compromiso

Una medida de la confirmación de red para el [bloque](#block).

## formato de array compacto

Un array compacto es un array serializado en el siguiente formato:

- La longitud del array (codificada como
  [compact-u16](https://github.com/anza-xyz/agave/blob/v2.1.13/short-vec/src/lib.rs))
- Los elementos del array listados uno tras otro

![Formato de array compacto](/assets/docs/core/transactions/compact_array_format.png)

El formato de array compacto se utiliza para codificar tanto el array de
[direcciones de cuenta](/docs/core/transactions/transaction-structure#account-addresses)
como el array de
[instrucciones](/docs/core/transactions/transaction-structure#instructions) en
mensajes de transacción.

## clúster

Un conjunto de [validadores](#validator) que mantienen un único
[libro mayor](#ledger).

## presupuesto de cómputo

El número máximo de [unidades de cómputo](#compute-units) consumidas por
transacción.

## unidades de cómputo

La unidad de medida más pequeña para el consumo de recursos computacionales de
la blockchain. Las unidades de cómputo se utilizan para medir los recursos
computacionales usados para procesar una transacción.

## tiempo de confirmación

La duración de tiempo real entre un [líder](#leader) creando una
[entrada de tick](#tick) y creando un [bloque confirmado](#confirmed-block).

## bloque confirmado

Un [bloque](#block) que ha recibido una [supermayor­ía](#supermajority) de
[votos del libro mayor](#ledger-vote).

## plano de control

Una red de gossip que conecta todos los [nodos](#node) de un
[clúster](#cluster).

## período de enfriamiento

Un número de [epochs](#epoch) después de que el [stake](#stake) ha sido
desactivado mientras progresi­vamente se vuelve disponible para retiro. Durante
este período, el stake se considera "desactivándose". Más información sobre:
[calentamiento y enfriamiento](https://docs.anza.xyz/consensus/stake-delegation-and-rewards#stake-warmup-cooldown-withdrawal)

## unidades de costo

También conocidas como `transaction cost`, es una estimación integral de todos
los recursos necesarios para procesar una transacción, medidos en
[unidades de cómputo](#compute-units). Más información
[aquí](https://www.anza.xyz/blog/why-solana-transaction-costs-and-compute-units-matter-for-developers).

## crédito

Ver [crédito de voto](#vote-credit).

## Cross Program Invocation (CPI)

Una llamada de un [programa onchain](#onchain-program) a otro. Para más
información, consulte [llamadas entre programas](/docs/core/cpi).

## plano de datos

Una red multicast utilizada para validar eficientemente [entradas](#entry) y
alcanzar consenso.

## drone

Un servicio offchain que actúa como custodio de la clave privada de un usuario.
Típicamente sirve para validar y firmar transacciones.

## entrada

Una entrada en el [libro mayor](#ledger), ya sea un [tick](#tick) o una
[entrada de transacción](#transactions-entry).

## id de entrada

Un [hash](#hash) resistente a preimagen sobre el contenido final de una entrada,
que actúa como el identificador globalmente único de la [entrada](#entry). El
hash sirve como evidencia de:

- La entrada siendo generada después de una duración de tiempo
- Las [transacciones](#transaction) especificadas son las incluidas en la
  entrada
- La posición de la entrada con respecto a otras entradas en el
  [libro mayor](#ledger)

Ver [prueba de historial](#proof-of-history-poh).

## epoch

El tiempo, es decir, el número de slots, durante el cual un
[programa de líder](#leader-schedule) es válido.

## cuenta de comisión

La cuenta de comisión en la transacción es la cuenta que paga el costo de
incluir la transacción en el libro mayor. Esta es la primera cuenta en la
transacción. Esta cuenta debe declararse como Lectura-Escritura (escribible) en
la transacción, ya que pagar por la transacción reduce el saldo de la cuenta.

## finalidad

Cuando los nodos que representan 2/3 del [stake](#stake) tienen una
[raíz](#root) común.

## bifurcación

Un [libro mayor](#ledger) derivado de entradas comunes pero que luego divergió.

## bloque génesis

El primer [bloque](#block) en la cadena.

## configuración génesis

El archivo de configuración que prepara el [libro mayor](#ledger) para el
[bloque génesis](#genesis-block).

## hash

Una huella digital de una secuencia de bytes.

## inflación

Un aumento en el suministro de tokens a lo largo del tiempo utilizado para
financiar recompensas por validación y para financiar el desarrollo continuo de
Solana.

## instrucción interna

Ver [Cross Program Invocation](#cross-program-invocation-cpi).

## instrucción

Una llamada para invocar un [manejador de instrucciones](#instruction-handler)
específico en un [programa](#program). Una instrucción también especifica qué
cuentas desea leer o modificar, y datos adicionales que sirven como entrada
auxiliar para el [manejador de instrucciones](#instruction-handler). Un
[cliente](#client) debe incluir al menos una instrucción en una
[transacción](#transaction), y todas las instrucciones deben completarse para
que la transacción se considere exitosa.

## manejador de instrucciones

Los manejadores de instrucciones son funciones de [programa](#program) que
procesan [instrucciones](#instruction) de [transacciones](#transaction). Un
manejador de instrucciones puede contener una o más
[invocaciones entre programas](#cross-program-invocation-cpi).

## IDL

Interface Definition Language (Lenguaje de Definición de Interfaz). Un archivo
JSON generado por [Anchor](#anchor) que describe los
[manejadores de instrucciones](#instruction-handler), cuentas y tipos de un
programa. El IDL permite que las bibliotecas de cliente generen automáticamente
interfaces tipadas para interactuar con un [programa onchain](#onchain-program).

## keypair

Una [clave pública](#public-key-pubkey) y su correspondiente
[clave privada](#private-key) para acceder a una cuenta.

## lamport

Un [token nativo](#native-token) fraccionario con el valor de 0.000000001
[SOL](#sol).

> Dentro del presupuesto de cómputo, se utiliza una cantidad de
> _[micro-lamports](https://github.com/solana-labs/solana/blob/ced8f6a512c61e0dd5308095ae8457add4a39e94/program-runtime/src/prioritization_fee.rs#L1-L2)_
> en el cálculo de las [tarifas de priorización](#prioritization-fee).

## líder

El rol de un [validador](#validator) cuando está agregando [entradas](#entry) al
[libro mayor](#ledger).

## programación de líderes

Una secuencia de [claves públicas](#public-key-pubkey) de
[validadores](#validator) asignadas a [slots](#slot). El clúster utiliza la
programación de líderes para determinar qué validador es el [líder](#leader) en
cualquier momento.

## libro mayor

Una lista de [entradas](#entry) que contienen [transacciones](#transaction)
firmadas por [clientes](#client). Conceptualmente, esto se puede rastrear hasta
el [bloque génesis](#genesis-block), pero el libro mayor de un
[validador](#validator) real puede tener solo [bloques](#block) más recientes
para reducir el almacenamiento, ya que los más antiguos no son necesarios para
la validación de bloques futuros por diseño.

## voto del libro mayor

Un [hash](#hash) del [estado del validador](#bank-state) a una determinada
[altura de tick](#tick-height). Comprende la afirmación de un
[validador](#validator) de que un [bloque](#block) que ha recibido ha sido
verificado, así como una promesa de no votar por un [bloque](#block) conflictivo
\(es decir, [bifurcación](#fork)\) durante un período de tiempo específico, el
período de [bloqueo](#lockout).

## cliente ligero

Un tipo de [cliente](#client) que puede verificar que está apuntando a un
[clúster](#cluster) válido. Realiza más verificación del libro mayor que un
[cliente delgado](#thin-client) y menos que un [validador](#validator).

## cargador

Un [programa](#program) con la capacidad de interpretar la codificación binaria
de otros [programas en cadena](#onchain-program).

## bloqueo

La duración de tiempo durante la cual un [validador](#validator) no puede
[votar](#ledger-vote) en otra [bifurcación](#fork).

## mensaje

El contenido estructurado de una [transacción](#transaction). Generalmente
contiene un encabezado, un array de direcciones de cuentas, un
[blockhash](#blockhash) reciente y un array de [instrucciones](#instruction).

Obtén más información sobre el
[formato de mensajes dentro de las transacciones](/docs/core/transactions/transaction-structure#header)
aquí.

## micro-lamport

1,000,000 micro-lamports = 1 [lamport](#lamport)

## Coeficiente de Nakamoto

Una medida de descentralización, el Coeficiente de Nakamoto es el menor número
de entidades independientes que pueden actuar colectivamente para detener una
blockchain. El término fue acuñado por Balaji S. Srinivasan y Leland Lee en
[Quantifying Decentralization](https://news.earn.com/quantifying-decentralization-e39db233c28e).

## token nativo

El [token](#token) utilizado para rastrear el trabajo realizado por los
[nodos](#node) en un clúster.

## nodo

Una computadora que participa en un [clúster](#cluster).

## cantidad de nodos

El número de [validadores](#validator) que participan en un [clúster](#cluster).

## onchain

Describe datos o programas que existen o se ejecutan en una blockchain.
"Onchain" (sin guion) es la ortografía preferida.

## programa onchain

El código ejecutable en la blockchain de Solana que interpreta las
[instrucciones](#instruction) enviadas dentro de cada
[transacción](#transaction) para leer y modificar las cuentas sobre las cuales
tiene control. Estos programas a menudo se denominan
"[_smart contracts_](/docs/core/programs)" en otras blockchains.

## propietario

Puede referirse al [programa propietario](#owning-program) o a la
[billetera propietaria](#owning-wallet). Por ejemplo,
[getTokenAccountsByOwner](https://solana.com/docs/rpc/http/gettokenaccountsbyowner)
obtiene las cuentas de tokens que pertenecen a una billetera en particular; sin
embargo, el programa propietario de esas cuentas de tokens es el Token Program.

Para evitar confusiones, es preferible usar
[programa propietario](#owning-program) o
[billetera propietaria](#owning-wallet) en lugar del término genérico
'propietario'.

## programa propietario

La [dirección](#address) del [programa](#onchain-program) que posee la cuenta.
Solo el programa propietario es capaz de modificar la cuenta.

Ver también [autoridad](#authority).

## billetera propietaria

La billetera que está asociada con una cuenta. Por ejemplo, el Token Program se
utiliza para almacenar cuentas de tokens asociadas con una billetera y mint
particulares utilizando una dirección creada a partir de esa
[billetera](#wallet) y [mint](#token-mint).

## PoH

Ver [Proof of History](#proof-of-history-poh).

## punto

Un [crédito](#credit) ponderado en un régimen de recompensas. En el
[régimen de recompensas](https://docs.anza.xyz/consensus/stake-delegation-and-rewards)
del [validador](#validator), el número de puntos adeudados a un [stake](#stake)
durante el canje es el producto de los [créditos de voto](#vote-credit)
obtenidos y el número de [lamports](#lamport) apostados.

## clave privada

La clave privada de un [keypair](#keypair).

## programa

Ver [programa onchain](#onchain-program).

## Program Derived Address (PDA)

Una cuenta cuya dirección se crea basándose en un
[programa onchain](#onchain-program) y una o más [semillas](#seed) creadas por
el autor del programa. Las direcciones PDA no son claves públicas, y los PDA no
tienen claves privadas.

La autoridad de firma para un PDA es el [programa onchain](#onchain-program) que
es su [programa propietario](#owning-program).

## id del programa

La clave pública de la [cuenta](#account) que contiene un
[programa](#onchain-program).

## Proof of History (PoH)

Una secuencia de pruebas, cada una de las cuales demuestra que ciertos datos
existieron antes de que se creara la prueba y que transcurrió una duración
precisa de tiempo antes de la prueba anterior. Al igual que una
[VDF](#verifiable-delay-function-vdf), un Proof of History puede verificarse en
menos tiempo del que tomó producirlo.

## tarifa de priorización

Una tarifa adicional que el usuario puede especificar en la
[instrucción](#instruction) del presupuesto de cómputo para priorizar sus
[transacciones](#transaction).

La tarifa de priorización se calcula multiplicando las unidades de cómputo
máximas solicitadas por el precio de la unidad de cómputo (especificado en
incrementos de 0.000001 [lamports](#lamport) por unidad de cómputo) redondeado
hacia arriba al [lamport](#lamport) más cercano.

Las transacciones deben solicitar la cantidad mínima de unidades de cómputo
necesarias para la ejecución a fin de minimizar las tarifas.

## clave pública (pubkey)

La clave pública de un [keypair](#keypair). Una clave pública es uno de los
diferentes tipos de [dirección](#address) que una cuenta puede usar en Solana.

## rent

Tarifa pagada por [Cuentas](#account) y [Programas](#program) para almacenar
datos en la blockchain. Cuando las cuentas no tienen saldo suficiente para pagar
el rent, pueden ser eliminadas por el recolector de basura.

Ver también [exento de rent](#rent-exempt) a continuación. Obtén más información
sobre el rent aquí: [¿Qué es el rent?](/docs/core/accounts/account-structure).

## exento de rent

Cuentas que mantienen un saldo mínimo de lamport proporcional a la cantidad de
datos almacenados en la cuenta. Todas las cuentas recién creadas se almacenan en
cadena de forma permanente hasta que la cuenta se cierra. No es posible crear
una cuenta que quede por debajo del umbral de exención de rent.

## raíz

Un [bloque](#block) o [slot](#slot) que ha alcanzado el [lockout](#lockout)
máximo en un [validator](#validator). La raíz es el bloque más alto que es
ancestro de todas las bifurcaciones activas en un validator. Todos los bloques
ancestros de una raíz también son transitivamente una raíz. Los bloques que no
son ancestros ni descendientes de la raíz quedan excluidos de consideración para
el consenso y pueden descartarse.

## entorno de ejecución

El componente de un [validator](#validator) responsable de la ejecución de
[programas](#program).

## Sealevel

El entorno de ejecución paralelo de Solana para
[programas onchain](#onchain-program).

## seed

Un valor de byte o arreglo de valores de byte utilizado como entrada al derivar
una [dirección derivada de programa](#program-derived-address-pda). Un programa
puede usar cualquier combinación de cadenas estáticas, claves públicas u otros
datos como seeds. Los seeds se combinan con el ID del programa y un
[bump](#bump) para producir una dirección PDA única.

## shred

Una fracción de un [bloque](#block); la unidad más pequeña enviada entre
[validadores](#validator).

## firma

Una firma ed25519 de 64 bytes compuesta por R (32 bytes) y S (32 bytes). Con el
requisito de que R sea un punto Edwards empaquetado que no sea de orden pequeño
y S sea un escalar en el rango de `0 <= S < L`. Este requisito garantiza que no
haya maleabilidad de firma. Cada transacción debe tener al menos una firma para
la [cuenta de comisión](#fee-account). Por lo tanto, la primera firma en la
transacción puede ser tratada como
[identificador de transacción](#transaction-id)

## tasa de omisión

El porcentaje de [slots omitidos](#skipped-slot) del total de slots de líder en
el epoch actual. Esta métrica puede ser engañosa ya que tiene alta varianza
después del límite del epoch cuando el tamaño de la muestra es pequeño, así como
para validadores con un número bajo de slots de líder, sin embargo también puede
ser útil para identificar configuraciones incorrectas del nodo en ocasiones.

## slot omitido

Un [slot](#slot) pasado que no produjo un [bloque](#block), porque el líder
estaba desconectado o el [fork](#fork) que contenía el slot fue abandonado por
una alternativa mejor mediante el consenso del clúster. Un slot omitido no
aparecerá como ancestro para bloques en slots posteriores, ni incrementará la
[altura de bloque](#block-height), ni expirará el `recent_blockhash` más
antiguo.

Si un slot ha sido omitido solo puede determinarse cuando se vuelve más antiguo
que el último slot [enraizado](#root) (por lo tanto, no omitido).

## slot

El período de tiempo durante el cual cada [líder](#leader) procesa transacciones
y produce un [bloque](#block).

Colectivamente, los slots crean un reloj lógico. Los slots están ordenados
secuencialmente y no se superponen, comprendiendo un tiempo del mundo real
aproximadamente igual según [PoH](#proof-of-history-poh).

## contrato inteligente

Ver [programa onchain](#onchain-program).

## SOL

El [token nativo](#native-token) de un [clúster](#cluster) de Solana.

## Biblioteca de Programas de Solana (SPL)

Una [biblioteca de programas](https://www.solana-program.com/docs/) en Solana
como spl-token que facilita tareas como la creación y el uso de tokens.

## participación

Tokens que se pierden en el [clúster](#cluster) si se puede probar un
comportamiento malicioso del [validador](#validator).

## calidad de servicio ponderada por participación (SWQoS)

SWQoS permite
[trato preferencial para transacciones que provienen de validadores con participación](/developers/guides/advanced/stake-weighted-qos).

## supermayoría

2/3 de un [clúster](#cluster).

## System Program

Un programa nativo de Solana (ID de programa `11111111111111111111111111111111`)
responsable de crear nuevas [cuentas](#account), asignar datos de cuenta,
asignar cuentas a programas y transferir [SOL](#sol) entre cuentas. Todas las
cuentas nuevas en Solana son propiedad del System Program hasta que se
reasignen.

## sysvar

Una [cuenta](#account) del sistema. Los
[Sysvars](https://docs.anza.xyz/runtime/sysvars) proporcionan información del
estado del clúster como la altura actual del tick, valores de [puntos](#point)
de recompensas, etc. Los programas pueden acceder a Sysvars a través de una
cuenta Sysvar (pubkey) o consultando mediante una syscall.

## cliente ligero

Un tipo de [cliente](#client) que confía en que se está comunicando con un
[clúster](#cluster) válido.

## tick

Una [entrada](#entry) del libro mayor que estima la duración del reloj de pared.

## altura de tick

El enésimo [tick](#tick) en el [libro mayor](#ledger).

## token

Un activo transferible digitalmente.

## token account

Una [cuenta](#account) que mantiene un saldo de un [token](#token) específico.
Las cuentas de token son propiedad del [Token Program](#token-program) o del
[Token Extensions Program](#token-extensions-program) y almacenan el saldo del
token, el [token mint](#token-mint) asociado y la dirección de la billetera
propietaria.

## Token Extensions Program

El [Token Extensions Program](https://www.solana-program.com/docs/token-2022)
tiene el ID de programa `TokenzQdBNbLqP5VEhdkAS6EPFLC1PHnBqCXEpPxuEb` e incluye
todas las mismas características que el [Token Program](#token-program), pero
viene con extensiones como transferencias confidenciales, lógica de
transferencia personalizada, metadatos extendidos y mucho más.

## token mint

Una cuenta que puede producir (o 'acuñar') tokens. Los diferentes tokens se
distinguen por sus direcciones únicas de token mint.

## Token Program

El [Token Program](https://www.solana-program.com/docs/token) tiene el ID de
programa `TokenkegQfeZyiNwAJbNbGKPFXCWuBvf9Ss623VQ5DA` y proporciona las
capacidades básicas de transferir, congelar y acuñar tokens.

## tps

[Transacciones](#transaction) por segundo.

## tpu

[Unidad de procesamiento de transacciones](https://docs.anza.xyz/validator/tpu).

## transaction

Una o más [instrucciones](#instruction) firmadas por un [cliente](#client)
utilizando uno o más [keypairs](#keypair) y ejecutadas atómicamente con solo dos
resultados posibles: éxito o fracaso.

## transaction id

La primera [firma](#signature) en una [transacción](#transaction), que puede
usarse para identificar de forma única la transacción a través del
[libro mayor](#ledger) completo.

## transaction confirmations

El número de [bloques confirmados](#confirmed-block) desde que la transacción
fue aceptada en el [libro mayor](#ledger). Una transacción se finaliza cuando su
bloque se convierte en una [raíz](#root).

## transactions entry

Un conjunto de [transacciones](#transaction) que pueden ejecutarse en paralelo.

## tvu

## validator

Un participante completo en un [clúster](#cluster) de red de Solana que produce
nuevos [bloques](#block). Un validator valida las transacciones agregadas al
[libro mayor](#ledger)

## VDF

Ver [función de retardo verificable](#verifiable-delay-function-vdf).

## función de retardo verificable (VDF)

Una función que requiere una cantidad fija de tiempo para ejecutarse y que
produce una prueba de que se ejecutó, la cual puede ser verificada en menos
tiempo del que tomó producirla.

## transacción versionada

Un formato de transacción que soporta características adicionales más allá del
formato original de [transacción](#transaction). La versión 0 (`v0`) es el
primer formato versionado y añade soporte para
[tablas de búsqueda de direcciones](#address-lookup-table-alt), permitiendo que
las transacciones hagan referencia a más cuentas de las que el formato original
permite.

## voto

Ver [voto del libro mayor](#ledger-vote).

## crédito de voto

Una recompensa acumulada para [validadores](#validator). Se otorga un crédito de
voto a un validador en su cuenta de voto cuando el validador alcanza una
[raíz](#root).

## billetera

Una colección de [pares de claves](#keypair) que permite a los usuarios
administrar sus fondos.

## período de preparación

Un número determinado de [épocas](#epoch) después de que el [stake](#stake) ha
sido delegado mientras progesivamente se vuelve efectivo. Durante este período,
el stake se considera que está "activándose". Más información sobre:
[preparación y enfriamiento](https://docs.anza.xyz/consensus/stake-delegation-and-rewards#stake-warmup-cooldown-withdrawal)
