h1: Crea Tu Primer Programa de Solana
---

Este inicio rápido utiliza el proyecto base del
[framework Anchor](https://www.anchor-lang.com/docs) generado por `anchor init`.
Crearás el proyecto de forma local, ejecutarás sus pruebas, compilarás el
programa y revisarás el código del programa que Anchor genera.

## Requisitos previos

Antes de comenzar,
[instala las herramientas de desarrollo de Solana](/docs/intro/installation). La
instalación incluye Rust, la CLI de Solana y la CLI de Anchor.

Usa la versión 1.1.2 o superior de la CLI de Anchor para esta plantilla.
Verifica tu versión instalada:

```terminal
$ anchor --version
```

<Steps>

<Step>

### Crear el proyecto

Ejecuta los siguientes comandos en tu terminal:

```terminal
$ anchor init my-program
$ cd my-program
```

El proyecto base incluye un programa de Solana en `programs/my-program`. El
programa incluye dos instrucciones: una para inicializar una cuenta de contador
y otra para incrementar el contador.

Algunas partes de la plantilla demuestran patrones comunes de programas de
Solana: derivar direcciones de cuenta [PDA](/docs/core/pda), realizar un Cross
Program Invocation ([CPI](/docs/core/cpi)) para transferir SOL, y usar
verificaciones de error personalizadas para detener una instrucción cuando una
condición falla.

<Step>

### Compilar el programa

Ejecuta `anchor build` para compilar el programa base:

```terminal
$ anchor build
```

El programa compilado se escribe en `target/deploy/my_program.so`. Cuando el
programa se despliega, el contenido de este archivo `.so` se almacena en una
cuenta onchain.

</Step>

<Step>

### Ejecutar prueba

Ejecuta la prueba predeterminada:

```terminal
$ anchor test
```

El `Anchor.toml` de esta plantilla utiliza el comando de prueba de Rust:

```toml title="Anchor.toml"
skip_local_validator = true

[scripts]
test = "cargo test"
```

La prueba carga el programa compilado en [LiteSVM](https://www.litesvm.com/),
crea un pagador, envía las instrucciones _rs`initialize`_ y _rs`increment`_,
luego verifica el estado de la cuenta del contador.

Ejecutar `anchor test` también compila el programa, por lo que no es necesario
ejecutar `anchor build` primero al realizar pruebas localmente.

</Step>

<Step>

### Desplegar programa

Las pruebas locales son el ciclo de retroalimentación más rápido. Cuando estés
listo para desplegar en una red, por ejemplo devnet, primero compila y luego
despliega en un clúster.

Desplegar un programa de Solana requiere SOL porque el programa se almacena en
un program account, y dicha cuenta debe pagar por el espacio que utiliza. En
devnet, solicita SOL de devnet de forma gratuita desde el
[Solana Faucet](https://faucet.solana.com/) o con la CLI de Solana:

```terminal
$ solana airdrop 2 --url devnet
```

```terminal
$ anchor build
$ anchor deploy --provider.cluster devnet
```

</Step>

</Steps>

## Archivos fuente

El directorio `src` contiene el programa de Solana. La documentación de
[Estructura del programa](https://www.anchor-lang.com/docs/basics/program-structure)
de Anchor explica los macros principales utilizados aquí, incluyendo
_rs`declare_id!`_, _rs`#[program]`_, _rs`#[derive(Accounts)]`_ y
_rs`#[account]`_. Esta sección recorre los archivos de la plantilla.

### lib.rs

`lib.rs` es el punto de entrada del programa. Conecta los archivos fuente,
define la dirección del programa y define las instrucciones del programa que los
usuarios pueden invocar.

<WithNotes>

```rust title="programs/my-program/src/lib.rs" -w
// !tooltip[/pub mod constants/] constantsMod
pub mod constants;
// !tooltip[/pub mod error/] errorMod
pub mod error;
// !tooltip[/pub mod instructions/] instructionsMod
pub mod instructions;
// !tooltip[/pub mod state/] stateMod
pub mod state;

// !tooltip[/anchor_lang::prelude/] prelude
use anchor_lang::prelude::*;

pub use constants::*;
pub use instructions::*;
pub use state::*;

// !tooltip[/declare_id!/] declareId
declare_id!("82sFkffP9wxwpyfZyeaKHH2chQoJPUGsJZSPi9mrUuXd");

// !tooltip[/#\[program\]/] programMacro
#[program]
pub mod my_program {
    use super::*;

    // !tooltip[/pub fn initialize/] publicInstructions
    pub fn initialize(ctx: Context<Initialize>) -> Result<()> {
        // !tooltip[/handle_initialize/] handlers
        crate::instructions::initialize::handle_initialize(ctx)
    }

    // !tooltip[/pub fn increment/] publicInstructions
    pub fn increment(ctx: Context<Increment>) -> Result<()> {
        // !tooltip[/handle_increment/] handlers
        crate::instructions::increment::handle_increment(ctx)
    }
}
```

### !constantsMod

_rs`pub mod constants`_ incorpora `constants.rs` al crate del programa. Aquí es
donde la plantilla almacena los valores compartidos del programa, como el seed
del PDA, el monto de transferencia y el valor máximo del contador.

### !errorMod

_rs`pub mod error`_ incorpora `error.rs` al crate del programa. Este archivo
define los errores personalizados que el programa puede devolver cuando una
instrucción no es válida.

### !instructionsMod

_rs`pub mod instructions`_ incorpora `instructions.rs` al crate del programa.
Ese archivo funciona como un índice para los archivos específicos de
instrucciones en `src/instructions/`.

### !stateMod

_rs`pub mod state`_ incorpora `state.rs` al crate del programa. Este archivo
define la estructura de datos para las cuentas propiedad de este programa,
incluida la cuenta _rs`Counter`_.

### !prelude

_rs`anchor_lang::prelude::*`_ importa los tipos y macros de Anchor utilizados
para definir un programa de Solana, incluyendo _rs`Context`_, _rs`Result`_,
_rs`Pubkey`_, _rs`Account`_, _rs`Signer`_ y los atributos de cuenta de Anchor.

### !reexports

Las líneas _rs`pub use`_ hacen que los elementos públicos de los módulos
secundarios estén disponibles desde la raíz del crate. Esto mantiene los puntos
de entrada de instrucciones concisos, como _rs`Context<Initialize>`_ en lugar de
una ruta de módulo más larga.

### !declareId

_rs`declare_id!`_ define la dirección del programa. Debe coincidir con la
dirección en `Anchor.toml`, el program account desplegado y cualquier usuario o
prueba que llame al programa.

### !programMacro

_rs`#[program]`_ le indica a Anchor que genere el punto de entrada del programa
Solana y el código de despacho de instrucciones. Las funciones públicas de este
módulo se convierten en las instrucciones que los usuarios pueden invocar.

### !publicInstructions

_rs`initialize`_ y _rs`increment`_ son los dos nombres de instrucciones que los
usuarios pueden invocar. Cada una recibe un _rs`Context<T>`_. El tipo dentro de
_rs`Context`_ define las cuentas requeridas por esa instrucción, y Anchor valida
dichas cuentas antes de que se ejecute la instrucción.

### !handlers

La función manejadora contiene la lógica de la instrucción.

</WithNotes>

### constants.rs

`constants.rs` centraliza los valores compartidos en un solo lugar. En esta
plantilla, _rs`COUNTER_SEED`_ deriva el PDA del contador,
_rs`HELLO_WORLD_LAMPORTS`_ se transfiere durante la inicialización, y
_rs`MAX_COUNT`_ se verifica antes de incrementar.

<WithNotes>

```rust title="programs/my-program/src/constants.rs" -w
use anchor_lang::prelude::*;

// !tooltip[/#\[constant\]/] anchorConstant
// !tooltip[/COUNTER_SEED/] counterSeed
#[constant]
pub const COUNTER_SEED: &[u8] = b"counter";

// !tooltip[/HELLO_WORLD_LAMPORTS/] helloLamports
#[constant]
pub const HELLO_WORLD_LAMPORTS: u64 = 1;

// !tooltip[/MAX_COUNT/] maxCount
#[constant]
pub const MAX_COUNT: u64 = 10;
```

### !anchorConstant

_rs`#[constant]`_ indica a Anchor que incluya el valor en el IDL del programa.
Esto hace que las constantes importantes del programa sean visibles para los
clientes generados y las herramientas.

### !counterSeed

_rs`COUNTER_SEED`_ es la cadena de bytes utilizada para derivar el PDA del
contador. Tanto el programa como las pruebas usan este seed para que coincidan
en la dirección de la cuenta sin necesidad de almacenar un keypair separado.
Esta plantilla usa un seed fijo, por lo que el programa tiene una única
dirección de contador. Los programas reales suelen incluir la dirección de un
usuario u otro identificador en los seeds para crear múltiples cuentas
relacionadas.

### !helloLamports

_rs`HELLO_WORLD_LAMPORTS`_ es la cantidad de lamports transferidos desde el
pagador a la cuenta del contador dentro de _rs`initialize`_. Es deliberadamente
pequeña porque la plantilla demuestra un CPI al System Program.

### !maxCount

_rs`MAX_COUNT`_ es el límite superior verificado por _rs`increment`_. Mantenerlo
aquí proporciona al programa, las pruebas y el IDL generado un nombre compartido
para esta regla.

</WithNotes>

<SideBySide>

### !left

```rust title="initialize.rs" -w
#[derive(Accounts)]
pub struct Initialize<'info> {
    // ...
    #[account(
        init,
        payer = payer,
        space = 8 + Counter::INIT_SPACE,
        // !mark
        // !mention counter-seed
        seeds = [COUNTER_SEED],
        bump
    )]
    pub counter: Account<'info, Counter>,
    // ...
}

pub fn handle_initialize(ctx: Context<Initialize>) -> Result<()> {
    // ...

    // !mark
    // !mention hello-world-lamports
    anchor_lang::system_program::transfer(cpi_ctx, HELLO_WORLD_LAMPORTS)?;

    Ok(())
}
```

### !right

`initialize.rs` usa dos constantes:

- [_rs`COUNTER_SEED`_](mention:counter-seed) obtiene la dirección PDA del
  contador.
- [_rs`HELLO_WORLD_LAMPORTS`_](mention:hello-world-lamports) establece la
  cantidad transferida desde el pagador a la cuenta del contador.

</SideBySide>

<SideBySide>

### !left

```rust title="increment.rs" -w
pub fn handle_increment(ctx: Context<Increment>) -> Result<()> {
    // ...

    require!(
        // !mark
        // !mention max-count
        ctx.accounts.counter.count < MAX_COUNT,
        ErrorCode::CounterOverflow,
    );

    ctx.accounts.counter.count += 1;
    Ok(())
}
```

### !right

`increment.rs` usa [_rs`MAX_COUNT`_](mention:max-count) como límite superior del
contador. Si el conteo actual ya se encuentra en el máximo, _rs`require!`_
devuelve _rs`CounterOverflow`_ y los datos de la cuenta no se modifican.

</SideBySide>

### state.rs

`state.rs` define tipos de datos personalizados para las cuentas que el programa
crea y administra. El programa define instrucciones para crear, inicializar y
actualizar esos datos, pero los datos del contador no se almacenan dentro del
programa en sí. Se almacenan en una cuenta separada con su propia dirección.

<WithNotes>

```rust title="programs/my-program/src/state.rs" -w
use anchor_lang::prelude::*;

// !tooltip[/#\[account\]/] accountMacro
#[account]
// !tooltip[/InitSpace/] initSpace
#[derive(InitSpace)]
pub struct Counter {
    // !tooltip[/count/] count
    pub count: u64,
    // !tooltip[/authority/] authority
    pub authority: Pubkey,
}
```

### !accountMacro

_rs`#[account]`_ le indica a Anchor que esta estructura es un dato que puede
almacenarse en una cuenta onchain propiedad de este programa. Anchor la usa para
serializar, deserializar y validar las cuentas _rs`Counter`_.

### !initSpace

_rs`InitSpace`_ permite a Anchor calcular _rs`Counter::INIT_SPACE`_, el número
de bytes necesarios para los campos de esta estructura. El discriminador de
cuenta de 8 bytes de Anchor se agrega por separado cuando se crea la cuenta.

### !count

_rs`count`_ es el valor del contador almacenado dentro de los datos de la
cuenta.

### !authority

_rs`authority`_ almacena la dirección autorizada para incrementar el contador.
_rs`initialize`_ escribe esta dirección, y _rs`increment`_ requiere que esa
misma dirección firme.

</WithNotes>

<SideBySide>

### !left

```rust title="initialize.rs" -w
#[account(
    init,
    payer = payer,
    // !mark
    // !mention counter-space
    space = 8 + Counter::INIT_SPACE,
    seeds = [COUNTER_SEED],
    bump
)]
pub counter: Account<'info, Counter>,
// ...

pub fn handle_initialize(ctx: Context<Initialize>) -> Result<()> {
    // !mark(1:2)
    // !mention(1:2) initialize-fields
    ctx.accounts.counter.count = 0;
    ctx.accounts.counter.authority = ctx.accounts.payer.key();

    Ok(())
}
```

```rust title="increment.rs" -w
pub fn handle_increment(ctx: Context<Increment>) -> Result<()> {
    // ...

    // !mark
    // !mention increment-count
    ctx.accounts.counter.count += 1;
    Ok(())
}
```

### !right

`state.rs` define los datos de la cuenta _rs`Counter`_. Los archivos de
instrucciones utilizan ese tipo cuando crean y actualizan la cuenta:

- [_rs`Counter::INIT_SPACE`_](mention:counter-space) dimensiona la cuenta para
  los campos definidos en `state.rs`.
- [_rs`count`_ y _rs`authority`_](mention:initialize-fields) son los valores de
  los campos que se escriben cuando se inicializa la cuenta.
- [_rs`count += 1`_](mention:increment-count) actualiza el valor del contador
  almacenado una vez que la validación es exitosa.

</SideBySide>

### error.rs

`error.rs` define los errores personalizados del programa. En esta plantilla,
los errores demuestran cómo los manejadores de instrucciones se detienen cuando
un llamante no tiene permiso para actualizar el contador o cuando el contador ya
ha alcanzado _rs`MAX_COUNT`_.

<WithNotes>

```rust title="programs/my-program/src/error.rs" -w
use anchor_lang::prelude::*;

// !tooltip[/#\[error_code\]/] errorCode
#[error_code]
pub enum ErrorCode {
    // !tooltip[/Unauthorized/] unauthorized
    #[msg("Only the counter authority can update this counter")]
    Unauthorized,
    // !tooltip[/CounterOverflow/] counterOverflow
    #[msg("Counter has reached the maximum value")]
    CounterOverflow,
}
```

### !errorCode

_rs`#[error_code]`_ indica a Anchor que convierta estas variantes en errores de
programa personalizados. Los clientes y las pruebas pueden usar esos errores en
lugar de adivinar a partir de códigos de error sin procesar.

### !unauthorized

_rs`Unauthorized`_ se devuelve cuando el firmante de la transacción no es la
autoridad almacenada en la cuenta del contador.

### !counterOverflow

_rs`CounterOverflow`_ se devuelve cuando al incrementar se infringiría la regla
_rs`MAX_COUNT`_ del programa.

</WithNotes>

<SideBySide>

### !left

```rust title="increment.rs" -w
pub fn handle_increment(ctx: Context<Increment>) -> Result<()> {
    require_keys_eq!(
        ctx.accounts.counter.authority,
        ctx.accounts.authority.key(),
        // !mark
        // !mention unauthorized-error
        ErrorCode::Unauthorized,
    );
    require!(
        ctx.accounts.counter.count < MAX_COUNT,
        // !mark
        // !mention overflow-error
        ErrorCode::CounterOverflow,
    );

    ctx.accounts.counter.count += 1;
    msg!("Hello, world! Counter is now {}", ctx.accounts.counter.count);
    Ok(())
}
```

### !right

`error.rs` nombra los errores que la instrucción puede devolver:

- [_rs`ErrorCode::Unauthorized`_](mention:unauthorized-error) se devuelve cuando
  el firmante no es la autoridad almacenada en la cuenta del contador.
- [_rs`ErrorCode::CounterOverflow`_](mention:overflow-error) se devuelve cuando
  el contador ya ha alcanzado _rs`MAX_COUNT`_.

</SideBySide>

### instructions.rs

`instructions.rs` conecta los archivos de instrucciones con el crate del
programa para que `lib.rs` pueda acceder al código de instrucciones
_rs`initialize`_ y _rs`increment`_. Cada archivo de instrucciones define las
cuentas requeridas por esa instrucción y la lógica del manejador que se ejecuta
después de que Anchor valida dichas cuentas.

<WithNotes>

```rust title="programs/my-program/src/instructions.rs" -w
// !tooltip[/pub mod initialize/] moduleDeclarations
pub mod initialize;
pub mod increment;

// !tooltip[/pub use initialize/] reexports
pub use initialize::*;
pub use increment::*;
```

### !moduleDeclarations

Las líneas _rs`pub mod`_ hacen que `src/instructions/initialize.rs` e
`src/instructions/increment.rs` formen parte del crate del programa.

### !reexports

Las líneas _rs`pub use`_ reexportan elementos públicos de esos archivos. Esto
hace que las estructuras de cuentas de instrucciones y las funciones manejadoras
estén disponibles a través del módulo _rs`instructions`_.

</WithNotes>

### initialize.rs

`initialize.rs` define las cuentas necesarias para crear la cuenta del contador
y luego escribe los valores iniciales de la cuenta. La estructura
_rs`#[derive(Accounts)]`_ utiliza las
[restricciones de cuenta](https://www.anchor-lang.com/docs/references/account-constraints)
de Anchor para especificar qué cuentas son requeridas y cómo se crea la nueva
cuenta del contador.

<WithNotes>

```rust title="programs/my-program/src/instructions/initialize.rs" -w
use anchor_lang::prelude::*;

use crate::{constants::*, state::Counter};

// !tooltip[/derive\(Accounts\)/] accountsStruct
#[derive(Accounts)]
pub struct Initialize<'info> {
    // !tooltip[/#\[account\(mut\)\]/] payerMutable
    #[account(mut)]
    // !tooltip[/Signer<'info>/] payerSigner
    pub payer: Signer<'info>,
    // !tooltip[/#\[account\(/] counterConstraints
    #[account(
        // !tooltip[/init/] initConstraint
        init,
        // !tooltip[/payer = payer/] payerConstraint
        payer = payer,
        // !tooltip[/space = 8 \+ Counter::INIT_SPACE/] space
        space = 8 + Counter::INIT_SPACE,
        // !tooltip[/seeds = \[COUNTER_SEED\]/] seeds
        seeds = [COUNTER_SEED],
        // !tooltip[/bump/] bump
        bump
    )]
    // !tooltip[/Account<'info, Counter>/] counterAccountType
    pub counter: Account<'info, Counter>,
    // !tooltip[/Program<'info, System>/] systemProgram
    pub system_program: Program<'info, System>,
}

// !tooltip[/handle_initialize/] handler
pub fn handle_initialize(ctx: Context<Initialize>) -> Result<()> {
    // !tooltip[/counter.count = 0/] initialCount
    ctx.accounts.counter.count = 0;
    // !tooltip[/counter.authority = ctx.accounts.payer.key\(\)/] initialAuthority
    ctx.accounts.counter.authority = ctx.accounts.payer.key();

    // !tooltip[/system_program::Transfer/] cpiAccounts
    let cpi_accounts = anchor_lang::system_program::Transfer {
        from: ctx.accounts.payer.to_account_info(),
        to: ctx.accounts.counter.to_account_info(),
    };
    // !tooltip[/CpiContext::new/] cpiContext
    let cpi_ctx = CpiContext::new(anchor_lang::system_program::ID, cpi_accounts);
    // !tooltip[/system_program::transfer/] transfer
    anchor_lang::system_program::transfer(cpi_ctx, HELLO_WORLD_LAMPORTS)?;

    // !tooltip[/msg!/] programLog
    msg!("Hello, world! Counter initialized");
    Ok(())
}
```

## !!steps Contexto de Cuenta

La estructura _rs`Initialize`_ define las cuentas que deben incluirse cuando un
usuario llama a la instrucción _rs`initialize`_. Anchor verifica estas cuentas
antes de que se ejecute el manejador.

<CodePlaceholder title="initialize.rs" />

```rust !! title="initialize.rs"
// !focus(1:2)
#[derive(Accounts)]
pub struct Initialize<'info> {
    #[account(mut)]
    pub payer: Signer<'info>,
    #[account(
        init,
        payer = payer,
        space = 8 + Counter::INIT_SPACE,
        seeds = [COUNTER_SEED],
        bump
    )]
    pub counter: Account<'info, Counter>,
    pub system_program: Program<'info, System>,
}
```

## !!steps Cuenta del Pagador

La cuenta _rs`payer`_ paga la creación de la cuenta del contador. El tipo
_rs`Signer<'info>`_ significa que el pagador debe firmar la transacción, y
_rs`#[account(mut)]`_ indica que la cuenta del pagador puede modificarse porque
se deducirán lamports.

<CodePlaceholder title="initialize.rs" />

```rust !! title="initialize.rs"
#[derive(Accounts)]
pub struct Initialize<'info> {
    // !focus(1:2)
    #[account(mut)]
    pub payer: Signer<'info>,
    #[account(
        init,
        payer = payer,
        space = 8 + Counter::INIT_SPACE,
        seeds = [COUNTER_SEED],
        bump
    )]
    pub counter: Account<'info, Counter>,
    pub system_program: Program<'info, System>,
}
```

## !!steps Cuenta del Contador

La cuenta _rs`counter`_ almacena los datos _rs`Counter`_ de `state.rs`.
_rs`init`_ le indica a Anchor que cree esta cuenta antes de que se ejecute el
manejador, y _rs`payer = payer`_ le indica a Anchor qué cuenta paga por la
creación.

<CodePlaceholder title="initialize.rs" />

```rust !! title="initialize.rs"
#[derive(Accounts)]
pub struct Initialize<'info> {
    #[account(mut)]
    pub payer: Signer<'info>,
    // !focus(1:8)
    #[account(
        init,
        payer = payer,
        space = 8 + Counter::INIT_SPACE,
        seeds = [COUNTER_SEED],
        bump
    )]
    pub counter: Account<'info, Counter>,
    pub system_program: Program<'info, System>,
}
```

## !!steps Tamaño de la Cuenta

La restricción _rs`space`_ le indica a Anchor cuántos datos de cuenta debe
asignar. Anchor almacena primero un discriminador de 8 bytes, luego los bytes
necesarios para los campos _rs`Counter`_. El discriminador permite que Anchor
reconozca esta cuenta como una cuenta _rs`Counter`_ antes de deserializar los
datos de la cuenta.

<CodePlaceholder title="initialize.rs" />

```rust !! title="initialize.rs"
#[derive(Accounts)]
pub struct Initialize<'info> {
    #[account(mut)]
    pub payer: Signer<'info>,
    #[account(
        init,
        payer = payer,
        // !focus
        space = 8 + Counter::INIT_SPACE,
        seeds = [COUNTER_SEED],
        bump
    )]
```

### !accountsStruct

_rs`Initialize`_ es el contexto de cuentas para la instrucción _rs`initialize`_.

### !payerMutable

_rs`#[account(mut)]`_ le indica a Anchor que la cuenta del pagador debe ser
modificable. Esto es necesario porque tanto la creación de la cuenta como el CPI
de transferencia deducen lamports del pagador.

### !payerSigner

_rs`Signer<'info>`_ le indica a Anchor que el pagador debe firmar la
transacción.

### !counterConstraints

El bloque _rs`#[account(...)]`_ le indica a Anchor cómo crear y validar la
cuenta del contador antes de que se ejecute el handler.

### !initConstraint

_rs`init`_ le indica a Anchor que cree la cuenta _rs`counter`_ antes de que se
ejecute el handler. La nueva cuenta es propiedad de este programa, por lo que
este programa puede almacenar datos de _rs`Counter`_ en ella.

### !payerConstraint

_rs`payer = payer`_ especifica la cuenta _rs`payer`_ como pagadora de la
creación de la nueva cuenta del contador.

### !space

_rs`space = 8 + Counter::INIT_SPACE`_ reserva bytes para el discriminador de
cuenta de Anchor más los campos _rs`Counter`_ de `state.rs`. El discriminador
ayuda a Anchor a confirmar que la cuenta almacena el tipo esperado antes de
leerla.

### !seeds

_rs`seeds = [COUNTER_SEED]`_ indica que la dirección de la cuenta del contador
debe ser el PDA derivado de este seed y el ID del programa. Con un seed fijo,
hay un único PDA del contador para este programa. Para crear contadores
separados en un programa real, normalmente se añadiría otro seed, como la
dirección de un usuario.

### !bump

_rs`bump`_ le indica a Anchor que encuentre y valide el bump del PDA, que es el
byte adicional que hace que la dirección derivada sea válida como PDA.

### !counterAccountType

_rs`Account<'info, Counter>`_ le indica a Anchor que trate los datos de la
cuenta como el tipo _rs`Counter`_ de `state.rs`. Anchor también verifica que la
cuenta sea propiedad de este programa.

### !systemProgram

_rs`Program<'info, System>`_ requiere que el emisor proporcione el System
Program. Anchor lo utiliza para la restricción de creación de cuenta _rs`init`_.
El CPI de transferencia del handler también invoca el System Program.

### !handler

El handler se ejecuta después de que Anchor valide las cuentas y cree la cuenta
del contador, por lo que puede escribir de forma segura los datos iniciales de
la cuenta.

### !initialCount

Establece el campo _rs`count`_ en la cuenta del contador a _rs`0`_.

### !initialAuthority

### !cpiContext

_rs`CpiContext`_ empaqueta el programa de destino y las cuentas requeridas para

mutación.

### !mutateCount

Solo después de que ambas verificaciones pasen, el handler actualiza los datos
de la cuenta del contador. Si cualquiera de las verificaciones falla, la
transacción falla y la cuenta no se modifica.

### !programLog

_rs`msg!`_ escribe un mensaje en los registros del programa cuando este es
invocado.

</WithNotes>

## Archivo de prueba

`programs/my-program/tests/test_initialize.rs` es una prueba de integración en
Rust. No incia un validator local. En su lugar, carga el archivo compilado `.so`
en LiteSVM, construye transacciones que llaman al programa y lee la cuenta del
contador después de cada transacción. La prueba construye instrucciones para una
transacción de Solana especificando el ID del programa a invocar, proporcionando
instruction data y pasando las cuentas requeridas.

```rust title="programs/my-program/tests/test_initialize.rs" -w
use {
    anchor_lang::{
        prelude::Pubkey,
        solana_program::{instruction::Instruction, system_program},
        AccountDeserialize, InstructionData, ToAccountMetas,
    },
    // !tooltip[/litesvm::LiteSVM/] litesvm
    litesvm::LiteSVM,

    solana_signer::Signer,
    solana_transaction::versioned::VersionedTransaction,

#[test]
fn test_initialize() {
    // !tooltip[/let program_id/] setup
    let program_id = my_program::id();
    let payer = Keypair::new();
    // !tooltip[/Pubkey::find_program_address/] counterPda
    let counter = Pubkey::find_program_address(
        &[my_program::constants::COUNTER_SEED],
        &program_id,

    .0;

    let bytes = include_bytes!(concat!(

        "/../deploy/my_program.so"

    svm.add_program(program_id, bytes).unwrap();
    svm.airdrop(&payer.pubkey(), 1_000_000_000).unwrap();

    let instruction = Instruction::new_with_bytes(
        program_id,
        &my_program::instruction::Initialize {}.data(),

    let tx = VersionedTransaction::try_new(VersionedMessage::Legacy(msg), &[&payer]).unwrap();

    let res = svm.send_transaction(tx);

    let counter_account = svm.get_account(&counter).unwrap();
    let mut data: &[u8] = &counter_account.data;

    // !tooltip[/assert_eq!\(counter_state.count, 0\)/] initAssert
```

### !litesvm

_rs`LiteSVM`_ es una VM de Solana en proceso para pruebas. Permite que la prueba
registre el programa compilado, cree cuentas, envíe transacciones e inspeccione
el estado de las cuentas sin necesidad de iniciar un proceso de validator.

### !setup

La prueba obtiene la dirección del programa a partir del valor _rs`declare_id!`_
mediante _rs`my_program::id()`_ y crea un nuevo keypair de pagador para la
transacción de prueba.

### !counterPda

La dirección del contador se deriva del mismo _rs`COUNTER_SEED`_ utilizado por
el programa. Esto hace que la prueba apunte al PDA que `initialize.rs` espera.

### !programBytes

_rs`include_bytes!`_ lee el archivo `my_program.so` compilado generado por la
construcción. Esos bytes son el código ejecutable del programa que LiteSVM
ejecutará en la prueba.

### !addProgram

_rs`svm.add_program`_ registra los bytes del programa compilado en la dirección
del programa en LiteSVM. El airdrop le otorga al pagador suficiente SOL local
para pagar la transacción y la creación de la cuenta.

### !testInstruction

_rs`Instruction::new_with_bytes`_ crea una instrucción para invocar el programa.
Incluye el ID del programa, instruction data y las cuentas de la instrucción.

### !blockhash

Cada transacción de Solana necesita un blockhash reciente. En esta prueba,
LiteSVM proporciona un blockhash para usar en la transacción.

### !message

El mensaje contiene la instrucción, el pagador de comisiones y el blockhash.
Estos son los datos de la transacción que serán firmados.

### !transaction

_rs`VersionedTransaction::try_new`_ firma el mensaje con el keypair del pagador.
El pagador firma porque paga la comisión de la transacción y financia la nueva
cuenta del contador.

### !sendTx

_rs`svm.send_transaction`_ ejecuta la transacción en LiteSVM.

### !getCounter

Después de que se ejecuta la transacción, la prueba lee la cuenta del contador
directamente desde LiteSVM para poder inspeccionar los datos de la cuenta.

### !deserialize

La cuenta del contador almacena datos serializados.
_rs`Counter::try_deserialize`_ verifica el discriminador de cuenta de Anchor y
convierte los datos de la cuenta de vuelta al tipo _rs`Counter`_ desde
`state.rs`.

### !initAssert

Después de que _rs`initialize`_ se ejecuta correctamente, la prueba deserializa
la cuenta del contador y verifica que el estado inicial coincida con lo que
_rs`handle_initialize`_ escribió.

### !incrementAssert

Después de que _rs`increment`_ se ejecuta correctamente, la misma cuenta del
contador debería tener _rs`count`_ establecido en _rs`1`_ manteniendo la misma
autoridad.

</WithNotes>
