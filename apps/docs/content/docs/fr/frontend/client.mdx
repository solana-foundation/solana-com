`@solana/kit` fournit `createClient`, un petit hôte que vous étendez avec des
plugins. Chaque plugin ajoute une capacité — une connexion RPC, un signataire,
la planification des transactions — et le client composé prend en charge aussi
bien les scripts, les routes serveur que les interfaces utilisateur.

<Callout type="info">
  Vous développez une application React ? Le [guide
  React](/docs/frontend/react-hooks) encapsule ce même client avec
  `@solana/react` afin que les composants accèdent à l'état du portefeuille et
  aux données RPC via des hooks. Commencez par là et revenez ici lorsque vous
  avez besoin d'un contrôle plus fin.
</Callout>

## Installation

```terminal
$ npm install @solana/kit @solana/kit-plugin-rpc @solana/kit-plugin-signer
```

Utilisez le gestionnaire de paquets de votre choix. Kit est en v7+ et les
paquets `@solana/kit-plugin-*` sont en 0.13+.

## Créer un client

Composez des plugins avec `.use(...)`. Définissez d'abord un signataire, puis la
connexion RPC. `solanaDevnetRpc()` ajoute `client.rpc`, les abonnements, le
planificateur/exécuteur de transactions, ainsi que le helper
`client.sendTransaction` — de sorte qu'une application a rarement besoin de
câbler ces éléments elle-même.

```ts
import { createClient, generateKeyPairSigner } from "@solana/kit";
import { solanaDevnetRpc } from "@solana/kit-plugin-rpc";
import { signer } from "@solana/kit-plugin-signer";

const payer = await generateKeyPairSigner();

const client = createClient().use(signer(payer)).use(solanaDevnetRpc());

const slot = await client.rpc.getSlot().send();
console.log("current slot", slot);
```

Le plugin `signer` définit à la fois le payeur des frais et l'identité de
l'application. Le bundle RPC exige qu'un payeur soit défini au préalable, ce qui
explique pourquoi l'ordre est important.

### Variantes de signataire

`@solana/kit-plugin-signer` expose trois rôles afin que vous n'accordiez que ce
dont un client a besoin :

- `signer(x)` — définit à la fois `client.payer` et `client.identity`.
- `payer(x)` — payeur des frais uniquement.
- `identity(x)` — autorité de l'application uniquement.

Chacun dispose des variantes `*FromFile(path)` (Node), `generated*()`,
`generated*WithSol(amount)` et `airdrop*()` pour les tests et les scripts.

## Envoyer une transaction

Construisez des instructions avec un client de programme généré, puis
transmettez-les à `client.sendTransaction`. Le planificateur résout le
blockhash, estime le calcul, signe avec le payeur du client et soumet la
transaction.

```ts
import { getTransferSolInstruction } from "@solana-program/system";
import { address, lamports } from "@solana/kit";

const transfer = getTransferSolInstruction({
  source: client.payer,
  destination: address("Fke...address"),
  amount: lamports(10_000_000n) // 0.01 SOL
});

const { context } = await client.sendTransaction([transfer]);
console.log("submitted", context.signature);
```

`sendTransaction` se résout en un objet résultat ; la signature de la
transaction se trouve à `result.context.signature`. Il vérifie que le plan tient
dans une seule transaction et lève une exception dans le cas contraire —
utilisez `sendTransactions` lorsqu'un lot peut s'étendre sur plusieurs
transactions.

## Choisir un bundle RPC

`@solana/kit-plugin-rpc` fournit des préconfigurations de cluster :

- `solanaDevnetRpc()` — paramètres par défaut du devnet, airdrop activé
  (`client.airdrop`).
- `solanaMainnetRpc()` — mainnet ; l'airdrop est une erreur de compilation.
- `solanaLocalRpc()` — `http://127.0.0.1:8899` pour un validator local.
- `solanaRpc({ rpcUrl })` — n'importe quel endpoint.

```ts
import { solanaRpc } from "@solana/kit-plugin-rpc";

const client = createClient()
  .use(signer(payer))
  .use(solanaRpc({ rpcUrl: "https://your-rpc-provider.example" }));
```

### Composition granulaire

Les préconfigurations regroupent la connexion RPC, le planificateur, l'exécuteur
et les assistants d'envoi, de sorte que la plupart des applications n'ont jamais
besoin des éléments individuellement. N'utilisez les plugins granulaires que
pour personnaliser l'un d'eux — par exemple un planificateur personnalisé ou un
canal d'abonnement différent. Cette approche nécessite également
`@solana/kit-plugin-instruction-plan`, que les préconfigurations incluent sinon
automatiquement :

```ts
import {
  solanaRpcConnection,
  rpcAirdrop,
  rpcTransactionPlanner,
  rpcTransactionPlanExecutor
} from "@solana/kit-plugin-rpc";
import { planAndSendTransactions } from "@solana/kit-plugin-instruction-plan";

const client = createClient()
  .use(signer(payer))
  .use(solanaRpcConnection({ rpcUrl: "https://api.devnet.solana.com" }))
  .use(rpcAirdrop())
  .use(rpcTransactionPlanner())
  .use(rpcTransactionPlanExecutor())
  .use(planAndSendTransactions());
```

## Ajouter des clients de programme

Les packages `@solana-program/*` générés exposent des plugins client afin que
les appels de programme soient rattachés au client. Installez le programme dont
vous avez besoin et `.use()` son plugin :

```ts
import { tokenProgram } from "@solana-program/token";

const client = createClient()
  .use(signer(payer))
  .use(solanaDevnetRpc())
  .use(tokenProgram()); // adds client.token
```

<Callout type="warn" title="Plages de dépendances homologues">
  Certaines versions de `@solana-program/*` déclarent encore une dépendance
  homologue `@solana/kit ^6.x` alors que Kit est en 7.x. Elles fonctionnent avec
  Kit 7 — contournez la plage de dépendances homologues dans votre gestionnaire
  de paquets si celui-ci se plaint.
</Callout>

## Signature sans portefeuille navigateur

Vous exécutez un processus sans interface dans une route API, un worker ou un
script ? Signez avec un service de gestion de clés backend plutôt qu'avec un
keypair brut. Consultez [Keychain](/docs/tools/keychain) et
[Signature en production](/docs/core/transactions/signing-in-production).

## Modèles courants pour les développeurs Solana

- **Configuration partagée** : Construisez le client une seule fois dans un
  module central et importez-le dans les scripts, les routes serveur et votre
  interface utilisateur.
- **Lectures côté serveur** : Un client avec seulement `signer` + `solanaRpc`
  (sans plugin de portefeuille) lit les comptes et simule des transactions
  depuis le serveur.
- **Tests en local** : Remplacez le bundle RPC par `.use(litesvm())` depuis
  `@solana/kit-plugin-litesvm` pour exécuter sur un validator en mémoire (Node
  uniquement).
- **Testabilité** : Le client composé est facile à simuler — remplacez les
  réponses RPC ou les signataires sans avoir de portefeuille navigateur présent.
