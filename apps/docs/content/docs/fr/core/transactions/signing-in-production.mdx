Une transaction ne s'exécute que si elle contient une
[signature](/docs/core/transactions/transaction-structure#signatures) valide de
chaque signataire requis. Lors du développement local, vous détenez généralement
cette clé sous forme de [`Keypair`](/developers/cookbook/wallets/create-keypair)
et signez directement. En production, la question importante est différente :
**où réside la clé privée, et qui est autorisé à signer avec elle ?**

<Callout type="warn">
  N'intégrez jamais de clés privées dans du code côté client, ne les incluez pas
  dans un frontend, et ne les commitez pas dans un système de contrôle de
  version. Toute personne qui obtient une clé acquiert le contrôle total des
  fonds de ce compte.
</Callout>

## Approches de signature

| Approche                       | Où réside la clé                                                | Qui signe                                      | Idéal pour                                                             |
| ------------------------------ | --------------------------------------------------------------- | ---------------------------------------------- | ---------------------------------------------------------------------- |
| **keypair local**              | Un fichier ou une variable d'environnement sur la machine       | Votre code, directement                        | Développement local, tests, CI                                         |
| **Portefeuille de navigateur** | Le portefeuille de l'utilisateur final (ex. Phantom, Solflare)  | L'utilisateur final, par transaction           | dApps frontend où les utilisateurs signent leurs propres transactions  |
| **KMS / HSM cloud**            | Un service de clés à support matériel (AWS KMS, GCP KMS, Vault) | Votre backend demande une signature au service | Services backend, opérations de trésorerie, environnements réglementés |
| **Portefeuilles gérés / MPC**  | Répartis ou en garde chez l'infrastructure d'un fournisseur     | Le fournisseur co-signe selon votre politique  | Portefeuilles embarqués, flux d'approbation, garde institutionnelle    |

Le keypair local est la seule approche qui place le matériel de clé brut
directement dans votre application. Toutes les approches en production
conservent la clé dans une infrastructure dédiée et demandent à cette
infrastructure de signer en votre nom.

## Signature côté serveur

Lorsqu'un serveur doit signer — payer des frais, envoyer des transactions de
programme ou gérer une trésorerie — utilisez un backend de gestion de clés dédié
plutôt qu'un keypair brut. La Solana Foundation maintient
[Keychain](/docs/tools/keychain), une bibliothèque de signature unifiée (Rust et
TypeScript) qui expose une seule interface `SolanaSigner` compatible avec tous
les backends du tableau ci-dessus : Memory, HashiCorp Vault, AWS KMS, GCP KMS,
Fireblocks, Privy, Turnkey, CDP, Crossmint, Dfns, Para, Openfort et Utila.

L'interface étant identique pour tous les backends, vous pouvez développer
localement avec une clé en mémoire et basculer vers un backend de production via
la configuration, sans réécrire le code de votre application. Keychain est
compatible avec [`@solana/kit`](/docs/clients/official/javascript) et le
[SDK Rust](/docs/clients/official/rust).
