Deze quickstart gebruikt het
[Anchor framework](https://www.anchor-lang.com/docs) startersproject gegenereerd
door `anchor init`. Je maakt het project lokaal aan, voert de tests uit, bouwt
het programma en doorloopt de programmacode die Anchor generiert.

## Vereisten

Voordat je begint,
[installeer de Solana-ontwikkeltools](/docs/intro/installation). De installatie
omvat Rust, de Solana CLI en de Anchor CLI.

Gebruik Anchor CLI versie 1.1.2 of hoger voor dit sjabloon. Controleer je
geïnstalleerde versie:

```terminal
$ anchor --version
```

<Steps>

<Step>

### Maak het project aan

Voer de volgende opdrachten uit in je terminal:

```terminal
$ anchor init my-program
$ cd my-program
```

Het startersproject bevat één Solana-programma onder `programs/my-program`. Het
programma bevat twee instructies: één om een telleraccount te initialiseren en
één om de teller te verhogen.

Sommige onderdelen van het sjabloon demonstreren veelvoorkomende
Solana-programmapatronen: het afleiden van
[PDA](/docs/core/pda)-accountadressen, het uitvoeren van een Cross Program
Invocation ([CPI](/docs/core/cpi)) om SOL over te maken, en het gebruik van
aangepaste foutcontroles om een instructie te stoppen wanneer een conditie
mislukt.

### Programma Bouwen

Voer `anchor build` uit om het startersprogramma te compileren:

```terminal
$ anchor build
```

Het gecompileerde programma wordt geschreven naar `target/deploy/my_program.so`.
Wanneer het programma wordt uitgerold, wordt de inhoud van dit `.so`-bestand
opgeslagen in een account onchain.

</Step>

<Step>

### Test uitvoeren

Voer de standaardtest uit:

```terminal
$ anchor test
```

De `Anchor.toml` van dit sjabloon gebruikt de Rust-testopdracht:

```toml title="Anchor.toml"
skip_local_validator = true

[scripts]
test = "cargo test"
```

De test laadt het gecompileerde programma in
[LiteSVM](https://www.litesvm.com/), maakt een betaler aan, verstuurt de
_rs`initialize`_ en _rs`increment`_ instructies, en controleert vervolgens de
status van het telleraccount.

Het uitvoeren van `anchor test` compileert het programma ook, dus u hoeft
`anchor build` niet eerst uit te voeren bij lokaal testen.

</Step>

<Step>

### Programma implementeren

Lokale tests bieden de snelste feedbackcyclus. Wanneer u klaar bent om te
implementeren op een netwerk, bijvoorbeeld devnet, bouw dan eerst en
implementeer daarna op een cluster.

Voor het implementeren van een Solana-programma is SOL vereist, omdat het
programma wordt opgeslagen in een account, en het account moet betalen voor de
ruimte die het gebruikt. Op devnet kunt u gratis devnet SOL aanvragen via de
[Solana Faucet](https://faucet.solana.com/) of met de Solana CLI:

```terminal
$ solana airdrop 2 --url devnet
```

```terminal
$ anchor build
$ anchor deploy --provider.cluster devnet
```

</Step>

</Steps>

## Bronbestanden

De map `src` bevat het Solana-programma. De
[Programmastructuur](https://www.anchor-lang.com/docs/basics/program-structure)-documentatie
van Anchor legt de kernmacro's uit die hier worden gebruikt, waaronder
_rs`declare_id!`_, _rs`#[program]`_, _rs`#[derive(Accounts)]`_ en
_rs`#[account]`_. Dit gedeelte beschrijft de sjabloonbestanden.

### lib.rs

`lib.rs` is het toegangspunt van het programma. Het verbindt de bronbestanden,
definieert het programmaadres en definieert de programma-instructies die
gebruikers kunnen aanroepen.

<WithNotes>

```rust title="programs/my-program/src/lib.rs" -w
// !tooltip[/pub mod constants/] constantsMod
pub mod constants;
// !tooltip[/pub mod error/] errorMod
pub mod error;
// !tooltip[/pub mod instructions/] instructionsMod
pub mod instructions;
// !tooltip[/pub mod state/] stateMod
pub mod state;

// !tooltip[/anchor_lang::prelude/] prelude
use anchor_lang::prelude::*;

pub use constants::*;
pub use instructions::*;
pub use state::*;

// !tooltip[/declare_id!/] declareId
declare_id!("82sFkffP9wxwpyfZyeaKHH2chQoJPUGsJZSPi9mrUuXd");

// !tooltip[/#\[program\]/] programMacro
#[program]
pub mod my_program {
    use super::*;

    // !tooltip[/pub fn initialize/] publicInstructions
    pub fn initialize(ctx: Context<Initialize>) -> Result<()> {
        // !tooltip[/handle_initialize/] handlers
        crate::instructions::initialize::handle_initialize(ctx)
    }

    // !tooltip[/pub fn increment/] publicInstructions
    pub fn increment(ctx: Context<Increment>) -> Result<()> {
        // !tooltip[/handle_increment/] handlers
        crate::instructions::increment::handle_increment(ctx)
    }
}
```

### !constantsMod

_rs`pub mod constants`_ maakt `constants.rs` deel uit van de program crate. Dit
is waar de template gedeelde programmawaarden opslaat, zoals de PDA seed, het
overdrachtsbedrag en de maximale tellerwaarde.

### !errorMod

_rs`pub mod error`_ maakt `error.rs` deel uit van de program crate. Dit bestand
definieert de aangepaste fouten die het programma kan teruggeven wanneer een
instructie ongeldig is.

### !instructionsMod

_rs`pub mod instructions`_ maakt `instructions.rs` deel uit van de program
crate. Dat bestand fungeert als een index voor de instructiespecifieke bestanden
in `src/instructions/`.

### !stateMod

_rs`pub mod state`_ maakt `state.rs` deel uit van de program crate. Dit bestand
definieert de gegevensindeling voor accounts die eigendom zijn van dit
programma, inclusief het _rs`Counter`_ account.

### !prelude

_rs`anchor_lang::prelude::*`_ importeert de Anchor-typen en macro's die worden
gebruikt om een Solana-programma te definiëren, waaronder _rs`Context`_,
_rs`Result`_, _rs`Pubkey`_, _rs`Account`_, _rs`Signer`_, en
Anchor-accountattributen.

### !reexports

De _rs`pub use`_-regels maken publieke items uit de onderliggende modules
beschikbaar vanaf de crate-root. Dat houdt de instructie-ingangspunten kort,
zoals _rs`Context<Initialize>`_ in plaats van een langer modulepad.

### !declareId

_rs`declare_id!`_ definieert het adres van het programma. Dit moet overeenkomen
met het adres in `Anchor.toml`, het gedeployde program account, en alle
gebruikers of tests die het programma aanroepen.

### !programMacro

_rs`#[program]`_ instrueert Anchor om het Solana-programma-ingangspunt en de
instructie-verzendcode te genereren. Publieke functies in deze module worden de
instructies die gebruikers kunnen aanroepen.

### !publicInstructions

_rs`initialize`_ en _rs`increment`_ zijn de twee instructienamen die gebruikers
kunnen aanroepen. Elk ontvangt een _rs`Context<T>`_. Het type binnen
_rs`Context`_ definieert de accounts die vereist zijn door die instructie, en
Anchor valideert die accounts vóór de instructie wordt uitgevoerd.

### !handlers

De handleerfunctie bevat de logica voor de instructie.

</WithNotes>

### constants.rs

`constants.rs` bewaart gedeelde waarden op één plek. In dit sjabloon leidt
_rs`COUNTER_SEED`_ de teller-PDA af, wordt _rs`HELLO_WORLD_LAMPORTS`_
overgedragen tijdens initialisatie, en wordt _rs`MAX_COUNT`_ gecontroleerd vóór
het ophogen.

<WithNotes>

```rust title="programs/my-program/src/constants.rs" -w
use anchor_lang::prelude::*;

// !tooltip[/#\[constant\]/] anchorConstant
// !tooltip[/COUNTER_SEED/] counterSeed
#[constant]
pub const COUNTER_SEED: &[u8] = b"counter";

// !tooltip[/HELLO_WORLD_LAMPORTS/] helloLamports
#[constant]
pub const HELLO_WORLD_LAMPORTS: u64 = 1;

// !tooltip[/MAX_COUNT/] maxCount
#[constant]
pub const MAX_COUNT: u64 = 10;
```

### !anchorConstant

_rs`#[constant]`_ vertelt Anchor om de waarde op te nemen in de programma-IDL.
Dat maakt belangrijke programmaconstan­ten zichtbaar voor gegenereerde clients
en tooling.

### !counterSeed

_rs`COUNTER_SEED`_ is de byte-string die wordt gebruikt om de teller-PDA af te
leiden. Het programma en de test gebruiken beide dit seed zodat ze het eens zijn
over het accountadres zonder een aparte keypair op te slaan. Dit sjabloon
gebruikt één vast seed, zodat het programma éen teller-adres heeft. Echte
programma's nemen vaak het adres van een gebruiker of een andere identificator
op in de seeds om veel gerelateerde accounts aan te maken.

### !helloLamports

_rs`HELLO_WORLD_LAMPORTS`_ is de hoeveelheid lamports die van de betaler naar
het teller-account wordt overgedragen binnen _rs`initialize`_. Het is bewust
klein omdat het sjabloon een CPI naar het System Program demonstreert.

### !maxCount

_rs`MAX_COUNT`_ is de bovengrens die wordt gecontroleerd door _rs`increment`_.
Door het hier te bewaren heeft het programma, de tests en de gegenereerde IDL
één gedeelde naam voor de regel.

</WithNotes>

<SideBySide>

### !left

```rust title="initialize.rs" -w

    // ...
    #[account(
        init,
        payer = payer,
        space = 8 + Counter::INIT_SPACE,
        // !mark
        // !mention counter-seed
        seeds = [COUNTER_SEED],
        bump
    )]
    pub counter: Account<'info, Counter>,
    // ...
}

pub fn handle_initialize(ctx: Context<Initialize>) -> Result<()> {
    // ...

    // !mark
    // !mention hello-world-lamports
    anchor_lang::system_program::transfer(cpi_ctx, HELLO_WORLD_LAMPORTS)?;

    Ok(())
}
```

### !right

`initialize.rs` gebruikt twee constanten:

- [_rs`COUNTER_SEED`_](mention:counter-seed) leidt het PDA-adres van de teller
  af.
- [_rs`HELLO_WORLD_LAMPORTS`_](mention:hello-world-lamports) stelt het bedrag in
  dat van de betaler naar het telleraccount wordt overgemaakt.

</SideBySide>

<SideBySide>

### !left

```rust title="increment.rs" -w
pub fn handle_increment(ctx: Context<Increment>) -> Result<()> {
    // ...

    require!(
        // !mark
        // !mention max-count
        ctx.accounts.counter.count < MAX_COUNT,
        ErrorCode::CounterOverflow,
    );

    ctx.accounts.counter.count += 1;
    Ok(())
}
```

### !right

`increment.rs` gebruikt [_rs`MAX_COUNT`_](mention:max-count) als de bovengrens
van de teller. Als de huidige telling al op het maximum staat, geeft
_rs`require!`_ de waarde _rs`CounterOverflow`_ terug en worden de
accountgegevens niet gewijzigd.

</SideBySide>

### state.rs

`state.rs` definieert aangepaste gegevenstypen voor accounts die het programma
aanmaakt en beheert. Het programma definieert instructies om die gegevens aan te
maken, te initialiseren en bij te werken, maar de tellergegevens worden niet in
het programma zelf opgeslagen. Ze worden opgeslagen in een aparte account met
een eigen adres.

<WithNotes>

```rust title="programs/my-program/src/state.rs" -w
use anchor_lang::prelude::*;

// !tooltip[/#\[account\]/] accountMacro
#[account]
// !tooltip[/InitSpace/] initSpace
#[derive(InitSpace)]
pub struct Counter {
    // !tooltip[/count/] count
    pub count: u64,
    // !tooltip[/authority/] authority
    pub authority: Pubkey,
}
```

### !accountMacro

_rs`#[account]`_ vertelt Anchor dat deze struct gegevens zijn die kunnen worden
opgeslagen in een onchain account dat eigendom is van dit programma. Anchor
gebruikt dit om _rs`Counter`_-accounts te serialiseren, te deserialiseren en te
valideren.

### !initSpace

_rs`InitSpace`_ laat Anchor _rs`Counter::INIT_SPACE`_ berekenen: het aantal
bytes dat nodig is voor de velden van deze struct. De 8-byte
Anchor-accountdiscriminator wordt apart toegevoegd wanneer het account wordt
aangemaakt.

### !count

_rs`count`_ is de tellerwaarde die is opgeslagen in de accountgegevens.

### !authority

_rs`authority`_ slaat het adres op dat gemachtigd is om de teller te verhogen.
_rs`initialize`_ schrijft dit adres, en _rs`increment`_ vereist dat datzelfde
adres ondertekent.

</WithNotes>

<SideBySide>

### !left

```rust title="initialize.rs" -w
#[account(
    init,
    payer = payer,
    // !mark
    // !mention counter-space
    space = 8 + Counter::INIT_SPACE,
    seeds = [COUNTER_SEED],
    bump
)]
pub counter: Account<'info, Counter>,
// ...

pub fn handle_initialize(ctx: Context<Initialize>) -> Result<()> {
    // !mark(1:2)
    // !mention(1:2) initialize-fields
    ctx.accounts.counter.count = 0;
    ctx.accounts.counter.authority = ctx.accounts.payer.key();

    Ok(())
}
```

```rust title="increment.rs" -w
pub fn handle_increment(ctx: Context<Increment>) -> Result<()> {
    // ...

    // !mark
    // !mention increment-count
    ctx.accounts.counter.count += 1;
    Ok(())
}
```

### !right

`state.rs` definieert de _rs`Counter`_ accountgegevens. De instructiebestanden
gebruiken dat type wanneer ze het account aanmaken en bijwerken:

- [_rs`Counter::INIT_SPACE`_](mention:counter-space) bepaalt de grootte van het
  account voor de velden gedefinieerd in `state.rs`.
- [_rs`count`_ en _rs`authority`_](mention:initialize-fields) zijn de veld-
  waarden die worden geschreven wanneer het account wordt geïnitialiseerd.
- [_rs`count += 1`_](mention:increment-count) werkt de opgeslagen tellerwaarde
  bij nadat de validatie is geslaagd.

</SideBySide>

### error.rs

`error.rs` definieert de aangepaste fouten van het programma. In dit sjabloon
laten de fouten zien hoe instructie-handlers stoppen wanneer een aanroeper niet
bevoegd is om de teller bij te werken of de teller al _rs`MAX_COUNT`_ heeft
bereikt.

<WithNotes>

```rust title="programs/my-program/src/error.rs" -w
use anchor_lang::prelude::*;

// !tooltip[/#\[error_code\]/] errorCode
#[error_code]
pub enum ErrorCode {
    // !tooltip[/Unauthorized/] unauthorized
    #[msg("Only the counter authority can update this counter")]
    Unauthorized,
    // !tooltip[/CounterOverflow/] counterOverflow
    #[msg("Counter has reached the maximum value")]
    CounterOverflow,
}
```

### !errorCode

_rs`#[error_code]`_ vertelt Anchor om deze varianten om te zetten in aangepaste
programmafouten. Clients en tests kunnen die fouten gebruiken in plaats van te
raden op basis van ruwe foutcodes.

### !unauthorized

_rs`Unauthorized`_ wordt geretourneerd wanneer de transactieondertekenaar niet
de authority is die is opgeslagen in het telleraccount.

### !counterOverflow

_rs`CounterOverflow`_ wordt geretourneerd wanneer verhogen de
_rs`MAX_COUNT`_-regel van het programma zou schenden.

</WithNotes>

<SideBySide>

### !left

```rust title="increment.rs" -w
pub fn handle_increment(ctx: Context<Increment>) -> Result<()> {
    require_keys_eq!(
        ctx.accounts.counter.authority,
        ctx.accounts.authority.key(),
        // !mark
        // !mention unauthorized-error
        ErrorCode::Unauthorized,
    );
    require!(
        ctx.accounts.counter.count < MAX_COUNT,
        // !mark
        // !mention overflow-error
        ErrorCode::CounterOverflow,
    );

    ctx.accounts.counter.count += 1;
    msg!("Hello, world! Counter is now {}", ctx.accounts.counter.count);
    Ok(())
}
```

### !right

`error.rs` benoemt de fouten die de instructie kan retourneren:

- [_rs`ErrorCode::Unauthorized`_](mention:unauthorized-error) wordt
  geretourneerd wanneer de ondertekenaar niet de authority is die is opgeslagen
  in het telleraccount.
- [_rs`ErrorCode::CounterOverflow`_](mention:overflow-error) wordt geretourneerd
  wanneer de teller al _rs`MAX_COUNT`_ heeft bereikt.

</SideBySide>

`instructions.rs` verbindt de instructiebestanden met de programma-crate zodat

_rs`increment`_-instructiecode. Elk instructiebestand definieert de accounts die

De _rs`pub mod`_-regels maken `src/instructions/initialize.rs` en

### initialize.rs

`initialize.rs` definieert de accounts die nodig zijn om het tegenaccount aan te
maken, en schrijft vervolgens de eerste waarden van het account. De
_rs`#[derive(Accounts)]`_-struct gebruikt Anchor
[account constraints](https://www.anchor-lang.com/docs/references/account-constraints)
om aan te geven welke accounts vereist zijn en hoe het nieuwe tegenaccount wordt
aangemaakt.

<WithNotes>

```rust title="programs/my-program/src/instructions/initialize.rs" -w
use anchor_lang::prelude::*;

use crate::{constants::*, state::Counter};

// !tooltip[/derive\(Accounts\)/] accountsStruct
#[derive(Accounts)]
pub struct Initialize<'info> {
    // !tooltip[/#\[account\(mut\)\]/] payerMutable
    #[account(mut)]
    // !tooltip[/Signer<'info>/] payerSigner
    pub payer: Signer<'info>,
    // !tooltip[/#\[account\(/] counterConstraints
    #[account(
        // !tooltip[/init/] initConstraint
        init,
        // !tooltip[/payer = payer/] payerConstraint
        payer = payer,
        // !tooltip[/space = 8 \+ Counter::INIT_SPACE/] space
        space = 8 + Counter::INIT_SPACE,
        // !tooltip[/seeds = \[COUNTER_SEED\]/] seeds
        seeds = [COUNTER_SEED],
        // !tooltip[/bump/] bump
        bump
    )]
    // !tooltip[/Account<'info, Counter>/] counterAccountType
    pub counter: Account<'info, Counter>,
    // !tooltip[/Program<'info, System>/] systemProgram
    pub system_program: Program<'info, System>,
}

// !tooltip[/handle_initialize/] handler
pub fn handle_initialize(ctx: Context<Initialize>) -> Result<()> {
    // !tooltip[/counter.count = 0/] initialCount
    ctx.accounts.counter.count = 0;
    // !tooltip[/counter.authority = ctx.accounts.payer.key\(\)/] initialAuthority
    ctx.accounts.counter.authority = ctx.accounts.payer.key();

    // !tooltip[/system_program::Transfer/] cpiAccounts
    let cpi_accounts = anchor_lang::system_program::Transfer {
        from: ctx.accounts.payer.to_account_info(),
        to: ctx.accounts.counter.to_account_info(),
    };
    // !tooltip[/CpiContext::new/] cpiContext
    let cpi_ctx = CpiContext::new(anchor_lang::system_program::ID, cpi_accounts);
    // !tooltip[/system_program::transfer/] transfer
    anchor_lang::system_program::transfer(cpi_ctx, HELLO_WORLD_LAMPORTS)?;

    // !tooltip[/msg!/] programLog
    msg!("Hello, world! Counter initialized");
    Ok(())
}
```

## !!steps Account Context

De _rs`Initialize`_-struct definieert de accounts die moeten worden meegestuurd
wanneer een gebruiker de _rs`initialize`_-instructie aanroept. Anchor

```rust !! title="initialize.rs"
pub struct Initialize<'info> {

        payer = payer,
        space = 8 + Counter::INIT_SPACE,
        seeds = [COUNTER_SEED],
        bump

    pub counter: Account<'info, Counter>,
    pub system_program: Program<'info, System>,
}
```

## !!steps Payer Account

Het _rs`payer`_-account betaalt voor het aanmaken van het tegenaccount. Het
_rs`Signer<'info>`_-type betekent dat de betaler de transactie moet
ondertekenen, en _rs`#[account(mut)]`_ betekent dat het betaleraccount kan
worden gewijzigd omdat er lamports zullen worden afgetrokken.

<CodePlaceholder title="initialize.rs" />

```rust !! title="initialize.rs"

pub struct Initialize<'info> {
    // !focus(1:2)
    #[account(mut)]
    pub payer: Signer<'info>,

        init,
        payer = payer,
        space = 8 + Counter::INIT_SPACE,
        seeds = [COUNTER_SEED],
        bump

    pub counter: Account<'info, Counter>,
    pub system_program: Program<'info, System>,
```

## !!steps Counter Account

Het _rs`counter`_-account slaat de _rs`Counter`_-gegevens op uit `state.rs`.
_rs`init`_ vertelt Anchor dit account aan te maken vóór de handler wordt
uitgevoerd, en _rs`payer = payer`_ vertelt Anchor welk account de aanmaak
betaalt.

<CodePlaceholder title="initialize.rs" />

```rust !! title="initialize.rs"
#[derive(Accounts)]
pub struct Initialize<'info> {
    #[account(mut)]
    pub payer: Signer<'info>,
    // !focus(1:8)
    #[account(
        init,
        payer = payer,
        space = 8 + Counter::INIT_SPACE,
        seeds = [COUNTER_SEED],
        bump
    )]
    pub counter: Account<'info, Counter>,
    pub system_program: Program<'info, System>,
}
```

## !!steps Accountgrootte

De _rs`space`_ constraint vertelt Anchor hoeveel accountdata er gealloceerd moet
worden. Anchor slaat eerst een 8-byte discriminator op, gevolgd door de bytes
die nodig zijn voor de _rs`Counter`_ velden. De discriminator stelt Anchor in
staat dit account te herkennen als een _rs`Counter`_ account voordat de
accountdata gedeserialiseerd wordt.

<CodePlaceholder title="initialize.rs" />

```rust !! title="initialize.rs"
#[derive(Accounts)]
pub struct Initialize<'info> {
    #[account(mut)]
    pub payer: Signer<'info>,
```

### !imports

Dit bestand importeert de gedeelde constanten en het _rs`Counter`_-accounttype.
De instructie gebruikt deze definities om de PDA af te leiden, het
overdrachtsbedrag te kiezen en het account te alloceren met de juiste
gegevensgrootte.

### !accountsStruct

_rs`Initialize`_ is de accountcontext voor de _rs`initialize`_-instructie. Elk
veld in de struct vertegenwoordigt een afzonderlijk account dat moet worden
meegestuurd wanneer de instructie wordt aangeroepen. Anchor gebruikt de struct
om die inkomende accounts te valideren voordat de handler wordt uitgevoerd.

### !payerMutable

_rs`#[account(mut)]`_ vertelt Anchor dat het betalersaccount schrijfbaar moet
zijn. Dit is vereist omdat zowel het aanmaken van accounts als de transfer CPI
lamports van de betaler aftrekken.

### !payerSigner

_rs`Signer<'info>`_ vertelt Anchor dat de betaler de transactie moet
ondertekenen.

### !counterConstraints

Het _rs`#[account(...)]`_-blok vertelt Anchor hoe het teileraccount aangemaakt
en gevalideerd moet worden voordat de handler wordt uitgevoerd.

### !initConstraint

_rs`init`_ vertelt Anchor om het _rs`counter`_ account aan te maken voordat de
handler wordt uitgevoerd. Het nieuwe account is eigendom van dit programma,
zodat dit programma _rs`Counter`_ gegevens erin kan opslaan.

### !payerConstraint

_rs`payer = payer`_ specificeert het _rs`payer`_ account als betaler voor het
aanmaken van het nieuwe counter account.

### !space

_rs`space = 8 + Counter::INIT_SPACE`_ reserveert bytes voor Anchor's account
discriminator plus de _rs`Counter`_ velden uit `state.rs`. De discriminator
helpt Anchor bevestigen dat het account het verwachte type opslaat voordat het
wordt gelezen.

### !seeds

_rs`seeds = [COUNTER_SEED]`_ bepaalt dat het adres van het counter account de
PDA moet zijn die is afgeleid van deze seed en het programma-ID. Met één vaste
seed is er één counter PDA voor dit programma. Om afzonderlijke tellers aan te
maken in een echt programma, zou je gewoonlijk een andere seed toevoegen, zoals
het adres van een gebruiker.

### !bump

_rs`bump`_ vertelt Anchor om de PDA bump te vinden en te valideren. Dit is de
extra byte die het afgeleide adres geldig maakt als PDA.

### !counterAccountType

_rs`Account<'info, Counter>`_ vertelt Anchor om de accountgegevens te behandelen
als het _rs`Counter`_ type uit `state.rs`. Anchor controleert ook of het account
eigendom is van dit programma.

### !systemProgram

_rs`Program<'info, System>`_ vereist dat de aanroeper het System Program
aanlevert. Anchor gebruikt dit voor de _rs`init`_ account aanmaakbeperking. De
transfer CPI van de handler roept ook het System Program aan.

### !handler

De handler wordt uitgevoerd nadat Anchor de accounts heeft gevalideerd en het
counter account heeft aangemaakt, zodat de initiële gegevens van het account
veilig kunnen worden geschreven.

### !initialCount

Stelt het _rs`count`_ veld in het counter account in op _rs`0`_.

### !initialAuthority

Stelt het _rs`authority`_ veld in het counter account in op het adres van de
betaler.

### !cpiAccounts

Dit zijn de accounts die worden doorgegeven aan de System Program
transfer-instructie. Deze transfer CPI is alleen opgenomen om te demonstreren
hoe een CPI accounts doorgeeft aan een ander programma.

### !cpiContext

_rs`CpiContext`_ verpakt het doelprogramma en de vereiste accounts voor een
Cross Program Invocation. Het aanmaken van de context bereidt de CPI voor door
het programma dat je wilt aanroepen te koppelen aan de accounts die dat
programma verwacht.

### !transfer

Deze CPI roept de transfer-instructie van het System Program aan en verplaatst
_rs`HELLO_WORLD_LAMPORTS`_ van de betaler naar het telleraccount.

### !programLog

_rs`msg!`_ schrijft een bericht naar de programmalogboeken wanneer het programma
wordt aangeroepen.

tenzij het adres van de ondertekenaar overeenkomt met de autoriteit die is
opgeslagen in het teller-account. Dit is de autorisatieregel op applicatieniveau
van de template.

### !maxCheck

_rs`require!`_ controleert een booleaanse regel en stopt de instructie met
_rs`CounterOverflow`_ als de teller al _rs`MAX_COUNT`_ heeft bereikt. Deze
templateregel demonstreert aangepaste fouten en het mislukken vóór mutatie.

### !mutateCount

Pas nadat beide controles zijn geslaagd, werkt de handler de
teller-accountgegevens bij. Als een van de controles mislukt, mislukt de
transactie en wordt het account niet gewijzigd.

### !programLog

_rs`msg!`_ schrijft een bericht naar de programmalogboeken wanneer het programma
wordt aangeroepen.

</WithNotes>

## Testbestand

`programs/my-program/tests/test_initialize.rs` is een Rust-integratietest. Er
wordt geen lokale validator gestart. In plaats daarvan laadt het het
gecompileerde `.so`-bestand in LiteSVM, bouwt het transacties die het programma
aanroepen en leest het het teller-account na elke transactie. De test bouwt
instructies voor een Solana-transactie door het programma-ID van het aan te
roepen programma op te geven, instruction data te verstrekken en de vereiste
accounts door te geven.

```rust title="programs/my-program/tests/test_initialize.rs" -w
use {
    anchor_lang::{
        prelude::Pubkey,
        solana_program::{instruction::Instruction, system_program},
```

### !litesvm

_rs`LiteSVM`_ is een in-process Solana VM voor tests. Het stelt de test in staat
het gecompileerde programma te registreren, accounts aan te maken, transacties
te verzenden en de accountstatus te inspecteren zonder een validator-proces te
starten.

### !setup

De test haalt het programmaadres op uit de _rs`declare_id!`_ waarde via
_rs`my_program::id()`_ en maakt een nieuw payer keypair aan voor de
testtransactie.

### !counterPda

Het counteradres wordt afgeleid van dezelfde _rs`COUNTER_SEED`_ die door het
programma wordt gebruikt. Dit zorgt ervoor dat de test de PDA target die
`initialize.rs` verwacht.

### !programBytes

_rs`include_bytes!`_ leest het gecompileerde `my_program.so`-bestand dat door de
build is gegenereerd. Die bytes zijn de uitvoerbare programmacode die LiteSVM in
de test zal uitvoeren.

### !addProgram

_rs`svm.add_program`_ registreert de gecompileerde programmabytes op het
programmaadres in LiteSVM. De airdrop geeft de betaler genoeg lokale SOL om de
transactie en het aanmaken van het account te betalen.

### !testInstruction

_rs`Instruction::new_with_bytes`_ maakt een instructie aan om het programma aan
te roepen. Het bevat de programma-ID, instruction data en instructieaccounts.

### !blockhash

Elke Solana-transactie heeft een recente blockhash nodig. In deze test levert
LiteSVM een blockhash voor gebruik bij de transactie.

### !message

Het bericht bevat de instructie, de vergoedingsbetaler en de blockhash. Dit is
de transactiedata die ondertekend zal worden.

### !transaction

_rs`VersionedTransaction::try_new`_ ondertekent het bericht met de payer
keypair. De betaler ondertekent omdat hij de transactievergoeding betaalt en het
nieuwe counter-account financiert.

### !sendTx

_rs`svm.send_transaction`_ voert de transactie uit in LiteSVM.

### !getCounter

Nadat de transactie is uitgevoerd, leest de test het counteraccount rechtstreeks
uit LiteSVM om de accountdata te inspecteren.

### !deserialize

Het counteraccount slaat geserialiseerde data op. _rs`Counter::try_deserialize`_
controleert de accountdiscriminator van Anchor en zet de accountdata terug om
naar het _rs`Counter`_-type uit `state.rs`.

### !initAssert

Nadat _rs`initialize`_ is geslaagd, deserialiseert de test het counteraccount en
controleert of de begintoestand overeenkomt met wat _rs`handle_initialize`_
heeft geschreven.

### !incrementAssert

Nadat _rs`increment`_ is geslaagd, moet hetzelfde counteraccount _rs`count`_
hebben ingesteld op _rs`1`_ terwijl de authority hetzelfde blijft.

</WithNotes>

### !skipValidator

`skip_local_validator = true` vertelt Anchor om geen validator te starten voor
`anchor test`. De Rust-test gebruikt in plaats daarvan LiteSVM.

### !programs

`[programs.localnet]` koppelt de naam van de Rust-crate aan het programmaadres
dat wordt gebruikt voor lokale builds en tests.

### !provider

`[provider]` stelt het standaardcluster en de standaardwallet in voor opdrachten
zoals `anchor deploy`.

### !scripts

`[scripts]` bepaalt wat `anchor test` uitvoert. In dit sjabloon voert
`anchor test` `cargo test` uit.

</WithNotes>
