`@solana/kit` levert `createClient`, een kleine host die je uitbreidt met
plugins. Elke plugin voegt een mogelijkheid toe — een RPC-verbinding, een
ondertekenaar, transactieplanning — en de samengestelde client ondersteunt
scripts, serverroutes en UI's gelijkelijk.

<Callout type="info">
  Een React-app bouwen? De [React-gids](/docs/frontend/react-hooks) omhult
  dezelfde client met `@solana/react` zodat componenten walletstatus en
  RPC-gegevens via hooks kunnen lezen. Begin daar en kom hier terug wanneer je
  meer controle nodig hebt.
</Callout>

## Installeren

```terminal
$ npm install @solana/kit @solana/kit-plugin-rpc @solana/kit-plugin-signer
```

Gebruik een willekeurige pakketbeheerder. Kit is v7+ en de
`@solana/kit-plugin-*`-pakketten zijn 0.13+.

## Een client aanmaken

Combineer plugins met `.use(...)`. Stel eerst een ondertekenaar in, dan de
RPC-verbinding. `solanaDevnetRpc()` voegt `client.rpc`, abonnementen, de
transactie- planner/uitvoerder en de hulpfunctie `client.sendTransaction` toe —
zodat een app die onderdelen zelden zelf hoeft te verbinden.

```ts
import { createClient, generateKeyPairSigner } from "@solana/kit";
import { solanaDevnetRpc } from "@solana/kit-plugin-rpc";
import { signer } from "@solana/kit-plugin-signer";

const payer = await generateKeyPairSigner();

const client = createClient().use(signer(payer)).use(solanaDevnetRpc());

const slot = await client.rpc.getSlot().send();
console.log("current slot", slot);
```

De plugin `signer` stelt zowel de vergoedingsbetaler als de app-identiteit in.
Het RPC-pakket vereist dat er eerst een betaler aanwezig is, daarom is de
volgorde van belang.

### Ondertekenaarsvarianten

`@solana/kit-plugin-signer` biedt drie rollen, zodat je alleen verleent wat een
client nodig heeft:

- `signer(x)` — stelt zowel `client.payer` als `client.identity` in.
- `payer(x)` — alleen vergoedingsbetaler.
- `identity(x)` — alleen app-autoriteit.

Elk heeft varianten `*FromFile(path)` (Node), `generated*()`,
`generated*WithSol(amount)` en `airdrop*()` voor tests en scripts.

## Een transactie versturen

Bouw instructies met een gegenereerde programma-client en geef ze door aan
`client.sendTransaction`. De planner lost de blockhash op, schat de rekenkracht
in, ondertekent met de betaler van de client en verzendt de transactie.

```ts
import { getTransferSolInstruction } from "@solana-program/system";
import { address, lamports } from "@solana/kit";

const transfer = getTransferSolInstruction({
  source: client.payer,
  destination: address("Fke...address"),
  amount: lamports(10_000_000n) // 0.01 SOL
});

const { context } = await client.sendTransaction([transfer]);
console.log("submitted", context.signature);
```

`sendTransaction` wordt omgezet naar een resultaatobject; de
transactiehandtekening bevindt zich op `result.context.signature`. Het
controleert of het plan in één transactie past en gooit een fout als dat niet
het geval is — gebruik `sendTransactions` wanneer een batch meerdere transacties
kan omvatten.

## Kies een RPC-bundel

`@solana/kit-plugin-rpc` levert cluster-presets:

- `solanaDevnetRpc()` — devnet-standaarden, airdrop ingeschakeld
  (`client.airdrop`).
- `solanaMainnetRpc()` — mainnet; airdrop is een compilatiefout.
- `solanaLocalRpc()` — `http://127.0.0.1:8899` voor een lokale validator.
- `solanaRpc({ rpcUrl })` — elk eindpunt.

```ts
import { solanaRpc } from "@solana/kit-plugin-rpc";

const client = createClient()
  .use(signer(payer))
  .use(solanaRpc({ rpcUrl: "https://your-rpc-provider.example" }));
```

### Granulair samenstellen

De presets bundelen de RPC-verbinding, planner, executor en verzendhelpers,
zodat de meeste apps de afzonderlijke onderdelen nooit nodig hebben. Gebruik de
granulaire plugins alleen om één onderdeel aan te passen — bijvoorbeeld een
aangepaste planner of een ander subscriptiekanaal. Dit pad vereist ook
`@solana/kit-plugin-instruction-plan`, dat de presets anders voor je meenemen:

```ts
import {
  solanaRpcConnection,
  rpcAirdrop,
  rpcTransactionPlanner,
  rpcTransactionPlanExecutor
} from "@solana/kit-plugin-rpc";
import { planAndSendTransactions } from "@solana/kit-plugin-instruction-plan";

const client = createClient()
  .use(signer(payer))
  .use(solanaRpcConnection({ rpcUrl: "https://api.devnet.solana.com" }))
  .use(rpcAirdrop())
  .use(rpcTransactionPlanner())
  .use(rpcTransactionPlanExecutor())
  .use(planAndSendTransactions());
```

## Programmaclients toevoegen

Gegenereerde `@solana-program/*`-pakketten bieden client-plugins zodat
programma-aanroepen aan de client hangen. Installeer het gewenste programma en
`.use()` zijn plugin:

```ts
import { tokenProgram } from "@solana-program/token";

const client = createClient()
  .use(signer(payer))
  .use(solanaDevnetRpc())
  .use(tokenProgram()); // adds client.token
```

<Callout type="warn" title="Peer dependency ranges">
  Sommige `@solana-program/*`-releases declareren nog steeds een `@solana/kit
  ^6.x` peer terwijl Kit op versie 7.x zit. Ze werken met Kit 7 — overschrijf
  het peer-bereik in je pakketbeheerder als die klaagt.
</Callout>

## Ondertekenen zonder een browserwallet

Draait u headless in een API-route, worker of script? Onderteken met een backend
sleutelbeheerservice in plaats van een ruwe keypair. Zie
[Keychain](/docs/tools/keychain) en
[Ondertekenen in productie](/docs/core/transactions/signing-in-production).

## Veelvoorkomende patronen voor Solana-ontwikkelaars

- **Gedeelde setup**: Bouw de client eenmalig in een kernmodule en importeer hem
  in scripts, serverroutes en uw UI.
- **Server-side reads**: Een client met alleen `signer` + `solanaRpc` (zonder
  wallet- plugin) leest accounts en simuleert transacties vanaf de server.
- **Lokaal testen**: Vervang de RPC-bundel door `.use(litesvm())` van
  `@solana/kit-plugin-litesvm` om tegen een in-memory validator te draaien
  (alleen Node).
- **Testbaarheid**: De samengestelde client is eenvoudig te mocken — stub
  RPC-responses of ondertekenaars zonder dat er een browserwallet aanwezig is.
