`@solana/kit` liefert `createClient`, einen kleinen Host, den du mit Plugins
erweiterst. Jedes Plugin fügt eine Funktion hinzu — eine RPC-Verbindung, einen
Signer, Transaktionsplanung — und der zusammengesetzte Client unterstützt
gleichermaßen Skripte, Server-Routen und UIs.

<Callout type="info">
  Du baust eine React-App? Der [React-Leitfaden](/docs/frontend/react-hooks)
  kapselt denselben Client mit `@solana/react`, sodass Komponenten den
  Wallet-Status und RPC-Daten über Hooks lesen. Starte dort und kehre hierher
  zurück, wenn du mehr Kontrolle benötigst.
</Callout>

## Installation

```terminal
$ npm install @solana/kit @solana/kit-plugin-rpc @solana/kit-plugin-signer
```

Verwende einen beliebigen Paketmanager. Kit ist v7+ und die
`@solana/kit-plugin-*`-Pakete sind 0.13+.

## Einen Client erstellen

Kombiniere Plugins mit `.use(...)`. Lege zuerst einen Signer fest, dann die
RPC-Verbindung. `solanaDevnetRpc()` fügt `client.rpc`, Subscriptions, den
Transaktions- Planer/Executor und den `client.sendTransaction`-Helfer hinzu —
sodass eine App diese Komponenten selten selbst verdrahten muss.

```ts
import { createClient, generateKeyPairSigner } from "@solana/kit";
import { solanaDevnetRpc } from "@solana/kit-plugin-rpc";
import { signer } from "@solana/kit-plugin-signer";

const payer = await generateKeyPairSigner();

const client = createClient().use(signer(payer)).use(solanaDevnetRpc());

const slot = await client.rpc.getSlot().send();
console.log("current slot", slot);
```

Das `signer`-Plugin legt sowohl den Fee-Zahler als auch die App-Identität fest.
Das RPC-Bundle erfordert, dass zuerst ein Zahler vorhanden ist, weshalb die
Reihenfolge wichtig ist.

### Signer-Varianten

`@solana/kit-plugin-signer` bietet drei Rollen, sodass du nur das gewährst, was
ein Client benötigt:

- `signer(x)` — legt sowohl `client.payer` als auch `client.identity` fest.
- `payer(x)` — nur Fee-Zahler.
- `identity(x)` — nur App-Autorität.

Jede hat `*FromFile(path)` (Node), `generated*()`, `generated*WithSol(amount)`
und `airdrop*()`-Varianten für Tests und Skripte.

## Eine Transaktion senden

Erstelle Anweisungen mit einem generierten Programm-Client und übergib sie dann
an `client.sendTransaction`. Der Planer löst den Blockhash auf, schätzt den
Rechenaufwand, signiert mit dem Zahler des Clients und übermittelt die
Transaktion.

```ts
import { getTransferSolInstruction } from "@solana-program/system";
import { address, lamports } from "@solana/kit";

const transfer = getTransferSolInstruction({
  source: client.payer,
  destination: address("Fke...address"),
  amount: lamports(10_000_000n) // 0.01 SOL
});

const { context } = await client.sendTransaction([transfer]);
console.log("submitted", context.signature);
```

`sendTransaction` löst sich zu einem Ergebnisobjekt auf; die
Transaktionssignatur befindet sich unter `result.context.signature`. Es wird
sichergestellt, dass der Plan in eine einzelne Transaktion passt, andernfalls
wird ein Fehler ausgelöst — verwende `sendTransactions`, wenn ein Batch mehrere
Transaktionen umfassen kann.

## Wähle ein RPC-Bundle

`@solana/kit-plugin-rpc` wird mit Cluster-Voreinstellungen geliefert:

- `solanaDevnetRpc()` — Devnet-Standardwerte, Airdrop aktiviert
  (`client.airdrop`).
- `solanaMainnetRpc()` — Mainnet; Airdrop ist ein Kompilierungsfehler.
- `solanaLocalRpc()` — `http://127.0.0.1:8899` für einen lokalen validator.
- `solanaRpc({ rpcUrl })` — beliebiger Endpunkt.

```ts
import { solanaRpc } from "@solana/kit-plugin-rpc";

const client = createClient()
  .use(signer(payer))
  .use(solanaRpc({ rpcUrl: "https://your-rpc-provider.example" }));
```

### Granular zusammenstellen

Die Voreinstellungen bündeln die RPC-Verbindung, den Planer, den Executor und
die Send-Hilfsfunktionen, sodass die meisten Apps die einzelnen Komponenten nie
separat benötigen. Wechsle nur dann zu den granularen Plugins, wenn du eine
Komponente anpassen möchtest — zum Beispiel einen eigenen Planer oder einen
anderen Subscription-Kanal. Dieser Weg erfordert außerdem
`@solana/kit-plugin-instruction-plan`, das die Voreinstellungen andernfalls für
dich einbinden:

```ts
import {
  solanaRpcConnection,
  rpcAirdrop,
  rpcTransactionPlanner,
  rpcTransactionPlanExecutor
} from "@solana/kit-plugin-rpc";
import { planAndSendTransactions } from "@solana/kit-plugin-instruction-plan";

const client = createClient()
  .use(signer(payer))
  .use(solanaRpcConnection({ rpcUrl: "https://api.devnet.solana.com" }))
  .use(rpcAirdrop())
  .use(rpcTransactionPlanner())
  .use(rpcTransactionPlanExecutor())
  .use(planAndSendTransactions());
```

## Programm-Clients hinzufügen

Generierte `@solana-program/*`-Pakete stellen Client-Plugins bereit, sodass
Programmaufrufe am Client hängen. Installiere das benötigte Programm und binde
sein Plugin mit `.use()` ein:

```ts
import { tokenProgram } from "@solana-program/token";

const client = createClient()
  .use(signer(payer))
  .use(solanaDevnetRpc())
  .use(tokenProgram()); // adds client.token
```

<Callout type="warn" title="Peer-Dependency-Bereiche">
  Einige `@solana-program/*`-Releases deklarieren noch einen `@solana/kit
  ^6.x`-Peer, während Kit bereits bei Version 7.x ist. Sie funktionieren mit Kit
  7 — überschreibe den Peer-Bereich in deinem Paketmanager, falls dieser sich
  beschwert.
</Callout>

## Signieren ohne Browser-Wallet

Läuft dein Code headless in einer API-Route, einem Worker oder einem Skript?
Signiere mit einem Backend- Key-Management-Service statt mit einem rohen
keypair. Siehe [Keychain](/docs/tools/keychain) und
[Signieren in der Produktion](/docs/core/transactions/signing-in-production).

## Häufige Muster für Solana-Entwickler

- **Gemeinsames Setup**: Erstelle den Client einmal in einem Kernmodul und
  importiere ihn in Skripte, Server-Routen und deine UI.
- **Serverseitige Lesevorgänge**: Ein Client mit nur `signer` + `solanaRpc`
  (ohne Wallet- Plugin) liest Konten und simuliert Transaktionen vom Server aus.
- **Lokales Testen**: Ersetze das RPC-Bundle durch `.use(litesvm())` aus
  `@solana/kit-plugin-litesvm`, um gegen einen In-Memory-validator zu testen
  (nur Node).
- **Testbarkeit**: Der zusammengestellte Client lässt sich leicht mocken — stub
  RPC-Antworten oder Signer ohne eine Browser-Wallet.
