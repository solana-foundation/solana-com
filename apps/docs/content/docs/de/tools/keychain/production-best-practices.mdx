Keychain abstrahiert das Signing-Backend, aber der Betrieb eines Signers im
Produktionsbetrieb liegt weiterhin in Ihrer Verantwortung. Diese Praktiken
gelten für alle Backends; siehe
[Backend auswählen](/docs/tools/keychain/choosing-a-backend) für die Auswahl
eines geeigneten Backends.

## Hot- und Cold-Signer trennen

Verwenden Sie einen Signer mit niedrigem Guthaben ("Hot") für häufige,
automatisierte Transaktionen und einen separaten High-Value-Signer ("Cold") für
Treasury-Bewegungen. Sichern Sie den Cold-Signer mit MPC-Verwahrung oder einem
Cloud-HSM ab und verlangen Sie Genehmigungsrichtlinien, bevor er signiert. Da
die Schnittstelle identisch ist, kann jede Rolle ein anderes Backend verwenden,
ohne Aufrufstellen zu ändern.

## Schlüssel gezielt rotieren

Bei Solana ist der Signing-Schlüssel _die_ Kontenadresse, daher erzeugt eine
Schlüsselrotation eine neue Adresse. Planen Sie die Rotation als Migration:
Stellen Sie einen neuen Schlüssel bereit, übertragen Sie die relevanten
Berechtigungen (oder verschieben Sie Guthaben) auf die neue Adresse und ziehen
Sie den alten Schlüssel dann zurück. Verwenden Sie pro Umgebung separate
Schlüssel, damit Devnet-Schlüssel niemals im Mainnet wiederverwendet werden.

## Least-Privilege-Zugriff gewähren

Beschränken Sie die Anmeldeinformationen jedes Backends auf das Minimum, das zum
Signieren erforderlich ist. Für AWS KMS sind das `kms:Sign` und
`kms:DescribeKey` für den jeweiligen Schlüssel; bevorzugen Sie IAM-Rollen
gegenüber statischen Anmeldeinformationen. Wenden Sie dasselbe Prinzip auf
Vault-Richtlinien und verwaltete Wallet-API-Schlüssel an.

## Verfügbarkeit einplanen

Remote-Backends sind auf einen Netzwerkdienst angewiesen. Rufen Sie
`isAvailable()` auf, bevor Sie sich auf einen Signer verlassen, wenden Sie
Wiederholungsversuche mit Backoff bei vorübergehenden Fehlern an und definieren
Sie einen Fallback-Pfad für Ausfälle. Keychain erzwingt HTTPS für
Remote-Backends und wendet einen Standard-Anfrage-Timeout an.

## Geheimnisse und Übertragung schützen

- Alle Remote-Backends kommunizieren über HTTPS; Nicht-HTTPS-Endpunkte werden
  abgelehnt.
- Stellen Sie API-Schlüssel, Tokens und Anmeldeinformationen über einen
  Secret-Manager oder Umgebungsvariablen bereit — niemals fest im Code
  hinterlegen oder in die Versionsverwaltung einchecken.
- Keychain bereinigt Fehlerantworten, um das Durchsickern von Geheimnissen in
  Logs zu verhindern; halten Sie Ihr eigenes Logging frei von unverschlüsselten
  Anmeldeinformationen und Schlüsselmaterial.

## Signierungsvorgänge überwachen und auditieren

Verfolgen Sie Erfolgsrate, Latenz und Fehleranzahl bei Signierungsvorgängen und
richten Sie Benachrichtigungen bei Ausreißern ein — siehe die Metriken unter
[Produktionsbereitschaft](/docs/payments/production-readiness#monitoring).
Aktivieren Sie anbieterseite Audit-Logs (KMS, Vault oder Custodian) und gleichen
Sie diese mit den Transaktionen ab, die Ihre Anwendung einreicht.
