Die folgenden Begriffe werden in der gesamten Solana-Dokumentation und im
Entwicklungs- ökosystem verwendet.

## account

Ein Eintrag im Solana-Ledger, der entweder Daten enthält oder ein ausführbares
Programm ist.

Wie ein Konto bei einer traditionellen Bank kann ein Solana-Konto Guthaben
namens [Lamports](#lamport) halten. Jedes Konto in einem [Cluster](#cluster) hat
eine eindeutige [Adresse](#address).

## App

Eine Frontend-Anwendung, die mit einem Solana-Cluster interagiert.

## Adresse

Der Speicherort in Solana, der verwendet werden kann, um ein Konto zu finden.
Jede Adresse ist einzigartig im Solana-[Cluster](#cluster).

Adressen in Solana können entweder ein Ed25519-[Public Key](#public-key-pubkey)
oder eine [Program Derived Address](#program-derived-address) sein.

## Berechtigung

Die Adresse eines Benutzers, der eine bestimmte Berechtigung über ein Konto hat.

Zum Beispiel:

- Die Fähigkeit, neue Token zu prägen, wird dem Konto gewährt, das die „Mint-
  Berechtigung“ für das Token-Mint ist.
- Die Fähigkeit, ein Programm zu aktualisieren, wird dem Konto gewährt, das die
  „Upgrade- Berechtigung“ eines Programms ist.

## Bankstatus

Das Ergebnis der Interpretation aller Programme im Ledger bei einer gegebenen
[Tick-Höhe](#tick-height). Es umfasst mindestens die Menge aller
[Konten](#account), die Token mit einem Wert ungleich null halten.

## Block

Eine zusammenhängende Menge von [Einträgen](#entry) im Ledger, die durch eine
[Abstimmung](#ledger-vote) abgedeckt sind. Ein [Leader](#leader) erzeugt
höchstens einen Block pro [Slot](#slot).

## Blockhash

Ein eindeutiger Wert ([Hash](#hash)), der einen Datensatz (Block) identifiziert.
Solana berechnet einen Blockhash aus der letzten [Entry-ID](#entry-id) des
Blocks.

## Blockhöhe

Die Anzahl der [Blöcke](#block) unterhalb des aktuellen Blocks. Der erste Block
nach dem [Genesis-Block](#genesis-block) hat die Höhe eins.

## Bootstrap-Validator

Der [Validator](#validator), der den Genesis-(ersten) [Block](#block) einer
Blockchain erzeugt.

## BPF-Loader

Das Solana-Programm, das [BPF](/docs/core/programs#berkeley-packet-filter-bpf)
[Onchain-Programme](#onchain-program) besitzt und lädt und es dem Programm
ermöglicht, mit der Laufzeitumgebung zu interagieren.

## Client

Ein Computerprogramm, das auf das Solana-Server-Netzwerk [Cluster](#cluster)
zugreift.

## Commitment

Ein Maß für die Netzwerkbestätigung des [Blocks](#block).

## Kompaktes Array-Format

Ein kompaktes Array ist ein Array, das im folgenden Format serialisiert wird:

- Die Array-Länge (kodiert als
  [compact-u16](https://github.com/anza-xyz/agave/blob/v2.1.13/short-vec/src/lib.rs))
- Die Array-Elemente nacheinander aufgelistet

![Kompaktes Array-Format](/assets/docs/core/transactions/compact_array_format.png)

Das kompakte Array-Format wird verwendet, um sowohl das Array der
[Kontoadressen](/docs/core/transactions#account-addresses) als auch das Array
der [ Anweisungen](/docs/core/transactions#instructions) in
Transaktionsnachrichten zu kodieren.

## Cluster

Eine Gruppe von [Validatoren](#validator), die ein einziges [Ledger](#ledger)
pflegen.

## Compute-Budget

Die maximale Anzahl von [Compute-Einheiten](#compute-units), die pro Transaktion
verbraucht werden.

## Compute-Einheiten

Die kleinste Maßeinheit für den Verbrauch von Rechenressourcen der Blockchain.
Compute-Einheiten werden verwendet, um die Rechenressourcen zu messen, die für
die Verarbeitung einer Transaktion verwendet werden.

## Bestätigungszeit

Die Wanduhr-Dauer zwischen der Erstellung eines [Tick-Eintrags](#tick) durch
einen [Leader](#leader) und der Erstellung eines
[bestätigten Blocks](#confirmed-block).

## Bestätigter Block

Ein [Block](#block), der eine [Supermehrheit](#supermajority) von
[Ledger-Stimmen](#ledger-vote) erhalten hat.

## Kontrollebene

Ein Gossip-Netzwerk, das alle [Knoten](#node) eines [Clusters](#cluster)
verbindet.

## Abkühlungsperiode

Eine Anzahl von [Epochen](#epoch), nachdem der [Stake](#stake) deaktiviert
wurde, während er schrittweise für Abhebungen verfügbar wird. Während dieser
Zeit gilt der Stake als "deaktivierend". Weitere Informationen über:
[Aufwärm- und Abkühlungsphase](https://docs.anza.xyz/implemented-proposals/staking-rewards#stake-warmup-cooldown-withdrawal)

## Kosteneinheiten

Auch als `transaction cost` bezeichnet, ist eine umfassende Schätzung aller
Ressourcen, die zur Verarbeitung einer Transaktion erforderlich sind, gemessen
in [Compute-Einheiten](#compute-units). Weitere Informationen
[hier](https://www.anza.xyz/blog/why-solana-transaction-costs-and-compute-units-matter-for-developers).

## credit

Siehe [vote credit](#vote-credit).

## Cross Program Invocation (CPI)

Ein Aufruf von einem [Onchain-Programm](#onchain-program) an ein anderes.
Weitere Informationen finden Sie unter
[Aufrufe zwischen Programmen](/docs/core/cpi).

## Datenebene

Ein Multicast-Netzwerk, das verwendet wird, um [Einträge](#entry) effizient zu
validieren und Konsens zu erzielen.

## Drone

Ein Offchain-Dienst, der als Verwalter für den privaten Schlüssel eines
Benutzers fungiert. Er dient in der Regel dazu, Transaktionen zu validieren und
zu signieren.

## Eintrag

Ein Eintrag im [Ledger](#ledger), entweder ein [Tick](#tick) oder ein
[Transaktionseintrag](#transactions-entry).

## Eintrags-ID

Ein vorbildresistenter [Hash](#hash) über den endgültigen Inhalt eines Eintrags,
der als global eindeutiger Identifikator des [Eintrags](#entry) dient. Der Hash
dient als Nachweis für:

- Der Eintrag wurde nach einer bestimmten Zeitspanne generiert
- Die angegebenen [Transaktionen](#transaction) sind diejenigen, die im Eintrag
  enthalten sind
- Die Position des Eintrags in Bezug auf andere Einträge im [Ledger](#ledger)

Siehe [Proof of History](#proof-of-history-poh).

## epoch

Die Zeit, d. h. die Anzahl der [Slots](#slot), für die ein
[Leader-Schedule](#leader-schedule) gültig ist.

## Fee-Konto

Das Fee-Konto in der Transaktion ist das Konto, das die Kosten für die Aufnahme
der Transaktion in das Ledger übernimmt. Dies ist das erste Konto in der
Transaktion. Dieses Konto muss in der Transaktion als Read-Write (beschreibbar)
deklariert werden, da die Zahlung für die Transaktion den Kontostand reduziert.

## Finalität

Wenn Knoten, die 2/3 des [Stakes](#stake) repräsentieren, eine gemeinsame
[Root](#root) haben.

## Fork

Ein [Ledger](#ledger), das von gemeinsamen Einträgen abgeleitet wurde, sich dann
aber aufgespalten hat.

## Genesis-Block

Der erste [Block](#block) in der Kette.

## Genesis-Konfiguration

Die Konfigurationsdatei, die das [Ledger](#ledger) für den
[Genesis-Block](#genesis-block) vorbereitet.

## Hash

Ein digitaler Fingerabdruck einer Byte-Sequenz.

## Inflation

Eine Erhöhung des Token-Angebots im Laufe der Zeit, die zur Finanzierung von
Validierungsbelohnungen und zur Fortführung der Entwicklung von Solana verwendet
wird.

## Innere Anweisung

Siehe [Cross Program Invocation](#cross-program-invocation-cpi).

## Anweisung

Ein Aufruf zum Aufrufen eines bestimmten
[Anweisungshandlers](#instruction-handler) in einem [Programm](#program). Eine
Anweisung gibt auch an, welche Konten sie lesen oder ändern möchte, sowie
zusätzliche Daten, die als ergänzende Eingabe für den
[Anweisungshandler](#instruction-handler) dienen. Ein [Client](#client) muss
mindestens eine Anweisung in eine [Transaktion](#transaction) aufnehmen, und
alle Anweisungen müssen abgeschlossen werden, damit die Transaktion als
erfolgreich gilt.

## Anweisungshandler

Anweisungshandler sind [Programm](#program)-Funktionen, die
[Anweisungen](#instruction) aus [Transaktionen](#transaction) verarbeiten. Ein
Anweisungshandler kann eine oder mehrere
[Cross Program Invocations](#cross-program-invocation-cpi) enthalten.

## keypair

Ein [öffentlicher Schlüssel](#public-key-pubkey) und der zugehörige
[private Schlüssel](#private-key) für den Zugriff auf ein Konto.

## lamport

Ein Bruchteil eines [nativen Tokens](#native-token) mit dem Wert von 0,000000001
[SOL](#sol).

> Innerhalb des Rechenbudgets wird eine Menge von
> _[Mikro-lamports](https://github.com/solana-labs/solana/blob/ced8f6a512c61e0dd5308095ae8457add4a39e94/program-runtime/src/prioritization_fee.rs#L1-L2)_
> bei der Berechnung von [Priorisierungsgebühren](#prioritization-fee)
> verwendet.

## Leader

Die Rolle eines [Validators](#validator), wenn er [Einträge](#entry) an das
[Ledger](#ledger) anfügt.

## Leader-Zeitplan

Eine Sequenz von [öffentlichen Schlüsseln](#public-key-pubkey) der
[Validatoren](#validator), die [Slots](#slot) zugeordnet sind. Der Cluster
verwendet den Leader-Zeitplan, um zu bestimmen, welcher Validator zu einem
bestimmten Zeitpunkt der [Leader](#leader) ist.

## Ledger

Eine Liste von [Einträgen](#entry), die [Transaktionen](#transaction) enthalten,
die von [Clients](#client) signiert wurden. Konzeptionell lässt sich dies bis
zum [Genesis-Block](#genesis-block) zurückverfolgen, aber das tatsächliche
Ledger eines [Validators](#validator) kann nur neuere [Blöcke](#block)
enthalten, um Speicherplatz zu reduzieren, da ältere für die Validierung
zukünftiger Blöcke nicht benötigt werden.

## Ledger Vote

Ein [Hash](#hash) des [Validator-Zustands](#bank-state) zu einer bestimmten
[Tick-Höhe](#tick-height). Es umfasst die Bestätigung eines
[Validators](#validator), dass ein empfangener [Block](#block) verifiziert
wurde, sowie ein Versprechen, nicht für einen widersprüchlichen [Block](#block)
\(d.h. [Fork](#fork)\) für einen bestimmten Zeitraum zu stimmen, die
[Lockout](#lockout)-Periode.

## Light Client

Ein [Client](#client)-Typ, der überprüfen kann, ob er auf ein gültiges
[Cluster](#cluster) verweist. Er führt mehr Ledger-Verifizierung durch als ein
[Thin Client](#thin-client) und weniger als ein [Validator](#validator).

## Loader

Ein [Programm](#program) mit der Fähigkeit, die Binärcodierung anderer
Onchain-Programme zu interpretieren.

## Lockout

Der Zeitraum, in dem ein [Validator](#validator) nicht in der Lage ist, für
einen anderen [Fork](#fork) zu [voten](#ledger-vote).

## Nachricht

Der strukturierte Inhalt einer [Transaktion](#transaction). Enthält im
Allgemeinen einen Header, ein Array von Kontoadressen, einen aktuellen
[Blockhash](#blockhash) und ein Array von [ Anweisungen](#instruction).

Erfahren Sie mehr über die
[Nachrichtenformatierung innerhalb von Transaktionen](/docs/core/transactions#message-header)
hier.

## Micro-Lamport

1.000.000 Micro-Lamports = 1 Lamport

## Nakamoto-Koeffizient

Ein Maß für Dezentralisierung, der Nakamoto-Koeffizient ist die kleinste Anzahl
unabhängiger Einheiten, die gemeinsam handeln können, um eine Blockchain
stillzulegen. Der Begriff wurde von Balaji S. Srinivasan und Leland Lee in
[Quantifying Decentralization](https://news.earn.com/quantifying-decentralization-e39db233c28e)
geprägt.

## Native Token

Der [Token](#token), der verwendet wird, um die von [Nodes](#node) in einem
Cluster geleistete Arbeit zu verfolgen.

## Node

Ein Computer, der an einem [Cluster](#cluster) teilnimmt.

## Node-Anzahl

Die Anzahl der [Validatoren](#validator), die an einem [Cluster](#cluster)
teilnehmen.

## Onchain-Programm

Der ausführbare Code auf der Solana-Blockchain, der die
[ Anweisungen](#instruction) interpretiert, die innerhalb jeder
[Transaktion](#transaction) gesendet werden, um Konten zu lesen und zu ändern,
über die er Kontrolle hat. Diese Programme werden auf anderen Blockchains oft
als "[_Smart Contracts_](/docs/core/programs)" bezeichnet.

## Eigentümer

Kann sich entweder auf das [besitzende Programm](#owning-program) oder auf die
[besitzende Wallet](#owning-wallet) beziehen. Beispielsweise ruft
[getTokenAccountsByOwner](https://solana.com/docs/rpc/http/gettokenaccountsbyowner)
Token-Konten ab, die einer bestimmten Wallet gehören, wobei das besitzende
Programm für diese Token-Konten das [Token Program](#token-program) ist.

Um Verwirrung zu vermeiden, ist es vorzuziehen, entweder
[besitzendes Programm](#owning-program) oder [besitzende Wallet](#owning-wallet)
zu verwenden, anstatt des allgemeinen Begriffs „Eigentümer“.

## besitzendes Programm

Die [Adresse](#address) des [Programms](#onchain-program), das das Konto
besitzt. Nur das besitzende Programm ist in der Lage, das Konto zu ändern.

Siehe auch [Berechtigung](#authority).

## besitzende Wallet

Die Wallet, die einem Konto zugeordnet ist. Beispielsweise wird das Token
Program verwendet, um Token-Konten zu speichern, die einer bestimmten Wallet und
Mint zugeordnet sind, unter Verwendung einer Adresse, die aus dieser
[Wallet](#wallet) und [Mint)(#token-mint) erstellt wurde.

## PoH

Siehe [Proof of History](#proof-of-history-poh).

## Punkt

Ein gewichteter [Credit](#credit) in einem Belohnungssystem. Im
[Validator](#validator)-
[Belohnungssystem](https://docs.anza.xyz/consensus/stake-delegation-and-rewards)
ist die Anzahl der Punkte, die einem [Stake](#stake) bei der Einlösung zustehen,
das Produkt aus den verdienten [Vote Credits](#vote-credit) und der Anzahl der
gestakten Lamports.

## privater Schlüssel

Der private Schlüssel eines [Keypairs](#keypair).

## Programm

Siehe [Onchain-Programm](#onchain-program).

## programmabgeleitetes Konto (PDA)

Ein Konto, dessen Adresse basierend auf einem
[Onchain-Programm](#onchain-program) und einem oder mehreren [Seeds](#seed)
erstellt wird, die vom Programmautor erstellt wurden. PDA-Adressen sind keine
öffentlichen Schlüssel, und PDAs haben keine privaten Schlüssel.

Die Signierungsberechtigung für ein PDA liegt beim
[Onchain-Programm](#onchain-program), das ihr
[besitzendes Programm](#owning-program) ist.

## Programm-ID

Der öffentliche Schlüssel des [Kontos](#account), das ein
[Programm](#onchain-program) enthält.

## Proof of History (PoH)

Ein Stapel von Beweisen, von denen jeder beweist, dass bestimmte Daten
existierten, bevor der Beweis erstellt wurde, und dass eine präzise Zeitspanne
vor dem vorherigen Beweis verstrichen ist. Wie eine
[VDF](#verifiable-delay-function-vdf) kann ein Proof of History in weniger Zeit
verifiziert werden, als seine Erstellung dauerte.

## priority fee

Eine zusätzliche Gebühr, die Benutzer in der Compute-Budget- Anweisungen angeben
können, um ihre [Transaktionen](#transaction) zu priorisieren.

Die Priorisierungsgebühr wird berechnet, indem die angeforderten maximalen
Compute-Einheiten mit dem Compute-Einheitenpreis (angegeben in Schritten von
0,000001 lamport pro Compute-Einheit) multipliziert und auf den nächsten lamport
aufgerundet werden.

Transaktionen sollten die minimal erforderliche Anzahl an Compute-Einheiten für
die Ausführung anfordern, um die Gebühren zu minimieren.

## public key (pubkey)

Der öffentliche Schlüssel eines [keypair](#keypair). Ein öffentlicher Schlüssel
ist eine der verschiedenen Arten von [Adressen](#address), die ein Konto auf
Solana verwenden kann.

## rent

Fee, die von [Konten](#account) und [Programmen](#program) für die Speicherung
von Daten auf der Blockchain gezahlt wird. Wenn Konten nicht über ausreichend
Guthaben zur Zahlung der rent verfügen, können sie durch Garbage Collection
entfernt werden.

Siehe auch [rent exempt](#rent-exempt) unten. Erfahren Sie hier mehr über rent:
[Was ist rent?](/docs/intro/rent).

## rent exempt

Konten, die ein minimales lamport-Guthaben aufrechterhalten, das proportional
zur Menge der auf dem Konto gespeicherten Daten ist. Alle neu erstellten Konten
werden dauerhaft on-chain gespeichert, bis das Konto geschlossen wird. Es ist
nicht möglich, ein Konto zu erstellen, das unter die Schwelle der Befreiung von
rent fällt.

## root

Ein [Block](#block) oder [slot](#slot), der die maximale [Sperrfrist](#lockout)
auf einem [Validatoren](#validator) erreicht hat. Die root ist der höchste
Block, der ein Vorfahre aller aktiven Forks auf einem validator ist. Alle
Vorfahrenblöcke einer root sind ebenfalls transitiv eine root. Blöcke, die weder
ein Vorfahre noch ein Nachkomme der root sind, werden vom Konsens ausgeschlossen
und können verworfen werden.

## Laufzeit

Die Komponente eines [Validatoren](#validator), die für die Ausführung von
[Programmen](#program) verantwortlich ist.

## Sealevel

Solanas parallele Laufzeitumgebung für [Onchain-Programme](#onchain-program).

## Shred

Ein Bruchteil eines [Blocks](#block); die kleinste Einheit, die zwischen
[Validatoren](#validator) übertragen wird.

## Signatur

Eine 64-Byte-ed25519-Signatur von R (32 Bytes) und S (32 Bytes). Mit der
Anforderung, dass R ein gepackter Edwards-Punkt ohne kleine Ordnung ist und S
ein Skalar im Bereich von `0 <= S < L` liegt. Diese Anforderung gewährleistet,
dass keine Signatur-Manipulierbarkeit besteht. Jede Transaktion muss mindestens
eine Signatur für das [Gebührenkonto](#fee-account) haben. Daher kann die erste
Signatur in der Transaktion als [Transaktions-ID](#transaction-id) behandelt
werden.

## Ausfallrate

Der Prozentsatz der [übersprungenen Slots](#skipped-slot) von den gesamten
Leader-Slots in der aktuellen Epoche. Diese Metrik kann irreführend sein, da sie
eine hohe Varianz nach der Epochengrenze aufweist, wenn die Stichprobengröße
klein ist, sowie für Validatoren mit einer geringen Anzahl von Leader-Slots. Sie
kann jedoch auch nützlich sein, um zeitweise Fehlkonfigurationen von Knoten zu
identifizieren.

## Übersprungener Slot

Ein vergangener [Slot](#slot), der keinen [Block](#block) erzeugt hat, weil der
Leader offline war oder der [Fork](#fork), der den Slot enthielt, durch
Cluster-Konsens zugunsten einer besseren Alternative aufgegeben wurde. Ein
übersprungener Slot wird nicht als Vorgänger für Blöcke in nachfolgenden Slots
erscheinen, die [Blockhöhe](#block-height) nicht erhöhen und den ältesten
`recent_blockhash` nicht ablaufen lassen.

Ob ein Slot übersprungen wurde, kann erst festgestellt werden, wenn er älter ist
als der neueste [verwurzelte](#root) (also nicht übersprungene) Slot.

## Slot

Der Zeitraum, in dem jeder [Leader](#leader) Transaktionen aufnimmt und einen
[Block](#block) erzeugt.

Zusammengenommen bilden Slots eine logische Uhr. Slots sind sequenziell geordnet
und überschneiden sich nicht, wobei sie gemäß [PoH](#proof-of-history-poh)
annähernd gleiche Zeitspannen in der realen Welt umfassen.

## Smart Contract

Siehe [Onchain-Programm](#onchain-program).

## SOL

Der [native Token](#native-token) eines Solana-[Clusters](#cluster).

## Solana Program Library (SPL)

Eine [Programmbibliothek](https://spl.solana.com/) auf Solana, wie z. B.
spl-token, die Aufgaben wie das Erstellen und Verwenden von Token erleichtert.

## Stake

Token, die dem [Cluster](#cluster) verfallen, wenn böswilliges Verhalten eines
[Validatoren](#validator) nachgewiesen werden kann.

## Stake-Weighted Quality of Service (SWQoS)

SWQoS ermöglicht eine
[bevorzugte Behandlung für Transaktionen, die von gestakten Validatoren stammen](/developers/guides/advanced/stake-weighted-qos).

## Supermajorität

2/3 eines [Clusters](#cluster).

## Sysvar

Ein System-[Konto](#account). [Sysvars](https://docs.anza.xyz/runtime/sysvars)
stellen Cluster-Zustandsinformationen bereit, wie z. B. die aktuelle Tick-Höhe,
Rewards-[Punkte](#point)-Werte usw. Programme können auf Sysvars über ein
Sysvar-Konto (pubkey) oder durch Abfrage über einen Syscall zugreifen.

## Thin Client

Eine Art von [Client](#client), der darauf vertraut, dass er mit einem gültigen
[Cluster](#cluster) kommuniziert.

## Tick

Ein Ledger-[Eintrag](#entry), der die Wanduhr-Dauer schätzt.

## Tick-Höhe

Der N-te [Tick](#tick) im [Ledger](#ledger).

## Token

Ein digital übertragbarer Vermögenswert.

## Token Extensions Program

Das [Token Extensions Program](https://spl.solana.com/token-2022) hat die
Programm-ID `TokenzQdBNbLqP5VEhdkAS6EPFLC1PHnBqCXEpPxuEb` und umfasst alle
Funktionen des [Token Program](#token-program), bietet aber zusätzlich
Erweiterungen wie vertrauliche Transfers, benutzerdefinierte Transferlogik,
erweiterte Metadaten und vieles mehr.

## Token-Mint

Ein Konto, das Token erzeugen (oder „minten“) kann. Verschiedene Token werden
durch ihre eindeutigen Token-Mint-Adressen unterschieden.

## Token Program

Das [Token Program](https://spl.solana.com/token) hat die Programm-ID
`TokenkegQfeZyiNwAJbNbGKPFXCWuBvf9Ss623VQ5DA` und bietet die grundlegenden
Funktionen zum Übertragen, Einfrieren und Minten von Token.

## tps

[Transaktionen](#transaction) pro Sekunde.

## tpu

[Transaction Processing Unit](https://docs.anza.xyz/validator/tpu).

## Transaktion

Eine oder mehrere Anweisungen, die von einem [Client](#client) mit einem oder
mehreren [Keypairs](#keypair) signiert und atomar mit nur zwei möglichen
Ergebnissen ausgeführt werden: Erfolg oder Fehlschlag.

## Transaktions-ID

Die erste [Signatur](#signature) in einer [Transaktion](#transaction), die zur
eindeutigen Identifizierung der Transaktion im gesamten [Ledger](#ledger)
verwendet werden kann.

## Transaktionsbestätigungen

Die Anzahl der [bestätigten Blöcke](#confirmed-block) seit dem Zeitpunkt, zu dem
die Transaktion in das [Ledger](#ledger) aufgenommen wurde. Eine Transaktion ist
finalisiert, wenn ihr Block zu einem [Root](#root) wird.

## Transaktionseintrag

Eine Menge von [Transaktionen](#transaction), die parallel ausgeführt werden
können.

## tvu

[Transaction Validation Unit](https://docs.anza.xyz/validator/tvu).

## Validatoren

Ein vollwertiger Teilnehmer in einem Solana-Netzwerk-[Cluster](#cluster), der
neue [Blöcke](#block) erzeugt. Ein Validator validiert die Transaktionen, die
dem [Ledger](#ledger) hinzugefügt werden

## VDF

Siehe [Verifiable Delay Function](#verifiable-delay-function-vdf).

## Verifiable Delay Function (VDF)

Eine Funktion, die eine feste Zeitspanne zur Ausführung benötigt und einen
Beweis erzeugt, dass sie ausgeführt wurde, der dann in kürzerer Zeit verifiziert
werden kann, als die Erzeugung dauerte.

## Vote

Siehe [Ledger-Abstimmung](#ledger-vote).

## Abstimmungsguthaben

Eine Belohnungsbilanz für [Validatoren](#validator). Ein Abstimmungsguthaben
wird einem Validator in seinem Abstimmungskonto gutgeschrieben, wenn der
Validator eine [Root](#root) erreicht.

## Wallet

Eine Sammlung von [Keypairs](#keypair), die es Nutzern ermöglicht, ihre Guthaben
zu verwalten.

## Aufwärmphase

Eine bestimmte Anzahl von [Epochen](#epoch), nachdem [Stake](#stake) delegiert
wurde, während dieser progressiv wirksam wird. Während dieser Phase gilt der
Stake als "aktivierend". Weitere Informationen zu:
[Aufwärm- und Abkühlphase](https://docs.anza.xyz/consensus/stake-delegation-and-rewards#stake-warmup-cooldown-withdrawal)
