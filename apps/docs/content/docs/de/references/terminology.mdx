Die folgenden Begriffe werden in der gesamten Solana-Dokumentation und im
Entwicklungs- ökosystem verwendet.

## account

Ein Eintrag im Solana-Ledger, der entweder Daten enthält oder ein ausführbares
Programm ist.

Wie ein Konto bei einer traditionellen Bank kann ein Solana-Konto Guthaben
namens [Lamports](#lamport) halten. Jedes Konto in einem [Cluster](#cluster) hat
eine eindeutige [Adresse](#address).

## App

Eine Frontend-Anwendung, die mit einem Solana-Cluster interagiert.

## Adresse

Der Speicherort in Solana, der verwendet werden kann, um ein Konto zu finden.
Jede Adresse ist einzigartig im Solana-[Cluster](#cluster).

Adressen in Solana können entweder ein Ed25519-[Public Key](#public-key-pubkey)
oder eine [Program Derived Address](#program-derived-address-pda) sein.

## Address Lookup Table (ALT)

Eine Datenstruktur, die eine Liste von Kontoadressen speichert und es
[versionierten Transaktionen](#versioned-transaction) ermöglicht, auf Konten
mithilfe kompakter 1-Byte-Indizes anstelle vollständiger 32-Byte-Public-Keys zu
verweisen. Address Lookup Tables ermöglichen es einer einzelnen Transaktion, auf
mehr Konten zu verweisen, als das ursprüngliche Transaktionsformat erlaubt.

## Anchor

Ein Framework zum Erstellen von Solana-[Onchain-Programmen](#onchain-program) in
Rust. Anchor bietet Makros, die Boilerplate-Code reduzieren, ein
standardisiertes Kontovalidierungssystem und generiert eine [IDL](#idl), die die
Schnittstelle des Programms beschreibt. Es ist das am häufigsten verwendete
Framework für die Entwicklung von Solana-Programmen.

## Associated Token Account (ATA)

Ein [token account](#token-account), dessen Adresse deterministisch von einer
Wallet-Adresse und einem [Token Mint](#token-mint) abgeleitet wird. Das
Associated Token Program stellt sicher, dass jede Wallet ein kanonisches
Token-Konto pro Token-Typ hat.

## Autorisierung

Die Adresse eines Benutzers, der eine bestimmte Berechtigung über ein Konto hat.

Zum Beispiel:

- Die Berechtigung zum Prägen neuer Token wird dem Konto erteilt, das die
  'Mint-Autorisierung' für den Token Mint ist.
- Die Berechtigung zum Aktualisieren eines Programms wird dem Konto erteilt, das
  die 'Upgrade-Autorisierung' eines Programms ist.

## Bank State

Das Ergebnis der Interpretation aller Programme im Ledger bei einer bestimmten
[Tick Height](#tick-height). Es umfasst mindestens die Gesamtheit aller
[Konten](#account), die nicht-null [Native Token](#native-token) halten.

## Block

Eine zusammenhängende Menge von [Einträgen](#entry) im Ledger, die durch eine
[Abstimmung](#ledger-vote) abgedeckt sind. Ein [Leader](#leader) produziert
höchstens einen Block pro [slot](#slot).

## Blockhash

Ein eindeutiger Wert ([Hash](#hash)), der einen Datensatz (Block) identifiziert.
Solana berechnet einen Blockhash aus der letzten [Entry-ID](#entry-id) des
Blocks.

## Blockhöhe

Die Anzahl der [Blöcke](#block) unterhalb des aktuellen Blocks. Der erste Block
nach dem [Genesis-Block](#genesis-block) hat die Höhe eins.

## Bootstrap-Validator

Der [Validator](#validator), der den Genesis-Block (ersten [Block](#block))
einer Blockchain erzeugt.

## BPF-Loader

Das Solana-Programm, das [BPF](/docs/core/programs)
[Onchain-Programme](#onchain-program) besitzt und lädt, wodurch das Programm mit
der Laufzeitumgebung interagieren kann.

## Bump

Ein einzelnes Byte (0–255), das an [Seeds](#seed) angehängt wird, wenn eine
[Programmabgeleitete Adresse](#program-derived-address-pda) abgeleitet wird, um
sicherzustellen, dass die resultierende Adresse außerhalb der ed25519-Kurve
liegt und somit eine gültige PDA ist. Der kanonische Bump ist der höchste Wert,
der eine gültige PDA erzeugt, und wird normalerweise in den Konten daten
gespeichert, um eine Neuberechnung zu vermeiden.

## Client

Ein Computerprogramm, das auf das Solana-Server-Netzwerk [Cluster](#cluster)
zugreift.

## Commitment

Ein Maß für die Netzwerkbestätigung des [Blocks](#block).

## Kompaktes Array-Format

Ein kompaktes Array ist ein Array, das im folgenden Format serialisiert wird:

- Die Array-Länge (kodiert als
  [compact-u16](https://github.com/anza-xyz/agave/blob/v2.1.13/short-vec/src/lib.rs))
- Die Array-Elemente nacheinander aufgelistet

![Kompaktes Array-Format](/assets/docs/core/transactions/compact_array_format.png)

Das kompakte Array-Format wird verwendet, um sowohl das
[Konten -Adressen](/docs/core/transactions/transaction-structure#account-addresses)-Array
als auch das
[ Anweisungen](/docs/core/transactions/transaction-structure#instructions)-Array
in Transaktionsnachrichten zu kodieren.

## Cluster

Eine Gruppe von [Validatoren](#validator), die ein einzelnes [Ledger](#ledger)
pflegen.

## Rechenbudget

Die maximale Anzahl an [Recheneinheiten](#compute-units), die pro Transaktion
verbraucht werden.

## Recheneinheiten

Die kleinste Maßeinheit für den Verbrauch von Rechenressourcen der Blockchain.
Recheneinheiten werden verwendet, um die Rechenressourcen zu messen, die zur
Verarbeitung einer Transaktion verwendet werden.

## Bestätigungszeit

Die Echtzeit-Dauer zwischen der Erstellung eines [Tick-Eintrags](#tick) durch
einen [Leader](#leader) und der Erstellung eines
[bestätigten Blocks](#confirmed-block).

## bestätigter Block

Ein [Block](#block), der eine [Supermehrheit](#supermajority) von
[Ledger-Stimmen](#ledger-vote) erhalten hat.

## Kontrollebene

Ein Gossip-Netzwerk, das alle [Nodes](#node) eines [Clusters](#cluster)
verbindet.

## Abkühlungsphase

Eine bestimmte Anzahl von epoch, nachdem der [Stake](#stake) deaktiviert wurde,
während er schrittweise für Abhebungen verfügbar wird. Während dieser Phase gilt
der Stake als "deaktivierend". Weitere Informationen über:
[Aufwärm- und Abkühlungsphase](https://docs.anza.xyz/consensus/stake-delegation-and-rewards#stake-warmup-cooldown-withdrawal)

## Kosteneinheiten

Auch als `transaction cost` bezeichnet, ist eine umfassende Schätzung aller
Ressourcen, die zur Verarbeitung einer Transaktion erforderlich sind, gemessen
in [Recheneinheiten](#compute-units). Weitere Informationen
[hier](https://www.anza.xyz/blog/why-solana-transaction-costs-and-compute-units-matter-for-developers).

## Credit

Siehe [Vote-Credit](#vote-credit).

## Cross Program Invocation (CPI)

Ein Aufruf von einem [Onchain-Programm](#onchain-program) zu einem anderen.
Weitere Informationen finden Sie unter
[Aufrufe zwischen Programmen](/docs/core/cpi).

## Datenebene

Ein Multicast-Netzwerk, das verwendet wird, um [Einträge](#entry) effizient zu
validieren und Konsens zu erzielen.

## Drone

Ein Offchain-Dienst, der als Verwahrer für den privaten Schlüssel eines
Benutzers fungiert. Er dient typischerweise dazu, Transaktionen zu validieren
und zu signieren.

## Eintrag

Ein Eintrag im [Ledger](#ledger), entweder ein [Tick](#tick) oder ein
[Transaktionseintrag](#transactions-entry).

## Eintrags-ID

Ein preimage-resistenter [Hash](#hash) über den finalen Inhalt eines Eintrags,
der als global eindeutiger Identifikator des [Eintrags](#entry) dient. Der Hash
dient als Nachweis für:

- Die Generierung des Eintrags nach einer bestimmten Zeitdauer
- Die angegebenen [Transaktionen](#transaction) sind diejenigen, die im Eintrag
  enthalten sind
- Die Position des Eintrags in Bezug auf andere Einträge im [Ledger](#ledger)

Siehe [Proof of History](#proof-of-history-poh).

## epoch

Die Zeit, d. h. die Anzahl der [Slots](#slot), für die ein
[Leader-Zeitplan](#leader-schedule) gültig ist.

## Fee-Konto

Das Fee-Konto in der Transaktion ist das Konto, das für die Kosten der Aufnahme
der Transaktion in das Ledger aufkommt. Dies ist das erste Konto in der
Transaktion. Dieses Konto muss in der Transaktion als Read-Write (beschreibbar)
deklariert werden, da die Bezahlung der Transaktion den Kontostand reduziert.

## Finalität

Wenn Knoten, die 2/3 des [Stakes](#stake) repräsentieren, eine gemeinsame
[Root](#root) haben.

## Fork

Ein [Ledger](#ledger), das von gemeinsamen Einträgen abgeleitet wurde, sich dann
aber verzweigt hat.

## Genesis-Block

Der erste [Block](#block) in der Chain.

## Genesis-Konfiguration

Die Konfigurationsdatei, die das [Ledger](#ledger) für den
[Genesis-Block](#genesis-block) vorbereitet.

## Hash

Ein digitaler Fingerabdruck einer Bytefolge.

## Inflation

Eine Erhöhung des Token-Angebots im Laufe der Zeit, die zur Finanzierung von
Belohnungen für die Validierung und zur Finanzierung der fortlaufenden
Entwicklung von Solana verwendet wird.

## Inner Instruction

Siehe [Cross Program Invocation](#cross-program-invocation-cpi).

## Anweisungen

Ein Aufruf, um einen bestimmten [Instruction-Handler](#instruction-handler) in
einem [Programm](#program) aufzurufen. Eine Instruction gibt auch an, welche
Konten sie lesen oder ändern möchte, sowie zusätzliche Daten, die als
zusätzliche Eingabe für den [Instruction-Handler](#instruction-handler) dienen.
Ein [Client](#client) muss mindestens eine Instruction in einer
[Transaktion](#transaction) einschließen, und alle Anweisungen müssen
abgeschlossen werden, damit die Transaktion als erfolgreich gilt.

## Instruction-Handler

Instruction-Handler sind [Programm](#program)-Funktionen, die
[Anweisungen](#instruction) aus [Transaktionen](#transaction) verarbeiten. Ein
Instruction-Handler kann eine oder mehrere
[Cross-Program Invocations](#cross-program-invocation-cpi) enthalten.

## IDL

Interface Definition Language. Eine JSON-Datei, die von [Anchor](#anchor)
generiert wird und die [Instruction-Handler](#instruction-handler), Konten und
Typen eines Programms beschreibt. Die IDL ermöglicht es Client-Bibliotheken,
automatisch typisierte Schnittstellen für die Interaktion mit einem
[Onchain-Programm](#onchain-program) zu generieren.

## keypair

Ein [öffentlicher Schlüssel](#public-key-pubkey) und der zugehörige
[private Schlüssel](#private-key) für den Zugriff auf ein Konto.

## lamport

Ein Bruchteil eines [nativen Tokens](#native-token) mit dem Wert von 0,000000001
[SOL](#sol).

> Innerhalb des Rechenbudgets wird eine Menge von
> _[Mikro-Lamports](https://github.com/solana-labs/solana/blob/ced8f6a512c61e0dd5308095ae8457add4a39e94/program-runtime/src/prioritization_fee.rs#L1-L2)_
> bei der Berechnung von [Priorisierungsgebühren](#prioritization-fee)
> verwendet.

## leader

Die Rolle eines [Validatoren](#validator), wenn er [Einträge](#entry) an das
[Ledger](#ledger) anhängt.

## Leader-Zeitplan

Eine Abfolge von [öffentlichen Schlüsseln](#public-key-pubkey) der
[Validatoren](#validator), die [Slots](#slot) zugeordnet sind. Das Cluster
verwendet den Leader-Zeitplan, um zu bestimmen, welcher Validator zu einem
bestimmten Zeitpunkt der [Leader](#leader) ist.

## Ledger

Eine Liste von [Einträgen](#entry), die [Transaktionen](#transaction) enthalten,
welche von [Clients](#client) signiert wurden. Konzeptionell lässt sich dies bis
zum [Genesis-Block](#genesis-block) zurückverfolgen, aber das tatsächliche
Ledger eines [Validatoren](#validator) enthält möglicherweise nur neuere
[Blöcke](#block), um Speicherplatz zu sparen, da ältere Blöcke für die
Validierung zukünftiger Blöcke systembedingt nicht benötigt werden.

## Ledger-Vote

Ein [Hash](#hash) des [Validator-Zustands](#bank-state) zu einer bestimmten
[Tick-Höhe](#tick-height). Es stellt die Bestätigung eines
[Validatoren](#validator) dar, dass ein empfangener [Block](#block) verifiziert
wurde, sowie ein Versprechen, für einen bestimmten Zeitraum, die
[Lockout-Periode](#lockout), nicht für einen widersprüchlichen [Block](#block)
\(d. h. [Fork](#fork)\) zu stimmen.

## Light Client

Ein Typ von [Client](#client), der überprüfen kann, ob er auf ein gültiges
[Cluster](#cluster) zeigt. Er führt mehr Ledger-Verifizierung durch als ein
[Thin Client](#thin-client), aber weniger als ein [Validator](#validator).

## Loader

Ein [Programm](#program) mit der Fähigkeit, die binäre Kodierung anderer
[Onchain-Programme](#onchain-program) zu interpretieren.

## Lockout

Die Zeitspanne, während der ein [Validator](#validator) nicht in der Lage ist,
für einen anderen [Fork](#fork) zu [voten](#ledger-vote).

## Nachricht

Der strukturierte Inhalt einer [Transaktion](#transaction). Enthält in der Regel
einen Header, ein Array von Kontoadressen, einen aktuellen
[Blockhash](#blockhash) und ein Array von [Anweisungen](#instruction).

Erfahren Sie mehr über die
[Nachrichtenformatierung innerhalb von Transaktionen](/docs/core/transactions/transaction-structure#header)
hier.

## Mikro-Lamport

1.000.000 Mikro-Lamports = 1 [Lamport](#lamport)

## Nakamoto-Koeffizient

Ein Maß für die Dezentralisierung. Der Nakamoto-Koeffizient ist die kleinste
Anzahl unabhängiger Einheiten, die gemeinsam handeln können, um eine Blockchain
zu stoppen. Der Begriff wurde von Balaji S. Srinivasan und Leland Lee in
[Quantifying Decentralization](https://news.earn.com/quantifying-decentralization-e39db233c28e)
geprägt.

## Native Token

Der [Token](#token), der verwendet wird, um die von [Knoten](#node) in einem
Cluster geleistete Arbeit zu verfolgen.

## Knoten

Ein Computer, der an einem [Cluster](#cluster) teilnimmt.

## Knotenanzahl

Die Anzahl der [Validatoren](#validator), die an einem [Cluster](#cluster)
teilnehmen.

## Onchain

Beschreibt Daten oder Programme, die auf einer Blockchain existieren oder
ausgeführt werden. "Onchain" (ohne Bindestrich) ist die bevorzugte Schreibweise.

## Onchain-Programm

Der ausführbare Code auf der Solana-Blockchain, der die
[Anweisungen](#instruction) interpretiert, die innerhalb jeder
[Transaktion](#transaction) gesendet werden, um Konten zu lesen und zu ändern,
über die er die Kontrolle hat. Diese Programme werden auf anderen Blockchains
oft als "[_Smart Contracts_](/docs/core/programs)" bezeichnet.

## Eigentümer

Kann sich entweder auf das [eigentümerführende Programm](#owning-program) oder
die [eigentümerführende Wallet](#owning-wallet) beziehen. Beispielsweise ruft
[getTokenAccountsByOwner](https://solana.com/docs/rpc/http/gettokenaccountsbyowner)
Token-Konten ab, die einer bestimmten Wallet gehören, jedoch ist das
eigentümerführende Programm für diese Token-Konten das
[Token-Programm](#token-program).

Um Verwirrung zu vermeiden, ist es vorzuziehen, entweder
[eigentümerführendes Programm](#owning-program) oder
[eigentümerführende Wallet](#owning-wallet) anstelle des allgemeinen Begriffs
'Eigentümer' zu verwenden.

## Eigentümerführendes Programm

Die [Adresse](#address) des [Programms](#onchain-program), das das Konto
besitzt. Nur das eigentümerführende Programm ist in der Lage, das Konto zu
ändern.

Siehe auch [Autorität](#authority).

## Eigentümer-Wallet

Die Wallet, die mit einem Konto verknüpft ist. Zum Beispiel wird das Token
Program verwendet, um token accounts zu speichern, die mit einer bestimmten
Wallet und Mint verbunden sind, unter Verwendung einer Adresse, die aus dieser
[Wallet](#wallet) und [Mint](#token-mint) erstellt wurde.

## PoH

Siehe [Proof of History](#proof-of-history-poh).

## Punkt

Ein gewichtetes [Guthaben](#credit) in einem Belohnungssystem. Im
[Validatoren](#validator)-[Belohnungssystem](https://docs.anza.xyz/consensus/stake-delegation-and-rewards)
ist die Anzahl der Punkte, die einem [Stake](#stake) bei der Einlösung zustehen,
das Produkt aus den verdienten [Vote-Credits](#vote-credit) und der Anzahl der
[Lamports](#lamport), die gestaked wurden.

## Private Key

Der private Schlüssel eines [Keypairs](#keypair).

## Programm

Siehe [Onchain-Programm](#onchain-program).

## Program Derived Address (PDA)

Ein Konto, dessen Adresse auf Basis eines [Onchain-Programms](#onchain-program)
und eines oder mehrerer [Seeds](#seed) erstellt wird, die vom Programmautor
definiert wurden. PDA-Adressen sind keine öffentlichen Schlüssel, und PDAs haben
keine privaten Schlüssel.

Die Signierungsberechtigung für eine PDA liegt beim
[Onchain-Programm](#onchain-program), das ihr
[Eigentümerprogramm](#owning-program) ist.

## Programm-ID

Der öffentliche Schlüssel des [Kontos](#account), das ein
[Programm](#onchain-program) enthält.

## Proof of History (PoH)

Ein Stapel von Beweisen, von denen jeder nachweist, dass bestimmte Daten
existierten, bevor der Beweis erstellt wurde, und dass eine präzise Zeitspanne
seit dem vorherigen Beweis vergangen ist. Wie eine
[VDF](#verifiable-delay-function-vdf) kann ein Proof of History in kürzerer Zeit
verifiziert werden, als er zur Erstellung benötigte.

## Priorisierungsgebühr

Eine zusätzliche Gebühr, die Nutzer in den Anweisungen des Rechenbudgets angeben
können, um ihre [Transaktionen](#transaction) zu priorisieren.

Die Priorisierungsgebühr wird berechnet, indem die angeforderten maximalen
Recheneinheiten mit dem Recheneinheitenpreis (angegeben in Schritten von
0,000001 [Lamports](#lamport) pro Recheneinheit) multipliziert und auf den
nächsten [Lamport](#lamport) aufgerundet werden.

Transaktionen sollten die minimal erforderliche Menge an Recheneinheiten für die
Ausführung anfordern, um die Fee zu minimieren.

## Öffentlicher Schlüssel (pubkey)

Der öffentliche Schlüssel eines [keypair](#keypair). Ein öffentlicher Schlüssel
ist eine der verschiedenen Arten von [Adressen](#address), die ein Konto auf
Solana verwenden kann.

## rent

Fee, die von [Konten](#account) und [Programmen](#program) gezahlt wird, um
Daten auf der Blockchain zu speichern. Wenn Konten nicht über ausreichend
Guthaben verfügen, um rent zu zahlen, können sie durch Garbage Collection
entfernt werden.

Siehe auch [rent exempt](#rent-exempt) weiter unten. Erfahren Sie hier mehr über
rent: [Was ist rent?](/docs/core/accounts/account-structure).

## rent exempt

Konten, die ein minimales lamport-Guthaben aufrechterhalten, das proportional
zur Menge der auf dem Konto gespeicherten Daten ist. Alle neu erstellten Konten
werden dauerhaft onchain gespeichert, bis das Konto geschlossen wird. Es ist
nicht möglich, ein Konto zu erstellen, das unter die rent-Befreiungsschwelle
fällt.

## Root

Ein [Block](#block) oder [slot](#slot), der auf einem [Validatoren](#validator)
die maximale [Sperrfrist](#lockout) erreicht hat. Der Root ist der höchste
Block, der ein Vorfahre aller aktiven Forks auf einem validator ist. Alle
Vorfahren-Blöcke eines Root sind transitiv ebenfalls ein Root. Blöcke, die weder
ein Vorfahre noch ein Nachkomme des Root sind, werden von der Konsensbildung
ausgeschlossen und können verworfen werden.

## Runtime

Die Komponente eines [Validatoren](#validator), die für die Ausführung von
[Programmen](#program) verantwortlich ist.

## Sealevel

Solanas parallele Laufzeitumgebung für [Onchain-Programme](#onchain-program).

## seed

Ein Byte-Wert oder ein Array von Byte-Werten, das als Eingabe bei der Ableitung
einer [Program Derived Address](#program-derived-address-pda) verwendet wird.
Ein Programm kann jede Kombination aus statischen Zeichenketten, öffentlichen
Schlüsseln oder anderen Daten als seed verwenden. Seeds werden mit der ID des
Programms und einem [bump](#bump) kombiniert, um eine eindeutige PDA-Adresse zu
erzeugen.

## shred

Ein Bruchteil eines [Block](#block); die kleinste Einheit, die zwischen
[Validatoren](#validator) übertragen wird.

## Signatur

Eine 64-Byte-ed25519-Signatur aus R (32 Bytes) und S (32 Bytes). Mit der
Anforderung, dass R ein gepackter Edwards-Punkt niedriger Ordnung ist und S ein
Skalar im Bereich von `0 <= S < L`. Diese Anforderung stellt sicher, dass keine
Signatur- Formbarkeit besteht. Jede Transaktion muss mindestens eine Signatur
für das [Gebührenkonto](#fee-account) haben. Somit kann die erste Signatur einer
Transaktion als [Transaktions-ID](#transaction-id) behandelt werden

## Ausfallrate

Der Prozentsatz der [übersprungenen Slots](#skipped-slot) aus der Gesamtzahl der
Leader-Slots in der aktuellen Epoche. Diese Metrik kann irreführend sein, da sie
eine hohe Varianz nach der Epochengrenze aufweist, wenn die Stichprobengröße
klein ist, sowie für Validatoren mit einer geringen Anzahl von Leader-Slots. Sie
kann jedoch auch nützlich sein, um Fehlkonfigurationen von Knoten zu
identifizieren.

## übersprungener Slot

Ein vergangener [Slot](#slot), der keinen [Block](#block) erzeugt hat, weil der
Leader offline war oder die [Fork](#fork), die den Slot enthält, zugunsten einer
besseren Alternative durch den Cluster-Konsens aufgegeben wurde. Ein
übersprungener Slot erscheint nicht als Vorgänger für Blöcke in nachfolgenden
Slots, erhöht nicht die [Blockhöhe](#block-height) und lässt nicht die älteste
`recent_blockhash` ablaufen.

Ob ein Slot übersprungen wurde, kann erst festgestellt werden, wenn er älter als
der neueste [verwurzelte](#root) (also nicht übersprungene) Slot ist.

## slot

Der Zeitraum, in dem jeder [Leader](#leader) Transaktionen aufnimmt und einen
[Block](#block) erzeugt.

Gemeinsam erzeugen Slots eine logische Uhr. Slots sind sequenziell angeordnet
und überlappen sich nicht, wobei sie ungefähr gleiche reale Zeit gemäß
[PoH](#proof-of-history-poh) umfassen.

## Smart Contract

Siehe [Onchain-Programm](#onchain-program).

## SOL

Der [native Token](#native-token) eines Solana-[Clusters](#cluster).

## Solana Program Library (SPL)

Eine [Programmbibliothek](https://www.solana-program.com/docs/) auf Solana, wie
z. B. spl-token, die Aufgaben wie das Erstellen und Verwenden von Token
erleichtert.

## Stake

Token, die dem [Cluster](#cluster) verfallen, wenn böswilliges
[Validatoren](#validator)-Verhalten nachgewiesen werden kann.

## Stake-Weighted Quality of Service (SWQoS)

SWQoS ermöglicht eine
[bevorzugte Behandlung für Transaktionen, die von gestakten Validatoren stammen](/developers/guides/advanced/stake-weighted-qos).

## Supermajority

2/3 eines [Clusters](#cluster).

## System Program

Ein natives Solana-Programm (Programm-ID `11111111111111111111111111111111`),
das für die Erstellung neuer [Konten](#account), die Zuweisung von Kontodaten,
die Zuweisung von Konten an Programme und die Übertragung von [SOL](#sol)
zwischen Konten verantwortlich ist. Alle neuen Konten auf Solana gehören dem
System Program, bis sie neu zugewiesen werden.

## Sysvar

Ein System-[Konto](#account). [Sysvars](https://docs.anza.xyz/runtime/sysvars)
stellen Cluster-Zustandsinformationen wie die aktuelle Tick-Höhe, Belohnungs-
[Punkte](#point)-Werte usw. bereit. Programme können auf Sysvars über ein
Sysvar-Konto (Pubkey) oder durch Abfrage über einen Syscall zugreifen.

## Thin Client

Eine Art von [Client](#client), der darauf vertraut, dass er mit einem gültigen
[Cluster](#cluster) kommuniziert.

## Tick

Ein Ledger-[Eintrag](#entry), der die Wanduhr-Dauer schätzt.

## Tick-Höhe

Der N-te [Tick](#tick) im [Ledger](#ledger).

## Token

Ein digital übertragbares Asset.

## token account

Ein [Konto](#account), das ein Guthaben eines bestimmten [Tokens](#token) hält.
Token-Konten gehören dem [Token Program](#token-program) oder dem
[Token Extensions Program](#token-extensions-program) und speichern das
Token-Guthaben, die zugehörige [Token Mint](#token-mint) und die Adresse der
besitzenden Wallet.

## Token Extensions Program

Das [Token Extensions Program](https://www.solana-program.com/docs/token-2022)
hat die Programm-ID `TokenzQdBNbLqP5VEhdkAS6EPFLC1PHnBqCXEpPxuEb` und umfasst
alle Funktionen des [Token Program](#token-program), bietet jedoch zusätzliche
Erweiterungen wie vertrauliche Überweisungen, benutzerdefinierte Transferlogik,
erweiterte Metadaten und vieles mehr.

## Token Mint

Ein Konto, das Token produzieren (oder „minten“) kann. Verschiedene Token werden
durch ihre eindeutigen Token-Mint-Adressen unterschieden.

## Token Program

Das [Token Program](https://www.solana-program.com/docs/token) hat die
Programm-ID `TokenkegQfeZyiNwAJbNbGKPFXCWuBvf9Ss623VQ5DA` und bietet die
grundlegenden Funktionen zum Übertragen, Einfrieren und Minten von Token.

## tps

[Transaktionen](#transaction) pro Sekunde.

## tpu

[Transaction Processing Unit](https://docs.anza.xyz/validator/tpu).

## Transaktion

Eine oder mehrere [Anweisungen](#instruction), die von einem [Client](#client)
mit einem oder mehreren [Keypairs](#keypair) signiert und atomar mit nur zwei
möglichen Ergebnissen ausgeführt werden: Erfolg oder Fehler.

## Transaktions-ID

Die erste [Signatur](#signature) in einer [Transaktion](#transaction), die zur
eindeutigen Identifizierung der Transaktion im gesamten [Ledger](#ledger)
verwendet werden kann.

## Transaktionsbestätigungen

Die Anzahl der [bestätigten Blöcke](#confirmed-block) seit der Aufnahme der
Transaktion in das [Ledger](#ledger). Eine Transaktion ist finalisiert, wenn ihr
Block zu einem [Root](#root) wird.

## Transaktionseintrag

Eine Gruppe von [Transaktionen](#transaction), die parallel ausgeführt werden
können.

## tvu

[Transaction Validation Unit](https://docs.anza.xyz/validator/tvu).

## Validatoren

Ein vollwertiger Teilnehmer in einem Solana-Netzwerk-[Cluster](#cluster), der
neue [Blöcke](#block) produziert. Ein Validator validiert die Transaktionen, die
dem [Ledger](#ledger) hinzugefügt werden.

## VDF

Siehe [Verifiable Delay Function](#verifiable-delay-function-vdf).

## Verifiable Delay Function (VDF)

Eine Funktion, die eine feste Zeitspanne zur Ausführung benötigt und einen
Beweis erzeugt, dass sie ausgeführt wurde, der anschließend in kürzerer Zeit
verifiziert werden kann, als die Erzeugung gedauert hat.

## Versionierte Transaktion

Ein Transaktionsformat, das zusätzliche Funktionen über das ursprüngliche
[Transaktionsformat](#transaction) hinaus unterstützt. Version 0 (`v0`) ist das
erste versionierte Format und bietet Unterstützung für
[Address Lookup Tables](#address-lookup-table-alt), wodurch Transaktionen mehr
Konten referenzieren können, als das ursprüngliche Format erlaubt.

## Vote

Siehe [Ledger Vote](#ledger-vote).

## Vote Credit

Eine Belohnungsbilanz für [Validatoren](#validator). Ein Vote Credit wird einem
Validator in seinem Vote-Konto gutgeschrieben, wenn der Validator eine
[Root](#root) erreicht.

## Wallet

Eine Sammlung von [Keypairs](#keypair), die es Benutzern ermöglicht, ihre
Guthaben zu verwalten.

## Warmup-Periode

Eine bestimmte Anzahl von [Epochs](#epoch), nachdem [Stake](#stake) delegiert
wurde, während dessen er schrittweise wirksam wird. Während dieser Periode gilt
der Stake als "aktivierend". Weitere Informationen über:
[Warmup und Cooldown](https://docs.anza.xyz/consensus/stake-delegation-and-rewards#stake-warmup-cooldown-withdrawal)
