Die folgenden Begriffe werden in der gesamten Solana-Dokumentation und im
Entwicklungs- ökosystem verwendet.

## account

Ein Eintrag im Solana-Ledger, der entweder Daten enthält oder ein ausführbares
Programm ist.

Wie ein Konto bei einer traditionellen Bank kann ein Solana-Konto Guthaben
namens [Lamports](#lamport) halten. Jedes Konto in einem [Cluster](#cluster) hat
eine eindeutige [Adresse](#address).

## App

Eine Frontend-Anwendung, die mit einem Solana-Cluster interagiert.

## Adresse

Der Speicherort in Solana, der verwendet werden kann, um ein Konto zu finden.
Jede Adresse ist einzigartig im Solana-[Cluster](#cluster).

Adressen in Solana können entweder ein Ed25519-[Public Key](#public-key-pubkey)
oder eine [Program Derived Address](#program-derived-address-pda) sein.

## Address Lookup Table (ALT)

Eine Datenstruktur, die eine Liste von Kontoadressen speichert und es
[versionierten Transaktionen](#versioned-transaction) ermöglicht, auf Konten
mithilfe kompakter 1-Byte-Indizes anstelle vollständiger 32-Byte-Public-Keys zu
verweisen. Address Lookup Tables ermöglichen es einer einzelnen Transaktion, auf
mehr Konten zu verweisen, als das ursprüngliche Transaktionsformat erlaubt.

## Anchor

Ein Framework zum Erstellen von Solana-[Onchain-Programmen](#onchain-program) in
Rust. Anchor bietet Makros, die Boilerplate-Code reduzieren, ein
standardisiertes Kontovalidierungssystem und generiert eine [IDL](#idl), die die
Schnittstelle des Programms beschreibt. Es ist das am häufigsten verwendete
Framework für die Entwicklung von Solana-Programmen.

## Associated Token Account (ATA)

Ein [token account](#token-account), dessen Adresse deterministisch von einer
Wallet-Adresse und einem [Token Mint](#token-mint) abgeleitet wird. Das
Associated Token Program stellt sicher, dass jede Wallet ein kanonisches
Token-Konto pro Token-Typ hat.

## Autorisierung

Die Adresse eines Benutzers, der eine bestimmte Berechtigung über ein Konto hat.

Zum Beispiel:

- Die Berechtigung zum Prägen neuer Token wird dem Konto erteilt, das die
  'Mint-Autorisierung' für den Token Mint ist.
- Die Berechtigung zum Aktualisieren eines Programms wird dem Konto erteilt, das
  die 'Upgrade-Autorisierung' eines Programms ist.

## Automated Market Maker (AMM)

Ein Onchain-Börsenmodell, bei dem Nutzer gegen einen Pool von Token-Reserven
handeln, der durch eine Preisformel gesteuert wird, anstatt gegen die offenen
Orders einer Gegenpartei. Die Preise bewegen sich als Funktion der Reserven,
sodass Trades einen [Slippage](#slippage) verursachen, der proportional zur
Größe ist.

## Bank State

Das Ergebnis der Interpretation aller Programme im Ledger bei einer gegebenen
[Tick Height](#tick-height). Es umfasst mindestens die Menge aller
[Konten](#account), die nicht-null [Native Tokens](#native-token) halten.

## Block

Eine zusammenhängende Menge von [Entries](#entry) im Ledger, die durch ein
[Vote](#ledger-vote) abgedeckt wird. Ein [Leader](#leader) erzeugt höchstens
einen Block pro [Slot](#slot).

## Blockhash

Ein eindeutiger Wert ([Hash](#hash)), der einen Datensatz (Block) identifiziert.
Solana berechnet einen Blockhash aus der letzten [Entry ID](#entry-id) des
Blocks.

## Block Height

Die Anzahl der [Blocks](#block) unterhalb des aktuellen Blocks. Der erste Block
nach dem [Genesis Block](#genesis-block) hat die Höhe eins.

## Bootstrap Validator

Der [Validator](#validator), der den Genesis-(ersten) [Block](#block) einer
Blockchain erzeugt.

## BPF Loader

Das Solana-Programm, das
[BPF](/docs/core/programs)-[Onchain-Programme](#onchain-program) besitzt und
lädt, wodurch das Programm mit der Runtime interagieren kann.

## Bump

Ein einzelnes Byte (0–255), das an [Seeds](#seed) angehängt wird, wenn eine
[Program Derived Address](#program-derived-address-pda) abgeleitet wird, um
sicherzustellen, dass die resultierende Adresse außerhalb der ed25519-Kurve
liegt und somit eine gültige PDA ist. Der kanonische Bump ist der höchste Wert,
der eine gültige PDA erzeugt, und wird typischerweise in den Kontodaten
gespeichert, um eine erneute Berechnung zu vermeiden.

## Central Limit Order Book (CLOB)

Ein Börsenmodell, das eine sortierte Liste offener Kauf- und Verkaufsaufträge
führt; ein Trade erfolgt, wenn eine eingehende Order eine Order auf der
gegenüberliegenden Seite kreuzt. Im Gegensatz zum
[Automated Market Maker](#automated-market-maker-amm).

## Client

Ein Computerprogramm, das auf das Solana-Servernetzwerk [Cluster](#cluster)
zugreift.

## Commitment

Ein Maß für die Netzwerkbestätigung des [Blocks](#block).

## Kompaktes Array-Format

Ein kompaktes Array ist ein Array, das im folgenden Format serialisiert wird:

- Die Array-Länge (kodiert als
  [compact-u16](https://github.com/anza-xyz/agave/blob/v2.1.13/short-vec/src/lib.rs))
- Die Array-Elemente nacheinander aufgelistet

![Kompaktes Array-Format](/assets/docs/core/transactions/compact_array_format.png)

Das kompakte Array-Format wird verwendet, um sowohl das Array der
[Kontoadressen](/docs/core/transactions/transaction-structure#account-addresses)
als auch das Array der
[Anweisungen](/docs/core/transactions/transaction-structure#instructions) in
Transaktionsnachrichten zu kodieren.

## Cluster

Eine Gruppe von [Validatoren](#validator), die ein einziges [Ledger](#ledger)
pflegen.

## Compute-Budget

Die maximale Anzahl an [Compute-Einheiten](#compute-units), die pro Transaktion
verbraucht werden.

## Compute-Einheiten

Die kleinste Maßeinheit für den Verbrauch von Rechenressourcen der Blockchain.
Compute-Einheiten werden verwendet, um die Rechenressourcen zu messen, die zur
Verarbeitung einer Transaktion benötigt werden.

## Bestätigungszeit

Die Wanduhrzeit zwischen der Erstellung eines [Tick-Eintrags](#tick) durch einen
[Leader](#leader) und der Erstellung eines
[bestätigten Blocks](#confirmed-block).

## Bestätigter Block

Ein [Block](#block), der eine [Supermehrheit](#supermajority) von
[Ledger-Stimmen](#ledger-vote) erhalten hat.

## Kontrollebene

Ein Gossip-Netzwerk, das alle [Nodes](#node) eines [Clusters](#cluster)
verbindet.

## Abkühlungsperiode

Eine bestimmte Anzahl von [Epochen](#epoch), nachdem [Stake](#stake) deaktiviert
wurde, während er schrittweise für Auszahlungen verfügbar wird. Während dieser
Zeit gilt der Stake als "deaktivierend". Weitere Informationen:
[Warmup und Cooldown](https://docs.anza.xyz/consensus/stake-delegation-and-rewards#stake-warmup-cooldown-withdrawal)

## Kosteneinheiten

Auch bezeichnet als `transaction cost`, ist eine umfassende Schätzung aller
Ressourcen, die zur Verarbeitung einer Transaktion erforderlich sind, gemessen
in [Compute-Einheiten](#compute-units). Weitere Informationen
[hier](https://www.anza.xyz/blog/why-solana-transaction-costs-and-compute-units-matter-for-developers).

## credit

Siehe [vote credit](#vote-credit).

## Cross Program Invocation (CPI)

Ein Aufruf von einem [Onchain-Programm](#onchain-program) zu einem anderen.
Weitere Informationen finden Sie unter
[Aufrufe zwischen Programmen](/docs/core/cpi).

## Datenebene

Ein Multicast-Netzwerk, das zur effizienten Validierung von [Einträgen](#entry)
und zur Konsensbildung verwendet wird.

## Drone

Ein Offchain-Dienst, der als Verwahrer für den privaten Schlüssel eines
Benutzers fungiert. Er dient in der Regel dazu, Transaktionen zu validieren und
zu signieren.

## Eintrag

Ein Eintrag im [Ledger](#ledger), entweder ein [Tick](#tick) oder ein
[Transaktionseintrag](#transactions-entry).

## Eintrags-ID

Ein vorbildresistenter [Hash](#hash) über die finalen Inhalte eines Eintrags,
der als global eindeutiger Identifikator des [Eintrags](#entry) dient. Der Hash
dient als Nachweis für:

- Die Generierung des Eintrags nach einer bestimmten Zeitdauer
- Die angegebenen [Transaktionen](#transaction) sind diejenigen, die im Eintrag
  enthalten sind
- Die Position des Eintrags in Bezug auf andere Einträge im [Ledger](#ledger)

Siehe [Proof of History](#proof-of-history-poh).

## epoch

Die Zeit, d. h. die Anzahl der [Slots](#slot), für die ein
[Leader Schedule](#leader-schedule) gültig ist.

## Escrow

Ein Muster, bei dem ein [Onchain-Programm](#onchain-program) Token in einem
[Vault](#vault) im Namen von zwei oder mehr Parteien hält, bis die vom Programm
definierten Bedingungen erfüllt sind, und sie dann freigibt. Wird verwendet, um
die Notwendigkeit zu beseitigen, dass die Parteien einander direkt vertrauen
müssen.

## Fee-Konten

Das Fee-Konten in der Transaktion ist das Konten, das für die Kosten der
Aufnahme der Transaktion in das Ledger bezahlt. Dies ist das erste Konten in der
Transaktion. Dieses Konten muss in der Transaktion als Read-Write (beschreibbar)
deklariert werden, da die Bezahlung der Transaktion den Kontostand reduziert.

## Finalität

Wenn Knoten, die 2/3 des [Stake](#stake) repräsentieren, eine gemeinsame
[Root](#root) haben.

## Fork

Ein [Ledger](#ledger), der von gemeinsamen Einträgen abgeleitet wurde, aber dann
divergiert ist.

## Genesis-Block

Der erste [Block](#block) in der Kette.

## Genesis-Config

Die Konfigurationsdatei, die das [Ledger](#ledger) für den
[Genesis-Block](#genesis-block) vorbereitet.

## Hash

Ein digitaler Fingerabdruck einer Bytefolge.

## Inflation

Eine Erhöhung des Token-Angebots im Laufe der Zeit, die zur Finanzierung von
Belohnungen für die Validierung und zur Finanzierung der fortlaufenden
Entwicklung von Solana verwendet wird.

## Inner Instruction

Siehe [Cross Program Invocation](#cross-program-invocation-cpi).

## Instruction

Ein Aufruf zum Aufrufen eines bestimmten
[Instruction-Handlers](#instruction-handler) in einem [Programm](#program). Eine
Instruction gibt auch an, welche Konten sie lesen oder ändern möchte, sowie
zusätzliche Daten, die als zusätzliche Eingabe für den
[Instruction-Handler](#instruction-handler) dienen. Ein [Client](#client) muss
mindestens eine Instruction in einer [Transaktion](#transaction) einschließen,
und alle Anweisungen müssen erfolgreich abgeschlossen werden, damit die
Transaktion als erfolgreich gilt.

## Instruction-Handler

Instruction-Handler sind [Programm](#program)-Funktionen, die
[Anweisungen](#instruction) aus [Transaktionen](#transaction) verarbeiten. Ein
Instruction-Handler kann eine oder mehrere
[Cross-Program Invocations](#cross-program-invocation-cpi) enthalten.

## IDL

Interface Definition Language. Eine JSON-Datei, die von [Anchor](#anchor)
generiert wird und die [Instruction-Handler](#instruction-handler), Konten und
Typen eines Programms beschreibt. Die IDL ermöglicht es Client-Bibliotheken,
automatisch typisierte Schnittstellen für die Interaktion mit einem
[Onchain-Programm](#onchain-program) zu generieren.

## Keypair

Ein [Public Key](#public-key-pubkey) und der entsprechende
[Private Key](#private-key) für den Zugriff auf ein Konto.

## Lamport

Ein Bruchteil eines [Native Token](#native-token) mit dem Wert von 0,000000001
[SOL](#sol).

> Innerhalb des Compute-Budgets wird eine Menge von
> _[Micro-Lamports](https://github.com/solana-labs/solana/blob/ced8f6a512c61e0dd5308095ae8457add4a39e94/program-runtime/src/prioritization_fee.rs#L1-L2)_
> bei der Berechnung von [Prioritization Fee](#prioritization-fee) verwendet.

## Leader

Die Rolle eines [Validatoren](#validator) beim Anhängen von [Einträgen](#entry)
an das [Ledger](#ledger).

## Leader-Zeitplan

Eine Sequenz von
[Validator](#validator)-[öffentlichen Schlüsseln](#public-key-pubkey), die
[Slots](#slot) zugeordnet sind. Das Cluster verwendet den Leader-Zeitplan, um zu
bestimmen, welcher Validator zu jedem Zeitpunkt der [Leader](#leader) ist.

## Ledger

Eine Liste von [Einträgen](#entry), die [Transaktionen](#transaction) enthalten,
die von [Clients](#client) signiert wurden. Konzeptionell lässt sich dies bis
zum [Genesis-Block](#genesis-block) zurückverfolgen, aber das tatsächliche
Ledger eines [Validatoren](#validator) enthält möglicherweise nur neuere
[Blöcke](#block), um Speicherplatz zu reduzieren, da ältere Blöcke aufgrund des
Designs nicht für die Validierung zukünftiger Blöcke benötigt werden.

## Ledger-Vote

Ein [Hash](#hash) des [Validator-Zustands](#bank-state) bei einer bestimmten
[Tick-Höhe](#tick-height). Es umfasst die Bestätigung eines
[Validatoren](#validator), dass ein empfangener [Block](#block) verifiziert
wurde, sowie ein Versprechen, nicht für einen widersprüchlichen [Block](#block)
\(d. h. [Fork](#fork)\) für einen bestimmten Zeitraum, die
[Lockout](#lockout)-Periode, zu stimmen.

## Light Client

Eine Art von [Client](#client), der verifizieren kann, dass er auf ein gültiges
[Cluster](#cluster) verweist. Er führt mehr Ledger-Verifizierung durch als ein
[Thin Client](#thin-client) und weniger als ein [Validator](#validator).

## Liquidität

Die Tiefe der verfügbaren Orders oder gepoolten Reserven, die zum oder nahe dem
aktuellen Preis gehandelt werden können. Tiefere Liquidität reduziert
[Slippage](#slippage). Bei einem [CLOB](#central-limit-order-book-clob) wird
Liquidität durch [Maker-Orders](#maker) bereitgestellt; bei einem
[AMM](#automated-market-maker-amm) wird sie von Liquiditätsanbietern
bereitgestellt, die Token-Paare in den Pool einzahlen.

## Loader

Ein [Programm](#program) mit der Fähigkeit, die binäre Kodierung anderer
[Onchain-Programme](#onchain-program) zu interpretieren.

## Lockout

Die Zeitdauer, für die ein [Validator](#validator) nicht in der Lage ist, über
einen anderen [Fork](#fork) zu [voten](#ledger-vote).

## Maker

Ein Händler, dessen Order [Liquidität](#liquidity) zu einem
[CLOB](#central-limit-order-book-clob) hinzufügt, indem sie im Orderbuch
verbleibt, anstatt sofort die Gegenseite zu kreuzen. Eine solche Order wird als
Maker-Order bezeichnet. Maker zahlen in der Regel keine Gebühr oder erhalten
eine Rückvergütung von der Handelsplattform.

## Market Maker

Eine Einheit, die [Liquidität](#liquidity) bereitstellt, indem sie
kontinuierlich sowohl Geld- als auch Briefkurse stellt. Auf Solana sind Market
Maker entweder Programme (wie [AMMs](#automated-market-maker-amm)) oder
Offchain-Firmen, die Bots betreiben, welche [Maker-Orders](#maker) auf
[CLOBs](#central-limit-order-book-clob) platzieren.

## Nachricht

Der strukturierte Inhalt einer [Transaktion](#transaction). Enthält in der Regel
einen Header, ein Array von Kontoadressen, einen aktuellen
[Blockhash](#blockhash) und ein Array von [ Anweisungen](#instruction).

Erfahren Sie mehr über die
[Nachrichtenformatierung innerhalb von Transaktionen](/docs/core/transactions/transaction-structure#header)
hier.

## Micro-Lamport

1.000.000 Micro-Lamports = 1 [Lamport](#lamport)

## Nakamoto-Koeffizient

Ein Maß für Dezentralisierung, der Nakamoto-Koeffizient ist die kleinste Anzahl
unabhängiger Einheiten, die gemeinsam handeln können, um eine Blockchain
stillzulegen. Der Begriff wurde von Balaji S. Srinivasan und Leland Lee in
[Quantifying Decentralization](https://news.earn.com/quantifying-decentralization-e39db233c28e)
geprägt.

## Nativer Token

Der [Token](#token), der verwendet wird, um die von [Nodes](#node) in einem
[Cluster](#cluster) geleistete Arbeit zu erfassen.

## Node

Ein Computer, der an einem [Cluster](#cluster) teilnimmt.

## Node-Anzahl

Die Anzahl der Validatoren, die an einem [Cluster](#cluster) teilnehmen.

## Offchain

Beschreibt Daten, Prozesse oder Dienste, die außerhalb der Solana-Blockchain
existieren oder ausgeführt werden, wie z. B. RPC-Clients, Indexer, Frontends und
Offchain-Orderbuch-Maker. Gegenteil von [Onchain](#onchain). "Offchain" (ohne
Bindestrich) ist die bevorzugte Schreibweise.

## Onchain

Beschreibt Daten oder Programme, die auf einer Blockchain existieren oder
ausgeführt werden. "Onchain" (ohne Bindestrich) ist die bevorzugte Schreibweise.

## Onchain-Programm

Der ausführbare Code auf der Solana-Blockchain, der die
[Anweisungen](#instruction) interpretiert, die in jeder
[Transaktion](#transaction) gesendet werden, um Konten zu lesen und zu ändern,
über die er die Kontrolle hat. Diese Programme werden auf anderen Blockchains
oft als "[_Smart Contracts_](/docs/core/programs)" bezeichnet.

## Oracle

Ein [Onchain-Programm](#onchain-program), das Daten von außerhalb der Chain
veröffentlicht – meist Preise von realen Vermögenswerten –, damit andere
Programme diese lesen können.

## Eigentümer

Kann sich entweder auf das [besitzende Programm](#owning-program) oder die
[besitzende Wallet](#owning-wallet) beziehen. Zum Beispiel ruft
[getTokenAccountsByOwner](https://solana.com/docs/rpc/http/gettokenaccountsbyowner)
Token-Konten ab, die einer bestimmten Wallet gehören, wobei das besitzende
Programm für diese Token-Konten das [Token Program](#token-program) ist.

Um Verwirrung zu vermeiden, sollte vorzugsweise entweder
[besitzendes Programm](#owning-program) oder [besitzende Wallet](#owning-wallet)
anstelle des allgemeinen Begriffs "Eigentümer" verwendet werden.

## Besitzendes Programm

Die [Adresse](#address) des [Programms](#onchain-program), das das Konto
besitzt. Nur das besitzende Programm ist in der Lage, das Konto zu ändern.

Siehe auch [Berechtigung](#authority).

## Besitzende Wallet

Die Wallet, die mit einem Konto verknüpft ist. Beispielsweise wird das Token
Program verwendet, um Token-Konten zu speichern, die mit einer bestimmten Wallet
und Mint verknüpft sind, wobei eine Adresse aus dieser [Wallet](#wallet) und
diesem [Mint](#token-mint) erstellt wird.

## PoH

Siehe [Proof of History](#proof-of-history-poh).

## Punkt

Ein gewichtetes [Guthaben](#credit) in einem Belohnungssystem. Im
[Validatoren](#validator)-[Belohnungssystem](https://docs.anza.xyz/consensus/stake-delegation-and-rewards)
ist die Anzahl der Punkte, die einem [Stake](#stake) bei der Einlösung zustehen,
das Produkt aus den verdienten [Vote Credits](#vote-credit) und der Anzahl der
gestakten [Lamports](#lamport).

## Privater Schlüssel

Der private Schlüssel eines [Keypairs](#keypair).

## Programm

Siehe [Onchain-Programm](#onchain-program).

## Program Derived Address (PDA)

Ein Konto, dessen Adresse auf Basis eines [Onchain-Programms](#onchain-program)
und einer oder mehrerer [Seeds](#seed) erstellt wird, die vom Programmautor
erstellt wurden. PDA-Adressen sind keine öffentlichen Schlüssel, und PDAs haben
keine privaten Schlüssel.

Die Signierungsberechtigung für eine PDA liegt beim
[Onchain-Programm](#onchain-program), das ihr
[besitzendes Programm](#owning-program) ist.

## Programm-ID

Der öffentliche Schlüssel des [Konten](#account), das ein
[Programm](#onchain-program) enthält.

## Proof of History (PoH)

Ein Stapel von Nachweisen, von denen jeder beweist, dass bestimmte Daten
existierten, bevor der Nachweis erstellt wurde, und dass eine präzise Zeitspanne
vor dem vorherigen Nachweis verstrichen ist. Wie eine
[VDF](#verifiable-delay-function-vdf) kann ein Proof of History in kürzerer Zeit
verifiziert werden, als seine Erstellung dauerte.

## Priorisierungsgebühr

Eine zusätzliche Gebühr, die Benutzer in der Compute-Budget- Anweisungen angeben
können, um ihre [Transaktionen](#transaction) zu priorisieren.

Die priority fee wird berechnet, indem die angeforderten maximalen
Recheneinheiten mit dem Recheneinheitspreis (angegeben in Schritten von 0,000001
[lamport](#lamport) pro Recheneinheit) multipliziert und auf den nächsten
[lamport](#lamport) aufgerundet werden.

Transaktionen sollten die minimal erforderliche Menge an Recheneinheiten für die
Ausführung anfordern, um die Fee zu minimieren.

## Öffentlicher Schlüssel (pubkey)

Der öffentliche Schlüssel eines [keypair](#keypair). Ein pubkey ist eine der
verschiedenen Arten von [Adressen](#address), die ein Konto auf Solana verwenden
kann.

## rent

Fee, die von [Konten](#account) und [Programmen](#program) zur Speicherung von
Daten auf der Blockchain bezahlt wird. Wenn Konten nicht über ausreichend
Guthaben zur Zahlung von rent verfügen, können sie per Garbage Collection
entfernt werden.

Siehe auch [rent-befreit](#rent-exempt) unten. Erfahren Sie hier mehr über rent:
[Was ist rent?](/docs/core/accounts/account-structure).

## rent-befreit

Konten, die ein Mindestguthaben in lamport aufrechterhalten, das proportional
zur auf dem Konto gespeicherten Datenmenge ist. Alle neu erstellten Konten
werden dauerhaft onchain gespeichert, bis das Konto geschlossen wird. Es ist
nicht möglich, ein Konto zu erstellen, das unter die rent-Befreiungsschwelle
fällt.

## root

Ein [Block](#block) oder slot, der die maximale [Sperrfrist](#lockout) auf einem
Validatoren erreicht hat. Der Root ist der höchste Block, der ein Vorfahre aller
aktiven Forks auf einem Validatoren ist. Alle vorherigen Blöcke eines Roots sind
ebenfalls transitiv ein Root. Blöcke, die weder ein Vorfahre noch ein Nachfolger
des Roots sind, werden vom Konsens ausgeschlossen und können verworfen werden.

## Laufzeitumgebung

Die Komponente eines Validatoren, die für die Ausführung von
[Programmen](#program) verantwortlich ist.

## Sealevel

Solanas parallele Laufzeitumgebung für [Onchain-Programme](#onchain-program).

## seed

Ein Byte-Wert oder ein Array von Byte-Werten, der als Eingabe bei der Ableitung
einer [Program Derived Address](#program-derived-address-pda) verwendet wird.
Ein Programm kann beliebige Kombinationen aus statischen Zeichenketten,
öffentlichen Schlüsseln oder anderen Daten als Seeds verwenden. Seeds werden mit
der Programm-ID und einem [Bump](#bump) kombiniert, um eine eindeutige
PDA-Adresse zu erzeugen.

## Shard

Ein Bruchteil eines [Blocks](#block); die kleinste Einheit, die zwischen
Validatoren übertragen wird.

## Signatur

Eine 64-Byte-ed25519-Signatur bestehend aus R (32 Bytes) und S (32 Bytes). Mit
der Anforderung, dass R ein gepackter Edwards-Punkt ist, der keine kleine
Ordnung aufweist, und S ein Skalar im Bereich von `0 <= S < L` ist. Diese
Anforderung stellt sicher, dass keine Signatur-Manipulierbarkeit vorliegt. Jede
Transaktion muss mindestens eine Signatur für das [Gebührenkonto](#fee-account)
haben. Daher kann die erste Signatur in einer Transaktion als
[Transaktions-ID](#transaction-id) behandelt werden.

## Signer

Ein Konten, das eine Anweisungen oder [Transaktion](#transaction) autorisiert
hat. Bei einem regulären keypair bedeutet dies eine kryptografische
[Signatur](#signature) über die Transaktion. Bei einer
[Program Derived Address](#program-derived-address-pda) wird die Autorisierung
durch das [besitzende Programm](#owning-program) bestätigt, wenn es eine Cross
Program Invocation durchführt, da eine PDA keinen privaten Schlüssel besitzt.

## Übersprungrate

Der Prozentsatz der [übersprungenen Slots](#skipped-slot) von den gesamten
Leader-Slots in der aktuellen epoch. Diese Kennzahl kann irreführend sein, da
sie eine hohe Varianz nach der Epochengrenze aufweist, wenn die Stichprobengröße
klein ist, sowie bei Validatoren mit einer geringen Anzahl von Leader-Slots. Sie
kann jedoch auch nützlich sein, um Fehlkonfigurationen von Knoten zu
identifizieren.

## übersprungener slot

Ein vergangener [slot](#slot), der keinen [Block](#block) erzeugt hat, weil der
Leader offline war oder die [Fork](#fork), die den slot enthielt, zugunsten
einer besseren Alternative durch Cluster-Konsens aufgegeben wurde. Ein
übersprungener slot erscheint nicht als Vorgänger für Blöcke in nachfolgenden
Slots, erhöht nicht die [Blockhöhe](#block-height) und lässt auch nicht den
ältesten `recent_blockhash` verfallen.

Ob ein slot übersprungen wurde, kann erst festgestellt werden, wenn er älter ist
als der neueste [verwurzelte](#root) (also nicht übersprungene) slot.

## Slippage

Die Differenz zwischen dem Preis, den ein Händler erwartet hat, und dem Preis,
zu dem sein Trade tatsächlich ausgeführt wurde. Bei
[AMMs](#automated-market-maker-amm) entsteht sie durch die Verschiebung der
Preiskurve bei sich ändernden Reserven; bei
[CLOBs](#central-limit-order-book-clob) entsteht sie dadurch, dass eine einzelne
Order mehrere ruhende Orders zu sukzessive schlechteren Preisen verbraucht.

## slot

Der Zeitraum, in dem jeder [Leader](#leader) Transaktionen verarbeitet und einen
[Block](#block) erzeugt.

Zusammen bilden Slots eine logische Uhr. Slots sind sequenziell angeordnet und
überschneiden sich nicht, wobei sie gemäß [PoH](#proof-of-history-poh) ungefähr
gleiche reale Zeitspannen umfassen.

## System Program

Ein natives Solana-Programm (Programm-ID `11111111111111111111111111111111`),
das für die Erstellung neuer [Konten](#account), die Zuweisung von Kontodaten,
die Zuweisung von Konten zu Programmen und die Übertragung von [SOL](#sol)
zwischen Konten verantwortlich ist. Alle neuen Konten auf Solana gehören dem
System Program, bis sie neu zugewiesen werden.

## Sysvar

Ein System-[Konto](#account). [Sysvars](https://docs.anza.xyz/runtime/sysvars)
stellen Cluster-Statusinformationen wie die aktuelle Tick-Höhe,
Belohnungs-[Punkte](#point)-Werte usw. bereit. Programme können auf Sysvars über
ein Sysvar-Konto (pubkey) oder durch Abfrage über einen Syscall zugreifen.

## Taker

Ein Händler, dessen Order [Liquidität](#liquidity) aus einem
[CLOB](#central-limit-order-book-clob) entfernt, indem sie die Gegenseite des
Orderbuchs kreuzt und mit einer oder mehreren ruhenden Orders übereinstimmt.
Eine solche Order wird als Taker-Order bezeichnet, und der Taker zahlt
typischerweise die Handelsgebühr der Plattform.

Token-Saldo, die zugehörige [Token-Mint](#token-mint) und die Adresse der

alle Funktionen des [Token Program](#token-program), bietet jedoch zusätzlich
Erweiterungen wie vertrauliche Transfers, benutzerdefinierte Transfer-Logik,

## Vault

Ein [token account](#token-account), das einem
[Onchain-Programm](#onchain-program) gehört und gepoolte Vermögenswerte im Namen
von Benutzern hält — zum Beispiel die Basis- und Quotierungsreserven eines
[AMM](#automated-market-maker-amm), die gesperrten Token in einem
[Escrow](#escrow) oder die ruhenden Guthaben und aufgelaufenen Gebühren eines
[CLOB](#central-limit-order-book-clob). Nur das besitzende Programm kann Token
aus dem Vault bewegen, typischerweise in das eigene
[token account](#token-account) eines Benutzers, sobald eine Handels- oder
Freigabebedingung erfüllt ist.

## Verifiable Delay Function (VDF)

Eine Funktion, die eine festgelegte Zeitspanne zur Ausführung benötigt und einen
Nachweis erzeugt, dass sie ausgeführt wurde, der dann in kürzerer Zeit
verifiziert werden kann, als die Erzeugung gedauert hat.

## versionierte Transaktion

Ein Transaktionsformat, das zusätzliche Funktionen über das ursprüngliche
[Transaktions](#transaction)-Format hinaus unterstützt. Version 0 (`v0`) ist das
erste versionierte Format und bietet Unterstützung für
[Address Lookup Tables](#address-lookup-table-alt), wodurch Transaktionen auf
mehr Konten verweisen können, als das ursprüngliche Format erlaubt.
