Eine Transaktion wird nur ausgeführt, wenn sie eine gültige
[Signatur](/docs/core/transactions/transaction-structure#signatures) von jedem
erforderlichen Signer enthält. Während der lokalen Entwicklung wird der Key in
der Regel als [`Keypair`](/developers/cookbook/wallets/create-keypair) gehalten
und direkt signiert. In der Produktion stellt sich eine andere wichtige Frage:
**Wo wird der private Key gespeichert, und wer ist berechtigt, eine Signatur
damit zu autorisieren?**

<Callout type="warn">
  Betten Sie private Keys niemals in clientseitigen Code ein, bündeln Sie sie
  nicht in einem Frontend und übertragen Sie sie nicht in die Versionskontrolle.
  Jeder, der einen Key erlangt, erhält die volle Kontrolle über die Mittel des
  betreffenden Konten.
</Callout>

## Signierungsansätze

| Ansatz                       | Wo der Key gespeichert ist                                    | Wer signiert                                    | Am besten geeignet für                                                  |
| ---------------------------- | ------------------------------------------------------------- | ----------------------------------------------- | ----------------------------------------------------------------------- |
| **Lokales keypair**          | Eine Datei oder Umgebungsvariable auf dem Rechner             | Ihr Code, direkt                                | Lokale Entwicklung, Tests, CI                                           |
| **Browser-Wallet**           | Das Wallet des Endbenutzers (z. B. Phantom, Solflare)         | Der Endbenutzer, pro Transaktion                | Frontend-dApps, bei denen Benutzer ihre eigenen Transaktionen signieren |
| **Cloud KMS / HSM**          | Ein hardwaregestützter Key-Dienst (AWS KMS, GCP KMS, Vault)   | Ihr Backend fordert eine Signatur vom Dienst an | Backend-Dienste, Treasury-Operationen, regulierte Umgebungen            |
| **Verwaltete / MPC-Wallets** | Aufgeteilt oder verwahrt in der Infrastruktur eines Anbieters | Der Anbieter co-signiert gemäß Ihrer Richtlinie | Eingebettete Wallets, Genehmigungsworkflows, institutionelle Verwahrung |

Das lokale keypair ist der einzige Ansatz, bei dem rohes Schlüsselmaterial
direkt in Ihrer Anwendung gespeichert wird. Jeder produktionsbereite Ansatz hält
den Schlüssel in dedizierter Infrastruktur und lässt diese Infrastruktur in
Ihrem Namen signieren.

## Backend-Signierung

Wenn ein Server signieren muss – Fee bezahlen, Programmtransaktionen senden oder
eine Treasury verwalten – verwenden Sie ein dediziertes
Schlüsselverwaltungs-Backend anstelle eines rohen keypair. Die Solana Foundation
pflegt [Keychain](/docs/tools/keychain), eine einheitliche Signierbibliothek
(Rust und TypeScript), die eine einzige `SolanaSigner`-Schnittstelle über alle
oben aufgeführten Backends hinweg bereitstellt: Memory, HashiCorp Vault, AWS
KMS, GCP KMS, Fireblocks, Privy, Turnkey, CDP, Crossmint, Dfns, Para, Openfort
und Utila.

Da die Schnittstelle über alle Backends hinweg identisch ist, können Sie lokal
mit einem In-Memory-Schlüssel entwickeln und durch eine Konfigurationsänderung
auf ein produktionsbereites Backend wechseln – ohne Anwendungscode neu schreiben
zu müssen. Keychain ist kompatibel mit
[`@solana/kit`](/docs/clients/official/javascript) und dem
[Rust SDK](/docs/clients/official/rust).
