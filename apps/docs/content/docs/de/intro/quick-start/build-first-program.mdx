h1: Dein erstes Solana-Programm erstellen
---

Dieses Schnellstart-Tutorial verwendet das
[Anchor-Framework](https://www.anchor-lang.com/docs) Starterprojekt, das von
`anchor init` generiert wird. Du erstellst das Projekt lokal, führst seine Tests
aus, baust das Programm und gehst den Programmcode durch, den Anchor generiert.

## Voraussetzungen

Bevor du beginnst,
[installiere die Solana-Entwicklungswerkzeuge](/docs/intro/installation). Die
Installation umfasst Rust, die Solana CLI und die Anchor CLI.

Verwende Anchor CLI Version 1.1.2 oder höher für diese Vorlage. Überprüfe deine
installierte Version:

```terminal
$ anchor --version
```

<Steps>

<Step>

### Projekt erstellen

Führe die folgenden Befehle in deinem Terminal aus:

```terminal
$ anchor init my-program
$ cd my-program
```

Das Starterprojekt enthält ein Solana-Programm unter `programs/my-program`. Das
Programm enthält zwei Anweisungen: eine zum Initialisieren eines Zähler-Konten
und eine zum Erhöhen des Zählers.

Einige Teile der Vorlage veranschaulichen gängige Solana-Programmierungsmuster:
Ableiten von [PDA](/docs/core/pda)-Konten-Adressen, Durchführen einer Cross
Program Invocation ([CPI](/docs/core/cpi)) zum Übertragen von SOL und Verwenden
benutzerdefinierter Fehlerprüfungen, um eine Anweisung zu stoppen, wenn eine
Bedingung nicht erfüllt ist.

<Step>

### Programm bauen

Führe `anchor build` aus, um das Starterprogramm zu kompilieren:

```terminal
$ anchor build
```

Das kompilierte Programm wird in `target/deploy/my_program.so` geschrieben. Wenn
das Programm bereitgestellt wird, werden die Inhalte dieser `.so`-Datei in einem
Konten onchain gespeichert.

</Step>

<Step>

### Test ausführen

Führen Sie den Standardtest aus:

```terminal
$ anchor test
```

Das `Anchor.toml` dieses Templates verwendet den Rust-Testbefehl:

```toml title="Anchor.toml"
skip_local_validator = true

[scripts]
test = "cargo test"
```

Der Test lädt das kompilierte Programm in [LiteSVM](https://www.litesvm.com/),
erstellt einen Zahler, sendet die _rs`initialize`_- und _rs`increment`_
Anweisungen und überprüft dann den Zustand des Zähler-Konten.

Das Ausführen von `anchor test` kompiliert das Programm ebenfalls, sodass Sie
beim lokalen Testen `anchor build` nicht zuerst ausführen müssen.

</Step>

<Step>

### Programm deployen

Lokale Tests bieten die schnellste Feedbackschleife. Wenn Sie bereit sind, auf
einem Netzwerk zu deployen, beispielsweise auf dem Devnet, erstellen Sie
zunächst einen Build und deployen Sie dann auf einem Cluster.

Das Deployment eines Solana-Programms erfordert SOL, da das Programm in einem
program account gespeichert wird und der program account für den genutzten
Speicherplatz bezahlen muss. Auf dem Devnet können Sie kostenloses Devnet-SOL
über den [Solana Faucet](https://faucet.solana.com/) oder mit der Solana CLI
anfordern:

```terminal
$ solana airdrop 2 --url devnet
```

```terminal
$ anchor build
$ anchor deploy --provider.cluster devnet
```

</Step>

</Steps>

## Quelldateien

Das Verzeichnis `src` enthält das Solana-Programm. Anchors
[Programmstruktur](https://www.anchor-lang.com/docs/basics/program-structure)
Dokumentation erklärt die hier verwendeten Kern-Makros, einschließlich
_rs`declare_id!`_, _rs`#[program]`_, _rs`#[derive(Accounts)]`_ und
_rs`#[account]`_. Dieser Abschnitt führt durch die Template-Dateien.

### lib.rs

`lib.rs` ist der Einstiegspunkt des Programms. Es verbindet die Quelldateien,
definiert die Programmadresse und definiert die Programm- Anweisungen, die
Benutzer aufrufen können.

<WithNotes>

```rust title="programs/my-program/src/lib.rs" -w
// !tooltip[/pub mod constants/] constantsMod
pub mod constants;
// !tooltip[/pub mod error/] errorMod
pub mod error;
// !tooltip[/pub mod instructions/] instructionsMod
pub mod instructions;
// !tooltip[/pub mod state/] stateMod
pub mod state;

// !tooltip[/anchor_lang::prelude/] prelude
use anchor_lang::prelude::*;

pub use constants::*;
pub use instructions::*;
pub use state::*;

// !tooltip[/declare_id!/] declareId
declare_id!("82sFkffP9wxwpyfZyeaKHH2chQoJPUGsJZSPi9mrUuXd");

// !tooltip[/#\[program\]/] programMacro
#[program]
pub mod my_program {
    use super::*;

    // !tooltip[/pub fn initialize/] publicInstructions
    pub fn initialize(ctx: Context<Initialize>) -> Result<()> {
        // !tooltip[/handle_initialize/] handlers
        crate::instructions::initialize::handle_initialize(ctx)
    }

    // !tooltip[/pub fn increment/] publicInstructions
    pub fn increment(ctx: Context<Increment>) -> Result<()> {
        // !tooltip[/handle_increment/] handlers
        crate::instructions::increment::handle_increment(ctx)
    }
}
```

### !constantsMod

_rs`pub mod constants`_ macht `constants.rs` zu einem Teil der Programm-Crate.
Hier speichert die Vorlage gemeinsam genutzte Programmwerte wie den PDA-seed,
den Überweisungsbetrag und den maximalen Zählerwert.

### !errorMod

_rs`pub mod error`_ macht `error.rs` zu einem Teil der Programm-Crate. Diese
Datei definiert die benutzerdefinierten Fehler, die das Programm zurückgeben
kann, wenn eine Anweisung ungültig ist.

### !instructionsMod

_rs`pub mod instructions`_ macht `instructions.rs` zu einem Teil der
Programm-Crate. Diese Datei fungiert als Index für die anweisungsspezifischen
Dateien in `src/instructions/`.

### !stateMod

_rs`pub mod state`_ macht `state.rs` zu einem Teil der Programm-Crate. Diese
Datei definiert das Datenlayout für Konten, die diesem Programm gehören,
einschließlich des _rs`Counter`_-Konten.

### !prelude

_rs`anchor_lang::prelude::*`_ importiert die Anchor-Typen und Makros, die zur
Definition eines Solana-Programms verwendet werden, darunter _rs`Context`_,
_rs`Result`_, _rs`Pubkey`_, _rs`Account`_, _rs`Signer`_ sowie
Anchor-Konten-Attribute.

### !reexports

Die _rs`pub use`_-Zeilen machen öffentliche Elemente aus den untergeordneten
Modulen vom Crate-Stammverzeichnis aus zugänglich. Dadurch bleiben die
Anweisungs-Einstiegspunkte kurz, z. B. _rs`Context<Initialize>`_ statt eines
längeren Modulpfads.

### !declareId

_rs`declare_id!`_ definiert die Adresse des Programms. Sie sollte mit der
Adresse in `Anchor.toml`, dem bereitgestellten program account und allen Nutzern
oder Tests, die das Programm aufrufen, übereinstimmen.

### !programMacro

_rs`#[program]`_ weist Anchor an, den Solana-Programm-Einstiegspunkt und den
Anweisungs-Dispatch-Code zu generieren. Öffentliche Funktionen in diesem Modul
werden zu Anweisungen, die Nutzer aufrufen können.

### !publicInstructions

_rs`initialize`_ und _rs`increment`_ sind die beiden Anweisungs-Namen, die
Nutzer aufrufen können. Jede erhält einen _rs`Context<T>`_. Der Typ innerhalb
von _rs`Context`_ definiert die für diese Anweisung erforderlichen Konten, und
Anchor validiert diese Konten, bevor die Anweisung ausgeführt wird.

### !handlers

Die Handler-Funktion enthält die Logik für die Anweisungen.

</WithNotes>

Programm für Sicherheitsprüfungen.

### constants.rs

`constants.rs` hält gemeinsam genutzte Werte an einem Ort. In dieser Vorlage
leitet _rs`COUNTER_SEED`_ die Counter-PDA ab, _rs`HELLO_WORLD_LAMPORTS`_ wird
bei der Initialisierung übertragen, und _rs`MAX_COUNT`_ wird vor dem
Inkrementieren geprüft.

<WithNotes>

```rust title="programs/my-program/src/constants.rs" -w
use anchor_lang::prelude::*;

// !tooltip[/#\[constant\]/] anchorConstant
// !tooltip[/COUNTER_SEED/] counterSeed
#[constant]
pub const COUNTER_SEED: &[u8] = b"counter";

// !tooltip[/HELLO_WORLD_LAMPORTS/] helloLamports
#[constant]
pub const HELLO_WORLD_LAMPORTS: u64 = 1;

// !tooltip[/MAX_COUNT/] maxCount
#[constant]
pub const MAX_COUNT: u64 = 10;
```

### !anchorConstant

_rs`#[constant]`_ weist Anchor an, den Wert in die Programm-IDL aufzunehmen.
Dadurch werden wichtige Programmkonstanten für generierte Clients und Tooling
sichtbar.

### !counterSeed

_rs`COUNTER_SEED`_ ist der Byte-String, der zur Ableitung der Counter-PDA
verwendet wird. Das Programm und der Test verwenden beide diesen seed, sodass
sie sich ohne Speicherung eines separaten keypair auf die Kontenadresse einigen.
Diese Vorlage verwendet einen festen seed, sodass das Programm eine
Counter-Adresse hat. Echte Programme enthalten oft die Adresse eines Benutzers
oder einen anderen Bezeichner in den seeds, um viele verwandte Konten zu
erstellen.

### !helloLamports

_rs`HELLO_WORLD_LAMPORTS`_ ist der Betrag an lamports, der vom Zahler an das
Counter-Konto innerhalb von _rs`initialize`_ übertragen wird. Er ist bewusst
klein gehalten, da die Vorlage einen CPI an das System Program demonstriert.

### !maxCount

_rs`MAX_COUNT`_ ist die obere Grenze, die von _rs`increment`_ geprüft wird. Die
Speicherung hier gibt dem Programm, den Tests und der generierten IDL einen
gemeinsamen Namen für die Regel.

</WithNotes>

<SideBySide>

### !left

```rust title="initialize.rs" -w
#[derive(Accounts)]
pub struct Initialize<'info> {
    // ...
    #[account(
        init,
        payer = payer,
        space = 8 + Counter::INIT_SPACE,
        // !mark
        // !mention counter-seed
        seeds = [COUNTER_SEED],
        bump
    )]
    pub counter: Account<'info, Counter>,
    // ...
}

pub fn handle_initialize(ctx: Context<Initialize>) -> Result<()> {
    // ...

    // !mark
    // !mention hello-world-lamports
    anchor_lang::system_program::transfer(cpi_ctx, HELLO_WORLD_LAMPORTS)?;

    Ok(())
}
```

### !right

`initialize.rs` verwendet zwei Konstanten:

- [_rs`COUNTER_SEED`_](mention:counter-seed) leitet die PDA-Adresse des Zählers
  ab.
- [_rs`HELLO_WORLD_LAMPORTS`_](mention:hello-world-lamports) legt den Betrag
  fest, der vom Zahler auf das Zählerkonto übertragen wird.

</SideBySide>

<SideBySide>

### !left

```rust title="increment.rs" -w
pub fn handle_increment(ctx: Context<Increment>) -> Result<()> {
    // ...

    require!(
        // !mark
        // !mention max-count
        ctx.accounts.counter.count < MAX_COUNT,
        ErrorCode::CounterOverflow,
    );

    ctx.accounts.counter.count += 1;
    Ok(())
}
```

### !right

`increment.rs` verwendet [_rs`MAX_COUNT`_](mention:max-count) als obere Grenze
des Zählers. Wenn der aktuelle Zählerstand bereits das Maximum erreicht hat,
gibt _rs`require!`_ den Wert _rs`CounterOverflow`_ zurück und die Kontodaten
werden nicht geändert.

</SideBySide>

### state.rs

`state.rs` definiert benutzerdefinierte Datentypen für Konten, die das Programm
erstellt und verwaltet. Das Programm definiert Anweisungen zum Erstellen,
Initialisieren und Aktualisieren dieser Daten, die Zählerdaten werden jedoch
nicht im Programm selbst gespeichert. Sie werden in einem separaten Konto mit
eigener Adresse gespeichert.

<WithNotes>

```rust title="programs/my-program/src/state.rs" -w
use anchor_lang::prelude::*;

// !tooltip[/#\[account\]/] accountMacro
#[account]
// !tooltip[/InitSpace/] initSpace
#[derive(InitSpace)]
pub struct Counter {
    // !tooltip[/count/] count
    pub count: u64,
    // !tooltip[/authority/] authority
    pub authority: Pubkey,
}
```

### !accountMacro

_rs`#[account]`_ teilt Anchor mit, dass diese Struct Daten sind, die in einem
Onchain-Konto gespeichert werden können, das diesem Programm gehört. Anchor
verwendet sie zum Serialisieren, Deserialisieren und Validieren von
_rs`Counter`_ Konten.

### !initSpace

_rs`InitSpace`_ ermöglicht es Anchor, _rs`Counter::INIT_SPACE`_ zu berechnen –
die Anzahl der Bytes, die für die Felder dieser Struct benötigt werden. Der
8-Byte-Anchor-Konto-Diskriminator wird beim Erstellen des Kontos separat
hinzugefügt.

### !count

_rs`count`_ ist der Zählerwert, der in den Kontodaten gespeichert ist.

### !authority

_rs`authority`_ speichert die Adresse, die berechtigt ist, den Zähler zu
erhöhen. _rs`initialize`_ schreibt diese Adresse, und _rs`increment`_ erfordert,
dass dieselbe Adresse signiert.

</WithNotes>

<SideBySide>

### !left

```rust title="initialize.rs" -w
#[account(
    init,
    payer = payer,
    // !mark
    // !mention counter-space
    space = 8 + Counter::INIT_SPACE,
    seeds = [COUNTER_SEED],
    bump
)]
pub counter: Account<'info, Counter>,
// ...

pub fn handle_initialize(ctx: Context<Initialize>) -> Result<()> {
    // !mark(1:2)
    // !mention(1:2) initialize-fields
    ctx.accounts.counter.count = 0;
    ctx.accounts.counter.authority = ctx.accounts.payer.key();

    Ok(())
}
```

```rust title="increment.rs" -w
pub fn handle_increment(ctx: Context<Increment>) -> Result<()> {
    // ...

    // !mark
    // !mention increment-count
    ctx.accounts.counter.count += 1;
    Ok(())
}
```

### !right

`state.rs` definiert die _rs`Counter`_ Kontendaten. Die Anweisungsdateien
verwenden diesen Typ beim Erstellen und Aktualisieren der Konten:

- [_rs`Counter::INIT_SPACE`_](mention:counter-space) legt die Größe der Konten
  für die in `state.rs` definierten Felder fest.
- [_rs`count`_ und _rs`authority`_](mention:initialize-fields) sind die
  Feldwerte, die beim Initialisieren der Konten geschrieben werden.
- [_rs`count += 1`_](mention:increment-count) aktualisiert den gespeicherten
  Zählerwert, nachdem die Validatoren erfolgreich abgeschlossen wurden.

</SideBySide>

### error.rs

`error.rs` definiert die benutzerdefinierten Fehler des Programms. In dieser
Vorlage zeigen die Fehler, wie Anweisungs-Handler abbrechen, wenn ein Aufrufer
nicht berechtigt ist, den Zähler zu aktualisieren, oder der Zähler bereits
_rs`MAX_COUNT`_ erreicht hat.

<WithNotes>

```rust title="programs/my-program/src/error.rs" -w
use anchor_lang::prelude::*;

// !tooltip[/#\[error_code\]/] errorCode
#[error_code]
pub enum ErrorCode {
    // !tooltip[/Unauthorized/] unauthorized
    #[msg("Only the counter authority can update this counter")]
    Unauthorized,
    // !tooltip[/CounterOverflow/] counterOverflow
    #[msg("Counter has reached the maximum value")]
    CounterOverflow,
}
```

### !errorCode

_rs`#[error_code]`_ weist Anchor an, diese Varianten in benutzerdefinierte
Programmfehler umzuwandeln. Clients und Tests können diese Fehler verwenden,
anstatt aus rohen Fehlercodes zu raten.

### !unauthorized

_rs`Unauthorized`_ wird zurückgegeben, wenn der Signer der Transaktion nicht die
im Zähler-Konten gespeicherte Autorität ist.

### !counterOverflow

_rs`CounterOverflow`_ wird zurückgegeben, wenn das Inkrementieren gegen die
_rs`MAX_COUNT`_-Regel des Programms verstoßen würde.

</WithNotes>

<SideBySide>

### !left

```rust title="increment.rs" -w
pub fn handle_increment(ctx: Context<Increment>) -> Result<()> {
    require_keys_eq!(
        ctx.accounts.counter.authority,
        ctx.accounts.authority.key(),
        // !mark
        // !mention unauthorized-error
        ErrorCode::Unauthorized,
    );
    require!(
        ctx.accounts.counter.count < MAX_COUNT,
        // !mark
        // !mention overflow-error
        ErrorCode::CounterOverflow,
    );

    ctx.accounts.counter.count += 1;
    msg!("Hello, world! Counter is now {}", ctx.accounts.counter.count);
    Ok(())
}
```

### !right

`error.rs` benennt die Fehler, die die Anweisung zurückgeben kann:

- [_rs`ErrorCode::Unauthorized`_](mention:unauthorized-error) wird
  zurückgegeben, wenn der Signer nicht die im Zähler-Konten gespeicherte
  Autorität ist.
- [_rs`ErrorCode::CounterOverflow`_](mention:overflow-error) wird zurückgegeben,
  wenn der Zähler bereits _rs`MAX_COUNT`_ erreicht hat.

</SideBySide>

### Anweisungen.rs

`instructions.rs` verbindet die Anweisungen-Dateien mit der Programm-Crate,
damit `lib.rs` auf den _rs`initialize`_- und _rs`increment`_-Anweisungs-Code
zugreifen kann. Jede Anweisungs-Datei definiert die Konten, die von dieser
Anweisung benötigt werden, sowie die Handler-Logik, die ausgeführt wird, nachdem
Anchor diese Konten validiert hat.

<WithNotes>

```rust title="programs/my-program/src/instructions.rs" -w
// !tooltip[/pub mod initialize/] moduleDeclarations
pub mod initialize;
pub mod increment;

// !tooltip[/pub use initialize/] reexports
pub use initialize::*;
pub use increment::*;
```

### !moduleDeclarations

Die _rs`pub mod`_-Zeilen machen `src/instructions/initialize.rs` und
`src/instructions/increment.rs` zu einem Teil der Programm-Crate.

### !reexports

Die _rs`pub use`_-Zeilen re-exportieren öffentliche Elemente aus diesen Dateien.
Dadurch werden die Anweisungen-Konto-Strukturen und Handler-Funktionen über das
_rs`instructions`_-Modul verfügbar gemacht.

</WithNotes>

### initialize.rs

`initialize.rs` definiert die Konten, die zum Erstellen des Zähler-Kontos
erforderlich sind, und schreibt dann die ersten Werte des Kontos. Die
_rs`#[derive(Accounts)]`_-Struktur verwendet Anchor
[account constraints](https://www.anchor-lang.com/docs/references/account-constraints),
um anzugeben, welche Konten erforderlich sind und wie das neue Zähler-Konto
erstellt wird.

<WithNotes>

```rust title="programs/my-program/src/instructions/initialize.rs" -w
use anchor_lang::prelude::*;

use crate::{constants::*, state::Counter};

// !tooltip[/derive\(Accounts\)/] accountsStruct
#[derive(Accounts)]
pub struct Initialize<'info> {
    // !tooltip[/#\[account\(mut\)\]/] payerMutable
    #[account(mut)]
    // !tooltip[/Signer<'info>/] payerSigner
    pub payer: Signer<'info>,
    // !tooltip[/#\[account\(/] counterConstraints
    #[account(
        // !tooltip[/init/] initConstraint
        init,
        // !tooltip[/payer = payer/] payerConstraint
        payer = payer,
        // !tooltip[/space = 8 \+ Counter::INIT_SPACE/] space
        space = 8 + Counter::INIT_SPACE,
        // !tooltip[/seeds = \[COUNTER_SEED\]/] seeds
        seeds = [COUNTER_SEED],
        // !tooltip[/bump/] bump
        bump
    )]
    // !tooltip[/Account<'info, Counter>/] counterAccountType
    pub counter: Account<'info, Counter>,
    // !tooltip[/Program<'info, System>/] systemProgram
    pub system_program: Program<'info, System>,
}

// !tooltip[/handle_initialize/] handler
pub fn handle_initialize(ctx: Context<Initialize>) -> Result<()> {
    // !tooltip[/counter.count = 0/] initialCount
    ctx.accounts.counter.count = 0;
    // !tooltip[/counter.authority = ctx.accounts.payer.key\(\)/] initialAuthority
    ctx.accounts.counter.authority = ctx.accounts.payer.key();

    // !tooltip[/system_program::Transfer/] cpiAccounts
    let cpi_accounts = anchor_lang::system_program::Transfer {
        from: ctx.accounts.payer.to_account_info(),
        to: ctx.accounts.counter.to_account_info(),
    };
    // !tooltip[/CpiContext::new/] cpiContext
    let cpi_ctx = CpiContext::new(anchor_lang::system_program::ID, cpi_accounts);
    // !tooltip[/system_program::transfer/] transfer
    anchor_lang::system_program::transfer(cpi_ctx, HELLO_WORLD_LAMPORTS)?;

    // !tooltip[/msg!/] programLog
    msg!("Hello, world! Counter initialized");
    Ok(())
}
```

## !!steps Account Context

Die _rs`Initialize`_-Struktur definiert die Konten, die angegeben werden müssen,
wenn ein Benutzer die _rs`initialize`_- Anweisung aufruft. Anchor prüft diese
Konten, bevor der Handler ausgeführt wird.

<CodePlaceholder title="initialize.rs" />

```rust !! title="initialize.rs"

        seeds = [COUNTER_SEED],
        bump

    pub counter: Account<'info, Counter>,

}
```

_rs`Signer<'info>`_ bedeutet, dass der Zahler die Transaktion unterzeichnen
muss, und _rs`#[account(mut)]`_ bedeutet, dass das Zahler-Konto geändert werden
kann, da lamports abgezogen werden.

## !!steps Counter Account

Das _rs`counter`_-Konto speichert die _rs`Counter`_-Daten aus `state.rs`.
_rs`init`_ weist Anchor an, dieses Konto vor der Ausführung des Handlers zu
erstellen, und _rs`payer = payer`_ teilt Anchor mit, welches Konto die
Erstellung bezahlt.

<CodePlaceholder title="initialize.rs" />

```rust !! title="initialize.rs"
#[derive(Accounts)]
pub struct Initialize<'info> {
    #[account(mut)]
    pub payer: Signer<'info>,
    // !focus(1:8)
    #[account(
        init,
```

### !imports

Diese Datei importiert die gemeinsamen Konstanten und den
_rs`Counter`_-Kontotyp. Die Anweisungen verwenden diese Definitionen, um die PDA
abzuleiten, den Transferbetrag festzulegen und die Konten mit der richtigen
Datengröße zu reservieren.

### !accountsStruct

_rs`Initialize`_ ist der Kontokontext für die _rs`initialize`_- Anweisungen.
Jedes Feld in der Struktur repräsentiert ein separates Konto, das beim Aufruf
der Anweisungen angegeben werden muss. Anchor verwendet die Struktur, um diese
eingehenden Konten zu validieren, bevor der Handler ausgeführt wird.

### !payerMutable

_rs`#[account(mut)]`_ teilt Anchor mit, dass das Zahler-Konto beschreibbar sein
muss. Dies ist erforderlich, da sowohl die Kontoerstellung als auch der
Transfer-CPI lamport vom Zahler abziehen.

### !payerSigner

_rs`Signer<'info>`_ teilt Anchor mit, dass der Zahler die Transaktion signieren
muss.

### !counterConstraints

Der _rs`#[account(...)]`_-Block teilt Anchor mit, wie das Zähler-Konto erstellt
und validiert werden soll, bevor der Handler ausgeführt wird.

### !initConstraint

_rs`init`_ weist Anchor an, das Konten _rs`counter`_ zu erstellen, bevor der
Handler ausgeführt wird. Das neue Konten gehört diesem Programm, sodass dieses
Programm _rs`Counter`_-Daten darin speichern kann.

### !payerConstraint

_rs`payer = payer`_ legt das Konten _rs`payer`_ als Zahler für die Erstellung
des neuen Zähler-Konten fest.

### !space

_rs`space = 8 + Counter::INIT_SPACE`_ reserviert Bytes für Anchors
Konten-Diskriminator sowie die _rs`Counter`_-Felder aus `state.rs`. Der
Diskriminator hilft Anchor, zu bestätigen, dass das Konten den erwarteten Typ
speichert, bevor es gelesen wird.

### !seeds

_rs`seeds = [COUNTER_SEED]`_ gibt an, dass die Adresse des Zähler-Konten die PDA
sein muss, die aus diesem seed und der Programm-ID abgeleitet wird. Mit einem
festen seed gibt es eine Zähler-PDA für dieses Programm. Um separate Zähler in
einem echten Programm zu erstellen, würde man in der Regel einen weiteren seed
hinzufügen, z. B. die Adresse eines Benutzers.

### !bump

_rs`bump`_ weist Anchor an, den PDA-Bump zu finden und zu validieren, also das
zusätzliche Byte, das die abgeleitete Adresse als gültige PDA kennzeichnet.

### !counterAccountType

_rs`Account<'info, Counter>`_ weist Anchor an, die Kontendaten als den
_rs`Counter`_-Typ aus `state.rs` zu behandeln. Anchor prüft außerdem, ob das
Konten diesem Programm gehört.

### !systemProgram

_rs`Program<'info, System>`_ erfordert, dass der Aufrufer das System Program
bereitstellt. Anchor verwendet es für die Konten-Erstellungsbedingung
_rs`init`_. Der Transfer-CPI des Handlers ruft ebenfalls das System Program auf.

### !handler

Der Handler wird ausgeführt, nachdem Anchor die Konten validiert und das
Zähler-Konten erstellt hat, sodass er die initialen Daten des Konten sicher
schreiben kann.

### !initialCount

Setzt das Feld _rs`count`_ im Zähler-Konten auf _rs`0`_.

### !initialAuthority

Setzt das Feld _rs`authority`_ im Zähler-Konten auf die Adresse des Zahlers.

### !cpiAccounts

Dies sind die Konten, die an die Transfer-Anweisung des System Program übergeben
werden. Dieses Transfer-CPI ist nur enthalten, um zu zeigen, wie ein CPI Konten
an ein anderes Programm weitergibt.

### !cpiContext

_rs`CpiContext`_ bündelt das Zielprogramm und die erforderlichen Konten für eine
Cross Program Invocation. Das Erstellen des Kontexts bereitet das CPI vor, indem
das aufzurufende Programm mit den von ihm erwarteten Konten verknüpft wird.

### !transfer

Dieses CPI ruft die Transfer-Anweisung des System Program auf und überträgt
_rs`HELLO_WORLD_LAMPORTS`_ vom Zahler auf das Zähler-Konto.

### !programLog

_rs`msg!`_ schreibt eine Nachricht in die Programm-Logs, wenn das Programm
aufgerufen wird.

_rs`Increment`_ ist der Kontenkontext für die Anweisung _rs`increment`_. Anchor
verwendet ihn, um das veränderliche Zählerkonto und den erforderlichen Signer zu
validieren.

### !counterMutable

_rs`mut`_ teilt Anchor mit, dass das Zählerkonto beschreibbar sein muss, da der
Handler den gespeicherten _rs`count`_-Wert aktualisiert.

### !counterSeeds

_rs`seeds = [COUNTER_SEED]`_ überprüft, dass das angegebene Zählerkonto das PDA
ist, das aus diesem seed und der Programm-ID abgeleitet wurde.

### !counterBump

_rs`bump`_ teilt Anchor mit, den PDA-Bump für das Zählerkonto zu finden und zu
validieren.

### !incrementCounterAccountType

_rs`Account<'info, Counter>`_ teilt Anchor mit, die Kontodaten als den Typ
_rs`Counter`_ aus `state.rs` zu deserialisieren. Anchor überprüft außerdem, dass
das Konten diesem Programm gehört.

### !authoritySigner

_rs`Signer`_ erfordert die Signatur der Autorität. Der Handler vergleicht dann
die Adresse dieses Signer mit der im Zählerkonto gespeicherten Autorität.

### !incrementHandler

Der Handler wird ausgeführt, nachdem Anchor die Kontenliste validiert, die
Eigentümerschaft geprüft und das Zählerkonto deserialisiert hat.

### !authorityCheck

_rs`require_keys_eq!`_ vergleicht zwei öffentliche Schlüssel und stoppt die
Anweisung, sofern die Adresse des Signer nicht mit der im Zählerkonto
gespeicherten Autorität übereinstimmt. Dies ist die anwendungsseitige
Autorisierungsregel der Vorlage.

### !maxCheck

_rs`require!`_ überprüft eine boolesche Bedingung und stoppt die Anweisung mit
_rs`CounterOverflow`_, wenn der Zähler bereits _rs`MAX_COUNT`_ erreicht hat.
Diese Vorlagenregel demonstriert benutzerdefinierte Fehler und das Abbrechen vor
einer Mutation.

### !mutateCount

Erst nachdem beide Prüfungen bestanden wurden, aktualisiert der Handler die
Zählerkontodaten. Schlägt eine der Prüfungen fehl, schlägt die Transaktion fehl
und das Konten wird nicht geändert.

### !programLog

_rs`msg!`_ schreibt eine Nachricht in die Programmlogs, wenn das Programm
aufgerufen wird.

</WithNotes>

## Testdatei

`programs/my-program/tests/test_initialize.rs` ist ein Rust-Integrationstest. Es
wird kein lokaler validator gestartet. Stattdessen wird die kompilierte
`.so`-Datei in LiteSVM geladen, Transaktionen erstellt, die das Programm
aufrufen, und das Zählerkonto nach jeder Transaktion ausgelesen. Der Test
erstellt Anweisungen für eine Solana-Transaktion, indem die Programm-ID des
aufzurufenden Programms angegeben, instruction data bereitgestellt und die

```rust title="programs/my-program/tests/test_initialize.rs" -w
use {
    anchor_lang::{
        prelude::Pubkey,
        solana_program::{instruction::Instruction, system_program},
        AccountDeserialize, InstructionData, ToAccountMetas,
    },
    // !tooltip[/litesvm::LiteSVM/] litesvm
    litesvm::LiteSVM,
    solana_keypair::Keypair,
    solana_message::{Message, VersionedMessage},
    solana_signer::Signer,
    solana_transaction::versioned::VersionedTransaction,

#[test]
fn test_initialize() {

    let program_id = my_program::id();
    let payer = Keypair::new();
    // !tooltip[/Pubkey::find_program_address/] counterPda
    let counter = Pubkey::find_program_address(
        &[my_program::constants::COUNTER_SEED],
        &program_id,
    )
    .0;

    let bytes = include_bytes!(concat!(
        env!("CARGO_TARGET_TMPDIR"),
        "/../deploy/my_program.so"

    svm.add_program(program_id, bytes).unwrap();
    svm.airdrop(&payer.pubkey(), 1_000_000_000).unwrap();

    let instruction = Instruction::new_with_bytes(
        program_id,

    let tx = VersionedTransaction::try_new(VersionedMessage::Legacy(msg), &[&payer]).unwrap();

    let res = svm.send_transaction(tx);

    let counter_account = svm.get_account(&counter).unwrap();

    // !tooltip[/assert_eq!\(counter_state.count, 0\)/] initAssert
```

### !litesvm

_rs`LiteSVM`_ ist eine In-Process-Solana-VM für Tests. Sie ermöglicht es dem
Test, das kompilierte Programm zu registrieren, Konten zu erstellen,
Transaktionen zu senden und den Kontostatus zu überprüfen, ohne einen
validator-Prozess zu starten.

### !setup

Der Test ruft die Programmadresse aus dem _rs`declare_id!`_ Wert über
_rs`my_program::id()`_ ab und erstellt ein neues keypair für die
Testtransaktion.

### !counterPda

Die Zähleradresse wird aus demselben _rs`COUNTER_SEED`_ abgeleitet, der vom
Programm verwendet wird. Dadurch zielt der Test auf die PDA ab, die
`initialize.rs` erwartet.

### !programBytes

_rs`include_bytes!`_ liest die kompilierte `my_program.so`-Datei, die beim Build
erzeugt wurde. Diese Bytes sind der ausführbare Programmcode, den LiteSVM im
Test ausführt.

### !addProgram

_rs`svm.add_program`_ registriert die kompilierten Programm-Bytes unter der
Programm- adresse in LiteSVM. Der Airdrop gibt dem Zahler genug lokales SOL, um
für die Transaktion und die Konten-Erstellung zu bezahlen.

### !testInstruction

_rs`Instruction::new_with_bytes`_ erstellt eine Anweisung zum Aufrufen des
Programms. Sie enthält die Programm-ID, instruction data und die
Anweisungs-Konten.

### !blockhash

Jede Solana-Transaktion benötigt einen aktuellen Blockhash. In diesem Test
stellt LiteSVM einen Blockhash für die Transaktion bereit.

### !message

Die Nachricht enthält die Anweisung, den Gebührenzahler und den Blockhash. Dies
sind die Transaktionsdaten, die signiert werden.

### !transaction

_rs`VersionedTransaction::try_new`_ signiert die Nachricht mit dem keypair des
Zahlers. Der Zahler signiert, weil er die Transaktionsgebühr bezahlt und das
neue Zähler- Konten finanziert.

### !sendTx

_rs`svm.send_transaction`_ führt die Transaktion in LiteSVM aus.

### !getCounter

Nachdem die Transaktion ausgeführt wurde, liest der Test das Zähler-Konten
direkt aus LiteSVM aus, um die Konten-Daten zu überprüfen.

### !deserialize

Das Zähler-Konten speichert serialisierte Daten. _rs`Counter::try_deserialize`_
prüft Anchor's Konten-Diskriminator und wandelt die Konten-Daten zurück in den
_rs`Counter`_-Typ aus `state.rs` um.

### !initAssert

Nachdem _rs`initialize`_ erfolgreich war, deserialisiert der Test das
Zähler-Konten und prüft, ob der Ausgangszustand mit dem übereinstimmt, was
_rs`handle_initialize`_ geschrieben hat.

### !incrementAssert

Nachdem _rs`increment`_ erfolgreich war, sollte dasselbe Zähler-Konten
_rs`count`_ auf _rs`1`_ gesetzt haben, während dieselbe Autorität beibehalten
wird.

</WithNotes>

### !skipValidator

`skip_local_validator = true` weist Anchor an, für `anchor test` keinen
validator zu starten. Der Rust-Test verwendet stattdessen LiteSVM.

### !programs

`[programs.localnet]` ordnet den Namen der Rust-Crate der Programmadresse zu,
die für lokale Builds und Tests verwendet wird.

### !provider

`[provider]` legt den Standard-Cluster und die Wallet für Befehle wie
`anchor deploy` fest.

### !scripts

`[scripts]` ändert, was `anchor test` ausführt. In dieser Vorlage führt
`anchor test` `cargo test` aus.

</WithNotes>
