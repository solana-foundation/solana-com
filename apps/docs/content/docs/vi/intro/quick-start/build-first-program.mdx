Quickstart này sử dụng dự án khởi đầu của
[Anchor framework](https://www.anchor-lang.com/docs) được tạo bởi `anchor init`.
Bạn sẽ tạo dự án trên máy cục bộ, chạy các bài kiểm tra, build chương trình và
xem qua mã nguồn chương trình mà Anchor tạo ra.

## Điều Kiện Tiên Quyết

Trước khi bắt đầu,
[hãy cài đặt các công cụ phát triển Solana](/docs/intro/installation). Bộ cài
đặt bao gồm Rust, Solana CLI và Anchor CLI.

Sử dụng Anchor CLI phiên bản 1.1.2 trở lên cho template này. Kiểm tra phiên bản
đã cài đặt của bạn:

```terminal
$ anchor --version
```

<Steps>

<Step>

### Tạo Dự Án

Chạy các lệnh sau trong terminal của bạn:

```terminal
$ anchor init my-program
$ cd my-program
```

Dự án khởi đầu bao gồm một chương trình Solana nằm trong `programs/my-program`.
Chương trình này có hai instruction: một để khởi tạo tài khoản counter và một để
tăng giá trị counter.

Một số phần của template minh họa các mẫu chương trình Solana phổ biến: dẫn xuất
địa chỉ tài khoản [PDA](/docs/core/pda), thực hiện Cross Program Invocation
([CPI](/docs/core/cpi)) để chuyển SOL, và sử dụng kiểm tra lỗi tùy chỉnh để dừng
một instruction khi một điều kiện không được thỏa mãn.

<Step>

### Build Chương Trình

Chạy `anchor build` để biên dịch chương trình khởi đầu:

```terminal
$ anchor build
```

Chương trình đã biên dịch được ghi vào `target/deploy/my_program.so`. Khi chương
trình được triển khai, nội dung của tệp `.so` này sẽ được lưu trữ trong một tài
khoản trên chuỗi.

</Step>

<Step>

### Chạy Kiểm Thử

Chạy kiểm thử mặc định:

```terminal
$ anchor test
```

`Anchor.toml` của template này sử dụng lệnh kiểm thử Rust:

```toml title="Anchor.toml"
skip_local_validator = true

[scripts]
test = "cargo test"
```

Kiểm thử tải chương trình đã biên dịch vào [LiteSVM](https://www.litesvm.com/),
tạo một payer, gửi các lệnh _rs`initialize`_ và _rs`increment`_, sau đó kiểm tra
trạng thái của tài khoản counter.

Chạy `anchor test` cũng sẽ biên dịch chương trình, vì vậy bạn không cần chạy
`anchor build` trước khi kiểm thử cục bộ.

</Step>

<Step>

### Triển Khai Chương Trình

Kiểm thử cục bộ là vòng phản hồi nhanh nhất. Khi bạn sẵn sàng triển khai lên một
mạng, ví dụ như devnet, hãy build trước, sau đó triển khai lên một cluster.

Việc triển khai một chương trình Solana yêu cầu SOL vì chương trình được lưu trữ
trong một account, và account đó phải trả phí cho không gian lưu trữ. Trên
devnet, hãy yêu cầu devnet SOL miễn phí từ
[Solana Faucet](https://faucet.solana.com/) hoặc qua Solana CLI:

```terminal
$ solana airdrop 2 --url devnet
```

```terminal
$ anchor build
$ anchor deploy --provider.cluster devnet
```

</Step>

</Steps>

## Các tệp nguồn

Thư mục `src` chứa chương trình Solana. Tài liệu
[Cấu Trúc Chương Trình](https://www.anchor-lang.com/docs/basics/program-structure)
của Anchor giải thích các macro cốt lõi được sử dụng ở đây, bao gồm
_rs`declare_id!`_, _rs`#[program]`_, _rs`#[derive(Accounts)]`_, và
_rs`#[account]`_. Phần này hướng dẫn qua các tệp template.

### lib.rs

`lib.rs` là điểm vào của chương trình. Nó kết nối các tệp nguồn, định nghĩa địa
chỉ chương trình, và định nghĩa các lệnh của chương trình mà người dùng có thể
gọi.

<WithNotes>

```rust title="programs/my-program/src/lib.rs" -w
// !tooltip[/pub mod constants/] constantsMod
pub mod constants;
// !tooltip[/pub mod error/] errorMod
pub mod error;
// !tooltip[/pub mod instructions/] instructionsMod
pub mod instructions;
// !tooltip[/pub mod state/] stateMod
pub mod state;

// !tooltip[/anchor_lang::prelude/] prelude
use anchor_lang::prelude::*;

pub use constants::*;
pub use instructions::*;
pub use state::*;

// !tooltip[/declare_id!/] declareId
declare_id!("82sFkffP9wxwpyfZyeaKHH2chQoJPUGsJZSPi9mrUuXd");

// !tooltip[/#\[program\]/] programMacro
#[program]
pub mod my_program {
    use super::*;

    // !tooltip[/pub fn initialize/] publicInstructions
    pub fn initialize(ctx: Context<Initialize>) -> Result<()> {
        // !tooltip[/handle_initialize/] handlers
        crate::instructions::initialize::handle_initialize(ctx)
    }

    // !tooltip[/pub fn increment/] publicInstructions
    pub fn increment(ctx: Context<Increment>) -> Result<()> {
        // !tooltip[/handle_increment/] handlers
        crate::instructions::increment::handle_increment(ctx)
    }
}
```

### !constantsMod

_rs`pub mod constants`_ đưa `constants.rs` vào crate chương trình. Đây là nơi
template lưu trữ các giá trị chung của chương trình như seed PDA, số tiền chuyển
và giá trị bộ đếm tối đa.

### !errorMod

_rs`pub mod error`_ đưa `error.rs` vào crate chương trình. File này định nghĩa
các lỗi tùy chỉnh mà chương trình có thể trả về khi một lệnh không hợp lệ.

### !instructionsMod

_rs`pub mod instructions`_ đưa `instructions.rs` vào crate chương trình. File đó
hoạt động như một chỉ mục cho các file theo từng lệnh trong `src/instructions/`.

### !stateMod

_rs`pub mod state`_ đưa `state.rs` vào crate chương trình. File này định nghĩa
cấu trúc dữ liệu cho các tài khoản thuộc sở hữu của chương trình này, bao gồm
tài khoản _rs`Counter`_.

### !prelude

_rs`anchor_lang::prelude::*`_ nhập các kiểu và macro của Anchor dùng để định
nghĩa một chương trình Solana, bao gồm _rs`Context`_, _rs`Result`_,
_rs`Pubkey`_, _rs`Account`_, _rs`Signer`_ và các thuộc tính tài khoản Anchor.

### !reexports

Các dòng _rs`pub use`_ công khai các mục từ các module con tại gốc crate. Điều
này giúp các điểm vào lệnh ngắn gọn hơn, chẳng hạn _rs`Context<Initialize>`_
thay vì một đường dẫn module dài hơn.

### !declareId

_rs`declare_id!`_ định nghĩa địa chỉ của chương trình. Nó phải khớp với địa chỉ
trong `Anchor.toml`, program account đã triển khai và bất kỳ người dùng hoặc bài
kiểm tra nào gọi chương trình.

### !programMacro

_rs`#[program]`_ yêu cầu Anchor tạo điểm vào chương trình Solana và mã điều phối
lệnh. Các hàm công khai trong module này trở thành các lệnh mà người dùng có thể
gọi.

### !publicInstructions

_rs`initialize`_ và _rs`increment`_ là hai tên lệnh mà người dùng có thể gọi.
Mỗi lệnh nhận một _rs`Context<T>`_. Kiểu bên trong _rs`Context`_ định nghĩa các
tài khoản mà lệnh đó yêu cầu, và Anchor xác thực các tài khoản đó trước khi lệnh
được thực thi.

### !handlers

Hàm xử lý chứa logic cho lệnh.

</WithNotes>

Địa chỉ chương trình giống nhau xuất hiện trong cả cấu hình và mã nguồn.
`Anchor.toml` cho Anchor biết địa chỉ cần triển khai hoặc gọi cho một cluster.
_rs`declare_id!`_ định nghĩa địa chỉ chương trình trong chương trình để kiểm tra
bảo mật.

### constants.rs

`constants.rs` lưu các giá trị dùng chung ở một nơi. Trong template này,
_rs`COUNTER_SEED`_ dẫn xuất PDA của bộ đếm, _rs`HELLO_WORLD_LAMPORTS`_ được
chuyển trong quá trình khởi tạo, và _rs`MAX_COUNT`_ được kiểm tra trước khi tăng
giá trị.

<WithNotes>

```rust title="programs/my-program/src/constants.rs" -w
use anchor_lang::prelude::*;

// !tooltip[/#\[constant\]/] anchorConstant
// !tooltip[/COUNTER_SEED/] counterSeed
#[constant]
pub const COUNTER_SEED: &[u8] = b"counter";

// !tooltip[/HELLO_WORLD_LAMPORTS/] helloLamports
#[constant]
pub const HELLO_WORLD_LAMPORTS: u64 = 1;

// !tooltip[/MAX_COUNT/] maxCount
#[constant]
pub const MAX_COUNT: u64 = 10;
```

### !anchorConstant

_rs`#[constant]`_ yêu cầu Anchor đưa giá trị vào IDL của chương trình. Điều này
giúp các hằng số quan trọng của chương trình hiển thị với các client được tạo tự
động và các công cụ hỗ trợ.

### !counterSeed

_rs`COUNTER_SEED`_ là chuỗi byte dùng để dẫn xuất PDA của bộ đếm. Cả chương
trình và bài kiểm tra đều sử dụng seed này để đồng thuận về địa chỉ tài khoản mà
không cần lưu trữ một keypair riêng biệt. Template này sử dụng một seed cố định
duy nhất, vì vậy chương trình có một địa chỉ bộ đếm. Các chương trình thực tế
thường bao gồm địa chỉ của người dùng hoặc một định danh khác trong các seeds để
tạo nhiều tài khoản có liên quan.

### !helloLamports

_rs`HELLO_WORLD_LAMPORTS`_ là lượng lamport được chuyển từ người thanh toán đến
tài khoản bộ đếm bên trong _rs`initialize`_. Giá trị này cố ý rất nhỏ vì
template đang minh họa một CPI đến System Program.

### !maxCount

_rs`MAX_COUNT`_ là giới hạn trên được kiểm tra bởi _rs`increment`_. Đặt nó ở đây
giúp chương trình, các bài kiểm tra và IDL được tạo tự động có một tên chung duy
nhất cho quy tắc này.

</WithNotes>

<SideBySide>

### !left

```rust title="initialize.rs" -w
#[derive(Accounts)]
pub struct Initialize<'info> {
    // ...
    #[account(
        init,
        payer = payer,
        space = 8 + Counter::INIT_SPACE,
        // !mark
        // !mention counter-seed
        seeds = [COUNTER_SEED],
        bump
    )]
    pub counter: Account<'info, Counter>,
    // ...
}

pub fn handle_initialize(ctx: Context<Initialize>) -> Result<()> {
    // ...

    // !mark
    // !mention hello-world-lamports
    anchor_lang::system_program::transfer(cpi_ctx, HELLO_WORLD_LAMPORTS)?;

    Ok(())
}
```

### !right

`initialize.rs` sử dụng hai hằng số:

- [_rs`COUNTER_SEED`_](mention:counter-seed) lấy địa chỉ PDA của bộ đếm.
- [_rs`HELLO_WORLD_LAMPORTS`_](mention:hello-world-lamports) đặt số lượng được
  chuyển từ người thanh toán sang tài khoản bộ đếm.

</SideBySide>

<SideBySide>

### !left

```rust title="increment.rs" -w
pub fn handle_increment(ctx: Context<Increment>) -> Result<()> {
    // ...

    require!(
        // !mark
        // !mention max-count
        ctx.accounts.counter.count < MAX_COUNT,
        ErrorCode::CounterOverflow,
    );

    ctx.accounts.counter.count += 1;
    Ok(())
}
```

### !right

`increment.rs` sử dụng [_rs`MAX_COUNT`_](mention:max-count) làm giới hạn trên
của bộ đếm. Nếu giá trị hiện tại đã đạt tối đa, _rs`require!`_ trả về
_rs`CounterOverflow`_ và dữ liệu tài khoản không bị thay đổi.

</SideBySide>

### state.rs

`state.rs` định nghĩa các kiểu dữ liệu tùy chỉnh cho các tài khoản mà chương
trình tạo ra và sở hữu. Chương trình định nghĩa các lệnh để tạo, khởi tạo và cập
nhật dữ liệu đó, nhưng dữ liệu bộ đếm không được lưu trữ bên trong chính chương
trình. Nó được lưu trữ trong một tài khoản riêng biệt với địa chỉ của chính nó.

<WithNotes>

```rust title="programs/my-program/src/state.rs" -w
use anchor_lang::prelude::*;

// !tooltip[/#\[account\]/] accountMacro
#[account]
// !tooltip[/InitSpace/] initSpace
#[derive(InitSpace)]
pub struct Counter {
    // !tooltip[/count/] count
    pub count: u64,
    // !tooltip[/authority/] authority
    pub authority: Pubkey,
}
```

### !accountMacro

_rs`#[account]`_ cho Anchor biết rằng struct này là dữ liệu có thể được lưu trữ
trong một tài khoản onchain do chương trình này sở hữu. Anchor sử dụng nó để
tuần tự hóa, giải tuần tự hóa, và xác thực các tài khoản _rs`Counter`_.

### !initSpace

_rs`InitSpace`_ cho phép Anchor tính toán _rs`Counter::INIT_SPACE`_, số byte cần
thiết cho các trường của struct này. Bộ phân biệt tài khoản Anchor 8 byte được
thêm riêng khi tài khoản được tạo.

### !count

_rs`count`_ là giá trị bộ đếm được lưu trữ bên trong dữ liệu tài khoản.

### !authority

_rs`authority`_ lưu trữ địa chỉ được phép tăng bộ đếm. _rs`initialize`_ ghi địa
chỉ này, và _rs`increment`_ yêu cầu cùng địa chỉ đó để ký.

</WithNotes>

<SideBySide>

### !left

```rust title="initialize.rs" -w
#[account(
    init,
    payer = payer,
    // !mark
    // !mention counter-space
    space = 8 + Counter::INIT_SPACE,
    seeds = [COUNTER_SEED],
    bump
)]
pub counter: Account<'info, Counter>,
// ...

pub fn handle_initialize(ctx: Context<Initialize>) -> Result<()> {
    // !mark(1:2)
    // !mention(1:2) initialize-fields
    ctx.accounts.counter.count = 0;
    ctx.accounts.counter.authority = ctx.accounts.payer.key();

    Ok(())
}
```

```rust title="increment.rs" -w
pub fn handle_increment(ctx: Context<Increment>) -> Result<()> {
    // ...

    // !mark
    // !mention increment-count
    ctx.accounts.counter.count += 1;
    Ok(())
}
```

### !right

`state.rs` định nghĩa dữ liệu tài khoản _rs`Counter`_. Các tệp lệnh sử dụng kiểu
đó khi tạo và cập nhật tài khoản:

- [_rs`Counter::INIT_SPACE`_](mention:counter-space) xác định kích thước tài
  khoản cho các trường được định nghĩa trong `state.rs`.
- [_rs`count`_ và _rs`authority`_](mention:initialize-fields) là các giá trị
  trường được ghi khi tài khoản được khởi tạo.
- [_rs`count += 1`_](mention:increment-count) cập nhật giá trị bộ đếm đã lưu sau
  khi xác thực thành công.

</SideBySide>

### error.rs

`error.rs` định nghĩa các lỗi tùy chỉnh của chương trình. Trong mẫu này, các lỗi
minh họa cách các trình xử lý lệnh dừng lại khi người gọi không được phép cập
nhật bộ đếm hoặc bộ đếm đã đạt đến _rs`MAX_COUNT`_.

<WithNotes>

```rust title="programs/my-program/src/error.rs" -w
use anchor_lang::prelude::*;

// !tooltip[/#\[error_code\]/] errorCode
#[error_code]
pub enum ErrorCode {
    // !tooltip[/Unauthorized/] unauthorized
    #[msg("Only the counter authority can update this counter")]
    Unauthorized,
    // !tooltip[/CounterOverflow/] counterOverflow
    #[msg("Counter has reached the maximum value")]
    CounterOverflow,
}
```

### !errorCode

_rs`#[error_code]`_ yêu cầu Anchor chuyển đổi các biến thể này thành các lỗi
chương trình tùy chỉnh. Các client và bài kiểm thử có thể sử dụng những lỗi đó
thay vì phải đoán từ mã lỗi thô.

### !unauthorized

_rs`Unauthorized`_ được trả về khi người ký giao dịch không phải là authority
được lưu trong tài khoản bộ đếm.

### !counterOverflow

_rs`CounterOverflow`_ được trả về khi việc tăng giá trị sẽ vi phạm quy tắc
_rs`MAX_COUNT`_ của chương trình.

</WithNotes>

<SideBySide>

### !left

```rust title="increment.rs" -w
pub fn handle_increment(ctx: Context<Increment>) -> Result<()> {
    require_keys_eq!(
        ctx.accounts.counter.authority,
        ctx.accounts.authority.key(),
        // !mark
        // !mention unauthorized-error
        ErrorCode::Unauthorized,
    );
    require!(
        ctx.accounts.counter.count < MAX_COUNT,
        // !mark
        // !mention overflow-error
        ErrorCode::CounterOverflow,
    );

    ctx.accounts.counter.count += 1;
    msg!("Hello, world! Counter is now {}", ctx.accounts.counter.count);
    Ok(())
}
```

### !right

`error.rs` liệt kê các lỗi mà lệnh có thể trả về:

- [_rs`ErrorCode::Unauthorized`_](mention:unauthorized-error) được trả về khi
  người ký không phải là authority được lưu trong tài khoản bộ đếm.
- [_rs`ErrorCode::CounterOverflow`_](mention:overflow-error) được trả về khi bộ
  đếm đã đạt đến _rs`MAX_COUNT`_.

</SideBySide>

### instructions.rs

`instructions.rs` kết nối các tệp lệnh với crate chương trình để `lib.rs` có thể
truy cập mã lệnh _rs`initialize`_ và _rs`increment`_. Mỗi tệp lệnh định nghĩa
các tài khoản mà lệnh đó yêu cầu và logic xử lý chạy sau khi Anchor xác thực các
tài khoản đó.

<WithNotes>

```rust title="programs/my-program/src/instructions.rs" -w
// !tooltip[/pub mod initialize/] moduleDeclarations
pub mod initialize;
pub mod increment;

// !tooltip[/pub use initialize/] reexports
pub use initialize::*;
pub use increment::*;
```

### initialize.rs

`initialize.rs` định nghĩa các tài khoản cần thiết để tạo tài khoản counter, sau
đó ghi các giá trị đầu tiên của tài khoản. Struct _rs`#[derive(Accounts)]`_ sử
dụng các
[ràng buộc tài khoản](https://www.anchor-lang.com/docs/references/account-constraints)
của Anchor để chỉ định các tài khoản nào là bắt buộc và cách tạo tài khoản
counter mới.

<WithNotes>

```rust title="programs/my-program/src/instructions/initialize.rs" -w
use anchor_lang::prelude::*;

use crate::{constants::*, state::Counter};

// !tooltip[/derive\(Accounts\)/] accountsStruct
#[derive(Accounts)]
pub struct Initialize<'info> {
    // !tooltip[/#\[account\(mut\)\]/] payerMutable
    #[account(mut)]
    // !tooltip[/Signer<'info>/] payerSigner
    pub payer: Signer<'info>,
    // !tooltip[/#\[account\(/] counterConstraints
    #[account(
        // !tooltip[/init/] initConstraint
        init,
        // !tooltip[/payer = payer/] payerConstraint
        payer = payer,
        // !tooltip[/space = 8 \+ Counter::INIT_SPACE/] space
        space = 8 + Counter::INIT_SPACE,
        // !tooltip[/seeds = \[COUNTER_SEED\]/] seeds
        seeds = [COUNTER_SEED],
        // !tooltip[/bump/] bump
        bump
    )]
    // !tooltip[/Account<'info, Counter>/] counterAccountType
    pub counter: Account<'info, Counter>,
    // !tooltip[/Program<'info, System>/] systemProgram
    pub system_program: Program<'info, System>,
}

// !tooltip[/handle_initialize/] handler
pub fn handle_initialize(ctx: Context<Initialize>) -> Result<()> {
    // !tooltip[/counter.count = 0/] initialCount
    ctx.accounts.counter.count = 0;
    // !tooltip[/counter.authority = ctx.accounts.payer.key\(\)/] initialAuthority
    ctx.accounts.counter.authority = ctx.accounts.payer.key();

    // !tooltip[/system_program::Transfer/] cpiAccounts
    let cpi_accounts = anchor_lang::system_program::Transfer {
        from: ctx.accounts.payer.to_account_info(),
        to: ctx.accounts.counter.to_account_info(),
    };
    // !tooltip[/CpiContext::new/] cpiContext
    let cpi_ctx = CpiContext::new(anchor_lang::system_program::ID, cpi_accounts);
    // !tooltip[/system_program::transfer/] transfer
    anchor_lang::system_program::transfer(cpi_ctx, HELLO_WORLD_LAMPORTS)?;

    // !tooltip[/msg!/] programLog
    msg!("Hello, world! Counter initialized");
    Ok(())
}
```

### !imports

Tệp này nhập các hằng số dùng chung và kiểu tài khoản _rs`Counter`_. Lệnh sử
dụng các định nghĩa đó để suy ra PDA, chọn số lượng transfer và cấp phát tài
khoản với kích thước dữ liệu phù hợp.

### !accountsStruct

_rs`Initialize`_ là ngữ cảnh tài khoản cho lệnh _rs`initialize`_. Mỗi trường
trong struct đại diện cho một tài khoản riêng biệt phải được đưa vào khi lệnh
được gọi. Anchor sử dụng struct này để xác thực các tài khoản đầu vào trước khi
handler chạy.

### !payerMutable

_rs`#[account(mut)]`_ báo cho Anchor biết tài khoản payer phải có quyền ghi.
Điều này là bắt buộc vì việc tạo tài khoản và CPI transfer đều khấu trừ lamport
từ payer.

### !payerSigner

_rs`Signer<'info>`_ báo cho Anchor biết payer phải ký giao dịch.

### !counterConstraints

Khối _rs`#[account(...)]`_ báo cho Anchor biết cách tạo và xác thực tài khoản
counter trước khi handler chạy.

### !initConstraint

_rs`init`_ yêu cầu Anchor tạo tài khoản _rs`counter`_ trước khi handler chạy.
Tài khoản mới thuộc sở hữu của chương trình này, do đó chương trình có thể lưu
trữ dữ liệu _rs`Counter`_ trong đó.

### !payerConstraint

_rs`payer = payer`_ chỉ định tài khoản _rs`payer`_ là người thanh toán cho việc
tạo tài khoản counter mới.

### !space

_rs`space = 8 + Counter::INIT_SPACE`_ phân bổ số byte cho discriminator tài
khoản của Anchor cộng với các trường _rs`Counter`_ từ `state.rs`. Discriminator
giúp Anchor xác nhận tài khoản lưu trữ đúng kiểu dữ liệu mong đợi trước khi đọc.

### !seeds

_rs`seeds = [COUNTER_SEED]`_ quy định rằng địa chỉ tài khoản counter phải là PDA
được suy ra từ seed này và program ID. Với một seed cố định, chương trình này
chỉ có một counter PDA. Để tạo các counter riêng biệt trong một chương trình
thực tế, bạn thường cần thêm một seed khác như địa chỉ của người dùng.

### !bump

_rs`bump`_ yêu cầu Anchor tìm và xác thực bump của PDA, đây là byte bổ sung giúp
địa chỉ được suy ra hợp lệ như một PDA.

### !counterAccountType

_rs`Account<'info, Counter>`_ yêu cầu Anchor xử lý dữ liệu tài khoản theo kiểu
_rs`Counter`_ từ `state.rs`. Anchor cũng kiểm tra rằng tài khoản thuộc sở hữu
của chương trình này.

### !systemProgram

_rs`Program<'info, System>`_ yêu cầu người gọi cung cấp System Program. Anchor
sử dụng nó cho ràng buộc tạo tài khoản _rs`init`_. CPI chuyển tiền của handler
cũng gọi System Program.

### !handler

Handler chạy sau khi Anchor xác thực các tài khoản và tạo tài khoản counter, do
đó nó có thể ghi dữ liệu khởi tạo của tài khoản một cách an toàn.

### !initialCount

Đặt trường _rs`count`_ trong tài khoản counter thành _rs`0`_.

### !initialAuthority

Đặt trường _rs`authority`_ trong tài khoản counter thành địa chỉ của người thanh
toán.

### !cpiAccounts

Đây là các tài khoản được truyền vào lệnh chuyển của System Program. CPI chuyển
này chỉ được đưa vào để minh họa cách một CPI truyền tài khoản sang chương trình
khác.

### !cpiContext

_rs`CpiContext`_ đóng gói chương trình đích và các tài khoản cần thiết cho một
Cross Program Invocation. Việc tạo context chuẩn bị CPI bằng cách ghép nối
chương trình bạn muốn gọi với các tài khoản mà chương trình đó yêu cầu.

### !transfer

CPI này gọi lệnh chuyển của System Program và chuyển _rs`HELLO_WORLD_LAMPORTS`_
từ người thanh toán sang tài khoản counter.

### !programLog

_rs`VersionedTransaction::try_new`_ ký message bằng keypair của payer. Payer ký
vì nó thanh toán phí giao dịch và cấp vốn cho tài khoản counter mới.

### !sendTx

_rs`svm.send_transaction`_ thực thi giao dịch trong LiteSVM.

### !getCounter

Sau khi giao dịch chạy, bài test đọc trực tiếp tài khoản counter từ LiteSVM để
có thể kiểm tra dữ liệu tài khoản.

### !deserialize

Tài khoản counter lưu trữ dữ liệu đã được serialize.
_rs`Counter::try_deserialize`_ kiểm tra bộ phân biệt tài khoản của Anchor và
chuyển đổi dữ liệu tài khoản trở lại thành kiểu _rs`Counter`_ từ `state.rs`.

### !initAssert

Sau khi _rs`initialize`_ thành công, bài test deserialize tài khoản counter và
kiểm tra rằng trạng thái ban đầu khớp với những gì _rs`handle_initialize`_ đã
ghi.

### !incrementAssert

Sau khi _rs`increment`_ thành công, cùng tài khoản counter đó sẽ có _rs`count`_
được đặt thành _rs`1`_ trong khi vẫn giữ nguyên authority.

</WithNotes>

## Cấu hình dự án

Các tệp dự án gốc cho Anchor và Cargo biết cách build, test và triển khai chương
trình. Để tham khảo đầy đủ, xem tài liệu Anchor cho
[cấu hình Anchor.toml](https://www.anchor-lang.com/docs/references/anchor-toml)
và [Anchor CLI](https://www.anchor-lang.com/docs/references/cli).

<WithNotes>

```toml title="Anchor.toml"
# !tooltip[/skip_local_validator/] skipValidator
skip_local_validator = true

[toolchain]

[features]
resolution = true
skip-lint = false

# !tooltip[/programs.localnet/] programs
[programs.localnet]
my_program = "82sFkffP9wxwpyfZyeaKHH2chQoJPUGsJZSPi9mrUuXd"

# !tooltip[/provider/] provider
[provider]
cluster = "localnet"
wallet = "~/.config/solana/id.json"

# !tooltip[/scripts/] scripts
[scripts]
test = "cargo test"

[hooks]
```

### !skipValidator

`skip_local_validator = true` yêu cầu Anchor không khởi động validator cho
`anchor test`. Bài kiểm tra Rust sử dụng LiteSVM thay thế.

### !programs

`[programs.localnet]` ánh xạ tên crate Rust tới địa chỉ program account được
dùng cho bản build và kiểm tra cục bộ.

### !provider

`[provider]` thiết lập cluster và ví mặc định cho các lệnh như `anchor deploy`.

### !scripts

`[scripts]` thay đổi những gì `anchor test` chạy. Trong template này,
`anchor test` chạy `cargo test`.

</WithNotes>
