이 퀵스타트는 `anchor init`로 생성된
[Anchor 프레임워크](https://www.anchor-lang.com/docs) 스타터 프로젝트를
사용합니다. 프로젝트를 로컬에 생성하고, 테스트를 실행하고, 프로그램을 빌드하며,
Anchor가 생성하는 프로그램 코드를 살펴봅니다.

## 사전 요구 사항

시작하기 전에 [Solana 개발 도구를 설치하세요](/docs/intro/installation). 설치
항목에는 Rust, Solana CLI, Anchor CLI가 포함됩니다.

이 템플릿에는 Anchor CLI 버전 1.1.2 이상을 사용하세요. 설치된 버전을 확인하려면:

```terminal
$ anchor --version
```

<Steps>

<Step>

### 프로젝트 생성

터미널에서 다음 명령어를 실행하세요:

```terminal
$ anchor init my-program
$ cd my-program
```

스타터 프로젝트에는 `programs/my-program` 아래에 Solana 프로그램이 하나 포함되어
있습니다. 이 프로그램에는 두 가지 명령어가 있습니다: 카운터 계정을 초기화하는
명령어와 카운터를 증가시키는 명령어입니다.

템플릿의 일부 코드는 일반적인 Solana 프로그램 패턴을 보여줍니다:
[PDA](/docs/core/pda) 계정 주소 도출, SOL을 전송하기 위한 Cross Program
Invocation([CPI](/docs/core/cpi)) 수행, 그리고 조건 실패 시 명령어를 중단하기
위한 커스텀 오류 검사 사용.

<Step>

### 프로그램 빌드

`anchor build`를 실행하여 스타터 프로그램을 컴파일하세요:

```terminal
$ anchor build
```

컴파일된 프로그램은 `target/deploy/my_program.so`에 저장됩니다. 프로그램이
배포되면 이 `.so` 파일의 내용이 온체인 계정에 저장됩니다.

</Step>

<Step>

### 테스트 실행

기본 테스트를 실행합니다:

```terminal
$ anchor test
```

이 템플릿의 `Anchor.toml`는 Rust 테스트 명령을 사용합니다:

```toml title="Anchor.toml"
skip_local_validator = true

[scripts]
test = "cargo test"
```

테스트는 컴파일된 프로그램을 [LiteSVM](https://www.litesvm.com/)에 로드하고,
페이어를 생성하여 _rs`initialize`_ 및 _rs`increment`_ 명령어를 전송한 후, 카운터
계정 상태를 확인합니다.

`anchor test`를 실행하면 프로그램도 함께 컴파일되므로, 로컬에서 테스트할 때 먼저
`anchor build`를 실행할 필요가 없습니다.

</Step>

<Step>

### 프로그램 배포

로컬 테스트는 가장 빠른 피드백 루프입니다. devnet과 같은 네트워크에 배포할
준비가 되었다면, 먼저 빌드한 후 클러스터에 배포하세요.

Solana 프로그램을 배포하려면 SOL이 필요합니다. 프로그램이 계정에 저장되고, 해당
계정은 사용하는 공간에 대한 비용을 지불해야 하기 때문입니다. devnet에서는
[Solana Faucet](https://faucet.solana.com/)을 이용하거나 Solana CLI로 무료
devnet SOL을 요청할 수 있습니다:

```terminal
$ solana airdrop 2 --url devnet
```

```terminal
$ anchor build
$ anchor deploy --provider.cluster devnet
```

</Step>

</Steps>

## 소스 파일

`src` 디렉터리에는 Solana 프로그램이 포함되어 있습니다. Anchor의
[Program Structure](https://www.anchor-lang.com/docs/basics/program-structure)
문서에서는 _rs`declare_id!`_, _rs`#[program]`_, _rs`#[derive(Accounts)]`_,
_rs`#[account]`_ 등 여기서 사용된 핵심 매크로를 설명합니다. 이 섹션에서는 템플릿
파일들을 살펴봅니다.

### lib.rs

`lib.rs`는 프로그램 진입점입니다. 소스 파일을 연결하고, 프로그램 주소를
정의하며, 사용자가 호출할 수 있는 프로그램 명령어를 정의합니다.

<WithNotes>

```rust title="programs/my-program/src/lib.rs" -w
// !tooltip[/pub mod constants/] constantsMod
pub mod constants;
// !tooltip[/pub mod error/] errorMod
pub mod error;
// !tooltip[/pub mod instructions/] instructionsMod
pub mod instructions;
// !tooltip[/pub mod state/] stateMod
pub mod state;

// !tooltip[/anchor_lang::prelude/] prelude
use anchor_lang::prelude::*;

pub use constants::*;
pub use instructions::*;
pub use state::*;

// !tooltip[/declare_id!/] declareId
declare_id!("82sFkffP9wxwpyfZyeaKHH2chQoJPUGsJZSPi9mrUuXd");

// !tooltip[/#\[program\]/] programMacro
#[program]
pub mod my_program {
    use super::*;

    // !tooltip[/pub fn initialize/] publicInstructions
    pub fn initialize(ctx: Context<Initialize>) -> Result<()> {
        // !tooltip[/handle_initialize/] handlers
        crate::instructions::initialize::handle_initialize(ctx)
    }

    // !tooltip[/pub fn increment/] publicInstructions
    pub fn increment(ctx: Context<Increment>) -> Result<()> {
        // !tooltip[/handle_increment/] handlers
        crate::instructions::increment::handle_increment(ctx)
    }
}
```

### !constantsMod

*rs`pub mod constants`*은(는) `constants.rs`을(를) 프로그램 크레이트의 일부로
만듭니다. 이 파일에는 PDA seed, 전송 금액, 최대 카운터 값과 같은 공유 프로그램
값이 저장됩니다.

### !errorMod

*rs`pub mod error`*은(는) `error.rs`을(를) 프로그램 크레이트의 일부로 만듭니다.
이 파일은 명령어가 유효하지 않을 때 프로그램이 반환할 수 있는 커스텀 오류를
정의합니다.

### !instructionsMod

*rs`pub mod instructions`*은(는) `instructions.rs`을(를) 프로그램 크레이트의
일부로 만듭니다. 해당 파일은 `src/instructions/` 내 명령어별 파일들의 인덱스
역할을 합니다.

### !stateMod

*rs`pub mod state`*은(는) `state.rs`을(를) 프로그램 크레이트의 일부로 만듭니다.
이 파일은 _rs`Counter`_ account를 포함하여 이 프로그램이 소유한 계정의 데이터
레이아웃을 정의합니다.

### !prelude

_rs`anchor_lang::prelude::_`*은(는) Solana 프로그램 정의에 사용되는 Anchor 타입과 매크로를 가져옵니다. 여기에는 _rs`Context`_, _rs`Result`_, _rs`Pubkey`_, _rs`Account`_, _rs`Signer`\_
및 Anchor 계정 속성이 포함됩니다.

### !reexports

_rs`pub use`_ 줄은 하위 모듈의 공개 항목을 크레이트 루트에서 사용할 수 있도록
합니다. 이를 통해 긴 모듈 경로 대신 *rs`Context<Initialize>`*와 같이 명령어
진입점을 간결하게 유지할 수 있습니다.

### !declareId

*rs`declare_id!`*은(는) 프로그램의 주소를 정의합니다. 이 주소는 `Anchor.toml`,
배포된 program account, 그리고 프로그램을 호출하는 모든 사용자 또는 테스트와
일치해야 합니다.

### !programMacro

*rs`#[program]`*은(는) Anchor에게 Solana 프로그램 진입점과 명령어 디스패치
코드를 생성하도록 지시합니다. 이 모듈의 공개 함수들이 사용자가 호출할 수 있는
명령어가 됩니다.

### !publicInstructions

*rs`initialize`*와 *rs`increment`*은(는) 사용자가 호출할 수 있는 두 가지 명령어
이름입니다. 각각은 *rs`Context<T>`*을(를) 받습니다. _rs`Context`_ 내부의 타입은
해당 명령어에 필요한 계정을 정의하며, Anchor는 명령어 실행 전에 해당 계정들을
검증합니다.

### !handlers

핸들러 함수에는 명령어에 대한 로직이 포함되어 있습니다.

</WithNotes>

### !left

```toml title="Anchor.toml" -w
[programs.localnet]
# !mark
my_program = "82sFkffP9wxwpyfZyeaKHH2chQoJPUGsJZSPi9mrUuXd"
```

### !right

```rust title="lib.rs" -w
// !mark
declare_id!("82sFkffP9wxwpyfZyeaKHH2chQoJPUGsJZSPi9mrUuXd");
```

</SideBySide>

동일한 프로그램 주소가 설정과 코드 모두에 나타납니다. `Anchor.toml`는 Anchor에게
클러스터에 배포하거나 호출할 주소를 알려줍니다. *rs`declare_id!`*는 보안 검사를
위해 프로그램 내에서 프로그램 주소를 정의합니다.

### constants.rs

`constants.rs`는 공유 값을 한 곳에 보관합니다. 이 템플릿에서
*rs`COUNTER_SEED`*는 카운터 PDA를 도출하고, *rs`HELLO_WORLD_LAMPORTS`*는 초기화
시 전송되며, *rs`MAX_COUNT`*는 증가 전에 확인됩니다.

<WithNotes>

```rust title="programs/my-program/src/constants.rs" -w
use anchor_lang::prelude::*;

// !tooltip[/#\[constant\]/] anchorConstant
// !tooltip[/COUNTER_SEED/] counterSeed
#[constant]
pub const COUNTER_SEED: &[u8] = b"counter";

// !tooltip[/HELLO_WORLD_LAMPORTS/] helloLamports
#[constant]
pub const HELLO_WORLD_LAMPORTS: u64 = 1;

// !tooltip[/MAX_COUNT/] maxCount
#[constant]
pub const MAX_COUNT: u64 = 10;
```

### !anchorConstant

*rs`#[constant]`*는 Anchor에게 해당 값을 프로그램 IDL에 포함하도록 지시합니다.
이를 통해 중요한 프로그램 상수를 생성된 클라이언트 및 툴링에서 확인할 수
있습니다.

### !counterSeed

*rs`COUNTER_SEED`*는 카운터 PDA를 도출하는 데 사용되는 바이트 문자열입니다.
프로그램과 테스트 모두 이 seed를 사용하므로 별도의 keypair를 저장하지 않고도
계정 주소에 동의할 수 있습니다. 이 템플릿은 하나의 고정된 seed를 사용하므로
프로그램에는 하나의 카운터 주소가 있습니다. 실제 프로그램에서는 여러 관련 계정을
생성하기 위해 사용자 주소나 다른 식별자를 seeds에 포함하는 경우가 많습니다.

### !helloLamports

*rs`HELLO_WORLD_LAMPORTS`*는 _rs`initialize`_ 내에서 지불자로부터 카운터
계정으로 전송되는 lamport의 양입니다. 템플릿이 System Program에 대한 CPI를
시연하고 있기 때문에 의도적으로 매우 작은 값으로 설정되어 있습니다.

### !maxCount

*rs`MAX_COUNT`*는 *rs`increment`*가 검사하는 상한값입니다. 이 값을 여기에 두면
프로그램, 테스트, 생성된 IDL이 해당 규칙에 대해 하나의 공통된 이름을 공유하게
됩니다.

</WithNotes>

<SideBySide>

### !left

```rust title="initialize.rs" -w
#[derive(Accounts)]
pub struct Initialize<'info> {
    // ...
    #[account(
        init,
        payer = payer,
        space = 8 + Counter::INIT_SPACE,
        // !mark
        // !mention counter-seed
        seeds = [COUNTER_SEED],
        bump
    )]
    pub counter: Account<'info, Counter>,
    // ...
}

pub fn handle_initialize(ctx: Context<Initialize>) -> Result<()> {
    // ...

    // !mark
    // !mention hello-world-lamports
    anchor_lang::system_program::transfer(cpi_ctx, HELLO_WORLD_LAMPORTS)?;

    Ok(())
}
```

### !right

`initialize.rs`는 두 가지 상수를 사용합니다:

- [_rs`COUNTER_SEED`_](mention:counter-seed)는 카운터 PDA 주소를 도출합니다.
- [_rs`HELLO_WORLD_LAMPORTS`_](mention:hello-world-lamports)는 지불자에서 카운터
  계정으로 전송되는 금액을 설정합니다.

</SideBySide>

<SideBySide>

### !left

```rust title="increment.rs" -w
pub fn handle_increment(ctx: Context<Increment>) -> Result<()> {
    // ...

    require!(
        // !mark
        // !mention max-count
        ctx.accounts.counter.count < MAX_COUNT,
        ErrorCode::CounterOverflow,
    );

    ctx.accounts.counter.count += 1;
    Ok(())
}
```

### !right

`increment.rs`는 [_rs`MAX_COUNT`_](mention:max-count)를 카운터의 상한값으로
사용합니다. 현재 카운트가 이미 최댓값에 도달한 경우, *rs`require!`*는
*rs`CounterOverflow`*를 반환하고 계정 데이터는 변경되지 않습니다.

</SideBySide>

### state.rs

`state.rs`는 프로그램이 생성하고 소유하는 계정에 대한 커스텀 데이터 타입을
정의합니다. 프로그램은 해당 데이터를 생성, 초기화 및 업데이트하는 명령어를
정의하지만, 카운터 데이터는 프로그램 자체에 저장되지 않습니다. 이는 고유한
주소를 가진 별도의 계정에 저장됩니다.

<WithNotes>

```rust title="programs/my-program/src/state.rs" -w
use anchor_lang::prelude::*;

// !tooltip[/#\[account\]/] accountMacro
#[account]
// !tooltip[/InitSpace/] initSpace
#[derive(InitSpace)]
pub struct Counter {
    // !tooltip[/count/] count
    pub count: u64,
    // !tooltip[/authority/] authority
    pub authority: Pubkey,
}
```

### !accountMacro

*rs`#[account]`*는 Anchor에게 이 구조체가 이 프로그램이 소유하는 온체인 계정에
저장될 수 있는 데이터임을 알립니다. Anchor는 이를 사용하여 _rs`Counter`_ 계정을
직렬화, 역직렬화 및 검증합니다.

### !initSpace

*rs`InitSpace`*를 사용하면 Anchor가 이 구조체의 필드에 필요한 바이트 수인
*rs`Counter::INIT_SPACE`*를 계산할 수 있습니다. 8바이트의 Anchor 계정 식별자는
계정이 생성될 때 별도로 추가됩니다.

### !count

*rs`count`*는 계정 데이터 내에 저장된 카운터 값입니다.

### !authority

*rs`authority`*는 카운터를 증가시킬 수 있는 권한이 부여된 주소를 저장합니다.
*rs`initialize`*는 이 주소를 기록하고, *rs`increment`*는 동일한 주소의 서명을
요구합니다.

</WithNotes>

<SideBySide>

### !left

```rust title="initialize.rs" -w
#[account(
    init,
    payer = payer,
    // !mark
    // !mention counter-space
    space = 8 + Counter::INIT_SPACE,
    seeds = [COUNTER_SEED],
    bump
)]
pub counter: Account<'info, Counter>,
// ...

pub fn handle_initialize(ctx: Context<Initialize>) -> Result<()> {
    // !mark(1:2)
    // !mention(1:2) initialize-fields
    ctx.accounts.counter.count = 0;
    ctx.accounts.counter.authority = ctx.accounts.payer.key();

    Ok(())
}
```

```rust title="increment.rs" -w
pub fn handle_increment(ctx: Context<Increment>) -> Result<()> {
    // ...

    // !mark
    // !mention increment-count
    ctx.accounts.counter.count += 1;
    Ok(())
}
```

### !right

`state.rs`는 _rs`Counter`_ 계정 데이터를 정의합니다. 명령어 파일은 계정을
생성하고 업데이트할 때 해당 타입을 사용합니다:

- [_rs`Counter::INIT_SPACE`_](mention:counter-space)는 `state.rs`에 정의된
  필드에 맞게 계정 크기를 조정합니다.
- [_rs`count`_ 및 _rs`authority`_](mention:initialize-fields)는 계정이 초기화될
  때 기록되는 필드 값입니다.
- [_rs`count += 1`_](mention:increment-count)는 유효성 검사를 통과한 후 저장된
  카운터 값을 업데이트합니다.

</SideBySide>

### error.rs

`error.rs`는 프로그램의 커스텀 오류를 정의합니다. 이 템플릿에서 오류는 호출자가
카운터를 업데이트할 권한이 없거나 카운터가 이미 *rs`MAX_COUNT`*에 도달했을 때
명령어 핸들러가 중단되는 방식을 보여줍니다.

<WithNotes>

```rust title="programs/my-program/src/error.rs" -w
use anchor_lang::prelude::*;

// !tooltip[/#\[error_code\]/] errorCode
#[error_code]
pub enum ErrorCode {
    // !tooltip[/Unauthorized/] unauthorized
    #[msg("Only the counter authority can update this counter")]
    Unauthorized,
    // !tooltip[/CounterOverflow/] counterOverflow
    #[msg("Counter has reached the maximum value")]
    CounterOverflow,
}
```

### !errorCode

*rs`#[error_code]`*는 Anchor에게 이 변형들을 커스텀 프로그램 오류로 변환하도록
지시합니다. 클라이언트와 테스트는 원시 오류 코드를 추측하는 대신 해당 오류를
사용할 수 있습니다.

### !unauthorized

*rs`Unauthorized`*는 트랜잭션 서명자가 카운터 계정에 저장된 권한자가 아닐 때
반환됩니다.

### !counterOverflow

*rs`CounterOverflow`*는 증가 시 프로그램의 _rs`MAX_COUNT`_ 규칙을 위반하게 될 때
반환됩니다.

</WithNotes>

### !left


### initialize.rs

`initialize.rs`는 카운터 계정을 생성하는 데 필요한 계정을 정의하고, 계정의 초기
값을 기록합니다. _rs`#[derive(Accounts)]`_ 구조체는 Anchor
[account constraints](https://www.anchor-lang.com/docs/references/account-constraints)를
사용하여 어떤 계정이 필요한지, 새로운 카운터 계정이 어떻게 생성되는지를
지정합니다.

<WithNotes>

```rust title="programs/my-program/src/instructions/initialize.rs" -w
use anchor_lang::prelude::*;

use crate::{constants::*, state::Counter};

// !tooltip[/derive\(Accounts\)/] accountsStruct
#[derive(Accounts)]
pub struct Initialize<'info> {
    // !tooltip[/#\[account\(mut\)\]/] payerMutable
    #[account(mut)]
    // !tooltip[/Signer<'info>/] payerSigner
    pub payer: Signer<'info>,
    // !tooltip[/#\[account\(/] counterConstraints
    #[account(
        // !tooltip[/init/] initConstraint
        init,
        // !tooltip[/payer = payer/] payerConstraint
        payer = payer,
        // !tooltip[/space = 8 \+ Counter::INIT_SPACE/] space
        space = 8 + Counter::INIT_SPACE,
        // !tooltip[/seeds = \[COUNTER_SEED\]/] seeds
        seeds = [COUNTER_SEED],
        // !tooltip[/bump/] bump
        bump
    )]
    // !tooltip[/Account<'info, Counter>/] counterAccountType
    pub counter: Account<'info, Counter>,
    // !tooltip[/Program<'info, System>/] systemProgram
    pub system_program: Program<'info, System>,
}

// !tooltip[/handle_initialize/] handler
pub fn handle_initialize(ctx: Context<Initialize>) -> Result<()> {
    // !tooltip[/counter.count = 0/] initialCount
    ctx.accounts.counter.count = 0;
    // !tooltip[/counter.authority = ctx.accounts.payer.key\(\)/] initialAuthority
    ctx.accounts.counter.authority = ctx.accounts.payer.key();

    // !tooltip[/system_program::Transfer/] cpiAccounts
    let cpi_accounts = anchor_lang::system_program::Transfer {
        from: ctx.accounts.payer.to_account_info(),
        to: ctx.accounts.counter.to_account_info(),
    };
    // !tooltip[/CpiContext::new/] cpiContext
    let cpi_ctx = CpiContext::new(anchor_lang::system_program::ID, cpi_accounts);
    // !tooltip[/system_program::transfer/] transfer
    anchor_lang::system_program::transfer(cpi_ctx, HELLO_WORLD_LAMPORTS)?;

    // !tooltip[/msg!/] programLog
    msg!("Hello, world! Counter initialized");
    Ok(())
}
```

### !imports

이 파일은 공유 상수와 _rs`Counter`_ 계정 타입을 가져옵니다. 명령어는 이러한
정의를 사용하여 PDA를 도출하고, transfer 금액을 결정하며, 올바른 데이터 크기로
계정을 할당합니다.

### !accountsStruct

*rs`Initialize`*는 _rs`initialize`_ 명령어의 계정 컨텍스트입니다. 구조체의 각
필드는 명령어가 호출될 때 포함되어야 하는 개별 계정을 나타냅니다. Anchor는
핸들러가 실행되기 전에 이 구조체를 사용하여 들어오는 계정을 검증합니다.

### !payerMutable

*rs`#[account(mut)]`*는 Anchor에게 payer 계정이 쓰기 가능해야 함을 알립니다.
계정 생성과 transfer CPI 모두 payer로부터 lamport를 차감하기 때문에 이것이
필요합니다.

### !payerSigner

*rs`Signer<'info>`*는 Anchor에게 payer가 트랜잭션에 서명해야 함을 알립니다.

### !counterConstraints

_rs`#[account(...)]`_ 블록은 핸들러가 실행되기 전에 Anchor가 counter 계정을
생성하고 검증하는 방법을 알려줍니다.

### !initConstraint

_rs`init`_ 는 Anchor에게 핸들러가 실행되기 전에 _rs`counter`_ 계정을 생성하도록
지시합니다. 새 계정은 이 프로그램이 소유하므로 이 프로그램이 그 안에
_rs`Counter`_ 데이터를 저장할 수 있습니다.

### !payerConstraint

_rs`payer = payer`_ 는 새 카운터 계정 생성 비용을 지불할 _rs`payer`_ 계정을
지정합니다.

### !space

_rs`space = 8 + Counter::INIT_SPACE`_ 는 Anchor의 계정 판별자와 `state.rs` 의
_rs`Counter`_ 필드를 위한 바이트를 할당합니다. 판별자는 Anchor가 계정을 읽기
전에 해당 계정이 예상된 타입을 저장하고 있는지 확인하는 데 도움을 줍니다.

### !seeds

_rs`seeds = [COUNTER_SEED]`_ 는 카운터 계정 주소가 이 seed와 프로그램 ID로부터
파생된 PDA여야 함을 나타냅니다. 하나의 고정 seed를 사용하면 이 프로그램에 대한
카운터 PDA가 하나만 존재합니다. 실제 프로그램에서 별도의 카운터를 생성하려면
일반적으로 사용자 주소와 같은 다른 seed를 추가해야 합니다.

### !bump

_rs`bump`_ 는 Anchor에게 PDA bump를 찾아 검증하도록 지시합니다. bump는 파생된
주소를 유효한 PDA로 만드는 추가 바이트입니다.

### !counterAccountType

_rs`Account<'info, Counter>`_ 는 Anchor에게 계정 데이터를 `state.rs` 의
_rs`Counter`_ 타입으로 처리하도록 지시합니다. Anchor는 또한 해당 계정이 이
프로그램에 의해 소유되어 있는지도 확인합니다.

### !systemProgram

_rs`Program<'info, System>`_ 는 호출자가 System Program을 제공할 것을
요구합니다. Anchor는 _rs`init`_ 계정 생성 제약 조건에 이를 사용합니다. 핸들러의
전송 CPI 또한 System Program을 호출합니다.

### !handler

핸들러는 Anchor가 계정을 검증하고 카운터 계정을 생성한 후 실행되므로, 계정의
초기 데이터를 안전하게 작성할 수 있습니다.

### !initialCount

카운터 계정의 _rs`count`_ 필드를 _rs`0`_ 으로 설정합니다.

### !initialAuthority

카운터 계정의 _rs`authority`_ 필드를 지불자의 주소로 설정합니다.

### !cpiAccounts

이 계정들은 System Program 전송 명령에 전달되는 계정들입니다. 이 전송 CPI는
CPI가 다른 프로그램에 계정을 전달하는 방식을 보여주기 위한 예시로만
포함되었습니다.

### !cpiContext

*rs`CpiContext`*는 Cross Program Invocation을 위한 대상 프로그램과 필요한
계정들을 묶어줍니다. 컨텍스트를 생성하면 호출하려는 프로그램과 해당 프로그램이
필요로 하는 계정들을 연결하여 CPI를 준비합니다.

### !transfer

이 CPI는 System Program의 전송 명령을 호출하여 *rs`HELLO_WORLD_LAMPORTS`*를
지불자에서 카운터 계정으로 이동시킵니다.

### !programLog

*rs`msg!`*는 프로그램이 호출될 때 프로그램 로그에 메시지를 기록합니다.

### !incrementAccounts

*rs`Increment`*는 _rs`increment`_ 명령의 계정 컨텍스트입니다. Anchor는 이를
사용하여 변경 가능한 카운터 계정과 필요한 서명자를 검증합니다.

### !counterMutable

*rs`mut`*는 Anchor에게 핸들러가 저장된 _rs`count`_ 값을 업데이트하기 때문에
카운터 계정이 쓰기 가능해야 한다고 알려줍니다.

### !counterSeeds

*rs`seeds = [COUNTER_SEED]`*는 제공된 카운터 계정이 이 seed와 프로그램 ID로부터
파생된 PDA인지 검증합니다.

### !counterBump

*rs`bump`*는 Anchor에게 카운터 계정의 PDA 범프를 찾아 검증하도록 지시합니다.

*rs`Account<'info, Counter>`*는 Anchor에게 계정 데이터를 `state.rs`의
_rs`Counter`_ 타입으로 역직렬화하도록 지시합니다. 또한 Anchor는 해당 계정이 이
프로그램에 의해 소유되어 있는지 확인합니다.

### !authoritySigner

*rs`Signer`*는 권한자의 서명을 요구합니다. 그런 다음 핸들러는 이 서명자의 주소를

### !incrementHandler

핸들러는 Anchor가 계정 목록을 검증하고, 소유권을 확인하고, 카운터 계정을
역직렬화한 후에 실행됩니다.

### !authorityCheck

*rs`require_keys_eq!`*는 두 개의 공개 키를 비교하여 서명자의 주소가 카운터
계정에 저장된 권한자와 일치하지 않으면 명령어를 중단합니다. 이것이 이 템플릿의
애플리케이션 수준 인가 규칙입니다.

### !maxCheck

*rs`require!`*는 부울 규칙을 확인하고, 카운터가 이미 *rs`MAX_COUNT`*에 도달한
경우 *rs`CounterOverflow`*와 함께 명령어를 중단합니다. 이 템플릿 규칙은 커스텀
오류 처리와 변경 전 실패 처리를 보여줍니다.

### !mutateCount

### !programLog

*rs`msg!`*는 프로그램이 호출될 때 프로그램 로그에 메시지를 기록합니다.

instruction data를 생성합니다. 여기서 명령어 인수가 인코딩되지만, 이
_rs`increment`_ 명령어는 인수가 필요하지 않습니다.

</SideBySide>

### !litesvm

*rs`LiteSVM`*는 테스트를 위한 인프로세스 Solana VM입니다. validator 프로세스를
시작하지 않고도 컴파일된 프로그램을 등록하고, 계정을 생성하며, 트랜잭션을
전송하고, 계정 상태를 검사할 수 있습니다.

### !setup

테스트는 *rs`my_program::id()`*를 통해 _rs`declare_id!`_ 값에서 프로그램 주소를
가져오고, 테스트 트랜잭션을 위한 새 payer keypair를 생성합니다.

### !counterPda

카운터 주소는 프로그램이 사용하는 동일한 *rs`COUNTER_SEED`*에서 파생됩니다. 이를
통해 테스트가 `initialize.rs`가 예상하는 PDA를 대상으로 합니다.

### !programBytes

*rs`include_bytes!`*는 빌드에서 생성된 컴파일된 `my_program.so` 파일을 읽습니다.
해당 바이트는 LiteSVM이 테스트에서 실행할 실행 가능한 프로그램 코드입니다.

### !addProgram

*rs`svm.add_program`*는 컴파일된 프로그램 바이트를 LiteSVM의 프로그램 주소에
등록합니다. 에어드롭을 통해 지불자는 트랜잭션 및 계정 생성에 필요한 충분한 로컬
SOL을 받습니다.

### !testInstruction

*rs`Instruction::new_with_bytes`*는 프로그램을 호출하는 명령을 생성합니다.
여기에는 프로그램 ID, instruction data, 그리고 명령 계정이 포함됩니다.

### !blockhash

모든 Solana 트랜잭션에는 최근 블록해시가 필요합니다. 이 테스트에서 LiteSVM은
트랜잭션에 사용할 블록해시를 제공합니다.

### !message

메시지에는 명령, 수수료 지불자, 블록해시가 포함됩니다. 이것이 서명될 트랜잭션
데이터입니다.

### !transaction

*rs`VersionedTransaction::try_new`*는 지불자 keypair로 메시지에 서명합니다.
지불자는 트랜잭션 수수료를 지불하고 새 카운터 계정에 자금을 제공하기 때문에
서명합니다.

### !sendTx

*rs`svm.send_transaction`*는 LiteSVM에서 트랜잭션을 실행합니다.

### !getCounter

트랜잭션이 실행된 후, 테스트는 계정 데이터를 검사하기 위해 LiteSVM에서 직접
카운터 계정을 읽습니다.

### !deserialize

카운터 계정은 직렬화된 데이터를 저장합니다. *rs`Counter::try_deserialize`*는
Anchor의 계정 판별자를 확인하고 계정 데이터를 `state.rs`의 _rs`Counter`_
타입으로 다시 변환합니다.

### !initAssert

*rs`initialize`*가 성공한 후, 테스트는 카운터 계정을 역직렬화하고 초기 상태가
*rs`handle_initialize`*가 기록한 내용과 일치하는지 확인합니다.

### !incrementAssert

*rs`increment`*가 성공한 후, 동일한 카운터 계정의 *rs`count`*가 *rs`1`*로
설정되고 동일한 권한이 유지되어야 합니다.

</WithNotes>

## 프로젝트 구성

루트 프로젝트 파일은 Anchor와 Cargo에게 프로그램을 빌드, 테스트 및 배포하는
방법을 알려줍니다. 전체 참조 문서는
[Anchor.toml 구성](https://www.anchor-lang.com/docs/references/anchor-toml) 및
[Anchor CLI](https://www.anchor-lang.com/docs/references/cli)에 대한 Anchor
문서를 참조하세요.

<WithNotes>

```toml title="Anchor.toml"
# !tooltip[/skip_local_validator/] skipValidator
skip_local_validator = true

[toolchain]

[features]
resolution = true
skip-lint = false

# !tooltip[/programs.localnet/] programs
[programs.localnet]
my_program = "82sFkffP9wxwpyfZyeaKHH2chQoJPUGsJZSPi9mrUuXd"

# !tooltip[/provider/] provider
[provider]
cluster = "localnet"
wallet = "~/.config/solana/id.json"

# !tooltip[/scripts/] scripts

[hooks]
```

### !skipValidator

`skip_local_validator = true`는 Anchor에게 `anchor test`에 대해 validator를
시작하지 않도록 지시합니다. Rust 테스트는 대신 LiteSVM을 사용합니다.

### !programs

`[programs.localnet]`는 Rust 크레이트 이름을 로컬 빌드 및 테스트에 사용되는
프로그램 주소에 매핑합니다.

### !provider

`[provider]`는 `anchor deploy`와 같은 명령어에 사용할 기본 클러스터와 지갑을
설정합니다.

### !scripts

`[scripts]`는 `anchor test`의 실행 내용을 변경합니다. 이 템플릿에서
`anchor test`는 `cargo test`를 실행합니다.

</WithNotes>
