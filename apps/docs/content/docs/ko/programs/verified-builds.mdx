이 가이드는 솔라나에서 자신의 프로그램에 대한 검증된 빌드를 구현하고자 하는
개발자를 위한 참고 자료입니다. 검증된 빌드가 무엇인지, 어떻게 사용하는지, 특별한
고려사항 및 온체인 프로그램의 신뢰성을 보장하기 위한 모범 사례를 다룰 것입니다.

# 검증된 빌드란 무엇인가요?

검증된 빌드는 솔라나 네트워크에 배포하는 실행 가능한 프로그램이 저장소의 소스
코드와 일치하는지 확인합니다. 이를 통해 개발자와 사용자는 온체인에서 실행되는
프로그램이 공개 코드베이스와 정확히 일치한다는 확신을 가질 수 있어 투명성과
보안을 촉진합니다.

검증 과정은 온체인 프로그램의 해시와 소스 코드에서 로컬로 빌드된 프로그램의
해시를 비교하는 것을 포함합니다. 이는 두 버전 간에 불일치가 없음을 보장합니다.

> 검증된 빌드가 검증되지 않은 빌드보다 더 안전하다고 간주되어서는 안 되지만, 이
> 빌드는 개발자가 소스 코드가 온체인에 배포된 것과 일치하는지 스스로 확인할 수
> 있게 합니다. 소스 코드를 사용하여 개발자는 트랜잭션을 보낼 때 코드가 실행하는
> 내용을 검증할 수 있습니다.

검증된 빌드 파이프라인은 [Ellipsis Labs](https://ellipsislabs.xyz/)와
[OtterSec](https://osec.io/)에 의해 고안되고 유지되고 있습니다. 자세한 내용은
[원본 검증된 빌드](https://github.com/Ellipsis-Labs/solana-verifiable-build)
저장소의 가이드를 따르고, 지원되면 [Anza](https://www.anza.xyz/) 도구 모음에
직접 빌드 프로세스를 검증하세요.

# 어떻게 작동하나요?

검증 프로세스는 온체인 프로그램의 해시와 소스 코드에서 로컬로 빌드된 프로그램의
해시를 비교하여 수행됩니다. Solana Verify CLI와 Docker를 사용하여 제어된
환경에서 프로그램을 빌드합니다. 이를 통해 빌드 프로세스가 다양한 시스템에서
결정적이고 일관되게 유지됩니다. 실행 파일을 얻은 후에는 Solana 네트워크에 배포할
수 있습니다. 빌드 프로세스 중에
[verify 프로그램](https://github.com/otter-sec/otter-verify)의
[PDA](https://explorer.solana.com/address/63XDCHrwZu3mXsw2RUFb4tbNpChuUHx4eA5aJMnHkpQQ/anchor-account)가
생성됩니다. 이 PDA에는 프로그램을 검증하는 데 필요한 모든 데이터가 포함되어
있습니다. PDA에는 프로그램 주소, git URL, 커밋 해시 및 프로그램을 빌드하는 데
사용된 인수가 포함됩니다.

PDA의 데이터를 사용하여 누구나 로컬에서 verify 프로그램 명령을 실행하고
프로그램이 제공된 소스 코드에서 빌드되었는지 확인할 수 있습니다. 그런 다음 모든
사람이 완전히 신뢰 없이 직접 검증하거나
[OtterSec](https://github.com/otter-sec)에서 유지 관리하는 자체
[verify API](https://github.com/otter-sec/solana-verified-programs-api)를
실행하여 사용자가 검증을 쉽게 확인할 수 있는 접근점을 제공할 수 있습니다. 이미
[Solana Explorer](https://explorer.solana.com/address/E1fcPKuV1UJEsTJPpab2Jr8y87ZN73i4CHTPLbWQE6CA/verified-build)와
[SolanaFM](https://solana.fm/address/E1fcPKuV1UJEsTJPpab2Jr8y87ZN73i4CHTPLbWQE6CA/transactions?cluster=mainnet-alpha)
등 여러 곳에서 이러한
[API 호출](https://verify.osec.io/status/PhoeNiXZ8ByJGLkxNfZRnkUfjvmuYqLR89jjFHGqdXY)이
사용되는 것을 볼 수 있습니다.

# 왜 검증된 빌드를 사용해야 하나요?

검증된 빌드를 사용하면 다음과 같은 이점이 있습니다:

- 보안: 온체인에서 실행 중인 프로그램이 소스 코드와 일치하는지 보장하여 악의적인
  변경을 방지합니다.

- 투명성: 다른 사용자와 개발자가 온체인 프로그램을 공개 코드베이스와 비교하여
  신뢰할 수 있는지 검증할 수 있습니다.

- 신뢰: 검증된 빌드는 프로그램의 온체인 동작이 공개 코드와 일치함을 보여주므로
  사용자 신뢰도를 높입니다. 검증 가능한 프로그램을 구축할 때 승인되지 않거나
  악의적인 코드 실행과 관련된 위험을 최소화합니다. 또한 모범 사례를 준수하고
  보안 연구원이 쉽게 연락할 수 있는 방법을 제공합니다. 또한 지갑 및 기타 도구는
  프로그램이 검증되는 한 해당 프로그램의 트랜잭션을 더 쉽게 허용할 수 있습니다.

- 발견 가능성: 프로그램의 검증된 빌드를 제공하면 모든 사람이 소스 코드, 문서,
  프로그램 SDK 또는 IDL을 찾을 수 있으며, 문제가 있을 경우 GitHub을 통해 쉽게
  연락할 수 있습니다.

# 검증된 빌드는 어떻게 만드나요?

검증된 빌드를 만들려면 다음 단계를 따라야 합니다:

요약:

- 코드를 공개 저장소에 커밋하기
- Docker에서 검증된 빌드 만들기
- 검증된 빌드 배포하기
- 배포된 프로그램을 공개 API와 대조하여 검증하기

Docker 컨테이너에서 빌드되지 않은 프로그램을 검증하려고 하면 대부분 실패할
것입니다. 이는 Solana 프로그램 빌드가 서로 다른 시스템에서 결정적이지 않기
때문입니다.

<Steps>
<Step>

### Docker와 Cargo 설치하기

필요한 도구를 설치하고 Docker와 Cargo가 설치되어 있는지 확인하세요. Docker는
일관성을 보장하기 위한 제어된 빌드 환경을 제공하며, Cargo는 Rust 패키지 관리에
사용됩니다.

- Docker: [Docker 웹사이트](https://docs.docker.com/engine/install/)에서
  플랫폼에 맞는 Docker 설치 단계를 따르세요. 설치 후, 이 가이드를 계속 따라
  Docker 서비스가 실행 중인지 확인하세요.
- Cargo: 아직 Cargo가 설치되어 있지 않다면, 다음 명령어를 실행하여 설치할 수
  있습니다:

```bash
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh
```

</Step>
<Step>

### Solana Verify CLI 설치하기

Solana Verify CLI는 빌드를 검증하는 데 사용되는 주요 도구입니다. Solana Verify
CLI는 현재 [Ellipsis Labs](https://ellipsislabs.xyz/)에서 유지 관리하고 있으며
Cargo를 사용하여 설치할 수 있습니다.

다음 명령어를 실행하여 설치할 수 있습니다:

```bash
cargo install solana-verify
```

특정 버전의 CLI가 필요한 경우, 다음과 같이 버전을 지정할 수 있습니다:

```bash
cargo install solana-verify --version $VERSION
```

원하는 경우, 특정 커밋에서 직접 버전을 설치할 수 있습니다:

```bash
cargo install solana-verify --git https://github.com/Ellipsis-Labs/solana-verifiable-build --rev 13a1db2
```

</Step>
<Step>

### 프로젝트 준비

저장소에 대해 검증하려면 저장소의 루트 디렉토리에 `Cargo.lock` 파일이 있어야
합니다. 저장소에 프로그램이 하나만 있고 루트에 `cargo.lock` 파일이 있는 경우
다음 단계로 바로 이동하여 프로그램을 빌드할 수 있습니다.

프로그램이 하위 폴더에 있고 rust 워크스페이스가 있는 경우 저장소의 루트
디렉토리에 워크스페이스 `Cargo.toml` 파일을 생성해야 합니다.

이 `Cargo.toml` 예제를 프리셋으로 사용할 수 있습니다:

```toml title="Cargo.toml"
[workspace]
members = ["program/programs/*"]
resolver = "2"

[profile.release]
overflow-checks = true
lto = "fat"
codegen-units = 1

[profile.release.build-override]
opt-level = 3
incremental = false
codegen-units = 1
```

프로그램이 `workspace/members` 배열에 있고 프로그램의 `Cargo.toml`에 올바른
`lib` 이름이 구성되어 있는지 확인하세요.

> 중요한 것은 `lib name`이지 패키지 이름이 아닙니다!

다음과 같이 구성됩니다:

```toml title="waffle/Cargo.toml"
[package]
name = "waffle"
version = "0.1.0"
edition = "2021"

[lib]
name = "waffle"
crate-type = ["cdylib", "lib"]

[dependencies]
solana-program = "2.1.0"
```

이 [저장소](https://github.com/solana-developers/verified-program)에서 하위
폴더에 프로그램이 있는 워크스페이스의 예제를 볼 수 있습니다. 또한 프로그램이
하위 폴더에 있는 경우 나중에 이 폴더를 `verify-from-repo` 명령에
`--mount-path`로 추가해야 한다는 점에 유의하세요.

이 [저장소](https://github.com/solana-developers/solana-game-preset)에서 Anchor
예제를 찾을 수 있습니다. 이
[저장소](https://github.com/solana-developers/verified-program-root)에서는
네이티브 러스트 예제를 찾을 수 있습니다.

이 `Cargo.toml` 파일이 준비되면 `cargo generate-lockfile`를 실행하여 락 파일을
생성하고 프로그램 빌드를 계속할 수 있습니다.

</Step>
<Step>

### 검증 가능한 프로그램 빌드하기

Solana 프로그램을 검증 가능하게 빌드하려면 워크스페이스의 `Cargo.toml` 파일이
있는 디렉토리로 이동하여 다음을 실행하세요:

```bash
solana-verify build
```

이렇게 하면 환경이 도커 컨테이너로 복사되어 결정적인 방식으로 빌드됩니다.

> 실제로 검증된 빌드를 배포하고 `anchor build` 또는 `cargo build-sbf`로 실수로
> 덮어쓰지 않도록 주의하세요. 이러한 명령은 동일한 해시를 생성하지 않을 가능성이
> 높아 검증이 실패할 수 있습니다.

여러 프로그램이 있는 프로젝트의 경우, 라이브러리 이름(패키지 이름이 아님)을
사용하여 특정 프로그램을 빌드할 수 있습니다:

```bash
solana-verify build --library-name $PROGRAM_LIB_NAME
```

이 과정은 결정적 빌드를 보장하며, 특히 특정 시스템(예: M1 맥북)에서는 도커
컨테이너 내에서 실행되기 때문에 시간이 걸릴 수 있습니다. 더 빠른 빌드를 위해서는
x86 아키텍처를 실행하는 리눅스 머신을 사용하는 것이 좋습니다.

빌드가 완료되면 다음 명령어를 사용하여 실행 파일의 해시를 검색할 수 있습니다:

```bash
solana-verify get-executable-hash target/deploy/$PROGRAM_LIB_NAME.so
```

</Step>
<Step>

### 검증 가능한 프로그램 배포하기

프로그램을 빌드하고 해시를 검색한 후에는 Solana 네트워크에 배포할 수 있습니다.
안전한 배포를 위해 [Squads Protocol](https://squads.so/protocol)과 같은 다중
서명 또는 거버넌스 솔루션을 사용하는 것이 좋지만, 다음과 같이 직접 배포할 수도
있습니다:

```bash
solana program deploy -u $NETWORK_URL target/deploy/$PROGRAM_LIB_NAME.so --program-id $PROGRAM_ID --with-compute-unit-price 50000 --max-sign-attempts 100 --use-rpc
```

현재 적합한 낮은 우선순위 수수료는
[Quicknode](https://www.quicknode.com/gas-tracker/solana)와 같은 RPC
제공업체에서 요청할 수 있습니다.

배포된 프로그램이 빌드된 실행 파일과 일치하는지 확인하려면 다음을 실행하세요:

```bash
solana-verify get-program-hash -u $NETWORK_URL $PROGRAM_ID
```

> 다른 [Solana 클러스터](/docs/references/clusters)(예: devnet, testnet,
> mainnet)에 다른 버전이 배포되어 있을 수 있습니다. 프로그램을 검증하려는 Solana
> 클러스터에 맞는 올바른 네트워크 URL을 사용하세요. 원격 검증은 메인넷에서만
> 작동합니다.

</Step>
<Step>

### 저장소에 대한 검증

공개 저장소에 대해 프로그램을 검증하려면 다음을 사용하세요:

```bash
solana-verify verify-from-repo -u $NETWORK_URL --program-id $PROGRAM_ID https://github.com/$REPO_PATH --commit-hash $COMMIT_HASH --library-name $PROGRAM_LIB_NAME --mount-path $MOUNT_PATH
```

> 프로그램 디렉토리에서 검증된 빌드를 실행하는 동안 `verify-from-repo`를 실행할
> 때 `--mount-path` 플래그를 추가해야 합니다. 이는 프로그램의 라이브러리 이름이
> 포함된 `Cargo.toml`가 있는 폴더의 경로입니다.

이 명령은 온체인 프로그램 해시와 지정된 커밋 해시의 소스에서 빌드된 실행 파일
해시를 비교합니다.

명령 마지막에 검증 데이터를 온체인에 업로드할지 물어볼 것입니다. 업로드하면
Solana 익스플로러에 즉시 프로그램의 검증 데이터가 표시됩니다. 원격 빌드로 검증될
때까지는 미검증 상태로 표시됩니다. 다음 단계에서 공개 API를 통해 프로그램을
검증하는 방법을 알아보세요.

특정 릴리스로 검증을 고정하려면 명령에 `--commit-hash` 플래그를 추가할 수
있습니다.

</Step>
<Step>

### 공개 API를 통한 검증

`verify-from-repo`로 검증 PDA를 온체인에 업로드한 후,
[OtterSec API](https://github.com/otter-sec/solana-verified-programs-api)에 원격
검증 작업을 제출하세요:

```bash
solana-verify remote submit-job --program-id <program-id> --uploader <address>
```

`--uploader`는 검증 PDA를 업로드한 주소로, 일반적으로 프로그램의 업그레이드
권한입니다. 프로그램이 멀티시그로 제어되는 경우, 이 가이드 아래의
[멀티시그 검증](#how-to-verify-your-program-when-its-controlled-by-a-multisig-like-squads)
섹션을 참조하세요.

> `verify-from-repo`의 레거시 `--remote` 플래그는 더 이상 사용되지 않습니다.
> 먼저 PDA를 업로드한 후 `remote submit-job`를 실행하세요.

이 명령은 OtterSec API에 작업을 제출합니다. 다음 명령으로 작업 상태를 확인할 수
있습니다:

```bash
solana-verify remote get-job --job-id <job-id>
```

검증이 성공적으로 완료되면(다소 시간이 걸릴 수 있음),
[단일 프로그램용 OtterSec API](https://verify.osec.io/status/PhoeNiXZ8ByJGLkxNfZRnkUfjvmuYqLR89jjFHGqdXY),
[Solana Explorer](https://explorer.solana.com/address/PhoeNiXZ8ByJGLkxNfZRnkUfjvmuYqLR89jjFHGqdXY/verified-build),

## 멀티시그(예: Squads)로 제어되는 프로그램을 검증하는 방법

원격 검증이 작동하려면 프로그램 권한이 서명한 PDA에 검증 데이터를 기록해야
합니다. 프로그램이 멀티시그로 제어되는 경우, 이 PDA 쓰기 트랜잭션을 내보내어
[Squads Protocol](https://squads.so/protocol) 또는 원하는 다른 멀티시그 솔루션을
통해 제출할 수 있습니다.

<Steps>
<Step>

### 1. 검증 가능한 프로그램 빌드

먼저 프로그램을 빌드하세요:

```bash

```

이 명령은 `Cargo.lock` 파일에 지정된 Solana 버전을 사용하는 Docker 컨테이너로
검증 가능한 빌드를 생성합니다.

</Step>
<Step>

### 2. 프로그램 배포

```bash

```

`6XBGfP17P3KQAKoJb2s5M5fR4aFTXzPeuC1af2GYkvhD`를 예시 프로그램 ID로 사용합니다.

</Step>
<Step>

https://github.com/solana-developers/verify-squads

선택 사항: 먼저 로컬에서 검증할 수 있는지 확인해 보세요 (이 명령은 예시 프로그램
ID `6XBGfP17P3KQAKoJb2s5M5fR4aFTXzPeuC1af2GYkvhD`를 사용합니다):

## 검증된 빌드 예시

다음은 이 [저장소](https://github.com/solana-developers/verified-program)의 소스

프로그램을 검증하는 예시입니다:


저장소에서 계속 작업하려는 경우 `commit-hash` 매개변수를 사용하여 특정 커밋을
지정할 수도 있습니다: `--commit-hash 5b82b86f02afbde330dff3e1847bed2d42069f4e`

프롬프트가 표시되면 검증 PDA를 온체인에 업로드하기 위해 yes를 입력하세요. 그런
다음 OtterSec API에 원격 검증 작업을 제출합니다:

```bash
solana-verify remote submit-job --program-id FWEYpBAf9WsemQiNbAewhyESfR38GBBHLrCaU3MpEKWv --uploader <your-upgrade-authority>
```

# 자주 묻는 질문

## 검증이 실패합니다. 어떻게 해야 하나요?

다음과 같은 일반적인 문제를 확인하세요:

- **잘못된 서명자**: `solana program show YourProgramId`를 실행하여 서명자가
  프로그램의 업그레이드 권한자인지 확인하세요.
- **온체인 PDA 없음**: `solana-verify verify-from-repo -um`를 실행하고
  **프롬프트가 표시될 때 YES를 선택하세요**. PDA를 업로드하지 않으면 API가 검증
  메타데이터를 가져올 수 없습니다.
- **PDA 데이터 불일치**: 프로그램을 재배포한 경우 PDA를 업데이트하세요. PDA
  데이터는 배포된 프로그램과 일치해야 합니다.
- **잘못된 커밋 해시**: 배포 시 사용한 정확한 커밋 해시로 PDA를 생성하세요.
- **빌드 환경 차이**: PDA 생성 시 solana-verify와 함께 Docker를 사용하세요.

## 로컬 빌드 해시가 온체인 해시와 일치하지 않습니다. 왜 그런가요?

보통 다음과 같은 이유 때문입니다:

- 다른 Rust/Solana 툴체인 버전을 사용하고 있습니다.
- 빌드 사이에 의존성이 업데이트되었습니다.
- Docker 컨테이너 내에서 빌드하지 않았습니다.
- 잘못된 커밋을 체크아웃했습니다.

Docker에서 `solana-verify build`를 사용하여 배포한 정확한 커밋으로 빌드하면 이
문제를 해결할 수 있습니다.

## 검증은 얼마나 걸리나요?

프로그램 크기에 따라 예상 소요 시간은 다음과 같습니다:

- 간단한 프로그램: 1~5분
- 복잡한 프로그램: 5~15분
- 매우 큰 프로그램: 최대 30분

작업 상태 엔드포인트를 사용하여 진행 상황을 추적하세요.

## 프로그램이 불변(업그레이드 권한 없음) 상태입니다. 어떻게 검증할 수 있나요?

프로그램에 업그레이드 권한이 없거나 PDA를 생성하기 전에 불변 상태로 설정된 경우,
이 상황을 위한 화이트리스트 주소가 있습니다. contact@osec.io로 문의해 주시면
프로그램 검증을 도와드리겠습니다.

## PDA란 무엇이며 왜 중요한가요?

귀하의 PDA(프로그램 파생 계정)는 신뢰 없는 검증을 가능하게 합니다:

- **온체인 스토리지**: 검증 메타데이터(저장소 URL, 커밋 해시, 빌드 매개변수)를
  Otter Verify program (`verifycLy8mB96wd9wqq3WDXQwM4oU6r42Th37Db9fC`)이 소유한
  PDA에 온체인으로 저장합니다
- **암호화 연결**: 귀하의 PDA는 프로그램 주소에서 파생되어, 검증 데이터에 대한
  불변 링크를 생성합니다
- **탈중앙화 신뢰**: 누구든지 귀하의 PDA를 읽고 독립적으로 프로그램을 검증할 수
  있습니다

## API를 사용하기 전에 반드시 PDA를 생성해야 하는 이유는 무엇인가요?

API는 온체인 PDA에서만 작동합니다. 그 이유는 다음과 같습니다:

- **신뢰 불필요**: API는 임의의 데이터를 거부하며, 업그레이드 권한자가 온체인에
  저장한 데이터만 사용합니다
- **간편함**: 서명자와 program_id만 제공하면 API가 나머지 모든 정보를 귀하의
  PDA에서 가져옵니다
- **변조 불가**: 귀하의 PDA는 누구든지 독립적으로 검증할 수 있는 불변 기록을
  생성합니다
- **소유권 증명**: 서명자가 업그레이드 권한자여야 하므로, 프로그램 제어권을
  암호학적으로 증명합니다

## 프로그램을 얼마나 자주 검증해야 하나요?

다음과 같은 경우에 프로그램을 검증하세요:

- 배포 또는 업그레이드 후 매번
- 소스 저장소를 업데이트할 때
- 그 외의 경우에는 재검증에 대해 걱정하지 않아도 됩니다 - API가 자동으로 모든
  프로그램을 24시간마다 재검증합니다

## 프로그램을 업그레이드하면 어떻게 되나요?

업그레이드 후 다음 단계를 따르세요:

1. API가 업그레이드를 감지하고 프로그램 검증을 해제합니다.
2. 새로운 검증 메타데이터로 PDA를 업데이트합니다:

```bash
solana-verify verify-from-repo -um \
  --program-id YourProgramId... \
  https://github.com/your-org/your-program
```

3. 업그레이드 권한자를 사용하여 새로운 검증 요청을 제출합니다
4. API가 업데이트된 PDA를 기준으로 새 버전을 검증합니다

**중요**: 업그레이드된 프로그램의 새 커밋 해시와 함께 항상 업그레이드 권한자를
사용하여 PDA를 업데이트하세요.

## 검증 결과를 신뢰할 수 있나요?

네 - 이 시스템은 신뢰가 필요 없으며 독립적으로 검증 가능하도록 설계되었습니다:

**신뢰할 수 있는 이유:**

- **온체인 PDA**: 검증 메타데이터는 어떤 중앙 기관의 통제도 받지 않고 온체인에
  저장됩니다
- **업그레이드 권한 증명**: 프로그램의 업그레이드 권한 보유자만 PDA를 생성하거나
  업데이트할 수 있습니다
- **독립적 검증**: PDA를 읽고 로컬에서 `solana-verify`를 실행하면 누구든지
  검증할 수 있습니다
- **지속적 재검증**: API는 모든 프로그램을 24시간마다 자동으로 재검증합니다

**다음 한계점을 이해하세요:**

- 검증은 소스가 배포본과 일치함을 확인하는 것이지, 코드가 **안전하다는 것을
  보장하지 않습니다**
- 프로그램과 상호작용하기 전에 항상 코드를 검토하세요
- 검증됨 ≠ 감사됨 또는 안전함
- PDA의 리포지토리와 커밋을 확인하여 신뢰할 수 있는 출처인지 확인하세요

## 프로그램을 독립적으로 어떻게 검증할 수 있나요?

온체인 PDA를 읽고 로컬에서 검증을 실행하여 직접 모든 프로그램을 검증할 수
있습니다:

**1단계: 온체인 PDA 읽기**

```bash
# Install solana-verify if you haven't
cargo install solana-verify

# Get the PDA data
solana-verify list-program-pdas --program-id YourProgramId...
```

**2단계: 로컬에서 검증**

```bash
# Verify using the repository and commit & other arguments from the PDA
solana-verify verify-from-repo \
  --program-id YourProgramId... \
  https://github.com/your-org/your-program
  --commit-hash <commit-hash>
  ... (other arguments from the PDA)
# Confirm the hash output matches the onchain program hash
```

**이를 통해 다음이 증명됩니다:**

1. PDA 메타데이터가 진본임을 확인합니다 (온체인에 저장됨)
2. PDA 리포지토리의 소스 코드가 배포된 프로그램과 일치합니다
3. API를 신뢰할 필요 없이 온체인에서 모든 것을 직접 검증할 수 있습니다

## 다른 사람이 허가 없이 내 프로그램을 검증할 수 있나요?

네, 그렇기 때문에 서명자가 업그레이드 권한 보유자여야 합니다. 서명자가
업그레이드 권한 보유자인 경우에만 검증이 유효한 것으로 간주합니다.

## 검증 가능한 빌드를 생성하려면 무엇이 필요한가요?

다음 도구들을 설치하세요:

- Docker (결정론적 빌드를 위한)
- Cargo (Rust 패키지 매니저)
- Solana Verify CLI: `cargo install solana-verify`
- 소스 코드가 있는 공개 Git 리포지토리

## 비공개 저장소도 검증할 수 있나요?

아니요 - 검증을 위해서는 공개 소스 코드가 필요합니다:

- PDA에는 누구나 접근할 수 있는 공개 저장소 URL이 저장됩니다
- 신뢰 없는 검증은 공개 코드 접근에 의존합니다
- 사용자는 프로그램이 무엇을 하는지 이해하기 위해 소스 코드를 읽어야 합니다
- 이 시스템의 핵심 목적은 사용자가 소스 코드와 배포본이 일치하는지 독립적으로
  검증할 수 있도록 하는 것입니다

비공개 저장소는 검증 시스템의 핵심 신뢰 모델을 무너뜨립니다.

## Squads Multisig으로 제어되는 프로그램을 어떻게 검증하나요?

멀티시그로 제어되는 프로그램의 경우 다음 단계를 따르세요:

```bash
# 1. Build and deploy normally
solana-verify build
solana program deploy <your-program.so> --program-id YourProgramId...

# 2. Verify locally first - confirm the hash matches
solana-verify verify-from-repo -um \
  --program-id YourProgramId... \
  https://github.com/your-org/your-program

# 3. Export the PDA creation transaction
solana-verify export-pda-tx \
  --program-id YourProgramId... \
  https://github.com/your-org/your-program

# 4. Execute the PDA transaction through your Squads Multisig interface

# 5. After multisig execution, trigger remote verification
solana-verify remote submit-job \
  --program-id YourProgramId... \
  --uploader YourMultisigAddress...
```

**중요**: PDA 트랜잭션을 내보내기 전에 반드시 로컬에서 검증(2단계)하여 빌드
해시가 일치하는지 확인하세요.

# 결론

[Solana의 검증된 빌드](/docs/programs/verified-builds)를 사용하면 네트워크상의
프로그램 무결성과 신뢰성을 보장하고 개발자가 Solana Explorer에서 직접 SDK를 찾을
수 있습니다. Solana Verify CLI 및 Docker와 같은 도구를 활용하여 소스 코드와
일치하는 검증 가능하고 안전한 빌드를 유지할 수 있습니다. 항상 일관된 환경을
사용하기 위한 필요한 주의 조치를 취하고, 안전한 업그레이드 및 배포를 위한
거버넌스 솔루션을 고려하세요.

## 보안 및 면책 조항

검증된 빌드는 Solana 프로그램의 무결성을 보장하는 강력한 도구이지만, 기본
설정에서는 완전히 신뢰 없는 방식이 아닙니다. Docker 이미지는 Solana
Foundation에서 빌드하고 호스팅합니다.

다운로드한 Docker 이미지 내에서 프로젝트를 빌드한다는 점과, 빌드 과정에서 민감한
정보를 포함한 전체 설정이 해당 Docker 이미지로 복사된다는 점에 유의하세요.

완전히 신뢰 없는 설정을 원한다면 Docker 이미지를 직접 빌드하여 자체 인프라에
호스팅할 수 있습니다. 이렇게 하면 Docker 이미지가 변조되지 않았음을 확인할 수
있습니다. 자체 Docker 이미지를 만들기 위한 스크립트는
[Verified builds 저장소](https://github.com/Ellipsis-Labs/solana-verifiable-build)에서
찾을 수 있으며, 포크하여 GitHub Actions를 직접 실행하거나 올바른지 검증할 수
있습니다.

또한 원격 검증의 경우 OtterSec API와
[Solana Explorer](https://explorer.solana.com/address/PhoeNiXZ8ByJGLkxNfZRnkUfjvmuYqLR89jjFHGqdXY)를
어느 정도 신뢰해야 합니다.

API 또는 Solana Explorer가 손상된 경우 잘못된 정보를 표시할 수 있습니다.

완전히 신뢰가 필요 없는 환경을 구축하려면
[Verify API](https://github.com/otter-sec/solana-verified-programs-api)를 직접
실행하거나, 배포 권한과
[verify program](https://explorer.solana.com/address/verifycLy8mB96wd9wqq3WDXQwM4oU6r42Th37Db9fC)에서
파생된
[PDA](https://explorer.solana.com/address/63XDCHrwZu3mXsw2RUFb4tbNpChuUHx4eA5aJMnHkpQQ/anchor-account)에
저장된 온체인 검증 데이터를 사용하여 `verify-from-repo` 명령으로 프로그램 검증을
로컬에서 직접 실행할 수 있습니다.

verify program은 [OtterSec 팀](https://osec.io/)이 배포했으며 아직 동결되지 않아
언제든지 업그레이드될 수 있습니다.

Solana Foundation, OtterSec 및 Ellipsis Labs 팀은 검증된 빌드 파이프라인
사용으로 인해 발생할 수 있는 손실이나 손해에 대해 책임을 지지 않습니다.

# Solana 프로그램을 위한 Security.txt

검증된 빌드 외에도 프로그램에 `security.txt` 파일을 추가할 수 있습니다. 향후
구현되면 `security.txt`는 검증 PDA에 저장된 검증 데이터에 쉽게 접근할 수 있도록
검증자 공개 키를 보유하게 됩니다. 프로그램을 빌드하고 검증하는 데 필요한 모든
정보를 포함하는 PDA는 프로그램 주소와 검증자 pubkey에서 파생됩니다. 기본적으로
이는 프로그램을 빌드하고 배포한 것과 동일한 pubkey입니다. 하지만
`security.txt`에서 지정할 수 있는 다른 pubkey가 될 수도 있습니다.

`security.txt` 기능을 통해 개발자는 Solana 스마트 컨트랙트 내에 연락처 및 보안
정보를 직접 삽입할 수 있습니다. [securitytxt.org](https://securitytxt.org)에서
영감을 받은 이 접근 방식은 보안 연구원들이 컨트랙트 주소만 알고 있는 경우에도
프로젝트 관리자에게 연락할 수 있는 표준화된 방법을 제공합니다.

## security.txt를 사용하는 이유

많은 프로젝트, 특히 소규모 또는 비공개 프로젝트의 경우 컨트랙트 주소만으로
개발자를 식별하는 것은 어렵고 시간이 많이 걸릴 수 있습니다. 프로그램 내에
`security.txt` 파일을 삽입하면 보안 연구자들이 담당자에게 쉽게 연락할 수 있어
잠재적인 익스플로잇을 예방하고 신속한 버그 보고가 가능해집니다.

## security.txt 구현 방법

Solana 프로그램에 `security.txt`를 추가하려면 다음 단계를 따르세요:

`Cargo.toml`에 `solana-security-txt` 의존성을 추가하세요:

```toml title="Cargo.toml"
[dependencies]
solana-security-txt = "1.1.1"
```

컨트랙트에서 `security_txt!` 매크로를 사용하여 보안 정보를 정의하세요. 연락처,
프로젝트 URL, 보안 정책 등을 포함할 수 있습니다. 예시는 다음과 같습니다:

```rust
#[cfg(not(feature = "no-entrypoint"))]
use {default_env::default_env, solana_security_txt::security_txt};

#[cfg(not(feature = "no-entrypoint"))]
security_txt! {
    name: "MyProject",
    project_url: "https://myproject.com",
    contacts: "email:security@myproject.com,discord:security#1234",
    policy: "https://myproject.com/security-policy",

    // Optional Fields
    preferred_languages: "en,de",
    source_code: "https://github.com/solana-developers/solana-game-preset",
    source_revision: "5vJwnLeyjV8uNJSp1zn7VLW8GwiQbcsQbGaVSwRmkE4r",
    source_release: "",
    encryption: "",
    auditors: "Verifier pubkey: 5vJwnLeyjV8uNJSp1zn7VLW8GwiQbcsQbGaVSwRmkE4r",
    acknowledgements: "Thank you to our bug bounty hunters!"
}
```

`security.txt` 정보가 프로그램에 삽입되면 Solana Explorer와 같은 도구를 통해
쉽게 조회할 수 있으며, 잠재적인 문제를 보고하려는 누구에게나 연락처 및 보안
정보가 제공됩니다.

## 모범 사례

- 링크 활용: 변경될 가능성이 있는 정보(예: 연락처)의 경우, 컨트랙트에 직접
  하드코딩하는 대신 웹 페이지 링크를 사용하는 것이 권장됩니다. 이렇게 하면 잦은
  프로그램 업그레이드를 피할 수 있습니다.

- 검증: 배포 전에 `query-security-txt` 도구를 사용하여 형식과 내용을 검증하세요.
  이 도구는 온체인 프로그램과 로컬 바이너리 모두 검증할 수 있습니다:

```bash
query-security-txt target/bpfel-unknown-unknown/release/my_contract.so
```

보안 연락처 정보를 컨트랙트에 직접 삽입함으로써 연구자들이 더 쉽게 연락할 수
있게 되어 Solana 생태계 내에서 더 나은 보안과 커뮤니케이션을 도모할 수 있습니다.

다음은
[Solana Explorer에서 security.txt가 표시되는 예시](https://explorer.solana.com/address/HPxKXnBN4vJ8RjpdqDCU7gvNQHeeyGnSviYTJ4fBrDt4/security?cluster=devnet)입니다.

`security.txt` 프로젝트는 [Neodyme Labs](https://github.com/neodyme-labs)에서
관리합니다.

[verify.osec.io](https://verify.osec.io)에서 검증 상태를 확인하고 검증된
프로그램을 탐색할 수 있습니다.
