Se provieni da `@solana/web3.js` (v1), `@solana/wallet-adapter-*`, o dai
precedenti pacchetti framework-kit (`@solana/client`, `@solana/react-hooks`,
`@solana/web3-compat`), questi approcci sono stati superati. Le nuove app
vengono costruite direttamente su `@solana/kit` e i plugin kit. Questa pagina
mappa i concetti che già conosci sui loro equivalenti in Kit.

## Mappatura dei concetti

| web3.js v1 / wallet-adapter            | Kit                                                                       |
| -------------------------------------- | ------------------------------------------------------------------------- |
| `new Connection(url)`                  | `createSolanaRpc(url)`, oppure `.use(solanaRpc({ rpcUrl }))` su un client |
| `PublicKey`                            | `Address` — converti una stringa con `address("...")`                     |
| `Keypair`                              | `generateKeyPairSigner()` o i plugin `@solana/kit-plugin-signer`          |
| `SystemProgram.transfer(...)`          | `getTransferSolInstruction(...)` da `@solana-program/system`              |
| `LAMPORTS_PER_SOL` math                | `lamports(...)` da `@solana/kit`                                          |
| `Transaction` / `VersionedTransaction` | Messaggi di transazione costruiti e pianificati dal transaction planner   |
| `sendAndConfirmTransaction(...)`       | `client.sendTransaction([...])` (il planner firma, invia e conferma)      |
| `@solana/wallet-adapter-*`             | `@solana/kit-plugin-wallet` + scoperta tramite Wallet Standard            |

## Provieni da web3.js?

Se mantieni un ampio codebase `@solana/web3.js` v1 e una riscrittura completa a
Kit non è ancora praticabile, web3.js v3 rappresenta il ponte. Attualmente viene
distribuito come `@solana/web3.js@rc` (la linea `3.0.0-rc`) e ricostruisce la
classica API basata su classi — `Connection`, `Keypair`, `Transaction` — sopra
gli internals di `@solana/kit`. Poiché si appoggia su Kit, l'interoperabilità è
stretta: `PublicKey` è un alias deprecato di `Address`, e un `Keypair` v3
soddisfa strutturalmente `KeyPairSigner` di Kit, quindi puoi passarlo
direttamente alle API di Kit, ai plugin e ai client generati da Codama.

v3 è ancora una **release candidate** — blocca le versioni esatte e aspettati
alcune variazioni dell'API tra le RC. È un percorso di interoperabilità con il
codice legacy, non una scelta predefinita per i nuovi progetti; le nuove app
dovrebbero essere costruite direttamente su Kit. Per la migrazione stessa,
consulta la guida ufficiale
[web3.js v1 → v3 migration guide](https://github.com/solana-foundation/solana-web3.js/blob/v3.x/docs/web3js-v1-to-v3-migration.md).

## Portafogli

Kit non utilizza wallet-adapter. `@solana/kit-plugin-wallet` rileva i portafogli
Wallet Standard e li espone tramite `client.wallet` e gli
[hook React](/docs/frontend/react-hooks). I portafogli moderni (Phantom,
Solflare, Backpack e altri) si pubblicizzano tramite Wallet Standard, quindi non
è necessario alcun pacchetto adattatore per singolo portafoglio.

Puoi anche utilizzare strumenti preconfigurati come
[@solana/connector](https://www.npmjs.com/package/@solana/connector), che usa
direttamente le primitive di Kit per connettersi ai portafogli.

<Callout type="info" title="Stai ancora usando wallet-adapter?">
  Se gestisci un'app basata su `@solana/wallet-adapter-*` e non sei ancora
  pronto a migrare, il [repository wallet-adapter
  ](https://github.com/anza-xyz/wallet-adapter) rimane il riferimento per quello
  stack. Per qualsiasi nuovo progetto, preferisci il plugin wallet di Kit.
</Callout>
