h1: Crea il Tuo Primo Programma Solana
---

Questo avvio rapido utilizza il progetto iniziale del
[framework Anchor](https://www.anchor-lang.com/docs) generato da `anchor init`.
Creerai il progetto in locale, eseguirai i suoi test, compilerai il programma ed
esaminerai il codice che Anchor genera.

## Prerequisiti

Prima di iniziare,
[installa gli strumenti di sviluppo Solana](/docs/intro/installation).
L'installazione include Rust, la Solana CLI e la Anchor CLI.

Usa la versione 1.1.2 o superiore della Anchor CLI per questo template. Verifica
la versione installata:

```terminal
$ anchor --version
```

<Steps>

<Step>

### Crea il progetto

Esegui i seguenti comandi nel tuo terminale:

```terminal
$ anchor init my-program
$ cd my-program
```

Il progetto iniziale include un programma Solana sotto `programs/my-program`. Il
programma include due istruzioni: una per inizializzare un account contatore e
una per incrementare il contatore.

Alcune parti del template illustrano pattern comuni dei programmi Solana:
derivare indirizzi di account [PDA](/docs/core/pda), eseguire una Cross Program
Invocation ([CPI](/docs/core/cpi)) per trasferire SOL e utilizzare controlli di
errore personalizzati per bloccare un'istruzione quando una condizione non viene
soddisfatta.

<Step>

### Compila il Programma

Esegui `anchor build` per compilare il programma iniziale:

```terminal
$ anchor build
```

Il programma compilato viene scritto in `target/deploy/my_program.so`. Quando il
programma viene distribuito, il contenuto di questo file `.so` viene memorizzato
in un account onchain.

</Step>

<Step>

### Esegui il test

Esegui il test predefinito:

```terminal
$ anchor test
```

Il `Anchor.toml` di questo template utilizza il comando di test Rust:

```toml title="Anchor.toml"
skip_local_validator = true

[scripts]
test = "cargo test"
```

Il test carica il programma compilato in [LiteSVM](https://www.litesvm.com/),
crea un pagante, invia le istruzioni _rs`initialize`_ e _rs`increment`_, quindi
verifica lo stato dell'account del contatore.

L'esecuzione di `anchor test` compila anche il programma, quindi non è
necessario eseguire `anchor build` prima durante i test in locale.

</Step>

<Step>

### Distribuisci il programma

I test in locale rappresentano il ciclo di feedback più rapido. Quando sei
pronto per distribuire su una rete, ad esempio devnet, prima compila, poi
distribuisci su un cluster.

La distribuzione di un programma Solana richiede SOL perché il programma è
memorizzato in un account, e l'account deve pagare per lo spazio che utilizza.
Su devnet, richiedi SOL devnet gratuito dal
[Solana Faucet](https://faucet.solana.com/) oppure tramite la Solana CLI:

```terminal
$ solana airdrop 2 --url devnet
```

```terminal
$ anchor build
$ anchor deploy --provider.cluster devnet
```

</Step>

</Steps>

## File sorgente

La directory `src` contiene il programma Solana. La documentazione di Anchor
sulla
[Struttura del programma](https://www.anchor-lang.com/docs/basics/program-structure)
spiega le macro principali utilizzate qui, tra cui _rs`declare_id!`_,
_rs`#[program]`_, _rs`#[derive(Accounts)]`_ e _rs`#[account]`_. Questa sezione
illustri i file del template.

### lib.rs

`lib.rs` è il punto di ingresso del programma. Collega i file sorgente,
definisce l'indirizzo del programma e definisce le istruzioni del programma che
gli utenti possono richiamare.

<WithNotes>

```rust title="programs/my-program/src/lib.rs" -w
// !tooltip[/pub mod constants/] constantsMod
pub mod constants;
// !tooltip[/pub mod error/] errorMod
pub mod error;
// !tooltip[/pub mod instructions/] instructionsMod
pub mod instructions;
// !tooltip[/pub mod state/] stateMod
pub mod state;

// !tooltip[/anchor_lang::prelude/] prelude
use anchor_lang::prelude::*;

pub use constants::*;
pub use instructions::*;
pub use state::*;

// !tooltip[/declare_id!/] declareId
declare_id!("82sFkffP9wxwpyfZyeaKHH2chQoJPUGsJZSPi9mrUuXd");

// !tooltip[/#\[program\]/] programMacro
#[program]
pub mod my_program {
    use super::*;

    // !tooltip[/pub fn initialize/] publicInstructions
    pub fn initialize(ctx: Context<Initialize>) -> Result<()> {
        // !tooltip[/handle_initialize/] handlers
        crate::instructions::initialize::handle_initialize(ctx)
    }

    // !tooltip[/pub fn increment/] publicInstructions
    pub fn increment(ctx: Context<Increment>) -> Result<()> {
        // !tooltip[/handle_increment/] handlers
        crate::instructions::increment::handle_increment(ctx)
    }
}
```

### !constantsMod

_rs`pub mod constants`_ include `constants.rs` nel crate del programma. Qui il
template conserva i valori condivisi del programma, come il seed PDA, l'importo
del trasferimento e il valore massimo del contatore.

### !errorMod

_rs`pub mod error`_ include `error.rs` nel crate del programma. Questo file
definisce gli errori personalizzati che il programma può restituire quando
un'istruzione non è valida.

### !instructionsMod

_rs`pub mod instructions`_ include `instructions.rs` nel crate del programma.
Quel file funge da indice per i file specifici delle istruzioni presenti in
`src/instructions/`.

### !stateMod

_rs`pub mod state`_ include `state.rs` nel crate del programma. Questo file
definisce la struttura dei dati per gli account di proprietà di questo
programma, incluso l'account _rs`Counter`_.

### !prelude

_rs`anchor_lang::prelude::*`_ importa i tipi e le macro di Anchor utilizzati per
definire un programma Solana, tra cui _rs`Context`_, _rs`Result`_, _rs`Pubkey`_,
_rs`Account`_, _rs`Signer`_ e gli attributi degli account di Anchor.

### !reexports

Le righe _rs`pub use`_ rendono pubblici gli elementi dei moduli figli a partire
dalla radice del crate. Ciò mantiene brevi gli entrypoint delle istruzioni, ad
esempio _rs`Context<Initialize>`_ invece di un percorso di modulo più lungo.

### !declareId

_rs`declare_id!`_ definisce l'indirizzo del programma. Deve corrispondere
all'indirizzo in `Anchor.toml`, al program account distribuito e a qualsiasi
utente o test che chiami il programma.

### !programMacro

_rs`#[program]`_ indica ad Anchor di generare l'entrypoint del programma Solana
e il codice di dispatch delle istruzioni. Le funzioni pubbliche in questo modulo
diventano le istruzioni che gli utenti possono chiamare.

### !publicInstructions

_rs`initialize`_ e _rs`increment`_ sono i due nomi delle istruzioni che gli
utenti possono chiamare. Ciascuna riceve un _rs`Context<T>`_. Il tipo
all'interno di _rs`Context`_ definisce gli account richiesti da
quell'istruzione, e Anchor li valida prima che l'istruzione venga eseguita.

### !handlers

La funzione handler contiene la logica per l'istruzione.

</WithNotes>

### constants.rs

`constants.rs` mantiene i valori condivisi in un unico posto. In questo
template, _rs`COUNTER_SEED`_ deriva il PDA del contatore,
_rs`HELLO_WORLD_LAMPORTS`_ viene trasferito durante l'inizializzazione, e
_rs`MAX_COUNT`_ viene verificato prima dell'incremento.

<WithNotes>

```rust title="programs/my-program/src/constants.rs" -w
use anchor_lang::prelude::*;

// !tooltip[/#\[constant\]/] anchorConstant
// !tooltip[/COUNTER_SEED/] counterSeed
#[constant]
pub const COUNTER_SEED: &[u8] = b"counter";

// !tooltip[/HELLO_WORLD_LAMPORTS/] helloLamports
#[constant]
pub const HELLO_WORLD_LAMPORTS: u64 = 1;

// !tooltip[/MAX_COUNT/] maxCount
#[constant]
pub const MAX_COUNT: u64 = 10;
```

### !anchorConstant

_rs`#[constant]`_ indica ad Anchor di includere il valore nell'IDL del
programma. Questo rende le costanti importanti del programma visibili ai client
generati e agli strumenti.

### !counterSeed

_rs`COUNTER_SEED`_ è la stringa di byte utilizzata per derivare il PDA del
contatore. Il programma e il test utilizzano entrambi questo seed in modo da
concordare sull'indirizzo dell'account senza memorizzare un keypair separato.
Questo template utilizza un seed fisso, quindi il programma ha un unico
indirizzo per il contatore. I programmi reali spesso includono l'indirizzo di un
utente o un altro identificatore nei seed per creare molti account correlati.

### !helloLamports

_rs`HELLO_WORLD_LAMPORTS`_ è la quantità di lamport trasferiti dal pagante
all'account del contatore all'interno di _rs`initialize`_. È volutamente piccola
perché il template sta dimostrando una CPI al System Program.

### !maxCount

_rs`MAX_COUNT`_ è il limite superiore verificato da _rs`increment`_. Mantenerlo
qui dà al programma, ai test e all'IDL generato un unico nome condiviso per la
regola.

</WithNotes>

<SideBySide>

### !left

```rust title="initialize.rs" -w
#[derive(Accounts)]
pub struct Initialize<'info> {
    // ...
    #[account(
        init,
        payer = payer,
        space = 8 + Counter::INIT_SPACE,
        // !mark
        // !mention counter-seed
        seeds = [COUNTER_SEED],
        bump
    )]
    pub counter: Account<'info, Counter>,
    // ...
}

pub fn handle_initialize(ctx: Context<Initialize>) -> Result<()> {
    // ...

    // !mark
    // !mention hello-world-lamports
    anchor_lang::system_program::transfer(cpi_ctx, HELLO_WORLD_LAMPORTS)?;

    Ok(())
}
```

### !right

`initialize.rs` usa due costanti:

- [_rs`COUNTER_SEED`_](mention:counter-seed) ricava l'indirizzo PDA del
  contatore.
- [_rs`HELLO_WORLD_LAMPORTS`_](mention:hello-world-lamports) imposta la quantità
  trasferita dal pagante all'account del contatore.

</SideBySide>

<SideBySide>

### !left

```rust title="increment.rs" -w
pub fn handle_increment(ctx: Context<Increment>) -> Result<()> {
    // ...

    require!(
        // !mark
        // !mention max-count
        ctx.accounts.counter.count < MAX_COUNT,
        ErrorCode::CounterOverflow,
    );

    ctx.accounts.counter.count += 1;
    Ok(())
}
```

### !right

`increment.rs` usa [_rs`MAX_COUNT`_](mention:max-count) come limite superiore
del contatore. Se il conteggio attuale ha già raggiunto il massimo,
_rs`require!`_ restituisce _rs`CounterOverflow`_ e i dati dell'account non
vengono modificati.

</SideBySide>

### state.rs

`state.rs` definisce tipi di dati personalizzati per gli account che il
programma crea e gestisce. Il programma definisce istruzioni per creare,
inizializzare e aggiornare tali dati, ma i dati del contatore non sono
memorizzati nel programma stesso. Sono memorizzati in un account separato con il
proprio indirizzo.

<WithNotes>

```rust title="programs/my-program/src/state.rs" -w
use anchor_lang::prelude::*;

// !tooltip[/#\[account\]/] accountMacro
#[account]
// !tooltip[/InitSpace/] initSpace
#[derive(InitSpace)]
pub struct Counter {
    // !tooltip[/count/] count
    pub count: u64,
    // !tooltip[/authority/] authority
    pub authority: Pubkey,
}
```

### !accountMacro

_rs`#[account]`_ indica ad Anchor che questa struct è un dato che può essere
memorizzato in un account onchain di proprietà di questo programma. Anchor la
usa per serializzare, deserializzare e validare gli account _rs`Counter`_.

### !initSpace

_rs`InitSpace`_ consente ad Anchor di calcolare _rs`Counter::INIT_SPACE`_,
ovvero il numero di byte necessari per i campi di questa struct.
L'discriminatore di account Anchor a 8 byte viene aggiunto separatamente al
momento della creazione dell'account.

### !count

_rs`count`_ è il valore del contatore memorizzato nei dati dell'account.

### !authority

_rs`authority`_ memorizza l'indirizzo autorizzato a incrementare il contatore.
_rs`initialize`_ scrive questo indirizzo, e _rs`increment`_ richiede che lo
stesso indirizzo firmi.

</WithNotes>

<SideBySide>

### !left

```rust title="initialize.rs" -w
#[account(
    init,
    payer = payer,
    // !mark
    // !mention counter-space
    space = 8 + Counter::INIT_SPACE,
    seeds = [COUNTER_SEED],
    bump
)]
pub counter: Account<'info, Counter>,
// ...

pub fn handle_initialize(ctx: Context<Initialize>) -> Result<()> {
    // !mark(1:2)
    // !mention(1:2) initialize-fields
    ctx.accounts.counter.count = 0;
    ctx.accounts.counter.authority = ctx.accounts.payer.key();

    Ok(())
}
```

```rust title="increment.rs" -w
pub fn handle_increment(ctx: Context<Increment>) -> Result<()> {
    // ...

    // !mark
    // !mention increment-count
    ctx.accounts.counter.count += 1;
    Ok(())
}
```

### !right

`state.rs` definisce i dati dell'account _rs`Counter`_. I file delle istruzioni
utilizzano quel tipo quando creano e aggiornano l'account:

- [_rs`Counter::INIT_SPACE`_](mention:counter-space) dimensiona l'account per i
  campi definiti in `state.rs`.
- [_rs`count`_ e _rs`authority`_](mention:initialize-fields) sono i valori dei
  campi scritti quando l'account viene inizializzato.
- [_rs`count += 1`_](mention:increment-count) aggiorna il valore del contatore
  memorizzato dopo che la validazione è superata.

</SideBySide>

### error.rs

`error.rs` definisce gli errori personalizzati del programma. In questo
template, gli errori dimostrano come i gestori delle istruzioni si interrompono
quando a un chiamante non è consentito aggiornare il contatore o il contatore ha
già raggiunto _rs`MAX_COUNT`_.

<WithNotes>

```rust title="programs/my-program/src/error.rs" -w
use anchor_lang::prelude::*;

// !tooltip[/#\[error_code\]/] errorCode
#[error_code]
pub enum ErrorCode {
    // !tooltip[/Unauthorized/] unauthorized
    #[msg("Only the counter authority can update this counter")]
    Unauthorized,
    // !tooltip[/CounterOverflow/] counterOverflow
    #[msg("Counter has reached the maximum value")]
    CounterOverflow,
}
```

### !errorCode

_rs`#[error_code]`_ indica ad Anchor di trasformare queste varianti in errori
personalizzati del programma. I client e i test possono utilizzare quegli errori
invece di dover dedurre dai codici di errore grezzi.

### !unauthorized

_rs`Unauthorized`_ viene restituito quando il firmatario della transazione non è
l'autorità memorizzata nell'account del contatore.

### !counterOverflow

_rs`CounterOverflow`_ viene restituito quando un incremento violerebbe la regola
_rs`MAX_COUNT`_ del programma.

</WithNotes>

<SideBySide>

### !left

```rust title="increment.rs" -w
pub fn handle_increment(ctx: Context<Increment>) -> Result<()> {
    require_keys_eq!(
        ctx.accounts.counter.authority,
        ctx.accounts.authority.key(),
        // !mark
        // !mention unauthorized-error
        ErrorCode::Unauthorized,
    );
    require!(
        ctx.accounts.counter.count < MAX_COUNT,
        // !mark
        // !mention overflow-error
        ErrorCode::CounterOverflow,
    );

    ctx.accounts.counter.count += 1;
    msg!("Hello, world! Counter is now {}", ctx.accounts.counter.count);
    Ok(())
}
```

### !right

`error.rs` elenca gli errori che l'istruzione può restituire:

- [_rs`ErrorCode::Unauthorized`_](mention:unauthorized-error) viene restituito
  quando il firmatario non è l'autorità memorizzata nell'account del contatore.
- [_rs`ErrorCode::CounterOverflow`_](mention:overflow-error) viene restituito
  quando il contatore ha già raggiunto _rs`MAX_COUNT`_.

</SideBySide>

### instructions.rs

`instructions.rs` collega i file delle istruzioni al crate del programma, così
`lib.rs` può accedere al codice delle istruzioni _rs`initialize`_ e
_rs`increment`_. Ciascun file di istruzione definisce gli account richiesti da
quell'istruzione e la logica del gestore che viene eseguita dopo che Anchor ha
validato quegli account.

<WithNotes>

```rust title="programs/my-program/src/instructions.rs" -w
// !tooltip[/pub mod initialize/] moduleDeclarations
pub mod initialize;
pub mod increment;

// !tooltip[/pub use initialize/] reexports
pub use initialize::*;
pub use increment::*;
```

### !moduleDeclarations

Le righe _rs`pub mod`_ rendono `src/instructions/initialize.rs` e
`src/instructions/increment.rs` parte del crate del programma.

### !reexports

Le righe _rs`pub use`_ ri-esportano gli elementi pubblici da quei file. Ciò
rende le struct degli account delle istruzioni e le funzioni gestore disponibili
attraverso il modulo _rs`instructions`_.

</WithNotes>

### initialize.rs

`initialize.rs` definisce gli account necessari per creare l'account del
contatore, quindi scrive i valori iniziali dell'account. La struct
_rs`#[derive(Accounts)]`_ utilizza i
[vincoli degli account](https://www.anchor-lang.com/docs/references/account-constraints)
di Anchor per specificare quali account sono richiesti e come viene creato il
nuovo account del contatore.

<WithNotes>

```rust title="programs/my-program/src/instructions/initialize.rs" -w
use anchor_lang::prelude::*;

use crate::{constants::*, state::Counter};

// !tooltip[/derive\(Accounts\)/] accountsStruct
#[derive(Accounts)]
pub struct Initialize<'info> {
    // !tooltip[/#\[account\(mut\)\]/] payerMutable
    #[account(mut)]
    // !tooltip[/Signer<'info>/] payerSigner
    pub payer: Signer<'info>,
    // !tooltip[/#\[account\(/] counterConstraints
    #[account(
        // !tooltip[/init/] initConstraint
        init,
        // !tooltip[/payer = payer/] payerConstraint
        payer = payer,
        // !tooltip[/space = 8 \+ Counter::INIT_SPACE/] space
        space = 8 + Counter::INIT_SPACE,
        // !tooltip[/seeds = \[COUNTER_SEED\]/] seeds
        seeds = [COUNTER_SEED],
        // !tooltip[/bump/] bump
        bump
    )]
    // !tooltip[/Account<'info, Counter>/] counterAccountType
    pub counter: Account<'info, Counter>,
    // !tooltip[/Program<'info, System>/] systemProgram
    pub system_program: Program<'info, System>,
}

// !tooltip[/handle_initialize/] handler
pub fn handle_initialize(ctx: Context<Initialize>) -> Result<()> {
    // !tooltip[/counter.count = 0/] initialCount
    ctx.accounts.counter.count = 0;
    // !tooltip[/counter.authority = ctx.accounts.payer.key\(\)/] initialAuthority
    ctx.accounts.counter.authority = ctx.accounts.payer.key();

    // !tooltip[/system_program::Transfer/] cpiAccounts
    let cpi_accounts = anchor_lang::system_program::Transfer {
        from: ctx.accounts.payer.to_account_info(),
        to: ctx.accounts.counter.to_account_info(),
    };
    // !tooltip[/CpiContext::new/] cpiContext
    let cpi_ctx = CpiContext::new(anchor_lang::system_program::ID, cpi_accounts);
    // !tooltip[/system_program::transfer/] transfer
    anchor_lang::system_program::transfer(cpi_ctx, HELLO_WORLD_LAMPORTS)?;

    // !tooltip[/msg!/] programLog
    msg!("Hello, world! Counter initialized");
    Ok(())
}
```

## !!steps Contesto dell'Account

La struct _rs`Initialize`_ definisce gli account che devono essere inclusi
quando un utente richiama l'istruzione _rs`initialize`_. Anchor verifica questi
account prima che il gestore venga eseguito.

<CodePlaceholder title="initialize.rs" />

```rust !! title="initialize.rs"
// !focus(1:2)
#[derive(Accounts)]
pub struct Initialize<'info> {
    #[account(mut)]
    pub payer: Signer<'info>,
    #[account(
        init,
        payer = payer,
        space = 8 + Counter::INIT_SPACE,
        seeds = [COUNTER_SEED],
        bump
    )]
    pub counter: Account<'info, Counter>,
    pub system_program: Program<'info, System>,
}
```

## !!steps Account del Pagante

L'account _rs`payer`_ paga per creare l'account del contatore. Il tipo
_rs`Signer<'info>`_ indica che il pagante deve firmare la transazione, e
_rs`#[account(mut)]`_ indica che l'account del pagante può essere modificato
perché i lamport verranno sottratti.

<CodePlaceholder title="initialize.rs" />

```rust !! title="initialize.rs"
#[derive(Accounts)]
pub struct Initialize<'info> {
    // !focus(1:2)
    #[account(mut)]
    pub payer: Signer<'info>,
    #[account(
        init,
        payer = payer,
        space = 8 + Counter::INIT_SPACE,
        seeds = [COUNTER_SEED],
        bump
    )]
    pub counter: Account<'info, Counter>,
    pub system_program: Program<'info, System>,
}
```

## !!steps Account del Contatore

L'account _rs`counter`_ memorizza i dati _rs`Counter`_ da `state.rs`. _rs`init`_
indica ad Anchor di creare questo account prima che il gestore venga eseguito, e
_rs`payer = payer`_ indica ad Anchor quale account paga per la creazione.

<CodePlaceholder title="initialize.rs" />

```rust !! title="initialize.rs"
#[derive(Accounts)]
pub struct Initialize<'info> {
    #[account(mut)]
    pub payer: Signer<'info>,
    // !focus(1:8)
    #[account(
        init,
        payer = payer,
        space = 8 + Counter::INIT_SPACE,
        seeds = [COUNTER_SEED],
        bump
    )]
    pub counter: Account<'info, Counter>,
    pub system_program: Program<'info, System>,
}
```

## !!steps Dimensione dell'Account

Il vincolo _rs`space`_ indica ad Anchor quanti dati allocare per l'account.
Anchor memorizza prima un discriminatore a 8 byte, poi i byte necessari per i
campi _rs`Counter`_. Il discriminatore consente ad Anchor di riconoscere questo
account come un account _rs`Counter`_ prima di deserializzarne i dati.

<CodePlaceholder title="initialize.rs" />

```rust !! title="initialize.rs"
#[derive(Accounts)]
pub struct Initialize<'info> {
    #[account(mut)]
    pub payer: Signer<'info>,
    #[account(
        init,
        payer = payer,
        // !focus
        space = 8 + Counter::INIT_SPACE,
        seeds = [COUNTER_SEED],
        bump
    )]
    pub counter: Account<'info, Counter>,
    pub system_program: Program<'info, System>,
}
```

### !imports

Questo file importa le costanti condivise e il tipo di account _rs`Counter`_.
L'istruzione utilizza quelle definizioni per derivare il PDA, scegliere
l'importo del trasferimento e allocare l'account con la dimensione dei dati
corretta.

### !accountsStruct

_rs`Initialize`_ è il contesto degli account per l'istruzione _rs`initialize`_.
Ogni campo nella struct rappresenta un account separato che deve essere incluso
quando l'istruzione viene chiamata. Anchor utilizza la struct per validare
quegli account in entrata prima che il handler venga eseguito.

### !payerMutable

_rs`#[account(mut)]`_ indica ad Anchor che l'account del pagante deve essere
scrivibile. Questo è necessario perché sia la creazione dell'account che la CPI
di trasferimento deducono lamport dal pagante.

### !payerSigner

_rs`Signer<'info>`_ indica ad Anchor che il pagante deve firmare la transazione.

### !counterConstraints

Il blocco _rs`#[account(...)]`_ indica ad Anchor come creare e validare
l'account counter prima che il handler venga eseguito.

### !initConstraint

_rs`init`_ indica ad Anchor di creare l'account _rs`counter`_ prima che
l'handler venga eseguito. Il nuovo account è di proprietà di questo programma,
quindi il programma può memorizzare i dati _rs`Counter`_ al suo interno.

### !payerConstraint

_rs`payer = payer`_ specifica l'account _rs`payer`_ come pagante per la
creazione del nuovo account contatore.

### !space

_rs`space = 8 + Counter::INIT_SPACE`_ alloca i byte per il discriminatore di
account di Anchor più i campi _rs`Counter`_ da `state.rs`. Il discriminatore
aiuta Anchor a confermare che l'account contenga il tipo atteso prima di
leggerlo.

### !seeds

_rs`seeds = [COUNTER_SEED]`_ indica che l'indirizzo dell'account contatore deve
essere il PDA derivato da questo seed e dall'ID del programma. Con un solo seed
fisso, esiste un unico PDA contatore per questo programma. Per creare contatori
separati in un programma reale, di solito si aggiunge un altro seed, come
l'indirizzo di un utente.

### !bump

_rs`bump`_ indica ad Anchor di trovare e validare il bump del PDA, ovvero il
byte aggiuntivo che rende valido l'indirizzo derivato come PDA.

### !counterAccountType

_rs`Account<'info, Counter>`_ indica ad Anchor di trattare i dati dell'account
come il tipo _rs`Counter`_ da `state.rs`. Anchor verifica inoltre che l'account
sia di proprietà di questo programma.

### !systemProgram

_rs`Program<'info, System>`_ richiede che il chiamante fornisca il System
Program. Anchor lo utilizza per il vincolo di creazione dell'account _rs`init`_.
Anche il CPI di trasferimento dell'handler invoca il System Program.

### !handler

L'handler viene eseguito dopo che Anchor ha validato gli account e creato
l'account contatore, quindi può scrivere in sicurezza i dati iniziali
dell'account.

### !initialCount

Imposta il campo _rs`count`_ nell'account contatore a _rs`0`_.

### !initialAuthority

Imposta il campo _rs`authority`_ nell'account contatore all'indirizzo del
pagante.

### !cpiAccounts

Questi sono gli account passati all'istruzione di trasferimento del System
Program. Questo CPI di trasferimento è incluso solo per dimostrare come un CPI
passa gli account a un altro programma.

### !cpiContext

_rs`CpiContext`_ racchiude il programma di destinazione e gli account necessari
per una Cross Program Invocation. La creazione del contesto prepara il CPI
abbinando il programma che si desidera chiamare con gli account che quel
programma si aspetta.

### !transfer

Questo CPI invoca l'istruzione di trasferimento del System Program e sposta
_rs`HELLO_WORLD_LAMPORTS`_ dal pagante all'account contatore.

### !programLog

_rs`msg!`_ scrive un messaggio nei log del programma quando il programma viene
invocato.

fallimento prima della mutazione.

### !mutateCount

Solo dopo che entrambi i controlli sono superati il gestore aggiorna i dati
dell'account del contatore. Se uno dei controlli non va a buon fine, la
transazione fallisce e l'account non viene modificato.

### !programLog

_rs`msg!`_ scrive un messaggio nei log del programma quando il programma viene
invocato.

</WithNotes>

```rust title="programs/my-program/tests/test_initialize.rs" -w
use {
    anchor_lang::{
        prelude::Pubkey,
        solana_program::{instruction::Instruction, system_program},
```

### !litesvm

_rs`LiteSVM`_ è una Solana VM in-process per i test. Permette al test di
registrare il programma compilato, creare account, inviare transazioni e
ispezionare lo stato degli account senza avviare un processo validator.

### !setup

Il test ottiene l'indirizzo del programma dal valore _rs`declare_id!`_ tramite
_rs`my_program::id()`_ e crea un nuovo keypair pagante per la transazione di
test.

### !counterPda

L'indirizzo del contatore è derivato dallo stesso _rs`COUNTER_SEED`_ utilizzato
dal programma. Questo fa sì che il test punti al PDA che `initialize.rs` si
aspetta.

### !programBytes

_rs`include_bytes!`_ legge il file compilato `my_program.so` prodotto dalla
build. Quei byte sono il codice eseguibile del programma che LiteSVM eseguirà
nel test.

### !addProgram

_rs`svm.add_program`_ registra i byte del programma compilato all'indirizzo del
programma in LiteSVM. L'airdrop fornisce al pagante abbastanza SOL locale per
pagare la transazione e la creazione dell'account.

### !testInstruction

_rs`Instruction::new_with_bytes`_ crea un'istruzione per invocare il programma.
Include l'ID del programma, instruction data e gli account dell'istruzione.

### !blockhash

Ogni transazione Solana necessita di un blockhash recente. In questo test,
LiteSVM fornisce un blockhash da utilizzare per la transazione.

### !message

Il messaggio contiene l'istruzione, il pagante delle commissioni e il blockhash.
Questi sono i dati della transazione che verranno firmati.

### !transaction

_rs`VersionedTransaction::try_new`_ firma il messaggio con il keypair del
pagante. Il pagante firma perché paga la commissione della transazione e
finanzia il nuovo account contatore.

### !sendTx

_rs`svm.send_transaction`_ esegue la transazione in LiteSVM.

### !getCounter

Dopo l'esecuzione della transazione, il test legge l'account contatore
direttamente da LiteSVM per poter ispezionare i dati dell'account.

### !deserialize

L'account contatore memorizza dati serializzati. _rs`Counter::try_deserialize`_
verifica il discriminatore di account di Anchor e riconverte i dati dell'account
nel tipo _rs`Counter`_ da `state.rs`.

### !initAssert

Dopo che _rs`initialize`_ ha successo, il test deserializza l'account contatore
e verifica che lo stato iniziale corrisponda a ciò che _rs`handle_initialize`_
ha scritto.

### !incrementAssert

Dopo che _rs`increment`_ ha successo, lo stesso account contatore dovrebbe avere
_rs`count`_ impostato su _rs`1`_ mantenendo la stessa authority.

</WithNotes>
