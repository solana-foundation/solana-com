You do not need to become a protocol engineer before building a financial
product on Solana. You do need a clear model of where balances and ownership
live, how they change, and how your systems observe those changes.

## State lives in accounts

Solana stores state in [accounts](/docs/core/accounts). An account has an
address, an owner program, a SOL balance, and optional data. Programs contain
the rules for changing state; the mutable state itself remains in accounts.

For financial applications, the important distinction is between a wallet
address and the accounts it controls. A wallet can authorize activity across
many accounts, including a separate token account for every asset it holds.

## Assets use token accounts

An asset is identified by a mint account. Balances are stored in token accounts
associated with that mint, and a Token Program enforces transfers, issuance,
burning, authorities, and optional extensions.

Start with [Assets on Solana](/docs/tokens) to understand mints and token
accounts. If you plan to issue an asset, continue to
[Issuance & Tokenization](/docs/tokenization).

## Transactions change state

A [transaction](/docs/core/transactions) contains one or more instructions that
execute atomically: either every instruction succeeds or none of their state
changes are committed. This is what makes workflows such as delivery versus
payment and multi-step trades possible without a separate settlement process.

Transaction status progresses through processed, confirmed, and finalized
commitment levels. Financial systems should choose a commitment level based on
the risk of the operation and reconcile finalized results rather than treating
submission as settlement.

## Applications read the ledger through RPC

Applications use the [Solana JSON-RPC API](/docs/rpc) to read account state,
submit transactions, and check their status. Production systems typically pair
RPC calls with [indexing](/docs/tools/indexing) for history, reporting, and
reconciliation.

Use the [Solana Explorer](https://explorer.solana.com) to inspect an address,
transaction signature, program, or token mint while developing and debugging.
The explorer is a view into onchain data, not a separate source of truth.
