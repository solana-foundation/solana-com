h1: Build Your First Solana Program
---

This quickstart uses the Anchor starter project generated by `anchor init`. You
will create the project locally, run its tests, build the program, and walk
through the program code that Anchor generates.

Before you start,
[install the Solana development tools](/docs/intro/installation). The
installation includes Rust, the Solana CLI, and the Anchor CLI.

## Create the project

Run the following commands in your terminal:

```terminal
$ anchor init my-program
$ cd my-program
```

The starter project includes one Rust program under `programs/my-program`. The
program stores a counter account, initializes the counter, and increments the
counter.

```text title="Project files"
my-program/
  Anchor.toml
  Cargo.toml
  programs/
    my-program/
      Cargo.toml
      src/
        lib.rs
        constants.rs
        error.rs
        instructions.rs
        instructions/
          initialize.rs
          increment.rs
        state.rs
      tests/
        test_initialize.rs
```

## Run locally

Run the generated test:

```terminal
$ anchor test
```

This starter's `Anchor.toml` uses the Rust test command:

```toml title="Anchor.toml"
skip_local_validator = true

[scripts]
test = "cargo test"
```

The test loads the compiled program into LiteSVM, creates a payer, sends the
`initialize` and `increment` instructions, then checks the counter account
state.

Build the program when you want the compiled deployable artifact:

```terminal
$ anchor build
```

The compiled program is written to `target/deploy/my_program.so`.

## Deploy later

Local tests are the fastest feedback loop. When you are ready to deploy to a
network, build first, then deploy to a cluster.

```terminal
$ anchor build
$ anchor deploy --provider.cluster devnet
```

Deploy to `mainnet-beta` only after you have tested, reviewed, and funded the
deployer wallet for real transaction fees and rent.

```terminal
$ anchor deploy --provider.cluster mainnet-beta
```

## Program entrypoint

The `src/lib.rs` file is the program entrypoint. It declares the program
address, imports the program modules, and exposes the instructions that clients
can call.

<WithNotes>

```rust title="programs/my-program/src/lib.rs"
// !tooltip[/pub mod constants/] modules
pub mod constants;
pub mod error;
pub mod instructions;
pub mod state;

use anchor_lang::prelude::*;

pub use constants::*;
pub use instructions::*;
pub use state::*;

// !tooltip[/declare_id!/] declareId
declare_id!("82sFkffP9wxwpyfZyeaKHH2chQoJPUGsJZSPi9mrUuXd");

// !tooltip[/#\[program\]/] program
#[program]
pub mod my_program {
    use super::*;

    // !tooltip[/initialize/] initialize
    pub fn initialize(ctx: Context<Initialize>) -> Result<()> {
        crate::instructions::initialize::handle_initialize(ctx)
    }

    // !tooltip[/increment/] increment
    pub fn increment(ctx: Context<Increment>) -> Result<()> {
        crate::instructions::increment::handle_increment(ctx)
    }
}
```

### !modules

The starter separates constants, errors, instruction handlers, and account state
into different files. `lib.rs` imports those modules and re-exports their public
types.

### !declareId

`declare_id!` sets the program address. Your generated project will have a
different address.

### !program

`#[program]` marks the module that Anchor exposes as callable program
instructions.

### !initialize

`initialize` is the instruction that creates and initializes the counter
account.

### !increment

`increment` is the instruction that reads the existing counter account and
increments its `count` field.

</WithNotes>

## !!steps Modules

The top of `lib.rs` declares the source files that make up the program. This
keeps the entrypoint small while the instruction logic lives in
`instructions/initialize.rs` and `instructions/increment.rs`.

<CodePlaceholder title="lib.rs" />

```rust !! title="lib.rs"
// !focus(1:4)
pub mod constants;
pub mod error;
pub mod instructions;
pub mod state;

use anchor_lang::prelude::*;

pub use constants::*;
pub use instructions::*;
pub use state::*;

declare_id!("82sFkffP9wxwpyfZyeaKHH2chQoJPUGsJZSPi9mrUuXd");

#[program]
pub mod my_program {
    use super::*;

    pub fn initialize(ctx: Context<Initialize>) -> Result<()> {
        crate::instructions::initialize::handle_initialize(ctx)
    }

    pub fn increment(ctx: Context<Increment>) -> Result<()> {
        crate::instructions::increment::handle_increment(ctx)
    }
}
```

## !!steps Program Address

`declare_id!` embeds the program address in the Rust crate. Anchor uses this
address when generating clients, account validation code, and instruction data.

<CodePlaceholder title="lib.rs" />

```rust !! title="lib.rs"
pub mod constants;
pub mod error;
pub mod instructions;
pub mod state;

use anchor_lang::prelude::*;

pub use constants::*;
pub use instructions::*;
pub use state::*;

// !focus
declare_id!("82sFkffP9wxwpyfZyeaKHH2chQoJPUGsJZSPi9mrUuXd");

#[program]
pub mod my_program {
    use super::*;

    pub fn initialize(ctx: Context<Initialize>) -> Result<()> {
        crate::instructions::initialize::handle_initialize(ctx)
    }

    pub fn increment(ctx: Context<Increment>) -> Result<()> {
        crate::instructions::increment::handle_increment(ctx)
    }
}
```

## !!steps Instructions

Functions inside the `#[program]` module are the instructions clients can
invoke. Each instruction receives a `Context<T>` that contains the accounts
required by that instruction.

<CodePlaceholder title="lib.rs" />

```rust !! title="lib.rs"
pub mod constants;
pub mod error;
pub mod instructions;
pub mod state;

use anchor_lang::prelude::*;

pub use constants::*;
pub use instructions::*;
pub use state::*;

declare_id!("82sFkffP9wxwpyfZyeaKHH2chQoJPUGsJZSPi9mrUuXd");

#[program]
pub mod my_program {
    use super::*;

    // !focus(1:7)
    pub fn initialize(ctx: Context<Initialize>) -> Result<()> {
        crate::instructions::initialize::handle_initialize(ctx)
    }

    pub fn increment(ctx: Context<Increment>) -> Result<()> {
        crate::instructions::increment::handle_increment(ctx)
    }
}
```

## !!steps Handlers

The entrypoint delegates to handler functions in the `instructions` module. This
keeps account validation and instruction logic close together in separate files.

<CodePlaceholder title="lib.rs" />

pub mod instructions;
pub mod state;
