This quickstart uses the Anchor starter project generated by `anchor init`. You
will create the project locally, run its tests, build the program, and walk
through the program code that Anchor generates.

Before you start,
[install the Solana development tools](/docs/intro/installation). The
installation includes Rust, the Solana CLI, and the Anchor CLI.

## Create the project

Run the following commands in your terminal:

```terminal
$ anchor init my-program
$ cd my-program
```

The starter project includes one Solana program under `programs/my-program`. The
program includes two instructions: one to initialize a counter account and one
to increment the counter.

## Build Program

Run `anchor build` to compile the starter program:

```terminal
$ anchor build
```

The compiled program is written to `target/deploy/my_program.so`. When the
program is deployed, the contents of this `.so` file are stored in an account
onchain.

## Run Test

Run the default test:

```terminal
$ anchor test
```

This template's `Anchor.toml` uses the Rust test command:

```toml title="Anchor.toml"
skip_local_validator = true

[scripts]
test = "cargo test"
```

The test loads the compiled program into LiteSVM, creates a payer, sends the
`initialize` and `increment` instructions, then checks the counter account
state.

Running `anchor test` also compiles the program, so you do not need to run
`anchor build` first when testing locally.

## Deploy Program

Local tests are the fastest feedback loop. When you are ready to deploy to a
network, for example devnet, build first, then deploy to a cluster.

Deploying a Solana program requires SOL because the program is stored in an
account, and the account must pay for the space it uses. On devnet, request free
devnet SOL from the [Solana Faucet](https://faucet.solana.com/) or with the
Solana CLI:

```terminal
$ solana airdrop 2 --url devnet
```

```terminal
$ anchor build
$ anchor deploy --provider.cluster devnet
```

## Project configuration

The root project files tell Anchor and Cargo how to build, test, and deploy the
program. For a full reference, see the Anchor docs for
[Anchor.toml configuration](https://www.anchor-lang.com/docs/references/anchor-toml)
and the [Anchor CLI](https://www.anchor-lang.com/docs/references/cli).

<WithNotes>

```toml title="Anchor.toml"
# !tooltip[/skip_local_validator/] skipValidator
skip_local_validator = true

[toolchain]

[features]
resolution = true
skip-lint = false

# !tooltip[/programs.localnet/] programs

# !tooltip[/provider/] provider

# !tooltip[/scripts/] scripts

[hooks]
```

### !skipValidator

`skip_local_validator = true` tells Anchor not to start a validator for
`anchor test`. The Rust test uses LiteSVM instead.

### !programs

`[programs.localnet]` maps the Rust crate name to the program address used for
local builds and tests.

### !provider

`[provider]` sets the default cluster and wallet for commands such as
`anchor deploy`.

### !scripts

`[scripts]` changes what `anchor test` runs. In this template, `anchor test`
runs `cargo test`.

</WithNotes>

The root `Cargo.toml` defines a Rust workspace. The program crate has its own
`Cargo.toml` under `programs/my-program`. The workspace file controls shared
Rust settings; the program crate file controls how the Solana program is
compiled and tested.

The two `Cargo.toml` files work together:

- The root `members = ["programs/*"]` line tells Cargo to include
  `programs/my-program` in the workspace.
- The program crate inherits `edition` and `rust-version` from the workspace.
- The `cdylib` crate type produces the `.so` file that can be deployed on
  Solana. The `lib` crate type lets the Rust test import the same program code.
- The `name = "my_program"` line becomes the Rust crate name used in
  `test_initialize.rs`, such as `my_program::id()`.
- `anchor-lang` provides Anchor's macros and account types. The dev dependencies
  provide the LiteSVM test environment and Solana transaction types.

## Source files

The `src` directory contains the Solana program. Anchor's
[Program Structure](https://www.anchor-lang.com/docs/basics/program-structure)
docs explain the core macros used here, including `declare_id!`, `#[program]`,
`#[derive(Accounts)]`, and `#[account]`. This section follows the template files
in the order a new reader should inspect them.

Keep one picture in mind while reading the files: the program is code, and the
counter is data stored in a separate account. The program does not hold the
counter inside itself. The program only defines instructions that say which
accounts must be provided and how those accounts may change.

### `lib.rs`

`lib.rs` is the program entrypoint. It declares the source modules, imports the
public types those modules expose, sets the program address, and exposes the
instructions clients can invoke.

<WithNotes>

```rust title="programs/my-program/src/lib.rs" -w
// !tooltip[/pub mod constants/] modules
pub mod constants;
pub mod error;
pub mod instructions;
pub mod state;

// !tooltip[/anchor_lang::prelude/] prelude
use anchor_lang::prelude::*;

// !tooltip[/pub use constants/] reexports
pub use constants::*;
pub use instructions::*;
pub use state::*;

// !tooltip[/declare_id!/] declareId
declare_id!("82sFkffP9wxwpyfZyeaKHH2chQoJPUGsJZSPi9mrUuXd");

// !tooltip[/#\[program\]/] programMacro
#[program]
pub mod my_program {
    use super::*;

    // !tooltip[/pub fn initialize/] publicInstructions
    pub fn initialize(ctx: Context<Initialize>) -> Result<()> {
        // !tooltip[/handle_initialize/] handlers
        crate::instructions::initialize::handle_initialize(ctx)
    }

    // !tooltip[/pub fn increment/] publicInstructions
    pub fn increment(ctx: Context<Increment>) -> Result<()> {
        // !tooltip[/handle_increment/] handlers
        crate::instructions::increment::handle_increment(ctx)
    }
}
```

### !modules

The `pub mod` lines load the other files in `src`. Without these declarations,
Rust would not know that `constants.rs`, `error.rs`, `instructions.rs`, and
`state.rs` are part of this crate.

### !prelude

`anchor_lang::prelude::*` imports the Anchor types and macros used throughout
the program, including `Context`, `Result`, `Pubkey`, `Account`, `Signer`, and
the Anchor attributes.

### !reexports

The `pub use` lines make public items from the child modules available from the
crate root. That is why the `#[program]` module can write `Context<Initialize>`
instead of a longer path.

### !declareId

`declare_id!` embeds the program address in the Rust crate. The address should
match the address configured for the program in `Anchor.toml`.

### !programMacro

`#[program]` marks the module that Anchor exposes as callable program
instructions. Public functions in this module become instruction entrypoints.

### !publicInstructions

`initialize` and `increment` are the two instruction names clients can call.
Each function receives a `Context<T>` with the accounts required by that
instruction.

### !handlers

The public instruction functions delegate to handler functions in the
instruction files. This keeps `lib.rs` short while the account validation and
business logic stay next to each other.

</WithNotes>

### `constants.rs`

`constants.rs` keeps shared values in one place. The seed is used to derive the
counter PDA, the lamports value is transferred during initialization, and the
maximum count is used by the increment guard.

<WithNotes>

```rust title="programs/my-program/src/constants.rs" -w

// !tooltip[/HELLO_WORLD_LAMPORTS/] helloLamports
#[constant]
pub const HELLO_WORLD_LAMPORTS: u64 = 1;

// !tooltip[/MAX_COUNT/] maxCount
#[constant]
pub const MAX_COUNT: u64 = 10;
```

### !counterSeed

`COUNTER_SEED` is the byte string used to derive the counter PDA. The program
and the test both use this seed so they agree on the same counter address.

### !helloLamports

`HELLO_WORLD_LAMPORTS` is the amount transferred from the payer to the counter
account inside `initialize`. It is deliberately tiny because the template is
only demonstrating a CPI to the System Program.

### !maxCount

`MAX_COUNT` is the upper bound checked by `increment`. Keeping it here means the
limit has one name and one definition.

</WithNotes>

### `state.rs`

`state.rs` defines the account data owned by this program. A `Counter` account
stores a count and the authority allowed to update that count.

<WithNotes>

```rust title="programs/my-program/src/state.rs" -w
use anchor_lang::prelude::*;

// !tooltip[/#\[account\]/] accountMacro
#[account]
// !tooltip[/InitSpace/] initSpace
#[derive(InitSpace)]
pub struct Counter {
    // !tooltip[/count/] count
    pub count: u64,
    // !tooltip[/authority/] authority
    pub authority: Pubkey,
}
```

### !accountMacro

`#[account]` tells Anchor this struct is an account data type. Anchor can then
serialize, deserialize, and validate accounts that store `Counter` data.

### !initSpace

`InitSpace` lets Anchor calculate `Counter::INIT_SPACE`, the number of bytes
needed to store the fields in this struct.

### !count

`count` is the number this program increments.

### !authority

`authority` stores the address allowed to increment the counter. The
`initialize` instruction writes this value, and the `increment` instruction
checks it.

</WithNotes>

### `error.rs`

`error.rs` defines the program's custom errors. The instruction handlers return
these errors when a caller is not allowed to update the counter or the counter
has already reached `MAX_COUNT`.

<WithNotes>

```rust title="programs/my-program/src/error.rs" -w

pub enum ErrorCode {
    // !tooltip[/Unauthorized/] unauthorized
    #[msg("Only the counter authority can update this counter")]
    Unauthorized,
    // !tooltip[/CounterOverflow/] counterOverflow
    #[msg("Counter has reached the maximum value")]
    CounterOverflow,
}
```

### !errorCode

`#[error_code]` tells Anchor to turn enum variants into program errors that
clients and tests can recognize.

### !unauthorized

`Unauthorized` is returned when the signer is not the address stored in
`counter.authority`.

### !counterOverflow

`CounterOverflow` is returned when the counter has already reached `MAX_COUNT`.

</WithNotes>

### `instructions/initialize.rs`

`initialize.rs` defines the accounts required to create the counter account, and
then initializes the account's fields. The `#[derive(Accounts)]` struct uses
Anchor
[account constraints](https://www.anchor-lang.com/docs/references/account-constraints)
to say which accounts are required and how the new counter account is created.

<WithNotes>

```rust title="programs/my-program/src/instructions/initialize.rs" -w
use anchor_lang::prelude::*;

// !tooltip[/constants::\*/] imports
use crate::{constants::*, state::Counter};

// !tooltip[/derive\(Accounts\)/] accountsStruct
#[derive(Accounts)]
pub struct Initialize<'info> {
    // !tooltip[/Signer/] payer
    #[account(mut)]
    pub payer: Signer<'info>,
    #[account(
        // !tooltip[/init/] initConstraint
        init,
        payer = payer,
        // !tooltip[/Counter::INIT_SPACE/] space
        space = 8 + Counter::INIT_SPACE,
        // !tooltip[/COUNTER_SEED/] seeds
        seeds = [COUNTER_SEED],
        // !tooltip[/bump/] bump
        bump
    )]
    pub counter: Account<'info, Counter>,
    // !tooltip[/Program<'info, System>/] systemProgram
    pub system_program: Program<'info, System>,
}

// !tooltip[/handle_initialize/] handler
pub fn handle_initialize(ctx: Context<Initialize>) -> Result<()> {
    // !tooltip[/counter.count = 0/] startingState
    ctx.accounts.counter.count = 0;
    ctx.accounts.counter.authority = ctx.accounts.payer.key();

    // !tooltip[/system_program::Transfer/] cpiAccounts
    let cpi_accounts = anchor_lang::system_program::Transfer {
        from: ctx.accounts.payer.to_account_info(),
        to: ctx.accounts.counter.to_account_info(),
    };
    // !tooltip[/CpiContext::new/] cpiContext
    let cpi_ctx = CpiContext::new(anchor_lang::system_program::ID, cpi_accounts);
    // !tooltip[/system_program::transfer/] transfer
    anchor_lang::system_program::transfer(cpi_ctx, HELLO_WORLD_LAMPORTS)?;

    msg!("Hello, world! Counter initialized");
    Ok(())
}
```

### !imports

This file imports shared constants and the `Counter` account type. The
instruction cannot derive the PDA or allocate the account size without those
definitions.

### !accountsStruct

`Initialize` lists every account the client must provide to call the
`initialize` instruction.

### !payer

The payer signs the transaction and pays for the new account.

### !initConstraint

`init` tells Anchor to create the `counter` account before the handler runs.

### !space

`space = 8 + Counter::INIT_SPACE` allocates enough bytes for Anchor's account
discriminator plus the `Counter` fields from `state.rs`.

### !seeds

`seeds = [COUNTER_SEED]` says the counter account must be the PDA derived from
the seed in `constants.rs`.

### !bump

`bump` tells Anchor to find and validate the PDA bump for the seed.

### !systemProgram

The System Program is required because creating an account is done through the
System Program.

### !handler

The handler runs after Anchor validates the accounts and creates the counter
account.

### !startingState

The first two writes set the account's initial state: count starts at `0`, and
the payer becomes the authority that may increment later.

### !cpiAccounts

These are the accounts passed to the System Program transfer instruction.

### !cpiContext

`CpiContext` packages the target program and accounts for a cross-program
invocation.

### !transfer

This CPI transfers `HELLO_WORLD_LAMPORTS` from the payer to the new counter
account.

</WithNotes>

### `instructions/increment.rs`

`increment.rs` defines the accounts required to update the existing counter
account. The handler checks that the signer is the stored authority, checks that
the count has not reached the maximum, and then increments the count.

<WithNotes>

```rust title="programs/my-program/src/instructions/increment.rs" -w
use anchor_lang::prelude::*;

// !tooltip[/error::ErrorCode/] imports
use crate::{constants::*, error::ErrorCode, state::Counter};

// !tooltip[/derive\(Accounts\)/] incrementAccounts
#[derive(Accounts)]
pub struct Increment<'info> {
    // !tooltip[/mut, seeds/] counterAccount
    #[account(mut, seeds = [COUNTER_SEED], bump)]
    pub counter: Account<'info, Counter>,
    // !tooltip[/Signer/] authoritySigner
    pub authority: Signer<'info>,
}

// !tooltip[/handle_increment/] incrementHandler
pub fn handle_increment(ctx: Context<Increment>) -> Result<()> {
    // !tooltip[/require_keys_eq!/] authorityCheck
    require_keys_eq!(
        ctx.accounts.counter.authority,
        ctx.accounts.authority.key(),
        ErrorCode::Unauthorized,
    );
    // !tooltip[/require!/] maxCheck
    require!(
        ctx.accounts.counter.count < MAX_COUNT,
        ErrorCode::CounterOverflow,
    );

    // !tooltip[/count \+= 1/] mutateCount
    ctx.accounts.counter.count += 1;
    msg!("Hello, world! Counter is now {}", ctx.accounts.counter.count);
    Ok(())
}
```

### !imports

This file imports shared constants, custom errors, and the `Counter` account
type. The authority check needs `ErrorCode`, the PDA validation needs
`COUNTER_SEED`, and the account type is `Counter`.

### !incrementAccounts

`Increment` lists the accounts needed to call the `increment` instruction.

### !counterAccount

The counter account must be mutable because the handler changes `count`. The
same `COUNTER_SEED` verifies that the provided counter account is the expected
PDA.

### !authoritySigner

The authority must sign the transaction. The handler then compares this signer
to the authority stored in the counter account.

### !incrementHandler

The handler runs after Anchor validates the accounts.

### !authorityCheck

`require_keys_eq!` stops the instruction unless the signer matches the stored
authority.

### !maxCheck

`require!` stops the instruction if the counter has already reached `MAX_COUNT`.

### !mutateCount

Only after both checks pass does the handler increment the counter.

</WithNotes>

<SideBySide>

### !left

```rust title="error.rs" -w

    // !mark
```

### !right

The increment handler is a good example of the program's safety model: validate
first, mutate last. If either rule fails, the counter account is not changed.

does not start a local validator. Instead, it loads the compiled `.so` file into
LiteSVM, sends the two instructions, and reads the counter account after each
transaction.

### !anchorTraits

These Anchor traits connect the Rust program to the test:

- `InstructionData` turns `Initialize {}` and `Increment {}` into instruction
  bytes.
- `ToAccountMetas` turns the generated account helper structs into Solana
  account metadata.
- `AccountDeserialize` turns account bytes back into the `Counter` Rust type.

### !litesvm

`LiteSVM` is an in-process Solana VM for tests. It lets the test load a compiled
program, create accounts, send transactions, and inspect account state without
starting a validator process.

### !setup

The test gets the program address from `my_program::id()` and creates a new
payer keypair for the test transaction.

### !counterPda

The counter address is derived from the same `COUNTER_SEED` used by the program.
This makes the test target the PDA that `initialize.rs` expects.

### !programBytes

`include_bytes!` reads the compiled `my_program.so` file produced by the build.

### !addProgram

`svm.add_program` loads the compiled program into LiteSVM. The airdrop gives the
payer enough local SOL to pay for the transaction and account creation.

### !initializeData

Anchor generates instruction data and account metadata helpers for the program.
The test uses those helpers to build the `initialize` instruction.

### !blockhash

Every transaction needs a recent blockhash. In this test, LiteSVM provides the
latest blockhash from its local bank.

### !message

The message contains the instruction, the fee payer, and the blockhash. This is
the transaction data that will be signed.

### !transaction

`VersionedTransaction::try_new` signs the message with the payer keypair. The
payer signs because the payer pays for the transaction and creates the counter
account.

### !sendTx

`svm.send_transaction` executes the transaction against LiteSVM. If account
validation or instruction logic fails, this result is an error.

### !getCounter

After the transaction runs, the test reads the counter account directly from
LiteSVM.

### !deserialize

The counter account stores bytes. `Counter::try_deserialize` turns those bytes
back into the `Counter` type from `state.rs` so the test can inspect `count` and
`authority`.

### !initAssert

After `initialize` succeeds, the test deserializes the counter account and
checks the starting state.

### !incrementData

The second instruction uses the same counter account and the payer as the
authority.

### !incrementAssert

After `increment` succeeds, the same counter account should have `count` set to
`1`.
