To build on Solana, it's essential to understand the core concepts that make the
network unique. These pages are ordered so each concept builds on the previous.

## Accounts

An account is Solana's fundamental data unit. All state on the Solana
network--balances, program code, token metadata--is stored in accounts. The
network stores data in a key-value store where the key is a 32-byte address and
the value is an account.

Learn more about [Accounts](/docs/core/accounts).

## Programs

On Solana, "smart contracts" are called programs. A program is an account that
contains executable code and is marked as executable. Programs are stateless:
they do not store mutable data in their own account but instead read and write to
separate data accounts passed via instructions.

Learn more about [Programs on Solana](/docs/core/programs).

## Instructions

An instruction is a request to execute a specific function on a Solana program.
Each instruction specifies a single program to call, the accounts it needs, and
a byte array containing the function selector and serialized arguments.

Learn more about [Instructions](/docs/core/instructions).

## Transactions

A transaction is the atomic unit of interaction with the Solana network. It
packages one or more instructions along with the metadata required to execute
them, including cryptographic signatures and a recent blockhash.

Learn more about [Transactions](/docs/core/transactions).

## Fees

Every Solana transaction requires a fee paid in SOL. The fee has two parts:

- **Base Fee**: 5,000 lamports per signature, split 50% burned / 50% to the
  validator.
- **Prioritization Fee**: An optional fee based on compute units to increase
  the chance that the current leader processes your transaction ahead of others.

Learn more about [Fees on Solana](/docs/core/fees).

## Program Derived Address

A Program Derived Address (PDA) is a 32-byte address deterministically derived
from a program ID and a set of seeds. A PDA does not lie on the Ed25519 curve,
so it has no corresponding private key. PDAs enable two key capabilities:

- **Deterministic Addresses**: Given the same seeds and program ID, the same
  address is always derived, enabling predictable account lookups.
- **Program Signing**: Only the deriving program can authorize operations on a
  PDA's behalf through `invoke_signed` during cross-program invocations.

Learn more about [Program Derived Address](/docs/core/pda).

## Cross-program Invocation

A Cross-Program Invocation (CPI) is when one program calls an instruction on
another program during execution. CPIs enable composability: any program's
instructions can be invoked by any other program on the network.

Learn more about [Cross Program Invocation](/docs/core/cpi).

## Tokens

Tokens are digital assets that represent ownership over diverse categories of
assets. Tokenization enables the digitalization of property rights--a fundamental
component for managing fungible and non-fungible assets.

- **Fungible Tokens**: Interchangeable and divisible assets of the same type and
  value (ex. USDC).
- **Non-fungible Tokens (NFT):** Ownership of indivisible assets (e.g. artwork).

Learn more about [Tokens on Solana](/docs/core/tokens).

## Clusters and Endpoints

The Solana blockchain has several different groups of validators, known as
[Clusters](/docs/references/clusters). Each cluster serves a different purpose
and contains dedicated nodes to fulfill [JSON-RPC](/docs/rpc/) requests. There
are three primary clusters on the Solana network, shown in the table below.

| Name    | Endpoint                         | Use                       |
| ------- | -------------------------------- | ------------------------- |
| Mainnet | `https://api.mainnet.solana.com` | Production                |
| Devnet  | `https://api.devnet.solana.com`  | Developer experimentation |
| Testnet | `https://api.testnet.solana.com` | Validator testing         |

Learn more about [Clusters and Endpoints](/docs/references/clusters).
