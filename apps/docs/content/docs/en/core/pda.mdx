Program Derived Addresses (PDAs) are 32-byte account addresses that are
deterministically derived from a program ID and a set of seeds. They are
guaranteed to not lie on the Ed25519 curve, which means no private key exists for
them. Only the program whose ID was used in the derivation can "sign" for a PDA,
and it does so through [`invoke_signed`](/docs/core/cpi) during cross-program
invocations (CPIs).

Key properties:

- **Deterministic**: The same seeds and program ID always produce the same
  address.
- **Off-curve**: The derived address is verified to not be a valid Ed25519 public
  key. If the hash happens to land on the curve, the derivation fails and a
  different bump seed is tried.
- **No private key**: Because the address is off-curve, no one can produce a
  cryptographic signature for it. The program "signs" via the runtime's
  `invoke_signed` mechanism instead.

| Constraint | Value | Source |
| --- | --- | --- |
| Max seeds | 16 | [`MAX_SEEDS`](https://github.com/anza-xyz/solana-sdk/blob/f2d15de6f7a1715ff806f0c39bba8f64bf6a587d/address/src/lib.rs#L56) |
| Max seed length | 32 bytes each | [`MAX_SEED_LEN`](https://github.com/anza-xyz/solana-sdk/blob/f2d15de6f7a1715ff806f0c39bba8f64bf6a587d/address/src/lib.rs#L54) |
| Bump range | 0-255 (1 byte) | Appended as the final seed element |
| `create_program_address` cost | 1,500 CUs | [`create_program_address_units`](https://github.com/anza-xyz/agave/blob/v3.1.8/program-runtime/src/execution_budget.rs#L200) |
| `find_program_address` worst-case cost | 384,000 CUs (1,500 entry + 255 x 1,500 iterations) | 1,500 on entry + 1,500 per failed bump |
| Max PDA signers per CPI | 16 | [`MAX_SIGNERS`](https://github.com/anza-xyz/agave/blob/v3.1.8/program-runtime/src/cpi.rs#L62) |

**Depends on**: [Accounts](/docs/core/accounts) (account addresses),
[Programs](/docs/core/programs) (program ID).
**Used by**: [CPI](/docs/core/cpi) (`invoke_signed` to sign on behalf of PDAs).

## Background

Solana
[`Keypair`](https://github.com/anza-xyz/solana-sdk/blob/sdk%40v2.2.2/keypair/src/lib.rs#L26)
values are points on the [Ed25519 curve](https://ed25519.cr.yp.to/). A keypair
consists of a public key (used as the account address) and a secret key (used to
produce signatures). Anyone with the secret key can sign transactions for that
address.

![Two accounts with on-curve addresses](/assets/docs/core/pda/address-on-curve.svg)

A PDA is intentionally derived to fall _off_ the Ed25519 curve. Because it is
not a valid curve point, no secret key exists, and no external party can produce
a signature. Only the deriving program can authorize operations on the PDA
through `invoke_signed`.

![Off Curve Address](/assets/docs/core/pda/address-off-curve.svg)

### PDA vs keypair accounts

| Property | Keypair account | PDA account |
| --- | --- | --- |
| Address type | On Ed25519 curve | Off Ed25519 curve |
| Has private key | Yes | No |
| Can sign transactions | Yes (with private key) | No |
| Can sign during CPI | No (unless keypair is provided) | Yes (via `invoke_signed`) |
| Derivation | Random or from seed phrase | Deterministic from seeds + program ID |
| Typical use | User wallets, validators | Program-owned data accounts, authority accounts, vault accounts |

## Derivation algorithm

The PDA derivation is implemented in the SDK's
[`create_program_address`](https://github.com/anza-xyz/solana-sdk/blob/f2d15de6f7a1715ff806f0c39bba8f64bf6a587d/address/src/syscalls.rs#L390-L440)
function. The on-chain syscall wrapper (`SyscallCreateProgramAddress`) adds its
own seed count check before delegating to this SDK function. The algorithm works
as follows:

1. Validate that the number of seeds does not exceed `MAX_SEEDS` (16) and no
   individual seed exceeds `MAX_SEED_LEN` (32 bytes). If either check fails,
   return `AddressError::MaxSeedLengthExceeded`.
2. Initialize a SHA-256 hasher.
3. Hash each seed sequentially: `hash(seed[0])`, `hash(seed[1])`, ...,
   `hash(seed[n])`.
4. Append the program ID bytes (32 bytes) and the domain separator string
   `"ProgramDerivedAddress"` (21 bytes) to the same hasher via sequential
   `.update()` calls, matching the actual implementation.
5. Finalize the hash to produce a 32-byte result.
6. Check if the result is a valid point on the Ed25519 curve using
   [`bytes_are_curve_point`](https://github.com/anza-xyz/solana-sdk/blob/f2d15de6f7a1715ff806f0c39bba8f64bf6a587d/address/src/lib.rs#L177-L189),
   which attempts to decompress the bytes as a `CompressedEdwardsY` point. If
   decompression succeeds, the point is on the curve.
7. If the result IS on the curve, return `AddressError::InvalidSeeds` (the
   address would have a corresponding private key, which violates the PDA
   security property).
8. If the result is NOT on the curve, return it as the PDA.

```
create_program_address(seeds, program_id):
    if len(seeds) > 16:
        return Error(MaxSeedLengthExceeded)
    for seed in seeds:
        if len(seed) > 32:
            return Error(MaxSeedLengthExceeded)

    hasher = SHA-256()
    for seed in seeds:
        hasher.update(seed)
    hasher.update(program_id)           // 32 bytes
    hasher.update("ProgramDerivedAddress")  // 21 bytes
    hash = hasher.finalize()            // 32 bytes

    if is_on_ed25519_curve(hash):
        return Error(InvalidSeeds)
    return hash as Address
```

The `"ProgramDerivedAddress"` suffix is a domain separator that prevents
collisions between PDAs and regular Ed25519 public keys. It also prevents
`create_with_seed` (a separate address derivation mechanism, not PDA-related)
from producing addresses that overlap with PDAs. `create_with_seed` uses
`SHA-256(base, seed, owner)` -- the domain separator prevents these two
derivation methods from producing overlapping addresses. The
[`create_with_seed`](https://github.com/anza-xyz/solana-sdk/blob/f2d15de6f7a1715ff806f0c39bba8f64bf6a587d/address/src/lib.rs#L245-L263)
function explicitly rejects owners whose bytes end with `"ProgramDerivedAddress"`.

### Bump seed search

[`find_program_address`](https://github.com/anza-xyz/solana-sdk/blob/f2d15de6f7a1715ff806f0c39bba8f64bf6a587d/address/src/syscalls.rs#L298-L341)
wraps `create_program_address` with a search loop. It appends a single-byte
"bump seed" to the user-provided seeds and tries values from 255 down to 1:

<Callout type="info">
  Both the on-chain syscall (`try_find_program_address`) and the off-chain SDK
  function (`find_program_address`) loop 255 times, trying bumps from 255 down
  to 1. Bump 0 is never tried by either implementation.
</Callout>

```
find_program_address(seeds, program_id):
    for bump in 255, 254, 253, ..., 2, 1:
        result = create_program_address([...seeds, [bump]], program_id)
        if result is Ok:
            return (result, bump)    // canonical bump found
    return Error                     // statistically near-impossible
```

The first bump value that produces an off-curve address is called the
**canonical bump**. Programs should always use the canonical bump to ensure a
unique, deterministic mapping from seeds to address. The search starts at 255
because approximately 50% of random 32-byte values are off the Ed25519 curve, so
the first attempt almost always succeeds, keeping CU costs minimal.

<Callout type="warn">
  Always use the canonical bump when deriving PDAs. Using a non-canonical bump
  creates a second valid address for the same seeds, which can lead to
  vulnerabilities where an attacker substitutes a different account than
  expected.
</Callout>

### Compute unit costs

The
[on-chain syscall](https://github.com/anza-xyz/agave/blob/v3.1.8/syscalls/src/lib.rs#L798-L834)
for `create_program_address` charges
[1,500 CUs](https://github.com/anza-xyz/agave/blob/v3.1.8/program-runtime/src/execution_budget.rs#L200)
per call.

The
[`try_find_program_address` syscall](https://github.com/anza-xyz/agave/blob/v3.1.8/syscalls/src/lib.rs#L836-L886)
charges 1,500 CUs on entry (before the loop), then an additional 1,500 CUs for
each failed bump attempt within the loop. If the canonical bump is 255, the
total cost is 1,500 CUs (entry only). If every bump fails, the worst case is
1,500 + 255 x 1,500 = 384,000 CUs.

Best practice: derive PDAs off-chain or in a client, store the canonical bump,
and use `create_program_address` with the known bump on-chain to pay a flat
1,500 CUs.

## Derive a PDA

Deriving a PDA computes an address only. It does not create an on-chain account
at that address. The account must be explicitly created through a separate
instruction (typically `create_account` via CPI).

The Solana SDKs provide functions for PDA derivation. Each function takes:

- **Program ID**: The address of the program used to derive the PDA. This
  program can sign on behalf of the PDA.
- **Optional seeds**: Predefined inputs such as strings, numbers, or other
  account addresses.

| SDK | Function |
| --- | --- |
| `@solana/kit` (TypeScript) | [`getProgramDerivedAddress`](https://github.com/anza-xyz/kit/blob/v2.1.0/packages/addresses/src/program-derived-address.ts#L157) |
| `@solana/web3.js` (TypeScript) | [`findProgramAddressSync`](https://github.com/solana-foundation/solana-web3.js/blob/v1.98.0/src/publickey.ts#L212) |
| `solana_sdk` (Rust) | [`find_program_address`](https://github.com/anza-xyz/solana-sdk/blob/sdk%40v2.2.2/pubkey/src/lib.rs#L617) |

Each function iterates through bump values starting at 255, decrementing by 1,
until a valid off-curve address is found. It returns both the PDA and the bump
seed used.

<Callout>
  The bump seed is an extra byte appended to the optional seeds to push the
  derived address off the Ed25519 curve.
</Callout>

![PDA Derivation](/assets/docs/core/pda/pda-derivation.svg)

### Canonical bump

The canonical bump is the highest bump value (starting from 255, decrementing)
that produces a valid off-curve address. `find_program_address` always returns
the canonical bump. Programs should always use the canonical bump to ensure a
unique, deterministic mapping from seeds to address.

The following examples show PDA derivation using all possible bump seeds (255 to
0):

<Callout type="info">
  Kit example is not included because the
  [`createProgramDerivedAddress`](https://github.com/anza-xyz/kit/blob/v2.1.0/packages/addresses/src/program-derived-address.ts#L101)
  function isn't exported.
</Callout>

<CodeTabs storage="pda-examples" flags="r">

```ts !! title="Legacy"
import { PublicKey } from "@solana/web3.js";

const programId = new PublicKey("11111111111111111111111111111111");
const optionalSeed = "helloWorld";

// Loop through all bump seeds (255 down to 0)
for (let bump = 255; bump >= 0; bump--) {
  try {
    const PDA = PublicKey.createProgramAddressSync(
      [Buffer.from(optionalSeed), Buffer.from([bump])],
      programId
    );
    console.log("bump " + bump + ": " + PDA);
  } catch (error) {
    console.log("bump " + bump + ": " + error);
  }
}
```

```rs !! title="Rust"
use solana_sdk::pubkey::Pubkey;
use std::str::FromStr;

#[tokio::main]

    let program_id = Pubkey::from_str("11111111111111111111111111111111")?;
    let optional_seed = b"helloWorld";

    // Loop through all bump seeds (255 down to 0)
    for bump in (0..=255).rev() {
        match Pubkey::create_program_address(&[optional_seed.as_ref(), &[bump]], &program_id) {
            Ok(pda) => println!("bump {}: {}", bump, pda),
            Err(err) => println!("bump {}: {}", bump, err),
        }
    }

    Ok(())
}
```

### Common seed patterns

Seeds are application-specific. Common patterns include:

| Pattern | Seeds | Use case |
| --- | --- | --- |
| Global singleton | `["global"]` | Single program-wide config account |
| Per-user account | `["user", user_pubkey]` | One account per user per program |
| Per-user-per-entity | `["vault", user_pubkey, mint_pubkey]` | Token vaults, per-user-per-token |
| Authority PDA | `["authority"]` | Program authority for signing CPIs |
| Counter / sequential | `["order", user_pubkey, &order_id.to_le_bytes()]` | Sequential records per user |

<Callout type="warn">
  Seeds are hashed sequentially, so `["ab", "cd"]` and `["abcd"]` produce
  the same PDA. Use a fixed-length seed or a separator to avoid collisions.
  For example, using `["ab", "-", "cd"]` is unambiguous.
</Callout>

## How `invoke_signed` works

When a program calls [`invoke_signed`](/docs/core/cpi) during a CPI, it passes
signer seeds to the runtime. The runtime uses these seeds to reconstruct the
PDA and verify that the calling program is authorized to sign for it. Here is the
exact verification flow:

1. The program calls
   [`invoke_signed`](https://github.com/anza-xyz/agave/blob/v3.1.8/program-runtime/src/cpi.rs#L802-L846)
   with the instruction, account infos, and an array of signer seed slices.
2. The runtime
   [translates the signer seeds](https://github.com/anza-xyz/agave/blob/v3.1.8/program-runtime/src/cpi.rs#L619-L661)
   from the program's memory and validates:
   - No more than
     [`MAX_SIGNERS` (16)](https://github.com/anza-xyz/agave/blob/v3.1.8/program-runtime/src/cpi.rs#L62)
     signer seed sets.
   - Each seed set has no more than `MAX_SEEDS` (16) individual seeds.
3. For each signer seed set, the runtime calls
   [`Pubkey::create_program_address`](https://github.com/anza-xyz/solana-sdk/blob/f2d15de6f7a1715ff806f0c39bba8f64bf6a587d/address/src/syscalls.rs#L390-L440)
   with the seeds and the **calling program's ID** to reconstruct the PDA.
4. The reconstructed PDA is added to the signers list.
5. When the runtime
   [checks privilege escalation](https://github.com/anza-xyz/agave/blob/v3.1.8/program-runtime/src/invoke_context.rs#L395-L399)
   for the callee instruction, an account marked as a signer is allowed if it
   was either already a signer in the caller's context **or** is present in the
   reconstructed signers list: `caller_is_signer || signers.contains(account_key)`.
6. If the account is not in either set, the runtime returns
   `InstructionError::PrivilegeEscalation`.

The critical security property: the runtime always uses the **calling program's
ID** (not an attacker-supplied program ID) when reconstructing the PDA. This
means only the program that actually derived the PDA can sign for it.

## Create a PDA account

Deriving a PDA and creating an account at a PDA are separate operations. You
must explicitly create the account after deriving the address.

To create an account at a PDA, the deriving program invokes the System Program's
`create_account` instruction via [`invoke_signed`](/docs/core/cpi), passing the
PDA's seeds so the runtime can verify the program's authority over that address.

<WithMentions>

The example below uses the [Anchor framework](https://www.anchor-lang.com/docs)
to create a new account with a program-derived address. The program includes a
single [`initialize`](mention:initialize) instruction to create the new account,
which will store the [user address](mention:user-address) and
[bump seed](mention:bump) used to derive the PDA.

```rs title="Program"
use anchor_lang::prelude::*;

declare_id!("75GJVCJNhaukaa2vCCqhreY31gaphv7XTScBChmr1ueR");

#[program]
pub mod pda_account {
    use super::*;

    // !mention initialize
    pub fn initialize(ctx: Context<Initialize>) -> Result<()> {
        let account_data = &mut ctx.accounts.pda_account;
        // store the address of the `user`
        // !mark
        // !mention user-address
        account_data.user = *ctx.accounts.user.key;
        // store the canonical bump
        // !mark
        // !mention bump
        account_data.bump = ctx.bumps.pda_account;
        Ok(())
    }
}

#[derive(Accounts)]
pub struct Initialize<'info> {
    #[account(mut)]
    // !mention user-address
    pub user: Signer<'info>,

    #[account(
        init,
        // define the seeds to derive the PDA
        // !mark
        // !mention user-address
        seeds = [b"data", user.key().as_ref()],
        // use the canonical bump
        // !mark
        // !mention bump
        bump,
        payer = user,
        space = 8 + DataAccount::INIT_SPACE // 8 bytes for Anchor account discriminator
    )]
    pub pda_account: Account<'info, DataAccount>,
    pub system_program: Program<'info, System>,
}

#[account]
#[derive(InitSpace)]
pub struct DataAccount {
    // !mark
    // !mention user-address
    pub user: Pubkey,
    // !mark
    // !mention bump
    pub bump: u8,
}
```

</WithMentions>

<WithMentions>

The [`init`](mention:init) constraint tells Anchor to
[invoke the System Program](/docs/core/programs#the-system-program) to create a
new account using the PDA as the address. The [seeds](mention:seeds) used to
create the PDA are:

- The fixed string: "data"
- The address of the user account provided in the instruction
- The canonical [bump seed](mention:bump)

In this example, the bump constraint is not assigned a value, so Anchor will use
`find_program_address` to derive the PDA and find the bump.

```rust title="pda_account"
#[account(
    // !mention init
    init,
    // !mention seeds
    seeds = [b"data", user.key().as_ref()],
    // !mention bump
    bump,
    payer = user,
    space = 8 + DataAccount::INIT_SPACE // 8 bytes for Anchor account discriminator
)]
pub pda_account: Account<'info, DataAccount>,
```

## Common pitfalls

- **Non-canonical bump**: Using a non-canonical bump allows multiple valid PDAs
  for the same seeds. Always use the canonical bump (the first one found by
  `find_program_address`) and store it for later verification.
- **Deriving does not create**: Deriving a PDA does not create an account. You
  must explicitly create the account via the System Program's `create_account`
  instruction.
- **Seed limits**: Seeds are limited to
  [`MAX_SEEDS` (16)](https://github.com/anza-xyz/solana-sdk/blob/f2d15de6f7a1715ff806f0c39bba8f64bf6a587d/address/src/lib.rs#L56),
  each up to
  [`MAX_SEED_LEN` (32 bytes)](https://github.com/anza-xyz/solana-sdk/blob/f2d15de6f7a1715ff806f0c39bba8f64bf6a587d/address/src/lib.rs#L54).
  Exceeding these limits causes `create_program_address` to return
  `AddressError::MaxSeedLengthExceeded`.
- **Seed collision**: Seeds are hashed sequentially so `["abc", "def"]` and
  `["abcdef"]` produce the same address. Use fixed-length seeds or separators
  to prevent collisions.
- **No external signing**: PDAs cannot sign transactions externally. Only the
  owning program can sign for a PDA via `invoke_signed` during a
  [CPI](/docs/core/cpi).
- **CU cost of `find_program_address`**: This function iterates from bump 255
  downward, costing up to 384,000 CUs in the worst case. Store the canonical
  bump and use `create_program_address` with the known bump to save CUs.
- **On-chain verification**: When verifying a PDA on-chain, use
  `create_program_address` with the stored bump instead of
  `find_program_address` to avoid unnecessary iteration.
