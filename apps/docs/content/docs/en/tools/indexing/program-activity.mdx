---
title: Index program activity
description:
  Capture events, logs, and account state from onchain programs - for your own
  program or third-party protocols you integrate.
---

Program activity indexing answers "what did this program do, and what state is
it in now?" - needed whether it's your program (user activity, analytics) or a
protocol you build on (fills, liquidations, pool state).

## Events and logs

Programs emit events as logs (Anchor's `emit!` serializes events into log
messages). Capture them by:

- Subscribing to [logsSubscribe](/docs/rpc/websocket/logssubscribe) with the
  program ID - real-time, but pair with polling backfill for gaps
- Polling [getSignaturesForAddress](/docs/rpc/http/getsignaturesforaddress) on
  the program ID and decoding logs from
  [getTransaction](/docs/rpc/http/gettransaction)

Decoding needs the program's IDL - see [IDLs](/docs/programs/idls) for how IDLs
describe instruction and event layouts, and
[Codama](/docs/programs/codama/clients) for generating typed clients from them.

## Account state

- Snapshot: [getProgramAccounts](/docs/rpc/http/getprogramaccounts) with
  size/memcmp filters (use sparingly on public RPC)
- Live: [programSubscribe](/docs/rpc/websocket/programsubscribe) /
  [accountSubscribe](/docs/rpc/websocket/accountsubscribe)
- Full-firehose: Geyser streaming - see the
  [indexing overview](/docs/tools/indexing#geyser-and-streaming-pipelines)

## Don't forget failures

Failed transactions still land onchain and carry logs. For trading systems
especially, index failures - they distinguish congestion loss from logic bugs.
Both polling and streaming approaches include failed transactions; filter on the
`err` field rather than discarding.
