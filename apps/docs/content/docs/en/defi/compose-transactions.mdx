---
title: Compose atomic transactions
description:
  Chain instructions across multiple programs in one all-or-nothing transaction
  — the primitive that makes Solana DeFi composable.
---

A Solana transaction is an ordered list of instructions, each potentially
targeting a different program, executed atomically: if any instruction fails,
the entire transaction reverts. This is the core DeFi primitive — flash-like
sequences (borrow → swap → repay), rebalances, and settlement flows don't need
special protocol support, just instruction composition.

## The mechanics

- Instructions execute in order within one transaction; each sees the state
  changes of the previous ones. See
  [Transactions and instructions](/docs/core/transactions).
- Programs can also invoke other programs directly via
  [cross-program invocation](/docs/core/cpi) when logic must be enforced onchain
  rather than assembled client-side.
- All accounts touched by all instructions must be declared up front — this is
  what lets the runtime parallelize non-conflicting transactions.

## Practical limits and how to work within them

| Limit                          | Work-around                                                                                                   |
| ------------------------------ | ------------------------------------------------------------------------------------------------------------- |
| Transaction size (1232 bytes)  | [Address lookup tables](/developers/cookbook/transactions/lookup-tables) to reference many accounts compactly |
| Compute budget per transaction | [Optimize compute](/developers/cookbook/transactions/optimize-compute); split flows that don't need atomicity |
| Legacy vs versioned formats    | Use [versioned transactions](/developers/cookbook/transactions/versions) — required for lookup tables         |

## Patterns

- **Client-side composition** — build the instruction list in your client and
  sign once. Right for user-facing flows and bots.
- **Program-side composition (CPI)** — a program orchestrates calls to other
  programs, enforcing invariants onchain. Right for vaults and anything holding
  user funds.
- **Durable offline flows** — when a transaction must be signed now and
  submitted later (custody approval chains), use
  [durable nonces](/developers/cookbook/transactions/durable-nonces).
