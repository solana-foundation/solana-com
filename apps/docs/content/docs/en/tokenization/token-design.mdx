---
title: Choose a token design
description:
  Match your asset type to the right Solana token configuration — token program,
  extensions, and authority structure — before you mint anything.
---

Every tokenization decision downstream — compliance, privacy, settlement,
indexing — is shaped by three choices you make up front: which token program to
use, which extensions to enable at mint creation, and who holds which
authorities. Extensions are set **when the mint is created** and most cannot be
added later, so it pays to design first.

## Which token program

|                   | Token Program                                            | Token-2022 (Token Extensions)                                                                                                                                                                                                          |
| ----------------- | -------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Best for          | Simple fungible tokens, maximum wallet/DEX compatibility | Stablecoins, RWAs, and any asset needing compliance or policy controls                                                                                                                                                                 |
| Transfer controls | None built in                                            | [Transfer hooks](/docs/tokens/extensions/transfer-hook), [permanent delegate](/docs/tokens/extensions/permanent-delegate), [default frozen state](/docs/tokens/extensions/default-state), [pausable](/docs/tokens/extensions/pausable) |
| Privacy           | None                                                     | [Confidential transfers](/docs/tokens/extensions/confidential-transfer)                                                                                                                                                                |
| Metadata          | External (Metaplex)                                      | [Native metadata extension](/docs/tokens/extensions/metadata)                                                                                                                                                                          |

For institutional assets, Token-2022 is almost always the right choice. See
[Tokens on Solana](/docs/tokens) for the underlying account model.

## Common designs by asset type

### Fiat-backed stablecoin

Mint and burn against reserves, freeze for sanctions compliance, pause in an
incident:

- [Metadata](/docs/tokens/extensions/metadata) for name, symbol, and issuer URI
- Freeze authority +
  [default account state](/docs/tokens/extensions/default-state) or
  [Token ACL](/docs/tokenization/token-acl) if holders must be approved
- [Pausable](/docs/tokens/extensions/pausable) for emergency stops
- [Permanent delegate](/docs/tokens/extensions/permanent-delegate) if you need
  seize/clawback powers
- Walkthrough: [Issue a stablecoin](/docs/tokenization/stablecoins)

### Money market fund / yield-bearing token

Share price changes over time; holders are usually KYC'd:

- [Scaled UI amount](/docs/tokens/extensions/scaled-ui-amount) or
  [interest-bearing](/docs/tokens/extensions/interest-bearing-tokens) to reflect
  NAV without rebasing every account
- Allowlist enforcement via [Token ACL](/docs/tokenization/token-acl)
- NAV update workflow: [NAV strikes](/docs/tokenization/nav-strikes)

### Tokenized security / RWA

Restricted transfers, reporting obligations, and settlement guarantees:

- [Token ACL](/docs/tokenization/token-acl) allowlists for eligible holders
- [Confidential transfers](/docs/tokenization/private-transfers) where position
  sizes are sensitive
- Atomic settlement patterns: [Delivery vs Payment](/docs/tokenization/dvp)
- Overview: [Tokenize a real-world asset](/docs/tokenization/real-world-assets)

## Authority structure

Decide before launch who controls each authority, and use multisigs (e.g.
[Squads](https://squads.so)) rather than single keys:

- **Mint authority** — can create supply. For stablecoins this is the issuance
  pipeline; consider a program-controlled PDA.
- **Freeze authority** — can freeze individual token accounts; required for
  sanctions response.
- **Extension authorities** — pause authority, permanent delegate, transfer hook
  program authority, and confidential transfer auditor are each set separately.

Key management and operational controls are covered in
[production readiness](/docs/tokenization/production-readiness).
