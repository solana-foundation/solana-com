h1: Crie Seu Primeiro Programa Solana
---

Este guia de início rápido usa o projeto inicial do
[framework Anchor](https://www.anchor-lang.com/docs) gerado por `anchor init`.
Você criará o projeto localmente, executará seus testes, compilará o programa e
percorrerá o código que o Anchor gera.

## Pré-requisitos

Antes de começar,
[instale as ferramentas de desenvolvimento Solana](/docs/intro/installation). A
instalação inclui Rust, a CLI do Solana e a CLI do Anchor.

Use a versão 1.1.2 ou superior da CLI do Anchor para este template. Verifique
sua versão instalada:

```terminal
$ anchor --version
```

<Steps>

<Step>

### Criar o projeto

Execute os seguintes comandos no seu terminal:

```terminal
$ anchor init my-program
$ cd my-program
```

O projeto inicial inclui um programa Solana em `programs/my-program`. O programa
inclui duas instruções: uma para inicializar uma conta contadora e outra para
incrementar o contador.

Algumas partes do template demonstram padrões comuns de programas Solana:
derivação de endereços de conta [PDA](/docs/core/pda), realização de um Cross
Program Invocation ([CPI](/docs/core/cpi)) para transferir SOL, e uso de
verificações de erro personalizadas para interromper uma instrução quando uma
condição falha.

<Step>

### Compilar o Programa

Execute `anchor build` para compilar o programa inicial:

```terminal
$ anchor build
```

O programa compilado é gravado em `target/deploy/my_program.so`. Quando o
programa é implantado, o conteúdo deste arquivo `.so` é armazenado em uma conta
na blockchain.

</Step>

<Step>

### Executar Teste

Execute o teste padrão:

```terminal
$ anchor test
```

O `Anchor.toml` deste template usa o comando de teste do Rust:

```toml title="Anchor.toml"
skip_local_validator = true

[scripts]
test = "cargo test"
```

O teste carrega o programa compilado no [LiteSVM](https://www.litesvm.com/),
cria um pagador, envia as instruções _rs`initialize`_ e _rs`increment`_, em
seguida verifica o estado da conta do contador.

Executar `anchor test` também compila o programa, portanto não é necessário
executar `anchor build` primeiro ao testar localmente.

</Step>

<Step>

### Implantar Programa

Os testes locais são o ciclo de feedback mais rápido. Quando estiver pronto para
implantar em uma rede, por exemplo devnet, primeiro faça o build e depois
implante em um cluster.

Implantar um programa Solana requer SOL porque o programa é armazenado em um
program account, e o program account deve pagar pelo espaço que utiliza. Na
devnet, solicite SOL gratuito da devnet no
[Solana Faucet](https://faucet.solana.com/) ou com a Solana CLI:

```terminal
$ solana airdrop 2 --url devnet
```

```terminal
$ anchor build
$ anchor deploy --provider.cluster devnet
```

</Step>

</Steps>

## Arquivos de origem

O diretório `src` contém o programa Solana. A documentação de
[Estrutura de Programa](https://www.anchor-lang.com/docs/basics/program-structure)
do Anchor explica as macros principais usadas aqui, incluindo _rs`declare_id!`_,
_rs`#[program]`_, _rs`#[derive(Accounts)]`_ e _rs`#[account]`_. Esta seção
percorre os arquivos do template.

### lib.rs

`lib.rs` é o ponto de entrada do programa. Ele conecta os arquivos de origem,
define o endereço do programa e define as instruções do programa que os usuários
podem chamar.

<WithNotes>

```rust title="programs/my-program/src/lib.rs" -w
// !tooltip[/pub mod constants/] constantsMod
pub mod constants;
// !tooltip[/pub mod error/] errorMod
pub mod error;
// !tooltip[/pub mod instructions/] instructionsMod
pub mod instructions;
// !tooltip[/pub mod state/] stateMod
pub mod state;

// !tooltip[/anchor_lang::prelude/] prelude
use anchor_lang::prelude::*;

pub use constants::*;
pub use instructions::*;
pub use state::*;

// !tooltip[/declare_id!/] declareId
declare_id!("82sFkffP9wxwpyfZyeaKHH2chQoJPUGsJZSPi9mrUuXd");

// !tooltip[/#\[program\]/] programMacro
#[program]
pub mod my_program {
    use super::*;

    // !tooltip[/pub fn initialize/] publicInstructions
    pub fn initialize(ctx: Context<Initialize>) -> Result<()> {
        // !tooltip[/handle_initialize/] handlers
        crate::instructions::initialize::handle_initialize(ctx)
    }

    // !tooltip[/pub fn increment/] publicInstructions
    pub fn increment(ctx: Context<Increment>) -> Result<()> {
        // !tooltip[/handle_increment/] handlers
        crate::instructions::increment::handle_increment(ctx)
    }
}
```

### !constantsMod

_rs`pub mod constants`_ inclui `constants.rs` no crate do programa. É aqui que o
template mantém valores compartilhados do programa, como o seed do PDA, o valor
de transferência e o valor máximo do contador.

### !errorMod

_rs`pub mod error`_ inclui `error.rs` no crate do programa. Este arquivo define
os erros personalizados que o programa pode retornar quando uma instrução é
inválida.

### !instructionsMod

_rs`pub mod instructions`_ inclui `instructions.rs` no crate do programa. Esse
arquivo funciona como um índice para os arquivos específicos de instrução em
`src/instructions/`.

### !stateMod

_rs`pub mod state`_ inclui `state.rs` no crate do programa. Este arquivo define
o layout de dados para contas pertencentes a este programa, incluindo a conta
_rs`Counter`_.

### !prelude

_rs`anchor_lang::prelude::*`_ importa os tipos e macros do Anchor usados para
definir um programa Solana, incluindo _rs`Context`_, _rs`Result`_, _rs`Pubkey`_,
_rs`Account`_, _rs`Signer`_ e atributos de conta do Anchor.

### !reexports

As linhas _rs`pub use`_ tornam públicos os itens dos módulos filhos a partir da
raiz do crate. Isso mantém os pontos de entrada das instruções concisos, como
_rs`Context<Initialize>`_, em vez de um caminho de módulo mais longo.

### !declareId

_rs`declare_id!`_ define o endereço do programa. Deve corresponder ao endereço
em `Anchor.toml`, ao program account implantado e a qualquer usuário ou teste
que chame o programa.

### !programMacro

_rs`#[program]`_ instrui o Anchor a gerar o entrypoint do programa Solana e o
código de despacho de instruções. As funções públicas neste módulo tornam-se as
instruções que os usuários podem chamar.

### !publicInstructions

_rs`initialize`_ e _rs`increment`_ são os dois nomes de instrução que os
usuários podem chamar. Cada um recebe um _rs`Context<T>`_. O tipo dentro de
_rs`Context`_ define as contas exigidas por essa instrução, e o Anchor valida
essas contas antes que a instrução seja executada.

### !handlers

A função handler contém a lógica da instrução.

</WithNotes>

### !left

```toml title="Anchor.toml" -w
[programs.localnet]
# !mark
my_program = "82sFkffP9wxwpyfZyeaKHH2chQoJPUGsJZSPi9mrUuXd"
```

### !right

```rust title="lib.rs" -w
// !mark

```

</SideBySide>

O mesmo endereço de programa aparece na configuração e no código. `Anchor.toml`
informa ao Anchor qual endereço implantar ou chamar para um cluster.
_rs`declare_id!`_ define o endereço do programa para verificações de segurança.

### constants.rs

`constants.rs` mantém valores compartilhados em um único lugar. Neste template,
_rs`COUNTER_SEED`_ deriva o PDA do contador, _rs`HELLO_WORLD_LAMPORTS`_ é
transferido durante a inicialização, e _rs`MAX_COUNT`_ é verificado antes de
incrementar.

<WithNotes>

```rust title="programs/my-program/src/constants.rs" -w
use anchor_lang::prelude::*;

// !tooltip[/#\[constant\]/] anchorConstant
// !tooltip[/COUNTER_SEED/] counterSeed
#[constant]
pub const COUNTER_SEED: &[u8] = b"counter";

// !tooltip[/HELLO_WORLD_LAMPORTS/] helloLamports
#[constant]
pub const HELLO_WORLD_LAMPORTS: u64 = 1;

// !tooltip[/MAX_COUNT/] maxCount
#[constant]
pub const MAX_COUNT: u64 = 10;
```

### !anchorConstant

_rs`#[constant]`_ instrui o Anchor a incluir o valor no IDL do programa. Isso
torna as constantes importantes do programa visíveis para clientes gerados e
ferramentas.

### !counterSeed

_rs`COUNTER_SEED`_ é a sequência de bytes usada para derivar o PDA do contador.
O programa e o teste usam esse mesmo seed para que ambos concordem com o
endereço da conta sem armazenar um keypair separado. Este template usa um seed
fixo, portanto o programa tem um único endereço de contador. Programas reais
frequentemente incluem o endereço de um usuário ou outro identificador nos seeds
para criar muitas contas relacionadas.

### !helloLamports

_rs`HELLO_WORLD_LAMPORTS`_ é a quantidade de lamports transferida do pagador
para a conta do contador dentro de _rs`initialize`_. É deliberadamente pequena
porque o template demonstra um CPI para o System Program.

### !maxCount

_rs`MAX_COUNT`_ é o limite superior verificado por _rs`increment`_. Mantê-lo
aqui dá ao programa, aos testes e ao IDL gerado um nome compartilhado para a
regra.

</WithNotes>

<SideBySide>

### !left

```rust title="initialize.rs" -w
#[derive(Accounts)]
pub struct Initialize<'info> {
    // ...
    #[account(
        init,
        payer = payer,
        space = 8 + Counter::INIT_SPACE,
        // !mark
        // !mention counter-seed
        seeds = [COUNTER_SEED],
        bump
    )]
    pub counter: Account<'info, Counter>,
    // ...
}

pub fn handle_initialize(ctx: Context<Initialize>) -> Result<()> {
    // ...

    // !mark
    // !mention hello-world-lamports
    anchor_lang::system_program::transfer(cpi_ctx, HELLO_WORLD_LAMPORTS)?;

    Ok(())
}
```

### !right

`initialize.rs` usa duas constantes:

- [_rs`COUNTER_SEED`_](mention:counter-seed) deriva o endereço PDA do contador.
- [_rs`HELLO_WORLD_LAMPORTS`_](mention:hello-world-lamports) define o valor
  transferido do pagador para a conta do contador.

</SideBySide>

<SideBySide>

### !left

```rust title="increment.rs" -w
pub fn handle_increment(ctx: Context<Increment>) -> Result<()> {
    // ...

    require!(
        // !mark
        // !mention max-count
        ctx.accounts.counter.count < MAX_COUNT,
        ErrorCode::CounterOverflow,
    );

    ctx.accounts.counter.count += 1;
    Ok(())
}
```

### !right

`increment.rs` usa [_rs`MAX_COUNT`_](mention:max-count) como limite superior do
contador. Se a contagem atual já estiver no máximo, _rs`require!`_ retorna
_rs`CounterOverflow`_ e os dados da conta não são alterados.

</SideBySide>

### state.rs

`state.rs` define tipos de dados personalizados para as contas que o programa
cria e gerencia. O programa define instruções para criar, inicializar e
atualizar esses dados, mas os dados do contador não são armazenados dentro do
próprio programa. Eles são armazenados em uma conta separada com seu próprio
endereço.

<WithNotes>

```rust title="programs/my-program/src/state.rs" -w
use anchor_lang::prelude::*;

// !tooltip[/#\[account\]/] accountMacro
#[account]
// !tooltip[/InitSpace/] initSpace
#[derive(InitSpace)]
pub struct Counter {
    // !tooltip[/count/] count
    pub count: u64,
    // !tooltip[/authority/] authority
    pub authority: Pubkey,
}
```

### !accountMacro

_rs`#[account]`_ informa ao Anchor que esta struct é um dado que pode ser
armazenado em uma conta onchain de propriedade deste programa. O Anchor a
utiliza para serializar, desserializar e validar contas _rs`Counter`_.

### !initSpace

_rs`InitSpace`_ permite que o Anchor calcule _rs`Counter::INIT_SPACE`_, o número
de bytes necessários para os campos desta struct. O discriminador de conta
Anchor de 8 bytes é adicionado separadamente quando a conta é criada.

### !count

_rs`count`_ é o valor do contador armazenado nos dados da conta.

### !authority

_rs`authority`_ armazena o endereço autorizado a incrementar o contador.
_rs`initialize`_ registra esse endereço, e _rs`increment`_ exige que esse mesmo
endereço assine.

</WithNotes>

<SideBySide>

### !left

```rust title="initialize.rs" -w
#[account(
    init,
    payer = payer,
    // !mark
    // !mention counter-space
    space = 8 + Counter::INIT_SPACE,
    seeds = [COUNTER_SEED],
    bump
)]
pub counter: Account<'info, Counter>,
// ...

pub fn handle_initialize(ctx: Context<Initialize>) -> Result<()> {
    // !mark(1:2)
    // !mention(1:2) initialize-fields
    ctx.accounts.counter.count = 0;
    ctx.accounts.counter.authority = ctx.accounts.payer.key();

    Ok(())
}
```

```rust title="increment.rs" -w
pub fn handle_increment(ctx: Context<Increment>) -> Result<()> {
    // ...

    // !mark
    // !mention increment-count
    ctx.accounts.counter.count += 1;
    Ok(())
}
```

### !right

`state.rs` define os dados da conta _rs`Counter`_. Os arquivos de instrução usam
esse tipo quando criam e atualizam a conta:

- [_rs`Counter::INIT_SPACE`_](mention:counter-space) dimensiona a conta para os
  campos definidos em `state.rs`.
- [_rs`count`_ e _rs`authority`_](mention:initialize-fields) são os valores dos
  campos gravados quando a conta é inicializada.
- [_rs`count += 1`_](mention:increment-count) atualiza o valor do contador
  armazenado após a validação ser concluída.

</SideBySide>

### error.rs

`error.rs` define os erros personalizados do programa. Neste template, os erros
demonstram como os handlers de instrução são interrompidos quando um chamador
não tem permissão para atualizar o contador ou quando o contador já atingiu
_rs`MAX_COUNT`_.

<WithNotes>

```rust title="programs/my-program/src/error.rs" -w
use anchor_lang::prelude::*;

// !tooltip[/#\[error_code\]/] errorCode
#[error_code]
pub enum ErrorCode {
    // !tooltip[/Unauthorized/] unauthorized
    #[msg("Only the counter authority can update this counter")]
    Unauthorized,
    // !tooltip[/CounterOverflow/] counterOverflow
    #[msg("Counter has reached the maximum value")]
    CounterOverflow,
}
```

### !errorCode

_rs`#[error_code]`_ instrui o Anchor a converter essas variantes em erros de
programa personalizados. Clientes e testes podem usar esses erros em vez de
tentar adivinhar a partir de códigos de erro brutos.

### !unauthorized

_rs`Unauthorized`_ é retornado quando o assinante da transação não é a
autoridade armazenada na conta do contador.

### !counterOverflow

_rs`CounterOverflow`_ é retornado quando um incremento violaria a regra
_rs`MAX_COUNT`_ do programa.

</WithNotes>

<SideBySide>

### !left

```rust title="increment.rs" -w
pub fn handle_increment(ctx: Context<Increment>) -> Result<()> {
    require_keys_eq!(
        ctx.accounts.counter.authority,
        ctx.accounts.authority.key(),
        // !mark
        // !mention unauthorized-error
        ErrorCode::Unauthorized,
    );
    require!(
        ctx.accounts.counter.count < MAX_COUNT,
        // !mark
        // !mention overflow-error
        ErrorCode::CounterOverflow,
    );

    ctx.accounts.counter.count += 1;
    msg!("Hello, world! Counter is now {}", ctx.accounts.counter.count);
    Ok(())
}
```

### !right

`error.rs` nomeia os erros que a instrução pode retornar:

- [_rs`ErrorCode::Unauthorized`_](mention:unauthorized-error) é retornado quando
  o assinante não é a autoridade armazenada na conta do contador.
- [_rs`ErrorCode::CounterOverflow`_](mention:overflow-error) é retornado quando
  o contador já atingiu _rs`MAX_COUNT`_.

</SideBySide>

### instructions.rs

`instructions.rs` conecta os arquivos de instrução ao crate do programa para que
`lib.rs` possa acessar o código de instrução _rs`initialize`_ e _rs`increment`_.
Cada arquivo de instrução define as contas exigidas por essa instrução e a
lógica do handler que é executada após o Anchor validar essas contas.

<WithNotes>

```rust title="programs/my-program/src/instructions.rs" -w
// !tooltip[/pub mod initialize/] moduleDeclarations
pub mod initialize;
pub mod increment;

// !tooltip[/pub use initialize/] reexports
pub use initialize::*;
pub use increment::*;
```

### !moduleDeclarations

As linhas _rs`pub mod`_ tornam `src/instructions/initialize.rs` e
`src/instructions/increment.rs` parte do crate do programa.

### !reexports

As linhas _rs`pub use`_ reexportam itens públicos desses arquivos. Isso torna as
estruturas de contas de instrução e as funções handler disponíveis através do
módulo _rs`instructions`_.

</WithNotes>

### initialize.rs

`initialize.rs` define as contas necessárias para criar a conta do contador e,
em seguida, grava os primeiros valores da conta. A struct
_rs`#[derive(Accounts)]`_ utiliza as
[restrições de conta](https://www.anchor-lang.com/docs/references/account-constraints)
do Anchor para especificar quais contas são necessárias e como a nova conta do
contador é criada.

<WithNotes>

```rust title="programs/my-program/src/instructions/initialize.rs" -w
use anchor_lang::prelude::*;

use crate::{constants::*, state::Counter};

// !tooltip[/derive\(Accounts\)/] accountsStruct
#[derive(Accounts)]
pub struct Initialize<'info> {
    // !tooltip[/#\[account\(mut\)\]/] payerMutable
    #[account(mut)]
    // !tooltip[/Signer<'info>/] payerSigner
    pub payer: Signer<'info>,
    // !tooltip[/#\[account\(/] counterConstraints
    #[account(
        // !tooltip[/init/] initConstraint
        init,
        // !tooltip[/payer = payer/] payerConstraint
        payer = payer,
        // !tooltip[/space = 8 \+ Counter::INIT_SPACE/] space
        space = 8 + Counter::INIT_SPACE,
        // !tooltip[/seeds = \[COUNTER_SEED\]/] seeds
        seeds = [COUNTER_SEED],
        // !tooltip[/bump/] bump
        bump
    )]
    // !tooltip[/Account<'info, Counter>/] counterAccountType
    pub counter: Account<'info, Counter>,
    // !tooltip[/Program<'info, System>/] systemProgram
    pub system_program: Program<'info, System>,
}

// !tooltip[/handle_initialize/] handler
pub fn handle_initialize(ctx: Context<Initialize>) -> Result<()> {
    // !tooltip[/counter.count = 0/] initialCount
    ctx.accounts.counter.count = 0;
    // !tooltip[/counter.authority = ctx.accounts.payer.key\(\)/] initialAuthority
    ctx.accounts.counter.authority = ctx.accounts.payer.key();

    // !tooltip[/system_program::Transfer/] cpiAccounts
    let cpi_accounts = anchor_lang::system_program::Transfer {
        from: ctx.accounts.payer.to_account_info(),
        to: ctx.accounts.counter.to_account_info(),
    };
    // !tooltip[/CpiContext::new/] cpiContext
    let cpi_ctx = CpiContext::new(anchor_lang::system_program::ID, cpi_accounts);
    // !tooltip[/system_program::transfer/] transfer
    anchor_lang::system_program::transfer(cpi_ctx, HELLO_WORLD_LAMPORTS)?;

    // !tooltip[/msg!/] programLog
    msg!("Hello, world! Counter initialized");
    Ok(())
}
```

## !!steps Account Context

A struct _rs`Initialize`_ define as contas que devem ser incluídas quando um
usuário chama a instrução _rs`initialize`_. O Anchor verifica essas contas antes
de o handler ser executado.

<CodePlaceholder title="initialize.rs" />

```rust !! title="initialize.rs"
// !focus(1:2)
#[derive(Accounts)]
pub struct Initialize<'info> {
    #[account(mut)]
    pub payer: Signer<'info>,
    #[account(
        init,
        payer = payer,
        space = 8 + Counter::INIT_SPACE,
        seeds = [COUNTER_SEED],
        bump
    )]
    pub counter: Account<'info, Counter>,
    pub system_program: Program<'info, System>,
}
```

## !!steps Payer Account

A conta _rs`payer`_ paga pela criação da conta do contador. O tipo
_rs`Signer<'info>`_ significa que o pagador deve assinar a transação, e
_rs`#[account(mut)]`_ indica que a conta do pagador pode ser alterada, pois
lamports serão deduzidos.

<CodePlaceholder title="initialize.rs" />

```rust !! title="initialize.rs"
#[derive(Accounts)]
pub struct Initialize<'info> {
    // !focus(1:2)
    #[account(mut)]
    pub payer: Signer<'info>,
    #[account(
        init,
        payer = payer,
        space = 8 + Counter::INIT_SPACE,
        seeds = [COUNTER_SEED],
        bump
    )]
    pub counter: Account<'info, Counter>,
    pub system_program: Program<'info, System>,
}
```

## !!steps Counter Account

A conta _rs`counter`_ armazena os dados _rs`Counter`_ de `state.rs`. _rs`init`_
instrui o Anchor a criar essa conta antes de o handler ser executado, e
_rs`payer = payer`_ informa ao Anchor qual conta paga pela criação.

<CodePlaceholder title="initialize.rs" />

```rust !! title="initialize.rs"
#[derive(Accounts)]
pub struct Initialize<'info> {
    #[account(mut)]
    pub payer: Signer<'info>,
    // !focus(1:8)
    #[account(
        init,
        payer = payer,
        space = 8 + Counter::INIT_SPACE,
        seeds = [COUNTER_SEED],
        bump
    )]
    pub counter: Account<'info, Counter>,
    pub system_program: Program<'info, System>,
}
```

## !!steps Tamanho da Conta

A restrição _rs`space`_ informa ao Anchor quanto espaço de dados da conta deve
ser alocado. O Anchor armazena primeiro um discriminador de 8 bytes, seguido dos
bytes necessários para os campos _rs`Counter`_. O discriminador permite que o
Anchor reconheça esta conta como uma conta _rs`Counter`_ antes de desserializar
os dados da conta.

<CodePlaceholder title="initialize.rs" />

```rust !! title="initialize.rs"
#[derive(Accounts)]
pub struct Initialize<'info> {
    #[account(mut)]
    pub payer: Signer<'info>,
    #[account(
        init,
        payer = payer,
        // !focus
        space = 8 + Counter::INIT_SPACE,
        seeds = [COUNTER_SEED],
        bump
    )]
    pub counter: Account<'info, Counter>,
    pub system_program: Program<'info, System>,
}
```

## !!steps Endereço do Contador

As restrições _rs`seeds`_ e _rs`bump`_ definem o endereço PDA esperado para a
conta do contador. O Anchor verifica se a conta _rs`counter`_ fornecida
corresponde a esse endereço. O template usa um PDA para que os utilizadores
possam derivar o endereço do contador a partir do ID do programa e do seed,

### !imports

Este arquivo importa as constantes compartilhadas e o tipo de conta
_rs`Counter`_. A instrução usa essas definições para derivar o PDA, escolher o
valor da transferência e alocar a conta com o tamanho de dados correto.

### !accountsStruct

_rs`Initialize`_ é o contexto de contas para a instrução _rs`initialize`_. Cada
campo na struct representa uma conta separada que deve ser incluída quando a
instrução é chamada. O Anchor usa a struct para validar essas contas recebidas
antes de o handler ser executado.

### !payerMutable

_rs`#[account(mut)]`_ informa ao Anchor que a conta do pagador deve ser
gravável. Isso é necessário porque a criação de conta e o CPI de transferência
ambos deduzem lamports do pagador.

### !payerSigner

_rs`Signer<'info>`_ informa ao Anchor que o pagador deve assinar a transação.

### !counterConstraints

O bloco _rs`#[account(...)]`_ informa ao Anchor como criar e validar a conta do
contador antes de o handler ser executado.

### !initConstraint

_rs`init`_ instrui o Anchor a criar a conta _rs`counter`_ antes de o handler ser
executado. A nova conta é de propriedade deste programa, portanto ele pode
armazenar dados _rs`Counter`_ nela.

### !payerConstraint

_rs`payer = payer`_ especifica a conta _rs`payer`_ como pagadora pela criação da
nova conta de contador.

### !space

_rs`space = 8 + Counter::INIT_SPACE`_ aloca bytes para o discriminador de conta
do Anchor mais os campos _rs`Counter`_ de `state.rs`. O discriminador ajuda o
Anchor a confirmar que a conta armazena o tipo esperado antes de lê-la.

### !seeds

_rs`seeds = [COUNTER_SEED]`_ determina que o endereço da conta do contador deve
ser o PDA derivado deste seed e do ID do programa. Com um único seed fixo, há um
único PDA de contador para este programa. Para criar contadores separados em um
programa real, você normalmente adicionaria outro seed, como o endereço de um
usuário.

### !bump

_rs`bump`_ instrui o Anchor a encontrar e validar o bump do PDA, que é o byte
extra que torna o endereço derivado válido como um PDA.

### !counterAccountType

_rs`Account<'info, Counter>`_ instrui o Anchor a tratar os dados da conta como o
tipo _rs`Counter`_ de `state.rs`. O Anchor também verifica se a conta é de
propriedade deste programa.

### !systemProgram

_rs`Program<'info, System>`_ exige que o chamador forneça o System Program. O
Anchor o utiliza para a restrição de criação de conta _rs`init`_. A
transferência CPI do handler também invoca o System Program.

### !handler

O handler é executado após o Anchor validar as contas e criar a conta do
contador, podendo assim escrever com segurança os dados iniciais da conta.

### !initialCount

Define o campo _rs`count`_ na conta do contador como _rs`0`_.

### !initialAuthority

Define o campo _rs`authority`_ na conta do contador como o endereço do pagador.

### !cpiAccounts

Estas são as contas passadas para a instrução de transferência do System
Program. Esta CPI de transferência é incluída apenas para demonstrar como uma
CPI passa contas para outro programa.

### !cpiContext

_rs`CpiContext`_ empacota o programa de destino e as contas necessárias para uma
Cross Program Invocation. A criação do contexto prepara a CPI ao associar o
programa que você deseja chamar com as contas que esse programa espera.

### !transfer

Esta CPI invoca a instrução de transferência do System Program e move
_rs`HELLO_WORLD_LAMPORTS`_ do pagador para a conta do contador.

### !programLog

_rs`msg!`_ escreve uma mensagem nos logs do programa quando o programa é
invocado.

### !litesvm

_rs`LiteSVM`_ é uma VM Solana em processo para testes. Permite que o teste
registre o programa compilado, crie contas, envie transações e inspecione o
estado das contas sem iniciar um processo de validator.

### !setup

O teste obtém o endereço do programa a partir do valor _rs`declare_id!`_ através
de _rs`my_program::id()`_ e cria um novo keypair de pagador para a transação de
teste.

### !counterPda

O endereço do contador é derivado do mesmo _rs`COUNTER_SEED`_ usado pelo
programa. Isso faz com que o teste aponte para o PDA que `initialize.rs` espera.

### !programBytes

_rs`include_bytes!`_ lê o arquivo `my_program.so` compilado gerado pelo build.
Esses bytes são o código executável do programa que o LiteSVM executará no
teste.

### !addProgram

_rs`svm.add_program`_ registra os bytes do programa compilado no endereço do
programa no LiteSVM. O airdrop fornece ao pagador SOL local suficiente para
pagar pela transação e pela criação da conta.

### !testInstruction

_rs`Instruction::new_with_bytes`_ cria uma instrução para invocar o programa.
Ela inclui o ID do programa, instruction data e as contas da instrução.

### !blockhash

Toda transação Solana precisa de um blockhash recente. Neste teste, o LiteSVM
fornece um blockhash para ser usado na transação.

### !message

A mensagem contém a instrução, o pagador da taxa e o blockhash. Estes são os
dados da transação que serão assinados.

### !transaction

_rs`VersionedTransaction::try_new`_ assina a mensagem com o keypair do pagador.
O pagador assina porque paga a taxa de transação e financia a nova conta do
contador.

### !sendTx

_rs`svm.send_transaction`_ executa a transação no LiteSVM.

### !getCounter

Após a execução da transação, o teste lê a conta do contador diretamente do
LiteSVM para inspecionar os dados da conta.

### !deserialize

A conta do contador armazena dados serializados. _rs`Counter::try_deserialize`_
verifica o discriminador de conta do Anchor e converte os dados da conta de
volta para o tipo _rs`Counter`_ de `state.rs`.

### !initAssert

Após _rs`initialize`_ ser concluído com sucesso, o teste desserializa a conta do
contador e verifica se o estado inicial corresponde ao que
_rs`handle_initialize`_ escreveu.

### !incrementAssert

Após _rs`increment`_ ser concluído com sucesso, a mesma conta do contador deve
ter _rs`count`_ definido como _rs`1`_ mantendo a mesma autoridade.

</WithNotes>
