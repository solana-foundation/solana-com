Os seguintes termos são usados em toda a documentação e ecossistema de
desenvolvimento da Solana.

## conta

Um registro no livro-razão da Solana que contém dados ou é um programa
executável.

Como uma conta em um banco tradicional, uma conta Solana pode conter fundos
chamados [lamports](#lamport). Como um arquivo no Linux, é endereçável por uma
chave, frequentemente referida como [chave pública](#public-key-pubkey) ou
pubkey.

A chave pode ser uma das seguintes:

- uma chave pública ed25519
- um endereço de conta derivado de programa (valor de 32 bytes forçado fora da
  curva ed25519)
- um hash de uma chave pública ed25519 com uma string de 32 caracteres

## proprietário da conta

O endereço do programa que possui a conta. Apenas o programa proprietário é
capaz de modificar a conta.

Veja também [autoridade](#authority).

## aplicativo

Uma aplicação front-end que interage com um cluster Solana.

## autoridade

O endereço de um usuário que tem algum tipo de permissão sobre uma conta.

Por exemplo:

- A capacidade de criar novos tokens é dada à conta que é a 'autoridade de
  emissão' para a emissão de token.
- A capacidade de atualizar um programa é dada à conta que é a 'autoridade de
  atualização' de um programa.

## estado do banco

O resultado da interpretação de todos os programas no livro-razão em uma
determinada [altura de tick](#tick-height). Inclui pelo menos o conjunto de
todas as [contas](#account) que possuem [tokens nativos](#native-token)
diferentes de zero.

## bloco

Um conjunto contíguo de [entradas](#entry) no ledger coberto por um
[voto](#ledger-vote). Um [líder](#leader) produz no máximo um bloco por
[slot](#slot).

## blockhash

Um valor único ([hash](#hash)) que identifica um registro (bloco). A Solana
calcula um blockhash a partir do último [id de entrada](#entry-id) do bloco.

## altura do bloco

O número de [blocos](#block) abaixo do bloco atual. O primeiro bloco após o
[bloco gênesis](#genesis-block) tem altura um.

## validator bootstrap

O [validator](#validator) que produz o bloco gênesis (primeiro) [bloco](#block)
de uma blockchain.

## BPF loader

O programa Solana que possui e carrega
[programas BPF](/docs/core/programs#berkeley-packet-filter-bpf)
[onchain](#onchain-program), permitindo que o programa interaja com o runtime.

## cliente

Um programa de computador que acessa a rede de servidores Solana
[cluster](#cluster).

## commitment

Uma medida da confirmação da rede para o [bloco](#block).

## formato de array compacto

Um array compacto é um array serializado no seguinte formato:

- O comprimento do array (codificado como
  [compact-u16](https://github.com/anza-xyz/agave/blob/v2.1.13/short-vec/src/lib.rs))
- Os itens do array listados um após o outro

![Formato de array compacto](/assets/docs/core/transactions/compact_array_format.png)

O formato de array compacto é usado para codificar tanto o array de
[endereços de conta](/docs/core/transactions#account-addresses) quanto o array
de [instruções](/docs/core/transactions#instructions) nas mensagens de
transação.

## cluster

Um conjunto de [validadores](#validator) mantendo um único [ledger](#ledger).

## orçamento computacional

O número máximo de [unidades computacionais](#compute-units) consumidas por
transação.

## unidades computacionais

A menor unidade de medida para o consumo de recursos computacionais da
blockchain. Unidades computacionais são usadas para medir os recursos
computacionais utilizados para processar uma transação.

## tempo de confirmação

A duração de tempo real entre um [líder](#leader) criar uma
[entrada de tick](#tick) e criar um [bloco confirmado](#confirmed-block).

## bloco confirmado

Um [bloco](#block) que recebeu uma [super maioria](#supermajority) de
[votos do ledger](#ledger-vote).

## plano de controle

Uma rede de gossip conectando todos os [nós](#node) de um [cluster](#cluster).

## período de resfriamento

Alguns números de [epochs](#epoch) após o [stake](#stake) ter sido desativado
enquanto progressivamente se torna disponível para retirada. Durante este
período, o stake é considerado como "desativando". Mais informações sobre:
[aquecimento e resfriamento](https://docs.anza.xyz/implemented-proposals/staking-rewards#stake-warmup-cooldown-withdrawal)

## unidades de custo

Também referido como `transaction cost`, é uma estimativa abrangente de todos os
recursos necessários para processar uma transação, medidos em
[unidades de computação](#compute-units). Mais informações
[aqui](https://www.anza.xyz/blog/why-solana-transaction-costs-and-compute-units-matter-for-developers).

## crédito

Veja [crédito de voto](#vote-credit).

## invocação entre programas (CPI)

Uma chamada de um [programa onchain](#onchain-program) para outro. Para mais
informações, veja [chamadas entre programas](/docs/core/cpi).

## plano de dados

Uma rede multicast usada para validar eficientemente [entradas](#entry) e obter
consenso.

## drone

Um serviço off-chain que atua como custódia da chave privada de um usuário.
Geralmente serve para validar e assinar transações.

## entrada

Uma entrada no [livro-razão](#ledger) que pode ser um [tick](#tick) ou uma
[entrada de transação](#transactions-entry).

## id de entrada

Um [hash](#hash) resistente à pré-imagem sobre o conteúdo final de uma entrada,
que atua como identificador globalmente único da [entrada](#entry). O hash serve
como evidência de:

- A entrada foi gerada após um período de tempo
- As [transações](#transaction) especificadas são aquelas incluídas na entrada
- A posição da entrada em relação a outras entradas no [livro-razão](#ledger)

Veja [prova de história](#proof-of-history-poh).

## epoch

O tempo, ou seja, o número de [slots](#slot), para o qual uma
[programação de líder](#leader-schedule) é válida.

## conta de taxa

A conta de taxa na transação é a conta que paga pelo custo de incluir a
transação no livro-razão. Esta é a primeira conta na transação. Esta conta deve
ser declarada como Leitura-Escrita (gravável) na transação, já que pagar pela
transação reduz o saldo da conta.

## finalidade

Quando nós representando 2/3 do [stake](#stake) têm uma [raiz](#root) comum.

## fork

Um [livro-razão](#ledger) derivado de entradas comuns, mas que depois divergiu.

## bloco gênesis

O primeiro [bloco](#block) na cadeia.

## configuração gênesis

O arquivo de configuração que prepara o [livro-razão](#ledger) para o
[bloco gênesis](#genesis-block).

## hash

Uma impressão digital digital de uma sequência de bytes.

## inflação

Um aumento no fornecimento de tokens ao longo do tempo usado para financiar
recompensas pela validação e para financiar o desenvolvimento contínuo da
Solana.

## instrução interna

Veja [invocação entre programas](#cross-program-invocation-cpi).

## instrução

Uma chamada para invocar um [manipulador de instrução](#instruction-handler)
específico em um [programa](#program). Uma instrução também especifica quais
contas deseja ler ou modificar, e dados adicionais que servem como entrada
auxiliar para o [manipulador de instrução](#instruction-handler). Um
[cliente](#client) deve incluir pelo menos uma instrução em uma
[transação](#transaction), e todas as instruções devem ser concluídas para que a
transação seja considerada bem-sucedida.

## manipulador de instrução

Manipuladores de instrução são funções de [programa](#program) que processam
[instruções](#instruction) de [transações](#transaction). Um manipulador de
instrução pode conter uma ou mais
[invocações entre programas](#cross-program-invocation-cpi).

## keypair

Uma [chave pública](#public-key-pubkey) e a correspondente
[chave privada](#private-key) para acessar uma conta.

## lamport

Um [token nativo](#native-token) fracionário com o valor de 0,000000001
[sol](#sol).

> Dentro do orçamento computacional, uma quantidade de
> _[micro-lamports](https://github.com/solana-labs/solana/blob/ced8f6a512c61e0dd5308095ae8457add4a39e94/program-runtime/src/prioritization_fee.rs#L1-L2)_
> é usada no cálculo de [taxas de priorização](#prioritization-fee).

## líder

O papel de um [validador](#validator) quando está anexando [entradas](#entry) ao
[ledger](#ledger).

## programação de líderes

Uma sequência de [chaves públicas](#public-key-pubkey) de
[validadores](#validator) mapeadas para [slots](#slot). O cluster usa a
programação de líderes para determinar qual validador é o [líder](#leader) em
qualquer momento.

## ledger

Uma lista de [entradas](#entry) contendo [transações](#transaction) assinadas
por [clientes](#client). Conceitualmente, isso pode ser rastreado até o
[bloco gênese](#genesis-block), mas o ledger de um [validador](#validator) real
pode ter apenas blocos mais recentes para reduzir o armazenamento, já que os
mais antigos não são necessários para a validação de blocos futuros por design.

## voto do ledger

Um [hash](#hash) do [estado do validador](#bank-state) em uma determinada
[altura de tick](#tick-height). Compreende uma afirmação de um
[validador](#validator) de que um [bloco](#block) que recebeu foi verificado,
bem como uma promessa de não votar em um [bloco](#block) conflitante (ou seja,
[fork](#fork)) por um período específico de tempo, o período de
[bloqueio](#lockout).

## light client

Um tipo de [cliente](#client) que pode verificar se está apontando para um
[cluster](#cluster) válido. Ele realiza mais verificações do ledger do que um
[thin client](#thin-client) e menos do que um [validator](#validator).

## loader

Um [programa](#program) com a capacidade de interpretar a codificação binária de
outros programas onchain.

## lockout

O período de tempo durante o qual um [validator](#validator) está
impossibilitado de [votar](#ledger-vote) em outro [fork](#fork).

## message

O conteúdo estruturado de uma [transação](#transaction). Geralmente contém um
cabeçalho, matriz de endereços de contas, [blockhash](#blockhash) recente e uma
matriz de [instruções](#instruction).

Saiba mais sobre a
[formatação de mensagens dentro das transações](/docs/core/transactions#message-header)
aqui.

## micro-lamport

1.000.000 micro-lamports = 1 lamport

## coeficiente de Nakamoto

Uma medida de descentralização, o Coeficiente de Nakamoto é o menor número de
entidades independentes que podem agir coletivamente para desligar uma
blockchain. O termo foi cunhado por Balaji S. Srinivasan e Leland Lee em
[Quantifying Decentralization](https://news.earn.com/quantifying-decentralization-e39db233c28e).

## token nativo

O [token](#token) usado para rastrear o trabalho realizado pelos [nós](#node) em
um cluster.

## node

Um computador que participa de um [cluster](#cluster).

## contagem de nós

O número de [validators](#validator) participando em um [cluster](#cluster).

## programa onchain

O código executável na blockchain Solana que interpreta as
[instruções](#instruction) enviadas dentro de cada [transação](#transaction)
para ler e modificar contas sobre as quais tem controle. Esses programas são
frequentemente referidos como "[_contratos inteligentes_](/docs/core/programs)"
em outras blockchains.

## PoH

Veja [Prova de História](#proof-of-history-poh).

## point

Um [crédito](#credit) ponderado em um regime de recompensas. No regime de
[recompensas](https://docs.anza.xyz/consensus/stake-delegation-and-rewards) do
[validator](#validator), o número de pontos devidos a um [stake](#stake) durante
o resgate é o produto dos [créditos de voto](#vote-credit) ganhos e o número de
lamports em stake.

## chave privada

A chave privada de um [keypair](#keypair).

## programa

Veja [programa onchain](#onchain-program).

## conta derivada de programa (PDA)

Uma conta cuja autoridade de assinatura é um programa e, portanto, não é
controlada por uma chave privada como outras contas.

## id do programa

A chave pública da [conta](#account) contendo um [programa](#program).

## prova de história (PoH)

Uma pilha de provas, cada uma das quais comprova que alguns dados existiam antes
da prova ser criada e que um período preciso de tempo passou antes da prova
anterior. Como um [VDF](#verifiable-delay-function-vdf), uma Prova de História
pode ser verificada em menos tempo do que levou para ser produzida.

## taxa de priorização

Uma taxa adicional que o usuário pode especificar na [instrução](#instruction)
de orçamento computacional para priorizar suas [transações](#transaction).

A taxa de priorização é calculada multiplicando as unidades de computação
máximas solicitadas pelo preço da unidade de computação (especificado em
incrementos de 0,000001 lamports por unidade de computação) arredondado para
cima para o lamport mais próximo.

As transações devem solicitar a quantidade mínima de unidades de computação
necessárias para a execução para minimizar as taxas.

## chave pública (pubkey)

A chave pública de um [keypair](#keypair).

## rent

Taxa paga por [Contas](#account) e [Programas](#program) para armazenar dados na
blockchain. Quando as contas não têm saldo suficiente para pagar o rent, elas
podem ser coletadas pelo Garbage Collector.

Veja também [isento de rent](#rent-exempt) abaixo. Saiba mais sobre rent aqui:
[O que é rent?](/docs/intro/rent).

## isento de rent

Contas que mantêm um saldo mínimo de lamports proporcional à quantidade de dados
armazenados na conta. Todas as contas recém-criadas são armazenadas
permanentemente na cadeia até que a conta seja fechada. Não é possível criar uma
conta que fique abaixo do limite de isenção de rent.

## root

Um [bloco](#block) ou [slot](#slot) que atingiu o máximo de [lockout](#lockout)
em um [validator](#validator). O root é o bloco mais alto que é ancestral de
todos os forks ativos em um validator. Todos os blocos ancestrais de um root
também são transitivamente um root. Blocos que não são ancestrais e nem
descendentes do root são excluídos da consideração para consenso e podem ser
descartados.

## runtime

O componente de um [validator](#validator) responsável pela execução do
[programa](#program).

## Sealevel

O runtime paralelo da Solana para [programas onchain](#onchain-program).

## shred

Uma fração de um [bloco](#block); a menor unidade enviada entre
[validators](#validator).

## signature

Uma assinatura ed25519 de 64 bytes de R (32 bytes) e S (32 bytes). Com o
requisito de que R seja um ponto Edwards compactado que não seja de ordem
pequena e S seja um escalar no intervalo de `0 <= S < L`. Este requisito garante
que não haja maleabilidade de assinatura. Cada transação deve ter pelo menos uma
assinatura para a [conta de taxa](#fee-account). Assim, a primeira assinatura na
transação pode ser tratada como [ID de transação](#transaction-id)

## skip rate

A porcentagem de [slots ignorados](#skipped-slot) do total de slots de líder no
epoch atual. Esta métrica pode ser enganosa, pois tem alta variância após o
limite do epoch quando o tamanho da amostra é pequeno, bem como para validators
com um baixo número de slots de líder, no entanto, também pode ser útil na
identificação de configurações incorretas de nós em determinados momentos.

## skipped slot

Um [slot](#slot) passado que não produziu um [bloco](#block), porque o líder
estava offline ou o [fork](#fork) contendo o slot foi abandonado por uma melhor
alternativa pelo consenso do cluster. Um slot ignorado não aparecerá como
ancestral para blocos em slots subsequentes, nem incrementará a
[altura do bloco](#block-height), nem expirará o `recent_blockhash` mais antigo.

Se um slot foi pulado só pode ser determinado quando ele se torna mais antigo
que o último slot [enraizado](#root) (portanto não pulado).

## slot

O período de tempo durante o qual cada [líder](#leader) ingere transações e
produz um [bloco](#block).

Coletivamente, os slots criam um relógio lógico. Os slots são ordenados
sequencialmente e não se sobrepõem, compreendendo aproximadamente o mesmo tempo
real conforme o [PoH](#proof-of-history-poh).

## contrato inteligente

Veja [programa onchain](#onchain-program).

## SOL

O [token nativo](#native-token) de um [cluster](#cluster) Solana.

## Biblioteca de Programas Solana (SPL)

Uma [biblioteca de programas](https://spl.solana.com/) na Solana, como o
spl-token, que facilita tarefas como criar e usar tokens.

## stake

Tokens confiscados para o [cluster](#cluster) se um comportamento malicioso do
[validador](#validator) puder ser comprovado.

## qualidade de serviço ponderada por stake (SWQoS)

SWQoS permite
[tratamento preferencial para transações que vêm de validadores com stake](/developers/guides/advanced/stake-weighted-qos).

## supermaioria

2/3 de um [cluster](#cluster).

## sysvar

Uma [conta](#account) do sistema.
[Sysvars](https://docs.anza.xyz/runtime/sysvars) fornecem informações sobre o
estado do cluster, como altura atual do tick, valores de [pontos](#point) de
recompensas, etc. Os programas podem acessar Sysvars através de uma conta Sysvar
(pubkey) ou consultando via syscall.

## cliente leve

Um tipo de [cliente](#client) que confia que está se comunicando com um
[cluster](#cluster) válido.

## tick

Uma [entrada](#entry) do ledger que estima a duração do tempo real.

## altura do tick

O enésimo [tick](#tick) no [ledger](#ledger).

## token

Um ativo digitalmente transferível.

## Token Extensions Program

O [Token Extensions Program](https://spl.solana.com/token-2022) tem o ID de
programa `TokenzQdBNbLqP5VEhdkAS6EPFLC1PHnBqCXEpPxuEb` e inclui todos os mesmos
recursos que o [Token Program](#token-program), mas vem com extensões como
transferências confidenciais, lógica de transferência personalizada, metadados
estendidos e muito mais.

## token mint

Uma conta que pode produzir (ou 'cunhar') tokens. Diferentes tokens são
distinguidos por seus endereços únicos de token mint.

## Token Program

O [Token Program](https://spl.solana.com/token) tem o ID de programa
`TokenkegQfeZyiNwAJbNbGKPFXCWuBvf9Ss623VQ5DA`, e fornece as capacidades básicas
de transferir, congelar e cunhar tokens.

## tps

[Transações](#transaction) por segundo.

## tpu

[Unidade de processamento de transações](https://docs.anza.xyz/validator/tpu).

## transaction

Uma ou mais [instruções](#instruction) assinadas por um [cliente](#client)
usando um ou mais [pares de chaves](#keypair) e executadas atomicamente com
apenas dois resultados possíveis: sucesso ou falha.

## transaction id

A primeira [assinatura](#signature) em uma [transação](#transaction), que pode
ser usada para identificar exclusivamente a transação em todo o
[ledger](#ledger).

## transaction confirmations

O número de [blocos confirmados](#confirmed-block) desde que a transação foi
aceita no [ledger](#ledger). Uma transação é finalizada quando seu bloco se
torna uma [raiz](#root).

## transactions entry

Um conjunto de [transações](#transaction) que podem ser executadas em paralelo.

## tvu

[Unidade de validação de transações](https://docs.anza.xyz/validator/tvu).

## validator

Um participante completo em uma rede Solana [cluster](#cluster) que produz novos
[blocos](#block). Um validator valida as transações adicionadas ao
[ledger](#ledger)

## VDF

Veja [função de atraso verificável](#verifiable-delay-function-vdf).

## função de atraso verificável (VDF)

Uma função que leva uma quantidade fixa de tempo para executar e produz uma
prova de que foi executada, que pode então ser verificada em menos tempo do que
levou para ser produzida.

## voto

Veja [voto do ledger](#ledger-vote).

## crédito de voto

Um registro de recompensas para [validators](#validator). Um crédito de voto é
concedido a um validator em sua conta de voto quando o validator alcança uma
[raiz](#root).

## carteira

Uma coleção de [pares de chaves](#keypair) que permite aos usuários gerenciar
seus fundos.

## período de aquecimento

Alguns números de [épocas](#epoch) após o [stake](#stake) ter sido delegado
enquanto progressivamente se torna efetivo. Durante este período, o stake é
considerado como "ativando". Mais informações sobre:
[aquecimento e resfriamento](https://docs.anza.xyz/consensus/stake-delegation-and-rewards#stake-warmup-cooldown-withdrawal)
