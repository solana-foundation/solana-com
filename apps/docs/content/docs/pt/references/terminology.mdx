Os seguintes termos são usados em toda a documentação e ecossistema de
desenvolvimento da Solana.

## conta

Um registro no livro-razão da Solana que contém dados ou é um programa
executável.

Como uma conta em um banco tradicional, uma conta Solana pode conter fundos
chamados [lamports](#lamport). Cada conta em um [cluster](#cluster) possui um
[endereço](#address) único.

## app

Uma aplicação front-end que interage com um cluster Solana.

## endereço

A localização na Solana que pode ser usada para encontrar uma conta. Cada
endereço é único no [cluster](#cluster) Solana.

Endereços no Solana podem ser uma [chave pública](#public-key-pubkey) Ed25519 ou
um [endereço derivado de programa](#program-derived-address-pda).

## tabela de pesquisa de endereços (ALT)

Uma estrutura de dados que armazena uma lista de endereços de contas, permitindo
que [transações versionadas](#versioned-transaction) referenciem contas usando
índices compactos de 1 byte em vez de chaves públicas completas de 32 bytes. As
tabelas de pesquisa de endereços permitem que uma única transação referencie
mais contas do que o formato de transação original permite.

## Anchor

Um framework para construir [programas onchain](#onchain-program) no Solana em
Rust. O Anchor fornece macros que reduzem código repetitivo, um sistema
padronizado de validação de contas e gera uma [IDL](#idl) que descreve a
interface do programa. É o framework mais amplamente utilizado para
desenvolvimento de programas no Solana.

## conta de token associada (ATA)

Uma [conta de token](#token-account) cujo endereço é derivado de forma
determinística a partir de um endereço de carteira e de um
[mint de token](#token-mint). O programa de conta de token associada garante que
cada carteira tenha uma conta de token canônica por tipo de token.

## autoridade

O endereço de um usuário que possui algum tipo de permissão sobre uma conta.

Por exemplo:

- A capacidade de criar novos tokens é concedida à conta que é a 'autoridade de
  cunhagem' para o mint do token.
- A capacidade de atualizar um programa é concedida à conta que é a 'autoridade
  de atualização' de um programa.

## formador de mercado automatizado (AMM)

Um modelo de exchange onchain onde os usuários negociam contra um pool de
reservas de tokens regido por uma fórmula de precificação, em vez de negociar
contra a ordem pendente de uma contraparte. Os preços se movem em função das
reservas, portanto as negociações incorrem em [slippage](#slippage) proporcional
ao tamanho.

## estado do banco

O resultado da interpretação de todos os programas no ledger em uma determinada
[altura de tick](#tick-height). Inclui pelo menos o conjunto de todas as
[contas](#account) que possuem [tokens nativos](#native-token) diferentes de
zero.

## bloco

Um conjunto contíguo de [entradas](#entry) no ledger coberto por um
[voto](#ledger-vote). Um [líder](#leader) produz no máximo um bloco por
[slot](#slot).

## blockhash

Um valor único ([hash](#hash)) que identifica um registro (bloco). O Solana
calcula um blockhash a partir do último [ID de entrada](#entry-id) do bloco.

## altura do bloco

O número de [blocos](#block) abaixo do bloco atual. O primeiro bloco após o
[bloco gênese](#genesis-block) tem altura um.

## validador de bootstrap

O [validator](#validator) que produz o bloco gênese (primeiro) [bloco](#block)
de uma blockchain.

## BPF loader

O programa Solana que possui e carrega [programas onchain](/docs/core/programs)
[BPF](#onchain-program), permitindo que o programa interaja com o runtime.

## bump

Um único byte (0–255) anexado às [seeds](#seed) ao derivar um
[Program Derived Address](#program-derived-address-pda) para garantir que o
endereço resultante saia da curva ed25519 e seja, portanto, um PDA válido. O
bump canônico é o valor mais alto que produz um PDA válido e normalmente é salvo
nos dados da conta para evitar recalculá-lo.

## livro de ordens de limite central (CLOB)

Um modelo de exchange que mantém uma lista ordenada de ofertas e pedidos
pendentes; uma negociação ocorre quando uma ordem recebida cruza com uma ordem
do lado oposto. Contraste com
[formador de mercado automatizado](#automated-market-maker-amm).

## cliente

Um programa de computador que acessa a [cluster](#cluster) da rede de servidores
Solana.

## compromisso

Uma medida de confirmação da rede para o [bloco](#block).

## formato de array compacto

Um array compacto é um array serializado no seguinte formato:

- O comprimento do array (codificado como
  [compact-u16](https://github.com/anza-xyz/agave/blob/v2.1.13/short-vec/src/lib.rs))
- Os itens do array listados um após o outro

![Formato de array compacto](/assets/docs/core/transactions/compact_array_format.png)

O formato de array compacto é usado para codificar tanto o array de
[endereços de conta](/docs/core/transactions/transaction-structure#account-addresses)
quanto o array de
[instruções](/docs/core/transactions/transaction-structure#instructions) nas
mensagens de transação.

## cluster

Um conjunto de [validadores](#validator) mantendo um único [ledger](#ledger).

## orçamento computacional

O número máximo de [unidades computacionais](#compute-units) consumidas por
transação.

## unidades computacionais

A menor unidade de medida para consumo de recursos computacionais da blockchain.
As unidades computacionais são usadas para medir os recursos computacionais
utilizados para processar uma transação.

## tempo de confirmação

A duração de tempo real entre um [líder](#leader) criar uma
[entrada de tick](#tick) e criar um [bloco confirmado](#confirmed-block).

## bloco confirmado

Um [bloco](#block) que recebeu uma [supermaioria](#supermajority) de
[votos do ledger](#ledger-vote).

## plano de controle

Uma rede de fofoca (gossip) conectando todos os [nós](#node) de uma
[cluster](#cluster).

## período de resfriamento

Um determinado número de [epochs](#epoch) após o [stake](#stake) ter sido
desativado, durante o qual ele progressivamente fica disponível para saque.
Durante este período, o stake é considerado como "desativando". Mais informações
sobre:
[aquecimento e resfriamento](https://docs.anza.xyz/consensus/stake-delegation-and-rewards#stake-warmup-cooldown-withdrawal)

## unidades de custo

Também referidas como `transaction cost`, é uma estimativa abrangente de todos
os recursos necessários para processar uma transação, medidos em
[unidades computacionais](#compute-units). Mais informações
[aqui](https://www.anza.xyz/blog/why-solana-transaction-costs-and-compute-units-matter-for-developers).

## crédito

Consulte [crédito de voto](#vote-credit).

## Cross Program Invocation (CPI)

Uma chamada de um [programa onchain](#onchain-program) para outro. Para mais
informações, consulte [chamadas entre programas](/docs/core/cpi).

## plano de dados

Uma rede multicast usada para validar [entradas](#entry) de forma eficiente e
obter consenso.

## drone

Um serviço offchain que atua como custodiante da chave privada de um usuário.
Ele tipicamente serve para validar e assinar transações.

## entrada

Uma entrada no [ledger](#ledger), seja um [tick](#tick) ou uma
[entrada de transação](#transactions-entry).

## id de entrada

Um [hash](#hash) resistente a pré-imagem sobre o conteúdo final de uma entrada,
que atua como o identificador globalmente único da [entrada](#entry). O hash
serve como evidência de:

- A entrada sendo gerada após um período de tempo
- As [transações](#transaction) especificadas são aquelas incluídas na entrada
- A posição da entrada em relação a outras entradas no [ledger](#ledger)

Consulte [prova de história](#proof-of-history-poh).

## epoch

O tempo, ou seja, número de [slots](#slot), durante o qual um
[cronograma de líder](#leader-schedule) é válido.

## custódia

Um padrão no qual um [programa onchain](#onchain-program) mantém tokens em um
[cofre](#vault) em nome de duas ou mais partes até que as condições definidas
pelo programa sejam atendidas, e então os libera. Usado para eliminar a
necessidade de as partes confiarem diretamente umas nas outras.

## conta de taxa

A conta de taxa na transação é a conta que paga pelo custo de incluir a
transação no ledger. Esta é a primeira conta na transação. Esta conta deve ser
declarada como Leitura-Escrita (gravável) na transação, pois pagar pela
transação reduz o saldo da conta.

## finalidade

Quando os nós que representam 2/3 do [stake](#stake) têm uma [raiz](#root)
comum.

## fork

Um [ledger](#ledger) derivado de entradas comuns mas que depois divergiu.

## bloco genesis

O primeiro [bloco](#block) na cadeia.

## configuração genesis

O arquivo de configuração que prepara o [ledger](#ledger) para o
[bloco genesis](#genesis-block).

## hash

Uma impressão digital de uma sequência de bytes.

## inflação

Um aumento no fornecimento de tokens ao longo do tempo usado para financiar
recompensas pela validação e para financiar o desenvolvimento contínuo da
Solana.

## instrução interna

Veja [Cross Program Invocation](#cross-program-invocation-cpi).

## instrução

Uma chamada para invocar um [manipulador de instruções](#instruction-handler)
específico em um [programa](#program). Uma instrução também especifica quais
contas deseja ler ou modificar, e dados adicionais que servem como entrada
auxiliar para o [manipulador de instruções](#instruction-handler). Um
[cliente](#client) deve incluir pelo menos uma instrução em uma
[transação](#transaction), e todas as instruções devem ser concluídas para que a
transação seja considerada bem-sucedida.

## manipulador de instruções

Manipuladores de instruções são funções de [programa](#program) que processam
[instruções](#instruction) de [transações](#transaction). Um manipulador de
instruções pode conter uma ou mais
[Cross Program Invocation](#cross-program-invocation-cpi).

## IDL

Interface Definition Language. Um arquivo JSON gerado pelo [Anchor](#anchor) que
descreve os [manipuladores de instruções](#instruction-handler), contas, e tipos
de um programa. O IDL permite que bibliotecas cliente gerem automaticamente
interfaces tipadas para interagir com um [programa onchain](#onchain-program).

## keypair

Uma [chave pública](#public-key-pubkey) e [chave privada](#private-key)
correspondente para acessar uma conta.

## lamport

Um [token nativo](#native-token) fracionário com o valor de 0,000000001
[SOL](#sol).

> Dentro do orçamento computacional, uma quantidade de
> _[micro-lamports](https://github.com/solana-labs/solana/blob/ced8f6a512c61e0dd5308095ae8457add4a39e94/program-runtime/src/prioritization_fee.rs#L1-L2)_
> é usada no cálculo das [taxas de priorização](#prioritization-fee).

## líder

O papel de um [validator](#validator) quando está anexando [entradas](#entry) ao
[ledger](#ledger).

## cronograma do líder

Uma sequência de [chaves públicas](#public-key-pubkey) de
[validator](#validator) mapeadas para [slots](#slot). O cluster usa o cronograma
do líder para determinar qual validator é o [líder](#leader) em qualquer
momento.

## ledger

Uma lista de [entradas](#entry) contendo [transações](#transaction) assinadas
por [clientes](#client). Conceitualmente, isso pode ser rastreado até o
[bloco gênesis](#genesis-block), mas o ledger de um [validator](#validator) real
pode ter apenas [blocos](#block) mais recentes para reduzir o armazenamento, já
que os mais antigos não são necessários para validação de blocos futuros por
design.

## voto do ledger

Um [hash](#hash) do [estado do validator](#bank-state) em uma determinada
[altura de tick](#tick-height). Compreende a afirmação de um
[validator](#validator) de que um [bloco](#block) que ele recebeu foi
verificado, bem como uma promessa de não votar em um [bloco](#block) conflitante
\(ou seja, [fork](#fork)\) por um período específico de tempo, o período de
[lockout](#lockout).

## cliente leve

Um tipo de [cliente](#client) que pode verificar se está apontando para um
[cluster](#cluster) válido. Ele executa mais verificação de ledger do que um
[cliente thin](#thin-client) e menos do que um [validator](#validator).

## liquidez

A profundidade de ordens ou reservas agrupadas disponíveis para negociação no
preço atual ou próximo dele. Uma liquidez mais profunda reduz o
[deslizamento](#slippage). Em um [CLOB](#central-limit-order-book-clob) a
liquidez é fornecida por [ordens maker](#maker); em um
[AMM](#automated-market-maker-amm) ela é fornecida por provedores de liquidez
que depositam pares de tokens no pool.

## carregador

Um [programa](#program) com a capacidade de interpretar a codificação binária de
outros [programas onchain](#onchain-program).

## lockout

A duração de tempo durante a qual um [validator](#validator) é incapaz de
[votar](#ledger-vote) em outro [fork](#fork).

## maker

Um trader cuja ordem adiciona [liquidez](#liquidity) a um
[CLOB](#central-limit-order-book-clob) ao permanecer no livro de ofertas em vez
de cruzar imediatamente o lado oposto. Tal ordem é chamada de ordem maker.
Makers geralmente não pagam taxa ou recebem um reembolso da plataforma.

## market maker

Uma entidade que fornece [liquidez](#liquidity) ao publicar continuamente tanto
compras quanto vendas. Na Solana, os market makers são programas (como
[AMMs](#automated-market-maker-amm)) ou empresas offchain executando bots que
publicam [ordens maker](#maker) em [CLOBs](#central-limit-order-book-clob).

## mensagem

O conteúdo estruturado de uma [transação](#transaction). Geralmente contém um
cabeçalho, array de endereços de contas, [blockhash](#blockhash) recente e um
array de [instruções](#instruction).

Saiba mais sobre a
[formatação de mensagens dentro de transações](/docs/core/transactions/transaction-structure#header)
aqui.

## micro-lamport

1.000.000 micro-lamports = 1 [lamport](#lamport)

## Coeficiente de Nakamoto

Uma medida de descentralização, o Coeficiente de Nakamoto é o menor número de
entidades independentes que podem agir coletivamente para desligar uma
blockchain. O termo foi cunhado por Balaji S. Srinivasan e Leland Lee em
[Quantifying Decentralization](https://news.earn.com/quantifying-decentralization-e39db233c28e).

## token nativo

O [token](#token) usado para rastrear o trabalho realizado por [nós](#node) em
um cluster.

## nó

Um computador participando de um [cluster](#cluster).

## contagem de nós

O número de [validadores](#validator) participando de um [cluster](#cluster).

## offchain

Descreve dados, processos ou serviços que existem ou executam fora da blockchain
Solana, como clientes RPC, indexadores, frontends e criadores de livros de
ofertas offchain. Complemento de [onchain](#onchain). "Offchain" (sem hífen) é a
grafia preferida.

## onchain

Descreve dados ou programas que existem ou executam em uma blockchain. "Onchain"
(sem hífen) é a grafia preferida.

## programa onchain

O código executável na blockchain Solana que interpreta as
[instruções](#instruction) enviadas dentro de cada [transação](#transaction)
para ler e modificar contas sobre as quais tem controle. Esses programas são
frequentemente referidos como "[_contratos inteligentes_](/docs/core/programs)"
em outras blockchains.

## oráculo

Um [programa onchain](#onchain-program) que publica dados de fora da cadeia — na
maioria das vezes preços de ativos do mundo real — para que outros programas
possam ler.

## proprietário

Pode referir-se ao [programa proprietário](#owning-program) ou à
[carteira proprietária](#owning-wallet). Por exemplo,
[getTokenAccountsByOwner](https://solana.com/docs/rpc/http/gettokenaccountsbyowner)
obtém token accounts de propriedade de uma carteira específica, no entanto o
programa proprietário dessas token accounts é o [Token Program](#token-program).

Para evitar confusão, é preferível usar [programa proprietário](#owning-program)
ou [carteira proprietária](#owning-wallet) em vez do termo genérico
'proprietário'.

## programa proprietário

O [endereço](#address) do [programa](#onchain-program) que possui a conta.
Apenas o programa proprietário é capaz de modificar a conta.

Veja também [autoridade](#authority).

## carteira proprietária

A carteira que está associada a uma conta. Por exemplo, o token program é usado
para armazenar token accounts associadas a uma carteira específica e mint usando
um endereço criado a partir dessa [carteira](#wallet) e [mint](#token-mint).

## PoH

Veja [Prova de História](#proof-of-history-poh).

## ponto

Um [crédito](#credit) ponderado em um regime de recompensas. No regime de
[recompensas](https://docs.anza.xyz/consensus/stake-delegation-and-rewards) do
[validator](#validator), o número de pontos devidos a um [stake](#stake) durante
o resgate é o produto dos [créditos de voto](#vote-credit) ganhos e o número de
[lamports](#lamport) em stake.

## chave privada

A chave privada de um [keypair](#keypair).

## programa

Veja [programa onchain](#onchain-program).

## Program Derived Address (PDA)

Uma conta cujo endereço é criado com base em um
[programa onchain](#onchain-program) e uma ou mais [seeds](#seed) criadas pelo
autor do programa. Endereços PDA não são chaves públicas, e PDAs não possuem
chaves privadas.

A autoridade de assinatura para um PDA é o [programa onchain](#onchain-program)
que é seu [programa proprietário](#owning-program).

## program id

A chave pública da [conta](#account) que contém um [programa](#onchain-program).

## proof of history (PoH)

Uma pilha de provas, cada uma das quais comprova que alguns dados existiam antes
da criação da prova e que um período preciso de tempo passou antes da prova
anterior. Como uma [VDF](#verifiable-delay-function-vdf), uma Proof of History
pode ser verificada em menos tempo do que levou para ser produzida.

## prioritization fee

Uma taxa adicional que o usuário pode especificar na [instrução](#instruction)
de orçamento computacional para priorizar suas [transações](#transaction).

A taxa de priorização é calculada multiplicando as unidades computacionais
máximas solicitadas pelo preço por unidade computacional (especificado em
incrementos de 0,000001 [lamport](#lamport) por unidade computacional)
arredondado para cima até o [lamport](#lamport) mais próximo.

As transações devem solicitar a quantidade mínima de unidades computacionais
necessárias para execução a fim de minimizar as taxas.

## public key (pubkey)

A chave pública de um [keypair](#keypair). Uma chave pública é um dos diferentes
tipos de [endereço](#address) que uma conta pode usar na Solana.

## rent

Taxa paga por [Contas](#account) e [Programas](#program) para armazenar dados na
blockchain. Quando as contas não possuem saldo suficiente para pagar o rent,
elas podem ser removidas pela Coleta de Lixo.

Veja também [rent exempt](#rent-exempt) abaixo. Saiba mais sobre rent aqui:
[O que é rent?](/docs/core/accounts/account-structure).

## rent exempt

Contas que mantêm um saldo mínimo de lamport proporcional à quantidade de dados
armazenados na conta. Todas as contas recém-criadas são armazenadas onchain
permanentemente até que a conta seja fechada. Não é possível criar uma conta que
fique abaixo do limite de isenção de rent.

## root

Um [bloco](#block) ou [slot](#slot) que atingiu o [lockout](#lockout) máximo em
um [validator](#validator). O root é o bloco mais alto que é ancestral de todos
os forks ativos em um validator. Todos os blocos ancestrais de um root também
são transitivamente um root. Blocos que não são ancestrais nem descendentes do
root são excluídos da consideração para consenso e podem ser descartados.

## runtime

O componente de um [validator](#validator) responsável pela execução de
[programas](#program).

## Sealevel

O ambiente de execução paralela da Solana para
[programas onchain](#onchain-program).

## seed

Um valor em bytes ou array de valores em bytes usado como entrada ao derivar um
[endereço derivado de programa](#program-derived-address-pda). Um programa pode
usar qualquer combinação de strings estáticas, chaves públicas ou outros dados
como seeds. Os seeds são combinados com o ID do programa e um [bump](#bump) para
produzir um endereço PDA único.

## shred

Uma fração de um [bloco](#block); a menor unidade enviada entre
[validators](#validator).

## signature

Uma assinatura ed25519 de 64 bytes de R (32 bytes) e S (32 bytes). Com o
requisito de que R seja um ponto Edwards compactado não de ordem pequena e S
seja um escalar no intervalo de `0 <= S < L`. Este requisito garante que não
haja maleabilidade de assinatura. Cada transação deve ter pelo menos uma
assinatura para a [conta de taxa](#fee-account). Assim, a primeira assinatura na
transação pode ser tratada como [ID de transação](#transaction-id)

## signer

Uma [conta](#account) que autorizou uma [instrução](#instruction) ou
[transação](#transaction). Para um [keypair](#keypair) regular, isso significa
uma [assinatura](#signature) criptográfica sobre a transação. Para um
[endereço derivado de programa](#program-derived-address-pda), a autorização é
afirmada pelo [programa proprietário](#owning-program) quando ele faz uma
[invocação entre programas](#cross-program-invocation-cpi), já que um PDA não
possui chave privada.

## taxa de pulos

A porcentagem de [slots pulados](#skipped-slot) do total de slots de líder na
epoch atual. Esta métrica pode ser enganosa, pois apresenta alta variância após
o limite da epoch quando o tamanho da amostra é pequeno, bem como para
validadores com um número baixo de slots de líder, mas também pode ser útil para
identificar configurações incorretas do nó em determinados momentos.

## slot pulado

Um [slot](#slot) passado que não produziu um [bloco](#block), porque o líder
estava offline ou o [fork](#fork) contendo o slot foi abandonado em favor de uma
alternativa melhor pelo consenso do cluster. Um slot pulado não aparecerá como
ancestral para blocos em slots subsequentes, nem incrementará a
[altura do bloco](#block-height), nem expirará o `recent_blockhash` mais antigo.

Se um slot foi pulado só pode ser determinado quando ele se tornar mais antigo
que o último slot [enraizado](#root) (portanto, não pulado).

## derrapagem

A diferença entre o preço que um trader esperava e o preço pelo qual sua
negociação realmente foi executada. Em [AMMs](#automated-market-maker-amm),
surge do movimento da curva de precificação conforme as reservas mudam; em
[CLOBs](#central-limit-order-book-clob), surge de uma única ordem consumindo
várias ordens em espera a preços sucessivamente piores.

## slot

O período de tempo durante o qual cada [líder](#leader) ingere transações e
produz um [bloco](#block).

Coletivamente, os slots criam um relógio lógico. Os slots são ordenados
sequencialmente e não se sobrepõem, compreendendo aproximadamente tempo real
equivalente conforme [PoH](#proof-of-history-poh).

## contrato inteligente

Veja [programa onchain](#onchain-program).

## SOL

O [token nativo](#native-token) de um [cluster](#cluster) Solana.

## Solana Program Library (SPL)

Uma [biblioteca de programas](https://www.solana-program.com/docs/) na Solana,
como o spl-token, que facilita tarefas como criar e usar tokens.

## stake

Tokens perdidos para o [cluster](#cluster) se um comportamento malicioso de
[validator](#validator) puder ser comprovado.

## qualidade de serviço ponderada por stake (SWQoS)

O SWQoS permite
[tratamento preferencial para transações provenientes de validadores com stake](/developers/guides/advanced/stake-weighted-qos).

## supermaioria

2/3 de um [cluster](#cluster).

## System Program

Um programa nativo da Solana (ID do programa `11111111111111111111111111111111`)
responsável por criar novas [contas](#account), alocar dados de contas, atribuir
contas a programas e transferir [SOL](#sol) entre contas. Todas as novas contas
na Solana são de propriedade do System Program até serem reatribuídas.

## sysvar

Uma [conta](#account) do sistema.
[Sysvars](https://docs.anza.xyz/runtime/sysvars) fornecem informações sobre o
estado do cluster, como a altura atual do tick, valores de [pontos](#point) de
recompensa, etc. Os programas podem acessar Sysvars através de uma conta Sysvar
(pubkey) ou consultando via syscall.

## taker

Um trader cuja ordem remove [liquidez](#liquidity) de um
[CLOB](#central-limit-order-book-clob) ao cruzar o lado oposto do livro de
ofertas e corresponder a uma ou mais ordens em repouso. Essa ordem é chamada de
ordem taker, e o taker normalmente paga a taxa de negociação da plataforma.

## cliente leve

Um tipo de [cliente](#client) que confia estar se comunicando com um
[cluster](#cluster) válido.

## tick

Uma [entrada](#entry) do ledger que estima a duração do relógio de parede.

## altura do tick

O enésimo [tick](#tick) no [ledger](#ledger).

## token

Um ativo transferível digitalmente.

## token account

Uma [conta](#account) que mantém um saldo de um [token](#token) específico. As
token accounts são de propriedade do [Token Program](#token-program) ou
[Token Extensions Program](#token-extensions-program) e armazenam o saldo do
token, o [token mint](#token-mint) associado e o endereço da carteira
proprietária.

## Token Extensions Program

O [Token Extensions Program](https://www.solana-program.com/docs/token-2022) tem
o ID de programa `TokenzQdBNbLqP5VEhdkAS6EPFLC1PHnBqCXEpPxuEb` e inclui todos os
mesmos recursos do [Token Program](#token-program), mas vem com extensões como
transferências confidenciais, lógica de transferência personalizada, metadados
estendidos e muito mais.

## token mint

Uma conta que pode produzir (ou 'cunhar') tokens. Tokens diferentes são
distinguidos pelos seus endereços únicos de token mint.

## Token Program

O [Token Program](https://www.solana-program.com/docs/token) possui o ID de
programa `TokenkegQfeZyiNwAJbNbGKPFXCWuBvf9Ss623VQ5DA` e fornece as capacidades
básicas de transferência, congelamento e cunhagem de tokens.

## tps

[Transações](#transaction) por segundo.

## tpu

[Unidade de processamento de transações](https://docs.anza.xyz/validator/tpu).

## transação

Uma ou mais [instruções](#instruction) assinadas por um [cliente](#client)
usando um ou mais [keypairs](#keypair) e executadas atomicamente com apenas dois
resultados possíveis: sucesso ou falha.

## id de transação

A primeira [assinatura](#signature) em uma [transação](#transaction), que pode
ser usada para identificar exclusivamente a transação em todo o
[livro-razão](#ledger).

## confirmações de transação

O número de [blocos confirmados](#confirmed-block) desde que a transação foi
aceita no [livro-razão](#ledger). Uma transação é finalizada quando seu bloco se
torna uma [raiz](#root).

## entrada de transações

Um conjunto de [transações](#transaction) que podem ser executadas em paralelo.

## tvu

[Unidade de validação de transações](https://docs.anza.xyz/validator/tvu).

## validator

Um participante completo em um [cluster](#cluster) de rede Solana que produz
novos [blocos](#block). Um validator valida as transações adicionadas ao
[livro-razão](#ledger)

## cofre

Uma [token account](#token-account) pertencente a um
[programa onchain](#onchain-program) que mantém ativos agrupados em nome dos
usuários — por exemplo, as reservas base e de cotação de um
[AMM](#automated-market-maker-amm), os tokens bloqueados em um
[escrow](#escrow), ou os saldos em repouso e taxas acumuladas de um
[CLOB](#central-limit-order-book-clob). Apenas o programa proprietário pode
mover tokens para fora do cofre, geralmente para a própria
[token account](#token-account) de um usuário quando uma condição de negociação
ou liberação é atendida.

## VDF

Ver [função de atraso verificável](#verifiable-delay-function-vdf).

## função de atraso verificável (VDF)

Uma função que leva um tempo fixo para executar e que produz uma prova de que
foi executada, a qual pode ser verificada em menos tempo do que levou para ser
produzida.

## transação versionada

Um formato de transação que suporta recursos adicionais além do formato de
[transação](#transaction) original. A versão 0 (`v0`) é o primeiro formato
versionado e adiciona suporte para
[tabelas de pesquisa de endereços](#address-lookup-table-alt), permitindo que as
transações referenciem mais contas do que o formato original permite.

## voto

Consulte [voto do ledger](#ledger-vote).

## crédito de voto

Uma contagem de recompensas para [validadores](#validator). Um crédito de voto é
concedido a um validator em sua conta de voto quando o validator atinge uma
[raiz](#root).

## carteira

Uma coleção de [pares de chaves](#keypair) que permite aos usuários gerenciar
seus fundos.

## período de aquecimento

Um certo número de [épocas](#epoch) após o [stake](#stake) ter sido delegado,
durante as quais ele progressivamente se torna efetivo. Durante este período, o
stake é considerado como estando "ativando". Mais informações sobre:
[aquecimento e resfriamento](https://docs.anza.xyz/consensus/stake-delegation-and-rewards#stake-warmup-cooldown-withdrawal)
