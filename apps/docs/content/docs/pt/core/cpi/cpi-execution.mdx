<Callout type="info" title="Resumo">
  CPIs passam por 11 etapas de runtime incluindo verificação de privilégios,
  tradução de contas e sincronização de dados. Profundidade máxima de chamada: 5
  (9 com SIMD-0268). Regras de privilégio impedem que o programa chamado escale
  além do que o chamador concedeu.
</Callout>

## Regras de privilégio

CPIs estendem os privilégios de conta do chamador para o programa chamado com
aplicação rigorosa. O runtime verifica essas regras em
[`prepare_next_instruction`](https://github.com/anza-xyz/agave/blob/v3.1.8/program-runtime/src/invoke_context.rs#L293-L432):

| Cenário                                                                                                                         | Permitido? | Ponto de aplicação                                                                                                          | Erro                   |
| ------------------------------------------------------------------------------------------------------------------------------- | ---------- | --------------------------------------------------------------------------------------------------------------------------- | ---------------------- |
| Chamador passa conta como gravável, programa chamado marca como gravável                                                        | Sim        | --                                                                                                                          | --                     |
| Chamador passa conta como somente leitura, programa chamado marca como gravável                                                 | Não        | [`prepare_next_instruction`](https://github.com/anza-xyz/agave/blob/v3.1.8/program-runtime/src/invoke_context.rs#L388-L390) | `PrivilegeEscalation`  |
| Chamador passa conta como gravável, programa chamado marca como somente leitura                                                 | Sim        | --                                                                                                                          | --                     |
| Chamador passa conta como signatário, programa chamado marca como signatário                                                    | Sim        | --                                                                                                                          | --                     |
| Chamador passa conta como não signatário, programa chamado marca como signatário, conta é um PDA derivado das seeds do chamador | Sim        | [`prepare_next_instruction`](https://github.com/anza-xyz/agave/blob/v3.1.8/program-runtime/src/invoke_context.rs#L395-L399) | --                     |
| Chamador passa conta como não signatário, programa chamado marca como signatário, conta NÃO é um PDA do chamador                | Não        | [`prepare_next_instruction`](https://github.com/anza-xyz/agave/blob/v3.1.8/program-runtime/src/invoke_context.rs#L395-L399) | `PrivilegeEscalation`  |
| Chamador passa conta como signatário, programa chamado marca como não signatário                                                | Sim        | --                                                                                                                          | --                     |
| Programa A chama a si mesmo diretamente (A -> A)                                                                                | Sim        | [`push()`](https://github.com/anza-xyz/agave/blob/v3.1.8/program-runtime/src/invoke_context.rs#L243-L259)                   | --                     |
| Programa A chama B que chama A (reentrância indireta)                                                                           | Não        | [`push()`](https://github.com/anza-xyz/agave/blob/v3.1.8/program-runtime/src/invoke_context.rs#L243-L259)                   | `ReentrancyNotAllowed` |
| CPI para native loader, bpf_loader, bpf_loader_deprecated ou precompile                                                         | Não        | [`check_authorized_program`](https://github.com/anza-xyz/agave/blob/v3.1.8/program-runtime/src/cpi.rs#L193-L222)            | `ProgramNotSupported`  |
| Conta não encontrada na transação                                                                                               | Não        | [`prepare_next_instruction`](https://github.com/anza-xyz/agave/blob/v3.1.8/program-runtime/src/invoke_context.rs#L310-L321) | `MissingAccount`       |

As regras de privilégio podem ser resumidas como:

1. **Privilégio de escrita não pode escalar.** Se o chamador marcar uma conta
   como somente leitura, o chamado não pode marcá-la como gravável.
2. **Privilégio de assinante requer autorização.** Uma conta pode ser assinante
   no chamado apenas se (a) já era assinante no chamador, OU (b) é um PDA
   derivado das seeds do programa chamador via _rs`invoke_signed`_.
3. **Redução de privilégio é sempre permitida.** O chamado pode usar menos
   privilégios do que o chamador concedeu.

## Fluxo de execução CPI

Um CPI passa por várias camadas de runtime. Esta seção documenta o pipeline
completo desde a chamada SDK do programa através do limite de syscall até o
runtime e de volta. Cada etapa referencia o arquivo fonte que a implementa.

<Callout>
  A altura máxima da invocação de instrução do programa é chamada de
  [`max_instruction_stack_depth`](https://github.com/anza-xyz/agave/blob/v3.1.8/program-runtime/src/execution_budget.rs#L66)
  e é definida como a
  constante [`MAX_INSTRUCTION_STACK_DEPTH`](https://github.com/anza-xyz/agave/blob/v3.1.8/program-runtime/src/execution_budget.rs#L8)
  de 5. Com [`MAX_INSTRUCTION_STACK_DEPTH_SIMD_0268`](https://github.com/anza-xyz/agave/blob/v3.1.8/program-runtime/src/execution_budget.rs#L10) ativo, isso aumenta para 9.

Altura de pilha 1 é a instrução de transação inicial. Cada CPI incrementa a
altura em 1. Um máximo de 5 significa que um programa pode fazer CPIs até 4
níveis de profundidade (8 níveis de profundidade com SIMD-0268).

</Callout>

### Etapa 1: Programa chama `invoke` ou `invoke_signed`

O programa chama
[`invoke`](https://github.com/anza-xyz/solana-sdk/blob/clock%40v2.2.3/program/src/program.rs#L26)
ou
[`invoke_signed`](https://github.com/anza-xyz/solana-sdk/blob/clock%40v2.2.3/program/src/program.rs#L51).
_rs`invoke`_ é um wrapper fino que chama _rs`invoke_signed`_ com um array vazio
de seeds de assinante. A função SDK serializa a _rs`Instruction`_, slice
_rs`AccountInfo`_ e seeds de assinante na memória da VM, depois aciona a
syscall.

### Etapa 2: Entrada de syscall

A VM SBF despacha para o handler de syscall
[`sol_invoke_signed_rust`](https://github.com/anza-xyz/agave/blob/v3.1.8/syscalls/src/cpi.rs#L11-L33),
que chama o ponto de entrada compartilhado:
[`cpi_common`](https://github.com/anza-xyz/agave/blob/v3.1.8/program-runtime/src/cpi.rs#L802-L925).

### Passo 3: Consumir custo de invocação

A primeira ação dentro de _rs`cpi_common`_ é
[cobrar o custo fixo de invocação](https://github.com/anza-xyz/agave/blob/v3.1.8/program-runtime/src/cpi.rs#L815-L818)
do medidor de computação compartilhado:
[`invoke_units`](https://github.com/anza-xyz/agave/blob/v3.1.8/program-runtime/src/execution_budget.rs#L108)
= 1.000 CUs (ou 946 CUs com SIMD-0339).

### Passo 4: Traduzir instrução da memória da VM

O manipulador de syscall traduz a instrução do espaço de endereçamento da VM do
programa para tipos Rust do lado do host através de
[`translate_instruction_rust`](https://github.com/anza-xyz/agave/blob/v3.1.8/program-runtime/src/cpi.rs#L529-L590),
que lê uma estrutura _rs`StableInstruction`_, valida o comprimento dos dados
contra
[`MAX_INSTRUCTION_DATA_LEN`](https://github.com/anza-xyz/agave/blob/v3.1.8/transaction-context/src/lib.rs#L33)
(10.240 bytes), e então cobra o
[custo de serialização de dados](https://github.com/anza-xyz/agave/blob/v3.1.8/program-runtime/src/cpi.rs#L551-L569).

### Passo 5: Traduzir seeds de assinante e derivar PDAs

O manipulador chama
[`translate_signers_rust`](https://github.com/anza-xyz/agave/blob/v3.1.8/program-runtime/src/cpi.rs#L619-L661).
Para cada conjunto de seeds de assinante, o runtime:

1. Verifica o número de conjuntos de seeds de assinante contra
   [`MAX_SIGNERS`](https://github.com/anza-xyz/agave/blob/v3.1.8/program-runtime/src/cpi.rs#L62)
   (16).
2. Verifica o comprimento de cada conjunto de seeds contra
   [`MAX_SEEDS`](https://github.com/anza-xyz/solana-sdk/blob/clock%40v2.2.3/pubkey/src/lib.rs#L47)
   (16 seeds por conjunto).
3. Chama
   [`Pubkey::create_program_address`](https://github.com/anza-xyz/agave/blob/v3.1.8/program-runtime/src/cpi.rs#L653-L654)
   com as seeds e o ID do programa do **chamador**. Se as seeds não produzirem
   um PDA válido, o CPI falha com
   [`BadSeeds`](https://github.com/anza-xyz/agave/blob/v3.1.8/program-runtime/src/cpi.rs#L35-L36).
4. Coleta as pubkeys PDA resultantes num vec `signers`.

Estes PDAs derivados são tratados como assinantes válidos para a instrução do
chamado.

### Passo 6: Verificar programa autorizado

Antes de prosseguir, o runtime chama
[`check_authorized_program`](https://github.com/anza-xyz/agave/blob/v3.1.8/program-runtime/src/cpi.rs#L193-L222)
para verificar se o programa alvo é permitido para CPI. Os seguintes programas
são bloqueados:

- O carregador nativo
- `bpf_loader` e `bpf_loader_deprecated`
- `bpf_loader_upgradeable` (exceto instruções específicas de gestão: `upgrade`,
  `set_authority`, `set_authority_checked` (com feature gate),
  `extend_program_checked` (com feature gate), `close`)
- Programas de pré-compilação (ed25519, secp256k1, etc.)

A violação retorna
[`ProgramNotSupported`](https://github.com/anza-xyz/agave/blob/v3.1.8/program-runtime/src/cpi.rs#L55).

### Passo 7: Verificação de privilégios (`prepare_next_instruction`)

O runtime chama
[`prepare_next_instruction`](https://github.com/anza-xyz/agave/blob/v3.1.8/program-runtime/src/invoke_context.rs#L293-L432)
que constrói a lista _rs`InstructionAccount`_ do chamado e aplica as regras de
privilégios. Consulte [Regras de privilégios](#privilege-rules) abaixo para a
tabela de decisão completa.

### Passo 8: Traduzir informações da conta

O manipulador chama `translate_accounts` que:

1. [Valida a contagem de informações da conta](https://github.com/anza-xyz/agave/blob/v3.1.8/program-runtime/src/cpi.rs#L164-L190)
   contra _rs`MAX_CPI_ACCOUNT_INFOS`_ (128, ou 255 com SIMD-0339).
2. [Cobra o custo de tradução de informações da conta](https://github.com/anza-xyz/agave/blob/v3.1.8/program-runtime/src/cpi.rs#L969-L981)
   (apenas SIMD-0339): `(num_account_infos * 80) / 250` CUs.
3. Para cada conta não executável e não duplicada, constrói um
   [`CallerAccount`](https://github.com/anza-xyz/agave/blob/v3.1.8/program-runtime/src/cpi.rs#L228-L246)
   traduzindo ponteiros da memória da VM para a memória do host. Isso inclui
   [cobrar o custo de serialização de dados por conta](https://github.com/anza-xyz/agave/blob/v3.1.8/program-runtime/src/cpi.rs#L370-L374):
   `account_data_len / cpi_bytes_per_unit` CUs.

### Passo 9: Sincronização de conta pré-CPI (chamador para chamado)

Antes de executar o chamado, o runtime sincroniza as modificações da conta do
chamador para que o chamado possa vê-las. A função
[`update_callee_account`](https://github.com/anza-xyz/agave/blob/v3.1.8/program-runtime/src/cpi.rs#L1145-L1215)
é chamada para cada conta traduzida, copiando lamports, dados e proprietário.
Consulte
[Sincronização de dados da conta](/docs/core/cpi/cpi-cost-model#account-data-synchronization)
para o mapeamento detalhado dos campos.

### Passo 10: Empilhar contexto de instrução, executar chamado e desempilhar

O runtime chama
[`process_instruction`](https://github.com/anza-xyz/agave/blob/v3.1.8/program-runtime/src/invoke_context.rs#L477-L488),
que:

1. Chama
   [`push()`](https://github.com/anza-xyz/agave/blob/v3.1.8/program-runtime/src/invoke_context.rs#L233-L265)
   para adicionar um novo frame à pilha de instruções. _rs`push()`_ impõe a
   [regra de reentrada](https://github.com/anza-xyz/agave/blob/v3.1.8/program-runtime/src/invoke_context.rs#L243-L259):
   um programa só pode chamar a si mesmo se for o chamador direto (ou seja, o
   programa A pode chamar A, mas A não pode chamar B que chama A). A violação
   retorna _rs`ReentrancyNotAllowed`_.
2. Chama
   [`process_executable_chain`](https://github.com/anza-xyz/agave/blob/v3.1.8/program-runtime/src/invoke_context.rs#L507-L602)
   que resolve o ponto de entrada do programa chamado e o invoca. O chamado
   executa com o mesmo medidor de computação compartilhado. Todo o consumo de CU
   pelo chamado reduz o orçamento restante do chamador.
3. Chama
   [`pop()`](https://github.com/anza-xyz/agave/blob/v3.1.8/transaction-context/src/lib.rs#L366-L398)
   para remover o frame do chamado e verificar se os saldos de lamport
   permanecem inalterados (_rs`UnbalancedInstruction`_ caso contrário).

### Passo 11: Sincronização de conta pós-CPI (chamado para chamador)

Após `process_instruction` retornar (o que inclui o desempilhamento), o runtime
sincroniza as alterações de volta para o chamador via
[`update_caller_account`](https://github.com/anza-xyz/agave/blob/v3.1.8/program-runtime/src/cpi.rs#L1266-L1352)
para cada conta gravável. Além disso,
[`update_caller_account_region`](https://github.com/anza-xyz/agave/blob/v3.1.8/program-runtime/src/cpi.rs#L1217-L1252)
atualiza os mapeamentos de região de memória da VM para contas cujas regiões de
dados foram alteradas. Consulte
[Sincronização de dados da conta](/docs/core/cpi/cpi-cost-model#account-data-synchronization)
para o mapeamento detalhado dos campos.

O syscall CPI retorna `0` (sucesso) ao programa chamador.
