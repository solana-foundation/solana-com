This week brought new releases across Agave, Firedancer, Solana Go, and Surfpool, plus work on validator costs, network interface support, offchain message signing, and program performance. Teams also open sourced more developer and validator tooling across the Solana ecosystem.

## Releases

Several core and developer tools published new versions this week. Builders and operators can review the release notes for each project before upgrading.

- Anza shipped Agave [v4.1.0-beta.3](https://github.com/anza-xyz/agave/releases/tag/v4.1.0-beta.3) and [v4.0.2](https://github.com/anza-xyz/agave/releases/tag/v4.0.2).
- Firedancer released [Testnet v0.1002.40103](https://github.com/firedancer-io/firedancer/releases/tag/v0.1002.40103) and [Mainnet v0.911.40002](https://github.com/firedancer-io/firedancer/releases/tag/v0.911.40002).
- The Solana Foundation shipped [Solana Go v1.21.0](https://github.com/solana-foundation/solana-go/releases/tag/v1.21.0).
- The Solana Foundation released [Surfpool v1.3.1](https://github.com/solana-foundation/surfpool/releases/tag/v1.3.1).

## Solana Improvement Documents

A new Solana Improvement Documents (SIMDs) discussion proposes adding a Sysvar account for upcoming leader information. Sysvar accounts expose read-only internal cluster state that Solana programs can call, such as the current epoch and slot, rewards parameters, and stake activations.

- The proposal would create a new Sysvar account with [details about the upcoming leader](https://github.com/solana-foundation/solana-improvement-documents/discussions/558), storing the slot numbers and addresses of the current and next validator leaders.

## Validator client work

Validator client teams continued work across Agave, Firedancer-adjacent tooling, and Mithril. The updates focus on validator cost accounting, broader hardware support, and replay implementation work.

- Anza is preparing Agave to [scale down validator costs per epoch](https://github.com/anza-xyz/agave/pull/12990) as shorter slot times get adopted by the network. Shorter slot times mean more frequent epochs; validators vote more often, and voting cost is roughly 1.6 SOL per epoch, or about every 2 days at the time of writing. The change aims to make sure validators do not pay more in the same time frame when slot times decrease.
- Anza proposed Agave support for [tagging in VLAN interfaces](https://github.com/anza-xyz/agave/pull/12920) in its XDP implementation. Agave already supports most network interfaces when routing data through XDP, including normal ethernet and GRE; VLAN support would allow Agave to work with more network hardware used by potential validators.
- Mithril is working to [implement Replay](https://github.com/Overclock-Validator/mithril/pull/243) from Solana Turbine. Replay is the section of Turbine where validators take the block assembled from shreds and replay the transactions in that block to determine whether it is valid.

Teams are also working on standards and implementations for offchain message signing across Solana clients and wallet tooling. The effort includes Anza, Solana Mobile, wallet applications, and the Solana Foundation.

- Solana teams are creating offchain message signing [standards](https://github.com/anza-xyz/wallet-standard/pull/92), [implementations](https://github.com/anza-xyz/agave/pull/12965), and [changes to the current signing hooks](https://github.com/anza-xyz/wallet-standard/pull/93).

## Solana program frameworks

Program framework work this week centered on Pinocchio and performance improvements. Both updates are aimed at developer experience and reduced compute unit usage.

- Pinocchio will use [direct account pointers](https://github.com/anza-xyz/pinocchio/pull/421) to take advantage of [SIMD-0449](https://github.com/solana-foundation/solana-improvement-documents/pull/449). Direct account pointers allow programs to read instruction accounts from memory, standardize the instruction memory layout, and improve performance with decreased compute unit usage.
- Pinocchio is putting [error conversion on the cold path](https://github.com/anza-xyz/pinocchio/pull/422). Converting error codes to error messages is not an operation that should be prioritized while the CPU runs the program; putting errors on the cold path lets the compiler prioritize “hot” operations, such as running program instructions, to improve performance and decrease compute unit usage.
