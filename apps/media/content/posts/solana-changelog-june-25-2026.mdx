## Proposed protocol changes

New Solana Improvement Documents (SIMDs) and related discussions focused on program performance, cryptography precompiles, and cryptography syscalls. The proposals are early-stage changes for the community to review.

- A new SIMD proposes swapping the current SPL Associated Token Account program for a [Pinocchio-optimized ATA program](https://github.com/solana-foundation/solana-improvement-documents/pull/567). With the successful mainnet deployment of [p-token](https://github.com/solana-foundation/solana-improvement-documents/blob/main/proposals/0266-efficient-token-program.md), the next logical step would be to similarly migrate the Associated Token Account program, which creates token accounts on Solana.
- A new SIMD proposes deprecating [three cryptography precompiles](https://github.com/solana-foundation/solana-improvement-documents/pull/568) in favor of a syscall implementation. The runtime currently handles cryptography with precompiles, and moving to system calls could address several issues while improving performance and the developer experience for Solana applications.
- A new discussion proposes to [reimplement Poseidon hash syscalls](https://github.com/solana-foundation/solana-improvement-documents/discussions/569). Poseidon hash syscalls allow more cryptography signatures and mechanisms to be used on Solana, and implementing them as system calls would make them cheaper to run in both performance and compute unit terms than deployed-program implementations.

## Validator client work

Agave and Mithril work continued across networking and validator participation. The Agave updates focus on gossip networking and XDP testing, while Mithril moved toward block verification and Alpenglow participation.

- Agave is [restricting networking addresses to IPv4 in its gossip implementation](https://github.com/anza-xyz/agave/pull/13213). Gossip initially supported IPv4 and IPv6 network addresses, but IPv6 support is not universal across devices and complicates XDP adoption by validators; sticking to IPv4 also lets Agave developers focus on other parts of the stack.
- Agave is proposing an [XDP test harness](https://github.com/anza-xyz/agave/pull/13359) to test its XDP implementation end to end. Since Agave 4.1, more validators have started adopting XDP, and integration testing helps validators quickly scope issues when they arise.
- Mithril will be verifying blocks, officially becoming a proper validator to the chain, and participating in Alpenglow, according to [Overclock’s update on X](https://x.com/OverclockSol/status/2069816448592052245?s=20).

## RPC and developer library updates

RPC 2.0 and Solana language client work advanced across read-layer performance, program metadata, React data fetching, and hardware signing support. These changes target builders working with Solana data, clients, and program integrations.

- Superbank is [adding a RocksDB Disk Cache](https://github.com/solana-rpc/superbank/pull/6). Superbank ingests Solana transaction data into a ClickHouse database and serves as an RPC read layer from that data; the RocksDB Disk Cache improves performance under heavy load.
- Solana Foundation added a web library for [fetching program metadata](https://github.com/solana-foundation/idl), including the IDL and Security.txt. These libraries allow web applications to query a published program IDL so they can interact with Solana programs properly and securely.
- Solana Kit will [use TanStack Query underneath its React implementation](https://github.com/anza-xyz/kit/pull/1759). TanStack Query supports state management for data fetches, and exposing it as a hook lets React users use it instead of hand-rolling their own implementations.
- Solana has a new [Keychain Kit Plugin](https://github.com/solana-foundation/solana-keychain/pull/162). Keychain allows users to create signers from any backend, including Ledger and YubiKey, so more hardware can sign transactions and offline messages beyond wallet extensions.

## Program framework updates

Solana program framework work included an Anchor change for teams whose program IDs are not backed by locally available private key JSON files.

- Anchor will [no longer abort program builds on private key and program ID mismatches](https://github.com/otter-sec/anchor/pull/4705). Anchor’s template generates a private key and program ID for new Solana programs, but some users deploy programs with a multisig program ID, where the private key is not available in JSON; this change allows developers to build programs regardless of whether the key was generated locally.
