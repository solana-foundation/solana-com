Na Solanie wszystkie dane są przechowywane w tzw. "kontach". Możesz myśleć o
danych na Solanie jak o publicznej bazie danych z jedną tabelą "Konta", gdzie
każdy rekord w tej tabeli to "konto". Każde konto na Solanie korzysta z tego
samego podstawowego
[typu konta](https://github.com/anza-xyz/agave/blob/v2.1.13/sdk/account/src/lib.rs#L48-L60).

![Konta](/assets/docs/core/accounts/accounts.png)

## Kluczowe informacje

- Konta mogą przechowywać do
  [10MiB](https://github.com/anza-xyz/agave/blob/v2.1.13/sdk/program/src/system_instruction.rs#L85)
  danych, które zawierają kod programu wykonywalnego lub stan programu.
- Konta wymagają
  [depozytu rent](https://github.com/anza-xyz/agave/blob/v2.1.13/sdk/rent/src/lib.rs#L93-L97)
  w lamportach (SOL), który jest proporcjonalny do ilości przechowywanych
  danych, a pełny zwrot depozytu jest możliwy po zamknięciu konta.
- Każde konto ma programowego
  [właściciela](https://github.com/anza-xyz/agave/blob/v2.1.13/sdk/account/src/lib.rs#L55).
  Tylko program będący właścicielem konta może zmieniać jego dane lub odejmować
  saldo w lamportach. Jednak każdy może zwiększyć saldo.
- **Konta Sysvar** to specjalne konta przechowujące stan klastra sieciowego.
- **Konta programowe** przechowują kod wykonywalny smart kontraktów.
- **Konta danych** są tworzone przez programy w celu przechowywania i
  zarządzania stanem programu.

## Konto

Każde konto na Solanie ma unikalny 32-bajtowy adres, często przedstawiany jako
ciąg zakodowany w base58 (np. `14grJpemFaf88c8tiVb77W7TYg2W3ir6pfkKz3YjhhZ5`).

Relacja między kontem a jego adresem działa jak para klucz-wartość, gdzie adres
jest kluczem do zlokalizowania odpowiadających danych on-chain konta. Adres
konta działa jak "unikalny identyfikator" dla każdego wpisu w tabeli "Konta".

![Adres konta](/assets/docs/core/accounts/account-address.svg)

Większość kont Solana używa klucza publicznego
[Ed25519](https://ed25519.cr.yp.to/) jako swojego adresu.

<CodeTabs storage="accounts" flags="r">

```ts !! title="Kit"
import { generateKeyPairSigner } from "@solana/kit";

// Kit does not enable extractable private keys
const keypairSigner = await generateKeyPairSigner();
console.log(keypairSigner);
```

```ts !! title="Legacy"
import { Keypair } from "@solana/web3.js";

const keypair = Keypair.generate();
console.log(`Public Key: ${keypair.publicKey}`);
console.log(`Secret Key: ${keypair.secretKey}`);
```

```rs !! title="Rust"

#[tokio::main]
async fn main() {
    let keypair = Keypair::new();
    println!("Public Key: {}", keypair.pubkey());
    println!("Secret Key: {:?}", keypair.to_bytes());
}
```

</CodeTabs>

Chociaż klucze publiczne są powszechnie używane jako adresy kont, Solana
obsługuje również funkcję zwaną [Program Derived Addresses](/docs/core/pda)
(PDA). PDA to specjalne adresy, które można deterministycznie wyprowadzić z
identyfikatora programu i opcjonalnych danych wejściowych (seeds).

<CodeTabs storage="accounts" flags="r">

```ts !! title="Kit"
import { Address, getProgramDerivedAddress } from "@solana/kit";

const programAddress = "11111111111111111111111111111111" as Address;

const seeds = ["helloWorld"];
const [pda, bump] = await getProgramDerivedAddress({
  programAddress,
  seeds
});

console.log(`PDA: ${pda}`);
console.log(`Bump: ${bump}`);
```

```ts !! title="Legacy"
import { PublicKey } from "@solana/web3.js";

const programAddress = new PublicKey("11111111111111111111111111111111");

const seeds = [Buffer.from("helloWorld")];
const [pda, bump] = await PublicKey.findProgramAddressSync(
  seeds,
  programAddress
);

console.log(`PDA: ${pda}`);
console.log(`Bump: ${bump}`);
```

```rs !! title="Rust"
use solana_sdk::pubkey; // macro
use solana_sdk::pubkey::Pubkey;

#[tokio::main]
async fn main() {
    let program_address = pubkey!("11111111111111111111111111111111");
    let seeds = [b"helloWorld".as_ref()];
    let (pda, bump) = Pubkey::find_program_address(&seeds, &program_address);
    println!("PDA: {}", pda);
    println!("Bump: {}", bump);
}
```

</CodeTabs>

### Typ konta

Konta mają maksymalny rozmiar
[10MiB](https://github.com/anza-xyz/agave/blob/v2.1.13/sdk/program/src/system_instruction.rs#L85)
i każde konto na Solanie korzysta z tego samego podstawowego typu
[Account](https://github.com/anza-xyz/agave/blob/v2.1.13/sdk/account/src/lib.rs#L48-L60).

![Typ konta](/assets/docs/core/accounts/account-type.svg)

Każde konto na Solanie zawiera następujące pola:

- `data`: Tablica bajtów przechowująca dowolne dane dla konta. W przypadku kont
  nieegzekwowalnych często przechowuje stan, który ma być odczytywany. W
  przypadku kont programowych (smart contracts) zawiera wykonywalny kod
  programu. Pole danych jest często nazywane "danymi konta".
- `executable`: Ten znacznik wskazuje, czy konto jest programem.
- `lamports`: Saldo konta w lamportach, najmniejszej jednostce SOL (1 SOL = 1
  miliard lamportów).
- `owner`: Identyfikator programu (klucz publiczny) programu, który jest
  właścicielem tego konta. Tylko program właściciela może zmieniać dane konta
  lub odejmować jego saldo w lamportach.
- `rent_epoch`: Pole dziedzictwa z czasów, gdy Solana miała mechanizm okresowego
  odejmowania lamportów z kont. Chociaż to pole nadal istnieje w typie konta,
  nie jest już używane, ponieważ pobieranie opłat za przechowywanie zostało
  wycofane.

```rust title="Base Account Type"
pub struct Account {
    /// lamports in the account
    pub lamports: u64,
    /// data held in this account
    #[cfg_attr(feature = "serde", serde(with = "serde_bytes"))]
    pub data: Vec<u8>,
    /// the program that owns this account. If executable, the program that loads this account.
    pub owner: Pubkey,
    /// this account's data contains a loaded program (and is now read-only)
    pub executable: bool,
    /// the epoch at which this account will next owe rent
    pub rent_epoch: Epoch,
}
```

### Opłata za przechowywanie

Aby przechowywać dane w łańcuchu, konta muszą również utrzymywać saldo lamportów
(SOL), które jest proporcjonalne do ilości danych przechowywanych na koncie (w
bajtach). To saldo nazywane jest "rentem", ale działa bardziej jak depozyt,
ponieważ można odzyskać całą kwotę po zamknięciu konta. Obliczenia można znaleźć
[tutaj](https://github.com/anza-xyz/agave/blob/v2.1.13/sdk/rent/src/lib.rs#L93-L97)
z użyciem tych
[stałych](https://github.com/anza-xyz/agave/blob/v2.1.13/sdk/rent/src/lib.rs#L47-L70).

Termin "rent" pochodzi z przestarzałego mechanizmu, który regularnie odejmował
lamporty z kont, które spadały poniżej progu rentu. Ten mechanizm nie jest już
aktywny.

### Właściciel programu

Na Solanie "smart kontrakty" nazywane są [programami](/docs/core/programs).
Własność programu jest kluczowym elementem modelu kont Solany. Każde konto ma
przypisany program jako właściciela. Tylko program-właściciel może:

- Zmieniać pole `data` konta
- Odejmować lamporty z salda konta

## System Program

Domyślnie wszystkie nowe konta są przypisane do
[System Program](https://github.com/anza-xyz/agave/tree/v2.1.13/programs/system/src).
System Program wykonuje kilka kluczowych zadań:

- [Tworzenie nowych kont](https://github.com/anza-xyz/agave/blob/v2.1.13/programs/system/src/system_processor.rs#L146):
  Tylko System Program może tworzyć nowe konta.
- [Alokacja przestrzeni](https://github.com/anza-xyz/agave/blob/v2.1.13/programs/system/src/system_processor.rs#L71):
  Ustawia pojemność w bajtach dla pola danych każdego konta.
- [Transfer / przypisanie własności programu](https://github.com/anza-xyz/agave/blob/v2.1.13/programs/system/src/system_processor.rs#L113):
  Po utworzeniu konta przez System Program, może on przypisać właściciela
  programu do innego konta programu. W ten sposób niestandardowe programy
  przejmują własność nowych kont utworzonych przez System Program.

Wszystkie konta "portfela" na Solanie to po prostu konta przypisane do System
Program. Saldo lamportów na tych kontach pokazuje ilość SOL posiadaną przez
portfel. Tylko konta przypisane do System Program mogą płacić opłaty
transakcyjne.

![Konto systemowe](/assets/docs/core/accounts/system-account.svg)

## Konta Sysvar

Konta Sysvar to specjalne konta o z góry określonych adresach, które zapewniają
dostęp do danych o stanie klastra. Te konta są dynamicznie aktualizowane o dane
dotyczące klastra sieciowego. Pełną listę kont Sysvar znajdziesz
[tutaj](https://docs.anza.xyz/runtime/sysvars).

## Program Account

Wdrożenie programu Solana tworzy wykonywalne konto programu. Konto programu
przechowuje wykonywalny kod programu.

Konta programów są własnością
[Loader Program](/docs/core/programs#loader-programs).

![Konto programu](/assets/docs/core/accounts/program-account-simple.svg)

Dla uproszczenia możesz traktować konto programu jako sam program. Gdy
wywołujesz instrukcje programu, określasz adres konta programu (często nazywany
"Program ID").

<Callout type="info">

Gdy wdrażasz program Solana, jest on przechowywany na koncie programu. Konta
programów są własnością [Loader Program](/docs/core/programs#loader-programs).
Istnieje kilka wersji loadera, ale wszystkie, z wyjątkiem loader-v3, przechowują
wykonywalny kod bezpośrednio na koncie programu. Loader-v3 przechowuje
wykonywalny kod w oddzielnym "koncie danych programu", a konto programu jedynie
wskazuje na nie. Gdy wdrażasz nowy program, domyślnie Solana CLI używa
najnowszej wersji loadera.

</Callout>

### Konto bufora

Loader-v3 ma specjalny typ konta do tymczasowego przechowywania programu podczas
wdrażania lub ponownego wdrażania/aktualizacji. W loader-v4 nadal istnieją
bufory, ale są to po prostu zwykłe konta programów.

### Konto danych programu

Loader-v3 działa inaczej niż wszystkie inne programy BPF Loader. Konto programu
zawiera jedynie adres konta danych programu, które przechowuje rzeczywisty
wykonywalny kod:

![Konto danych programu](/assets/docs/core/accounts/program-account-expanded.svg)

Nie myl tych kont danych programu z kontami danych programów (patrz poniżej).

## Konto danych

W Solanie kod wykonywalny programu jest przechowywany w innym koncie niż stan
programu. To podobne do tego, jak systemy operacyjne zazwyczaj mają oddzielne
pliki dla programów i ich danych.

Aby utrzymać stan, programy definiują instrukcje do tworzenia oddzielnych kont,
które są ich własnością. Każde z tych kont ma swój unikalny adres i może
przechowywać dowolne dane zdefiniowane przez program.

![Konto danych](/assets/docs/core/accounts/data-account.svg)

Zauważ, że tylko [System Program](/docs/core/accounts#system-program) może
tworzyć nowe konta. Gdy System Program utworzy konto, może następnie przekazać
lub przypisać własność nowego konta innemu programowi.

Innymi słowy, tworzenie konta danych dla niestandardowego programu wymaga dwóch
kroków:

1. Wywołanie System Program w celu utworzenia konta, a następnie przekazanie
   własności niestandardowemu programowi
2. Wywołanie niestandardowego programu, który teraz jest właścicielem konta, w
   celu zainicjowania danych konta zgodnie z instrukcją programu

Proces tworzenia konta jest często abstrakcyjny jako pojedynczy krok, ale warto
zrozumieć proces leżący u jego podstaw.

<CodeTabs storage="accounts" flags="r">

```ts !! title="Kit"
import {
  airdropFactory,
  appendTransactionMessageInstructions,
  createSolanaRpc,
  createSolanaRpcSubscriptions,
  createTransactionMessage,
  generateKeyPairSigner,
  getSignatureFromTransaction,
  lamports,
  pipe,
  sendAndConfirmTransactionFactory,
  setTransactionMessageFeePayerSigner,
  setTransactionMessageLifetimeUsingBlockhash,
  signTransactionMessageWithSigners
} from "@solana/kit";
import { getCreateAccountInstruction } from "@solana-program/system";
import {
  getInitializeMintInstruction,
  getMintSize,
  TOKEN_2022_PROGRAM_ADDRESS
} from "@solana-program/token-2022";

// Create Connection, local validator in this example
const rpc = createSolanaRpc("http://127.0.0.1:8899");
const rpcSubscriptions = createSolanaRpcSubscriptions("ws://localhost:8900");

// Generate keypairs for fee payer
const feePayer = await generateKeyPairSigner();

// Fund fee payer
await airdropFactory({ rpc, rpcSubscriptions })({
  recipientAddress: feePayer.address,
  lamports: lamports(1_000_000_000n),
  commitment: "confirmed"
});

// Generate keypair to use as address of mint
const mint = await generateKeyPairSigner();

// Get default mint account size (in bytes), no extensions enabled
const space = BigInt(getMintSize());

// Get minimum balance for rent exemption
const rent = await rpc.getMinimumBalanceForRentExemption(space).send();

// Instruction to create new account for mint (token 2022 program)
// Invokes the system program
const createAccountInstruction = getCreateAccountInstruction({
  payer: feePayer,
  newAccount: mint,
  lamports: rent,
  space,
  programAddress: TOKEN_2022_PROGRAM_ADDRESS
});

// Instruction to initialize mint account data
// Invokes the token 2022 program
const initializeMintInstruction = getInitializeMintInstruction({
  mint: mint.address,
  decimals: 9,
  mintAuthority: feePayer.address
});

const instructions = [createAccountInstruction, initializeMintInstruction];

// Get latest blockhash to include in transaction
const { value: latestBlockhash } = await rpc.getLatestBlockhash().send();

// Create transaction message
const transactionMessage = pipe(
  createTransactionMessage({ version: 0 }), // Create transaction message
  (tx) => setTransactionMessageFeePayerSigner(feePayer, tx), // Set fee payer
  (tx) => setTransactionMessageLifetimeUsingBlockhash(latestBlockhash, tx), // Set transaction blockhash
  (tx) => appendTransactionMessageInstructions(instructions, tx) // Append instructions
);

// Sign transaction message with required signers (fee payer and mint keypair)
const signedTransaction =
  await signTransactionMessageWithSigners(transactionMessage);

// Send and confirm transaction
await sendAndConfirmTransactionFactory({ rpc, rpcSubscriptions })(
  signedTransaction,
  { commitment: "confirmed" }
);

// Get transaction signature
const transactionSignature = getSignatureFromTransaction(signedTransaction);

console.log("Mint Address:", mint.address);
console.log("Transaction Signature:", transactionSignature);

const accountInfo = await rpc.getAccountInfo(mint.address).send();
console.log(accountInfo);
```

```ts !! title="Legacy"
import {
  Connection,
  Keypair,
  sendAndConfirmTransaction,
  SystemProgram,
  Transaction,
  LAMPORTS_PER_SOL
} from "@solana/web3.js";
import {
  createInitializeMintInstruction,
  TOKEN_2022_PROGRAM_ID,
  MINT_SIZE,
  getMinimumBalanceForRentExemptMint
} from "@solana/spl-token";

// Create connection to local validator
const connection = new Connection("http://localhost:8899", "confirmed");
const recentBlockhash = await connection.getLatestBlockhash();

// Generate a new keypair for the fee payer
const feePayer = Keypair.generate();

// Airdrop 1 SOL to fee payer
const airdropSignature = await connection.requestAirdrop(
  feePayer.publicKey,
  LAMPORTS_PER_SOL
);
await connection.confirmTransaction({
  blockhash: recentBlockhash.blockhash,
  lastValidBlockHeight: recentBlockhash.lastValidBlockHeight,
  signature: airdropSignature
});

// Generate keypair to use as address of mint
const mint = Keypair.generate();

const createAccountInstruction = SystemProgram.createAccount({
  fromPubkey: feePayer.publicKey,
  newAccountPubkey: mint.publicKey,
  space: MINT_SIZE,
  lamports: await getMinimumBalanceForRentExemptMint(connection),
  programId: TOKEN_2022_PROGRAM_ID
});

const initializeMintInstruction = createInitializeMintInstruction(
  mint.publicKey, // mint pubkey
```

```rs !! title="Rust"
use anyhow::Result;
use solana_client::nonblocking::rpc_client::RpcClient;
use solana_sdk::{
    commitment_config::CommitmentConfig,
    program_pack::Pack,
    signature::{Keypair, Signer},
    system_instruction::create_account,
    transaction::Transaction,
};
use spl_token_2022::{id as token_2022_program_id, instruction::initialize_mint, state::Mint};

#[tokio::main]
async fn main() -> Result<()> {
    // Create connection to local validator
    let client = RpcClient::new_with_commitment(
        String::from("http://127.0.0.1:8899"),
        CommitmentConfig::confirmed(),
    );
    let recent_blockhash = client.get_latest_blockhash().await?;

    // Generate a new keypair for the fee payer
    let fee_payer = Keypair::new();

    // Airdrop 1 SOL to fee payer
    let airdrop_signature = client
        .request_airdrop(&fee_payer.pubkey(), 1_000_000_000)
        .await?;
    client.confirm_transaction(&airdrop_signature).await?;

    loop {
        let confirmed = client.confirm_transaction(&airdrop_signature).await?;
        if confirmed {
            break;
        }
    }

    // Generate keypair to use as address of mint
    let mint = Keypair::new();

    let space = Mint::LEN;
    let rent = client.get_minimum_balance_for_rent_exemption(space).await?;

    // Create account instruction
    let create_account_instruction = create_account(
        &fee_payer.pubkey(),      // fee payer
        &mint.pubkey(),           // mint address
        rent,                     // rent
        space as u64,             // space
        &token_2022_program_id(), // program id
    );

    // Initialize mint instruction
    let initialize_mint_instruction = initialize_mint(
        &token_2022_program_id(),
        &mint.pubkey(),            // mint address
        &fee_payer.pubkey(),       // mint authority
        Some(&fee_payer.pubkey()), // freeze authority
        9,                         // decimals
    )?;

    // Create transaction and add instructions
    let transaction = Transaction::new_signed_with_payer(
        &[create_account_instruction, initialize_mint_instruction],
        Some(&fee_payer.pubkey()),
        &[&fee_payer, &mint],
        recent_blockhash,
    );

    // Send and confirm transaction
    let transaction_signature = client.send_and_confirm_transaction(&transaction).await?;

    println!("Mint Address: {}", mint.pubkey());
    println!("Transaction Signature: {}", transaction_signature);

    let account_info = client.get_account(&mint.pubkey()).await?;
    println!("{:#?}", account_info);

    Ok(())
}
```
