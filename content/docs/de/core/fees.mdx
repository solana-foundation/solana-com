Jede Solana-Transaktion erfordert eine Basisgebühr (SOL), um die Validatoren für
verarbeitende Transaktionen zu entschädigen. Eine optionale Priorisierungsgebühr ist ebenfalls verfügbar,
um die Wahrscheinlichkeit zu erhöhen, dass die Transaktion vom aktuellen Führer
(Validator) verarbeitet wird.

## Wichtige Punkte

- Die **Basisgebühr** für eine Transaktion beträgt mindestens 5000 Lamports pro Signatur
  auf der Transaktion.
- Die **Priorisierungsgebühr** (optional) ist eine zusätzliche Gebühr, die an den
  Validator gezahlt wird, um die Wahrscheinlichkeit zu erhöhen, dass die Transaktion verarbeitet wird.
- Die Priorisierungsgebühr wird berechnet als: **(Recheneinheitenlimit \* Recheneinheitspreis)**.
- Das **Recheneinheitenlimit** ist die maximale Anzahl von Recheneinheiten, die die
  Transaktion verbrauchen kann.
- Der **Recheneinheitspreis** ist der Preis pro Recheneinheit, in Mikro-Lamports.
- 1.000.000 Mikro-Lamports = 1 Lamport
- Der Zahler der Transaktionsgebühr muss ein Konto sein, das dem Systemprogramm gehört.

## Basis-Transaktionsgebühr

Die Basisgebühr ist die Mindestkosten für das Senden einer Transaktion. Die Kosten betragen 5000
Lamports pro Signatur, die in der Transaktion enthalten ist.

Die Basisgebühr wird automatisch vom Zahler der Transaktionsgebühr bezahlt, der der
erste Unterzeichner der Transaktion ist. Der Zahler der Gebühr muss ein Konto sein, das dem
Systemprogramm gehört.

- [50% Verbrennung](https://github.com/anza-xyz/agave/blob/v2.1.13/sdk/fee-calculator/src/lib.rs#L70):
  Die Hälfte der Basisgebühr wird verbrannt.
- [50% Verteilung](https://github.com/anza-xyz/agave/blob/e621336acad4f5d6e5b860eaa1b074b01c99253c/runtime/src/bank/fee_distribution.rs#L58-L62):
  Die Hälfte wird an den Validator gezahlt, der die Transaktion verarbeitet hat.

## Priorisierungsgebühr

Die Priorisierungsgebühr ist eine optionale Gebühr, die an den Validator gezahlt wird, um die
Wahrscheinlichkeit zu erhöhen, dass die Transaktion verarbeitet wird.

- [SIMD-0096](https://github.com/solana-foundation/solana-improvement-documents/blob/main/proposals/0096-reward-collected-priority-fee-in-entirety.md):
  100% der Prioritätsgebühr wird direkt an den Validator gezahlt, der die
  Transaktion verarbeitet.

### Recheneinheiten und Limits

Wenn eine Transaktion ausgeführt wird, verbraucht sie Rechenressourcen, die in
Recheneinheiten (CU) gemessen werden. Jede Anweisung zieht von dem Recheneinheitenbudget der
Transaktion ab.

- Maximales Limit: Eine Transaktion kann bis zu
  [1,4 Millionen Recheneinheiten](https://github.com/anza-xyz/agave/blob/v2.1.13/compute-budget/src/compute_budget_limits.rs#L14) verwenden.
- Standardlimit: Standardmäßig ist jede Anweisung auf
  [200.000 Recheneinheiten](https://github.com/anza-xyz/agave/blob/v2.1.13/compute-budget/src/compute_budget_limits.rs#L10) beschränkt.
- Benutzerdefiniertes Limit: Sie können ein spezifisches Recheneinheitenlimit anfordern, indem Sie eine
  `SetComputeUnitLimit`-Anweisung in Ihre Transaktion einfügen.

Für zusätzliche Details zur Nutzung von Recheneinheiten:

- [ComputeBudget Typ](https://github.com/anza-xyz/agave/blob/v2.1.13/compute-budget/src/compute_budget.rs#L22-L130)
- [Standardwerte für den Verbrauch von Recheneinheiten](https://github.com/anza-xyz/agave/blob/v2.1.13/compute-budget/src/compute_budget.rs#L149-L197)

Verweisen Sie auf den
[Wie man optimale Rechenleistung anfordert](/developers/guides/advanced/how-to-request-optimal-compute)
Leitfaden für weitere Details zur Nutzung von Recheneinheiten.

### Recheneinheitspreis

Der Recheneinheitspreis, ausgedrückt in Mikro-Lamports, ist der optionale Preis,
der pro Recheneinheit gezahlt wird, um die Priorisierungsgebühr zu berechnen.

<Callout>1.000.000 Mikro-Lamports = 1 Lamport</Callout>

Verwenden Sie die folgenden Ressourcen, um Echtzeitempfehlungen zum aktuellen
Recheneinheitspreis zu erhalten:

- [Prioritätsgebühren-API](https://docs.helius.dev/solana-apis/priority-fee-api) von
  Helius
- [Globaler Prioritätsgebühren-Tracker](https://triton.one/solana-prioritization-fees/)
  von Triton

Verweisen Sie auf den
[Wie man Prioritätsgebühren verwendet](/developers/guides/advanced/how-to-use-priority-fees)
Leitfaden für weitere Details zu Prioritätsgebühren.

### Berechnung der Priorisierungsgebühr

Die Priorisierungsgebühr wird berechnet als:

**Priorisierungsgebühr = Recheneinheitenlimit × Recheneinheitspreis**

Verwenden Sie die folgenden Anweisungen, um das Recheneinheitenlimit und den Preis für eine
Transaktion festzulegen:

- [`SetComputeUnitLimit`](https://github.com/anza-xyz/agave/blob/v2.1.13/sdk/src/compute_budget.rs#L42-L44)
  um ein spezifisches Einheitlimit festzulegen.
- [`SetComputeUnitPrice`](https://github.com/anza-xyz/agave/blob/v2.1.13/sdk/src/compute_budget.rs#L48-L50)
  um den Preis pro Recheneinheit zu definieren.

<Callout>

Wenn diese Anweisungen nicht bereitgestellt werden, verwendet die Transaktion automatisch das
Standard-Recheneinheitenlimit mit einem Recheneinheitspreis von 0 (keine Priorisierungsgebühr).

Die Prioritätsgebühr basiert auf den Recheneinheiten, die für die Transaktion angefordert werden, nicht
auf den tatsächlich von der Transaktion verwendeten Recheneinheiten. Wenn Sie ein Recheneinheitenlimit
festlegen, das zu hoch ist oder den Standardbetrag verwenden, zahlen Sie möglicherweise für ungenutzte
Recheneinheiten.

</Callout>

### Beispiele

Nachfolgend finden Sie Beispiele für die Anweisungen, die verwendet werden, um das Recheneinheitenlimit und
den Preis für eine Transaktion festzulegen.

#### Rust

Das `solana-sdk` bietet Methoden über den
[`ComputeBudgetInstruction`](https://github.com/anza-xyz/agave/blob/v2.1.13/sdk/src/compute_budget.rs#L17-L31)
Typ:

```rust
let limit_instruction = ComputeBudgetInstruction::set_compute_unit_limit(300_000);

let price_instruction = ComputeBudgetInstruction::set_compute_unit_price(1);
```

#### JavaScript

Die `@solana/web3.js`-Bibliothek (v1) bietet Methoden über die
[`ComputeBudgetProgram`](https://solana-labs.github.io/solana-web3.js/v1.x/classes/ComputeBudgetProgram.html)
Klasse:

```js
const limitInstruction = ComputeBudgetProgram.setComputeUnitLimit({
  units: 300_000,
});

const priceInstruction = ComputeBudgetProgram.setComputeUnitPrice({
  microLamports: 1,
});
```
