В Solana все данные хранятся в так называемых "аккаунтах". Вы можете представить
данные в Solana как публичную базу данных с единственной таблицей "Аккаунты",
где каждая запись в этой таблице является "аккаунтом". Каждый аккаунт Solana
имеет одинаковый базовый
[тип Account](https://github.com/anza-xyz/agave/blob/v2.1.13/sdk/account/src/lib.rs#L48-L60).

![Аккаунты](/assets/docs/core/accounts/accounts.png)

## Ключевые моменты

- Аккаунты могут хранить до
  [10MiB](https://github.com/anza-xyz/agave/blob/v2.1.13/sdk/program/src/system_instruction.rs#L85)
  данных, которые содержат либо исполняемый код программы, либо состояние
  программы.
- Аккаунты требуют
  [депозит rent](https://github.com/anza-xyz/agave/blob/v2.1.13/sdk/rent/src/lib.rs#L93-L97)
  в lamport (SOL), который пропорционален объему хранимых данных, и вы можете
  полностью вернуть его при закрытии аккаунта.
- Каждый аккаунт имеет программу-
  [владельца](https://github.com/anza-xyz/agave/blob/v2.1.13/sdk/account/src/lib.rs#L55).
  Только программа, владеющая аккаунтом, может изменять его данные или уменьшать
  его баланс в lamport. Но увеличить баланс может кто угодно.
- **Аккаунты Sysvar** — это специальные аккаунты, которые хранят состояние
  сетевого кластера.
- **Программные аккаунты** хранят исполняемый код смарт-контрактов.
- **Аккаунты данных** создаются программами для хранения и управления состоянием
  программы.

## Аккаунт

Каждый аккаунт в Solana имеет уникальный 32-байтовый адрес, часто отображаемый
как строка в кодировке base58 (например,
`14grJpemFaf88c8tiVb77W7TYg2W3ir6pfkKz3YjhhZ5`).

Отношение между аккаунтом и его адресом работает как пара ключ-значение, где
адрес является ключом для поиска соответствующих данных аккаунта в блокчейне.
Адрес аккаунта действует как "уникальный идентификатор" для каждой записи в
таблице "Аккаунты".

![Адрес аккаунта](/assets/docs/core/accounts/account-address.svg)

Большинство аккаунтов Solana используют открытый ключ
[Ed25519](https://ed25519.cr.yp.to/) в качестве своего адреса.

<CodeTabs storage="accounts" flags="r">

```ts !! title="Kit"
import { generateKeyPairSigner } from "@solana/kit";

// Kit does not enable extractable private keys
const keypairSigner = await generateKeyPairSigner();
console.log(keypairSigner);
```

```ts !! title="Legacy"
import { Keypair } from "@solana/web3.js";

const keypair = Keypair.generate();
console.log(`Public Key: ${keypair.publicKey}`);
console.log(`Secret Key: ${keypair.secretKey}`);
```

```rs !! title="Rust"
use solana_sdk::signer::{keypair::Keypair, Signer};

#[tokio::main]
async fn main() {
    let keypair = Keypair::new();
    println!("Public Key: {}", keypair.pubkey());
    println!("Secret Key: {:?}", keypair.to_bytes());
}
```

</CodeTabs>

Хотя открытые ключи обычно используются в качестве адресов аккаунтов, Solana
также поддерживает функцию, называемую
[Program Derived Addresses](/docs/core/pda) (PDA). PDA — это специальные адреса,
которые можно детерминированно получить из идентификатора программы и
дополнительных входных данных (seed).

<CodeTabs storage="accounts" flags="r">

```ts !! title="Kit"
import { Address, getProgramDerivedAddress } from "@solana/kit";

const programAddress = "11111111111111111111111111111111" as Address;

const seeds = ["helloWorld"];
const [pda, bump] = await getProgramDerivedAddress({
  programAddress,
  seeds
});

console.log(`PDA: ${pda}`);
console.log(`Bump: ${bump}`);
```

```ts !! title="Legacy"
import { PublicKey } from "@solana/web3.js";

const programAddress = new PublicKey("11111111111111111111111111111111");

const seeds = [Buffer.from("helloWorld")];
const [pda, bump] = await PublicKey.findProgramAddressSync(
  seeds,
  programAddress
);

console.log(`PDA: ${pda}`);
console.log(`Bump: ${bump}`);
```

```rs !! title="Rust"
use solana_sdk::pubkey; // macro
use solana_sdk::pubkey::Pubkey;

#[tokio::main]
async fn main() {
    let program_address = pubkey!("11111111111111111111111111111111");
    let seeds = [b"helloWorld".as_ref()];
    let (pda, bump) = Pubkey::find_program_address(&seeds, &program_address);
    println!("PDA: {}", pda);
    println!("Bump: {}", bump);
}
```

</CodeTabs>

### Тип аккаунта

Аккаунты имеют максимальный размер
[10MiB](https://github.com/anza-xyz/agave/blob/v2.1.13/sdk/program/src/system_instruction.rs#L85),
и каждый аккаунт в Solana имеет один и тот же базовый тип
[Account](https://github.com/anza-xyz/agave/blob/v2.1.13/sdk/account/src/lib.rs#L48-L60).

![Тип аккаунта](/assets/docs/core/accounts/account-type.svg)

Каждый аккаунт в Solana имеет следующие поля:

- `data`: Массив байтов, который хранит произвольные данные для аккаунта. Для
  неисполняемых аккаунтов это часто хранит состояние, которое предназначено для
  чтения. Для программных аккаунтов (смарт-контрактов) это содержит исполняемый
  программный код. Поле данных обычно называют "данными аккаунта".
- `executable`: Этот флаг показывает, является ли аккаунт программой.
- `lamports`: Баланс аккаунта в lamport, наименьшей единице SOL (1 SOL = 1
  миллиард lamport).
- `owner`: Идентификатор программы (открытый ключ) программы, которая владеет
  этим аккаунтом. Только программа-владелец может изменять данные аккаунта или
  уменьшать его баланс в lamport.
- `rent_epoch`: Устаревшее поле со времен, когда в Solana существовал механизм,
  периодически списывающий lamport с аккаунтов. Хотя это поле все еще существует
  в типе Account, оно больше не используется с тех пор, как сбор rent был
  отменен.

```rust title="Base Account Type"
pub struct Account {
    /// lamports in the account
    pub lamports: u64,
    /// data held in this account
    #[cfg_attr(feature = "serde", serde(with = "serde_bytes"))]
    pub data: Vec<u8>,
    /// the program that owns this account. If executable, the program that loads this account.
    pub owner: Pubkey,
    /// this account's data contains a loaded program (and is now read-only)
    pub executable: bool,
    /// the epoch at which this account will next owe rent
    pub rent_epoch: Epoch,
}
```

<CodeTabs storage="accounts" flags="r">

```ts !! title="Kit"
import {
  airdropFactory,
  createSolanaRpc,
  createSolanaRpcSubscriptions,
  generateKeyPairSigner,
  lamports
} from "@solana/kit";

// Create a connection to Solana cluster
const rpc = createSolanaRpc("http://localhost:8899");
const rpcSubscriptions = createSolanaRpcSubscriptions("ws://localhost:8900");

// Generate a new keypair
const keypair = await generateKeyPairSigner();
console.log(`Public Key: ${keypair.address}`);

// Funding an address with SOL automatically creates an account
const signature = await airdropFactory({ rpc, rpcSubscriptions })({
  recipientAddress: keypair.address,
  lamports: lamports(1_000_000_000n),
  commitment: "confirmed"
});

const accountInfo = await rpc.getAccountInfo(keypair.address).send();
console.log(accountInfo);
```

```ts !! title="Legacy"
import { Keypair, Connection, LAMPORTS_PER_SOL } from "@solana/web3.js";

// Generate a new keypair
const keypair = Keypair.generate();
console.log(`Public Key: ${keypair.publicKey}`);

// Create a connection to the Solana cluster
const connection = new Connection("http://localhost:8899", "confirmed");

// Funding an address with SOL automatically creates an account
const signature = await connection.requestAirdrop(
  keypair.publicKey,
  LAMPORTS_PER_SOL
);
await connection.confirmTransaction(signature, "confirmed");

const accountInfo = await connection.getAccountInfo(keypair.publicKey);
console.log(JSON.stringify(accountInfo, null, 2));
```

```rs !! title="Rust"
use anyhow::Result;
use solana_client::nonblocking::rpc_client::RpcClient;
use solana_sdk::{
    commitment_config::CommitmentConfig,
    native_token::LAMPORTS_PER_SOL,
    signer::{keypair::Keypair, Signer},
};

#[tokio::main]
async fn main() -> Result<()> {
    // Generate a new keypair
    let keypair = Keypair::new();
    println!("Public Key: {}", keypair.pubkey());

    // Create a connection to Solana cluster
    let connection = RpcClient::new_with_commitment(
        "http://localhost:8899".to_string(),
        CommitmentConfig::confirmed(),
    );

    // Funding an address with SOL automatically creates an account
    let signature = connection
        .request_airdrop(&keypair.pubkey(), LAMPORTS_PER_SOL)
        .await?;
    connection.confirm_transaction(&signature).await?;

    let account_info = connection.get_account(&keypair.pubkey()).await?;
    println!("{:#?}", account_info);

    Ok(())
}
```

### Rent

Для хранения данных в блокчейне аккаунты также должны поддерживать баланс в
lamport (SOL), пропорциональный объему данных, хранящихся в аккаунте (в байтах).
Этот баланс называется "rent", но работает скорее как депозит, поскольку вы
можете вернуть полную сумму при закрытии аккаунта. Вы можете найти расчет
[здесь](https://github.com/anza-xyz/agave/blob/v2.1.13/sdk/rent/src/lib.rs#L93-L97)
с использованием этих
[констант](https://github.com/anza-xyz/agave/blob/v2.1.13/sdk/rent/src/lib.rs#L47-L70).

Термин "rent" происходит от устаревшего механизма, который регулярно вычитал
lamport из аккаунтов, которые падали ниже порога аренды. Этот механизм больше не
активен.

### Владелец программы

В Solana "умные контракты" называются [программами](/docs/core/programs).
Владение программой является ключевой частью модели аккаунтов Solana. У каждого
аккаунта есть назначенная программа в качестве владельца. Только
программа-владелец может:

- Изменять поле `data` аккаунта
- Вычитать lamport из баланса аккаунта

## Системная программа

По умолчанию все новые аккаунты принадлежат
[Системной программе](https://github.com/anza-xyz/agave/tree/v2.1.13/programs/system/src).
Системная программа выполняет несколько ключевых функций:

- [Создание нового аккаунта](https://github.com/anza-xyz/agave/blob/v2.1.13/programs/system/src/system_processor.rs#L146):
  Только Системная программа может создавать новые аккаунты.
- [Выделение пространства](https://github.com/anza-xyz/agave/blob/v2.1.13/programs/system/src/system_processor.rs#L71):
  Устанавливает емкость в байтах для поля данных каждого аккаунта.
- [Передача / Назначение владения программой](https://github.com/anza-xyz/agave/blob/v2.1.13/programs/system/src/system_processor.rs#L113):
  После того как Системная программа создает аккаунт, она может переназначить
  указанного владельца программы на другой программный аккаунт. Именно так
  пользовательские программы берут на себя владение новыми аккаунтами,
  созданными Системной программой.

Все аккаунты "кошельков" в Solana — это просто аккаунты, принадлежащие Системной
программе. Баланс lamport в этих аккаунтах показывает количество SOL,
принадлежащих кошельку. Только аккаунты, принадлежащие Системной программе,
могут оплачивать комиссии за транзакции.

![Системный аккаунт](/assets/docs/core/accounts/system-account.svg)

## Аккаунты Sysvar

Аккаунты Sysvar — это специальные аккаунты с предопределенными адресами, которые
предоставляют доступ к данным о состоянии кластера. Эти аккаунты динамически
обновляются данными о сетевом кластере. Полный список аккаунтов Sysvar можно
найти [здесь](https://docs.anza.xyz/runtime/sysvars).

<CodeTabs storage="accounts" flags="r">

```ts !! title="Kit"
import { Address, createSolanaRpc } from "@solana/kit";

const rpc = createSolanaRpc("https://api.mainnet-beta.solana.com");

const SYSVAR_CLOCK_ADDRESS =
  "SysvarC1ock11111111111111111111111111111111" as Address;

const accountInfo = await rpc
  .getAccountInfo(SYSVAR_CLOCK_ADDRESS, { encoding: "base64" })
  .send();
console.log(accountInfo);
```


## Программный аккаунт

Развертывание программы Solana создает исполняемый программный аккаунт.
Программный аккаунт хранит исполняемый код программы.

Программные аккаунты принадлежат
[Загрузчику программ](/docs/core/programs#loader-programs).

![Программный аккаунт](/assets/docs/core/accounts/program-account-simple.svg)

Для простоты вы можете рассматривать программный аккаунт как саму программу.
Когда вы вызываете инструкции программы, вы указываете адрес программного
аккаунта (обычно называемый "Program ID").

<CodeTabs storage="accounts" flags="r">

```ts !! title="Kit"
import { Address, createSolanaRpc } from "@solana/kit";

const rpc = createSolanaRpc("https://api.mainnet-beta.solana.com");

const programId = "TokenkegQfeZyiNwAJbNbGKPFXCWuBvf9Ss623VQ5DA" as Address;

const accountInfo = await rpc
  .getAccountInfo(programId, { encoding: "base64" })
  .send();
console.log(accountInfo);
```

<Callout type="info">

Когда вы развертываете программу Solana, она сохраняется в программном аккаунте.
Программные аккаунты принадлежат
[Загрузчику программ](/docs/core/programs#loader-programs). Существует несколько
версий загрузчика, но все, кроме loader-v3, хранят исполняемый код
непосредственно в программном аккаунте. Loader-v3 хранит исполняемый код в
отдельном "аккаунте данных программы", а программный аккаунт просто указывает на
него. При развертывании новой программы Solana CLI по умолчанию использует
последнюю версию загрузчика.

</Callout>

### Буферный аккаунт

Loader-v3 имеет специальный тип аккаунта для временного размещения загрузки
программы во время развертывания или повторного развертывания/обновления. В
loader-v4 буферы все еще существуют, но они являются обычными программными
аккаунтами.

### Аккаунт данных программы

Loader-v3 работает иначе, чем все остальные программы загрузчика BPF. Аккаунт
программы содержит только адрес аккаунта данных программы, который хранит
фактический исполняемый код:

![Аккаунт данных программы](/assets/docs/core/accounts/program-account-expanded.svg)

Не путайте эти аккаунты данных программы с аккаунтами данных программ (см.
ниже).

## Аккаунт данных

В Solana исполняемый код программы хранится в отдельном аккаунте, отличном от
состояния программы. Это похоже на то, как операционные системы обычно имеют
отдельные файлы для программ и их данных.

Для поддержания состояния программы определяют инструкции для создания отдельных
аккаунтов, которыми они владеют. Каждый из этих аккаунтов имеет свой уникальный
адрес и может хранить любые произвольные данные, определенные программой.

![Аккаунт данных](/assets/docs/core/accounts/data-account.svg)

Обратите внимание, что только
[Системная программа](/docs/core/accounts#system-program) может создавать новые
аккаунты. После того как Системная программа создает аккаунт, она может передать
или назначить владение новым аккаунтом другой программе.

Другими словами, создание аккаунта данных для пользовательской программы
включает два шага:

1. Вызвать Системную программу для создания аккаунта, затем передать владение
   пользовательской программе
2. Вызвать пользовательскую программу, которая теперь владеет аккаунтом, чтобы
   инициализировать данные аккаунта в соответствии с инструкцией программы

Этот процесс создания аккаунта часто абстрагируется как один шаг, но полезно
понимать его базовый процесс.

<CodeTabs storage="accounts" flags="r">

```ts !! title="Kit"
import {
  airdropFactory,
  appendTransactionMessageInstructions,
  createSolanaRpc,
  createSolanaRpcSubscriptions,
  createTransactionMessage,
  generateKeyPairSigner,
  getSignatureFromTransaction,
  lamports,
  pipe,
  sendAndConfirmTransactionFactory,
  setTransactionMessageFeePayerSigner,
  setTransactionMessageLifetimeUsingBlockhash,
  signTransactionMessageWithSigners
} from "@solana/kit";
import { getCreateAccountInstruction } from "@solana-program/system";
import {
  getInitializeMintInstruction,
  getMintSize,
  TOKEN_2022_PROGRAM_ADDRESS
} from "@solana-program/token-2022";

// Create Connection, local validator in this example
const rpc = createSolanaRpc("http://127.0.0.1:8899");
const rpcSubscriptions = createSolanaRpcSubscriptions("ws://localhost:8900");

// Generate keypairs for fee payer
const feePayer = await generateKeyPairSigner();

// Fund fee payer
await airdropFactory({ rpc, rpcSubscriptions })({
  recipientAddress: feePayer.address,
  lamports: lamports(1_000_000_000n),
  commitment: "confirmed"
});
```

```rs !! title="Rust"
#[tokio::main]
async fn main() -> Result<()> {
    // Create connection to local validator
    let client = RpcClient::new_with_commitment(
        String::from("http://127.0.0.1:8899"),
        CommitmentConfig::confirmed(),
    );
    let recent_blockhash = client.get_latest_blockhash().await?;

    // Generate a new keypair for the fee payer
    let fee_payer = Keypair::new();

    // Airdrop 1 SOL to fee payer
    let airdrop_signature = client
        .request_airdrop(&fee_payer.pubkey(), 1_000_000_000)
        .await?;
    client.confirm_transaction(&airdrop_signature).await?;

    loop {
        let confirmed = client.confirm_transaction(&airdrop_signature).await?;
        if confirmed {
            break;
        }
    }

    // Generate keypair to use as address of mint
    let mint = Keypair::new();

    let space = Mint::LEN;
    let rent = client.get_minimum_balance_for_rent_exemption(space).await?;

    // Create account instruction
    let create_account_instruction = create_account(
        &fee_payer.pubkey(),      // fee payer
        &mint.pubkey(),           // mint address

        space as u64,             // space
        &token_2022_program_id(), // program id
    );

    // Initialize mint instruction
    let initialize_mint_instruction = initialize_mint(
        &token_2022_program_id(),
        &mint.pubkey(),            // mint address
        &fee_payer.pubkey(),       // mint authority
        Some(&fee_payer.pubkey()), // freeze authority
        9,                         // decimals
    )?;

    // Create transaction and add instructions
    let transaction = Transaction::new_signed_with_payer(
        &[create_account_instruction, initialize_mint_instruction],
        Some(&fee_payer.pubkey()),
        &[&fee_payer, &mint],

    );

    let transaction_signature = client.send_and_confirm_transaction(&transaction).await?;

    println!("Transaction Signature: {}", transaction_signature);

    Ok(())
}
```
